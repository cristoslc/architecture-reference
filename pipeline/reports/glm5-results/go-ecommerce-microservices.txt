## Architecture Classification Analysis

### Primary: **Microservices Architecture**

The repository exhibits clear microservices architecture with the following evidence:

**Deployment Independence:**
- Three independently deployable services: `catalogreadservice`, `catalogwriteservice`, `orderservice`
- Each service has its own `go.mod` file, `Dockerfile`, and can be built/deployed independently
- Docker compose orchestration with separate infrastructure services (RabbitMQ, PostgreSQL, MongoDB, EventStoreDB, Redis, Elasticsearch)

**Database-per-Service:**
- `catalogwriteservice` → PostgreSQL (`catalogs_write_service` database)
- `catalogreadservice` → MongoDB + Redis + Elasticsearch  
- `orderservice` → MongoDB + EventStoreDB
- Each service owns its data completely

**Service Communication:**
- Asynchronous messaging via RabbitMQ (integration events like `ProductCreatedV1`, `OrderCreatedV1`)
- Synchronous gRPC for internal service calls (protobuf definitions in `api/protobuf/`)
- REST endpoints for external APIs

**Evidence from configs** (`config.development.json` files):
```json
"appOptions": { "serviceName": "catalogwriteservice" }
"grpcOptions": { "port": ":6003" }
"echoHttpOptions": { "port": ":7000" }
```

### Not Classified As:

- **Layered Architecture**: No horizontal layering (presentation/business/data). Instead, vertical slices organize by features.
- **Modular Monolith**: Services are independently deployable with separate databases, not a single deployable unit.
- **Hexagonal Architecture**: While there are ports/adapters patterns for infrastructure (repositories, gRPC), the primary organization is by vertical slices and bounded contexts, not ports-and-adapters.
- **Serverless**: No serverless functions (AWS Lambda, etc.), all services are containerized applications.

### Architecture Summary

This is a **Microservices architecture** with strong **Event-Driven**, **CQRS**, and **DDD** influences. Each service is independently deployable, owns its database, and communicates asynchronously via RabbitMQ integration events. The OrderService implements Event Sourcing with projections to read models, while the Catalog services follow a simpler CRUD pattern with separate read/write services. Code is organized using **Vertical Slice Architecture** within each service, grouping by features/use cases rather than technical layers.
