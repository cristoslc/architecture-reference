## Architecture Analysis

**Forem** is a classic **Layered Architecture** with Event-Driven elements for background processing.

### Primary Style: Layered

The codebase exhibits clear horizontal layering characteristic of Rails MVC applications:

1. **Presentation Layer** (`app/views/`, `app/decorators/`, `app/helpers/`) - 55+ view directories with templates, decorators for view-specific logic, and helper modules
2. **Controller Layer** (`app/controllers/`) - 91+ controllers organized by domain (articles, comments, users, admin, api/v0, api/v1)
3. **Service/Business Logic Layer** (`app/services/`) - 268 service objects organized into domain namespaces like `Articles::Creator`, `Notifications::NewFollower::Send`, `Users::Update`
4. **Data Access Layer** (`app/models/`) - 119 ActiveRecord models including Article, User, Comment, Organization with relationships and data validations
5. **Policy Layer** (`app/policies/`) - Authorization using Pundit with ApplicationPolicy and domain-specific policies

Evidence:
- Standard Rails directory structure: `app/{controllers,models,views,services,policies,decorators,helpers}`
- ApplicationController inherits from ActionController::Base (app/controllers/application_controller.rb:1)
- ApplicationRecord as base model class (app/models/application_record.rb)
- Service objects follow pattern: `Module::Class.call(...)` (app/services/articles/creator.rb:3-4)

### Secondary Style: Event-Driven

Forem uses background job processing extensively via Sidekiq:

**Evidence:**
- `app/workers/` contains 48 worker directories with namespaced workers
- Workers use `include Sidekiq::Job` (app/workers/notifications/new_follower_worker.rb:3)
- Priority queue configuration (config/sidekiq.yml:1-6):
  ```yaml
  :queues:
    - ["default", 1]
    - ["low_priority", 10]
    - ["medium_priority", 100]
    - ["high_priority", 1000]
    - ["scheduler", 1000]
    - ["mailers", 1000]
  ```
- Scheduled jobs via sidekiq-cron (config/schedule.yml) for feeds import, metrics recording, badge awards
- Event-driven notification system (`Notifications::NewFollowerWorker`, `Notifications::NewCommentWorker`, etc.)
- Redis for job queueing (docker-compose.yml:111-122)

### What It's NOT

**Not Microservices**: Single Dockerfile, single deployment unit, one PostgreSQL database, one Redis instance. All code runs in one Rails application.

**Not Modular Monolith**: While organized by domain concepts (articles, users, notifications), there are no enforced module boundaries. Models directly associate with each other without module-scoped isolation. The `app/services/` and `app/models/` directories use namespace organization but share the same database and don't have strict boundaries.

**Not Microkernel/Plugin**: The Liquid tag system (62 liquid tags in app/liquid_tags/) provides content embedding extensibility, but this is user-facing content functionality, not a plugin architecture for extending the core system. Liquid tags are registered at build time via `Liquid::Template.register_tag()`.

**Not Hexagonal Architecture**: While services encapsulate business logic, the architecture doesn't have explicit ports/adapters pattern. Controllers directly use models and services without an application core isolated by interfaces.

**Not CQRS**: Single model per domain concept; no separate read/write models. Models handle both reading and writing.

### Key Architectural Features

1. **Service Object Pattern**: Business logic extracted into service objects (app/services/) following `Module::Class.call(...)` pattern
2. **Policy-based Authorization**: Pundit policies (app/policies/) for authorization
3. **Background Jobs**: Extensive use of Sidekiq workers for async processing
4. **Liquid Templating**: User content supports embedded liquid tags for external content
5. **API Versioning**: REST API with v0 and v1 versions (app/controllers/api/v0/, app/controllers/api/v1/)
6. **Admin Interface**: Comprehensive admin controllers under `Admin::` namespace

### Deployment

- Single Rails application container (Dockerfile)
- PostgreSQL 13 for database
- Redis 7 for caching and job queues
- Puma as web server
- Sidekiq for background job processing
