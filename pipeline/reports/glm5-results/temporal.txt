## Analysis

### Primary Style: Service-Based Architecture

The Temporal server exhibits clear Service-Based architecture characteristics:

**Evidence:**

1. **Coarse-grained services (4 services)**: The system is decomposed into four well-defined services:
   - `service/frontend/` - API gateway handling external gRPC requests
   - `service/history/` - Core workflow execution state management with sharding
   - `service/matching/` - Task queue management and dispatch
   - `service/worker/` - Background processing (archival, replication)

   Defined in `temporal/server.go`:
   ```go
   var Services = []string{
       string(primitives.FrontendService),
       string(primitives.InternalFrontendService),
       string(primitives.HistoryService),
       string(primitives.MatchingService),
       string(primitives.WorkerService),
   }
   ```

2. **Shared database**: All services use the same persistence layer (Cassandra, MySQL, PostgreSQL, or SQLite). Configuration shows `numHistoryShards` as a shared persistence parameter in `config/development.yaml`:
   ```yaml
   persistence:
     defaultStore: sqlite-default
     visibilityStore: sqlite-visibility
     numHistoryShards: 1
   ```

3. **Inter-service gRPC communication**: Services communicate via generated gRPC clients, not in-process calls. The `client/client_bean.go` shows the client abstraction:
   ```go
   type Bean interface {
       GetHistoryClient() historyservice.HistoryServiceClient
       GetMatchingClient(namespaceIDToName NamespaceIDToNameFunc) (matchingservice.MatchingServiceClient, error)
       GetFrontendClient() workflowservice.WorkflowServiceClient
   }
   ```

4. **Single deployable with service selection**: `cmd/server/main.go` produces one binary that can run any combination of services. Default runs all together, but can be configured for independent deployment:
   ```go
   var DefaultServices = []string{
       string(primitives.FrontendService),
       string(primitives.HistoryService),
       string(primitives.MatchingService),
       string(primitives.WorkerService),
   }
   ```

5. **Service discovery via membership**: Ringpop-based membership system (`common/membership/ringpop/`) enables services to discover and communicate with each other.

### Secondary Style: Layered Architecture

Within each service, there's internal layering:

- **API/Handler layer**: gRPC handlers (e.g., `service/frontend/service.go`, `service/history/handler.go`)
- **Business logic layer**: Engines and processors (e.g., `service/history/history_engine.go`)
- **Data access layer**: Persistence abstractions in `common/persistence/` with ExecutionManager, MetadataManager, etc.

### Rejected Alternatives

- **NOT Microservices**: Services share a database rather than owning separate data stores. The 4 services are coarse-grained (not fine-grained), and the default deployment is a single process running all services together. This matches Service-Based, not Microservices.

- **NOT Modular Monolith**: Services communicate via gRPC with service discovery, not internal function calls. Each service has its own port configuration and can be independently deployed and scaled.

- **NOT Event-Driven**: While Temporal is a durable execution platform that processes workflow events, internal service-to-service communication is synchronous gRPC, not asynchronous messaging through a broker.

- **NOT Hexagonal/Ports-and-Adapters**: While there are interface abstractions, the primary architectural driver is service decomposition, not isolation of the application core via ports and adapters.

### Conclusion

Temporal exemplifies Service-Based architecture: a small number of coarse-grained services (4) that communicate via gRPC, share a database, can be deployed together or separately, and use membership-based service discovery. The ~108K lines of service code are organized into clear vertical slices with shared infrastructure in `common/`.
