## Architecture Classification Analysis

### Primary Style: Microservices

**Strong evidence for Microservices architecture:**

1. **Three independently deployable services** with separate go.mod files:
   - `internal/services/catalogwriteservice/go.mod` —独立的go模块
   - `internal/services/catalogreadservice/go.mod` —独立的go模块  
   - `internal/services/orderservice/go.mod` —独立的go模块

2. **Separate databases per service** (database-per-service pattern):
   - `catalogwriteservice`: PostgreSQL (`gormOptions` in config)
   - `catalogreadservice`: MongoDB + Elasticsearch + Redis (`mongoDbOptions`, `elasticOptions`, `redisOptions`)
   - `orderservice`: EventStoreDB + MongoDB + Elasticsearch (`eventStoreDbOptions`, `mongoDbOptions`)

3. **Individual Dockerfiles** for each service (`internal/services/*/Dockerfile`)

4. **Separate ports** for each service:
   - catalogwriteservice: HTTP 7000, gRPC 6003
   - catalogreadservice: HTTP 7001, gRPC 6004
   - orderservice: HTTP 8000, gRPC 6005

5. **Service-specific configurations** with distinct service names and infrastructure dependencies

### Secondary Style: Event-Driven Architecture

**Evidence:**

1. **RabbitMQ message broker** configured in `docker-compose.infrastructure.yaml`

2. **Integration events** published between services:
   ```go
   // internal/services/catalogwriteservice/internal/products/features/creatingproduct/v1/create_product_handler.go
   productCreated := integrationevents.NewProductCreatedV1(productDto)
   err = c.RabbitmqProducer.PublishMessage(ctx, productCreated, nil)
   ```

3. **Event bus abstraction** in `internal/pkg/core/messaging/bus/bus.go`:
   ```go

       producer.Producer

       consumer.ConsumerConnector
   }
   ```

4. **Projections** for handling events (`orderservice/internal/orders/projections/`)

### Secondary Style: CQRS

**Evidence:**

1. **Separate read/write services**: `catalogreadservice` and `catalogwriteservice` are physically separated services

2. **Command/Query interfaces** in `internal/pkg/core/cqrs/`:
   ```go
   type Command interface { isCommand(); Request; TypeInfo }
   type Query interface { isQuery(); Request; TypeInfo }
   ```

3. **Different database technologies** for read vs write sides:
   - Write side: PostgreSQL (relational, ACID transactions)
   - Read side: MongoDB + Elasticsearch (document stores, optimized for queries)

4. **MediatR pattern** used for dispatching commands and queries (`github.com/mehdihadeli/go-mediatr`)

### What This Is NOT

- **NOT a Modular Monolith**: Services are separate deployment units with their own Dockerfiles, not modules within a single application
- **NOT Layered Architecture**: While there's some layering within features, the primary organization is vertical slices per feature (`features/creatingproduct/`, `features/gettingproducts/`)
- **NOT Serverless**: No function-as-a-service deployments; services are containerized applications
- **NOT Hexagonal Architecture**: While there are ports/adapters patterns, this is secondary to the microservices organization


Each feature folder contains all the code it needs (handler, endpoint, DTOs, events), rather than organizing by technical layer.
