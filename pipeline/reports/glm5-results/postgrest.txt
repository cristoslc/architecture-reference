## Analysis

PostgREST exhibits a classic **Layered Architecture** with clear horizontal separation of concerns:

### Evidence for Layered Architecture

**1. Explicit Layer Separation in Documentation**

The architecture documentation (`docs/explanations/architecture.rst`) explicitly describes a processing pipeline with distinct stages:

- **Main/CLI** → Entry point
- **App** → "composing the different modules" (orchestration layer)
- **Auth** → Authentication
- **ApiRequest** → "parsing the URL query string, request headers, and request body" (presentation layer)
- **Plan** → "generates an internal AST" (business logic layer)
- **Query** → "generates the SQL queries" (data access layer)
- **MainTx** → "parametrizes, prepares, executes SQL queries and decodes their results"
- **Response** → HTTP response generation (presentation layer)

**2. Layered Import Structure**

In `src/PostgREST/App.hs`, the orchestration layer imports from layers below:

```haskell
import qualified PostgREST.Auth       as Auth
import qualified PostgREST.ApiRequest as ApiRequest
import qualified PostgREST.Plan       as Plan
import qualified PostgREST.Query      as Query
import qualified PostgREST.MainTx     as MainTx
import qualified PostgREST.Response   as Response
```

**3. Top-Down Request Flow**

Requests flow through layers in order:
1. HTTP (Warp) → ApiRequest (parsing)
2. Auth (authentication)
3. Plan (business logic, AST generation)
4. Query (SQL generation)
5. MainTx (database execution)
6. Response (formatting)

**4. Single Deployable Unit**

- `postgrest.cabal` defines one library with all modules
- Single executable produced (`main/Main.hs`)
- Dockerfile shows single static binary deployment

### What This Is NOT

**Not Modular Monolith**: There are no enforced module boundaries. All code is in a single package with no submodule isolation. The code is cohesive, not divided into separately versioned or packaged modules.

**Not Event-Driven**: While `Listener.hs` uses PostgreSQL LISTEN/NOTIFY, this is solely for schema cache invalidation notifications—not for core request handling or inter-service communication. The primary request path is synchronous HTTP request-response.

**Not Hexagonal**: There are no ports-and-adapters pattern. The code directly couples to PostgreSQL (Hasql library) and Warp HTTP server without abstraction layers.

**Not Microservices**: Single deployable binary, single connection pool, single process.

### Conclusion

PostgREST is a textbook example of a Layered Architecture: a single deployable application with horizontal layers where each layer has a specific responsibility and depends only on layers below it. The presentation layer handles HTTP, business logic handles request planning, and data access handles SQL generation and execution.
