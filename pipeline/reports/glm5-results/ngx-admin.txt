## Analysis

ngx-admin is a **Layered Architecture** with some **Hexagonal Architecture** patterns, implemented as a single-page Angular application.

### Evidence for Layered Architecture

**Clear horizontal layering is present:**

1. **Core Layer (`@core/`)**: Infrastructure and cross-cutting concerns
   - Abstract data interfaces (`src/app/@core/data/`) define contracts for data access
   - Service implementations (`src/app/@core/mock/`) provide concrete data
   - Utility services (`src/app/@core/utils/`) for analytics, layout, SEO, state management
   - Guard (`module-import-guard.ts`) enforces singleton pattern - CoreModule can only be loaded once

2. **Theme/Presentation Layer (`@theme/`)**: UI components and styling
   - Reusable components (Header, Footer, SearchInput)
   - Layout templates (OneColumn, TwoColumns, ThreeColumns layouts)
   - Theme definitions (default, cosmic, corporate, dark)
   - Shared pipes (capitalize, plural, round, timing)

3. **Application/Pages Layer (`pages/`)**: Feature modules organized by functionality
   - Dashboard, E-commerce, Charts, Forms, Tables, Maps, etc.
   - Each feature has its own module with routing and components
   - Lazy-loaded via Angular's `loadChildren` mechanism

4. **Single Deployment Unit**: `angular.json` defines one application project, building to a single `dist/` folder

### Evidence for Hexagonal Patterns (Secondary Style)

**Port/Adapter separation visible in data layer:**

```typescript
// src/app/@core/data/users.ts - Abstract "port"
export abstract class UserData {
  abstract getUsers(): Observable<User[]>;
  abstract getContacts(): Observable<Contacts[]>;
  abstract getRecentUsers(): Observable<RecentUsers[]>;
}

// src/app/@core/mock/users.service.ts - Concrete "adapter"
@Injectable()
export class UserService extends UserData {
  // Implementation returns mock data
}
```

**Dependency injection binding interfaces to implementations:**
```typescript
// src/app/@core/core.module.ts
const DATA_SERVICES = [
  { provide: UserData, useClass: UserService },
  { provide: ElectricityData, useClass: ElectricityService },
  // ... 18 more data interfaces bound to implementations
];
```

This pattern allows swapping the mock data layer for a real HTTP backend without changing consumers.

### What This Is NOT

- **NOT Microservices/Service-Based**: Single deployable Angular application, no separate services
- **NOT Modular Monolith**: No enforced module boundaries - Angular modules are organizational, not architecturally encapsulated. The `@core` and `@theme` folders use naming convention only, not actual boundary enforcement.
- **NOT Microkernel/Plugin**: No runtime plugin loading or extension points beyond Angular's standard module system
- **NOT Event-Driven**: No message brokers or event-based communication; uses direct service injection
- **NOT CQRS/Domain-Driven Design**: No aggregates, value objects, bounded contexts, or separate read/write models
