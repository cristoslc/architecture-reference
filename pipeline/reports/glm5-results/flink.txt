## Architecture Classification: Apache Flink

### Primary Style: Modular Monolith

Apache Flink is architected as a **Modular Monolith** with strong internal module boundaries enforced through Maven's multi-module structure and ArchUnit architecture tests.

**Evidence:**

1. **Single Deployable Unit**: The `flink-dist` module produces a unified distribution. The `bin.xml` assembly creates a single `flink-dist-{version}.jar` placed in `lib/` alongside core libraries, forming one cohesive runtime.

2. **Enforced Module Boundaries**: Flink uses a dedicated `flink-architecture-tests` module containing ArchUnit rules that enforce dependency constraints. Key rules include:
   - `ConnectorRules.java`: "Connector production code must depend only on public API when outside of connector packages"
   - `ApiAnnotationRules.java`: Enforces that `@Public` API methods only use `@Public` types
   - `TableApiRules.java`: Enforces visibility annotations and proper package structure

3. **Clear Module Hierarchy** (~40 modules in `pom.xml`):
   - Core: `flink-annotations`, `flink-core-api`, `flink-core`
   - Runtime: `flink-runtime`, `flink-runtime-web`
   - APIs: `flink-datastream-api`, `flink-streaming-java`, `flink-table-*`
   - Extensions: `flink-connectors`, `flink-formats`, `flink-metrics`, `flink-filesystems`, `flink-state-backends`

4. **API Visibility System**: Uses annotations (`@Public`, `@PublicEvolving`, `@Internal`, `@Experimental`) to define public API contracts, with architecture tests ensuring internal modules don't leak implementation details.

5. **Dependency Graph**: Clear module dependencies flow from `flink-core-api` → `flink-core` → `flink-runtime` → higher-level APIs. The `pom.xml` shows disciplined layering.

### Secondary Style: Microkernel (Plugin)

Flink incorporates a **Plugin Architecture** for runtime extensibility without modifying the core.

**Evidence:**

1. **Plugin System** (`flink-core/src/main/java/org/apache/flink/core/plugin/`):
   - `PluginManager` interface: Manages cluster plugins loaded via separate classloaders
   - `PluginLoader`: Uses `ServiceLoader` to discover SPI implementations
   - `Plugin` interface: Base interface for plugins with `configure()` lifecycle
   - `PluginClassLoader`: Custom classloader isolating plugin dependencies from Flink core

2. **Plugin Distribution**: The `plugins.xml` assembly shows plugins deployed to `plugins/` directory:
   - `plugins/metrics-jmx/`, `plugins/metrics-prometheus/`, `plugins/metrics-graphite/`, etc.
   - `plugins/external-resource-gpu/`
   - Each plugin is a self-contained JAR loaded at runtime

3. **Extension Points via SPI**:
   - **State Backends**: Multiple implementations (`flink-statebackend-rocksdb`, `flink-statebackend-heap-spillable`, `flink-statebackend-forst`) following `StateBackend` interface
   - **Metrics Reporters**: Pluggable implementations in `flink-metrics-*` modules
   - **Filesystems**: Pluggable filesystems (`flink-s3-fs-*`, `flink-azure-fs-*`, etc.)
   - **Connectors**: `flink-connector-base` defines SPI for external system connections

### What Was Rejected

- **NOT Microservices**: Flink distributes as a single monolithic deployment unit. The runtime itself is not decomposed into independently deployable services.

- **NOT Layered** (primarily): While there's API/implementation separation, the architecture emphasizes **module** boundaries over horizontal layers. Modules like `flink-connectors`, `flink-formats`, `flink-state-backends` are vertical domains, not horizontal layers.

- **NOT Event-Driven**: Though Flink processes streaming data (its business domain), the internal codebase structure is not organized around event-driven communication between components.

- **NOT Hexagonal**: No explicit ports/adapters pattern. The plugin system serves extensibility but doesn't follow hexagonal architecture patterns.

### Summary

Flink exemplifies a mature Modular Monolith with clear module boundaries enforced through tooling (ArchUnit, Maven modules, API annotations). Its plugin system provides the extensibility needed for a data processing framework—allowing users to add metrics reporters, filesystems, and state backends without touching core code. The architecture tests in `flink-architecture-tests` actively prevent erosion of module boundaries, making this a well-maintained modular monolith.
