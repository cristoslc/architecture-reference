This repository implements **Hexagonal Architecture (Ports and Adapters)**, also known as Clean Architecture. The evidence is unambiguous:

**Project Structure (`accounts-api/src/`):**
- `Domain/` - Core domain with entities, value objects, and interface definitions (ports)
- `Application/` - Use cases and service interfaces
- `Infrastructure/` - Implementation of ports (adapters)
- `WebApi/` - Presentation layer with controllers

**Dependency Direction (verified via `.csproj` files):**
- Domain has **zero project references** - pure and isolated
- Application → Domain only
- Infrastructure → Application only  
- WebApi → Application + Infrastructure

**Ports (interfaces in inner layers):**
- `IAccountRepository` (Domain/) - persistence port
- `IUnitOfWork`, `ICurrencyExchange`, `IUserService` (Application/Services/) - application ports
- `IOutputPort` interfaces in each use case folder - output ports for presenter pattern

**Adapters (implementations in outer layers):**
- `AccountRepository : IAccountRepository` (Infrastructure/DataAccess/Repositories/)
- `UnitOfWork : IUnitOfWork`, `CurrencyExchangeService : ICurrencyExchange` (Infrastructure/)
- Controllers implementing `IOutputPort` in WebApi/UseCases/

**Use Case Pattern:** Each use case folder contains `IXxxUseCase`, `XxxUseCase`, `IOutputPort`, and `XxxPresenter`, following the Controller-Presenter-Use Case pattern described in the README wiki.

**Domain-Driven Design tactical patterns** are used within the Domain layer (Aggregate Root pattern on `Account`, Value Objects like `Money` and `Currency`, Repository and Factory patterns), but these serve the Hexagonal structure rather than defining the overall architecture.

**Layered Architecture** is present as a secondary style due to the horizontal separation of concerns, but the defining characteristic is the **inward dependency direction** (dependency inversion principle), which distinguishes Hexagonal from traditional Layered.
