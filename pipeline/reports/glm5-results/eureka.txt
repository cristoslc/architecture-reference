## Architecture Analysis for Eureka Repository

### Primary Classification: Layered Architecture

The Eureka repository exhibits a classic **Layered Architecture** with clear horizontal separation of concerns:

**Evidence for Layered Architecture:**

1. **Presentation Layer (REST Resources)**
   - Located in `eureka-core/src/main/java/com/netflix/eureka/resources/`
   - JAX-RS annotated classes: `ApplicationsResource.java`, `InstanceResource.java`, `VIPResource.java`, `StatusResource.java`
   - Handles HTTP request/response, content negotiation, and routing
   - Delegates business logic to lower layers

2. **Business Logic Layer (Service/Registry)**
   - Core registry interfaces and implementations in `eureka-core/src/main/java/com/netflix/eureka/registry/`
   - Key classes: `InstanceRegistry`, `AbstractInstanceRegistry`, `PeerAwareInstanceRegistryImpl`
   - Handles registration, renewal, cancellation, eviction, and replication logic
   - Lease management: `LeaseManager` interface

3. **Data Access Layer (In-Memory Storage)**
   - `ConcurrentHashMap<String, Map<String, Lease<InstanceInfo>>>` as the primary registry storage
   - `ResponseCacheImpl` for cached responses
   - Read/write locks for concurrency control

**Module Structure Supporting Layers:**
```
eureka-server/     → Presentation/Web Layer (WAR deployment)
eureka-core/       → Business Logic + Data Access
eureka-client/     → Client Library (used by both external apps and server for peer replication)
eureka-resources/  → Static web resources (UI)
```

### Secondary Classification: Hexagonal Architecture (Ports and Adapters)

Evidence for hexagonal patterns:

1. **Core Interfaces (Ports):**
   - `LookupService<T>` - defines service discovery contract
   - `EurekaClient` - extends `LookupService`, defines client operations
   - `EurekaServerConfig` - server configuration interface
   - `InstanceRegistry` extends `LeaseManager<T>` and `LookupService<String>`

2. **Pluggable Implementations (Adapters):**
   - Multiple HTTP client implementations: `eureka-client-jersey2`, `eureka-core-jersey2`
   - Configuration adapters: `eureka-client-archaius2` (Archaius 2.x configuration system)
   - Dependency injection frameworks: `eureka-server-governator` (alternative DI using Netflix Governator)

3. **Dependency Injection:**
   - Heavy use of Google Guice: `@Inject`, `@Singleton` annotations throughout
   - `EurekaModule.java` provides bindings with clear separation of interface and implementation:
     ```java
     bind(EurekaClient.class).to(DiscoveryClient.class).in(Scopes.SINGLETON);
     bind(EurekaClientConfig.class).toProvider(DefaultEurekaClientConfigProvider.class);
     ```

### What This Is NOT:

- **Not Microservices**: This is a single deployable unit (WAR file). While Eureka *enables* microservice discovery, the repository itself is one monolithic application.

- **Not Modular Monolith**: While there are separate Gradle modules (`eureka-client`, `eureka-core`, `eureka-server`), these are library modules for reuse and packaging, not enforced bounded contexts. The `eureka-client` is used both as a standalone library AND embedded in the server for peer-to-peer communication.

- **Not Event-Driven**: Eureka uses synchronous REST calls for communication, not message brokers or event streams. Peer replication uses HTTP batch requests (`PeerEurekaNode`, `ReplicationTaskProcessor`).

- **Not Service-Based**: There's no concept of multiple coarse-grained services sharing a database. This is one service (the Eureka server) with its own in-memory data store.

### Architecture Characteristics:

- **REST API**: JAX-RS (Jersey) based HTTP endpoints
- **Peer-to-Peer Replication**: Eureka servers replicate state to each other (`PeerEurekaNodes`, `PeerEurekaNode`)
- **In-Memory Registry**: No persistent database; data is maintained in memory with lease-based expiration
- **Client Library**: The `DiscoveryClient` provides registration, heartbeat, and service discovery capabilities
- **Servlet Container Deployment**: Packaged as WAR file, runs in servlet containers

### Conclusion:

Eureka implements a **Layered Architecture** with **Hexagonal Architecture** influences. The primary structure is horizontal layering (presentation → business logic → data), while the dependency injection and interface-driven design allow for pluggable implementations of HTTP clients and configuration systems.
