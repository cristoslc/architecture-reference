## Analysis

**Primary Style: Modular Monolith**

Apache Geode is a textbook example of a well-structured modular monolith with strong evidence:

1. **Single Deployable Unit**: The `geode-assembly` and `geode-server-all` modules aggregate all components into one distribution. Docker packaging (`docker/Dockerfile`, `geode-assembly/Dockerfile`) produces a single container image.

2. **Explicit Module Boundaries**: The codebase is organized into 50+ Gradle subprojects with enforced dependencies:
   - Core modules: `geode-common`, `geode-logging`, `geode-serialization`, `geode-unsafe`
   - Main implementation: `geode-core` (the substantial implementation)
   - Feature modules: `geode-wan`, `geode-cq`, `geode-lucene`, `geode-memcached`
   - Extensions: `extensions/geode-modules`, `extensions/geode-modules-session`

3. **Enforced API Boundaries**: 
   - Gradle build files use `api` vs `implementation` dependencies for explicit visibility control
   - BOMs (`boms/geode-all-bom`, `boms/geode-client-bom`) provide dependency governance
   - `internal` packages throughout (e.g., `org.apache.geode.internal.*`, `org.apache.geode.cache.internal.*`) separate public API from implementation

4. **Code Organization Evidence** (`settings.gradle` lines 70-113):
   ```groovy
   include 'geode-common'
   include 'geode-core'
   include 'geode-wan'
   include 'geode-cq'
   include 'geode-lucene'
   ...
   ```

**Secondary Style: Layered**

Clear horizontal layering is present:

1. **Foundation Layer**: `geode-common` (annotations, utilities), `geode-logging`, `geode-serialization`, `geode-unsafe`

2. **Core Business Layer**: `geode-core` contains:
   - Cache abstraction (`org.apache.geode.cache.*`)
   - Distributed system implementation (`org.apache.geode.distributed.*`)
   - Internal cache implementation (`org.apache.geode.internal.cache.*`)

3. **Service Layer**: `geode-wan` (gateway/WAN), `geode-cq` (continuous queries), `geode-lucene` (search indexing)

4. **Presentation/Interface Layer**: `geode-gfsh` (CLI shell), `geode-web`, `geode-web-api`, `geode-web-management`

5. **Assembly Layer**: `geode-assembly`, `geode-server-all` (packaging)

**What This Is NOT:**

- **NOT Microservices**: Single JVM deployment, not multiple independent services. Geode nodes form a peer-to-peer distributed system, not separately deployed microservices.

- **NOT primarily Event-Driven**: While Geode has event capabilities (`CacheListener`, `CacheWriter`, continuous queries), these are features *within* the monolith, not the primary architectural organizing principle.

- **NOT Domain-Driven Design**: Code is organized by technical capability (serialization, logging, core, WAN) rather than bounded contexts.

- **NOT Microkernel/Plugin**: While `Declarable`, `Function`, `CacheLoader`, and `JarDeployer` provide extensibility, the core system isn't structured as a minimal kernel with plugins. The extension points are features of a complete product, not the fundamental architecture.

- `geode-server-all/build.gradle` - Aggregates all modules into single artifact
- `geode-core/src/main/java/org/apache/geode/cache/*.java` - Public API interfaces
- `geode-core/src/main/java/org/apache/geode/internal/*` - Internal implementation packages
