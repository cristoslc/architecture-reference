## Architecture Analysis

### Primary Style: Microservices

The eShop repository implements a textbook **Microservices architecture** with the following evidence:

**Multiple Independently Deployable Services:**
The AppHost (`src/eShop.AppHost/Program.cs`) orchestrates 10+ distinct services:
- `Basket.API` - Shopping basket service (gRPC, Redis-backed)
- `Catalog.API` - Product catalog service (PostgreSQL-backed)
- `Ordering.API` - Order management service (PostgreSQL-backed)
- `Identity.API` - Authentication/identity service
- `Webhooks.API` - Webhook management service
- `PaymentProcessor` - Background payment processing worker
- `OrderProcessor` - Background order processing worker
- `WebApp` - Web frontend (Blazor)
- `WebhookClient` - Webhook client application
- YARP-based mobile BFF (reverse proxy)

**Database Per Service Pattern:**
Each service owns its data with separate databases:
```
var catalogDb = postgres.AddDatabase("catalogdb");
var identityDb = postgres.AddDatabase("identitydb");
var orderDb = postgres.AddDatabase("orderingdb");
var webhooksDb = postgres.AddDatabase("webhooksdb");
```
Basket.API uses Redis for its state, not shared with other services.

**Independent Technology Choices:**
- Basket.API uses gRPC (`builder.Services.AddGrpc()`)
- Ordering.API uses REST with API versioning
- Services can be deployed and scaled independently via .NET Aspire

### Secondary Style: Event-Driven Architecture

Services communicate asynchronously through **integration events** via RabbitMQ:

**Event Bus Infrastructure:**
```
var rabbitMq = builder.AddRabbitMQ("eventbus")
```
(`src/eShop.AppHost/Program.cs`, lines 8-9)

**Integration Events Flow:**
- `OrderStartedIntegrationEvent` - published by Ordering.API when order created
- `OrderStatusChangedToAwaitingValidationIntegrationEvent` - triggers stock validation in Catalog.API
- `OrderStockConfirmedIntegrationEvent` / `OrderStockRejectedIntegrationEvent` - catalog stock validation results
- `OrderPaymentSucceededIntegrationEvent` / `OrderPaymentFailedIntegrationEvent` - payment processing results

**Event Handlers Subscribed via Event Bus:**
```csharp
// PaymentProcessor/Program.cs
builder.AddRabbitMqEventBus("EventBus")
    .AddSubscription<OrderStatusChangedToStockConfirmedIntegrationEvent, 
                      OrderStatusChangedToStockConfirmedIntegrationEventHandler>();
```

**Outbox Pattern Implementation:**
The `IntegrationEventLogService` (`src/IntegrationEventLogEF/`) implements reliable event publishing with transactional integrity - events are stored atomically with domain changes before being published.

### Rejected Styles

**Not Layered Monolith:** Services are independently deployable with separate databases, not horizontal layers in a single deployable. Each service has its own Program.cs entry point.

**Not Modular Monolith:** The solution has multiple deployable units (.csproj projects that become separate containers), not a single deployable with enforced module boundaries.

**Not Serverless:** Services are deployed as containers orchestrated by .NET Aspire, not as individual functions triggered by events. PaymentProcessor and OrderProcessor are background workers, not Azure Functions.

**Not CQRS (primary):** While MediatR is used for commands/queries in Ordering.API, this is an application pattern within the microservice, not the primary architectural characteristic. The read/write separation is not at the system level.

## Conclusion

eShop is a **Microservices reference application** built on .NET Aspire, using **Event-Driven** communication between services via RabbitMQ integration events, with **Domain-Driven Design** patterns applied within the Ordering bounded context. This combination is typical for mature microservices implementations where event-driven communication ensures loose coupling between services, while DDD patterns provide rich domain modeling within service boundaries.
