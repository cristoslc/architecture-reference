## Analysis

The Beats repository exhibits a clear **Microkernel (Plugin)** architecture, with libbeat serving as the core framework and individual Beats, modules, inputs, and processors functioning as plugins.

### Key Evidence for Microkernel (Plugin)

**1. libbeat as the Core System**

`libbeat/README.md` explicitly states:
> "libbeat is a Go library containing the common packages for all the Beats... If you want to create a new project that reads some sort of operational data and ships it to Elasticsearch, we suggest you make use of this library."

This is the defining characteristic of a microkernel — a core framework that provides infrastructure for building specialized applications.

**2. Plugin Registration Infrastructure**

`libbeat/plugin/plugin.go` provides the plugin loading mechanism:
- `PluginLoader` type for registering plugin handlers
- `MustRegisterLoader()` for registering plugin types
- `LoadPlugins()` for dynamically loading plugins from `.so` files (via Go's plugin package on Linux/Darwin)

**3. Registry Pattern with Factory Functions**

Multiple independent plugin registries exist:

- **Metricbeat modules** (`metricbeat/mb/registry.go`): `var Registry = NewRegister()` with `ModuleFactory` and `MetricSetFactory` types
- **Filebeat inputs** (`filebeat/input/registry.go`): `var registry = make(map[string]Factory)` with `Register()` and `GetFactory()`
- **Processors** (`libbeat/processors/registry.go`): `var registry = NewNamespace()` with `Constructor` type

Each module registers itself via `init()` functions. For example, in `metricbeat/module/apache/status/status.go`:
```go
func init() {
    mb.Registry.MustAddMetricSet("apache", "status", New,
        mb.WithHostParser(hostParser),
        mb.DefaultMetricSet(),
    )
}
```

**4. Beater Interface Contract**

`libbeat/beat/beat.go` defines the core contract:
```go
type Beater interface {
    Run(b *Beat) error
    Stop()
}
type Creator func(*Beat, *config.C) (Beater, error)
```

Each Beat (Filebeat, Metricbeat, etc.) implements this interface, providing a uniform extension point.

**5. Multiple Independently Deployable Beats**

The repository contains 7+ standalone Beats, each with its own:
- `main.go` entrypoint
- `Dockerfile` for containerized deployment
- Configuration files (`.yml`)

This demonstrates the plugin ecosystem built on top of libbeat.

**6. Community Beats Ecosystem**

`docs/extend/community-beats.md` lists dozens of community-created Beats (amazonbeat, apachebeat, browserbeat, cloudflarebeat, etc.) that extend the framework — definitive proof of a plugin architecture.

**7. Module Discovery and Dynamic Loading**

`filebeat/include/list.go` and similar files auto-generate module imports:
```go
// Code generated by beats/dev-tools/cmd/module_include_list/module_include_list.go
import (
    _ "github.com/elastic/beats/v7/filebeat/module/apache"
    _ "github.com/elastic/beats/v7/filebeat/module/auditd"
    // ... many more modules
)
```

This build-time plugin discovery pattern allows modules to be independently developed and integrated.

### Secondary: Layered Architecture

Within each Beat, there's a layered processing pipeline. As documented in `filebeat/main.go`:
```
input -> harvester -> spooler -> publisher -> registrar
```

Each stage has distinct responsibilities:
- **Input layer**: Discovers and configures data sources
- **Harvester layer**: Reads individual files/sources
- **Spooler layer**: Buffers events
- **Publisher layer**: Sends to outputs (Elasticsearch, Logstash, etc.)
- **Registrar layer**: Persists state

However, this layering is subordinate to the plugin architecture — it's how each Beat is internally organized, not the primary architectural pattern of the repository.

### What Was Rejected

- **Microservices**: The Beats are independently deployable, but they don't communicate with each other or share data. Each Beat ships data to external destinations (Elasticsearch/Logstash), not to other Beats. No service mesh, API gateway, or inter-service communication.

- **Modular Monolith**: While there are module boundaries, each Beat is a separate executable with its own main entrypoint — not a single deployable with internal modules.

- **Domain-Driven Design**: No evidence of aggregates, bounded contexts, or ubiquitous language. The modules are organized by data source technology (Apache, MySQL, Redis), not business domains.

- **Hexagonal Architecture**: While there are interfaces (Beater, Input, Processor), there's no explicit ports-and-adapters structure with application core isolation.

### Conclusion

The Beats repository is a textbook example of **Microkernel (Plugin)** architecture. libbeat provides the core framework, and individual Beats, modules, inputs, processors, and outputs are all plugins that register themselves via well-defined extension points. The registry pattern, factory functions, Beater interface contract, and community ecosystem all strongly support this classification.
