## Architecture Classification Analysis

### Primary Style: Microservices

The repository implements a **true microservices architecture** with clear evidence:

**Deployment Topology:**
- **5 independently deployable services** + API Gateway:
  - `product-api` - Product catalog service
  - `counter-api` - Order taking service  
  - `barista-api` - Beverage preparation service
  - `kitchen-api` - Food preparation service
  - `order-summary` - Read projection/audit service
  - `yarp` - API Gateway (reverse proxy)

- **Container orchestration** via `docker-compose.yaml` and Kubernetes manifests (`deployment-product-api.yaml`)
- **.NET Aspire orchestration** in `app-host/Program.cs` coordinating services, databases, message broker, and AI models

**Data Ownership:**
- Each service owns its data store:
  - `product-api` uses PostgreSQL with Entity Framework (`ProductDbContext`)
  - `order-summary` uses Marten event store with PostgreSQL
  - Services do not share databases

**Service Discovery & Communication:**
- YARP API Gateway (`yarp/appsettings.json`) routes `/p/*` → product-api, `/c/*` → counter-api
- HTTP-based service-to-service calls via `ItemHttpGateway` using service discovery
- Async communication via RabbitMQ message broker

**MassTransit/RabbitMQ Integration:**

- `order-summary/Consumers/OrderConsumer.cs`: Consumes both `BaristaOrderPlaced` and `KitchenOrderPlaced` for event sourcing

### What This Is NOT

- **NOT Layered Architecture**: No horizontal presentation/business/data layers across the system
- **NOT Modular Monolith**: Multiple deployable containers, not a single unit
- **NOT Serverless**: No function-as-a-service; uses containers and message queues
- **NOT CQRS**: Uses MediatR commands/queries but lacks separate read/write models (order-summary has event projections, but this is supplementary)
- **NOT Hexagonal Architecture**: No explicit ports/adapters pattern despite having gateways
