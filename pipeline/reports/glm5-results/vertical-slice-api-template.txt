### Evidence for CQRS (Secondary)

### What Was Rejected and Why

**NOT Microservices:**
- Single deployable, single database context, single solution
- Docker compose shows one API service, not multiple independent services

**NOT Layered Architecture:**
- No horizontal layering (no separate Controllers/Services/Repositories folders)
- Code organized by feature/slice, not by technical layer
- Each vertical slice contains its own endpoint, handler, DTOs, validation

**NOT Domain-Driven Design (primary):**
- Has DDD infrastructure (Aggregate, Entity, Domain Events base classes)
- But domain model is anemic: `Product.cs` is a POCO with no methods or invariants
- No rich aggregates with behavior, no bounded context separation
- DDD elements are scaffolding, not the primary architectural pattern

**NOT Event-Driven Architecture:**
- Integration events via MassTransit/RabbitMQ exist but are secondary
- Events published to external broker, not the primary communication pattern
- Synchronous request-response via Minimal APIs is the primary interaction model

### Summary

This repository is a **Modular Monolith** implementing **Vertical Slice Architecture** principles. The application is a single deployable with clearly defined internal module boundaries (Products module), where each feature/use case is a self-contained vertical slice containing all necessary code. CQRS via MediatR provides the command/query separation pattern within each slice. The template explicitly documents this: "Using Vertical Slice Architecture as a high-level architecture" and "Using CQRS Pattern on top of MediatR library."
