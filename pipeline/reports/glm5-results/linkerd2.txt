## Analysis

### Primary Classification: Microservices

The linkerd2 repository exhibits clear microservices architecture characteristics:

**Multiple Independently Deployable Services:**
1. **Controller** (`Dockerfile.controller`) - A multi-purpose binary that dispatches to several internal services:
   - `destination` - gRPC service discovery (port 8086)
   - `identity` - gRPC certificate authority (port 8080)
   - `proxy-injector` - Kubernetes admission webhook
   - `sp-validator` - Service profile validator webhook
   - `heartbeat` - Telemetry collection
   - `service-mirror` - Multicluster support

2. **Policy Controller** (`policy-controller/`) - Separate Rust binary providing gRPC policy API (port 8090), built with its own Cargo workspace

3. **Viz Extension** - Multiple independent services:
   - `metrics-api` (separate Dockerfile)
   - `tap` (separate Dockerfile)  
   - `web` dashboard (separate Dockerfile)

4. **CLI** - Standalone binary with its own Dockerfile

5. **Proxy** (`Dockerfile.proxy`) - Data plane sidecar proxy

**Evidence from `charts/linkerd-control-plane/templates/`:**
- Separate Kubernetes Deployments and Services for each component (destination.yaml, identity.yaml, proxy-injector.yaml, heartbeat.yaml)
- Each service exposes distinct gRPC endpoints
- Services communicate via gRPC (evident in `controller/api/destination/server.go:19` importing `google.golang.org/grpc`)

**State Isolation:**
- No shared database - each service uses Kubernetes API as its data store
- Services are stateless, storing state in Kubernetes CRDs

### Secondary Classification: Microkernel (Plugin)

The extension system documented in `EXTENSIONS.md` provides a clear plugin architecture:

**Plugin Contract:**
- Extensions are external executables named `linkerd-<name>`
- Must implement: `install`, `uninstall`, `check` commands
- Discovered via PATH at runtime

**Built-in Extensions:**
- `viz` - Observability extension (`viz/` directory with its own charts and commands)
- `multicluster` - Cross-cluster communication (`multicluster/` directory)

**Evidence from `EXTENSIONS.md:42-48`:**
```
The extension must implement these commands:
### `linkerd-name install`
This command must print the Kubernetes manifests for the extension as yaml
which is suitable to be passed to `kubectl apply -f`.
```

### What Was Rejected

- **Service-Based**: Rejected because services have much finer granularity (5+ control plane services, 3+ viz services) with independent deployment capabilities, each owning its own Kubernetes resources

- **Layered**: Rejected because there's no horizontal layering (presentation/business/data layers). The architecture is organized around vertical service boundaries with each service having its own full stack

- **Modular Monolith**: Rejected because while the controller binary contains multiple services, they are logically and operationally distinct services with separate Kubernetes Deployments, not internal modules in a single deployable

- **Event-Driven**: Rejected because inter-service communication is synchronous gRPC, not message broker based. Kubernetes watch streams are used for state synchronization but this is infrastructure-level, not application architecture

- **Hexagonal Architecture**: While there's some use of interfaces (e.g., `viz/metrics-api/grpc_server.go:24` defines a `Server` interface), ports-and-adapters is not the dominant organizational pattern

### Summary

Linkerd2 is fundamentally a microservices architecture for the control plane, with independent services communicating via gRPC and storing state in Kubernetes. The extension system adds a microkernel dimension, allowing third-party functionality to extend the core CLI through a well-defined plugin contract.
