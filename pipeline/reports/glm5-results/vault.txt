## Analysis

Vault implements a classic **Microkernel (Plugin)** architecture with a single deployable core system extended through a well-defined plugin contract.

### Core System Evidence

**Single Deployable:**
- `main.go` is the sole entry point, delegating to `command.Run()`
- Single `Dockerfile` produces a single `vault` binary
- Single `go.mod` at repository root - no multi-module workspace

**Core Orchestrator (`vault/core.go`):**
The `Core` struct (line 265) is the central coordinator containing:
- `router *Router` - routes requests to mounted backends
- `logicalBackends map[string]logical.Factory` - registered secrets engines
- `credentialBackends map[string]logical.Factory` - registered auth methods
- `pluginCatalogPath = "core/plugin-catalog/"` - plugin registry storage

### Plugin Contract Evidence

**Interface Definition (`sdk/logical/logical.go`):**
```go
type Backend interface {
    Initialize(context.Context, *InitializationRequest) error
    HandleRequest(context.Context, *Request) (*Response, error)
    SpecialPaths() *Paths
    System() SystemView
    Logger() log.Logger
    HandleExistenceCheck(context.Context, *Request) (bool, bool, error)
    Cleanup(context.Context)
    InvalidateKey(context.Context, string)
    Setup(context.Context, *BackendConfig) error
    Type() BackendType
}

type Factory func(context.Context, *BackendConfig) (Backend, error)
```

This interface is the **plugin contract** - all backends must implement it.

### What Was Rejected

**Not Microservices:**
- Single binary deployment
- No inter-service communication
- No independent deployability of components

**Not Modular Monolith:**
- No enforced module boundaries in code structure
- Plugins can be loaded externally, not just internal modules
- The extension mechanism is the primary architectural feature, not a secondary concern

**Not Layered:**
- No horizontal layering (presentation/business/data access)
- The `vault/`, `sdk/`, `api/` structure is about plugin development, not horizontal separation
- Core functionality and plugins operate at the same abstraction level

**Not Event-Driven:**
- Primary interaction model is synchronous request/response
- EventBus exists (`vault/eventbus/bus.go`) but is secondary, used for internal notifications
- Not the primary architectural characteristic

### Summary

Vault exemplifies the **Microkernel (Plugin)** pattern:
1. **Core System:** Minimal `Core` with router, mount table, and plugin catalog
2. **Plugin Contract:** `logical.Backend` interface with `Factory` function
3. **Plugin Types:** Auth methods (credential), secrets engines (logical), database plugins
4. **Extension Mechanism:** Both compile-time (builtin registry) and runtime (plugin catalog) loading
5. **Communication:** gRPC via `go-plugin` for external plugins, direct instantiation for builtin
