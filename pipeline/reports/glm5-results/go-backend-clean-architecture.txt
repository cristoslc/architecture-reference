## Architecture Classification Analysis

### Primary Style: Layered Architecture

The repository implements a classic **Layered Architecture** with clear horizontal separation of concerns:

| Layer | Directory | Responsibility |
|-------|-----------|----------------|
| Router | `api/route/` | HTTP route registration |
| Controller | `api/controller/` | Request handling, HTTP binding |
| Usecase | `usecase/` | Business logic orchestration |
| Repository | `repository/` | Data access abstraction |
| Domain | `domain/` | Entity definitions, interfaces |

**Evidence of layering:**
- `README.md` lines 27-33 explicitly document: "Router → Controller → Usecase → Repository → Domain"
- Request flow follows strict layer ordering (e.g., `api/route/task_route.go:17-18` wires controller → usecase → repository)

### Secondary Style: Hexagonal Architecture (Ports and Adapters)

The codebase implements **dependency inversion** characteristic of hexagonal architecture:

**Domain layer defines interfaces (ports):**
```go
// domain/task.go:19-22
type TaskRepository interface {
    Create(c context.Context, task *Task) error
    FetchByUserID(c context.Context, userID string) ([]Task, error)
}
```

**Outer layers implement adapters:**
```go
// repository/task_repository.go:17
func NewTaskRepository(db mongo.Database, collection string) domain.TaskRepository
```

**Database adapter pattern** (`mongo/mongo.go`):
- Defines abstract interfaces: `Database`, `Collection`, `Client`, `Cursor`, `SingleResult`
- Concrete implementations wrap MongoDB driver
- Repository depends on `mongo.Database` interface, not concrete driver
- Enables database substitution by implementing interfaces

**Dependency direction:** All dependencies point **inward** toward `domain`:
- `usecase` imports `domain` (line 7 in `usecase/task_usecase.go`)
- `repository` imports `domain` (line 6 in `repository/task_repository.go`)
- `controller` imports `domain` (line 6 in `api/controller/task_controller.go`)

### Deployment Model: Single Monolith

- Single `Dockerfile` builds one binary from `cmd/main.go`
- Single `docker-compose.yaml` with one `web` service
- Single Go module (`go.mod`) with no workspace/submodules
- Entry point (`cmd/main.go:11-26`) wires all dependencies in one process

### What This Is NOT

**Not Microservices:** Single deployable, single database connection, no inter-service communication.

**Not Modular Monolith:** Horizontal technical layers, not vertical business module boundaries. No module-specific packages or bounded contexts.

**Not Domain-Driven Design:** Lacks aggregates, value objects, domain events, bounded contexts. The `domain` package contains anemic data structures and repository interfaces, not rich domain logic.

**Not CQRS:** Single read/write model, no command/query separation.

**Not Event-Driven:** Synchronous request-response only, no message brokers or event handlers.

### Conclusion

This is a textbook implementation of **Clean Architecture** (Robert Martin's variant) applied to a layered system. It combines the horizontal layering of traditional N-tier architecture with the dependency inversion principle of hexagonal architecture. The innermost `domain` layer defines the core abstractions (ports), while `repository`, `usecase`, `controller`, and `route` serve as increasingly outward adapters.
