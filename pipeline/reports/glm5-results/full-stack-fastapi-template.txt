## Architecture Classification

### Primary Style: Service-Based

This repository exhibits a classic **Service-Based Architecture** with the following characteristics:

**Evidence from `compose.yml`:**
- **2 coarse-grained services** deployed independently: `backend` (FastAPI Python) and `frontend` (React/TypeScript served via Nginx)
- **Shared database**: Both services connect to a single PostgreSQL instance (`db` service)
- **Single Docker Compose deployment** with Traefik reverse proxy handling routing
- Typical of small-to-medium applications with 2-5 services sharing data storage

The deployment topology clearly shows services that are separately containerized and scalable, but they share a common PostgreSQL database—exactly matching the Service-Based pattern where coarse-grained services collaborate around a shared data store.

### Secondary Style: Layered

The backend demonstrates classic **horizontal layering**:

**From `backend/app/` directory structure:**
```
api/routes/     → Presentation Layer (FastAPI route handlers/controllers)
crud.py         → Business Logic Layer (service operations)
models.py       → Data Access Layer (SQLModel ORM entities)
core/           → Infrastructure Layer (config, db, security)
api/deps.py     → Cross-cutting concerns (dependency injection)
```

**Evidence from `backend/app/api/routes/users.py` (lines 56-77):**
- Routes call `crud.create_user()` and `crud.get_user_by_email()` rather than direct database access
- Clear separation between HTTP handling (routes) and business logic (crud)

**Evidence from `backend/app/crud.py`:**
- Pure data operations: `create_user()`, `update_user()`, `get_user_by_email()`, `authenticate()`
- No HTTP concerns, only Session and model operations

### What This Is NOT

- **NOT Microservices**: Only 2 services sharing a single database; no independent data ownership per service, no service mesh, no complex inter-service communication patterns
- **NOT Modular Monolith**: Two separately deployable containers (`backend/Dockerfile`, `frontend/Dockerfile`), not a single deployable unit
- **NOT Event-Driven**: No message brokers (Kafka, RabbitMQ, Redis, Celery) found in codebase; synchronous REST API only between frontend and backend
- **NOT Hexagonal Architecture**: No ports/adapters pattern; routes directly import and use SQLModel ORM with no abstraction layer
- **NOT Domain-Driven Design**: No bounded contexts, aggregates, or domain-focused organization; simple User/Item entities with basic CRUD operations

### Communication Pattern

The frontend generates an API client from the backend's OpenAPI spec (`frontend/src/client/sdk.gen.ts`), making synchronous REST calls to the backend API. This is standard request-response communication, not event-driven messaging.
