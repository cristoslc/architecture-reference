## Architecture Classification: Microkernel (Plugin)

**Moodle is a definitive example of a Microkernel (Plugin) architecture.** The codebase is structured as a core system with an extensive plugin infrastructure that provides nearly all functionality through modular, hot-pluggable extensions.

### Key Evidence

**1. Plugin Types System (48+ plugin types)**
The `lib/components.json` file defines 48 plugin types including:
- `mod` (activity modules like assign, forum, quiz)
- `auth` (authentication plugins like ldap, oauth2, shibboleth)
- `enrol` (enrollment methods)
- `block` (sidebar blocks)
- `theme` (visual themes)
- `filter` (content filters)
- `local` (site-specific customizations)
- And 40+ more types for search, cache, editors, payment gateways, etc.

**2. Plugin Manager and Lifecycle**
`lib/classes/plugin_manager.php` provides:
- Plugin discovery and registration
- Version management (`PLUGIN_STATUS_NEW`, `PLUGIN_STATUS_UPGRADE`, etc.)
- Dependency tracking between plugins
- Enable/disable/uninstall capabilities
- Source distinction (`PLUGIN_SOURCE_STANDARD` vs `PLUGIN_SOURCE_EXTENSION`)

**3. Plugin Base Class Infrastructure**
`lib/classes/plugininfo/base.php` defines the abstract contract all plugins must follow:
```php
abstract class base {
    public $type;      // Plugin type (mod, auth, etc.)
    public $name;      // Plugin name
    public $rootdir;   // Location on disk
    public $versiondisk;
    public $versiondb;
    // ... lifecycle methods
}
```

**4. Subplugin Architecture**
Plugins can define their own subplugins. For example, `mod/assign/db/subplugins.json`:
```json
{
    "subplugintypes": {
        "assignsubmission": "submission",
        "assignfeedback": "feedback"
    }
}
```
This allows the assignment module to have pluggable submission types (file, onlinetext) and feedback types (comments, editpdf).

**5. Hook System (PSR-14 Event Dispatcher)**
`lib/classes/hook/manager.php` implements a modern hook system:
- Implements PSR-14 `EventDispatcherInterface` and `ListenerProviderInterface`
- Plugins register callbacks in `db/hooks.php` files
- Decoupled communication between core and plugins

**6. Standard Plugin Structure**
Every plugin follows a consistent pattern:
- `version.php` - Plugin metadata and version
- `db/install.xml` - Database schema
- `db/upgrade.php` - Migration scripts
- `db/access.php` - Capability definitions
- `db/services.php` - External API definitions
- `lang/en/` - Localization strings
- `settings.php` - Admin configuration

**7. Core vs Plugin Separation**
- `public/lib/` - Core libraries (accesslib.php, adminlib.php, etc.)
- `public/mod/` - Activity module plugins
- `public/auth/` - Authentication plugins
- `public/blocks/` - Block plugins
- `public/local/` - Reserved for site-specific customizations

**8. Single Deployable Unit**
- One `config.php` connecting to a single database
- One `index.php` entry point
- Monolithic deployment via standard PHP web hosting

### Why Not Other Architectures

**Not Microservices**: Moodle is a single PHP application deployed as one unit with one database. No service boundaries, no inter-service communication, no independent deployability.

**Not Layered**: While there's presentation code, Moodle isn't organized horizontally into presentation/business/data layers. Instead, it's organized vertically by plugin type, with each plugin containing its own view, logic, and data handling.

**Not Modular Monolith**: Modular monoliths have internal modules with enforced boundaries. Moodle goes further—plugins are truly externalizable extensions that can be added/removed after deployment, discovered at runtime, and maintained independently.

**Not Event-Driven**: While Moodle has events and hooks, they're implementation details within the plugin system, not the primary architectural pattern. The hook system facilitates plugin communication but isn't the core organizing principle.

**Not Hexagonal**: While Moodle has external APIs, there's no explicit ports-and-adapters isolation. Plugins directly access core APIs, and the core doesn't have a clean domain layer separated from infrastructure.

### Conclusion

Moodle's architecture centers on a minimal core that provides essential services (database access, user authentication, session management, plugin loading) while delegating virtually all features to plugins. The sophisticated plugin infrastructure with typed plugin categories, lifecycle management, subplugins, and hook-based communication makes this a textbook **Microkernel (Plugin)** architecture with very high confidence (0.95).
