## Architecture Classification: Layered

Memcached implements a classic **Layered Architecture** organized as a single deployable binary with horizontal separation of concerns.

### Evidence for Layered Architecture

**1. Single Deployable Unit**
- The `Makefile.am` defines a single `memcached` binary (`bin_PROGRAMS = memcached`)
- All source files compile into one executable (~40,000 lines of C code across 56 `.c` files)
- No module boundaries, package structure, or independent deployment units

**2. Clear Horizontal Layering**

**Protocol/Presentation Layer:**
- `proto_text.c` (53,764 lines) - Text protocol handler
- `proto_bin.c` (44,152 lines) - Binary protocol handler  
- `proto_parser.c` (60,970 lines) - Protocol parsing
- `proto_proxy.c` (61,293 lines) - Proxy protocol (optional)

**Business Logic Layer:**
- `items.c` (62,307 lines) - Core caching logic: item allocation, LRU management, storage/retrieval
- `assoc.c` - Hash table implementation for key lookups
- `crawler.c` (30,589 lines) - Background expiration/crawling

**Data Access/Storage Layer:**
- `slabs.c` (829 lines) - Slab allocator for memory management
- `extstore.c` (31,784 lines) - External storage tier (flash/SSD)
- `storage.c` (1,601 lines) - Storage abstraction

**3. State Machine Request Processing**
The `memcached.c` main file (225,788 lines) orchestrates:
- Event-driven I/O via libevent (`event_handler`, `event_base`)
- Connection state machine (`drive_machine`)
- Request routing to appropriate protocol handlers

**4. Shared Global State**
`globals.c` exports shared state across layers:
```c
volatile rel_time_t current_time;
struct stats stats;
struct stats_state stats_state;
struct settings settings;
```

### What This Is NOT

**Not Modular Monolith:** No enforced module boundaries. All source files directly access shared global state (hash tables, LRU lists, slab classes). The codebase uses direct function calls and shared memory rather than module-scoped APIs.

**Not Event-Driven Architecture:** While memcached uses libevent for I/O multiplexing, internal components communicate via direct function calls, not event publication/subscription. The event loop handles network I/O, but the architecture doesn't use events for inter-component communication.

**Not Microservices:** Single binary, single process, shared memory space. No service decomposition.

**Not Hexagonal Architecture:** No ports/adapters pattern. Protocol handlers directly call business logic functions (`item_get`, `store_item`) which directly access data structures. No dependency inversion or interface abstraction layers.

**Not CQRS:** Read and write operations share the same item data structures and code paths.

### Threading Model

The threading model (documented in `doc/threads.txt`) supports the layered design:
- Listener thread accepts connections
- N worker threads run libevent loops, handle protocol parsing and data transfer
- Background threads for LRU maintenance, storage compaction, crawling
- Fine-grained locks (`item_locks`, `lru_locks`, `slabs_lock`) protect shared resources across threads

This is a performance-optimized layered architecture designed for high-throughput key-value caching, with clear separation between protocol handling, caching logic, and memory/storage management.
