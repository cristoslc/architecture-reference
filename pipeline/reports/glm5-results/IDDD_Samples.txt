## Architecture Analysis

This repository is the official reference implementation for Vaughn Vernon's book "Implementing Domain-Driven Design." The architecture is explicitly designed to demonstrate DDD patterns.

### Primary Style: Domain-Driven Design

**Clear evidence of DDD tactical patterns throughout:**

**1. Bounded Contexts**: The codebase is organized as separate Gradle modules, each representing a bounded context:
- `iddd_agilepm` - Agile Project Management context
- `iddd_collaboration` - Collaboration context  
- `iddd_identityaccess` - Identity & Access context
- `iddd_common` - Shared kernel with common domain infrastructure

**2. Domain Model Organization**: Each context follows the `domain/model/<aggregate>` package structure:
```
iddd_agilepm/domain/model/product/
iddd_agilepm/domain/model/team/
iddd_collaboration/domain/model/calendar/
iddd_collaboration/domain/model/forum/
iddd_identityaccess/domain/model/identity/
iddd_identityaccess/domain/model/access/
```

**3. Aggregates & Entities**: Clear aggregate roots with identity and lifecycle management:
- `Product` extends `Entity` base class (`iddd_agilepm/src/main/java/com/saasovation/agilepm/domain/model/product/Product.java`)
- `Calendar` extends `EventSourcedRootEntity` (`iddd_collaboration/src/main/java/com/saasovation/collaboration/domain/model/calendar/Calendar.java`)
- `Tenant`, `Group`, `User` as aggregate roots in identity context

**4. Value Objects**: `ValueObject` base class and value objects like `TenantId`, `ProductId`, `CalendarId` representing identity without lifecycle.

**5. Domain Events**: Robust event publishing via `DomainEventPublisher` (`iddd_common/src/main/java/com/saasovation/common/domain/model/DomainEventPublisher.java`):
```java
DomainEventPublisher.instance().publish(new ProductCreated(...));
DomainEventPublisher.instance().publish(new CalendarCreated(...));
```

**6. Repositories**: Interface-based repositories defined in domain layer, implemented in adapters:
- `ProductRepository` interface in domain model
- `CalendarRepository` interface in domain model
- Implementations like `LevelDBProductRepository`, `EventStoreCalendarRepository`, `HibernateUserRepository`

### Secondary Style: Hexagonal Architecture (Ports and Adapters)

The codebase explicitly follows the ports and adapters pattern:

**Application Layer**: Application services as primary ports:
- `ProductApplicationService`, `TeamApplicationService` in agilepm
- `CalendarApplicationService`, `ForumApplicationService` in collaboration
- `IdentityApplicationService`, `AccessApplicationService` in identityaccess

**Port/Adapter Structure**: Clear separation in `port/adapter` packages:
- `port/adapter/persistence/` - Persistence adapters (LevelDB, Hibernate, EventStore)
- `port/adapter/messaging/` - Messaging adapters (RabbitMQ, SlothMQ)
- `port/adapter/notification/` - Notification publishers
- `port/adapter/service/` - External service adapters

**Infrastructure Layer**: Separate `infrastructure/` packages for technical concerns:
- `iddd_identityaccess/infrastructure/persistence/` - Hibernate repositories
- `iddd_identityaccess/infrastructure/services/` - Infrastructure services

### Secondary Style: CQRS

The `iddd_collaboration` bounded context explicitly implements CQRS:

**Write Model**: Event-sourced domain model using `EventSourcedRootEntity`:
```java
public class Calendar extends EventSourcedRootEntity {
    protected void when(CalendarCreated anEvent) { ... }
    protected void when(CalendarDescriptionChanged anEvent) { ... }
}
```

**Read Model**: Separate query services with direct database queries:
```java
public class CalendarQueryService extends AbstractQueryService {
    public Collection<CalendarData> allCalendarsDataOfTenant(String aTenantId) {
        return this.queryObjects(CalendarData.class, "select ...");
    }
}
```

**Projections**: Event-driven projections updating read models:
```java
public class MySQLCalendarProjection extends AbstractProjection 
    implements EventDispatcher {
    protected void when(CalendarCreated anEvent) throws Exception { ... }
}
```

**README explicitly states**: "The iddd_collaboration project uses Event Sourcing and CQRS."

### Event Sourcing (within collaboration context)

Event sourcing infrastructure in `iddd_common`:
- `EventStore` interface with `LevelDBEventStore`, `HashMapEventStore` implementations
- `EventSourcedRootEntity` base class with `mutatingEvents()` and `apply()` methods
- Event stream versioning for optimistic concurrency

### Rejected Classifications

**Not Microservices**: Single deployable unit (Gradle build), no independent deployment, shared database in some contexts.

**Not Modular Monolith**: While there are modules, the primary organization principle is DDD bounded contexts, not just modular decomposition. The ports/adapters pattern adds hexagonal architecture on top.

**Not Layered (pure)**: While there are layers, the architecture is not primarily horizontal layering. The vertical slice of bounded contexts with domain model centrality is the dominant pattern.

**Not Serverless**: No function-based deployment model.

### Summary

This is a textbook reference implementation of **Domain-Driven Design** with **Hexagonal Architecture** as the structural foundation. **CQRS and Event Sourcing** are applied selectively within the collaboration bounded context. The architecture demonstrates proper separation of concerns with:
- Domain model isolated from infrastructure concerns
- Application services as entry points
- Ports (interfaces) in domain layer, adapters (implementations) in infrastructure
- Cross-context communication via domain events and RabbitMQ
