Looking at this Chatwoot repository, I need to analyze its architecture based on the provided evidence.

PRIMARY_STYLE: Modular Monolith
SECONDARY_STYLES: Event-Driven
CONFIDENCE: 0.9

REASONING:

This repository exhibits a **Modular Monolith** architecture as its primary style. The evidence is overwhelming:

1. **Clear Module Boundaries within Single Deployable Unit:**
   - `enterprise/` directory contains its own parallel app structure (`enterprise/app/models/`, `enterprise/app/services/`, `enterprise/app/controllers/`, etc.) - a separate module extending the core
   - `app/services/` contains 40+ distinct service modules: `auto_assignment/`, `messages/`, `imap/`, `widget/`, `sms/`, `notification/`, `mailbox/`, `google/`, `microsoft/`, `contacts/`, `linear/`, `crm/`, `labels/`, `facebook/`, `line/`, `telegram/`, `whatsapp/`, `twilio/`, etc.
   - `app/jobs/` organized by domain: `auto_assignment/`, `contact_ip_lookup_job.rb`, `webhook_job.rb`, `action_cable_broadcast_job.rb`, `channels/`, `campaigns/`, etc.
   - `app/listeners/` contains domain-specific event handlers: `webhook_listener.rb`, `agent_bot_listener.rb`, `notification_listener.rb`, `automation_rule_listener.rb`, `csat_survey_listener.rb`, `campaign_listener.rb`

2. **Single Codebase, Single Repository:**
   - The entire system is in one Rails monorepo (`app/` + `enterprise/` + `config/`)
   - Deployed as a unified application via Docker (see `docker-compose.yaml` - rails, sidekiq, vite services)
   - Database is shared (PostgreSQL)

3. **Secondary Style - Event-Driven:**
   - The codebase uses **Wisper** gem (`gem 'wisper', '2.0.0'` in Gemfile) for publish/subscribe pattern
   - `app/listeners/` directory with 12 listener classes responding to domain events
   - `app/dispatchers/` contains `async_dispatcher.rb`, `sync_dispatcher.rb`, `base_dispatcher.rb` - explicit event dispatching infrastructure
   - `lib/events/` directory exists with event definitions
   - ActionCable (WebSocket) is used for real-time features (`gem '@rails/actioncable'`)
   - Background jobs via Sidekiq handle asynchronous processing

4. **Other patterns present but not dominant:**
   - Layered architecture is inherent to Rails (models, controllers, services) but this is the standard Rails pattern, not the primary architectural choice
   - DDD concepts exist (services, builders, domain models) but not explicitly organized in bounded contexts

The modular monolith is the best fit because the system is a single deployable Rails application with clearly defined domain modules (services, jobs, listeners) that could theoretically be extracted into separate services in the future, but remain coupled in a single codebase.
