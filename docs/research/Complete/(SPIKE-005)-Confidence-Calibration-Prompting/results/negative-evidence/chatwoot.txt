The Chatwoot repository is a classic example of a **Layered** architecture, specifically following the Ruby on Rails "Model-View-Controller" (MVC) pattern. The codebase is organized by technical responsibility rather than business domain boundaries:

1.  **Presentation Layer:** Found in `app/controllers` (logic handling), `app/views` (templates), and a complex `app/javascript` directory (Frontend SPA using Vue.js/Vite as seen by `vite.sh` and `tailwind.config.js`).
2.  **Business Logic Layer:** Encapsulated within `app/models` (ActiveRecord models containing logic), `app/builders` (complex object construction), and `app/actions` (service-like objects for discrete operations like `contact_merge_action.rb`).
3.  **Data Access Layer:** Managed through `app/models` via ActiveRecord, which maps the domain objects to the database.

The project also heavily utilizes an **Event-Driven** secondary style for cross-module communication and background processing:
-   **Listeners:** `app/listeners/` contains multiple observers (e.g., `webhook_listener.rb`, `notification_listener.rb`) that react to events within the system without tightly coupling the trigger and the effect.
-   **Dispatchers:** `app/dispatchers/` (e.g., `async_dispatcher.rb`) manages the distribution of these events.
-   **Jobs:** `app/jobs/` implies an asynchronous task system (likely Sidekiq/Redis) for backgrounding work like `webhook_job.rb` and `notification/`.

### Rejected Styles:
-   **Microservices:** While the README mentions Docker and Kubernetes, the `app/` directory contains all business domains (Inboxes, Teams, Campaigns, Reports) in a single monolith. The `docker-compose.yaml` likely defines supporting services (Postgres, Redis, Rails, Sidekiq) rather than independent business services.
-   **Hexagonal Architecture:** There is no clear isolation of a "Core" from "Adapters." Controllers depend directly on Models, and Models often contain business logic and persistence concerns together.
-   **Modular Monolith:** While there is some separation, the structure follows standard Rails conventions where all models are in one flat directory (`app/models`) rather than encapsulated in isolated modules with enforced boundaries.
