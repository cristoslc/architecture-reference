# posthog ŌĆö Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/PostHog/posthog
**Classification:** Modular Monolith + Event-Driven + Service-Based
**Confidence:** 0.91

## Summary

PostHog is an all-in-one open-source product analytics platform structured as a deliberate Modular Monolith (a single Django application with ~40 enforced product boundaries via `tach` and facade contracts) layered over an Event-Driven ingestion backbone (Kafka/Redpanda at the center of the write path) and augmented by a constellation of coarse-grained specialized services (Rust, Node.js, Go). The dominant runtime coupling is through shared Kafka topics and shared PostgreSQL/ClickHouse databases ŌĆö which distinguishes it clearly from true Microservices, where services own isolated stores.

### Key Architectural Files

- `products/architecture.md`: Explicit documentation of the modular monolith target: facade-based inter-module communication, frozen dataclasses as contracts, tach-enforced import boundaries.
- `tach.toml`: 333-line dependency graph enforcing that each of 40+ `products.*` modules can only import through declared dependencies; products in `layer = "products"` declare explicit `interfaces = [...]` facade paths.
- `docker-compose.base.yml`: The Caddy proxy routes `/e`, `/batch` to `capture:3000` (Rust), `/flags` to `feature-flags:3001` (Rust), `/s` to `replay-capture` (Rust), `/i/v0/ai` to `capture-ai` (Rust), and all other paths to `web:8000` (Django). Kafka (Redpanda) is a first-class infrastructure service alongside PostgreSQL, ClickHouse, Redis, and MinIO/S3.
- `posthog/celery.py`: `app.autodiscover_tasks()` loads tasks from all Django apps. Prometheus instrumentation wraps every task. `CeleryQueue.LONG_RUNNING` separates cohort-calculation workers.
- `posthog/hogql_queries/query_runner.py`: Abstract `QueryRunner` base with 40+ concrete implementations ŌĆö one per analytics query type. This is a distinct read-path abstraction isolated from the write path.
- `rust/cymbal/src/stages/`: Named pipeline stages (`prep`, `person`, `group`, `geoip`, `exception`, `clean`) consumed by `pipeline.rs` ŌĆö a classic pipeline pattern within the error-tracking service.
- `ee/hogai/chat_agent/graph.py` + `ee/hogai/research_agent/graph.py`: LangGraph state-machine graphs implementing AI assistants with tool calling.
- `nodejs/src/ingestion/ingestion-consumer.ts`: Kafka consumer reading `events_plugin_ingestion`, processing through ordered pipeline steps, writing to ClickHouse and PostgreSQL.

### Patterns Found

**Modular Monolith (Primary Style)**

The core Django application (`posthog/` + `products/`) is a single deployable: one `manage.py`, one Django settings file, one Celery app, one set of Postgres/ClickHouse migrations. Despite this, the codebase is undergoing a deliberate modularization effort:

- `tach.toml` enforces that every `products.*` module declares its upstream imports explicitly. Products in `layer = "products"` (isolated) can only be consumed through their `facade/api.py` and `facade/contracts.py` ŌĆö frozen dataclasses with no Django or DRF imports. Products in `layer = "non_isolated"` are undergoing migration toward isolation.
- Each product under `products/` owns a `backend/` Django app with `apps.py`, `models.py`, `logic.py`, `tasks/`, `facade/`, and `presentation/` ŌĆö a full vertical slice registered via `INSTALLED_APPS`.
- Turborepo (`turbo.json`) enables selective testing: only products whose contracts changed need to rerun their test suite.
- The `ee/` directory extends `posthog/` with enterprise features using a sibling-module pattern; `tach.toml` treats `ee` as a peer module, not a separate service.

**Event-Driven (Strong Secondary Style)**

Kafka/Redpanda is the structural backbone of the data path, not just a messaging add-on:

- Rust `capture` service writes to `events_plugin_ingestion` Kafka topic.
- Node.js plugin server consumes `events_plugin_ingestion`, processes events through an ordered ingestion pipeline (preprocessing, person resolution, property enrichment), then writes to ClickHouse and emits downstream Kafka messages.
- `capture-logs` writes to `logs_ingestion` topic; `property-defs-rs` consumes events to maintain property definitions.
- `cyclotron-janitor` emits metrics to `clickhouse_app_metrics2` topic.
- Celery uses Redis as its broker for internal async tasks (a separate queue plane from the event ingestion path).
- Temporal handles long-running batch workflow orchestration (batch exports, data imports).
- Dagster DAGs handle data engineering operations (materialized columns, ClickHouse backfills).

**Service-Based (Secondary Style)**

Rather than microservices (fine-grained, database-per-service), PostHog deploys coarse-grained specialized services that share infrastructure:

- Rust services (`capture`, `feature-flags`, `cymbal`, `property-defs-rs`, `hook-worker`, `cyclotron-*`, `personhog-*`) are independently deployable but share PostgreSQL, ClickHouse, Redis, and Kafka with the Django monolith.
- Node.js plugin server is a separate process but shares all databases.
- `services/llm-gateway` (FastAPI) and `services/mcp` (Cloudflare Workers) are wholly separate deployments with no shared DB access; these are the closest to true microservices.
- The Caddy reverse proxy acts as the API gateway dispatching routes to the appropriate service.

**Pipeline (Significant Internal Pattern)**

Multiple internal subsystems are organized as discrete, ordered processing pipelines:

- `nodejs/src/ingestion/pipelines/`: ordered ingestion pipeline for event preprocessing and person resolution.
- `rust/cymbal/src/stages/` + `pipeline.rs`: named stages (`prep`, `person`, `group`, `geoip`, `exception`, `clean`) chained into a processing pipeline for error tracking.
- `posthog/cdp/`: Hog Function execution pipeline for real-time transformations.
- Temporal workflows for batch exports define sequences of activities (extract ŌåÆ transform ŌåÆ load).

**CQRS-Light (Read/Write Separation)**

PostHog separates its read and write paths architecturally, though it does not use CQRS as a primary organizing principle:

- Write path: Rust `capture` ŌåÆ Kafka ŌåÆ Node.js plugin server ŌåÆ ClickHouse/Postgres.
- Read path: Django DRF API ŌåÆ HogQL query runner ŌåÆ ClickHouse (analytics) or Postgres (operational data).
- The `hogql_queries/` directory contains 40+ `QueryRunner` implementations as a distinct read-path abstraction, compiled via the HogQL AST to ClickHouse SQL.

**AI Agent Pattern (Emerging)**

`ee/hogai/` implements LangGraph-based AI agents (Max AI assistant):

- `chat_agent/graph.py`: multi-node LangGraph state machine with tool-calling nodes.
- `research_agent/graph.py`: separate agent graph for research tasks.
- `MaxTool` plugin system (auto-discovered from `products/*/backend/max_tools.py`) allows each product to register callable tools into the agent's tool registry.

### What This Is NOT

- **NOT Microservices**: Services share PostgreSQL, ClickHouse, Redis, and Kafka. There is no database-per-service boundary. The Django monolith and all Rust/Node.js services read and write the same Postgres and ClickHouse databases directly. Service count is coarse-grained (~8 named services) not fine-grained.
- **NOT Hexagonal / Ports & Adapters**: The `facade/api.py` pattern within products is an inter-module boundary, not a ports-and-adapters abstraction over external dependencies. Django ORM models are not hidden behind repository ports.
- **NOT Layered**: The primary organization is vertical product slices, not horizontal layers. Although individual products have a `presentation ŌåÆ logic ŌåÆ models` internal stack, the system architecture is not a horizontal layers model.
- **NOT Serverless**: All services are long-running Docker containers. Cloudflare Workers in `services/mcp` and `services/oauth-proxy` are serverless edge functions but constitute a tiny fraction of the system.
- **NOT Space-Based**: No in-memory data grid, no tuple-space partitioning.

## Quality Attributes

- **Scalability**: Rust capture services handle ingest bursts independent of Django; ClickHouse provides columnar analytics at scale; Kafka decouples ingest rate from processing rate; Cyclotron provides a Postgres-backed job queue that scales processing workers independently; feature-flags service is stateless and horizontally scalable with Redis caching.
- **Modularity**: `tach.toml` enforces 40+ product boundaries at import time; facade contracts (frozen dataclasses) are the sole inter-product interface; Turborepo selective testing minimizes CI scope per product change.
- **Observability**: OpenTelemetry instrumentation throughout (OTLP collector in docker-compose, Jaeger for tracing, Prometheus metrics from every service); Celery Prometheus counters/histograms; structured logging via structlog; Sentry error capture.
- **Deployability**: Single `docker-compose.hobby.yml` for self-hosted; Kubernetes Helm charts for production; Rust services built from a shared workspace Dockerfile (`BIN` arg selects binary); Turborepo caches frontend and backend build artifacts.
- **Evolvability**: Active migration from `non_isolated` to `isolated` product modules with enforced facade boundaries; `products/architecture.md` documents the target structure; tach prevents regression; ADR-style internal documentation.
- **Fault Tolerance**: Kafka provides durable event buffering; Temporal provides workflow retry and durable execution for batch exports; Celery task retry semantics; Cyclotron janitor reclaims stalled jobs; Redis-backed rate limiting in capture and feature-flags services.
- **Performance**: Rust services (capture, feature-flags, cymbal) provide low-latency hot paths; ClickHouse columnar store for analytics; HogQL AST compiler generates optimized ClickHouse SQL; property-defs-rs asynchronously updates property definitions without blocking the critical path.
