# Architecture Report: ERPNext

**Date:** 2026-03-11
**Source:** https://github.com/frappe/erpnext
**Classification:** Modular Monolith (primary), Layered (secondary), Microkernel (tertiary)
**Confidence:** 0.92
**Method:** deep-analysis
**Model:** claude-sonnet-4-6

---

## Summary

ERPNext is an open-source ERP application built as a Frappe Framework "app." It is a large-scale **Modular Monolith** organized around 20+ named functional domains (Accounts, CRM, Selling, Buying, Stock, Manufacturing, Projects, Support, HR, etc.), all deployed as a single Python package. It also exhibits a strong **Layered** inheritance hierarchy across its shared controller stack, and it participates in a **Microkernel** extension model where the Frappe framework acts as the kernel and ERPNext (and further custom apps) act as installable plugins.

```
erpnext/
├── accounts/         # Financial accounting, GL, ledgers, invoicing
├── buying/           # Procurement, purchase orders, supplier management
├── selling/          # Quotations, sales orders, CRM pipeline
├── stock/            # Inventory, warehouse, stock ledger, valuation
├── manufacturing/    # BOMs, work orders, production planning
├── quality_management/
├── subcontracting/
├── support/          # Helpdesk, SLA, issues
├── crm/              # Leads, opportunities, email campaigns
├── projects/         # Tasks, timesheets
├── assets/           # Fixed asset lifecycle
├── regional/         # Country-specific tax overrides (Italy, UAE, Saudi Arabia, France)
├── erpnext_integrations/  # External integrations (Plaid)
├── controllers/      # Shared base controller hierarchy
├── utilities/        # Cross-cutting utilities
└── hooks.py          # Frappe app manifest (extension points, scheduler events)
```

Single deployable unit defined in `pyproject.toml` with one package (`erpnext`) depending on `frappe>=17.0.0-dev`.

## Architectural Evidence

### 1. Modular Monolith (Primary)

`erpnext/modules.txt` lists 20 named modules:

```
Accounts, CRM, Buying, Projects, Selling, Setup, Manufacturing,
Stock, Support, Utilities, Assets, Portal, Maintenance, Regional,
ERPNext Integrations, Quality Management, Communication, Telephony,
Bulk Transaction, Subcontracting, EDI
```

Each module under `erpnext/<module>/` contains:
- `doctype/` — schema-first entity definitions (JSON + Python + JS triads)
- `report/` — read-side reporting
- `workspace/` — UI workspace configuration
- `page/` — full-page views

Modules are co-located in a single Python package and share the same database instance and process. There are no runtime module boundaries, no inter-module APIs, and no module-specific schemas. Cross-module imports are pervasive and unrestricted:

```python
# erpnext/selling/doctype/sales_order/sales_order.py — 31 cross-module imports including:
from erpnext.accounts.doctype.sales_invoice.sales_invoice import ...
from erpnext.accounts.party import get_party_account
from erpnext.manufacturing.doctype.blanket_order.blanket_order import ...
from erpnext.stock.doctype.item.item import get_item_defaults
from erpnext.stock.stock_balance import get_reserved_qty
```

This is a **Modular Monolith** by organization (not a pure monolith), but the boundaries are soft — they are file-system conventions, not enforced module isolation.

### 2. Layered Architecture (Secondary)

The shared controller stack defines a strict horizontal layering by inheritance:

```
frappe.model.document.Document
    └── StatusUpdater          (erpnext/controllers/status_updater.py)
        └── TransactionBase    (erpnext/utilities/transaction_base.py)
            └── AccountsController  (erpnext/controllers/accounts_controller.py)
                └── StockController  (erpnext/controllers/stock_controller.py)
                    ├── SellingController  (erpnext/controllers/selling_controller.py)
                    └── SubcontractingController
                        └── BuyingController  (erpnext/controllers/buying_controller.py)
```

Each layer adds concerns:
- `StatusUpdater` — document lifecycle state transitions
- `TransactionBase` — posting date, UOM validation, cross-document consistency
- `AccountsController` — GL entry creation, pricing rules, tax and totals, payment terms (~1000 lines)
- `StockController` — SLE creation, bin updates, serial/batch bundle management
- `SellingController` / `BuyingController` — domain-specific validation

Each concrete DocType (e.g., `SalesOrder`, `PurchaseOrder`, `StockEntry`) inherits from its appropriate controller layer. This is textbook horizontal layering.

### 3. Microkernel / Plugin Pattern (Tertiary)

ERPNext is itself deployed as a "plugin" onto the Frappe kernel via the Bench tool. `hooks.py` is the app manifest wired into the Frappe extension system:

- `doc_events` — hook functions onto DocType lifecycle events (`validate`, `on_submit`, `on_cancel`, `after_insert`)
- `scheduler_events` — cron/daily/hourly background job registrations
- `regional_overrides` — region-specific function substitutions for Italy, UAE, Saudi Arabia, France
- `override_whitelisted_methods` — replacing Frappe core endpoints
- `extend_doctype_class` — subclassing Frappe base DocTypes

This mirrors a microkernel model: Frappe provides the kernel (ORM, router, background scheduler, auth, multi-tenancy), and ERPNext is the primary plugin. Third-party apps can layer on top of ERPNext using the same mechanism.

### 4. Ledger / Append-Only Pattern (Notable Feature, not a Primary Style)

`erpnext/accounts/general_ledger.py` and `stock/stock_ledger.py` use append-only write patterns: GL Entries and Stock Ledger Entries are never updated in place; corrections are posted as reversal entries. An opt-in `enable_immutable_ledger` setting (`is_immutable_ledger_enabled()`) makes this stricter. This is an architectural characteristic of accounting systems but does not make the overall system CQRS — there is no separate read model or command/query bus.

### 5. What This Is Not

- **Not Microservices**: Single `pyproject.toml`, single Python package, single database, single process.
- **Not Event-Driven**: No message broker (no Kafka, RabbitMQ, Celery). Background jobs use Frappe's Redis-backed `frappe.enqueue()` for deferred work, but this is operational queuing, not an event-driven architecture. There is no event bus, event sourcing, or domain event pattern.
- **Not CQRS**: GL Entry reports and queries are plain SQL via the Frappe ORM against the same write tables. No separate read model.
- **Not DDD**: No bounded contexts, aggregates, value objects, or domain repositories. Modules freely import each other's internals.
- **Not Hexagonal**: Business logic in controllers directly depends on `frappe.db.*` and ORM calls. No ports/adapters.
- **Not Service-Based**: No separately deployed service components communicating over a network.
- **Not Serverless**: Self-hosted, persistent server process.

## Conclusion

ERPNext is a **Modular Monolith** with a strong **Layered** controller hierarchy and a **Microkernel**-style plugin relationship with the Frappe framework. The 20+ functional modules provide domain organization but without enforced runtime boundaries. The tiered controller inheritance (StatusUpdater → TransactionBase → AccountsController → StockController → domain controllers) is the clearest architectural signal. The Frappe plugin model (hooks.py manifest, doc_events, regional_overrides) gives the system its extensibility characteristics. The previous classification of "Layered" is accurate as a secondary style but underweights the dominant modular decomposition and the microkernel relationship with Frappe.
