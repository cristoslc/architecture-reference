# Architecture Report: Apache Geode

**Date:** 2026-03-11
**Analyst:** claude-sonnet-4-6 (deep-analysis)
**Source:** https://github.com/apache/geode
**Commit:** depth-1 clone, HEAD as of 2026-03-11

## Summary

Apache Geode is a **Modular Monolith** at the code-organization level, with a strong secondary **Space-Based** architectural identity that reflects the runtime topology. The repository is a single Gradle multi-project build that packages into one distribution artifact (`apache-geode`). Internally the codebase is decomposed into 30+ enforced Gradle subprojects with explicit `api`/`implementation` visibility boundaries. At runtime, Geode nodes form a distributed data grid: in-memory data regions are partitioned or replicated across a cluster of JVM processes, each node owning a share of the total data and computing locally against it — the defining characteristic of Space-Based architecture. A ServiceLoader-based `CacheService` plugin mechanism provides a Microkernel-style extensibility layer, allowing optional capabilities (HTTP API, WAN gateway, Lucene indexing, CQ) to plug into the cache kernel without coupling to it at compile time.

## Evidence

### 1. Modular Monolith (primary)

**Single assembly artifact**

`geode-assembly/build.gradle` produces a single distribution (`apache-geode`) via the Gradle `distribution` plugin and `com.palantir.docker`. `geode-server-all/build.gradle` aggregates every module into one server JAR through `api` and `runtimeOnly` project dependencies:

```groovy
api(project(':geode-core'))
api(project(':geode-gfsh'))
api(project(':geode-lucene'))
api(project(':geode-management'))
api(project(':geode-wan'))
...
runtimeOnly(project(':geode-memcached'))
runtimeOnly(project(':geode-tcp-server'))
```

All modules run in the same JVM process. There are no cross-service network boundaries within a single Geode node.

**Enforced module boundaries via Gradle**

`settings.gradle` declares 30+ subprojects that define the module structure:

```groovy
include 'geode-common'
include 'geode-logging'
include 'geode-serialization'
include 'geode-membership'
include 'geode-tcp-server'
include 'geode-core'
include 'geode-wan'
include 'geode-cq'
include 'geode-lucene'
include 'geode-memcached'
include 'geode-gfsh'
include 'geode-web'
include 'geode-web-api'
include 'geode-web-management'
include 'geode-management'
include 'geode-connectors'
include 'geode-deployment'
include 'geode-server-all'
```

Build dependencies use `api` vs. `implementation` to enforce visibility. `geode-core/build.gradle` declares:

```groovy
api(project(':geode-common'))
api(project(':geode-serialization'))
api(project(':geode-management'))
implementation(project(':geode-logging'))
implementation(project(':geode-membership'))
implementation(project(':geode-tcp-server'))
```

`internal` packages throughout (`org.apache.geode.internal.*`, `org.apache.geode.internal.cache.*`) enforce the public-API/implementation boundary at the package level.

**Clear horizontal layer stack**

| Layer | Modules/Packages |
|---|---|
| Foundation | `geode-common`, `geode-logging`, `geode-serialization`, `geode-unsafe` |
| Transport/Membership | `geode-membership` (JGroups via `JGroupsMessenger`), `geode-tcp-server` |
| Core Data Grid | `geode-core` — `Cache`, `Region`, `PartitionedRegion`, `DistributedRegion`, `GemFireCacheImpl` |
| Feature Modules | `geode-wan`, `geode-cq`, `geode-lucene`, `geode-memcached`, `geode-connectors` |
| Management Plane | `geode-management`, `geode-gfsh`, `geode-web-management`, `geode-web-api`, `geode-pulse` |
| Assembly | `geode-assembly`, `geode-server-all` |

### 2. Space-Based (secondary)

The runtime topology is a textbook data grid. Every Geode node (a JVM process) holds a portion of the distributed data in-memory and executes operations locally against that data — the processing-unit concept from Space-Based architecture.

**Partitioned and replicated regions**

`geode-core/src/main/java/org/apache/geode/internal/cache/PartitionedRegion.java` (`extends LocalRegion`) is the implementation of distributed data partitioning. Data is split into buckets via `BucketRegion`, `BucketAdvisor`, and `PartitionedRegionDataStore`. `DistributedRegion.java` (`extends LocalRegion`) implements full replication across members.

The public API in `geode-core/src/main/java/org/apache/geode/cache/` exposes:
- `PartitionAttributes` / `PartitionAttributesFactory` — configure bucket count, redundancy copies, colocated-with
- `PartitionResolver` / `FixedPartitionResolver` — control data routing to buckets
- `DataPolicy` — PARTITION, REPLICATE, PERSISTENT_PARTITION, PERSISTENT_REPLICATE

**Shared-nothing persistence**

The README states explicitly: _"Combines redundancy, replication, and a 'shared nothing' persistence architecture to deliver fail-safe reliability."_ Each node operates on its own memory slice; there is no central database or shared storage tier.

**Compute-on-data (Function Execution)**

`geode-core/src/main/java/org/apache/geode/cache/execute/` defines the Function Execution SPI:
- `Function<T>` — user-defined code run co-located with the target data
- `FunctionService` — routes execution to members that hold the relevant buckets/keys
- `RegionFunctionContext` — gives the function access to the local data partition

This is the canonical space-based pattern: move computation to the data, not the other way around.

**Asynchronous event infrastructure**

`geode-core/src/main/java/org/apache/geode/cache/asyncqueue/` provides `AsyncEventQueue` and `AsyncEventListener` for reliable asynchronous notification of data mutations. `CacheListener`, `CacheWriter`, and `CacheLoader` are synchronous extension points wired directly into the Region mutation path.

**Cluster membership via JGroups**

`geode-membership/src/main/java/.../gms/messenger/JGroupsMessenger.java` — reliable group communication over TCP/UDP for node discovery, failure detection, and partition-based messaging. The locator process (configured via `DistributedSystem` / `Locator.java`) coordinates dynamic membership without a central coordinator.

### 3. Microkernel (tertiary)

The `CacheService` SPI provides a ServiceLoader-based plugin mechanism. Optional capabilities register themselves as `CacheService` implementations and are loaded at cache initialization by `GemFireCacheImpl.initializeServices()`:

```java
// GemFireCacheImpl.java line 1499
private void initializeServices() {
  ServiceLoader<CacheService> loader = ServiceLoader.load(CacheService.class);
  for (CacheService service : loader) {
    if (service.init(this)) {
      services.put(service.getInterface(), service);
    }
  }
}
```

Each optional module registers its `CacheService` implementation in `META-INF/services/org.apache.geode.internal.cache.CacheService`:

| Module | Registered service |
|---|---|
| `geode-http-service` | `InternalHttpService` (REST management API) |
| `geode-wan` | `WanCopyRegionFunctionService` (WAN gateway cross-datacenter) |
| `geode-lucene` | `LuceneServiceImpl` (full-text indexing) |

The `Declarable` interface in `geode-core/src/main/java/org/apache/geode/cache/Declarable.java` provides a secondary extensibility mechanism: user code (CacheLoader, CacheWriter, CacheListener, Function, PartitionResolver) is declared in XML and loaded by name, further decoupling user extensions from the kernel.

### What Was Ruled Out

**Microservices** — A single JVM process constitutes a Geode server. Nodes form a cluster through peer-to-peer membership, not independently deployed microservices with private data stores. All `Region` data within a cluster forms one logical namespace.

**Event-Driven** — While Geode provides `CacheListener`, `AsyncEventQueue`, and Continuous Queries, these are feature capabilities wired into the data mutation path. The primary organizing principle is partitioned in-memory data management, not message-broker-mediated event flow.

**Layered** — The codebase has layered internal structure (foundation → transport → core → features → management), but the dominant organizing principle is modular decomposition around capabilities (partitioning, WAN, Lucene, CQ), not strict horizontal tiers with upward-only dependencies across all modules. Layered is a supporting pattern within the Modular Monolith, not a separate primary style.

**Pipeline** — No sequential processing-stage topology. Data enters via client operations and is stored/replicated in regions; there is no chained transformation pipeline as a structural principle.

**CQRS** — There is no command/query separation. Read and write operations hit the same `Region` and `PartitionedRegionDataStore`. Continuous Queries (`geode-cq`) are a streaming query feature, not CQRS.

**Domain-Driven Design** — Code is organized by technical capability (partitioning, serialization, membership, WAN) rather than business bounded contexts.

**Hexagonal Architecture** — The management REST layer and client protocol layer are swappable via `CacheService` and `Declarable` extension points, but the code is not organized around explicit inbound/outbound port contracts in the Hexagonal sense.
