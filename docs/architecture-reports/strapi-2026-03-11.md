# Architecture Report: Strapi

**Date:** 2026-03-11
**Repository:** https://github.com/strapi/strapi
**Language:** TypeScript / JavaScript
**Domain:** Headless CMS
**Stars:** ~71,500 | **Forks:** ~9,500

---

## Executive Summary

Strapi is an open-source headless CMS framework built on Node.js (Koa) with a TypeScript monorepo structure managed by Nx and Yarn workspaces. Its dominant architectural style is **Microkernel (Plugin)**: a minimal core runtime that loads, registers, and orchestrates discrete functional plugins. Inside every plugin and the core itself, a consistent **Layered** MVC structure organizes routes → controllers → services → database. An internal **Event-Driven** backbone — the EventHub and lifecycle system — decouples content mutations from downstream side-effects such as webhooks. The result is a single-process, single-database CMS where extensibility is achieved through well-defined plugin contracts and provider swap-outs rather than through network decomposition.

## Architectural Styles

### 1. Microkernel (Primary)

The most defining structural property is Strapi's plugin system.

**Core runtime (`packages/core/core/src/`):**

- `Strapi` class extends `Container` — a simple service-locator map that stores named factories and lazily instantiates them on first `get()` call.
- `providers/registries.ts` registers all core registries (content-types, components, services, controllers, middlewares, hooks, plugins, etc.) into the container at `init()` time.
- `loaders/` discovers and loads plugins from both `INTERNAL_PLUGINS` (always present) and user-configured external plugins found in `node_modules` or local paths.
- Every plugin implements a well-defined lifecycle contract:

```typescript
// defaultPlugin contract (loaders/plugins/index.ts)
{
  bootstrap() {},
  destroy() {},
  register() {},
  config: { default: {}, validator() {} },
  routes: [],
  controllers: {},
  services: {},
  policies: {},
  middlewares: {},
  contentTypes: {},
}
```

**Internal plugins (always loaded):** content-manager, content-type-builder, email, upload, i18n, content-releases, review-workflows.

**External plugins:** graphql, users-permissions, sentry, color-picker, cloud, documentation — each published as a separate npm package implementing the same contract.

**User extensions:** An `applyUserExtension()` mechanism lets application developers override plugin schemas and inject custom `strapi-server.js` files to mutate plugin behaviour, following the microkernel "extension point" pattern without forking core.

**Provider pattern:** `packages/providers/` contains swappable implementations for email (Amazon SES, Mailgun, Sendgrid, Nodemailer, Sendmail) and upload (AWS S3, Cloudinary, local). Providers implement a minimal `init()` interface, enabling runtime replacement via config without touching core logic.

**Admin panel plugin system:** `StrapiApp.tsx` exposes `registerPlugin()`, injection zones, and hook overrides on the React side, mirroring the server-side plugin contract for the frontend.

### 2. Layered (Secondary — within core and every plugin)

Within the Microkernel host, every plugin and core API follows a consistent three-tier layered structure:

| Layer | Files |
|---|---|
| **HTTP / Routing** | `server/routes/`, `register-routes.ts`, Koa middleware stack |
| **Controllers** | `server/controllers/` — request/response shaping, sanitization, validation |
| **Services** | `server/services/` — domain operations, business rules |
| **Data** | `@strapi/database` — Knex-backed entity-manager, query builder, schema sync |

The **Document Service** (`services/document-service/`) is the current primary data-access API. It wraps the lower-level Entity Service (deprecated) and adds Draft & Publish state management, locale branching (i18n), component handling, and relation bidirectionality logic. It exposes repository-style methods (`findMany`, `create`, `update`, `publish`, `unpublish`, `discard`) that delegate to `strapi.db.query()`.

The **Core API** (`core-api/controller/`, `core-api/service/`) generates default CRUD controllers and services for every user-defined content type automatically, providing ready-made REST endpoints without user code.

---

### 3. Event-Driven (Secondary — internal, in-process)

Strapi embeds an internal event bus to decouple mutations from side-effects:

**EventHub (`services/event-hub.ts`):**
- Pub/sub with named events (`entry.create`, `entry.update`, `entry.delete`, `entry.publish`, `entry.unpublish`, `entry.draft-discard`).
- Supports `on`, `once`, `subscribe` (global interceptor), and `off` patterns.
- All calls are `async/await`, processed serially per subscriber.

**Document Service Event Manager (`services/document-service/events.ts`):**
- Emits entry lifecycle events **after** the wrapping database transaction commits (`onCommit` hook), ensuring consumers see consistent data.
- Populates and sanitizes the entry before emitting so event payloads carry full object state.

**Webhook Runner (`services/webhook-runner.ts`):**
- Subscribes to the EventHub.
- Routes events to registered webhook endpoints via a `WorkerQueue` (in-process, concurrency-limited to 5).
- Decouples CMS mutations from HTTP outbound calls without requiring a message broker.

**Database Lifecycle Provider (`packages/core/database/src/lifecycles/`):**
- Separate, lower-level subscriber system on raw database operations (`beforeCreate`, `afterCreate`, etc.).
- Used internally for timestamp injection and model-level hook dispatch.

This event system is in-process and lightweight — it is not a distributed event bus — but the architectural intent and decoupling pattern are unambiguously event-driven.

## Key Architectural Patterns and Qualifiers

### Dependency Injection / Service Locator

The `Container` class (`container.ts`) is a service-locator that resolves named services lazily. All core services (db, eventHub, documents, auth, cron, requestContext, etc.) are registered as named factories and retrieved via `strapi.get('name')`. Getters on the `Strapi` class (`strapi.db`, `strapi.documents`, etc.) are thin wrappers over this container. This provides loose coupling without a full DI framework.

### Decorator Pattern on Services

The deprecated Entity Service applies decorators to extend behaviour:

```typescript
service.decorate((oldService) => ({
  ...oldService,
  async findMany(...) { /* wrap */ }
}));
```

This allows the Document Service and EE (Enterprise Edition) features to augment core service behaviour without inheritance.

### Repository Pattern

The Document Service exposes a content-type-scoped repository (`createContentTypeRepository`) with methods aligned to the Repository pattern: `findMany`, `findFirst`, `findOne`, `create`, `update`, `delete`, `publish`, `unpublish`, `clone`, `discard`. The factory is called per content-type UID, producing isolated repositories with type-specific validation and transformation logic.

### Strategy / Provider Pattern

Upload and email capabilities are provided through swappable providers. Each provider exports an `init()` factory returning a standard interface. The platform is agnostic to the specific provider at build time; selection occurs at runtime via configuration.

### Pipeline in Data Transfer

`packages/core/data-transfer/src/engine/index.ts` implements a streaming pipeline (Node.js `stream/promises pipeline`) for import/export operations. Data flows through ordered stages: `entities → links → assets → schemas → configuration`. Source and destination providers implement `ISourceProvider` / `IDestinationProvider` interfaces, and transforms can be injected between them.

## Deployment Model

Strapi runs as a **single Node.js process**. There is no service-to-service RPC, no distributed runtime, and no separate worker services (the webhook WorkerQueue is in-memory). The monorepo packages are library modules co-deployed into one process. Database connectivity is a single Knex connection pool to a relational database (SQLite, MySQL, PostgreSQL).

The `@strapi/strapi` package provides the CLI (`strapi start`, `strapi develop`, `strapi build`, etc.) which bootstraps the single Koa HTTP server embedding all loaded plugins.

## Boundary Decisions and Trade-offs

1. **Microkernel over Microservices:** Strapi deliberately stays in-process. Extensibility is plugin-based, not service-based. This simplifies deployment for its target audience (developers building CMS-backed sites) but limits independent scaling of individual capabilities.

2. **Global `strapi` singleton:** The framework mounts the `Strapi` instance as `global.strapi`. This is a pragmatic developer-experience decision that enables plugin code to call `strapi.service(uid)` anywhere, but it constrains multi-instance scenarios and complicates pure unit testing.

3. **Entity Service deprecation → Document Service:** The original Entity Service operated at the database-row level. The Document Service adds a stable `documentId` abstraction that ties together draft/published versions and locales as a single logical document. This is an architectural upgrade that improves the content lifecycle model.

4. **EE (Enterprise Edition) overlay:** Enterprise features (Review Workflows, advanced permissions, SSO) are loaded conditionally via the `strapi.EE` flag and a separate `ee/` directory per package. The CE/EE split is implemented as an in-process feature flag, not a different deployment topology.

---

## Conclusion

Strapi's architecture is best classified as **Microkernel + Layered**, with a meaningful in-process **Event-Driven** component. The core provides a minimal runtime, registry, and lifecycle management surface; all functional capabilities arrive as plugins conforming to a documented contract. Inside the plugin boundary, a classic layered decomposition (routes → controllers → services → database) provides familiar structure. The event system decouples lifecycle side-effects (webhooks, lifecycle hooks) from the mutation path. This combination makes Strapi highly extensible for its target domain while remaining operationally simple to deploy.
