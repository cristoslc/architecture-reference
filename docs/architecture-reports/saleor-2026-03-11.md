## Style Rationales

**Modular Monolith (primary, 0.90):** Saleor is a single Django application deployed from one codebase and one process boundary (WSGI/ASGI + Celery workers). The `INSTALLED_APPS` list in `saleor/settings.py` enumerates 18 first-party Django apps, each a cohesive vertical slice of the commerce domain: `saleor.account`, `saleor.order`, `saleor.checkout`, `saleor.product`, `saleor.payment`, `saleor.discount`, `saleor.giftcard`, `saleor.warehouse`, `saleor.shipping`, `saleor.tax`, `saleor.channel`, `saleor.attribute`, `saleor.webhook`, `saleor.app`, `saleor.plugins`, `saleor.csv`, `saleor.invoice`, `saleor.thumbnail`. Every app follows Django's module conventions with its own `models.py`, `migrations/`, optional `tasks.py`, and `tests/`. All apps share a single PostgreSQL database instance (default + optional read replica via `DATABASE_ROUTERS = ["saleor.core.db_routers.PrimaryReplicaRouter"]`). The entire surface area is exposed through a single GraphQL endpoint at `/graphql/`, composed in `saleor/graphql/api.py` by assembling per-domain `Query`, `Mutation`, and `Subscription` mixins from each app's `schema.py`. Intra-domain calls are direct Python function calls â€” e.g., `checkout/complete_checkout.py` imports `order.actions.order_created` and `warehouse.management.decrease_stock` directly. There is no network boundary between domain modules.

**Event-Driven (secondary, 0.87):** A first-class webhook subsystem in `saleor/webhook/` provides asynchronous and synchronous event delivery to registered external apps. `WebhookEventAsyncType` defines over 100 named domain events (ORDER_CREATED, CHECKOUT_UPDATED, PRODUCT_VARIANT_OUT_OF_STOCK, etc.) and `WebhookEventSyncType` defines 20+ synchronous callback events (CHECKOUT_CALCULATE_TAXES, SHIPPING_LIST_METHODS_FOR_CHECKOUT, PAYMENT_AUTHORIZE, etc.). Celery (backed by an AMQP broker â€” CloudAMQP or AWS SQS) is the async task runtime: `saleor/celeryconf.py` autodiscovers tasks across all domain apps, and `saleor/webhook/transport/asynchronous/transport.py` implements Celery task delivery with retry-backoff (max 5 retries, 10-second backoff). Webhook delivery supports three transport schemes: HTTP/HTTPS (`send_webhook_using_http`), AWS SQS (`send_webhook_using_aws_sqs`), and Google Cloud Pub/Sub (`send_webhook_using_google_cloud_pubsub`) â€” defined in `saleor/webhook/transport/utils.py`. Synchronous webhook events are dispatched via `saleor/webhook/transport/synchronous/transport.py` and used for real-time tax computation, shipping rate resolution, and payment gateway delegation. A circuit breaker (`saleor/webhook/circuit_breaker/breaker_board.py`) guards sync webhook calls with configurable TTL, failure count thresholds, and cooldown periods backed by Redis.

**Microkernel (tertiary, 0.80):** The plugin system in `saleor/plugins/` acts as a microkernel extension mechanism. `BasePlugin` in `saleor/plugins/base_plugin.py` (1,899 lines) defines over 100 hook methods covering taxes, payments, email notifications, order lifecycle events, and more. The `PluginsManager` in `saleor/plugins/manager.py` acts as the kernel â€” it loads configured plugins from `settings.PLUGINS`, chains them via `__run_method_on_plugins`, and dispatches each hook to all registered plugins in sequence. Bundled plugins include `admin_email`, `user_email`, `sendgrid`, `openid_connect`, `avatax` (tax calculation), and a webhook plugin for app-defined handlers. External apps (Django App model in `saleor/app/`) can register webhooks that receive synchronous callbacks for tax/shipping/payment hooks, effectively acting as external plugin implementations. This architecture allows commerce behavior to be extended without modifying the core, which is the defining microkernel characteristic.

## Domain

GraphQL-native, API-only composable commerce platform. Core service domains: storefront catalog (products, variants, attributes), multi-channel pricing, checkout and order lifecycle, payment processing (legacy gateway + Transaction API), tax calculation, promotions and discounts, gift cards, warehouse and fulfillment, customer accounts, CMS pages, shipping zones, app extensions, and webhook-based composable integrations.

## Production Context

- Single Django process (WSGI + ASGI) + Celery worker pool; all in one repository and one deployable artifact
- GraphQL schema is the sole public interface â€” no REST endpoints except plugin webhook receivers and JWKS
- Webhook system is the composability seam: external services (tax engines, ERPs, payment processors, storefronts) integrate exclusively through async/sync webhook subscriptions, not shared database access
- The plugin system is the internal composability seam for Saleor-maintained integrations (Avatax, SendGrid, OIDC) that do not warrant external service deployment
- AWS SQS and GCP Pub/Sub transports allow the async event bus to be cloud-native without requiring a dedicated AMQP cluster
- Designed to be the commerce backend in a composable/headless stack â€” storefront, dashboard, and CMS are separate projects (`saleor-storefront`, `saleor-dashboard`)
