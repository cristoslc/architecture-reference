## Style Rationales

**Pipeline (primary, 0.93):** The pipeline abstraction is the foundational primitive of the entire system. Every unit of work is a `Block` (data loader, transformer, data exporter, sensor, callback, conditional, dbt, SQL, R, or custom). Blocks form a directed acyclic graph via explicit `upstream_blocks`/`downstream_blocks` relationships declared in `metadata.yaml` for each pipeline. The `PipelineType` enum (`python`, `streaming`, `integration`, `databricks`, `pyspark`, `execution_framework`) governs how that pipeline's blocks are executed. The `StreamingPipelineExecutor` enforces a strict linear structure: one source (data_loader) block fans into zero or more transformer chains, each terminating at a sink (data_exporter) block — any violation raises a structural exception at executor initialization. The `BlockExecutor` executes individual blocks with their output passed as inputs to downstream blocks. The `ExecutorFactory` selects among local, K8s, ECS, Azure Container Instance, GCP Cloud Run, and PySpark executors at the pipeline level, not at the infrastructure level — execution mode is a pipeline-level concern. The `run_blocks` function in `mage_ai/data_preparation/models/block/__init__.py` drives asynchronous concurrent execution of the DAG.

**Modular Monolith (primary, 0.87):** The entire system deploys as a single Python process (a Tornado async web server on port 6789) backed by a single SQLAlchemy-managed relational database (SQLite by default, PostgreSQL in production). All subsystems — `api`, `authentication`, `cache`, `cli`, `cluster_manager`, `data_preparation`, `data_integrations`, `extensions`, `frameworks`, `kernels`, `orchestration`, `server`, `services`, `streaming`, `version_control` — are Python packages within a single installable `mage-ai` package. There are no service boundaries or independent deployment units; the `docker-compose.yml` defines a single `server` container plus an optional `app` container for the Next.js frontend dev server. The 93 `api/resources/` classes expose all capabilities through a single REST API layer. The monorepo includes a separate `mage_integrations` package (Singer-protocol sources, destinations, and transformers) that is a standalone installable library, but it is imported directly into `mage_ai/data_integrations` rather than called over the network.

**Event-Driven (secondary, 0.72):** Three distinct event-handling mechanisms operate within the monolith. First, the `EventTrigger` in `mage_ai/orchestration/triggers/event_trigger.py` receives external events and calls `schedule_with_event()` to launch pipeline runs in response to cloud events (AWS EventBridge rules, etc.). Second, the `EventStreamHandler` in `mage_ai/server/events/stream.py` implements Server-Sent Events over HTTP long-polling so that the browser frontend can receive live pipeline execution output without polling. Third, the `WebSocketServer` in `mage_ai/server/websocket_server.py` tunnels Jupyter kernel I/O over WebSockets for the notebook-style interactive UI. The Tornado async I/O loop provides the event loop foundation for all three. Redis pub/sub is optionally used for distributed locking and activity tracking (`ActivityTracker` in `server.py`), and Celery (`mage_ai/orchestration/queue/celery_queue.py`) can replace the default `ProcessQueue` for distributed block execution. These mechanisms are significant but secondary to the pipeline model — the system can operate without an event broker, and the core execution path is synchronous pipeline dispatch.

## Structural Evidence

**Single-process Tornado server:** `mage_ai/server/server.py` imports and wires all subsystems into one `tornado.web.Application`. The `PeriodicCallback` scheduler loop runs inside this same process, polling for pending pipeline runs every `SCHEDULER_TRIGGER_INTERVAL` seconds.

**Pipeline-as-file DSL:** Each pipeline is a folder containing `metadata.yaml` (block graph, pipeline type, executor config) and block Python/SQL/R/YAML source files. The `Pipeline` model in `mage_ai/data_preparation/models/pipeline.py` reads this folder structure, resolves block dependencies, and drives execution.

**Executor abstraction:** `ExecutorFactory.get_pipeline_executor()` maps `PipelineType` and `ExecutorType` to a concrete executor class — `PipelineExecutor` (local), `K8sPipelineExecutor`, `EcsPipelineExecutor`, `GcpCloudRunBlockExecutor`, `PySparkPipelineExecutor`, `StreamingPipelineExecutor`. This lets the pipeline DAG definition remain executor-agnostic.

**Orchestration DB:** SQLAlchemy ORM models in `mage_ai/orchestration/db/models/` — `PipelineRun`, `BlockRun`, `PipelineSchedule`, `Backfill`, `EventMatcher` — track execution state. Alembic migrations in `mage_ai/orchestration/db/migrations/` manage schema evolution.

**Scheduling:** `PipelineScheduler` (in `pipeline_scheduler_original.py`) runs a `schedule_all()` loop, inspecting `PipelineSchedule` rows for cron/interval triggers, SLA checks, and event matchers. `TimeTrigger` invokes `schedule_all()` on a `PeriodicCallback`.

**Streaming connector library:** `mage_ai/streaming/` defines abstract `BaseSource` and `BaseSink` with concrete implementations for Kafka, Kinesis, RabbitMQ, Google Pub/Sub, Amazon SQS, ActiveMQ, Azure Event Hub, and others. These are loaded into the `StreamingPipelineExecutor` at runtime.

**Data integrations (Singer protocol):** `mage_integrations/` is a self-contained Singer-compatible ETL library with sources (databases, SaaS APIs), destinations (data warehouses), and transformers. It is mounted into blocks via `SourceBlock`, `TransformerBlock`, `DestinationBlock` in the integration pipeline type.

**Frontend:** A Next.js 14 / React TypeScript SPA lives in `mage_ai/frontend/`. In production it is compiled to static files served by the Tornado server from `mage_ai/server/frontend_dist/`. The dev compose stack runs it as a separate Node container on port 3000.

**AI layer:** `mage_ai/ai/` contains an `LLMPipelineWizard` with OpenAI and HuggingFace clients for AI-assisted pipeline and block generation — a feature layer layered on top of the pipeline model, not an architectural boundary.

## What Was Rejected and Why

**Not Microservices:** There are no independent services with their own data stores or network contracts. All subsystems communicate in-process. The ECS, K8s, and Cloud Run executors run individual blocks in remote compute, but these are ephemeral compute targets dispatched from the central server, not independent services with their own APIs.

**Not Microkernel:** While `ExecutorFactory` implements a factory pattern for executor selection and `BlockFactory` dispatches to block type classes, there is no formal plugin discovery mechanism (no entry points, no plugin registry). The block type classes and executor classes are statically imported within the package.

**Not pure Event-Driven:** The dominant coordination mechanism is synchronous pipeline-run dispatch through the scheduler loop and the SQLAlchemy ORM, not an event broker. Events (EventTrigger, SSE, WebSocket) are supplementary I/O mechanisms, not the primary execution coordination model.

**Not Serverless:** The system requires a persistent long-running process (the Tornado server) and a persistent relational database. Serverless execution of individual blocks (Cloud Run, Lambda) is an optional deployment target for block executors, not the architectural model.

## Quality Attributes Evidence

**Extensibility:** Pluggable block types (Python, SQL, R, dbt, Spark, custom); pluggable executors (local, K8s, ECS, Cloud Run, PySpark, Celery); pluggable streaming connectors; `mage_integrations` Singer connector library independently extensible with new sources and destinations.

**Deployability:** Single Docker image; Docker Compose for local development; Kubernetes deployment via `kube/` manifests; cloud-managed versions via `cluster_manager/` (AWS ECS, GCP, Kubernetes workspaces); pip/conda install for lightweight use.

**Scalability:** Block-level remote execution via K8s, ECS, or Cloud Run offloads compute from the central server; Celery queue enables distributed block execution across worker processes; streaming pipelines run continuously with configurable retry.

**Observability:** Prometheus metrics endpoint (`PrometheusMetricsHandler`); OpenTelemetry/OTLP export (`OTEL_EXPORTER_OTLP_ENDPOINT`); per-pipeline and per-block structured logging via `LoggerManagerFactory`; usage statistics logging; NewRelic integration via `services/newrelic/`.

**Testability:** `tests/` directory mirrors the main package structure; `pytest` configured in `pyproject.toml`; block-level unit test execution supported via `execute_block_function` in `BlockExecutor`.

**Operability:** Web-based notebook UI for interactive pipeline development and debugging; visual DAG editor; live block output streaming via SSE/WebSocket; version control integration (`version_control/`, `data_preparation/sync/git_sync.py`).
