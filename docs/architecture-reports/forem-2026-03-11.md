# Architecture Report: Forem

**Date:** 2026-03-11
**Repo URL:** https://github.com/forem/forem
**Classification:** Layered, Event-Driven
**Confidence:** 0.90

---

## Summary

Forem is the open-source platform powering DEV.to — a developer content-sharing community. It is a Ruby on Rails monolith deployed as two processes (Puma web server + Sidekiq worker) against a single PostgreSQL database and Redis instance. Its defining architecture is classic **Layered** (Rails MVC extended with service objects, query objects, decorators, and Pundit policies), with a strong **Event-Driven** secondary style expressed through an extensive Sidekiq background job subsystem (122 worker files, 6-tier priority queue). The codebase has no enforced module boundaries, no plugin/extension hooks, and no inter-service network calls — it is a disciplined, high-scale layered monolith with asynchronous event processing as the primary mechanism for domain side-effects (notifications, badge awards, cache invalidation, feed imports, metrics recording).

## Evidence from Code Exploration

### Deployment topology

- `Procfile` defines exactly two processes: `web: bundle exec puma -C config/puma.rb` and `sidekiq_worker: bundle exec sidekiq -t 25`. No service mesh, no inter-service HTTP, no gRPC.
- `docker-compose.yml` / `Containerfile` confirms a single Rails application container, `postgres:13`, and `redis` — three infrastructure nodes, all local to one deployment.
- `config/sidekiq.yml` defines six priority queues: `default (1)`, `low_priority (10)`, `medium_priority (100)`, `high_priority (1000)`, `scheduler (1000)`, `mailers (1000)`.
- `config/schedule.yml` contains 15+ cron-scheduled jobs: feed imports, metrics recording, daily usage stats, badge awards (yearly club, beloved comment, streak badges).

### Layered structure (`app/`)

The application follows strict Rails horizontal layering with additional sub-layers for scale:

| Authorization | `app/policies/` | 37 Pundit policy files |

Key structural evidence:

- `app/controllers/application_controller.rb` inherits from `ActionController::Base` and includes `Pundit::Authorization`, `CachingHeaders`, `SessionCurrentUser`, and `ValidRequest` — the controller boundary is well-defined and clean.
- `app/models/application_record.rb` extends `ActiveRecord::Base` with batch query helpers, decorator inference, and statement timeout controls. Models include Concerns (`Reactable`, `AlgoliaSearchable`, `Purgeable`, `Taggable`) to share cross-cutting behaviors.
- `app/services/application_service.rb` defines a `perform` class method that instantiates and delegates — the base pattern all 265 service objects build on (e.g., `Articles::Creator.call(user, params)`, `Users::Update.call(user, attrs)`, `Badges::Award.call(user_relation, slug, message)`).
- `app/policies/application_policy.rb` documents the Pundit authorization layer explicitly as "the canonical source for authorization" with clear examples of policy usage in controllers and views.

### Service object pattern (265 files)

The `app/services/` directory organizes business logic into domain namespaces without database or process isolation:
- `articles/creator.rb`, `articles/updater.rb` — article lifecycle
- `notifications/new_follower/send.rb`, `notifications/new_comment/send.rb`, `notifications/new_badge_achievement/send.rb` — notification dispatch services
- `users/update.rb`, `users/suspend.rb`, `users/delete.rb` — user management
- `badges/award.rb` — badge assignment
- `search/` — Algolia-backed search services
- `edge_cache/`, `fastly_config/` — CDN cache management services
- `push_notifications/`, `mailchimp/` — third-party integration services

This is Service Object Pattern (a design pattern applied within a Layered style), not Service-Based Architecture (no independent deployment units).

### Query objects (`app/queries/` — 18 files)

Dedicated query classes separate complex SQL/ActiveRecord logic from models:
- `articles/api_search_query.rb`, `articles/feed_query.rb`
- `users/listing_query.rb`, `comments/listing_query.rb`
- `billboards/listing_query.rb`, `audit_log/listing_query.rb`

### API versioning

- `app/controllers/api/v0/` and `app/controllers/api/v1/` — two versioned REST API trees (50 API controller files total)
- Full OpenAPI spec in `swagger/v1/swagger.yaml`
- Admin sub-namespace under `app/controllers/admin/` — comprehensive admin interface

### Routes organization

- `config/routes.rb` (510 lines) draws from three sub-files: `config/routes/admin.rb`, `config/routes/api.rb`, `config/routes/listing.rb`

### Event-Driven secondary style (122 worker files)

Sidekiq workers are organized into domain namespaces and handle all async domain side-effects:
- `notifications/` — `NewFollowerWorker`, `NewBadgeAchievementWorker`, `MilestoneWorker`, `UpdateWorker`, `RemoveAllWorker`, `WelcomeNotificationWorker`
- `articles/` — indexing, scoring, cache invalidation
- `feeds/` — RSS feed import workers
- `users/` — account management background tasks
- `mentions/` — mention notification workers
- `metrics/` — stats recording workers
- `spam/` — content moderation workers
- `push_notifications/`, `emails/`, `broadcasts/` — outbound communication

Workers delegate to corresponding service objects (e.g., `Notifications::NewFollowerWorker#perform` calls `Notifications::NewFollower::Send.call(follow_data, is_read: is_read)`), keeping business logic in the service layer and queue dispatch in workers.

Sidekiq extensions used: `sidekiq-cron` (scheduled jobs), `sidekiq-throttled` (concurrency control), `sidekiq-unique-jobs` (deduplication).

### Liquid Tag system (`app/liquid_tags/` — 64 files)

Forem implements a Liquid templating system with 64 embedded content tags (GitHub, CodePen, Twitter, YouTube, Asciinema, Bandcamp, etc.). All tags inherit from `LiquidTagBase` and are registered at boot time via `Liquid::Template.register_tag()`. This is a **user-facing content embedding feature**, not a plugin architecture for extending the application core. Tags cannot add routes, modify models, or inject middleware — they only render embedded HTML for article bodies.

### Frontend

- `app/javascript/` contains a Preact SPA (not React Router or Ember) with components organized by domain: `article-form/`, `articles/`, `billboard/`, `leftSidebar/`, `actionsPanel/`, etc.
- `package.json` confirms `"preact": "^10.20.2"` as the frontend framework.
- The frontend communicates with the backend exclusively via the versioned REST API.

- `.github/workflows/` contains 10 workflows including `ci.yml`, `cd.yml`, `buildkite.yml`, and `codeql-analysis.yml`.
- Full CI/CD pipeline is present.

## Architecture Styles Identified

### Primary: Layered

**Strength: Strong**

Forem is a textbook horizontal-layered Rails application, extended with additional sub-layers to manage the complexity of a high-scale production platform. The layering is enforced by Rails conventions and deepened by deliberate architectural choices:

1. **Presentation Layer**: `app/controllers/` (229 files) handle HTTP request/response. `app/decorators/` (10 files, Draper) add view-specific formatting without polluting models. `app/views/` provide server-rendered templates. `app/serializers/` format API responses.
2. **Authorization Layer**: `app/policies/` (37 files, Pundit) provide a standalone authorization boundary above the service layer. `ApplicationPolicy` is documented as the canonical authorization source.
3. **Business Logic Layer**: `app/services/` (265 files) contain all domain orchestration, following the `Module::Class.call(...)` pattern. This layer is the thickest and most mature.
4. **Query Layer**: `app/queries/` (18 files) encapsulate complex ActiveRecord queries, keeping models clean.
5. **Data Access Layer**: `app/models/` (147 files, ActiveRecord) manage persistence with concerns for cross-cutting behaviors.

The layers have clear unidirectional dependencies: controllers call services and policies, services call models and other services, models manage persistence. No layer reaches up.

### Secondary: Event-Driven

**Strength: Strong**

The 122 Sidekiq worker files and 6-tier priority queue system constitute a substantial event-driven subsystem for asynchronous domain side-effects. Domain events (user follows, article publications, comment creation, badge awards, page views) are enqueued as typed Sidekiq jobs and processed by worker classes that invoke the corresponding service objects. The `config/schedule.yml` adds time-driven events (cron jobs) for feed imports, metrics, and badge awards. `sidekiq-unique-jobs` ensures deduplication semantics — treating jobs as events, not tasks.

This is not incidental use of background jobs. The worker namespace mirrors the service namespace (`notifications/`, `articles/`, `feeds/`, `users/`, `badges/`, etc.) and the queue priority system reflects deliberate event prioritization decisions. The architecture processes domain events asynchronously to keep the request-response cycle fast.

### Rejected Classifications

- **Microkernel (Plugin)**: The Liquid tag system provides user-content extensibility but is not a plugin architecture for extending application behavior. There are no plugin hooks, no plugin lifecycle, no runtime enable/disable of features. The 64 liquid tags are registered at boot and render HTML — they do not add routes, models, or middleware.
- **Modular Monolith**: No enforced module boundaries. `app/models/`, `app/services/`, and `app/workers/` use namespace organization (Ruby modules) but share the same database, freely cross-reference each other's models, and have no isolation contracts.
- **Microservices / Service-Based**: Single deployable (one Containerfile), single PostgreSQL database, single Redis. No inter-service HTTP, gRPC, or message broker. `Procfile` shows two processes (web + sidekiq), not services.
- **CQRS**: Standard ActiveRecord read/write patterns. No separate read and write models. `app/queries/` objects are query encapsulation (a pattern), not a CQRS read model.
- **Hexagonal (Ports/Adapters)**: No ports/adapters abstraction. Controllers call services directly; services call models directly. No application core isolated by interfaces.
- **Pipeline**: No sequential data transformation stages. The Liquid tag rendering is a one-shot transformation, not a pipeline architecture.

## Quality Attributes

- **Scalability**: Strong horizontal scalability for background processing — Sidekiq worker count is independently configurable, 6-tier queue priority allows load shedding, `sidekiq-throttled` controls concurrency per queue. Puma web server supports multi-process and multi-thread modes (`WEB_CONCURRENCY` env var). PgSearch and Algolia handle full-text search load separately from the primary database. The single-database constraint is the primary horizontal scaling bottleneck.
- **Maintainability**: High. The service object pattern (265 files) keeps business logic in focused, named classes with clear `.call` interfaces. Query objects separate complex SQL from models. Pundit policies provide a single location for authorization logic. Draper decorators keep view-specific formatting out of models. 1,203 spec files provide safety net for changes.
- **Extensibility**: Medium. Adding new domain features requires touching multiple layers (controller, service, model, worker, policy) but the patterns are well-established and the directory structure is consistent. The Liquid tag system provides a clean extension point for content embedding. No plugin API for community-contributed application extensions.
- **Testability**: High. 1,203 RSpec spec files. Service objects are easily unit-testable without HTTP context. Pundit policies have dedicated matchers (`pundit-matchers`). Cypress integration tests cover end-to-end flows.
- **Operability**: High. Sidekiq Web UI for job monitoring. DataDog integration (`datadog/` directory, `ForemStatsClient`). `Metrics::Record*` background workers report queue stats every 10 minutes. PgHero dashboard for query analysis. GitHub Actions CI/CD with 10 workflow files.
- **Security**: Strong authorization model. Pundit policies applied in controllers and documented as canonical authorization source. Devise handles authentication. Rate limiting at the service layer (`RateLimitChecker`). `ApplicationPolicy::UserSuspendedError` propagated through the stack. `protect_from_forgery` enabled in ApplicationController.
- **Performance**: CDN-first caching strategy with Fastly integration (`app/services/edge_cache/`, `app/services/fastly_config/`). Redis-backed session storage and caching. PgSearch + Algolia for search. Background job priority queue prevents slow jobs from blocking fast ones. `statement_timeout` controls on `ApplicationRecord` prevent runaway queries.

## Classification Reasoning

Forem is classified as **Layered (primary) + Event-Driven (secondary)** with confidence 0.90.

The Layered classification is decisive. The codebase organizes every concern into an explicit horizontal layer: controllers at the HTTP boundary, Pundit policies for authorization, service objects for all business logic, query objects for complex queries, ActiveRecord models for data access, and decorators for view formatting. This is not default Rails scaffolding — it is a deliberate, consistent architectural commitment visible in 265 service objects, 18 query objects, 37 policies, and 10 decorators. The unidirectional dependency rule (each layer only calls the layer below) is enforced throughout.

The Event-Driven classification is earned by the scale and deliberateness of the Sidekiq subsystem. 122 worker files organized into domain namespaces, 6 priority queues, 3 Sidekiq extensions (cron, throttle, unique-jobs), and 15+ scheduled background events constitute an architectural commitment to async event processing for domain side-effects. This is not incidental use of ActiveJob — it is a core part of how Forem handles notifications, feed imports, badge awards, metrics, cache invalidation, and more.

The confidence cap at 0.90 reflects two considerations: (1) the Event-Driven pattern, while substantial, operates entirely within Sidekiq's queue model and does not use a message broker or pub/sub bus — some analysts would weight it as a significant implementation pattern rather than a co-primary style; (2) the Liquid tag system introduces a microkernel-adjacent extensibility concern (64 tags, registered at boot, open to community contribution) that is genuinely distinctive but does not reach the threshold of architectural style because it does not extend application behavior.
