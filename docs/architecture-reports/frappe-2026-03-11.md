## Summary

Frappe is an open-source, full-stack, low-code web framework written in Python and JavaScript. It is the kernel on which ERP and business applications (most notably ERPNext) are built as installable "apps." Its dominant architectural style is **Microkernel (Plugin)**: the framework ships a stable core system (ORM, HTTP routing, authentication, background scheduling, multi-tenancy, DocType engine) and exposes a comprehensive, hook-based extension mechanism through which any number of external apps can register behavior, DocTypes, routes, scheduled jobs, and lifecycle callbacks — without modifying the core. A strong **Layered** structure governs the internal organization of the framework itself, with clear horizontal separation between the data-access, model, API, desk, website, and infrastructure layers.

## Architectural Evidence

### 1. Microkernel (Primary)

The defining characteristic of Frappe is its **hooks-based plugin system**. Frappe provides the kernel; external "apps" registered via the Bench environment are the plugins.

**Hook discovery and aggregation** (`frappe/__init__.py:976`):

```python
def get_hooks(
    hook: str | None = None, default=..., app_name: str | None = None
) -> _dict:
    """Get hooks via `app/hooks.py`"""
    # Aggregates hooks from all installed apps' hooks.py files
    hooks = _request_cached_load_app_hooks(app_name) or _site_cached_load_app_hooks()
    ...
```

`_load_app_hooks()` iterates `get_installed_apps()` and merges all `hooks.py` manifests into a single registry. Apps are installed and uninstalled per-site via `installer.py:install_app()`.

**`frappe/hooks.py` — the canonical extension point dictionary.** The framework's own hooks file documents every extension point:

| Hook key | Purpose |
|---|---|
| `doc_events` | Lifecycle callbacks (`on_update`, `on_cancel`, `on_trash`, `after_insert`, etc.) on any DocType |
| `scheduler_events` | Cron/hourly/daily/weekly background job registrations |
| `permission_query_conditions` | Row-level permission SQL injection hooks |
| `has_permission` | Document-level permission overrides |
| `before_request` / `after_request` | HTTP request lifecycle hooks |
| `before_job` / `after_job` | Background job lifecycle hooks |
| `override_whitelisted_methods` | Replace any whitelisted HTTP endpoint |
| `override_doctype_class` | Replace or subclass any DocType controller class |
| `extend_bootinfo` | Inject data into the client boot payload |
| `after_install` / `after_migrate` | Installation and migration hooks |
| `app_include_js` / `app_include_css` | Asset bundle injection |

**DocType controller dispatch** (`frappe/model/base_document.py:116`):

```python
def get_controller(doctype):
    site_controllers = frappe.controllers.setdefault(frappe.local.site, {})
    if doctype not in site_controllers:
        site_controllers[doctype] = import_controller(doctype)
    return site_controllers[doctype]
```

`import_controller` checks `override_doctype_class` hooks first, allowing any installed app to substitute or subclass the controller for any DocType. This is a canonical microkernel plugin dispatch: the kernel resolves which plugin class to activate at runtime.

**Server Scripts — runtime code injection** (`frappe/utils/safe_exec.py`): The `Server Script` DocType allows administrators to inject Python code at runtime for DocType events, API endpoints, and scheduled jobs. The execution uses `RestrictedPython` as a security sandbox, providing a controlled script engine embedded in the kernel.

**App lifecycle** (`frappe/installer.py`):

```python
apps_to_install = ["frappe"] + (frappe.conf.get("install_apps") or []) + (list(install_apps or []))
for app in apps_to_install:
    install_app(app, verbose=verbose, ...)
```

Apps are installed on a per-site basis (`install_app()`), with lifecycle hooks (`after_app_install`, `after_app_uninstall`) notifying the app of its own lifecycle transitions.

**Plugin categories surfaced in `frappe/apps.py`**: `get_apps()` reads the `add_to_apps_screen` hook to build the app switcher menu — plugins register themselves for UI discovery.

### 2. Layered Architecture (Secondary)

Within the kernel itself, Frappe enforces strict horizontal layering with enforced no-upward-dependency rules.

**Layer stack (top to bottom):**

```
┌─────────────────────────────────────────────────────────────┐
│  Website Layer (frappe/website/)                            │
│  — PathResolver, page_renderers, WebsiteGenerator          │
├─────────────────────────────────────────────────────────────┤
│  Desk / UI Layer (frappe/desk/)                             │
│  — form, list, notifications, dashboard, report, kanban    │
├─────────────────────────────────────────────────────────────┤
│  API Layer (frappe/api/, frappe/handler.py)                 │
│  — REST v1/v2, method whitelisting, werkzeug routing       │
├─────────────────────────────────────────────────────────────┤
│  Model / Domain Layer (frappe/model/)                       │
│  — Document, BaseDocument, Meta, Naming, Workflow          │
├─────────────────────────────────────────────────────────────┤
│  Data Access Layer (frappe/database/)                       │
│  — Abstract Database class → MariaDB / Postgres / SQLite   │
├─────────────────────────────────────────────────────────────┤
│  Infrastructure (frappe/utils/background_jobs.py, cache)   │
│  — Redis/RQ queues, session management, rate limiting      │
└─────────────────────────────────────────────────────────────┘
```

**Request path through layers** (`frappe/app.py`):

```python
@after_response_wrapper
@Request.application
def application(request: Request):
    init_request(request)          # infrastructure init
    validate_auth()                # auth layer
    if request.path.startswith("/api/"):
        response = frappe.api.handle(request)   # API layer
    elif ...:
        response = get_response()               # website layer
```

**Data access abstraction** (`frappe/database/database.py:84`): `Database` is an abstract base class with three concrete implementations — `MariaDBDatabase`, `PostgresDatabase`, and `SQLiteDatabase`. The model layer interacts exclusively with the `Database` interface, never with driver-specific APIs.

**Query builder abstraction** (`frappe/query_builder/`): A PyPika-backed type-safe query builder (`frappe.qb`) provides an additional abstraction over raw SQL, with dialect-specific builders per backend (`MariaDB`, `Postgres`, `SQLite`).

**`frappe.local`** (`frappe/__init__.py`): A thread-local context object (`Local`) carries per-request state (`db`, `qb`, `conf`, `form_dict`, `session`, `flags`) that flows through all layers without coupling them to each other directly.

### 3. What This Is Not

- **Not Modular Monolith**: Unlike ERPNext (which Frappe hosts), Frappe itself is not organized around independently deployable or runtime-bounded functional domains. Its modules (`Core`, `Website`, `Workflow`, `Email`, `Automation`, etc.) are infrastructure concerns of the kernel, not business domain modules. The defining characteristic is the **plugin architecture** for hosting external apps, not internal module decomposition.
- **Not Microservices**: Single `pyproject.toml`, single Python package, single database per site, single process. No inter-service communication, no service mesh, no separately deployable service components.
- **Not Event-Driven**: Background jobs use Redis/RQ (`frappe/utils/background_jobs.py`) for operational task queuing. There is no event bus, no publish/subscribe messaging architecture, no event sourcing, and no domain event pattern. The `doc_events` hook system is a synchronous in-process callback registry, not an async event bus.
- **Not DDD**: No bounded contexts, aggregates, value objects, or domain repositories. DocTypes are generic metadata-driven entities. The `frappe.model` layer is a universal ORM abstraction, not a domain model.
- **Not Hexagonal**: Business logic in `Document` controllers directly calls `frappe.db.*` ORM APIs without a ports/adapters abstraction. There is no inversion of the dependency between domain logic and persistence.
- **Not CQRS**: No separate read/write models. The `frappe.db.get_value()` and write operations share the same database tables.
- **Not Service-Based**: No separately hosted service components communicating over a network layer within the framework itself.

## Conclusion

Frappe Framework is a **Microkernel** architecture with a **Layered** kernel. The plugin system — hooks aggregated from all installed apps' `hooks.py`, DocType controller dispatch via `override_doctype_class`, runtime Server Script injection via `RestrictedPython`, and per-site app install/uninstall lifecycle — is the primary and defining architectural characteristic. The kernel provides: a generic DocType ORM (the `Document` class), a multi-backend database abstraction (`Database` → MariaDB/Postgres/SQLite), a layered HTTP application (WSGI via Werkzeug), a background job scheduler (Redis/RQ), multi-tenancy (`frappe.local` per-request context, per-site configuration), and a permissions engine. Apps like ERPNext register into this kernel, extending it without modifying it. The layered internal structure provides predictable change isolation: the `database/` layer is independent of the `model/` layer, which is independent of the `api/` layer, which is independent of the `desk/` and `website/` layers. This is an archetypal microkernel framework: a compact, stable kernel hosting an ecosystem of installable application plugins.
