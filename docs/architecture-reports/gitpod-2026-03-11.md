# gitpod — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/gitpod-io/gitpod
**Spec:** SPEC-031
**Classification:** Microservices + Event-Driven
**Confidence:** 0.93
**Model:** claude-sonnet-4-6
**Method:** deep-analysis

## Summary

Gitpod is a cloud-native developer environment platform (now rebranded as Ona) built as a large-scale Microservices architecture deployed entirely on Kubernetes. The system decomposes into 25+ independently deployable services, each with its own build artifact (Docker image), Go module or Node.js package, and Kubernetes Deployment. Services communicate synchronously via gRPC (Protocol Buffers, 65 proto files spanning 55+ named gRPC services) and asynchronously via Redis Pub/Sub for fan-out event distribution to multi-instance server processes. Every workspace runs with two in-workspace agents (supervisor, workspacekit) and a node-level daemon (ws-daemon), making the workspace execution plane itself a nested microservices topology distinct from the control plane.

### Key Architectural Files

- `components/ws-manager-api/core.proto`: Defines the `WorkspaceManager` gRPC service with 13 RPCs (StartWorkspace, StopWorkspace, Subscribe streaming RPC, ControlPort, TakeSnapshot, etc.). This is the primary synchronous interface between the control plane and workspace execution plane.
- `components/server/src/messaging/redis-subscriber.ts`: `RedisSubscriber` subscribes to `WorkspaceInstanceUpdatesChannel`, `PrebuildUpdatesChannel`, `HeadlessUpdatesChannel` via ioredis. Each `server` replica receives all workspace state transitions — Redis Pub/Sub replaces a shared message queue for horizontal server scaling.
- `components/ws-manager-bridge/src/bridge-controller.ts`: `BridgeController` reconciles live workspace clusters from the database, creates a `WorkspaceManagerBridge` per cluster, each of which opens a long-lived gRPC `Subscribe` stream to the cluster's `ws-manager-mk2` and publishes received events to Redis.
- `components/supervisor/pkg/supervisor/supervisor.go`: In-workspace init daemon. Imports gRPC middleware, grpc-gateway, grpc-web, SSH, and cmux. Serves the `supervisor-api` gRPC interface consumed by IDE extensions, terminal multiplexer, and port forwarder. Uses `content-service` gRPC for workspace content init.
- `components/ws-daemon/pkg/daemon/daemon.go`: Node-level DaemonSet agent using `controller-runtime` (Kubernetes operator pattern). Registers `workspacev1` CRD scheme (ws-manager-mk2 CRD), manages cgroups, content, cpulimit, diskguard, netlimit, iws (in-workspace service for namespace operations), quota.
- `components/workspacekit/pkg/lift/lift.go`: Implements `ServeLift` — a Unix socket server allowing ring0 to execute commands in ring1/ring2 Linux namespaces, a namespace-isolation mechanism unique to Gitpod's rootless container model.
- `components/registry-facade/pkg/registry/registry.go`: Imports containerd content store, Docker distribution, IPFS kubo client, Redis. Implements a composable OCI registry facade that layers workspace-specific content (supervisor, IDE) onto base images at pull time, eliminating the need to pre-build fully assembled images.
- `components/content-service/pkg/storage/`: Four storage backend implementations (gcloud.go, s3.go, minio.go, noop.go) behind a common `DirectAccess` interface, enabling multi-cloud workspace content persistence.
- `components/public-api-server/pkg/server/server.go`: gRPC-Connect API gateway backed by a connection pool to the `server` WebSocket API (`proxy.NewConnectionPool(gitpodAPI, 500)`). Exposes 20+ v1 and experimental gRPC services to external clients.
- `components/image-builder-mk3/`: Builds workspace images on-demand from `.gitpod.yml` or `Dockerfile`, producing OCI artifacts pushed to the workspace registry.
- `install/installer/pkg/components/`: 30+ per-component deployers, each generating Kubernetes Deployment, Service, ConfigMap, and RBAC manifests. Services are independently scaled and configured — not a shared deployment unit.

### Patterns Found

**Microservices (Primary):** 25+ services each have independent build targets (`go.mod`, `package.json`, `leeway.Dockerfile`), independently pushed Docker images (evidenced by `install/installer/` per-component deployers), and separate Kubernetes Deployments. Services are fine-grained by business capability: workspace lifecycle (`ws-manager-mk2`), workspace content (`content-service`), IDE image assembly (`registry-facade`), usage metering (`usage`), authorization (`spicedb`), image building (`image-builder-mk3`). The installer confirms each as a distinct deployment unit with individual scaling knobs.

**Event-Driven (Secondary):** `ws-manager-mk2` emits workspace state transitions as gRPC streaming events. `ws-manager-bridge` converts these into Redis Pub/Sub messages on three channels. All `server` replicas subscribe, fan-out events to WebSocket clients. This event bus pattern is the mechanism enabling stateless horizontal scaling of the `server` layer — without it, each client would be pinned to a single server instance. The event vocabulary (`WorkspaceInstanceUpdatesChannel`, `PrebuildUpdatesChannel`, `HeadlessUpdatesChannel`) is defined in `gitpod-protocol` as a shared schema.

**Kubernetes Operator (Implementation mechanism):** `ws-daemon` and `ws-manager-mk2` use `controller-runtime` and CRDs for workspace lifecycle management. This is the control-plane substrate within which workspace microservices operate, not a separate architectural style.

**Sidecar/Agent (Implementation mechanism):** Each workspace pod runs `supervisor` (in-workspace gRPC daemon) and `workspacekit` (Linux namespace isolation) as mandatory sidecars. `ws-daemon` runs as a DaemonSet on every node. These are deployment-level concerns within the microservices topology.

**Multi-cloud Storage Adapter (Implementation pattern):** `content-service` presents a uniform `DirectAccess` interface over GCS, S3, MinIO, and noop backends. This hexagonal port-and-adapter pattern appears at the storage layer only — it does not generalize to the system architecture.

## Architecture Styles Identified

### Microservices (Primary)

Gitpod's defining architectural principle is decomposition into independently deployable, fine-grained services. Evidence is multi-dimensional:

1. **Independent build artifacts**: Each service in `components/` has its own module descriptor and produces an independent Docker image. The `install/installer/` directory contains a dedicated deployer per service generating separate Kubernetes manifests.

2. **gRPC service contracts**: 55+ named gRPC services across 65 proto files. Inter-service calls are typed contracts, not shared libraries. Key contracts: `WorkspaceManager` (ws-manager-mk2), `ContentService` (content-service), `ImageBuilder` (image-builder-mk3), `WorkspaceDaemon` (ws-daemon), `SupervisorAPI` (supervisor), `PublicAPI v1` (public-api-server).

3. **Polyglot implementation**: Control plane services in TypeScript (server, ws-manager-bridge, dashboard, gitpod-db), execution plane in Go (ws-manager-mk2, ws-daemon, supervisor, registry-facade, content-service, image-builder-mk3, ws-proxy, workspacekit). Language chosen per capability, not per team convention.

4. **Independent scaling**: The `server` service scales horizontally with Redis Pub/Sub enabling state sharing. `ws-manager-mk2` scales per workspace cluster. `registry-facade` is stateless and independently scalable. The installer deploys each with separate replica counts.

5. **Service-level data ownership**: `gitpod-db` is a shared TypeORM layer used by TypeScript services (server, ws-manager-bridge, public-api-server), but Go services have direct database access via GORM. This is a partial shared-database antipattern, mitigated by the fact that Go services access disjoint table sets for their bounded contexts.

### Event-Driven (Secondary)

The workspace state propagation path from execution plane to UI clients is fully event-driven:

- `ws-manager-mk2` (Kubernetes CRD controller) -> gRPC `Subscribe` streaming RPC
- `ws-manager-bridge` (consumes gRPC stream) -> Redis `PUBLISH` on typed channels
- All `server` replicas (subscribe via ioredis) -> WebSocket `sendInstanceUpdate` to browser clients

This is the backbone for real-time workspace status in the UI and for prebuild notifications. The event schema is formalized in `gitpod-protocol` shared types. The pattern enables stateless `server` replicas — each can handle any client connection because all instances receive all events.

### Not Service-Based

Despite the polyglot mix, the 25+ services are fine-grained (workspace lifecycle, content, IDE, proxy, image-building, usage, authorization each separate) rather than coarse-grained domain services typical of Service-Based architecture.

### Not Hexagonal

The storage adapter pattern in `content-service` shows ports-and-adapters locally, but services are not universally structured around an isolated application core with adapter layers. The `server` component uses IoC (InversifyJS) with direct gRPC client injection rather than port interfaces.

## Quality Attributes

- **Scalability**: Horizontal scaling of `server` (Redis fan-out removes sticky session requirement); per-cluster `ws-manager-mk2` instances for multi-region/multi-cloud workspace execution; stateless `registry-facade` and `ws-proxy` scale independently; `blobserve` and `openvsx-proxy` independently cacheable.
- **Security**: `workspacekit` provides Linux namespace isolation (user namespaces, network namespaces, seccomp profiles via `seccomp-profile-installer`); `spicedb` (Zanzibar-style) for fine-grained authorization; `agent-smith` (security enforcement); cryptographic workspace owner tokens for IDE access; rate limiting on all public API methods via `rate-limiter-flexible`.
- **Resilience**: gRPC retry policies with deadline propagation; `ws-manager-bridge` periodic cluster reconciliation (`wsClusterDBReconcileIntervalSeconds`); `BridgeController.reconcileQueue` serializes concurrent reconciliations; content-service multi-backend failover; service-waiter for startup ordering.
- **Observability**: Prometheus metrics instrumented at every service layer (`prometheus-metrics.ts` in server, `common-go/baseserver` metrics in Go services); gRPC interceptor metrics (`reportRedisUpdateReceived`, `reportRedisUpdateCompleted`); distributed tracing (`opentracing` in content-service, `common-go/tracing`); structured logging throughout via `logrus`/`common-go/log`.
- **Extensibility**: Registry-facade's composable image assembly allows workspace environment customization without pre-building full images; IDE service abstraction (`ide-service`) decouples editor selection from delivery; configurable workspace classes (`ListWorkspaceClasses` public API RPC) allow resource tier extension.
- **Portability**: Content-service storage backends for GCP, S3, MinIO allow deployment across cloud providers; installer generates Kubernetes-native manifests compatible with any conformant cluster; preview environment via k3d for local development.
- **Performance**: `registry-facade` uses IPFS for content-addressed layer sharing, reducing workspace startup time by avoiding redundant image transfers; `blobserve` pre-serves IDE assets with aggressive caching; `supervisor` starts IDE process in the workspace before the user connects (prebuild model).
- **Maintainability**: Custom Leeway build system manages cross-component dependencies and Docker image production with explicit dependency graphs; `common-go` shared library enforces consistent logging, tracing, gRPC options; proto-first API design enforces backward compatibility discipline.

## Classification Reasoning

The decisive architectural signal is the 25+ independently deployable services with distinct build artifacts, separate Kubernetes Deployments, and typed gRPC contracts between them. This is textbook Microservices architecture, not a modular monolith (shared-process deployment would be required for that classification), and not Service-Based (services are too fine-grained for that label).

The Event-Driven secondary classification is strongly supported by the Redis Pub/Sub event backbone, which is not merely an implementation optimization but the architectural mechanism enabling stateless horizontal scaling of the API server layer. Without it, the system would require sticky sessions or distributed session state. The event channel schema defined in `gitpod-protocol` and the three-tier event propagation path (ws-manager-mk2 -> ws-manager-bridge -> Redis -> server) is a first-class architectural design decision.

Confidence 0.93 (very high). Primary Microservices classification is multiply confirmed: build system, deployment manifests, gRPC contracts, polyglot implementation, independent scaling. Event-Driven secondary is clearly evidenced by the Redis Pub/Sub fan-out architecture with formalized event schema. The 0.07 gap reflects the shared-database antipattern (`gitpod-db` TypeORM layer shared across TypeScript services), which partially violates strict microservices database-per-service, and the Kubernetes operator pattern in `ws-daemon` and `ws-manager-mk2` which adds a reconciler dimension not captured by either primary style label.
