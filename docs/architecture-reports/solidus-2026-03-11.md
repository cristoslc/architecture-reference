# Architecture Report: Solidus

**Date:** 2026-03-11
**Repository:** https://github.com/solidusio/solidus
**Commit depth:** shallow clone (HEAD of main)
**Classifier:** claude-sonnet-4-6

---

## Summary

Solidus is a full-stack open-source e-commerce platform for Ruby on Rails, a maintained fork of Spree. It is deployed as a single Rails application composed of several cooperating Rails Engines packaged as gems. The dominant architectural style is **Modular Monolith**, with **Microkernel** as a strong secondary pattern (plugin/strategy registry system) and **Event-Driven** as a meaningful tertiary pattern (internal pub/sub bus using the Omnes library).

---

## Repository Structure

The monorepo root contains the following top-level components, each a standalone Ruby gem / Rails Engine:

| Directory | Gem | Role |
|---|---|---|
| `core/` | `solidus_core` | Domain models, state machines, business logic, pub/sub bus |
| `api/` | `solidus_api` | RESTful JSON API (OpenAPI-documented) |
| `admin/` | `solidus_admin` | Next-generation admin UI (ViewComponent + TailwindCSS + Turbo) |
| `backend/` | `solidus_backend` | Legacy admin UI (Erb + Sprockets) |
| `promotions/` | `solidus_promotions` | New promotion engine (optional replacement) |
| `legacy_promotions/` | `solidus_legacy_promotions` | Original promotion system |
| `sample/` | `solidus_sample` | Seed/sample data |

The top-level `solidus.gemspec` bundles `solidus_api`, `solidus_backend`, `solidus_core`, `solidus_legacy_promotions`, and `solidus_sample` as runtime dependencies, making the platform an umbrella gem over those engines. All engines share a single Rails application process and a single relational database.

## Architectural Style Classification

### Primary: Modular Monolith (confidence: 0.88)

Solidus runs as one Rails process against one database. Each major concern (API, admin, promotions, core) is separated into its own Rails Engine gem with its own routes, controllers, models (re-opened from core), migrations, assets, and specs. However, all engines are mounted into the same host application at boot time, share the same ActiveRecord connection and schema, and are loaded together into a single Ruby process. There is no inter-component communication via network or message queues — engines communicate by directly calling `Spree::*` domain classes from `core`.

The modular decomposition is deliberate and enforced: `Spree::Core::Engine` uses `isolate_namespace Spree` to avoid route collisions; admin engines expose their own route namespaces; the promotions module cleanly separates its models and subscribers from legacy promotions. The `solidus_promotions` module demonstrates genuine replaceability — it can substitute `solidus_legacy_promotions` without changing `solidus_core`.

### Secondary: Microkernel (confidence: 0.82)

Solidus implements a pervasive strategy/registry pattern that gives it a Microkernel character. The `Spree::Core::Environment` class maintains `ClassConstantizer::Set` collections for:

- `payment_methods` — swappable payment gateways
- `stock_splitters` — pluggable stock allocation strategies (`Spree::Stock::Splitter::*`)
- `subscribers` — event subscriber classes registered at boot
- `calculators` — pluggable pricing/tax/shipping calculators

`Spree::AppConfiguration` (backed by `Spree::Preferences::Configuration`) exposes dozens of swappable class references: `order_merger_class`, `order_mailer_class`, `carton_shipped_email_class`, `promotions.coupon_code_handler_class`, `stock.availability_validator_class`, and many more. Operators replace framework behavior by pointing these references at custom classes.

The `Spree::Core::EnvironmentExtension` mixin defines `add_class_set` and `add_nested_class_set` DSL methods that create typed, late-bound class collections, formalizing the extension points across the framework. This is a textbook Microkernel plugin contract.

Stock fulfillment further illustrates the pattern: `Spree::Stock::Splitter::Base` → `Backordered`, `ShippingCategory`, `Weight` are concrete splitters registered into the environment and composed at runtime via `Spree::Stock::SplitterChain`.

### Tertiary: Event-Driven (confidence: 0.72)

Solidus uses the [Omnes](https://github.com/nebulab/omnes) library to implement an internal publish/subscribe bus (`Spree::Bus`). Domain events are published synchronously from model methods:

- `order_finalized`, `order_canceled`, `order_emptied`, `order_recalculated`, `order_short_shipped` — from `Spree::Order`
- `carton_shipped` — from `Spree::OrderShipping`
- `reimbursement_reimbursed`, `reimbursement_errored` — from `Spree::Reimbursement`

Subscribers (`Spree::*MailerSubscriber`, `SolidusPromotions::OrderPromotionSubscriber`) use the `Omnes::Subscriber` mixin and declaratively bind handler methods to event names via `handle :event_name, with: :method_name`. Subscriber classes are registered into `Spree::Config.environment.subscribers` and wired to the bus at app boot.

This pattern is used for cross-module integration (e.g., the promotions module subscribes to `order_emptied` without core importing any promotions code) and for sending transactional emails without direct method calls. The events are synchronous in-process — there is no message broker — so this does not reach the level of a fully Event-Driven primary style. Nevertheless, the bus is a first-class architectural mechanism documented and extended by the framework.

## Patterns and Mechanisms Observed

**State Machines:** `Spree::Order`, `Spree::Payment`, `Spree::Shipment`, `Spree::Reimbursement`, `Spree::ReturnAuthorization`, `Spree::ReturnItem`, and `Spree::InventoryUnit` each have formal state machines (`core/lib/spree/core/state_machines/`). The order checkout flow is defined as a DSL (`checkout_flow do ... end`) that generates state machine transitions, and the state machine class itself is swappable via `Spree::Config.state_machines.order`.

**Strategy Pattern:** Stock splitters, tax calculators, availability validators, shipping rate selectors/sorters, reimbursement type engines — each is a pluggable strategy registered at boot and composed at runtime.

**Observer/Subscriber Pattern:** The Omnes bus + subscriber classes implement a formal observer pattern with named, typed events and declarative binding.

**Rails Engine Isolation:** Each gem uses `Rails::Engine` with `isolate_namespace` to prevent route and controller namespace conflicts while still sharing the host application's database.

**Preference System:** `Spree::Preferences::Configuration` provides a DSL for declaring typed preferences (`preference :name, :type, default: ...`) that serve as both configuration and as runtime-swappable class references.

**Permission Sets:** `Spree::PermissionSet::*` classes formalize role-based authorization, independently composable and assigned to user roles.

## Styles Considered and Ruled Out

**Layered:** Rails MVC imposes horizontal layering within each engine (routes → controllers → models → DB). However, the layering is a framework concern, not the dominant organizing principle of Solidus itself; the modular decomposition and plugin system are more architecturally significant.

**Hexagonal Architecture:** Not present. There are no formal ports/adapters. Domain models are ActiveRecord classes that mix persistence and business logic. No inversion of dependency at the infrastructure boundary.

**Domain-Driven Design:** The domain model is rich (~135 classes in `core/app/models/spree/`), but there are no explicit bounded contexts, aggregate roots, value objects, or domain events in the DDD sense. The event bus events are infrastructure/integration events, not DDD domain events.

**Microservices / Service-Based:** Ruled out. All engines deploy as one process. There is no inter-service network communication, no independent deployability of engine components, and no service discovery.

**CQRS:** Ruled out. No read-model/write-model separation. Standard ActiveRecord read and write paths.

**Pipeline:** Ruled out as a primary style. The checkout state machine and stock splitter chain have pipeline-like characteristics locally, but the overall system is not organized as a data processing pipeline.

**Event-Driven (primary):** The pub/sub bus is synchronous and in-process. No message broker or async event store. Insufficient for primary classification.

## Quality Attributes

- **Extensibility:** Pervasive plugin registry (payment methods, splitters, calculators, subscribers, swappable class references); Rails Engine isolation; Omnes bus allows cross-module integration without direct coupling.
- **Replaceability:** Entire subsystems are swappable — promotions engine, admin UI, tax calculator, order merger, state machine class.
- **Maintainability:** Clear module boundaries enforced by gem packaging; separate migration trees per engine; comprehensive RSpec test suite per engine.
- **Evolvability:** Versioned preferences (`Spree::Core::VersionedValue`); explicit deprecation framework (`Spree.deprecator`); `load_defaults(version)` mechanism lets applications pin to an older behavior profile while upgrading incrementally.
- **Testability:** Each engine has an independent spec suite; factory_bot fixtures; dedicated testing support module (`spree/testing_support`).
- **Observability:** State change tracking via `Spree::StateChange` model and `StateChangeTrackingJob`; `LogEntry` model for payment gateway responses; `Spree::Bus` events carry domain context usable for audit/monitoring.
- **Security:** Role-based permission sets; RBAC via `Spree::Ability`; sensitive parameter filtering registered in the engine initializer; credit card number/CVV scrubbed from logs.

---

## Confidence Notes

The Modular Monolith classification is high confidence because the structure is explicit, intentional, and well-documented. The Microkernel classification is high confidence due to the formalized plugin registry, swappable class contracts, and extension point DSL. The Event-Driven classification is moderate confidence — the bus is genuine and architectural but secondary to the module structure. No evidence of CQRS, Hexagonal, DDD, Microservices, or Pipeline as organizing styles.
