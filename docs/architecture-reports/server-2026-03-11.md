**One-line summary:** Bitwarden Server is a Service-Based architecture of independently deployed ASP.NET Core services (Api, Identity, Billing, Notifications, Events, EventsProcessor, Icons, Admin, SSO, SCIM) sharing a single-schema relational database via a Common Core library, augmented by an event-driven integration pipeline over Azure Service Bus and RabbitMQ and structured internally around DDD domains with CQRS commands and queries.

## Evidence Summary

### Deployment Topology — Service-Based

The solution (`bitwarden-server.sln`) contains ten independently deployable service entry points, each with its own `Dockerfile`, `Program.cs`, and `Startup.cs`:

- `src/Api` — main REST API (vault, auth, billing, secrets manager, tools)
- `src/Identity` — OAuth2/OpenID Connect token issuance via Duende IdentityServer
- `src/Billing` — Stripe/Braintree subscription management
- `src/Notifications` — real-time WebSocket push via SignalR hubs
- `src/Events` — audit event ingestion REST endpoint
- `src/EventsProcessor` — background Azure Queue consumer that drains the event queue
- `src/Icons` — favicon proxy service
- `src/Admin` — internal administration portal
- `bitwarden_license/src/Sso` — enterprise SSO (SAML/OIDC)
- `bitwarden_license/src/Scim` — SCIM 2.0 provisioning

Unlike a true Microservices architecture, all services share a single logical data store. The `Core` library exposes repository interfaces (`IUserRepository`, `ICipherRepository`, `IOrganizationRepository`, etc.) implemented by both `Infrastructure.Dapper` and `Infrastructure.EntityFramework`. Every service wires these through `AddDatabaseRepositories(globalSettings)` in `SharedWeb/Utilities/ServiceCollectionExtensions.cs`, connecting to the same SQL Server / PostgreSQL / MySQL / MariaDB schema. This shared-database characteristic is the decisive indicator of Service-Based rather than Microservices.

### Shared Core Library — Service-Based Backbone

`src/Core` is the shared kernel. It contains:

- Entity definitions and repository interfaces for every domain aggregate.
- Domain service interfaces (`IUserService`, `IEventService`, `IApplicationCacheService`, `IPushNotificationService`, etc.).
- Infrastructure-neutral business logic and settings.
- Mail, push, caching, and event write service abstractions.

All deployable services `ProjectReference` into `Core` (and the optional `Commercial.Core`). This hub-and-spoke dependency on a common library is a hallmark of Service-Based architecture.

### Event-Driven Integration Pipeline

The system uses two distinct async messaging layers:

**1. Audit Event Pipeline (Azure Queue)**

`EventService` (in `Core/Dirt/Services/Implementations/`) calls `IEventWriteService`, which in the cloud configuration is `AzureQueueEventWriteService`. This enqueues audit events to Azure Storage Queue. The standalone `EventsProcessor` service polls that queue via `AzureQueueHostedService` and writes events to Azure Table Storage or the database.

**2. Event Integration Pipeline (Azure Service Bus / RabbitMQ)**

When event integrations are enabled, `EventIntegrationEventWriteService` replaces the queue writer and publishes to a two-tier AMQP topology:

- **Tier 1 — Event Exchange/Topic:** Fan-out to `EventRepositoryHandler` (persistence) and `EventIntegrationHandler` (per-integration routing). Backed by Azure Service Bus (cloud) or RabbitMQ (self-hosted).
- **Tier 2 — Integration Exchange/Topic:** Per-integration queues/subscriptions consumed by `SlackIntegrationHandler`, `WebhookIntegrationHandler`, `DatadogIntegrationHandler`, `TeamsIntegrationHandler`, and `HecIntegrationHandler`. Each reports `IntegrationHandlerResult` for retry/DLQ decisions.

RabbitMQ supports either the delay plugin or retry-queue-with-timing-check for exponential backoff. Azure Service Bus uses native scheduled message delivery.

The `Notifications` service uses a separate Azure Storage Queue (`notifications` queue) consumed by its own `AzureQueueHostedService` to relay messages to connected SignalR clients.

Azure Service Bus is also used for cross-service cache invalidation: `InMemoryServiceBusApplicationCacheService` publishes `UpsertOrganizationAbility` and `DeleteOrganizationAbility` messages on a Service Bus topic; every Api instance subscribes and updates its local in-memory `OrganizationAbility` cache.

**Infrastructure in `dev/docker-compose.yml`:**
- `rabbitmq` (RabbitMQ with management UI)
- `service-bus` (Azure Service Bus emulator)
- `storage` (Azurite — Azure Storage emulator for queues and blobs)
- `redis` (distributed cache + FusionCache backplane)
- MSSQL / Postgres / MySQL / MariaDB (shared relational store)

### Domain-Driven Design (Internal)

`src/Core` is organized into explicit bounded contexts, each with its own `Commands`, `Queries`, `Repositories`, `Services`, and `Entities`:

- `Auth` — authentication, SSO, WebAuthn, emergency access, user keys
- `AdminConsole` — organizations, collections, groups, providers, policies
- `Vault` — ciphers, cipher tasks, security task notifications
- `SecretsManager` — machine accounts, projects, secrets
- `KeyManagement` — user key rotation, sends, Kdf management
- `NotificationCenter` — in-app notification entities
- `Billing` — Stripe/Braintree subscriptions, licensing, tax
- `Tools` — password generator, send, import features
- `Dirt` (Data Integrity, Reports & Telemetry) — event logging, integration handlers, audit reports

Each bounded context encapsulates domain entities, repository interfaces (ports), and command/query objects. Infrastructure implementations live in `Infrastructure.Dapper` and `Infrastructure.EntityFramework`, fulfilling the repository contracts.

### CQRS (Internal Pattern)

Multiple bounded contexts use the Command/Query separation pattern explicitly, without MediatR:

- `Core/Vault/Commands/`: `ArchiveCiphersCommand`, `CreateManyTasksCommand`, `MarkTaskAsCompletedCommand`, etc.
- `Core/Vault/Queries/`: `GetCipherPermissionsForUserQuery`, `GetTaskDetailsForUserQuery`, `GetTaskMetricsForOrganizationQuery`, etc.
- `Core/AdminConsole/OrganizationFeatures/Organizations/`: `CloudOrganizationSignUpCommand`, `OrganizationDeleteCommand`, `BulkUpdateOrganizationSubscriptionsCommand`, etc.
- `Core/KeyManagement/Commands/` and `Core/KeyManagement/Queries/`
- `Core/SecretsManager/Commands/` and `Core/SecretsManager/Queries/`
- `Core/NotificationCenter/Commands/` and `Core/NotificationCenter/Queries/`

This is a system-internal structural pattern, not a primary deployment-level style.

## Rejected Styles

- **Microservices:** Services share a single relational database schema via a common repository layer. No database-per-service ownership. The defining Microservices criterion — isolated datastores with inter-service APIs as the only integration point — is not met.
- **Modular Monolith:** Multiple separately deployable processes with distinct Docker images and process boundaries. Not a single deployable unit.
- **Layered:** The dominant organizing principle is bounded-context decomposition and per-service deployment, not a flat horizontal tier stack.
- **Hexagonal Architecture:** Port/adapter patterns are used tactically within bounded contexts (repository interfaces, push service abstractions) but do not define the primary framing of the overall system.
- **Pipeline:** EventsProcessor implements a queue drain loop, but this is a supporting infrastructure component, not the primary application topology.
- **Microkernel:** No plug-in host with a core kernel and extension registry. The license-gated `bitwarden_license` projects are conditional build references, not runtime plugins.
- **Serverless:** Services are long-running ASP.NET Core container processes, not FaaS functions.
- **Space-Based:** No distributed processing grid or tuple-space partitioning.
- **Multi-Agent:** No agent orchestration framework present.

## Quality Attributes

- **Security:** End-to-end encryption, WebAuthn/FIDO2, TOTP/Duo MFA, JWT bearer auth, SAML/OIDC SSO, SCIM provisioning, rate limiting via AspNetCoreRateLimit.
- **Extensibility:** Two-tier AMQP event integration pipeline designed for zero-friction addition of new integrations (Slack, Webhooks, Datadog, Teams, HEC) without altering the event dispatch path.
- **Reliability:** Exponential backoff retries with jitter, dead-letter queues for failed integrations; transactional event enqueue.
- **Scalability:** Stateless services; Redis-backed distributed cache with FusionCache multi-tier (L1 memory + L2 Redis) and pub/sub backplane for cache invalidation; Azure Service Bus for cross-instance cache coherence.
- **Maintainability:** Strongly bounded domains in Core; explicit command/query separation within each domain; infrastructure implementations isolated in separate projects.
- **Observability:** Serilog structured logging; health check endpoints (`/healthz`, `/healthz/extended`); LaunchDarkly feature flags.
- **Portability (Self-Host):** Supports SQL Server, PostgreSQL, MySQL, MariaDB; Azure Storage / Queue replaceable with local Azurite; Azure Service Bus replaceable with RabbitMQ; self-hosted Docker Compose configuration included.
