## Project Overview

Decidim is an open-source participatory democracy platform built in Ruby on Rails. It enables cities, organisations, and communities to run participatory processes, citizens' assemblies, citizen initiatives, and other democratic participation mechanisms. The project is structured as a mono-repository containing 27+ distinct Ruby gems, all deployed as a single cohesive Rails application.

---

## Codebase Structure

### Module Layout

The repository root contains one directory per gem:

```
decidim-core/          # Kernel — engine, manifest registry, base classes
decidim-proposals/     # Plugin — citizen proposals component
decidim-meetings/      # Plugin — meetings/assemblies component
decidim-budgets/       # Plugin — participatory budgeting component
decidim-assemblies/    # Participatory space — assemblies
decidim-participatory_processes/  # Participatory space — processes
decidim-verifications/ # Authorization workflow registry and built-in handlers
decidim-admin/         # Admin panel engine
decidim-api/           # GraphQL API engine
decidim-ai/            # AI/spam-detection strategy module
decidim-generators/    # Rails app and component scaffolding generators
... (17+ additional feature modules)
```

Each module follows the same internal Rails Engine convention:
- `lib/decidim/<module>/engine.rb` — Rails Engine definition
- `lib/decidim/<module>/component.rb` or `lib/decidim/<module>/<space>.rb` — self-registration DSL call
- `app/controllers/`, `app/models/`, `app/commands/`, `app/queries/`, `app/events/`, `app/forms/`, `app/cells/` — internal layered structure

### Deployment Model

`decidim.gemspec` aggregates all sibling gems as `add_dependency` entries. `decidim-generators` produces a Rails application that requires the top-level `decidim` gem, which in turn pulls in every module into a **single Ruby process** backed by a **single shared PostgreSQL database** with namespaced tables (via Rails Engine isolation).

## Key Architectural Patterns

### 1. Microkernel (Primary)

`decidim-core` acts as the kernel. It defines:

- **ComponentManifest** (`decidim-core/lib/decidim/component_manifest.rb`) — the complete plugin contract: engine reference, admin engine, settings DSL, lifecycle hooks (`:create`, `:publish`, `:unpublish`), actions, permissions class, GraphQL type, resource registration, export/import manifests.
- **ParticipatorySpaceManifest** — equivalent contract for top-level participation spaces.
- **ManifestRegistry** (`lib/decidim/manifest_registry.rb`) — global registry holding all registered manifests at runtime.
- **ViewHooks** (`lib/decidim/view_hooks.rb`) — named, priority-ordered view extension points so plugins can inject partials into core templates without modifying them.
- Multiple additional registries: `WorkflowRegistry` (verification workflows), `ContentBlockRegistry`, `PermissionsRegistry`, `MutationRegistry`, `StatsRegistry`, `MenuRegistry`.

Plugins self-register at boot via a DSL exposed by the kernel:

```ruby
# decidim-proposals/lib/decidim/proposals/component.rb
Decidim.register_component(:proposals) do |component|
  component.engine = Decidim::Proposals::Engine
  component.admin_engine = Decidim::Proposals::AdminEngine
  component.permissions_class_name = "Decidim::Proposals::Permissions"
  component.actions = %w(like vote create withdraw amend comment vote_comment)
  component.on(:create) { |instance| Decidim::Proposals.create_default_states!(instance, ...) }
  component.settings(:global) { |s| s.attribute :vote_limit, type: :integer, ... }
  component.register_resource(:proposal) { |r| r.model_class_name = "Decidim::Proposals::Proposal" }
end
```

The `Component` ActiveRecord model (`decidim-core/app/models/decidim/component.rb`) is polymorphic with respect to participatory spaces and looks up its manifest at runtime via `Decidim.find_component_manifest(manifest_name)` — the kernel never hard-codes knowledge of any specific component. Components (plugins) can be enabled, disabled, and configured by administrators at runtime within any participatory space.

### 2. Modular Monolith (Co-primary)

Despite the plugin facade, all 27 gems:
- Run in the same Ruby process.
- Share a single PostgreSQL schema (254 migrations in core alone; proposals adds 71 more, all landing in the same DB).
- Communicate via direct Ruby method calls and shared ActiveRecord models, not over the network.
- Are packaged together by `decidim.gemspec` and deployed as one unit.

The module boundaries are well-defined (each gem has its own namespace, engine, and gemspec) but are organisational rather than process/deployment boundaries. This is the defining trait of the Modular Monolith style.

### 3. Internal Layered Structure (Within Each Module)

Every module applies a consistent horizontal layering pattern inside its `app/` directory:

| Layer | Directory | Role |
|---|---|---|
| Presentation | `controllers/`, `views/`, `cells/` | HTTP request handling, template rendering |
| Application Services | `commands/`, `forms/` | Business operation orchestration, input validation |
| Query Layer | `queries/` | Read-side data retrieval objects |
| Domain | `models/` | ActiveRecord entities with domain rules |
| Events | `events/` | Event payload objects for notification dispatch |
| Infrastructure | `jobs/`, `mailers/`, `services/` | Background work, email, external integrations |

The `Decidim::Command` base class (from `lib/decidim/command.rb`) integrates `Wisper::Publisher` to provide intra-command pub/sub via `broadcast(:ok, resource)` / `broadcast(:invalid)`. The `Decidim::Query` base class provides composable query object primitives. This Command/Query separation is a design convention within the monolith; it does not constitute a standalone CQRS style because reads and writes share the same data model and database.

### 4. Event-Driven Notification Subsystem (Supplementary)

An asynchronous event bus is used specifically for user notifications and emails:

- **`Decidim::EventsManager`** wraps `ActiveSupport::Notifications` and registers a wildcard subscriber for `decidim.events.*` patterns at boot.
- Commands publish domain events via `Decidim::EventsManager.publish(event: "decidim.events.proposals.proposal_published", ...)` after completing write operations.
- **`EventPublisherJob`** (background job) routes incoming events to `EmailNotificationGeneratorJob` and `NotificationGeneratorJob` based on the event class's declared types.

This is a supplementary pattern serving the notification domain; the primary control flow remains synchronous Rails MVC + Commands.

## What the Architecture Is Not

- **Not Microservices:** All modules share a single process and database. No network-based inter-service communication exists.
- **Not Service-Based:** Modules are not deployed independently; they do not own separate data stores.
- **Not Hexagonal/Ports-and-Adapters:** There is no ports/adapters abstraction. Modules depend directly on Rails Engine conventions and shared ActiveRecord models.
- **Not CQRS:** Command and Query objects are separated by convention within each module, but they share the same write model. There is no separate read model or read store.
- **Not Event-Driven (primary):** Events drive only the notification subsystem. Core business flows are synchronous request-response cycles.
- **Not Domain-Driven Design (style):** While bounded-context thinking is visible in module naming, modules are not autonomous bounded contexts — they share the kernel's database schema and common base classes.

## Classification Summary

| Style | Confidence | Role |
|---|---|---|
| Microkernel | High | Primary — kernel + plugin registry + manifest DSL |
| Modular Monolith | High | Co-primary — single-process, shared DB, gem-based modules |
| Layered | Medium | Internal — uniform controller/command/query/model layering per module |

**Primary classification: Microkernel + Modular Monolith**

Decidim is an exemplary Microkernel architecture: `decidim-core` is the stable kernel providing extension points (manifests, registries, hooks), while all feature modules are plugins that self-register via the kernel's DSL and can be added, removed, or configured at runtime. The entire system is simultaneously a Modular Monolith because all plugins run in the same process and share a common database, making deployment a single unit. The Layered pattern is a consistent internal structuring discipline rather than a top-level architectural style.
