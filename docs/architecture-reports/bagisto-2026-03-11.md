## Summary

Bagisto is a Laravel-based open-source e-commerce platform organized as a **Modular Monolith** with strong **Microkernel (plugin-system)** and **Event-Driven** characteristics. Thirty-one discrete functional modules live under `packages/Webkul/`, each self-contained with its own models, migrations, routes, views, and service providers, and all are registered through the Concord module orchestration framework. The application deploys as a single Laravel process against a shared database. Module integration is achieved through two complementary mechanisms: a cross-module event bus (string-keyed Laravel events such as `checkout.order.save.after`, `sales.shipment.save.after`) and config-driven plugin registries for product types, payment methods, and shipping carriers.

## Styles Identified

### Modular Monolith (primary, 0.93)

The organizing principle of the codebase is vertical decomposition by business domain rather than horizontal technical layer. Thirty-one modules under `packages/Webkul/` — Admin, Attribute, BookingProduct, CartRule, CatalogRule, Category, Checkout, CMS, Core, Customer, DataGrid, DataTransfer, DebugBar, FPC, GDPR, Installer, Inventory, MagicAI, Marketing, Notification, Payment, Paypal, Product, Rule, Sales, Shipping, Shop, Sitemap, SocialLogin, SocialShare, Tax, Theme, User — are each independently structured and declared in `composer.json` as distinct PSR-4 autoload roots pointing to `packages/Webkul/<Module>/src`.

Central registration happens in `config/concord.php`, which lists each module's `ModuleServiceProvider` class. Every `ModuleServiceProvider` extends `CoreModuleServiceProvider` (itself extending `konekt/concord`'s `BaseModuleServiceProvider`), which enforces a consistent convention: migrations registered from `Database/Migrations`, views from `Resources/views`, routes from `Routes/`, and models from module-scoped definitions. There is no independent deployment, no inter-process communication, and no separate database per module — all modules share a single MySQL database accessed via Eloquent ORM. This is the canonical definition of a Modular Monolith.

**Key evidence:**
- `composer.json` — 31 distinct PSR-4 mappings to `packages/Webkul/*`
- `config/concord.php` — explicit central module registry
- `packages/Webkul/Core/src/Providers/CoreModuleServiceProvider.php` — enforced module lifecycle convention
- Each module contains `Contracts/`, `Models/`, `Repositories/`, `Providers/`, `Database/`, `Resources/` subdirectories

### Microkernel (secondary, 0.85)

Bagisto implements three distinct config-driven plugin registries that match the Microkernel pattern: the core application defines an abstract contract, and concrete implementations are registered via configuration keys and class references resolved at runtime through the Laravel service container.

**Product type plugin system:** `packages/Webkul/Product/src/Config/product_types.php` maps keys such as `simple`, `configurable`, `bundle`, `downloadable`, `grouped`, `virtual`, `booking` to concrete class names. Each concrete type extends `AbstractType` from `packages/Webkul/Product/src/Type/AbstractType.php`, which defines the extension contract. The active type is loaded dynamically: `app($config['product_types'][$type]['class'])`.

**Payment method plugin system:** `packages/Webkul/Payment/src/Payment.php` iterates `Config::get('payment_methods')` and instantiates each registered class via `app($paymentMethodConfig['class'])`. New payment gateways (e.g., Paypal in its own `Webkul/Paypal` module) register themselves into this config.

**Shipping carrier plugin system:** `packages/Webkul/Shipping/src/Shipping.php` iterates `Config::get('carriers')`, instantiating each carrier class. The abstract contract is `Carriers/AbstractShipping.php`.

**View hook plugin system:** `packages/Webkul/Theme/src/ViewRenderEventManager.php` provides named view-injection hooks (e.g., `bagisto.shop.layout.body.after`, `bagisto.admin.layout.head`) that any module can push templates into. This is an event-driven UI plugin registry extending the core rendering pipeline without modifying it.

**Key evidence:**
- `packages/Webkul/Product/src/Config/product_types.php` — product type registry
- `packages/Webkul/Product/src/Type/AbstractType.php` — plugin contract
- `packages/Webkul/Payment/src/Payment.php` — payment method plugin dispatcher
- `packages/Webkul/Shipping/src/Shipping.php` — shipping carrier plugin dispatcher
- `packages/Webkul/Theme/src/ViewRenderEventManager.php` — view-hook event system

### Event-Driven (secondary, 0.80)

Cross-module coordination is accomplished almost entirely through a synchronous string-keyed event bus built on Laravel's `Event::dispatch()` / `Event::listen()` infrastructure. Repositories in one module fire named domain events; listeners in other modules react independently. This decouples the Sales, Product, Marketing, CartRule, CatalogRule, FPC, Admin, Shop, and Notification modules without introducing direct dependencies between them.

The event surface spans the full order lifecycle: `checkout.order.save.before`, `checkout.order.save.after`, `checkout.order.orderitem.save.before`, `checkout.order.orderitem.save.after`, `sales.order.cancel.before`, `sales.order.cancel.after`, `sales.order.update-status.before`, `sales.order.update-status.after`, `sales.invoice.save.before`, `sales.invoice.save.after`, `sales.shipment.save.before`, `sales.shipment.save.after`, `sales.refund.save.before`, `sales.refund.save.after`, plus catalog events (`catalog.product.create.after`, `catalog.product.update.after`, `catalog.product.delete.before`, `catalog.category.*`), customer events (`customer.registration.after`, `customer.password.update.after`), and CMS/config events.

157 source files fire events; 14 packages declare `EventServiceProvider` classes registering listeners. The FPC (Full Page Cache) module registers 16 listeners reacting to domain events from Product, Category, CMS, Sales, and Core — a clean reactive cache invalidation topology.

Asynchronous processing exists for heavy-lifting tasks: `UpdateCreatePriceIndex`, `UpdateCreateInventoryIndex`, and ElasticSearch index jobs implement `ShouldQueue` and are dispatched to the Laravel queue (defaulting to database-backed queue).

**Key evidence:**
- `packages/Webkul/Sales/src/Repositories/OrderRepository.php` — fires 8 order lifecycle events
- `packages/Webkul/Admin/src/Providers/EventServiceProvider.php` — Admin module subscribes to checkout.order.save.after, sales.invoice.save.after, etc.
- `packages/Webkul/FPC/src/Providers/EventServiceProvider.php` — 16 cross-module cache invalidation listeners
- `packages/Webkul/Product/src/Jobs/UpdateCreatePriceIndex.php` — async ShouldQueue job
- 14 EventServiceProvider files across the package tree

## Classification Reasoning

Bagisto's primary architecture is **Modular Monolith**: the entire application ships and deploys as one process, shares a single database, and is organized into 31 well-bounded domain modules registered through the Concord framework. This is not a Layered architecture — the primary organizational axis is vertical (domain) rather than horizontal (technical layer), though Laravel's MVC stack provides a conventional request pipeline within each module.

The **Microkernel** secondary classification is justified by three fully realized plugin registries (product types, payment methods, shipping carriers) where the core defines an abstract contract and the framework dynamically loads registered implementations at runtime via config keys. This is distinct from ordinary strategy pattern use — the plugin points are configuration-driven, externally extensible, and match the Microkernel definition of a stable core with extensible plug-in components.

The **Event-Driven** secondary classification is supported by the pervasive use of string-keyed domain events as the sole cross-module coordination mechanism. Modules do not call each other directly; they fire named events from repositories and subscribe via `EventServiceProvider` declarations. With 157 event-firing files, 14 event subscriber providers, and a reactive cache invalidation module (FPC) built entirely on domain events, event-driven integration is an architectural-level concern rather than incidental usage.

The system is NOT Microservices (single deployment, shared database), NOT pure Layered (domain-vertical primary organization), NOT Hexagonal (Eloquent active-record models are tightly coupled to persistence with no port/adapter isolation), and NOT CQRS (no separate read/write models; Repositories serve both read and write paths, and the ElasticSearch flat index is an optimization layer rather than a CQRS read model with a separate command bus).
