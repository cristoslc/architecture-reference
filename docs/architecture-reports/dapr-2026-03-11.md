## Summary

Dapr (Distributed Application Runtime) is a CNCF-graduated platform runtime that provides distributed systems building blocks to application code via a language-agnostic sidecar process. Its internal architecture is a textbook **Microkernel** system — a stable runtime core that orchestrates a large, swappable plugin ecosystem — deployed as a set of independently running **Microservices** control-plane components.

**Primary style:** Microkernel
**Secondary style:** Microservices

## Primary Style: Microkernel

### Evidence

**1. Registry-per-capability plugin system**

Every building block is managed through a typed registry in `pkg/components/`:

```
pkg/components/
├── bindings/registry.go       — InputBinding / OutputBinding plugins
├── state/registry.go          — State store plugins
├── pubsub/registry.go         — Pub/Sub plugins
├── secretstores/registry.go   — Secret store plugins
├── lock/registry.go           — Distributed lock plugins
├── crypto/registry.go         — Cryptography provider plugins
├── nameresolution/registry.go — Service discovery plugins
├── configuration/registry.go  — Configuration store plugins
├── conversation/registry.go   — Conversation / AI plugins
└── middleware/http/registry.go — HTTP middleware plugins
```

The composite `pkg/runtime/registry/registry.go` aggregates all ten typed registries into a single `Registry` struct that the sidecar runtime holds at startup.

**2. Build-time plugin self-registration via Go `init()`**

`cmd/daprd/components/` contains 137 Go source files. Each file registers one concrete implementation into the appropriate typed registry using Go's `init()` function:

```go
// cmd/daprd/components/bindings_redis.go
func init() {
    bindingsLoader.DefaultRegistry.RegisterOutputBinding(redis.NewRedis, "redis")
}
```

Build tags (`//go:build allcomponents || stablecomponents`) allow selective inclusion. The daprd entrypoint triggers all registrations with a single anonymous import:

```go
// cmd/daprd/app/app.go
import (
    _ "github.com/dapr/dapr/cmd/daprd/components"
    ...
)
```

**3. Runtime-pluggable components via gRPC sockets**

Beyond compile-time registration, `pkg/components/pluggable/discovery.go` implements socket-based discovery: at startup, daprd scans a Unix socket directory, uses gRPC reflection to identify services, and dynamically wires them in as first-class components. `pkg/components/pluggable/grpc.go` provides the `GRPCConnector[TClient]` generic that wraps any gRPC client as a Dapr component. This extends the Microkernel pattern to support out-of-process plugins without recompilation.

**4. Stable kernel API surface**

`dapr/proto/runtime/v1/dapr.proto` defines 50+ RPC methods (state CRUD, pub/sub, secret retrieval, actor invocation, workflow, distributed lock, jobs, conversation) that application code calls. This surface is immutable regardless of which plugins are loaded — the canonical Microkernel contract.

**5. Stable plugin contracts**

`dapr/proto/components/v1/` defines protobuf interfaces for `state.proto`, `pubsub.proto`, `bindings.proto`, `secretstore.proto` — the "plugin API" that all runtime and socket-based implementations must satisfy.

## Secondary Style: Microservices

Dapr's control plane is not a single process. Six independently deployable services form the platform:

| Service | Binary | Role |
|---------|--------|------|
| daprd | cmd/daprd | Sidecar runtime (runs per application instance) |
| operator | cmd/operator | Manages Dapr CRDs and distributes component/resiliency configs |
| placement | cmd/placement | Raft-based actor placement and partition table distribution |
| sentry | cmd/sentry | SPIFFE/X.509 certificate authority for mTLS identity |
| injector | cmd/injector | Kubernetes mutating webhook for automatic sidecar injection |
| scheduler | cmd/scheduler | Job/cron scheduling service |

Each service has its own `cmd/` entrypoint, its own gRPC API defined in `dapr/proto/`, and is deployed as a separate Kubernetes Deployment in `charts/dapr/Chart.yaml`. The services communicate exclusively via protobuf-defined gRPC protocols — services are not in-process.

The placement service uses a Raft consensus implementation (`pkg/placement/internal/`) for distributed consistency, and the scheduler service has dedicated client/server packages (`pkg/scheduler/client/`, `pkg/scheduler/server/`).

## Supporting Observations

**Hexagonal influence (ports and adapters).** The universal API layer (`pkg/api/universal/`) implements the core domain logic once, shared between the HTTP adapter (`pkg/api/http/`) and the gRPC adapter (`pkg/api/grpc/`). Component registries define ports; concrete implementations are adapters. This is a subordinate pattern within the Microkernel's component management, not the primary organizing principle.

**Actors subsystem.** `pkg/actors/` implements a virtual actor model (Orleans-style), backed by distributed state and placement. This is an internal subsystem of the kernel, not a separate architectural style at the repository level.

**Workflow engine.** `pkg/runtime/wfengine/` implements durable workflows using the durabletask-go backend. The workflow engine is implemented as actors internally, another kernel subsystem.

**Resiliency.** `pkg/resiliency/` provides circuit breakers, retries, timeouts, and bulkheads declaratively via CRD — a cross-cutting concern integrated into the runtime layer.

## Rejected Styles

- **Layered:** Layers exist (API / runtime / components) but layering is not the primary organizing principle — the plugin registry mechanism is.
- **Modular Monolith:** Dapr services are separate deployment units that communicate over the network, not modules in one binary.
- **Hexagonal Architecture:** Ports/adapters exist, but they serve the Microkernel's component extension model; hexagonal isolation is not the dominant structural intent.
- **Event-Driven:** Dapr enables event-driven applications but its own internal component communication is synchronous gRPC, not event-driven messaging.
- **Domain-Driven Design:** No bounded contexts, aggregates, or domain-model-first organization. Organization is by infrastructure capability (state, pubsub, secrets) not business domain.
- **Pipeline:** No pipeline/filter topology within the runtime's own service interactions.
- **Service-Based:** The control-plane services do not share a monolithic core codebase behind a service layer; each is independently structured.

## Quality Attributes

- **Extensibility:** The plugin registry and gRPC socket discovery enable adding new infrastructure providers without modifying runtime core.
- **Portability:** Language-agnostic sidecar via standard HTTP/gRPC; runs on Kubernetes, self-hosted, IoT edge.
- **Resiliency:** Built-in circuit breaker, retry, timeout, and bulkhead policies via the resiliency subsystem.
- **Security:** mTLS with SPIFFE/X.509 certificates managed by the sentry CA; ACL policies per component.
- **Observability:** OpenTelemetry tracing, Prometheus metrics, and structured logging integrated throughout the runtime.
- **Scalability:** Stateless daprd sidecars; Raft-backed placement for actor distribution at scale.
