# grafana — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/grafana/grafana
**Classification:** Microkernel + Modular Monolith
**Confidence:** 0.91

## Summary

Grafana is an open-source observability and visualization platform whose dominant architectural characteristic is a formal **Microkernel** design: the core server is a minimal host runtime that extends its capabilities — datasources, panels, apps, and backend processes — entirely through a typed plugin registry. The plugin system is not a convenience feature; it is the primary integration seam for all data connectivity and visualization. Every datasource (Prometheus, Loki, Elasticsearch, CloudWatch, and 150+ community sources) is a plugin with a standardized `plugin.json` contract and, for backend plugins, a gRPC channel to the core via `hashicorp/go-plugin`. Alongside the plugin architecture, the backend codebase exhibits clear **Modular Monolith** characteristics: 74 domain-scoped Go service packages in `pkg/services/`, assembled by Google Wire dependency injection into a single deployable binary, sharing a unified SQL database (SQLite/MySQL/PostgreSQL) and Redis cache. A nascent module system (`pkg/modules/`) using `grafana/dskit` allows specific subsystems (StorageServer, SearchServer, ZanzanaServer, FrontendServer) to be split out and deployed as standalone gRPC services, but the primary mode and the defining identity of the system is the monolithic single-binary host with Microkernel extensibility.

### Key Architectural Files

- `pkg/plugins/manager/pipeline/doc.go`: Authoritative declaration of the plugin load pipeline — "A pipeline is a sequence of stages that are executed in order. Each stage is made up of a series of steps." Five stages: Discovery, Bootstrap, Validation, Initialization, Termination. This is the plugin lifecycle contract.
- `pkg/plugins/manager/pipeline/initialization/steps.go` (`BackendClientInit.Initialize`): For backend plugins, retrieves a `BackendFactoryFunc`, computes environment variables, and calls the factory to produce a `backendplugin.Plugin`. The factory ultimately uses `hashicorp/go-plugin` to launch the external binary and establish a gRPC channel. This is the core plugin activation mechanism.
- `pkg/plugins/manager/registry/in_memory.go` (`InMemory`): Thread-safe plugin registry keyed by plugin ID. Central component of the Microkernel pattern — the host maintains a runtime registry of all activated plugins.
- `pkg/modules/dependencies.go`: Nine named deployable modules with explicit dependency graph — `StorageServer` depends on `{UnifiedBackend, InstrumentationServer, GRPCServer, SearchServerRing}`, etc. The `All` module collapses the graph into the default monolithic deployment target.
- `pkg/server/module_server.go` (lines 231–287): All modules are registered via `m.RegisterModule(...)` callbacks. The `modules.All` catchall maps to `Core` which starts the full HTTP server. Enterprise builds inject additional modules via the `ModuleRegisterer` interface (`s.moduleRegisterer.RegisterModules(m)`).
- `pkg/server/wire.go` / `wire_gen.go`: Google Wire compile-time DI graph. A single `Initialize` function wires all 74 service providers into the `Server`. This is the monolith assembly point.
- `pkg/bus/bus.go` (`InProcBus`): Reflect-based in-process pub-sub. Listeners registered via `AddEventListener(handler HandlerFunc)` receive typed messages published via `Publish(ctx, msg)`. Used by alerting (`ngalert` imports `pkg/bus`) and storage events.
- `pkg/services/live/live.go`: Real-time streaming service using Centrifuge WebSocket broker. Enables live panel updates from streaming datasource plugins and backend streams.
- `pkg/storage/unified/resource/server.go` (`ResourceServer`): Kubernetes-style resource API implemented as a gRPC service. Runs in-process by default; can be split to `storage-server` standalone target (documented in `storage/unified/README.md`).
- `pkg/services/ngalert/ngalert.go`: Unified alerting — imports `pkg/bus`, `pkg/events`, dskit, gRPC, Centrifuge, external alerting library (`github.com/grafana/alerting`). Demonstrates the depth of in-process service integration; alerting is a large subsystem, not an independent service.
- `public/app/extensions/`: Frontend plugin extension point registry — panels and apps contribute UI extension points to each other through a structured extension registry pattern.
- `pkg/server/ring.go` + `memberlist.go`: Optional memberlist-based ring (dskit) for `search-server` sharding mode. Only active when `EnableSharding=true` and multiple `search-server` or `storage-server` instances are deployed.

### Patterns Found

**Microkernel (Primary):** The plugin system is the defining structural pattern. The core host provides: HTTP routing, authentication, RBAC, dashboard rendering, query fan-out, and alerting infrastructure. All data connectivity and visualization extensibility flows through the plugin registry. Plugin types: `datasource` (backend gRPC process or frontend-only), `panel` (React component), `app` (full-page extension), `renderer` (headless Chrome bridge). The five-stage load pipeline (Discovery → Bootstrap → Validation → Initialization → Termination) is a formal plugin lifecycle managed by the host, not by plugins themselves. Twenty built-in datasources in `pkg/tsdb/` are implemented as first-class plugins registered at startup — there is no privileged "core datasource" code path distinct from the plugin interface. The `@grafana/runtime` package exposes a formal SDK surface to frontend plugins. Backend plugins run as separate OS processes communicating via gRPC (hashicorp/go-plugin), making the plugin boundary a physical process isolation point.

**Modular Monolith (Secondary):** The backend is a single binary (`pkg/cmd/grafana/main.go`) assembled by Google Wire from 74 domain-scoped service packages. All services share a common SQL database (one schema, managed by xorm migrations), a Redis/database cache, and a configuration system (`pkg/setting/`). Services are loosely coupled via interfaces: each service declares its dependencies as constructor parameters (Wire provider functions), and cross-service calls are synchronous Go interface calls within the same process. The `pkg/modules/` system (dskit) allows named subsystems to be split to standalone processes, but the default `target = all` mode runs everything in one process. The `apps/` directory introduces a Kubernetes operator pattern for composable app-platform extensions, signaling a migration path toward more loosely coupled app delivery, but this is nascent.

**Supporting — Event-Driven (bus):** `InProcBus` provides lightweight in-process pub-sub used by a small number of subsystems (alerting, storage delete events). This is not a dominant integration mechanism — most cross-service calls are direct Go interface invocations via Wire injection. The bus supplements the synchronous call graph for decoupled side-effect notifications.

**Supporting — Pipeline (plugin load):** The plugin load pipeline (Discovery → Bootstrap → Validation → Initialization → Termination) is a formal pipe-and-filter structure, but it is confined to plugin lifecycle management, not to the primary data processing flow. Query execution is not pipelined — it is a fan-out to datasource plugins, aggregated by the expression engine.

## Architecture Styles Identified

### Microkernel (Primary)

The decisive evidence for Microkernel as the primary classification is the plugin registry pattern operating at the system's integration seam. Grafana's core value proposition — connecting to any data source and visualizing it any way — is delivered entirely through plugins, not through built-in code. The 20 built-in datasources in `pkg/tsdb/` are registered through the same `PluginSource` → `InMemory` registry as external community plugins. The `backendplugin.PluginFactoryFunc` is a universal adapter: whether a datasource is Prometheus (built-in Go), a community gRPC plugin, or a frontend-only REST plugin, it enters the system through the same pipeline and registration interface. The plugin load pipeline (`doc.go`) explicitly defines this as a multi-stage lifecycle managed by the host. The `@grafana/runtime` package formalizes the frontend plugin SDK contract. Extensions registry (`public/app/extensions/`) provides panel-to-panel and app-to-panel extension slots — a second-level plugin extensibility layer within the frontend.

Unlike informal plugin systems where plugins are optional add-ons, in Grafana the plugin boundary is load-bearing: remove the plugin system and Grafana cannot connect to any data source or render any visualization. This is the canonical Microkernel criterion (Richards & Ford Ch.12): the core system has minimal standalone function without plugins.

### Modular Monolith (Secondary)

The backend is deployed as a single Go binary. Wire generates a monolithic DI graph. All 74 services share a SQL database. Domain boundaries are enforced by Go package scoping (not enforced by tooling, but maintained by convention). The `pkg/modules/` system provides named module groups that can optionally be deployed independently (StorageServer, SearchServer, ZanzanaServer as standalone gRPC services), but the primary and default mode is `target = all`, i.e., everything in one process. This is a Modular Monolith with optional module extraction capability — the same structural pattern seen in Grafana Loki and Mimir, which share the same `dskit` module infrastructure.

### What This Is NOT

- **Not Microservices:** All 74 services share one database and one process. There are no independent deployment units in the default mode. The `storage-server` and `search-server` module targets are optional horizontal scaling features, not the architectural model.
- **Not Layered:** The service organization is domain-vertical (dashboards, datasources, alerting, auth), not horizontal layer separation (presentation → business → data across the whole application). There is no formal controller/service/repository layering.
- **Not Service-Based:** The module system produces only a handful of optional standalone targets (3-4), but these are operational concerns, not the architectural model. They do not have independent databases.
- **Not Event-Driven:** `InProcBus` covers a small number of side-effect notifications. The primary integration between services is synchronous Go interface calls assembled by Wire.

## Quality Attributes

- **Extensibility:** The core quality driver. Every data source, panel type, and app is a plugin. The formal plugin pipeline, gRPC plugin process isolation, and `@grafana/runtime` SDK make Grafana extensible to 1,000+ community plugins without modifying the core.
- **Observability:** Grafana instruments itself with OpenTelemetry (tracing spans across HTTP and gRPC layers), Prometheus metrics via `prometheus/client_golang`, and structured logging. Usage stats are collected and reported through the `UsageStatsProviders` registry.
- **Scalability:** The `pkg/modules/` dskit integration enables horizontal scaling of the storage and search subsystems (ring-based sharding via `memberlist`). The default single-binary mode scales vertically. Redis-backed caching (`pkg/infra/remotecache`) and database connection pooling (xorm) support moderate horizontal deployment.
- **Security:** RBAC access control (`pkg/services/accesscontrol/`) with optional OpenFGA/Zanzana backend for relationship-based authorization. Plugin signature verification (3 levels: public, private, unsigned) enforced at load time. Backend plugins run as separate OS processes (process isolation). OAuth, LDAP, SAML authentication via `pkg/services/authn/`.
- **Reliability:** Background service lifecycle management (registry pattern), graceful shutdown via dskit service manager, feature toggle system for progressive rollout, Kubernetes readiness/liveness probes in Helm chart.
- **Maintainability:** 74 service packages with Wire-enforced constructor-based dependency declaration. Plugin pipeline stages separate concerns cleanly. `@grafana/data` and `@grafana/schema` (CUE-based) provide schema versioning for dashboard models.
- **Real-time Capability:** Grafana Live service (`pkg/services/live/`) uses Centrifuge (WebSocket broker) for streaming panel updates, live tailing, and channel-based pub-sub from streaming datasource plugins (e.g., Loki tail, Graphite streaming).

## Classification Reasoning

The Microkernel + Modular Monolith classification is supported by multiple independent lines of evidence:

1. **Plugin registry as primary integration seam:** The `InMemory` plugin registry in `pkg/plugins/manager/registry/` is the runtime manifestation of the Microkernel pattern. All datasources and panels are registered here. The host's `pkg/services/query/` fan-outs to plugins via `plugins.BackendFactoryProvider` — no direct SQL or API calls to external systems exist in the core.

2. **Formal plugin lifecycle pipeline:** `pkg/plugins/manager/pipeline/doc.go` explicitly documents a five-stage pipeline managed by the host. This is not an informal callback system — it is a structured lifecycle contract between host and plugins.

3. **gRPC process isolation for backend plugins:** `pkg/plugins/backendplugin/grpcplugin/` uses `hashicorp/go-plugin` to launch backend plugin binaries as separate OS processes and communicate over gRPC. The plugin boundary is a physical process isolation point, not just a code interface.

4. **Single binary with Wire DI:** `pkg/server/wire_gen.go` assembles all services into one constructor call. The `pkg/server/module_server.go` `Run()` method starts the dskit module manager over this single-process graph.

5. **Shared database:** All services use `pkg/infra/db.DB` (xorm over SQLite/MySQL/PostgreSQL). No per-service databases. This is the definitive Modular Monolith marker, distinguishing it from Microservices or Service-Based.

6. **Module extraction as scaling option, not primary model:** The dskit `RegisterModule` calls in `module_server.go` document that `StorageServer`, `SearchServer`, and `ZanzanaServer` can be deployed standalone. But `modules.All → {Core}` is the default target, and `Core` runs the full HTTP server. The module system is an operational scaling lever, not the architecture model.

The pre-existing heuristic classification (`Microkernel + Modular Monolith`, confidence 0.0) is confirmed by code-level evidence. The Microkernel primary classification is strengthened by observing that the plugin system is not simply feature extensibility — it is the mandatory integration path for all data connectivity and visualization. Confidence is 0.91; the 0.09 gap reflects the nascent `apps/` operator-pattern layer and the optional dskit module standalone targets, which hint at a longer-term architectural migration whose final shape is not yet resolved in the codebase.
