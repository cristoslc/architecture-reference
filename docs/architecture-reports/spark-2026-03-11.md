## Executive Summary

Apache Spark is a unified analytics engine for large-scale distributed data processing. Its primary architectural style is **Pipeline**, expressed at two distinct levels: the per-query SQL compilation pipeline (parse → analyze → optimize → physical plan → execute) implemented through Catalyst's `RuleExecutor` batch framework, and the distributed runtime pipeline (Stages → ShuffleMapStage → ResultStage → tasks) executed by the `DAGScheduler`. A strong secondary style is **Layered**, visible in the clearly separated horizontal strata: language APIs / SQL / high-level libraries, Catalyst compiler, core execution and scheduling, cluster management adapters, and network/shuffle infrastructure. A third co-present style is **Microkernel**, evidenced by formal plugin extension points (`SparkPlugin`/`DriverPlugin`/`ExecutorPlugin` Java SPI, `SparkSessionExtensions` rule injection, `ShuffleManager` SPI, `DataSourceV2` SPI, `ExternalClusterManager` SPI) that allow external code to replace or extend almost every subsystem without touching core.

**Primary style:** Pipeline
**Secondary styles:** Layered, Microkernel
**Classification confidence:** 0.90

## Architecture Evidence

### 1. Pipeline Style — Primary

The Pipeline style is most concretely expressed in the SQL/DataFrame query compilation path, implemented in `QueryExecution`:

```
SQL text / DataFrame plan
  → Parser (ANTLR grammar → Unresolved LogicalPlan)
  → Analyzer (Catalyst RuleExecutor batches → resolved LogicalPlan)
  → Optimizer (43+ rule files, RuleExecutor batches → optimized LogicalPlan)
  → Physical Planner (SparkPlan strategies → SparkPlan tree)
  → Preparations (InsertAdaptiveSparkPlan, CollapseCodegenStages, etc.)
  → Execute (RDD DAG → distributed computation)
```

Each stage is a discrete transformation with a well-defined input/output contract. The `RuleExecutor` framework (`sql/catalyst/src/main/scala/org/apache/spark/sql/catalyst/rules/RuleExecutor.scala`) applies ordered `Batch` sequences of `Rule[TreeType]` until fixpoint, making each stage a pure filter in the classic pipeline sense. `QueryExecution` exposes `analyzed`, `optimizedPlan`, `sparkPlan`, and `executedPlan` as named pipeline checkpoints.

At the runtime level, `DAGScheduler` (`core/src/main/scala/org/apache/spark/scheduler/DAGScheduler.scala`) decomposes every job into a directed acyclic graph of `ShuffleMapStage` and `ResultStage` objects. Stages execute in dependency order, materializing shuffle outputs between stages as data moves through the pipeline. The comment in `DAGScheduler.scala` explicitly says: "computes the minimal schedule to run the job… submits stages as TaskSets to an underlying TaskScheduler."

The `sql/pipelines/` module (Declarative Pipelines, formerly Delta Live Tables style) formalizes the pipeline pattern further: `DataflowGraph`, `FlowPlanner`, `PipelineExecution`, and `GraphExecution` implement a DAG of named datasets connected by flows — a first-class declarative pipeline abstraction built on top of Spark's execution engine.

### 2. Layered Style — Secondary

Spark exhibits a clean horizontal layering enforced by module boundaries and import directionality:

**Layer 5 (User-facing APIs):** PySpark, SparkR, Spark Connect client, `org.apache.spark.sql.Dataset`, `org.apache.spark.rdd.RDD`
**Layer 4 (High-level libraries):** Spark SQL, Structured Streaming, MLlib, GraphX, Spark Streaming
**Layer 3 (Catalyst compiler):** `sql/catalyst/` — Parser, Analyzer, Optimizer, Planner (no dependency on core execution)
**Layer 2 (Core execution):** SparkContext, DAGScheduler, TaskScheduler, BlockManager, ShuffleManager, SparkEnv
**Layer 1 (Transport/Infrastructure):** `common/network-common/`, `common/network-shuffle/`, RPC (Netty-based), `common/unsafe/`
**Layer 0 (External resource managers):** YARN, Kubernetes — plugged in via `ExternalClusterManager` SPI

`SparkEnv` (`core/src/main/scala/org/apache/spark/SparkEnv.scala`) acts as a service locator for layer 2: it holds `RpcEnv`, `SerializerManager`, `MapOutputTracker`, `BroadcastManager`, `BlockManager`, `SecurityManager`, `OutputCommitCoordinator`, and `ShuffleManager` as a named set of platform services.

The `sql/catalyst/` module has no dependency on `core/`; it only depends on `common/`. This is a deliberate architectural boundary: the compiler is pure functional plan transformation, independent of how plans are eventually executed.

### 3. Microkernel Style — Secondary

Spark defines multiple formal extension points that follow the microkernel pattern — a small, stable core with peripheral plug-ins:

- **`SparkPlugin` SPI** (`core/src/main/java/org/apache/spark/api/plugin/SparkPlugin.java`): `driverPlugin()` and `executorPlugin()` return driver-side and executor-side plugin components, loaded via configuration. Used for custom metrics collectors, profilers, and runtime extensions.
- **`ShuffleManager` SPI**: Pluggable shuffle implementation; the default is `SortShuffleManager` but external implementations can be loaded.
- **`ExternalClusterManager` SPI**: Pluggable cluster managers (YARN, Kubernetes, custom) — the core scheduler does not know which cluster manager it talks to.
- **`SparkSessionExtensions`** (`sql/core/src/main/scala/org/apache/spark/sql/SparkSessionExtensions.scala`): Allows injection of custom Analyzer rules, Optimizer rules, Check Analysis rules, Optimizer strategies, Parsers, Columnar rules, Adaptive Query Execution rules, and post-planner strategy rules without modifying core code.
- **`DataSourceRegister` / DataSource V2 SPI**: External data sources (Kafka, Avro, Delta, Iceberg, etc.) plug in as named data sources without modifying SQL core.
- **`ResourceDiscoveryPlugin` SPI**: Custom resource discovery for GPUs, FPGAs, etc.

These extension points follow the classic microkernel contract: the core exposes stable interfaces, plug-ins are loaded dynamically, and the core delegates to them without embedding their logic.

### 4. Event-Driven Instrumentation (Implementation Mechanism)

The `LiveListenerBus` (`core/src/main/scala/org/apache/spark/scheduler/LiveListenerBus.scala`) provides an asynchronous, multi-queue event bus used for internal observability: UI updates, history server logging, metrics, and dynamic executor allocation. `SparkListenerEvent` sealed class hierarchy carries job, stage, task, executor, and storage events. This is not the primary organizational style — it is an internal decoupling mechanism used by infrastructure components — but it is a significant structural element worth noting.

## Classification Rationale

**Why Pipeline (primary)?** The most important organizing constraint in Spark is the sequence of transformations applied to a query or dataset before execution. The Catalyst pipeline is not merely an implementation detail — it is the reason the codebase is structured as it is. Catalyst's optimizer is a battery of 43+ rule files, each a filter in the pipeline. `QueryExecution` names and exposes each pipeline stage as a first-class concept. The DAGScheduler's stage graph is a runtime pipeline where data flows forward through shuffle boundaries. The Declarative Pipelines module (`sql/pipelines/`) makes this explicit for user-facing workflows. Pipeline is the dominant structural metaphor.

**Why Layered (secondary)?** The module structure enforces strict downward dependency: APIs depend on SQL, SQL depends on Catalyst, Catalyst does not depend on core, core depends on common. This is textbook layering, not coincidence — the `catalyst` module's independence from `core` is a documented design decision that enables the optimizer to be embedded in standalone tools (e.g., Spark Connect client can run the optimizer locally).

**Why Microkernel (secondary)?** Six distinct formal extension point systems — each with a named Java interface, configuration-driven loading, and documented lifecycle — constitute a genuine microkernel boundary. External ecosystem projects (Delta Lake, Iceberg, Hudi, third-party shuffle services, profilers) integrate exclusively through these points. This is distinguishable from incidental extensibility: the extension points are designed, versioned, and stabilized as `@DeveloperApi`.

**Why not Microservices?** All Spark modules deploy as a single JVM process (driver) plus per-application executor processes. There are no independently deployable services with separate data stores or network APIs between modules. Spark Connect adds a gRPC boundary between client and server, but this is a client protocol, not a decomposition of Spark internals.

**Why not Event-Driven (primary)?** The `LiveListenerBus` and `DAGSchedulerEventProcessLoop` use events, but events are internal coordination mechanisms, not the primary organizational principle. Data flow in Spark is sequential and transformation-based (Pipeline), not reactive to external events as the organizing structure.
