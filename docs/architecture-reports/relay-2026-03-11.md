# Architecture Report: Sentry Relay

**Date:** 2026-03-11
**Repository:** https://github.com/getsentry/relay
**Classification:** Pipeline (primary), Modular Monolith (secondary)
**Confidence:** 0.88

---

## Summary

Sentry Relay is a Rust-based event ingestion proxy that sits between SDKs and the Sentry backend. It is organized as a Modular Monolith — a single Rust workspace of ~28 crates deployed as one binary — whose internal runtime is governed by an explicit, multi-stage processing pipeline through which every ingested envelope passes. The pipeline metaphor is the dominant architectural force: envelopes enter through HTTP endpoints, queue in a partitioned buffer, pass through normalization/filtering/rate-limiting/PII-scrubbing stages in the EnvelopeProcessor pool, and are finally forwarded to either Kafka (processing mode) or an upstream HTTP relay. The modular monolith shape provides the physical structure (separate crates with strict dependency boundaries); the pipeline is the runtime execution model that gives the system its identity.

## Runtime Architecture

### Actor/Service System (`relay-system`)

The foundation is a bespoke actor-like service system built on Tokio. The core primitives are:

- **`Interface`** — a typed message enumeration for a service
- **`Addr<I>`** — a cloneable sender handle backed by a Tokio unbounded mpsc channel
- **`Receiver<I>`** — the inbound channel polled inside a service's `run` loop
- **`Service` trait** — every internal component implements this; `run(rx: Receiver<I>)` is the main loop
- **`BroadcastChannel`** — efficient fan-out for debounced/cached responses (used in ProjectCache)

Services are registered in a `ServiceRegistry` and started on a shared Tokio runtime. The `Controller` service listens for OS signals and coordinates graceful shutdown via `ShutdownHandle`.

### Service Registry (`relay-server/src/service.rs`)

`ServiceState::start()` wires together all services into a `Registry` struct that holds typed `Addr<T>` handles:

- `UpstreamRelayService` — HTTP client to the upstream Sentry or parent relay
- `GlobalConfigService` — polls upstream for global config, fan-outs via watch channel
- `ProjectCacheService` — maintains in-memory project config cache, backed by upstream or Redis
- `PartitionedEnvelopeBuffer` — N partitions of disk-backed (SQLite) + in-memory envelope queues
- `EnvelopeProcessorService` (pool) — CPU-thread pool executing the multi-stage processing pipeline
- `RouterService` (metrics) — routes metric buckets to namespace-scoped aggregators
- `OutcomeProducerService` — emits disposition outcomes to Kafka or upstream HTTP
- `OutcomeAggregator` — batches and deduplicates outcome records
- `RelayCacheService` — authenticates and caches downstream relay registrations
- `HealthCheckService` — `/health` endpoint backed by aggregator and upstream probes
- `StoreService` (pool, processing mode only) — serializes envelopes to Kafka topics
- `ObjectstoreService` (processing mode only) — stores large attachments to object storage
- `AutoscalingMetricService` — emits scaling signals based on buffer depth and memory pressure
- `CogsService` — COGS (cost-of-goods-sold) attribution tracking

Two operating modes are gated by the `processing` Cargo feature and the `RelayMode` enum:

- **Managed/Processing mode**: full pipeline — normalization, PII, sampling, Redis quotas, Kafka output
- **Proxy mode**: `ProxyProcessorService` — minimal processing, forward to upstream HTTP

### The Processing Pipeline

The envelope pipeline is the defining runtime pattern. Every ingested SDK payload traverses these ordered stages inside `EnvelopeProcessorService`:

1. **Receive** — HTTP endpoint accepts the envelope, decompresses, extracts event ID, returns 200 immediately
2. **Buffer** — `PartitionedEnvelopeBuffer` queues the envelope in a partition keyed by project DSN pair; persists to SQLite under memory pressure
3. **Project config fetch** — `ProjectCacheService` provides (or fetches) the project config; requests are batched and deduplicated via `BroadcastChannel`
4. **Parse** — envelope items are deserialized into typed structs
5. **Normalize** — `relay-event-normalization` applies schema normalization, timestamp correction, and platform-specific mutations
6. **Filter** — `relay-filter` applies inbound filters (IP, error message, crawlers)
7. **Dynamic sampling** — `relay-sampling` evaluates rules against trace context and issues a keep/drop decision
8. **Rate limit (quotas)** — Redis Lua scripts (`is_rate_limited.lua`) enforce per-project quotas; results written back to in-memory project cache
9. **PII scrub** — `relay-pii` applies data-scrubbing and PII config from project config
10. **Metrics extraction** — transaction, span, and session metrics are extracted and routed to `RouterService` for aggregation
11. **Forward** — processed envelope is either published to a typed Kafka topic (`ingest-events`, `ingest-transactions`, `ingest-spans`, `profiles`, `ingest-metrics`, etc.) via `StoreService`, or forwarded via HTTP to the upstream relay

Each stage is isolated in its own processing sub-module (`relay-server/src/processing/`) and implements the `Processor` trait, which produces typed `Output` implementing `Forward`. This composable trait chain is the explicit pipeline contract.

### Kafka Integration

When running in processing mode, Relay writes to ten Kafka topics via `relay-kafka`:

- `ingest-events`, `ingest-attachments`, `ingest-transactions`
- `ingest-spans`, `profiles`, `ingest-metrics` (sessions + generic)
- `outcomes`, `outcomes-billing`
- OpenTelemetry and replay-specific topics

The Kafka write is fire-and-forget; Relay does not consume from Kafka. This makes Relay the first half of an event-driven ingest pipeline, but Relay itself is not event-driven internally — inter-service communication is synchronous message-passing, not event pub/sub.

## Architectural Style Analysis

### Primary: Pipeline

The dominant runtime pattern is a deterministic, ordered sequence of stages through which every envelope item must pass before it reaches its destination. This is the explicit design intent documented in `relay-server/src/lib.rs` with both a Mermaid graph and sequence diagram. The `Processor` trait with its `UnitOfWork` / `Output` / `Forward` type chain formalizes this pipeline contract in the type system. Each processing sub-module is a pipeline stage with single-direction data flow. The order of stages is fixed and cannot be reconfigured by users.

### Secondary: Modular Monolith

The physical deployment unit is a single Rust binary compiled from one workspace. The ~28 crates provide strict module boundaries enforced at compile time by Rust's crate visibility rules, but there is no network boundary — all inter-crate communication is in-process function calls or the internal `Addr<I>` message bus. This is the defining characteristic of a Modular Monolith: strong internal modularity within a monolithic deployment unit.

### Why Not Microservices / Event-Driven

Relay is a single-process binary; there are no independently deployable service boundaries. While Relay *produces* to Kafka, it does not internally route work via an event bus — the inter-service communication uses typed point-to-point channels (`Addr<I>`), not a broker. The actor/service model is an implementation mechanism enabling async concurrency, not an architectural style that defines how the system is composed.

### Why Not Service-Based

Service-Based architecture requires multiple independently deployable coarse-grained services. Relay deploys as one binary. Horizontal scale is achieved by running multiple identical relay instances (process-level scale-out), not by decomposing the system into separate services.

## Quality Attributes

| Attribute | Mechanism |
|---|---|
| **Throughput** | Async Tokio runtime, CPU thread pool for processing, SQLite-backed envelope buffer absorbs bursts |
| **Reliability** | Durable envelope buffer (SQLite), graceful shutdown via `Controller`, retry logic in `UpstreamRelayService` |
| **Scalability** | Horizontal scale-out by running N relay instances; internal `AutoscalingMetricService` emits signals for autoscalers |
| **Security** | PII scrubbing at pipeline stage, relay registration authentication, rate limiting via Redis, inbound filters |
| **Observability** | StatsD metrics via `relay-statsd`, structured tracing via `tracing`, Sentry SDK integration for self-reporting errors |
| **Modifiability** | Crate-level boundaries enforced by Rust compiler; processing stages isolated in `processing/` sub-modules |
| **Performance** | mimalloc allocator, zstd/brotli/gzip compression, batched upstream requests, metric aggregation before forwarding |
| **Operability** | Health check endpoint, graceful shutdown, proxy/managed mode selection, feature-flagged `processing` capability |
