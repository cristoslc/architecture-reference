# gitlabhq — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/gitlabhq/gitlabhq
**Classification:** Modular Monolith + Layered + Event-Driven
**Confidence:** 0.92

## Summary

GitLab CE is a large-scale Ruby on Rails application whose defining architectural characteristic is a deliberate Modular Monolith: 69 explicitly governed bounded contexts (declared in `config/bounded_contexts.yml`, split into a domain layer and a platform layer) are enforced via namespace conventions across every layer of the Rails stack. The bounded contexts map to domain-namespaced subdirectories under `app/models/`, `app/services/`, `app/workers/`, `app/finders/`, and `app/graphql/`, and they map to separate PostgreSQL database schemas in `db/gitlab_schemas/`. This modular organization sits on top of a classic Layered Rails foundation (Controllers → Services/Finders/Policies → Models → Serializers/Presenters), augmented by a formal internal Event-Driven mechanism (`Gitlab::EventStore`) that publishes typed domain events and routes them to Sidekiq workers as subscribers to decouple bounded contexts from one another.

### Key Architectural Files

- `config/bounded_contexts.yml`: Authoritative declaration of 69 bounded contexts. Comments state "Only the defined bounded contexts are allowed" and that application/view layers are exempt from this constraint. Defines `domains:` section (67 contexts: Ai, Analytics, Authn, Authz, Ci, MergeRequests, WorkItems, etc.) and `platform:` section (Gitlab, Banzai). This is the primary modular governance document.
- `lib/gitlab/event_store.rb`: Internal pub-sub hub. Documents its purpose as "a simple pub-sub mechanism that lets you publish domain events and use Sidekiq workers as event handlers" and "decouple domains from different bounded contexts." Contains all event subscriptions — 45 `EventStore.publish` call sites in `app/`, 19 subscriber workers implementing `Gitlab::EventStore::Subscriber`.
- `lib/gitlab/event_store/store.rb`: Immutable subscription registry (`@subscriptions.freeze` after initialization), ensuring the event routing table cannot be modified at runtime.
- `app/events/ci/pipeline_created_event.rb`: Typed domain event with JSON Schema validation. Illustrates event authorship under the publishing bounded context namespace (`Ci::`).
- `config/database.yml.decomposed-postgresql`: Decomposes GitLab into separate `main`, `ci`, and `geo` PostgreSQL databases — an architectural boundary that aligns with the `gitlab_ci`, `gitlab_main`, and `gitlab_shared` schema files.
- `db/gitlab_schemas/`: 15 database schema files grouping tables by domain. Cross-schema foreign keys are managed through `gitlab_loose_foreign_keys.yml`.
- `app/models/project.rb` (4,146 lines) / `app/models/user.rb` (3,278 lines): These are acknowledged "God Objects" — a known legacy coupling point that bounded context enforcement is progressively correcting. They include 50+ Ruby modules each.
- `gems/` (23 gems): Platform layer extractions including `gitlab-database-load_balancing`, `gitlab-schema-validation`, `gitlab-utils`, `gitlab-backup-cli`. Demonstrates progressive extraction of platform concerns from the monolith.
- `app/workers/all_queues.yml`: 560 named Sidekiq queues organized by namespace (e.g., `activity_pub:*`, `authorized_project_update:*`, `ci:*`). Queue namespacing mirrors bounded context namespacing.
- `app/graphql/gitlab_schema.rb`: GraphQL schema with complexity/depth limits, ActionCable subscriptions, batch loading, and pagination. Provides a second API surface alongside the Grape REST API.

### Patterns Found

**Modular Monolith (Primary):** The bounded context system is the defining characteristic. `config/bounded_contexts.yml` governs which namespaces are permitted in the domain and platform layers, with tooling enforcing conformance (rubocop rules, lint checks). Each bounded context gets its own subdirectory across all application layers (`app/models/ci/`, `app/services/ci/`, `app/workers/ci/`, `app/finders/ci/`) and maps to distinct database schemas (`db/gitlab_schemas/gitlab_ci.yaml`). 23 extracted gems represent the advanced stage of the platform layer extraction program. The queue namespacing in Sidekiq (`all_queues.yml`) mirrors bounded contexts, making the modular decomposition visible even in the async processing layer.

**Layered (Secondary):** The Rails stack provides a strict horizontal layering: HTTP controllers handle routing and auth, a service layer (167+ service classes) encapsulates business logic, a dedicated finder layer (229 files) isolates complex queries, a policy layer (181 files using `DeclarativePolicy::Base`) manages authorization, ActiveRecord models handle persistence (315 root-level + domain subdirectories), and serializers/presenters (311 + 104 files) format responses. Both the Grape REST API (`lib/api/`, 173 files) and GraphQL API (`app/graphql/`, 1,343+ files) sit at the same presentation boundary. The layered structure holds within each bounded context — every domain replicates the same layer hierarchy.

**Event-Driven (Supporting):** `Gitlab::EventStore` provides a formal typed pub-sub mechanism for inter-domain communication. Events are defined with JSON Schema validation per domain (e.g., `Ci::PipelineCreatedEvent`, `MergeRequests::ApprovedEvent`, `Projects::ProjectCreatedEvent`). Subscribers are Sidekiq workers implementing `Gitlab::EventStore::Subscriber`. The subscription registry is frozen at initialization to prevent runtime mutation. This mechanism is explicitly designed to decouple bounded contexts — a domain publishes an event without knowing its subscribers. 45 publish call sites and 19 subscriber workers are currently registered. Additionally, ActionCable (`app/channels/`) provides WebSocket real-time delivery to browser clients, and Sidekiq (560 queues, 612 worker files) is the core async execution backbone.

## Architecture Styles Identified

### Modular Monolith (Primary)

The decisive classification signal for GitLab CE is the formal bounded context governance system. Unlike many large Rails applications that merely use namespace conventions informally, GitLab has elevated this to a tooling-enforced architectural constraint: `config/bounded_contexts.yml` is the authoritative registry, rubocop enforces namespace conformance, database schemas align with bounded contexts, and the platform layer extraction program (23 gems) demonstrates a deliberate roadmap toward extractable modules. The bounded context structure is present at every layer simultaneously — models, services, workers, finders, policies, and database schemas all carry the same namespace taxonomy. The queue naming in Sidekiq (`authorized_project_update:*`, `ci:pipeline_*`, `merge_requests:*`) extends this modularity into the async infrastructure.

The "monolith" aspect is clear: a single Rails application (`Gitlab::Application`) deployed as one unit (Puma + Sidekiq), sharing PostgreSQL and Redis, with no independent deployment units. There are no inter-service API calls between decomposed services. The decomposed PostgreSQL configuration (`database.yml.decomposed-postgresql`) is an advanced operational option, not the primary deployment model.

### Layered (Secondary)

GitLab's Rails skeleton follows a strict multi-layer hierarchy that has been extended well beyond standard MVC. The service layer, finder layer, policy layer, presenter layer, serializer layer, and the dual API surface (Grape REST + GraphQL) represent a mature, deliberately stratified architecture. Each layer has a clear single responsibility. The GraphQL API is particularly sophisticated — 323 mutations, 275 resolvers, 745 types — and is effectively a second presentation layer running in parallel with the REST API. The layered structure provides the internal organization within each bounded context.

### Event-Driven (Supporting)

`Gitlab::EventStore` is not a bolted-on integration adapter — it is a deliberately designed inter-domain communication mechanism whose primary stated purpose is bounded context decoupling. The immutable subscription registry, typed events with JSON Schema validation, and subscriber worker pattern represent a production-grade internal event bus. However, with 45 publish sites and 19 subscribers, it is not yet the dominant integration mechanism between all contexts (many cross-context calls still happen via direct service invocations). ActionCable adds real-time event delivery to the browser tier. Sidekiq provides the universal async execution backbone (560 queues, 612 workers, covering background processing for every domain).

## Quality Attributes

- **Modularity:** The dominant quality driver. The bounded context system, database schema decomposition, and platform layer gem extraction program all serve the goal of making a large, evolving codebase navigable and maintainable by distributed teams without cross-team coupling.
- **Maintainability:** Namespace enforcement, the finder/policy/service layer separation, rubocop rules, and feature-category ownership tracking (each file declares a `feature_category` for observability routing) collectively make a 64,000+ file codebase maintainable at scale.
- **Scalability:** Decomposed PostgreSQL databases (CI workloads isolated to `gitlab_ci` schema), Redis cluster support, 560 Sidekiq queues with independent scaling, GitLab Workhorse for Git-over-HTTP offloading, and Puma multi-process deployment. GitLab.com is the largest known deployment, running on Kubernetes with Helm charts.
- **Extensibility:** The GitLab EE/JH edition model is implemented via Ruby module prepending (`Gitlab::EventStore.prepend_mod_with`, `EE::` namespace injection via `config.class.prepend`) and conditional `Gitlab.ee { }` blocks, allowing enterprise features to augment the CE core without modifying CE source files.
- **Reliability:** Sidekiq reliable fetcher (vendored `sidekiq-reliable-fetch`), Sentry integration for error tracking, OpenTelemetry instrumentation across all key Rails layers (ActionPack, ActiveRecord, Grape, ActiveJob), Prometheus metrics via `prometheus-client-mmap`, and Rails database load balancing via the extracted `gitlab-database-load_balancing` gem.
- **Observability:** Structured JSON logging (`api_json.log` via GrapeLogging), OpenTelemetry tracing spanning web and Sidekiq layers, Prometheus metrics, Sentry error tracking, and per-worker `feature_category` labeling enabling ownership-based alerting.
- **Security:** `DeclarativePolicy::Base` authorization (181 policy files), `APIGuard` for REST API token validation, Grape-level authentication middleware, GraphQL complexity limits, audit event framework (`AuditEvents` bounded context), and secrets management (`SecretsManagement` bounded context).

## Classification Reasoning

The decisive classification question for GitLab CE is whether the bounded context governance system constitutes a true Modular Monolith or merely an informally organized Layered Rails application. The evidence decisively supports Modular Monolith as primary:

1. **Formal governance document:** `config/bounded_contexts.yml` is not a convention — it is the authoritative boundary registry. Comments state "Only the defined bounded contexts are allowed" in the domain and platform layers. This is a compile-time governance constraint, not a naming suggestion.

2. **Multi-layer consistency:** The bounded context taxonomy is enforced simultaneously at the model layer (`app/models/ci/`), service layer (`app/services/ci/`), worker layer (`app/workers/ci/`), finder layer (`app/finders/ci/`), database schema (`db/gitlab_schemas/gitlab_ci.yaml`), and Sidekiq queue naming (`ci:pipeline_*`). This is not accidental — it represents a deliberate, systemic architectural decision.

3. **Database decomposition:** The `db/gitlab_schemas/` directory contains 15 domain-scoped schema files, and `database.yml.decomposed-postgresql` provides a production-ready decomposed database configuration with separate `main` and `ci` databases. This mirrors the module boundary at the persistence layer — a hallmark of mature Modular Monolith design.

4. **Platform layer extraction program:** 23 extracted gems demonstrate a progressive modularization roadmap. The `platform:` section of `bounded_contexts.yml` explicitly frames the platform layer as a target for gem extraction. This is Modular Monolith architecture operating as a stepping stone toward potentially more granular decomposition.

5. **Event-Driven inter-domain decoupling:** `Gitlab::EventStore` explicitly names its purpose as "decouple domains from different bounded contexts." Its typed events, frozen subscription registry, and subscriber worker pattern are the inter-module communication mechanism of a Modular Monolith — analogous to how Modular Monolith frameworks provide internal event buses.

The Layered style is a genuine secondary classification. The depth of the service/finder/policy/serializer/presenter stratification, and the dual REST+GraphQL API surface, go well beyond standard Rails MVC and represent a deliberate architectural investment in horizontal layer isolation within each module.

The Event-Driven classification is real but supporting. `Gitlab::EventStore` is a production-quality typed event bus, but with 45 publish sites and 19 subscribers it covers only a portion of inter-domain communication. Direct service invocations between domains remain the dominant integration pattern. ActionCable and Sidekiq amplify the event-driven character but operate as infrastructure mechanisms within the modular structure.

Previous classification (deep-analysis-tooluse, GLM-5, 2026-03-08) identified `Layered + Domain-Driven Design`. This reclassification upgrades the primary style to `Modular Monolith` because the bounded context governance, database schema decomposition, and platform layer extraction program constitute enforcement infrastructure beyond what DDD alone implies — the system actively enforces module boundaries, not merely organizes code by domain. The `Layered` secondary classification is retained. `Domain-Driven Design` is dropped as a top-level style because while GitLab uses DDD vocabulary (bounded contexts, ubiquitous language per domain), the architecture style is defined by its enforcement mechanism (Modular Monolith) rather than the DDD methodology that informed it.

Confidence is 0.92. The Modular Monolith + Layered + Event-Driven classification is strongly supported by code-level evidence at multiple scales. The minor gap accounts for the residual coupling through `project.rb` and `user.rb` God Objects that partially undercut the module boundary claims, and the fact that the decomposed database deployment is optional rather than the only supported mode.
