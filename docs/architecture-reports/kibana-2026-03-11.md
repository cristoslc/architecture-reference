# Architecture Report: Kibana

**Date:** 2026-03-11
**Analyst:** claude-sonnet-4-6 (deep-analysis)

**SPEC:** SPEC-031

---

## Summary

Kibana is a **Microkernel** architecture, secondarily exhibiting strong **Modular Monolith** characteristics organized along **Domain-Driven Design** principles. The system is a single deployable Node.js process whose minimal core (`src/core/`) provides runtime services and orchestrates plugin lifecycle, while virtually all user-facing functionality (dashboards, discover, alerting, APM, security, ML, maps, fleet, and 1,294 registered plugins) is implemented as plugins. The Microkernel is the dominant structural pattern because the Core cannot deliver meaningful user functionality in isolation — plugins are not optional extensions but mandatory contributors of all application behavior. The Modular Monolith character is expressed through 56+ shared platform plugins, 60+ x-pack platform plugins, and 160+ solution plugins that share a common Elasticsearch persistence layer via Core's Saved Objects service and execute within the same server process. Domain-Driven organization is enforced at build time by the `kbn-eslint-plugin-imports` `no_group_crossing_imports` rule, which statically prevents `private`-visibility modules from being imported across group boundaries (`platform`, `security`, `observability`, `search`, `workplaceai`).

## Evidence

### 1. Microkernel (primary)

**Core as minimal kernel**

`src/core/packages/root/server-internal/src/server.ts` defines the `Server` class — the runtime kernel. The constructor instantiates 25 core services directly:

```
AnalyticsService, CoreInjectionService, ContextService, FeatureFlagsService,
HttpService, RenderingService, PluginsService, ElasticsearchService,
SavedObjectsService, UiSettingsService, CapabilitiesService,
EnvironmentService, NodeService, MetricsService, StatusService,
CoreAppsService, HttpRateLimiterService, HttpResourcesService,
LoggingService, CoreUsageDataService, I18nService, DeprecationsService,
UserActivityService, ExecutionContextService, PrebootService,
PricingService, DocLinksService, CustomBrandingService,
UserSettingsService, SecurityService, UserProfileService,
DataStreamsService
```

These services provide HTTP transport, Elasticsearch connectivity, saved object persistence, authentication primitives, and status reporting. They are the kernel: necessary infrastructure with no user-visible behavior.

**Plugin contract — three-phase lifecycle**

Every plugin must implement the `Plugin` interface from `@kbn/core-plugins-server`. The lifecycle phases — `preboot()`, `setup(core, deps)`, and `start(core, deps)` — are managed by `PluginsService` (`src/core/packages/plugins/server-internal/src/plugins_service.ts`) and orchestrated by `PluginsSystem` (`plugins_system.ts`). The kernel calls each phase in topological dependency order, computed by Kahn's algorithm (`getTopologicallySortedPluginNames`). Plugins cannot call each other outside this contract.

**Plugin manifest and explicit dependency declaration**

Each plugin has a `kibana.jsonc` manifest declaring:
- `requiredPlugins` — plugins that must be initialized before this one (enforced by topological sort)
- `optionalPlugins` — plugins used only if present
- `group` — the bounded context domain (`platform`, `security`, `observability`, `search`, `workplaceai`)
- `visibility` — `shared` (cross-group accessible) or `private` (same-group only)

Example from the Security Solution plugin (`x-pack/solutions/security/plugins/security_solution/kibana.jsonc`):

```jsonc
{
  "group": "security",
  "visibility": "private",
  "plugin": {
    "id": "securitySolution",
    "requiredPlugins": ["actions", "alerting", "cases", "data", "embeddable", "fleet", "ml", ...],
    "optionalPlugins": ["cloudExperiments", "discover", "maps", ...]
  }
}
```

**1,294 plugin manifests**

Running `find /tmp/classify-kibana -name "kibana.jsonc" -not -path "*/node_modules/*"` returns 1,294 files. Plugin search paths include:
- `src/platform/plugins/shared/` — 56 shared platform plugins (charts, data, discover, dashboard, console, embeddable, expressions, esql, files, inspector, ...)
- `x-pack/platform/plugins/shared/` — 60 x-pack platform plugins (actions, alerting, fleet, cases, ml, inference, index_management, ...)
- `x-pack/solutions/observability/plugins/` — 18 plugins (apm, infra, slo, synthetics, uptime, ...)
- `x-pack/solutions/security/plugins/` — 13 plugins (security_solution, cloud_security_posture, entity_store, ...)
- `x-pack/solutions/search/plugins/` — 11 plugins (content_connectors, data_catalog, ...)
- `x-pack/solutions/workplaceai/plugins/` — 2 plugins
- `examples/` — 44 example plugins for third-party developer onboarding

**The canonic Microkernel criterion, from official docs**

`dev_docs/key_concepts/kibana_platform_plugin_intro.mdx` (authoritative developer documentation):

> *"Did you know that almost everything you see in the Kibana UI is built inside a plugin? If you removed all plugins from Kibana, you'd be left with an empty navigation menu, and a set of developer tools. The Kibana platform is a blank canvas, just waiting for a developer to come along and create something!"*

This satisfies the canonical Richards & Ford criterion (Ch.12): "The core system of the microkernel pattern contains only the minimal functionality required to make the system operational."

**Plugin discovery and runtime loading**

`discover()` in `plugins_discovery.ts` scans `config.pluginSearchPaths` (an RxJS observable pipeline: `merge → concatMap → parseManifest`), creating `PluginWrapper` instances for each valid manifest. The `PluginWrapper` class (`plugin.ts`) wraps each plugin, managing its Inversify DI container (`@kbn/core-di`), lifecycle invocations, and contract exposure.

**Third-party plugin support**

`docs/extend/external-plugin-development.md` documents the process for third parties to develop, package, and install plugins. `packages/kbn-plugin-generator` provides a Yeoman-based scaffolding tool. External plugins are loaded from `config.additionalPluginPaths`.

### 2. Modular Monolith (secondary)

**Single deployable process**

Kibana is packaged and deployed as a single Node.js process. There is no service-to-service network boundary between plugins at runtime. All plugins share:
- The Core HTTP server and routing infrastructure
- The Core `ElasticsearchService` for all Elasticsearch connectivity
- The Core `SavedObjectsService` as the unified persistence layer — all plugins register their saved object types through this service, which manages schema migrations across all plugin types atomically
- A single process address space with shared V8 heap

**43 core packages + 130+ platform packages**

The core runtime is decomposed into 43 `@kbn/core-*` packages in `src/core/packages/` (one per service subdomain: `plugins`, `http`, `elasticsearch`, `saved-objects`, `ui-settings`, `capabilities`, `logging`, `metrics`, `security`, `rendering`, etc.), plus 130+ `@kbn/*` packages across `packages/`, `src/platform/packages/`, and `x-pack/platform/packages/`. This fine-grained package decomposition enforces physical boundaries within the monolith without introducing network hops.

**Node roles for horizontal scaling**

`src/core/packages/node/server/src/types.ts` defines `NodeRoles`:

```typescript
interface NodeRoles {
  backgroundTasks: boolean;  // operations not involving HTTP traffic from UI
  ui: boolean;               // handling browser HTTP traffic
  migrator: boolean;         // one-time migration run then shutdown
}
```

Kibana nodes can be specialized by role (e.g., running multiple `backgroundTasks` nodes without `ui` capability), enabling horizontal scaling at the process level. This is an operational scaling mechanism, not a decomposition into independent services — all roles share the same codebase and deployment artifact.

### 3. Boundary enforcement tooling (DDD-adjacent)

**`no_group_crossing_imports` ESLint rule**

`packages/kbn-eslint-plugin-imports/src/rules/no_group_crossing_imports.ts` enforces the group/visibility contract at build time. When a `security`-group plugin attempts to import a `platform`-group `private` module, the rule fires:

```
⚠ Illegal import: "securityPlugin" (security) is importing "someModule" (platform/private).
```

The groups (`platform`, `security`, `observability`, `search`, `workplaceai`) correspond directly to `x-pack/solutions/*/` directory boundaries. `shared` visibility modules are accessible cross-group; `private` modules are bounded-context-local. This is a static, tooling-enforced bounded context boundary — the structural hallmark of DDD organization applied to plugin grouping.

**`ModuleVisibility` type**

`src/platform/packages/private/kbn-projects-solutions-groups/src/module_groups.ts`:

```typescript
export type ModuleVisibility = 'private' | 'shared';
```

This type, together with the ESLint enforcement, constitutes a formal API visibility system at the group level, analogous to DDD bounded context public/private interface declarations.

### What Was Ruled Out

**Microservices** — There is no independent deployment unit for individual plugins. All plugins run in the same Node.js process. Kibana node roles (`ui`, `backgroundTasks`) allow process-level specialization but share the same artifact and codebase. No plugin has its own network endpoint or independent database schema.

**Layered** — The codebase is organized domain-vertically by plugin group (security, observability, search), not horizontally across presentation, business, and data layers. Individual plugins have `public/` (browser) and `server/` (server) subdirectories, but this is a client-server split within each plugin, not a system-wide layering axis.

**Service-Based** — While there are solution groups that could be described as coarse-grained, they are not independently deployed services. They share one Elasticsearch instance, one saved objects schema, and one process. There is no coarse-grained network boundary between solution groups at runtime.

**Event-Driven** — Kibana does not use a message broker or async event queue as its primary integration mechanism. Plugin-to-plugin communication is synchronous TypeScript interface calls through the contract system (setup/start return values). Some plugins use RxJS observables for reactive state sharing, which is event-reactive but not an Event-Driven architectural style.

**Pipeline** — There is no data processing pipeline as an organizing structural principle. The plugin load sequence uses RxJS pipelines internally (`merge → concatMap → parseManifest`), but this is a plugin discovery implementation detail, not the architectural topology.

## Quality Attributes

- **Extensibility:** The core quality driver. The plugin contract (`setup`/`start`/`stop` + manifest) is the universal integration seam. Third-party developers can build and install external plugins without modifying the Kibana source. The 1,294 internal plugin manifests demonstrate the pattern operates at industrial scale.
- **Maintainability:** Group/visibility boundaries enforced by `no_group_crossing_imports` ESLint rule prevent cross-domain coupling from accumulating silently. The 43 `@kbn/core-*` packages decompose the kernel into independently versionable units. Inversify DI containers per plugin decouple instantiation from usage.
- **Scalability:** Node roles (`ui`, `backgroundTasks`) allow process-level horizontal scaling. Multiple `backgroundTasks` nodes can run scheduled work without UI load. `migrator` role enables zero-downtime rolling upgrades by running migrations before traffic is enabled.
- **Security:** Plugin group/visibility system prevents unauthorized cross-domain access at build time. Core `SecurityService` provides authentication and authorization primitives. Saved objects type-level access control restricts which plugins can read/write each object type. Plugin manifests declare `requiredBundles` to prevent bundle sharing without explicit dependency.
- **Observability:** Kibana instruments itself with Elastic APM (`elastic-apm-node`) for distributed tracing, exposes metrics via `MetricsService`, and provides structured logging via `LoggingService`. `AnalyticsService` tracks usage statistics across plugins. Status API (`StatusService`) aggregates health across all registered plugin dependencies.
- **Reliability:** Three-phase lifecycle (`preboot → setup → start`) with graceful shutdown, topological plugin initialization order preventing dependency initialization races, and atomic saved object migrations across all plugin schema types. `migrator` node role enables safe database-level upgrades before serving traffic.
- **Portability:** Kibana runs in three deployment modes: self-managed (on-prem/cloud), Elastic Cloud Hosted, and Elastic Cloud Serverless (project types: `observability`, `security`, `elasticsearch`). The serverless mode uses `allowlistPluginGroups` in plugin config to load only the relevant solution group's plugins, dynamically narrowing the loaded plugin set without code changes.

## Classification Reasoning

The Microkernel classification rests on four independent lines of evidence:

1. **Official documentation confirms the canonical criterion:** The developer docs state explicitly that removing all plugins leaves an empty navigation menu. This satisfies Richards & Ford Ch.12's definition of a microkernel system — the core has no standalone user value.

2. **Plugin lifecycle is managed by the kernel, not by plugins:** `PluginsSystem.setupPlugins()` calls each plugin's `setup()` in topological dependency order, passing only the contracts declared as `requiredPlugins`. Plugins cannot self-initialize or bypass the lifecycle. The kernel controls when and with what context each plugin activates.

3. **1,294 plugin manifests across 6 search paths:** The scale of the plugin registry — covering observability, security, search, workplace AI, and platform capabilities — confirms that the plugin system is not a peripheral extensibility feature. It is the primary delivery mechanism for all Kibana functionality.

4. **Build-time boundary enforcement:** The `no_group_crossing_imports` ESLint rule enforces that `private`-visibility modules within a group cannot be imported by modules from other groups. This is not convention — it is a CI-enforced contract. The existence of this tooling demonstrates that Elastic treats the group/visibility boundary as a load-bearing architectural constraint, not a folder naming preference.

The Modular Monolith is the secondary classification because: (a) Kibana deploys as a single artifact; (b) all plugins share a single Elasticsearch cluster and saved objects schema; (c) the 43 core packages and 130+ platform packages impose physical module boundaries within the monolith without network isolation; and (d) node roles provide horizontal scaling at the process level without service decomposition.

Confidence is 0.94. The 0.06 gap reflects the serverless deployment mode's dynamic plugin allowlisting, which partially approaches a service-level concern, and the nascent node-role system, which could evolve toward a more formally service-based deployment topology in future versions.
