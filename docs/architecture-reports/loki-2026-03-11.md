# loki — Architecture Classification Report
**Date:** 2026-03-11
**Repository:** https://github.com/grafana/loki
**Primary Style(s):** Microservices, Modular Monolith
**Confidence:** 0.95
**Summary:** Loki is a horizontally-scalable log aggregation system built as a single binary that compiles all microservice components together, deployable as a true microservices cluster, a grouped simple-scalable deployment, or a single monolith — all from the same codebase.

## Evidence

- `pkg/loki/modules.go` defines 40+ named modules (Distributor, Ingester, QueryFrontend, QueryScheduler, Querier, Compactor, IndexGateway, Ruler, BloomPlanner, BloomBuilder, BloomGateway, PatternIngester, DataObjConsumer, etc.), each registered individually with explicit dependency graphs.
- `pkg/loki/loki.go` orchestrates all modules via `dskit/modules.Manager` with a `setupModuleManager()` function that declares dependency edges; modules are selectively activated via `-target` CLI flag.
- Three canonical deployment targets are declared: `Read = {QueryFrontend, Querier}`, `Write = {Ingester, Distributor, PatternIngester}`, `Backend = {QueryScheduler, Ruler, Compactor, IndexGateway, BloomPlanner, BloomBuilder, BloomGateway}` — and `All` runs everything.
- `docs/sources/get-started/deployment-modes.md` explicitly documents three modes: Monolithic (`-target=all`), Simple Scalable (read/write/backend split), and Microservices (each component as a separate process).
- `docs/sources/get-started/architecture.md` states: "Grafana Loki has a microservices-based architecture and is designed to run as a horizontally scalable, distributed system."
- Helm chart under `production/helm/loki/templates/` has separate StatefulSet/Deployment templates for: distributor, ingester, querier, query-frontend, query-scheduler, compactor, index-gateway, ruler, bloom-builder, bloom-gateway, bloom-planner, pattern-ingester — confirming independent Kubernetes deployment.
- Inter-component communication uses gRPC (protobuf-defined in `pkg/logproto/*.proto`, `pkg/scheduler/schedulerpb/`, `pkg/lokifrontend/frontend/v1/frontendv1pb/`, `pkg/lokifrontend/frontend/v2/frontendv2pb/`) and consistent hash rings via `dskit/ring` and Memberlist (gossip protocol).
- `pkg/kafka/` and `pkg/ingester/kafka_consumer.go` implement Kafka-backed ingest path: distributors can produce to Kafka partitions, ingesters consume from them — an optional event-driven ingest layer.
- `pkg/logql/log/pipeline.go` defines a composable `Pipeline` / `Stage` interface used throughout log processing; `pkg/querier/queryrange/queryrangebase/` implements a `Middleware` chain (tripperware) for query transformation — splitting, caching, retrying, sharding — following the Pipeline pattern for query execution.
- Consistent hash ring used across Distributor, Ingester, IndexGateway, QueryScheduler, BloomGateway, PatternIngester — each component registers into its own ring for peer discovery and sharding.
- WAL (Write-Ahead Log) in `pkg/ingester/wal/` provides durability guarantees.
- Object storage (S3, GCS, Azure Blob) is the unified backend for both index (TSDB/BoltDB via index shipper) and chunks.

## Architecture Styles

### Microservices (Primary)
Loki's canonical deployment model is microservices: each functional component (Distributor, Ingester, QueryFrontend, QueryScheduler, Querier, Compactor, IndexGateway, Ruler, BloomPlanner, BloomBuilder, BloomGateway, PatternIngester) is independently deployable, independently scalable, and communicates over well-defined gRPC or HTTP interfaces. The Helm chart provides separate Kubernetes workload definitions for each. Components discover each other via distributed hash rings (Memberlist gossip protocol). The official architecture documentation explicitly names this the "Microservices deployment mode."

### Modular Monolith (Primary — co-equal with Microservices)
Every microservice component is compiled into a single binary. The module system (`dskit/modules.Manager`) wires components together via dependency injection and lifecycle management within a single process when running `-target=all`. A blog post in the README is titled "How We Designed Loki to Work Easily Both as Microservices and as Monoliths." The module registration in `pkg/loki/loki.go` and the 40+ `initXxx()` functions in `pkg/loki/modules.go` make this the defining structural characteristic of the codebase.

### Pipeline (Secondary)
Two pipeline-pattern implementations are prominent: (1) `pkg/logql/log/pipeline.go` defines `Pipeline`, `StreamPipeline`, and `Stage` interfaces composing a chain of log-line transformation steps (filtering, label extraction, formatting, pattern matching); (2) `pkg/querier/queryrange/queryrangebase/` implements a `Middleware` / `Handler` chain (the "tripperware") that applies query splitting by time, query sharding, caching, retries, and instrumentation in sequential middleware stages. These are explicit Pipeline pattern implementations for both the write/process path and the query path.

### Event-Driven (Optional/Experimental)
`pkg/kafka/` and `pkg/ingester/kafka_consumer.go` implement an optional Kafka-backed ingest path. The distributor can publish log streams to partitioned Kafka topics; ingesters consume from those partitions asynchronously. This creates an event-driven decoupling between the write acknowledgment and the actual ingestion path, but it is not the default or required mode.

## Quality Attributes

- **Horizontal scalability**: each component scales independently; hash rings rebalance automatically as instances join/leave.
- **High availability**: replication factor (default 3) across ingesters; quorum consistency on reads and writes; WAL-backed durability.
- **Multi-tenancy**: tenant ID is a first-class concept propagated through all components; per-tenant rate limits, overrides, and isolation.
- **Cost efficiency**: label-only indexing (no full-text index); compressed chunk storage in object storage; bloom filters (experimental) for index acceleration.
- **Operational flexibility**: single binary deployable as monolith, grouped targets, or full microservices with no code changes.
- **Observability**: Prometheus metrics, distributed tracing, and structured logging throughout; Loki Canary for end-to-end monitoring.
- **Composability**: modular design with explicit dependency graph allows arbitrary subsets of components to be enabled per binary.
- **Consistency**: Dynamo-style quorum consistency on distributed writes; deduplication in querier for replicated data.
- **Durability**: WAL in ingesters combined with replication protects against data loss during restarts and partial failures.
- **Extensibility**: storage backends are pluggable (S3, GCS, Azure Blob, BoltDB, Cassandra, Bigtable); index formats are pluggable (TSDB, BoltDB).

## Classification Reasoning

Loki's architecture is best described as **Microservices / Modular Monolith** — both styles apply simultaneously and this duality is intentional design, not an accident.

The **Microservices** classification is strongly supported: components have single responsibilities, communicate only via network APIs (gRPC/HTTP), are independently deployable, independently scalable, and maintain no shared in-process state when deployed as separate processes. The Helm chart, documentation, and CLI flags all treat each component as a first-class independently operable service.

The **Modular Monolith** classification is equally valid: the codebase compiles all components into one binary, uses a dependency-injection module manager to wire them, and supports `-target=all` as a first-class production mode (not just for testing). The module boundary enforcement is explicit via the `dskit/modules.Manager` dependency graph. When run as a monolith, inter-component calls are in-process function calls, not network hops.

The **Pipeline** style is a strong secondary pattern: Loki's core value-add — log ingestion, processing, and querying — is implemented as composable pipelines both at the log-processing level (`pkg/logql/log/pipeline.go`) and at the query-execution level (the tripperware middleware chain). These are not incidental — they are the primary mechanism for applying transformations, caching, sharding, and routing to log queries.

The **Event-Driven** style is present but optional: the Kafka ingest path decouples write acknowledgment from ingestion, but since it is experimental/optional and the default path is synchronous gRPC push from distributor to ingester, it is not a primary classification.

The Cortex/Mimir lineage (Loki is architecturally derived from Cortex) explains the specific combination: a shared-nothing distributed system where every node is identical binary-wise but behaviorally differentiated by target flags. This "single binary, multiple personas" pattern is Loki's most distinctive architectural characteristic, sitting at the intersection of Microservices and Modular Monolith.
