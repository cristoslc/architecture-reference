## Summary

Apache Ignite 3 is a **Modular Monolith** at the code-organization level, with a strong secondary **Space-Based** architectural identity that describes the runtime topology, and a **Microkernel** extensibility layer for pluggable storage engines and configuration modules. The repository is a single Gradle multi-project build containing 78+ modules that compile and package into one distribution artifact (`ignite3-db`). At runtime, homogeneous Ignite nodes form a distributed in-memory data grid where partitioned data and compute colocate across a peer cluster — the defining signature of Space-Based architecture. A `java.util.ServiceLoader`-based plugin mechanism allows storage engines (RocksDB, page-memory) and configuration modules to be registered as service providers at runtime, following the Microkernel pattern.

This is a re-classification with deep-analysis replacing the earlier automated discovery. The primary and secondary style ordering has changed: Space-Based is now explicit as a secondary style given the in-memory data grid runtime model. Confidence is raised to 0.95.

## Evidence

### 1. Modular Monolith (primary)

**Single deployable unit**

`packaging/db/build.gradle` assembles a single distribution (`ignite3-db`) bundling all internal modules into one node binary. The Docker Compose in `packaging/docker/docker-compose.yml` confirms this: all three cluster nodes use the *same image* (`apacheignite/ignite:${IGNITE3_VERSION}`) with only a `--node-name` flag differentiating them. There are no distinct service images — nodes are replicas, not independent microservices.

```yaml
services:
  node1:
    << : *ignite-def
    command: --node-name node1
  node2:
    << : *ignite-def
    command: --node-name node2
  node3:
    << : *ignite-def
    command: --node-name node3
```

**Enforced internal module boundaries**

`modules/README.md` documents two explicit constraints:

> "Each module provides an exposed API which should be used by external modules. Not exposed APIs must not be used by external modules."

> "We prohibit cyclic dependencies between modules in order to simplify JIGSAW migration in the future."

`settings.gradle` includes 78+ named subprojects — a classic Gradle Modular Monolith structure. Each subsystem has an `*-api` module (public contract) paired with an implementation module:

- `ignite-network-api` / `ignite-network`
- `ignite-raft-api` / `ignite-raft`
- `ignite-metastorage-api` / `ignite-metastorage`
- `ignite-sql-engine-api` / `ignite-sql-engine`
- `ignite-storage-api` / `ignite-storage-rocksdb`, `ignite-storage-page-memory`
- `ignite-placement-driver-api` / `ignite-placement-driver`
- `ignite-security-api` / `ignite-security`
- `ignite-system-view-api` / `ignite-system-view`

**Runner as composition root**

`ignite-runner` declares 40+ direct `implementation project(...)` dependencies and acts as the single composition root that wires all components together. `modules/runner/README.md` states:

> "Components form an acyclic directed graph that is constructed in topological sort order wrt root."
> "Since metastorage watches can only be added during the component startup, the watch notification order is consistent with the component initialization order."

`IgniteImpl.java` (2274 lines) is the top-level assembly class, wiring all subsystems in a dependency-ordered startup sequence via `LifecycleManager`.

### 2. Space-Based (secondary)

Apache Ignite 3 is explicitly described as "a distributed database for high-performance computing" that provides:

- **In-memory data partitioning**: Data is partitioned across cluster nodes using distribution zones. Partition assignments are managed by the Placement Driver module and replicated via Raft consensus groups.
- **Colocated compute**: The `ignite-compute` module allows user code to execute on the node holding the relevant data partition — the canonical Space-Based compute colocation pattern.
- **Data replication without a shared database**: Each partition is an independent Raft group. There is no central database; data is the grid.
- **ACID distributed transactions**: The transaction protocol (IEP-91) uses 2-phase locking with Raft-replicated write-ahead logs across partition Raft groups — a sophisticated distributed commit without a central coordinator.
- **Distribution Zones**: The `ignite-distribution-zones` module manages how data is spread across nodes and handles rebalancing when nodes join or leave, demonstrating the dynamic processing-unit scaling characteristic of Space-Based architecture.

### 3. Microkernel (tertiary — extensibility layer)

**ServiceLoader-based plugin registration**

`IgniteImpl.java` uses `java.util.ServiceLoader` to discover pluggable components at runtime:

```java
ServiceLoader.load(DataStorageModule.class, serviceProviderClassLoader)
ChannelTypeRegistryProvider.loadByServiceLoader(serviceProviderClassLoader)
new SerializationRegistryServiceLoader(classLoader)
```

**Storage engine plugin interface**

`modules/storage-api/README.md` documents the explicit plugin contract for adding new storage backends:

> "To add a new data storage you need:
> - Add a new module
> - Implement interfaces: `DataStorageModule`, `StorageEngine`, `MvTableStorage`, `MvPartitionStorage`
> - Add services (which are loaded via `java.util.ServiceLoader.load(Class<S>)`):
>   - Implementation of `DataStorageModule`
>   - Implementation of `ConfigurationModule`"

Default implementations (RocksDB via `ignite-storage-rocksdb`, page-memory via `ignite-storage-page-memory`) are registered through this same mechanism. New engines can be contributed by third-party modules without modifying the core.

**Polymorphic configuration extensions**

`ConfigurationModule` uses `@PolymorphicConfigInstance` and `@ConfigurationExtension` annotations (processed at build time by `ignite-configuration-annotation-processor`) to extend the configuration schema at runtime. The `@AutoService(ConfigurationModule.class)` annotation wires implementations to the ServiceLoader mechanism automatically.

## What This Architecture Is Not

**Not Microservices**: A single deployment artifact, homogeneous nodes, data co-located in a shared cluster. Nodes are data-replicas and compute peers, not independently bounded-context services.

**Not Layered**: Module organization is not horizontal presentation/business/data layers. Modules represent vertical technical capabilities (raft, metastorage, sql-engine, transactions, compute), each with its own internal stack.

**Not Event-Driven (primary)**: The Raft consensus protocol and metastorage watches are message-passing, but the primary application interface is synchronous RPC via Netty (Thin Client protocol, JDBC, REST). There is no central event broker.

**Not Domain-Driven Design**: Modules are organized by technical capability, not business domain bounded contexts. No aggregate roots, domain events, or value objects in the DDD sense.

## Key Technologies

- **Consensus**: JRaft fork (Raft protocol implementation)
- **Cluster membership**: ScaleCube (gossip-based membership and failure detection)
- **Networking**: Netty (custom binary protocol for client-server and intra-cluster)
- **Persistent storage**: RocksDB (pluggable); page-memory (in-memory with optional persistence)
- **Configuration**: HOCON (Typesafe Config) with annotation-processed Java schema
- **SQL**: Apache Calcite (query planning and optimization)
- **Build**: Gradle multi-project with annotation processors (`ignite-network-annotation-processor`, `ignite-configuration-annotation-processor`)
- **Multi-language clients**: .NET (C#), C/C++, Python DB-API 2.0, JDBC, ODBC

---

## Notable Strengths

1. **API/implementation decoupling**: Every subsystem exposes a clean `-api` module; internal implementations are not reachable by consumers of the public API.
2. **Acyclic module graph**: Enforced at build time; eliminates circular dependency bugs and enables topological initialization ordering.
3. **Pluggable storage**: The ServiceLoader-based storage engine API allows RocksDB, page-memory, or future engines to be swapped or combined via storage profiles per table.
4. **Raft-native design**: Rather than bolting consensus on top, every stateful subsystem (metastorage, CMG, partition replication) uses dedicated Raft groups — making consistency properties predictable.
5. **Rich client ecosystem**: First-class native clients in four languages plus JDBC/ODBC, with a shared thin-client protocol.

## Notable Gaps

1. **CI/CD artifacts**: Only one GitHub Actions workflow file found (`python_dbapi_wheels.yml`); the main Java CI appears to run on Apache's internal infrastructure (not visible in the repo).
2. **ADR directory**: No `docs/adr/` or architecture decision records found in the repository; design decisions live in Apache JIRA/Confluence wiki (IEP documents).
3. **Documentation completeness**: Several README files contain TODO links (`TODO link to CLI readme`, `TODO link to metastorage watch`), indicating in-flight documentation.
4. **Spring integration limited**: The `ignite-spring` module exists but appears thin — Spring Boot auto-configuration is not fully fleshed out compared to competitors.

## Comparison with Apache Ignite 2.x

The earlier Ignite 2.x catalog entry classified that codebase as **Microkernel (primary), Modular Monolith (secondary)**. Ignite 3 inverts this: the Modular Monolith characteristics (strict Gradle modules, acyclic graph, single artifact, composition root runner) are primary, while Microkernel extensibility is tertiary. The Space-Based runtime identity — present but implicit in Ignite 2.x — is now a first-class secondary classification given the partition-first, colocated-compute design of Ignite 3.
