# logseq — Architecture Classification Report
**Date:** 2026-03-11
**Repository:** https://github.com/logseq/logseq
**Primary Style(s):** Modular Monolith, Microkernel, Layered
**Confidence:** 0.92
**Summary:** Privacy-first knowledge management app built as a layered modular monolith with a formal plugin system (Microkernel), a dedicated SQLite+DataScript web worker for DB isolation, and real-time collaboration via Cloudflare Durable Objects.

## Evidence

- **`CODEBASE_OVERVIEW.md`** explicitly describes the three-tier layout: `frontend/components/` (UI), `frontend/handler/` ("system component like code"), and `frontend/db/` (DataScript queries).
- **`deps/` directory** contains eight isolated local libraries (`common`, `db`, `db-sync`, `graph-parser`, `outliner`, `publish`, `publishing`, `shui`), each with its own `deps.edn`, enforcing hard module boundaries within the single deployable.
- **`shadow-cljs.edn`** defines separate build targets — `:app` (browser frontend), `:db-worker` (SQLite web worker, `web-worker: true`), `:electron` (desktop), `:mobile`, `:publishing` — confirming multi-target but single-codebase structure.
- **`src/main/logseq/api/`** contains the full plugin API surface (`app.cljs`, `block.cljs`, `db.cljs`, `editor.cljs`, `plugin.cljs`), with `handler/plugin.cljs` managing plugin lifecycle and hook installation — a textbook microkernel plugin registry.
- **`frontend/worker/db_worker.cljs`** implements a complete off-main-thread database worker using `comlink` for RPC, exposing `def-thread-api` endpoints consumed through `state/<invoke-db-worker` in the main thread.
- **`frontend/worker/pipeline.cljs`** defines a transaction post-processing pipeline (`transact-pipeline`, `invoke-hooks`) that runs after every DataScript transaction to rebuild refs, apply tag templates, validate journal invariants, and dispatch affected query keys.
- **`frontend/handler/events.cljs`** implements a core.async channel (`system/events`, capacity 1000) with a `go-loop` dispatcher and 43+ `defmulti handle` methods — an internal event bus decoupling producers from consumers within the main thread.
- **`deps/db-sync/src/logseq/db_sync/worker.cljs`** deploys a `SyncDO` Cloudflare Durable Object with WebSocket handling for real-time multi-device sync (RTC), showing a clear separation between the client application and the cloud sync backend.
- **`src/main/frontend/inference_worker/`** runs a third isolated worker (`InferenceWorker`) for on-device text embedding and HNSW vector search, exposed via `comlink`.
- **`deps.edn`** references `logseq/datascript` (fork), `missionary`, `datascript-transit`, `malli` — all integral to the reactive query-subscription model used across the app.

## Architecture Styles

### Primary: Modular Monolith
All application code compiles into a single deployable artifact per target platform. The eight `deps/` libraries are local-root Clojure dependencies, not independent services. Module boundaries are enforced structurally: `deps/outliner` depends on `deps/db`; `deps/graph-parser` depends on `deps/common`; there is no circular dependency. The `shadow-cljs.edn` build targets (`:app`, `:electron`, `:mobile`) all share the same source paths — they are different packaging of the same monolith, not independent deployables.

### Secondary: Microkernel
The plugin system is a first-class architectural concern. `src/main/logseq/api/` exposes 100+ ClojureScript functions organized by domain. `frontend/handler/plugin.cljs` manages the plugin registry: loading, lifecycle, hook registration, and cross-boundary event dispatch. Plugins communicate with the core via a defined API, not by importing internal namespaces. The `frontend.config/ENABLE-PLUGINS` build flag and iFrame-based plugin sandboxing (via Postmate in `libs/`) reinforce the plug-in isolation model.

### Secondary: Layered
The frontend follows a strict three-tier horizontal layering:
1. **Presentation** — `frontend/components/` (80+ Rum/React components, pure rendering)
2. **Application/Handler** — `frontend/handler/` (editor, page, query, plugin, sync, export handlers)
3. **Data** — `frontend/db/` + `deps/db/` + `deps/outliner/` (DataScript in-memory DB, Datalog queries, outliner transactions)

Dependencies flow strictly downward; components call handlers, handlers call DB APIs, DB layer has no UI imports.

## Quality Attributes

- **Privacy / Local-first**: Data stored on device in SQLite (OPFS in browser, file system on desktop); optional cloud sync (RTC) is additive, not required.
- **Extensibility**: Microkernel plugin architecture allows third-party extensions without modifying core.
- **Modularity**: Eight independently versioned `deps/` libraries enforce internal API surfaces.
- **Performance**: Dedicated web workers for DB I/O and ML inference prevent main-thread blocking.
- **Portability**: Single codebase targets web, Electron desktop, iOS (Capacitor), and Android; publishing target produces static read-only exports.
- **Consistency / Integrity**: Post-transaction pipeline enforces referential integrity (block refs, journal invariants, built-in property protection) on every DataScript commit.
- **Observability**: `frontend/modules/instrumentation/` integrates PostHog and Sentry; structured logging via `lambdaisland/glogi`.
- **Queryability**: DataScript + Datalog provides a highly expressive in-process query engine accessible to both core code and plugins.
- **Collaboration**: RTC (Real-Time Collaboration) via Cloudflare Durable Objects enables multi-device and multi-user sync.
- **Testability**: Separate `:test` shadow-cljs target with node-test runner; `clj-e2e/` for browser end-to-end tests.

## Classification Reasoning

Logseq is a **knowledge management application** that has undergone a major architectural refactoring (documented in the codebase) from a file-based graph model to a DB graph model backed by SQLite and DataScript. The architecture is best described as a **Modular Monolith** as its primary topology, with **Microkernel** and **Layered** as co-equal secondary styles.

**Why Modular Monolith (primary):** The entire application — editor, database layer, plugin host, sync client, vector search, publishing — ships as a single build artifact per platform. The `deps/` directory creates internally-enforced module seams (each module has its own `deps.edn` with explicit declared dependencies), but these are not independently deployable services. The architecture deliberately colocates all concerns in one codebase to enable local-first operation and cross-cutting optimizations (e.g., the outliner can call DataScript directly in the same process rather than over a network).

**Why Microkernel (secondary):** The plugin system is a genuine architectural concern, not a feature. The separation between core (stable kernel: editor, DB, outliner) and plugins (independently loaded, sandboxed extensions) matches the Richards & Ford definition of microkernel. The `logseq/api` namespace serves as the explicit plugin API contract; `handler/plugin.cljs` is the plugin manager; plugins run in iFrames or shadow DOMs with postMessage-based RPC.

**Why Layered (secondary):** The `CODEBASE_OVERVIEW.md` names this explicitly. The component→handler→DB flow is consistent and enforced. The `frontend/db/react.cljs` reactive query subscription system ties DataScript transactions to React re-renders through a query-key cache, forming a clean data-flow layer boundary.

**Why NOT Event-Driven (primary):** The internal core.async event channel (`system/events`) is a coordination mechanism within the single process, not an architectural event bus connecting independent components. The primary data-flow is synchronous: UI action → handler → DataScript transaction → reactive query refresh → React re-render. The channel dispatches cross-cutting lifecycle events (graph switch, page rename, plugin hooks) but is not the dominant communication pattern.

**Why NOT Microservices:** The `deps/db-sync/` cloud sync component could appear service-like, but it is a Cloudflare Worker backing a client application — the client and sync backend do not form an ensemble of peer microservices. The sync layer is an optional cloud extension of the local-first app, not the deployment unit.

**Distinguishing features of this classification:**
- The three-worker architecture (main thread + DB worker + inference worker) is a performance optimization within the monolith, not a service decomposition. Workers communicate via `comlink` RPC within the same browsing context.
- The post-transaction pipeline in `worker/pipeline.cljs` shows domain integrity logic (tag templates, ref rebuilding, journal protection) co-located with the transaction layer — characteristic of a modular monolith rather than distributed services.
- The DB graph refactoring (from file-based Markdown to SQLite+DataScript) is still in progress; the codebase contains both legacy file-format paths and the new DB-graph paths, adding internal complexity but not changing the primary topology.
