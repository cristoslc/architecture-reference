# Architecture Report: Synapse

**Date:** 2026-03-11
**Source:** https://github.com/element-hq/synapse
**Classification Method:** deep-analysis
**Model:** claude-sonnet-4-6
**Confidence:** 0.91

## Summary

Synapse is the reference Matrix homeserver, written in Python/Twisted with performance-critical modules in Rust. Its architecture is a **Layered Modular Monolith** as the default deployment, with an optional **Service-Based** horizontal scaling mode powered by a custom replication protocol over Redis pub/sub. A well-defined plugin/module API constitutes a **Microkernel** extension point. The federation protocol — the defining characteristic of Matrix — gives the system a strongly federated distribution model: every homeserver is a peer in a global decentralized network. All three architecture styles are active simultaneously in a scaled production deployment.

## Process Topology

In monolith mode, a single process runs everything. In worker mode, multiple specialized processes are deployed from the same codebase:

| Process | Entry Point | Responsibility |
|---|---|---|
| Main process | `synapse/app/homeserver.py` | Event persistence, auth writes, user directory maintenance, admin API |
| Generic worker | `synapse/app/generic_worker.py` | Client sync, federation reads, push, media, user directory search |
| Federation sender | `synapse/app/federation_sender.py` | Outbound federation transaction delivery |
| Media repository | `synapse/app/media_repository.py` | Media upload/download/thumbnail |
| Pusher | `synapse/app/pusher.py` | Push notifications (Sygnal / email) |

All processes share a single PostgreSQL database. A Redis pub/sub channel carries the internal replication stream between them. All processes also connect to Redis as a shared cache. An HTTP reverse proxy (nginx/haproxy) routes inbound requests to the appropriate worker.

## Primary Style Evidence: Layered

The layered architecture is explicit in the source tree and the `docs/architecture.md` diagram. Every process — whether the main homeserver or a generic worker — runs the same vertical stack:

### Layer 1 — REST/HTTP Layer (`synapse/rest/`)

HTTP servlets in `synapse/rest/client/`, `synapse/rest/admin/`, and `synapse/federation/transport/server.py` handle request parsing, authentication, and response serialization. Each servlet maps to a URL pattern and delegates to a handler. The layer is strictly input-only: no business logic lives here.

```
synapse/rest/
  client/         — Matrix Client-Server API (r0/v3/unstable)
  admin/          — Synapse Admin API
  media/          — Media upload/download endpoints
  key/            — Server key publication
  well_known.py   — .well-known/matrix/server discovery
```

### Layer 2 — Handler Layer (`synapse/handlers/`)

~50 handler modules implement all business logic. Each handler is injected with the `HomeServer` service locator and can call any other handler or storage component. Representative handlers:

- `message.py` — `EventCreationHandler`, `MessageHandler` (event creation pipeline)
- `federation.py` + `federation_event.py` — inbound federation event processing
- `room_member.py` — join/leave/invite/kick logic
- `sync.py` — `/sync` response assembly
- `sliding_sync/__init__.py` — MSC3575 Sliding Sync handler
- `state_deltas.py` — state resolution delta computation
- `auth.py` — login flows, token management
- `e2e_keys.py` — end-to-end encryption key management

The handler layer calls `synapse/state/` for state resolution (v1 and v2 algorithms) before persisting events.

### Layer 3 — Storage Layer (`synapse/storage/`)

`SQLBaseStore` is the base class for all store classes. Stores are organized by domain area under `synapse/storage/databases/main/`:

```
events.py, events_worker.py   — event reads/writes
roommember.py                 — room membership queries
devices.py, end_to_end_keys.py — device/key storage
push_rule.py, pusher.py       — push rule/pusher storage
sliding_sync.py               — sliding sync token tracking
```

A separate `synapse/storage/databases/state/` subtree holds state group management. Caching (`@cached`, `@cachedList` decorators) is applied heavily at the store layer. The `process_replication_rows` hook on every store is the integration point for replication: when a worker receives an RDATA command from Redis, each store invalidates affected caches.

### The HomeServer Service Locator (`synapse/server.py`)

`HomeServer` is a 300+ line class that lazily instantiates every handler, store, federation component, and HTTP client via `@cached_get_*` factory methods. This is the central dependency registry: it does not enforce layering, but the convention is that `rest/` → `handlers/` → `storage/` flows downward and never reverses.

## Primary Style Evidence: Service-Based

The worker architecture is documented explicitly in `docs/workers.md` and `docs/replication.md`. The key characteristics that qualify this as Service-Based (not Microservices):

- **Shared database:** All workers operate against the same PostgreSQL instance. Worker-specific stores (`*WorkerStore` variants in `synapse/app/generic_worker.py`) are mixin subsets of the full `DataStore`, not independent schemas.
- **Shared codebase:** Workers run the same Python package with different entry points. There is no separate repository per service.
- **Coarse-grained decomposition:** ~5 service types, not dozens. The primary split is between the main process (event writing) and generic workers (reading and secondary writes).
- **Optional:** The documentation explicitly states "For small instances it is recommended to run Synapse in the default monolith mode."

### Inter-Process Communication

Two channels carry inter-process traffic:

**Redis pub/sub (replication streams):** All processes subscribe to a Redis channel. The main process (or designated stream writer) publishes `RDATA` commands for named streams:

```
EventsStream, PresenceStream, TypingStream, ReceiptsStream,
AccountDataStream, ToDeviceStream, DeviceListsStream,
PushRulesStream, FederationStream, CachesStream, ...
```

Workers consume these streams to invalidate caches (`process_replication_rows`) and to push updates to connected clients via `synapse/notifier.py`.

**HTTP replication (`synapse/replication/http/`):** For operations requiring a synchronous reply — sending events, performing state resolution, updating membership — workers make authenticated HTTP requests to the main process's internal replication listener (default port 9093). This listener is never exposed publicly. Examples:

```
synapse/replication/http/send_events.py    — event persistence
synapse/replication/http/membership.py    — join/leave
synapse/replication/http/presence.py      — presence updates
synapse/replication/http/federation.py    — federation EDU forwarding
```

### Stream Writer Assignment

Configuration allows assigning write ownership of specific streams to named workers (`instance_map` in homeserver config). For example, the `to_device` stream can be written by a dedicated `to_device_worker` rather than the main process. This provides targeted write scaling without full Microservices decomposition.

## Secondary Style Evidence: Microkernel

`synapse/module_api/__init__.py` defines the `ModuleApi` class, a stable, versioned interface for external Python modules. The plugin surface includes:

- `register_web_resource()` — add HTTP endpoints served by Synapse
- `register_spam_checker_callbacks()` — intercept event creation
- `register_presence_router_callbacks()` — custom presence routing
- `register_password_auth_provider_callbacks()` — custom authentication
- `register_third_party_rules_callbacks()` — custom event validation rules
- `register_account_validity_callbacks()` — user account expiry logic
- `register_media_repository_callbacks()` — media upload quota enforcement
- `register_ratelimit_callbacks()` — per-user ratelimit overrides
- Cache management APIs for module-owned data

Modules are loaded from YAML configuration:

```yaml
modules:
  - module: my_module.MyClass
    config:
      setting: value
```

Callback registries in `synapse/module_api/callbacks/` hold lists of callables contributed by all registered modules, invoked at defined hook points in the handler pipeline. This is the classic Microkernel plugin-point pattern: a stable core with extension points, and extensions that do not require modifying core code.

## Federation Protocol Layer (`synapse/federation/`)

The Matrix federation protocol is implemented as a peer-to-peer HTTP layer between homeservers:

- `FederationServer` (`federation/federation_server.py`) — receives inbound federation transactions (PDUs and EDUs) from remote homeservers
- `FederationClient` (`federation/federation_client.py`) — sends outbound requests: event backfill, state queries, key exchange
- `TransportLayerServer` / `TransportLayerClient` — HTTP transport with signature verification
- `FederationSender` (`federation/sender/`) — queues and delivers outbound PDUs to remote servers, including retry logic and per-destination backoff
- `FederationRemoteSendQueue` — buffers outbound federation when a dedicated federation_sender worker is in use

Every event on a federated room is a node in a DAG (Directed Acyclic Graph), with `auth_events` and `prev_events` edges linking it to prior room history. State resolution (`synapse/state/v1.py`, `synapse/state/v2.py`) merges divergent DAG branches using the Matrix State Resolution algorithms.

## Rust Extension Layer (`rust/`)

Performance-critical components are implemented in Rust and exposed to Python via PyO3 bindings (`synapse/synapse_rust/`):

| Module | Function |
|---|---|
| `push/` | Push rule evaluation (matching rules against events at high throughput) |
| `events/` | Event internal ID generation and handling |
| `acl/` | Server ACL (access control list) matching |
| `http_client/` | HTTP client primitives |
| `identifier.rs` | Matrix identifier parsing and validation |
| `segmenter.rs` | Text segmentation (for search) |

Build integrity is enforced: `get_rust_file_digest()` returns a hash of compiled source files; Python checks this on startup to detect stale builds.

## What Was Rejected and Why

**Not Microservices:** All workers share one PostgreSQL database with no per-service schema. Workers are optional decompositions of a monolith, not independently developed services with independent data ownership. There is no API gateway or service mesh.

**Not Event-Driven (primary):** Redis pub/sub is used for internal cache invalidation (replication), not for application-level event routing between independent bounded contexts. The primary inter-worker coordination for writes uses synchronous HTTP, not async messaging. The Matrix protocol domain model uses events, but this is a domain concept, not an architectural style applied to the system's own structure.

**Not Modular Monolith:** While the monolith has clear internal organization, module boundaries are not enforced by the language or build system. The `HomeServer` service locator allows any component to reach any other component. Workers compose arbitrary subsets of stores via Python mixins, not through enforced module contracts.

**CQRS is a qualifier, not a style:** The worker split between event-writing (main process) and event-reading (generic workers) exhibits a read/write separation pattern. The stream writer assignment mechanism lets specific streams have designated writers. This is genuine CQRS at the infrastructure level, but it is implemented as an operational concern within the Service-Based worker topology, not as a domain-level command/query separation with separate models.

## Quality Attributes

### Scalability
- Worker mode enables horizontal scaling of read-heavy workloads (sync, federation reads, media) independently of the write path
- Redis shared cache reduces database read pressure across workers
- Stream writer assignment allows targeted write-side scaling for specific streams
- The `docs/workers.md` architectural diagram shows a validated production topology used by matrix.org

### Federation and Interoperability
- Full Matrix specification compliance; interoperates with all Matrix homeservers (Conduit, Dendrite, etc.)
- Implements State Resolution v1 (room versions 1–2) and v2 (all subsequent room versions)
- Implements Matrix federation transport including HTTP Signatures, key exchange, backfill, and state queries
- Application Service (AS) bridge protocol allows bots and bridges to connect external networks (IRC, Telegram, XMPP, etc.) via `synapse/appservice/`

### Observability
- Prometheus metrics throughout (`prometheus_client`), with per-stream and per-worker labeling
- OpenTracing/Jaeger integration (`synapse/logging/opentracing.py`)
- Structured logging with request context propagation (`synapse/logging/context.py`)
- Background process metrics wrapper (`synapse/metrics/background_process_metrics.py`)

### Extensibility
- Module API provides stable extension points without forking core
- Application Services allow external processes to hook into the event stream
- Experimental features flag (`synapse/storage/databases/main/experimental_features.py`) supports per-user feature rollout

### Resilience
- `ResponseCache` deduplicates in-flight identical requests (prevents thundering herd on federation)
- `Linearizer` prevents concurrent processing of the same room's events
- Worker processes reconnect to Redis automatically on disconnect
- Federation sender implements per-destination backoff and retry

### Security
- HTTP Signatures on all federation traffic (signed with server's ed25519 key)
- End-to-end encryption key management (`synapse/handlers/e2e_keys.py`)
- Server ACLs enforced via Rust-accelerated matcher
- Rate limiting at the REST layer (`synapse/api/ratelimiting.py`)
- MAS (Matrix Authentication Service) delegation support for OIDC-based auth
