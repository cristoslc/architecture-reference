# Architecture Report: Supabase Realtime

**Date:** 2026-03-11
**Version analyzed:** 2.78.12
**Source:** https://github.com/supabase/realtime
**Language:** Elixir (Phoenix Framework)
**Classification:** Event-Driven, Modular Monolith

---

## Executive Summary

Supabase Realtime is an Elixir/OTP application built on the Phoenix Framework that provides three real-time communication primitives: Broadcast (ephemeral pub/sub), Presence (shared state synchronization), and Postgres Changes (CDC-based database event streaming). The system is a single deployable unit with clear internal module boundaries, deployed as a clustered set of BEAM nodes coordinated via the `syn` distributed process registry. Its core architectural signature is **Event-Driven**: all message flow—from Postgres WAL events through the CDC pipeline to WebSocket subscribers—is driven by asynchronous events and Phoenix PubSub fanout. This sits within a **Modular Monolith** structural envelope: three well-bounded subsystems (realtime core, realtime_web, extensions) share one OTP application and one deployment artifact with no inter-service network calls.

## Architectural Evidence

### 1. Event-Driven Core

The entire message delivery path is event-driven rather than request/response:

**PostgreSQL WAL to WebSocket pipeline:**
- `Extensions.PostgresCdcRls.ReplicationPoller` polls a logical replication slot, emits `INSERT`/`UPDATE`/`DELETE` change records.
- Changes are dispatched via `TenantBroadcaster.pubsub_direct_broadcast/5` or `pubsub_broadcast/5`, which call `Phoenix.PubSub.local_broadcast` or `PubSub.broadcast` depending on the cluster topology.
- `RealtimeWeb.RealtimeChannel` subscribes to PubSub topics and forwards events to WebSocket transports via Phoenix Channel's fastlane mechanism (`subscriber_fastlane` metadata), bypassing the channel process entirely for throughput.

**Broadcast Changes path (v2 feature):**
- `Realtime.Tenants.ReplicationConnection` uses `Postgrex.ReplicationConnection` (streaming logical replication) to capture inserts into the `realtime.messages` table.
- On `handle_data/2` for a WAL write, `BatchBroadcast.broadcast/4` fans out to all subscribed WebSocket clients via PubSub.

**Inter-node event routing:**
- Cluster coordination uses `syn` (distributed Erlang process registry) and `libcluster`/`libcluster_postgres` for node discovery.
- `TenantBroadcaster` routes broadcasts to the correct BEAM node using either `Phoenix.PubSub.direct_broadcast` (gen_rpc adapter) or `Realtime.GenRpc.multicast` depending on the configured PubSub adapter.
- Node selection for tenant CDC connections uses a load-aware, time-bucketed deterministic algorithm in `Realtime.Nodes.launch_node/3`.

**Async message handling throughout:**
- All channel lifecycle events (`:postgres_subscribe`, `:confirm_token`, `:sync_presence`, `:check_connected_users`, `:recover_replication_connection`) are sent to GenServer/Channel processes via `Process.send_after/3`.
- No synchronous blocking calls in the hot path.

### 2. Modular Monolith Structure

Despite the event-driven runtime behavior, the codebase is organized as a modular monolith:

**Module boundaries:**
- `Realtime.*` — domain logic: tenant management, database connections, rate limiting, CDC abstraction, RLS authorization, telemetry.
- `RealtimeWeb.*` — Phoenix web layer: WebSocket channels, REST controllers, plugs, presence.
- `Extensions.*` — pluggable CDC drivers implementing the `Realtime.PostgresCdc` behaviour callback interface.

**Single OTP application:**
- `Realtime.Application.start/2` boots all subsystems in one supervision tree: database repos, cluster supervisor, PubSub, Cachex, registries, partition supervisors for migrations/connections/CDC, the Phoenix endpoint, and Presence.
- There is no service mesh, no inter-service HTTP calls, and no message broker. The only external dependencies at runtime are PostgreSQL (tenant databases and the internal repo) and the BEAM cluster.

**Extension behaviour pattern (Microkernel-adjacent):**
The `Realtime.PostgresCdc` module defines a behaviour with four callbacks (`handle_connect`, `handle_after_connect`, `handle_subscribe`, `handle_stop`). Extensions implement this interface and are registered at compile time via application config. This enables swappable CDC drivers without changing the core, but it is not a Microkernel architecture because the core is not a thin dispatcher—it carries full business logic.

### 3. Multi-Tenancy and Rate Limiting

- Each tenant gets its own isolated supervision subtree: `Connect.DynamicSupervisor`, `ReplicationConnection.DynamicSupervisor`, and `Migrations.DynamicSupervisor` use `PartitionSupervisor` with configurable partition slots.
- `Realtime.Tenants.Connect` manages per-tenant Postgrex connection pools and replication connections, shutting them down when no users are connected (`connected_users_bucket` sliding window).
- Rate limiting is implemented via `Realtime.RateCounter` (Cachex-backed sliding window counters) applied at joins per second, events per second, DB connection attempts, subscription errors, and authorization checks.

### 4. Authorization via PostgreSQL RLS

Authorization is enforced by running policy checks directly against the tenant's PostgreSQL database using Row Level Security:
- `Realtime.Tenants.Authorization.get_read_authorizations/4` and `get_write_authorizations/4` open a transaction, set session-scoped JWT claims via `set_config`, attempt inserts/selects, and use RLS policy outcomes to build a `Policies` struct.
- This pattern is deliberately not a separate auth service; it piggybacks on Postgres as the authorization engine.

### 5. Clustering and Regional Affinity

- `Realtime.Nodes` implements region-aware node selection using `:syn.members(RegionNodes, region)` and a time-bucketed two-candidate random selection with CPU load comparison.
- `Realtime.Tenants.Rebalancer` checks whether a tenant's database connection is on the optimal node and shuts it down for rebalancing if not.
- `GenRpc` (custom fork of `gen_rpc`) provides efficient Erlang term RPC across cluster nodes, used for cross-node database operations and authorization checks.

### 6. Observability

- `Realtime.PromEx` and `Realtime.TenantPromEx` export per-tenant and system-level Prometheus metrics.
- OpenTelemetry is set up for Cowboy, Phoenix, and Ecto spans.
- Logflare integration via `logflare_logger_backend` for structured log shipping.

## Architecture Style Classification

| Style | Verdict | Notes |
|---|---|---|
| **Event-Driven** | **Primary** | All message delivery uses PubSub fanout; CDC pipeline is event-stream-based; async GenServer messaging throughout |
| **Modular Monolith** | **Primary** | Single OTP app, three clear module namespaces, one deployment unit, explicit behaviour-based extension interface |
| Microkernel | Considered, rejected | `PostgresCdc` behaviour resembles plugin dispatch, but the core carries full business logic; not a thin kernel |
| Layered | Considered, rejected | Web/domain/infra layers exist but are not the dominant organizing principle; the event bus crosses them |
| Pipeline | Partial evidence | The WAL-to-WebSocket path is a logical pipeline, but it is embedded within the event-driven architecture, not a discrete pipeline topology |

**Final classification: Event-Driven, Modular Monolith**

The event-driven style is dominant at runtime: every significant operation—CDC event dispatch, broadcast fanout, presence synchronization, token revalidation—is driven by asynchronous messages over Phoenix PubSub and OTP message passing. The modular monolith is the structural reality: one deployable binary with bounded internal modules and no microservice decomposition.

## Quality Attributes

| Attribute | Assessment |
|---|---|
| **Scalability** | Horizontal: Erlang clustering with region-aware tenant placement and load-aware node selection. Vertical: per-tenant partition supervisors |
| **Fault Tolerance** | OTP supervision trees with restart strategies; replication connection recovery with exponential backoff (up to 2h window); CDC slot retry logic |
| **Low Latency** | Fastlane PubSub bypasses channel process; gen_rpc for cross-node calls; Cachex for tenant cache; Rust NIFs for hot paths |
| **Multi-tenancy** | Per-tenant supervision subtrees, isolated DB connection pools, RLS-based authorization, per-tenant rate counters |
| **Observability** | Prometheus/PromEx, OpenTelemetry, Logflare, Phoenix LiveDashboard, custom process dump page |
| **Extensibility** | PostgresCdc behaviour allows CDC driver plugins without core changes |
| **Security** | JWT validation on every channel join and re-validated periodically; PostgreSQL RLS for authorization; token blocklist support |
