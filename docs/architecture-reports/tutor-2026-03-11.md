# tutor — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/overhangio/tutor
**Classification:** Microkernel
**Confidence:** 0.93

## Summary

Tutor is the official Docker-based Open edX distribution tool — a Python CLI application that orchestrates the deployment, configuration, and lifecycle management of Open edX instances across local Docker Compose and Kubernetes environments. Its internal architecture is a textbook Microkernel (plugin) system: a minimal, hook-aware core is deliberately kept thin, while all customization and extension surface is owned by first- and third-party plugins that register callbacks against a typed Actions/Filters hook catalog. Tutor itself is a single Python package, not a collection of services; the microservices it deploys (LMS, CMS, Celery workers, Redis, Caddy) are the *output* of its rendering pipeline, not the architecture of Tutor itself.

### Key Architectural Files

- `tutor/core/hooks/actions.py`: `Action[T]` generic class. Callbacks registered via `@action.add(priority=N)` decorator. `do(*args)` fires all callbacks in priority+FIFO order within a process. `do_from_context(context, *args)` fires only callbacks registered under a specific context. Maintains a `WeakSet` of all instances for bulk clearing.
- `tutor/core/hooks/filters.py`: `Filter[T1, T2]` generic class. Implements a callback chain where the return value of each callback is passed as the first argument to the next — a typed pipe-and-filter within a single process. Convenience methods `add_item()`, `add_items()`, `iterate()` handle list-typed filters.
- `tutor/core/hooks/contexts.py`: `Context` context manager pushes/pops names onto `Context.CURRENT`. `Contextualized` base class snapshots active contexts at hook registration time; `is_in_context()` enables targeted plugin unloading without touching other hooks.
- `tutor/hooks/catalog.py`: Defines the public plugin API: Actions (`CORE_READY`, `CONFIG_LOADED`, `CONFIG_INTERACTIVE`, `COMPOSE_PROJECT_STARTED`, `PLUGIN_LOADED`, `PLUGIN_UNLOADED`, `PLUGINS_LOADED`, `DO_JOB`, `PROJECT_ROOT_READY`) and Filters (`CLI_COMMANDS`, `CLI_DO_COMMANDS`, `CLI_DO_INIT_TASKS`, `CONFIG_DEFAULTS`, `CONFIG_OVERRIDES`, `CONFIG_UNIQUE`, `CONFIG_USER`, `ENV_PATCHES`, `ENV_TEMPLATE_ROOTS`, `ENV_TEMPLATE_TARGETS`, `ENV_TEMPLATE_VARIABLES`, `ENV_TEMPLATE_FILTERS`, `IMAGES_BUILD`, `IMAGES_PULL`, `IMAGES_PUSH`, `COMPOSE_MOUNTS`, `MOUNTED_DIRECTORIES`, `PLUGINS_INFO`, `PLUGINS_INSTALLED`, `PLUGINS_LOADED`, `PLUGIN_INDEXES`, `LMS_WORKER_COMMAND`, `CMS_WORKER_COMMAND`, and more).
- `tutor/plugins/v1.py`: Two discovery paths — (1) Python packages declaring a `tutor.plugin.v1` setuptools entrypoint (discovered via `importlib_metadata`), (2) standalone `.py` files in the plugins root. Both paths dynamically import modules when a plugin is enabled. On unload, the module is evicted from `sys.modules` to allow clean re-enable within the same process.
- `tutor/plugins/__init__.py`: `load()` activates a plugin by entering nested `Context` scopes (`PLUGINS` + plugin-specific), then importing the plugin module. All plugin hooks are declared within this context, enabling full cleanup on `unload()`. `load_all()` loads in alphabetical order with per-plugin error isolation.
- `tutor/config.py`: Three-tier layered configuration: user `config.yml` → `base.yml` template → `defaults.yml`. Plugins inject settings via `CONFIG_DEFAULTS`, `CONFIG_UNIQUE`, `CONFIG_OVERRIDES` filters. `TUTOR_*` environment variables override any file-based value at runtime.
- `tutor/env.py`: `JinjaEnvironment` wraps `jinja2.FileSystemLoader` with plugin-contributed template roots (via `ENV_TEMPLATE_ROOTS` filter). `Renderer` applies config values, custom Jinja2 filters (from `ENV_TEMPLATE_FILTERS`), and renders entire directory trees. The `patch()` function in templates calls the `ENV_PATCHES` filter, allowing plugins to inject content at named splice points (e.g., `{{ patch("caddyfile") }}`).
- `tutor/tasks.py`: `BaseTaskRunner.run_task_from_template()` loads a bash script from a template file, renders it with config variables, then delegates to the abstract `run_task()`. `ComposeTaskRunner` and `K8sTaskRunner` implement `run_task()` by executing the rendered script inside the appropriate container runtime.
- `tutor/commands/compose.py`: `ComposeTaskRunner` wraps `docker compose` commands. Fires `COMPOSE_PROJECT_STARTED` action on start/up/restart/run. Service aliases (`openedx` → `lms, cms`) resolved at invocation time.
- `tutor/commands/k8s.py`: `K8sClients` singleton wraps the `kubernetes` Python client. `K8sTaskRunner` manages job submission, polling, and serialization via `kubectl apply --kustomize`. Declarative manifest rendering uses the same `env.py` pipeline as Docker Compose.
- `tutor/commands/cli.py`: `TutorCli(click.Group)` overrides `list_commands()` and `get_command()` to lazily iterate `hooks.Filters.CLI_COMMANDS`. Entry point: `hooks.Actions.CORE_READY.do()` → `cli()`. Plugins add subcommands through the filter without modifying core CLI files.

### Patterns Found

**Microkernel (primary):** The entire design is built around a thin, deliberately minimal core that exposes a typed hook catalog as the extension contract. Every functional concern beyond the hook framework itself — CLI commands, configuration values, template roots, Docker build parameters, Kubernetes resource patches, initialization jobs, worker command overrides, plugin indexes — is expressed as a registration against an Action or Filter. The plugin v1 API provides two standard discovery paths (entrypoints, module files) and a versioned base that has evolved from v0 to v1 with backward compatibility maintained. Context-scoped hooks enable clean plugin lifecycle management (load, unload, re-enable) without process restart. This is the defining characteristic of a microkernel: a minimal, stable kernel that delegates all feature surface to plugins through a well-specified extension protocol.

**Layered (secondary, internal):** Within the core, there is a clear functional layering: CLI commands layer → task runner layer → Docker/Kubernetes runtime layer, with configuration flowing downward. The config system itself is layered (user overrides base overrides defaults). This layering is an implementation detail of the core, subordinate to the microkernel extension model that cuts across all layers.

**Pipeline (implementation mechanism):** The `Filter` hook type implements a synchronous pipeline within a single process: each callback receives the previous callback's output as its first argument. The template rendering system (Jinja2 + `ENV_PATCHES`) is also pipeline-shaped: config loading → template resolution → Jinja2 rendering → file output. These are internal data-flow patterns, not a defining architectural style for the system as a whole.

## Architecture Styles Identified

### Microkernel (Primary, Sole Style)

All structural evidence converges on a single style. Tutor's core is intentionally minimal: the hooks framework (`actions.py`, `filters.py`, `contexts.py`), the config loader, the Jinja2 renderer, and the task runner base classes constitute essentially the entire "kernel." Every feature — including deployment targets (Docker Compose, Kubernetes), the CLI command tree, configuration defaults, Docker image builds, template content, and Open edX service worker commands — is either a built-in plugin or registered through the hook catalog by an external plugin.

The v0/v1 versioned plugin API, formal entrypoint-based discovery, context-scoped hook isolation, alphabetically ordered loading with per-plugin error isolation, and the `sys.modules` cache eviction on unload are all characteristic of a mature, deliberately designed microkernel plugin system, not incidental extensibility.

The project's own documentation states the design goal explicitly: "users should not have to fork Tutor to customise their deployments." This is the microkernel guarantee — the kernel never needs to change; all variation is handled by plugins.

### What This Architecture is Not

- **Not Microservices:** Tutor is a single Python package (one `pyproject.toml`, one `tutor` entry point). It deploys a multi-service Open edX stack, but Tutor itself is not that stack.
- **Not Modular Monolith:** There are no enforced module boundaries with restricted imports between peer modules. The plugin system is hierarchical (core → plugins), not peer-module isolation.
- **Not Event-Driven:** Actions and Filters are synchronous, in-process callback chains. There is no message broker, no async event bus, no eventual consistency between decoupled services. The hooks fire and return before the next line executes.
- **Not Layered (at the whole-system level):** The plugin system cuts across all functional layers — a plugin can simultaneously add CLI commands (presentation), configuration keys (data), template content (output), and initialization jobs (execution). This cross-cutting is antithetical to strict layering.
- **Not Service-Based or Space-Based:** Tutor is a single process CLI tool.

## Quality Attributes

- **Extensibility:** The microkernel design makes this the strongest attribute. 40+ hook extension points; versioned plugin API; plugin index with curated and contrib catalogs.
- **Maintainability:** Thin, well-typed core. Context-scoped hooks enable surgical plugin removal. Legacy plugin API (v0) maintained alongside v1 for backward compatibility.
- **Portability:** Same configuration and template pipeline targets Docker Compose and Kubernetes with minimal divergence.
- **Developer Experience:** `tutor local launch` as a single command for full Open edX deployment; plugin install/enable/disable without forking.
- **Testability:** Hook context system enables clean state reset between test cases (`clear_all()`). Test suite has dedicated plugin and hooks test files.
- **Observability:** Moderate. Logging present but no built-in metrics or distributed tracing for the tool itself (the deployed Open edX services have their own observability).
