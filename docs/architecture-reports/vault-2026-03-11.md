# vault — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/hashicorp/vault
**Classification:** Microkernel
**Confidence:** 0.96

## Summary

HashiCorp Vault is a secrets management platform that is the canonical implementation of the **Microkernel** architecture pattern in the Go ecosystem. The system is built around a slim, stable core (`vault/core.go`) that manages sealing/unsealing, request routing, policy enforcement, and lease management, while delegating all domain functionality — secret engines, auth methods, and audit sinks — to a richly defined plugin contract (`sdk/logical.Backend`). Plugins are first-class citizens: nine functional backend types ship as built-ins, and the external plugin system (backed by `go-plugin` + gRPC) allows any conforming binary to be registered and hot-loaded at runtime. The single deployable binary, the shared SDK module, and the radix-tree router are all architectural markers of a Microkernel, not a Microservices or Modular Monolith system.

### Key Architectural Files

- `vault/core.go`: The `Core` struct is the microkernel itself. Its key fields are `router *Router`, `logicalBackends map[string]logical.Factory`, `credentialBackends map[string]logical.Factory`, `auditBackends map[string]audit.Factory`, `barrier SecurityBarrier`, `physical physical.Backend`, `mounts *MountTable`, `auth *MountTable`, `ha physical.HABackend`, and `expiration *ExpirationManager`. The Core holds no domain logic — it only manages lifecycle, routing, and the plugin contract.
- `sdk/logical/logical.go`: The eleven-method `Backend` interface is the plugin contract. Every secret engine and auth method must implement `Initialize`, `HandleRequest`, `SpecialPaths`, `System`, `Logger`, `HandleExistenceCheck`, `Cleanup`, `InvalidateKey`, `Setup`, and `Type`. The `Factory func(context.Context, *BackendConfig) (Backend, error)` type is the plugin constructor signature.
- `vault/router.go`: The `Router` uses four concurrent-safe radix trees (`root`, `storagePrefix`, `mountUUIDCache`, `mountAccessorCache`) to map request paths to mounted backends. `routeCommon()` performs longest-prefix matching, strips namespace/mount prefixes, salts tokens per-mount, and calls `backend.HandleRequest()`.
- `vault/plugincatalog/plugin_catalog.go`: The `PluginCatalog` struct maintains both a `BuiltinRegistry` (compile-time) and persistent `catalogView` (runtime external plugins). `NewPluginClient()` spawns out-of-process plugins via `go-plugin` + gRPC, supporting multiplexed process sharing.
- `sdk/plugin/grpc_backend.go`: Implements the gRPC bridge that wraps `logical.Backend` for out-of-process execution. Call options allow up to `math.MaxInt32` message sizes. The `HandshakeConfig` with magic cookie `VAULT_BACKEND_PLUGIN` provides plugin identity verification.
- `helper/builtinplugins/registry.go`: An immutable, thread-safe registry struct containing five credential backends (approle, cert, jwt, oidc, userpass) and four logical backends (kv, pki, ssh, transit). External plugins from `builtin/credential` and `builtin/logical` (aws, consul, database, etc.) are registered via the same `Factory` contract.
- `vault/barrier_aes_gcm.go`: The `SecurityBarrier` wraps the physical storage backend with AES-256-GCM encryption. All plugin-owned data passes through the barrier before hitting the physical layer. The keyring supports key rotation without full re-encryption.
- `vault/expiration.go`: The `ExpirationManager` manages lease lifecycles across all plugins. Secrets issued by any backend get a lease registered here; revocation delegates back to the originating backend via the router, decoupling expiry logic from backend implementation.
- `sdk/framework/backend.go`: `framework.Backend` is the scaffolding layer that lets plugin authors declare `Paths` with regex patterns, field schemas, and operation callbacks instead of implementing low-level request dispatch manually.
- `audit/backend.go`: Defines the audit `Backend` interface as a node-based pipeline: filter → formatter → sink. Any conforming audit backend (file, syslog, socket) can be mounted at runtime.

### Patterns Found

**Microkernel (primary):** The core dispatches all functional requests to dynamically registered backends. The `Core` struct never encodes domain knowledge about any secret type or auth protocol — it only routes. The `Backend` interface + `Factory` function form an immutable extension contract. External plugins run as separate OS processes communicating via gRPC, providing true process isolation. Built-in plugins are registered at compile time using the same contract as external ones.

**Layered (secondary, within core):** The core request path has a clear top-to-bottom dependency: HTTP middleware chain (`http/handler.go`) → Core orchestration (`request_handling.go`) → ACL/token validation (`acl.go`, `token_store.go`) → Router dispatch (`router.go`) → Backend invocation → Security Barrier → Physical storage. This is a layered processing pipeline within the kernel, not a co-equal architectural style.

**Plugin Runtime Isolation:** External plugins are launched as child processes via `github.com/hashicorp/go-plugin`, negotiating via ALPN-secured gRPC. Multiplexed plugins share a single process; non-multiplexed plugins get one process per backend mount. The `PluginCatalog` manages the full external plugin lifecycle including versioning, health monitoring, and graceful reload.

**Event-Driven (peripheral):** `vault/eventbus/bus.go` implements an internal event broker using `eventlogger` for plugin-sourced events. Subscribers receive filtered, CloudEvents-formatted events via buffered Go channels. This is peripheral — it does not carry the primary request/response flow and exists to support plugin observability and cluster-wide event subscriptions.

**HA via Distributed Locking:** High-availability clustering uses distributed lock acquisition (over the HA physical backend) for leader election. Standby nodes forward requests to the active leader via gRPC over mTLS. The Raft integrated storage backend provides both consensus and persistence in deployments that do not use Consul for HA.

## Reasoning

Vault is classified as **Microkernel** with high confidence (0.96) based on the following decisive evidence:

**Core is a kernel, not a domain model.** The `Core` struct in `vault/core.go` contains no secrets domain logic whatsoever. It holds references to `logicalBackends map[string]logical.Factory`, `credentialBackends map[string]logical.Factory`, and the `Router` — all of which are indirection mechanisms, not business logic. Every domain capability (KV, PKI, Transit, AppRole, LDAP, etc.) lives in a plugin.

**The `logical.Backend` interface is a true plugin contract.** The eleven-method interface in `sdk/logical/logical.go` is an explicit, versioned extension point that all backends must satisfy. The `Factory` type is the plugin constructor signature. This is the canonical Microkernel pattern: the kernel defines a stable interface, plugins provide all behavior.

**External plugin isolation is architectural, not incidental.** Out-of-process plugins communicate via gRPC using `github.com/hashicorp/go-plugin` with HMAC-authenticated handshake (`VAULT_BACKEND_PLUGIN` magic cookie). Process isolation means a crashing plugin cannot bring down the core — a direct Microkernel quality attribute.

**Builtin and external plugins use identical contracts.** The builtin registry and the external `PluginCatalog` both produce `logical.Backend` instances via the same `Factory` type. There is no privileged path for built-in plugins. This is the clearest signal of true Microkernel design.

**What was rejected:**
- *Microservices*: Single binary deployment, no independent deployability, shared process memory for builtin plugins, no inter-service network boundary.
- *Modular Monolith*: The extension mechanism is the primary architectural identity, not an organizational convenience. External plugins run in separate processes outside the monolith boundary.
- *Layered*: No horizontal tiering of the whole system. The layered request pipeline is internal to the kernel and does not constitute the system's architectural identity.
- *Service-Based*: No coarse-grained independent services with separate deployment units.
- *Event-Driven*: The EventBus is peripheral; the primary request flow is synchronous RPC from HTTP handler through core to plugin backend.

**Qualifiers (not styles):** Vault uses Hexagonal/Ports-and-Adapters structuring within the plugin SDK (`sdk/logical/` as ports, `builtin/` as adapters), and the physical storage abstraction (`physical.Backend`) is a classic port-adapter boundary. These are qualifiers on the Microkernel classification per ADR-006.

Confidence is 0.96 rather than 1.0 because the layered processing pipeline within the kernel and the peripheral event-driven elements are real architectural features that add nuance, even though they do not displace the dominant Microkernel pattern.
