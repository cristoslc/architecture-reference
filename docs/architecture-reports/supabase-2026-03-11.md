## Style Rationales

**Microservices (primary, 0.92):** The `docker/docker-compose.yml` defines 13 independently deployable services, each with its own Docker image, health check, restart policy, and distinct technology stack. Services are:

- `studio` (Next.js, `supabase/studio`) — management dashboard
- `kong` (Kong 2.8.1) — API gateway
- `auth` (GoTrue, `supabase/gotrue:v2.186.0`) — authentication service in Go
- `rest` (PostgREST) — auto-generated REST API from Postgres schema
- `realtime` (Elixir, `supabase-realtime`) — WebSocket CDC server
- `storage` (`supabase/storage-api`) — file management API in TypeScript/Node.js
- `imgproxy` (darthsim/imgproxy) — on-the-fly image transformation
- `meta` (`supabase/postgres-meta:v0.95.2`) — database management REST API
- `functions` (`supabase/edge-runtime:v1.70.3`) — Deno-based edge functions runtime
- `analytics` (`supabase/logflare:1.31.2`) — log ingestion/query in Elixir
- `db` (`supabase/postgres:15.8.1.085`) — PostgreSQL with custom extensions
- `vector` (`timberio/vector:0.53.0-alpine`) — log collection and routing pipeline
- `supavisor` (`supabase/supavisor:2.7.4`) — Erlang/Elixir connection pooler

Each service is developed in a separate upstream repository (GoTrue at `github.com/supabase/gotrue`, Realtime at `github.com/supabase/realtime`, etc.). Kong acts as the API gateway routing by path prefix: `/auth/v1/* → auth:9999`, `/rest/v1/* → rest:3000`, `/realtime/v1/* → realtime:4000`, `/storage/v1/* → storage:5000`, `/functions/v1/* → functions:9000`, `/pg/* → meta:8080` (per `docker/volumes/api/kong.yml`). Services are independently scalable and communicate exclusively via HTTP/WebSocket through the gateway.

**Service-Based (secondary, 0.80):** Although the services are independently deployed containers, all backend services connect to the same PostgreSQL instance using distinct database users (`supabase_auth_admin`, `supabase_storage_admin`, `supabase_admin`). This shared-database pattern — where coarse-grained services partition by role but share a single data store — is the defining characteristic of Service-Based architecture. The `supabase/postgres` image includes all Supabase-specific extensions, schemas, and initialization scripts (`realtime.sql`, `webhooks.sql`, `roles.sql`, `jwt.sql`, `logs.sql`, `pooler.sql`) in one database cluster.

**Pipeline (supporting, 0.82):** The `docker/volumes/logs/vector.yml` defines a full log processing pipeline using Vector: a `docker_host` source subscribes to all container stdout/stderr; a `project_logs` transform normalizes fields; a `router` transform routes events to per-service channels (`kong`, `auth`, `rest`, `realtime`, `storage`, `functions`, `db`); per-service transforms (`kong_logs`, `kong_err`, `auth_logs`, `rest_logs`, `realtime_logs`, `storage_logs`, `db_logs`) parse and enrich each log stream; and six HTTP sinks fan out to the `analytics` (Logflare) service. This is a classic ETL pipeline architecture embedded within the observability layer. The `supabase_functions.http_request()` trigger function (`docker/volumes/db/webhooks.sql`) adds a secondary database-event-to-HTTP pipeline via `pg_net`, enabling Postgres triggers to invoke HTTP endpoints asynchronously.

## Domain

Postgres-centric Backend-as-a-Service (BaaS) and developer platform. Core service domains: PostgreSQL hosting with REST/GraphQL API auto-generation (PostgREST), authentication with OAuth2/SAML/MFA support (GoTrue), real-time database change subscriptions via CDC (Realtime), object storage with image transformation (Storage + imgproxy), serverless edge functions (Deno Edge Runtime), database management UI and observability (Studio + Logflare), and connection pooling (Supavisor).

## Production Context

- This repository serves as the orchestration layer (Docker Compose), the management frontend (Studio), and shared TypeScript packages. The actual backend service implementations live in separate repos (GoTrue, Realtime, PostgREST, etc.) and are consumed as pre-built Docker images.
- The self-hosted stack is a near-exact replica of Supabase Cloud, which runs these same services at scale on dedicated infrastructure per project.
- The Studio (Next.js) app supports a dual-mode architecture: `NEXT_PUBLIC_IS_PLATFORM=true` for cloud, `false` for self-hosted — with conditional routing, feature flags, and API endpoint switching.
- Edge functions run in isolated Deno sandbox containers, providing per-request isolation without separate container spin-up per invocation.
- Realtime CDC is implemented at the Postgres level via logical replication (`_realtime` schema, `supabase_realtime` publication), with the Elixir Realtime service subscribing to WAL events and broadcasting over WebSocket.
- The shared-database pattern (all services → same Postgres cluster) makes this closer to Service-Based than pure Microservices, though the independent deployment topology and polyglot technology choices clearly cross into Microservices territory.
