# eureka — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/Netflix/eureka
**Classification:** Layered + Microkernel
**Confidence:** 0.90

## Summary

Netflix Eureka is a REST-based service registry for service discovery, load balancing, and failover in AWS cloud environments. The repository produces a single WAR file (`eureka-server`) that runs inside a servlet container, plus a reusable client library (`eureka-client`). The codebase is organized as a **Layered** architecture — a strict three-tier horizontal separation from REST resources through business logic to the in-memory registry — combined with a **Microkernel** pattern where the core registry engine is extended via swappable adapter modules for HTTP transport, configuration, and dependency injection. The Gradle multi-project structure formalizes this: `eureka-core` and `eureka-client` are the stable kernel, while `eureka-client-jersey2`, `eureka-client-archaius2`, `eureka-core-jersey2`, and `eureka-server-governator` are the adapter modules.

### Key Architectural Files

- `eureka-client/src/main/java/com/netflix/discovery/shared/LookupService.java`: Root port interface for service discovery — defines the contract that both `EurekaClient` (client-side) and `InstanceRegistry` (server-side) implement.
- `eureka-client/src/main/java/com/netflix/discovery/EurekaClient.java`: Extends `LookupService`; annotated `@ImplementedBy(DiscoveryClient.class)` — the Guice port-to-adapter binding declared at the interface level.
- `eureka-client/src/main/java/com/netflix/discovery/shared/transport/EurekaHttpClient.java`: Transport port interface (`register`, `cancel`, `sendHeartBeat`, `getApplications`…) with multiple adapter implementations: Jersey 1 (`JerseyApplicationClient`), Jersey 2 (`Jersey2ApplicationClient`).
- `eureka-client/src/main/java/com/netflix/discovery/shared/transport/decorator/EurekaHttpClientDecorator.java`: Decorator chain over `EurekaHttpClient` — `RetryableEurekaHttpClient`, `SessionedEurekaHttpClient`, `RedirectingEurekaHttpClient`, `MetricsCollectingEurekaHttpClient` layered via composition.
- `eureka-client/src/main/java/com/netflix/discovery/guice/EurekaModule.java`: Guice module binding `EurekaClient → DiscoveryClient`, `EurekaInstanceConfig → CloudInstanceConfigProvider`, `EurekaClientConfig → DefaultEurekaClientConfigProvider`. Designed to be overridden with `Modules.override()`.
- `eureka-core/src/main/java/com/netflix/eureka/lease/LeaseManager.java`: Domain interface for `register`, `cancel`, `renew`, `evict` — the lease lifecycle contract.
- `eureka-core/src/main/java/com/netflix/eureka/registry/InstanceRegistry.java`: Extends both `LeaseManager<InstanceInfo>` and `LookupService<String>` — the central server-side port.
- `eureka-core/src/main/java/com/netflix/eureka/registry/AbstractInstanceRegistry.java`: Primary business logic implementation. Maintains `ConcurrentHashMap<String, Map<String, Lease<InstanceInfo>>>` as in-memory registry with `ReentrantReadWriteLock` for concurrency control. Implements registers, renewals, cancellations, evictions, and delta computation.
- `eureka-core/src/main/java/com/netflix/eureka/registry/PeerAwareInstanceRegistryImpl.java`: Extends `AbstractInstanceRegistry`; adds peer-to-peer replication by delegating writes to `PeerEurekaNodes`.
- `eureka-core/src/main/java/com/netflix/eureka/resources/ApplicationsResource.java`: JAX-RS resource (`@Path("/{version}/apps")`); delegates all logic to `PeerAwareInstanceRegistry` via `EurekaServerContext` — pure presentation layer.
- `eureka-core/src/main/java/com/netflix/eureka/EurekaBootStrap.java`: `ServletContextListener`; wires the full server context on startup — instantiates `DefaultEurekaServerConfig`, `DiscoveryClient` (embedded), `PeerAwareInstanceRegistryImpl`, `PeerEurekaNodes`, then starts the context.
- `eureka-core/src/main/java/com/netflix/eureka/EurekaServerContext.java`: Context interface aggregating `EurekaServerConfig`, `PeerEurekaNodes`, `ServerCodecs`, `PeerAwareInstanceRegistry` — the server's top-level composition root.
- `eureka-core/src/main/java/com/netflix/eureka/cluster/PeerEurekaNodes.java`: Manages a list of `PeerEurekaNode` instances; each node uses `JerseyReplicationClient` (HTTP) to batch-replicate registry mutations to sibling servers.

### Patterns Found

**Layered (primary):** Three horizontal tiers with strict downward dependency:

1. **REST / Presentation Layer** — `eureka-core/src/main/java/com/netflix/eureka/resources/`: JAX-RS annotated classes (`ApplicationsResource`, `InstanceResource`, `ApplicationResource`, `VIPResource`, `PeerReplicationResource`, `StatusResource`). Receives HTTP requests, delegates to the registry interface, returns responses. No business logic resides here.
2. **Business Logic / Registry Layer** — `eureka-core/src/main/java/com/netflix/eureka/registry/`: `AbstractInstanceRegistry` (registers, renewals, cancellations, lease eviction, delta tracking, response caching) and `PeerAwareInstanceRegistryImpl` (adds peer replication). The `com.netflix.eureka.cluster/` package (peer replication) and `com.netflix.eureka.aws/` package (cloud binding) sit at this layer.
3. **Data / State Layer** — The in-memory `ConcurrentHashMap<String, Map<String, Lease<InstanceInfo>>>` in `AbstractInstanceRegistry`, the `ResponseCacheImpl` (Guava cache of serialized responses), and the `Lease<T>` model are the data layer. No external database; Eureka is intentionally stateless-on-disk.

Classic layered dependency: `ApplicationsResource` → `PeerAwareInstanceRegistry` → `ConcurrentHashMap` registry store.

**Microkernel (co-primary):** The core registry engine (`eureka-core`, `eureka-client`) acts as the stable kernel with well-defined extension points:

- **Transport adapters**: `EurekaHttpClient` is a port interface; Jersey 1 (`JerseyApplicationClient`) is the default adapter. `eureka-client-jersey2` provides a Jersey 2 adapter as a drop-in replacement module.
- **Configuration adapters**: Default config reads from `eureka-client.properties` via Archaius 1.x (`DefaultEurekaClientConfig`). `eureka-client-archaius2` replaces this with Archaius 2.x config bindings — a separate Gradle module that replaces the config adapter without touching the core.
- **DI / Bootstrap adapters**: `eureka-server` wires everything with Jersey 1 + manual `EurekaBootStrap`. `eureka-server-governator` is an alternative WAR that wires the same core using Netflix Governator (Guice lifecycle management) — a separate deployable adapter around the same kernel.
- **AWS binding adapters**: `eureka-core/src/main/java/com/netflix/eureka/aws/` provides EIP, ENI, and Route 53 binding adapters; the `AwsBinder` interface is the port with `AwsBinderDelegate` as the default implementation.
- **Decorator chain on transport**: `EurekaHttpClientDecorator` enables wrapping the transport port with cross-cutting concerns (retry, sessions, redirects, metrics) without modifying the core.

**Guice DI as the glue**: `EurekaModule.configure()` uses `bind(EurekaClient.class).to(DiscoveryClient.class)` and `Modules.override()` as the runtime plugin mechanism — callers override individual bindings to substitute adapters.

## Quality Attributes

| Attribute | Rating | Justification |
|---|---|---|
| Availability | High | Peer-to-peer server replication via `PeerEurekaNodes`; self-preservation mode halts evictions during network partitions; client-side cache in `DiscoveryClient` provides discovery without server contact |
| Consistency | Low-Medium | AP system (favors availability over consistency); clients cache stale registry data; peer replication is asynchronous batch HTTP — no distributed consensus protocol |
| Scalability | Medium | Horizontal server scaling via peer replication (each server holds full registry copy); `ResponseCacheImpl` reduces per-request serialization cost; client-side delta fetches minimize bandwidth; full registry in memory caps instance count |
| Maintainability | High | Clean three-layer separation with interfaces at every boundary; Gradle multi-project enforces module isolation; adapter modules can be replaced without touching the core; ~409 Java files organized by clear package responsibility |
| Extensibility | High | Explicit adapter swap model: transport (Jersey1 vs Jersey2), config (Archaius1 vs Archaius2), DI (Bootstrap vs Governator), AWS binding all swappable as Gradle module replacements; `Modules.override()` for fine-grained Guice binding replacement |
| Performance | Medium | In-memory `ConcurrentHashMap` registry avoids I/O; `ResponseCacheImpl` pre-serializes responses using Guava cache; read-write lock allows concurrent reads; client heartbeat and delta refresh on scheduler threads minimizes latency impact |
| Operability | Medium | Servlet-based WAR deployment; basic admin UI in `eureka-resources`; Servo metrics instrumentation; status and health endpoints via JAX-RS resources; no out-of-box distributed tracing or structured logging |
| Resilience | High | Self-preservation mode prevents mass eviction during network partitions; `TimedSupervisorTask` wraps heartbeat/registry-refresh threads to survive transient failures; `BackupRegistry` interface allows fallback discovery source |

## Reasoning

Eureka's architecture is definitively **Layered** because the three tiers (REST resources, registry business logic, in-memory state) have zero cross-tier coupling violations — `ApplicationsResource` holds a reference to `EurekaServerContext`/`PeerAwareInstanceRegistry` but no registry data structures; `AbstractInstanceRegistry` contains no JAX-RS or servlet imports. The `EurekaBootStrap` is the composition root that assembles the tiers at startup.

The **Microkernel** classification is equally primary and structurally formalized by the Gradle module layout. The separation of `eureka-client-jersey2`, `eureka-client-archaius2`, and `eureka-server-governator` as independent Gradle modules that depend on the core without the core depending on them is the textbook microkernel pattern: a stable core with an explicit extension surface. The `EurekaHttpClient` interface and `EurekaModule.configure()` with its `Modules.override()` design comment make the extensibility contract explicit.

**What this is NOT:**
- Not Microservices: single WAR deployment; `eureka-client` is a library module embedded in apps and in the server itself, not an independent service.
- Not Modular Monolith: module boundaries here are transport/config adapter replacements organized by technical role, not domain-bounded contexts (there is no "orders module" or "users module" — Eureka has one domain: service registration).
- Not Hexagonal: the domain model (`AbstractInstanceRegistry`) directly references JAX-RS annotations via the resources layer that calls it, and `EurekaMonitors` (Servo metrics) imports permeate the registry layer — dependencies do not point strictly inward.
- Not Event-Driven: all communication is synchronous REST calls. Peer replication is HTTP batch requests; no message brokers or event buses.

Confidence is 0.90. The Layered pattern is unambiguous from direct code inspection. The Microkernel classification is strongly supported by the explicit multi-module adapter structure, though the "plugin" mechanism is dependency injection swapping rather than runtime class loading — a build-time microkernel variant that is well-established and deliberate given the module naming and `Modules.override()` design.
