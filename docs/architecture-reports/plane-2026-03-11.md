# Architecture Report: Plane

**Date:** 2026-03-11
**Source:** https://github.com/makeplane/plane
**Classification:** Service-Based + Event-Driven
**Confidence:** 0.88
**Method:** deep-analysis

---

## Summary

Plane is an open-source project management tool (work items, cycles, modules, pages, analytics) built as a **pnpm/Turborepo monorepo** with a Python/Django backend and multiple React Router v7 frontends. The system is deployed as a set of coarse-grained services sharing a single PostgreSQL database and Redis instance — a textbook Service-Based architecture. Background processing via Celery/RabbitMQ provides a secondary Event-Driven layer for async tasks, notifications, and webhook dispatch. A dedicated Node.js real-time service (`live`) handles collaborative document editing over WebSockets.

## Primary Style: Service-Based Architecture

The `docker-compose.yml` reveals Plane's deployment topology: multiple independently started processes, all sharing one PostgreSQL instance and one Redis/Valkey instance.

| Service | Role | Technology |
|---|---|---|
| `api` | Synchronous REST request handling | Django + DRF |
| `worker` | Async Celery task consumer | Django + Celery |
| `beat-worker` | Scheduled task scheduler (cron) | Django + Celery Beat |
| `migrator` | Database migration runner | Django `manage.py migrate` |
| `live` | Real-time collaborative editing | Node.js + Hocuspocus/Yjs |
| `web` | Main user-facing SPA | React Router v7 + MobX |
| `admin` | Instance administration SPA | React Router v7 + MobX |
| `space` | Public/embed SPA | React Router v7 + MobX |
| `proxy` | Reverse proxy / ingress | Caddy |

The `api`, `worker`, and `beat-worker` services are **all built from the same Django codebase** (`apps/api`), just invoked with different entry-point scripts (`docker-entrypoint-api.sh`, `docker-entrypoint-worker.sh`, `docker-entrypoint-beat.sh`). They share one PostgreSQL database (`plane-db`) and one Redis store (`plane-redis`). This is the defining characteristic of Service-Based architecture: coarse-grained service partitions with shared data storage.

The `live` service is a distinct technology (Node.js) deployed separately and talks to the same Redis for presence/state propagation.

The three frontends (`web`, `admin`, `space`) are independently deployable SPAs that consume the `api` service over HTTP. They are not microservices — they have no independent data stores and no service-to-service APIs beyond the shared REST backend.

## Internal Backend Structure: Layered Django Monolith

Within the `apps/api` Django application, the code is organized in classic horizontal layers:

| Layer | Path | Responsibility |
|---|---|---|
| URL routing | `plane/urls.py`, `plane/app/urls/`, `plane/api/urls/` | Request dispatch |
| Views (Controllers) | `plane/app/views/`, `plane/api/views/` | HTTP handling, permission checks, response shaping |
| Serializers | `plane/app/serializers/`, `plane/api/serializers/` | Input validation, output formatting |
| Models | `plane/db/models/` | Domain entities (Issue, Project, Workspace, Cycle, Module, Page, etc.) |
| Background tasks | `plane/bgtasks/` | Celery tasks — decoupled from request cycle |
| Utilities | `plane/utils/` | Filters, pagination, grouping, exception logging |

The backend serves two distinct API surfaces under one Django process:
- `api/` — internal application API (session-authenticated, full feature set)
- `api/v1/` — public REST API (token-authenticated, OpenAPI schema via drf-spectacular)
- `api/public/` — unauthenticated embed/space API
- `auth/` — authentication flows (magic link, OAuth, password)
- `api/instances/` — instance licensing API

## Secondary Style: Event-Driven (Celery/RabbitMQ)

Views dispatch background work using Celery's `.delay()` mechanism. From `plane/app/views/issue/base.py`:

```python
from plane.bgtasks.issue_activities_task import issue_activity
from plane.bgtasks.webhook_task import model_activity

# Inside a create/update handler:
issue_activity.delay(...)
model_activity.delay(...)
recent_visited_task.delay(...)
issue_description_version_task.delay(...)
```

The Celery beat scheduler (`celery.py`) registers ~12 periodic tasks covering:
- Email notification batching (every 5 minutes)
- Issue automation (archive/close stale issues, daily)
- Asset and log cleanup (daily, various hours UTC)
- Instance telemetry (every 6 hours)

RabbitMQ (`plane-mq`) is configured as the Celery broker in production, with Redis as the result backend and cache. This decouples mutation side-effects (notifications, activity logs, webhook delivery, version snapshots) from the synchronous request path, constituting a clear event-driven processing tier.

## Real-Time Collaboration (Live Service)

The `live` service is a standalone Node.js server using **Hocuspocus** (Y.js CRDT) for conflict-free concurrent document editing:

```typescript
// apps/live/src/hocuspocus.ts
this.server = new Hocuspocus({
  name: this.serverName,
  onAuthenticate,
  onStateless,
  extensions: getExtensions(),
  debounce: 10000,
});
```

It exposes a WebSocket endpoint behind the Caddy proxy. Redis is used for multi-instance coordination. This service is fully independent of the Django API at the transport level — it validates sessions against the API on connection but thereafter communicates peer-to-peer over WebSockets.

## Frontend Architecture

All three frontends share code through pnpm workspace packages:

- `@plane/services` — API service abstractions (one class per domain: IssueService, CycleService, etc.)
- `@plane/shared-state` — MobX stores shared across apps
- `@plane/types` — TypeScript type definitions
- `@plane/ui` — Design system components
- `@plane/editor` — Tiptap-based rich text editor

The main `web` app uses a deep **MobX** store hierarchy assembled in `CoreRootStore` (35+ domain stores: issue, cycle, module, project, workspace, analytics, etc.). The `ce/` directory provides Community Edition overrides and extensions on top of the core store, suggesting a plugin-like extension model at the frontend layer.

## Quality Attributes

- **Scalability:** Worker, beat-worker, and live services can scale horizontally independently of the API; Redis/RabbitMQ decouple async load
- **Real-time Collaboration:** Hocuspocus/Y.js with Redis adapter enables multi-instance CRDT document editing
- **Separation of Concerns:** Request handling, background processing, and real-time updates are in distinct service boundaries
- **Observability:** JSON-structured logging throughout (Django + pino on Node.js), Scout APM integration in production
- **Extensibility:** CE/EE edition split in both frontend (`ce/`) and backend (`license/`) enables feature gating
- **Reliability:** Beat scheduler and periodic cleanup tasks prevent unbounded data growth and ensure SLA-sensitive operations

## Why Not Other Styles

- **Not Microservices:** All backend services (`api`, `worker`, `beat-worker`) share one PostgreSQL database and one Redis instance. There are no bounded contexts with independent data ownership. The `live` service is the only technology-heterogeneous component, but it does not own a database.
- **Not Modular Monolith:** Multiple independently deployed processes with separate Dockerfiles, not a single deployment unit. The frontend apps are truly separate web applications.
- **Not Layered (primary):** The internal Django backend is layered, but the overall system spans multiple deployment units. The top-level classification must reflect the deployment topology.
- **Not Hexagonal/Ports-and-Adapters:** No explicit port interfaces. Django ORM models are accessed directly from views and tasks without adapter abstractions.
- **Not CQRS:** No read/write model split. Read replicas are optionally supported (`DATABASE_READ_REPLICA_URL`) but Django ORM models serve both reads and writes without separate query/command models.
- **Not Event-Driven (primary):** Celery tasks are triggered imperatively via `.delay()` calls from views, not via a domain event bus. The message queue is an implementation detail of background processing, not the primary integration mechanism between services.
