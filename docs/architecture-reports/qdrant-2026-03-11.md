## One-Line Summary

A single-binary Rust vector search engine organized as a Modular Monolith with a strict Layered request path (REST/gRPC → Dispatcher → Collection → Shard → Segment) and an optional Raft consensus overlay for distributed cluster deployments.

## Primary Style: Modular Monolith

Qdrant compiles to a **single binary** from one `src/main.rs` entry point and is packaged in a single Docker image. There is exactly one `[[bin]]` target named `qdrant` (plus three debug-mode utility binaries that require a feature flag). All functionality runs in-process with no network boundaries between internal components.

Module boundaries are enforced at **Rust compiler time** through the Cargo workspace. The `lib/` directory contains fourteen crates whose `Cargo.toml` files declare explicit path dependencies in a strict unidirectional acyclic graph:

```
api
 └── storage  ←  collection  ←  shard  ←  segment
                                           ├── quantization
                                           ├── sparse
                                           ├── gridstore
                                           └── posting_list
common/* (leaf, depended on by all)
```

No lower-level crate imports a higher-level crate. Rust's borrow checker and visibility system (`pub(crate)`, `pub(super)`) reinforce these boundaries at every API surface. This is the textbook definition of a modular monolith: single deployable unit, compiler-enforced module seams.

## Secondary Style: Layered

Within the monolith, there is a clear five-tier horizontal stack through which every request flows:

| Layer | Components | Responsibility |
|-------|------------|----------------|
| **API** | `src/actix/api/`, `src/tonic/api/` | HTTP/gRPC parsing, authentication (JWT/API key), CORS, TLS |
| **Dispatcher / Service** | `lib/storage/src/dispatcher.rs`, `lib/storage/src/content_manager/toc/` | Operation routing; consensus proposal injection when cluster mode is enabled |
| **Collection / Domain** | `lib/collection/src/collection/`, `lib/collection/src/shards/` | Collection lifecycle, shard fan-out, replica management, WAL, update handler, optimizers |
| **Shard / Data** | `lib/shard/src/`, `lib/collection/src/shards/local_shard/` | Segment holder, write-ahead log, search request batching, update pipeline |
| **Segment / Storage** | `lib/segment/src/` | HNSW graph index, payload index, vector storage (dense, sparse, multi-dense), mmap, quantization |

Requests enter the API layer, traverse down through the Dispatcher's `TableOfContent`, fan out across shards (local or remote via gRPC `RemoteShard`), reach individual `Segment` objects, and return results upward. No layer calls upward to a higher layer.

## Distributed Overlay: Raft Consensus (Not a Style)

When cluster mode is enabled (`cluster.enabled: true`), Qdrant runs an additional Raft thread (`src/consensus.rs` using the `raft` crate). This thread receives `ConsensusOperations` via an in-process `OperationSender` channel and replicates collection metadata changes across peer nodes. Shard data is replicated by the `ReplicaSet` within `lib/collection/`, which implements fan-out writes to `RemoteShard` instances over gRPC.

This Raft consensus overlay does **not** change the fundamental architecture style:
- Each node still runs the same single monolithic binary.
- Cluster peers are **identical replicas**, not decomposed services with different responsibilities.
- The Raft mechanism is a durability and replication facility within the monolith's service layer — analogous to WAL replay or leader election, not microservice decomposition.

The `ConsensusStateRef` is stored inside `Dispatcher` which is part of the `storage` layer. All consensus operations are committed synchronously before the `Dispatcher` returns a result to the API layer.

---

## Write-Ahead Log and Background Optimizers

Each local shard maintains a WAL (using the `wal` crate). The `UpdateHandler` in `lib/collection/src/update_handler.rs` receives `UpdateSignal` messages via a Tokio mpsc channel and applies operations to segments. A pool of background optimizer tasks (`IndexingOptimizer`, `MergeOptimizer`, `VacuumOptimizer`, `ConfigMismatchOptimizer`) run against the segment holder to rebuild HNSW indices, compact segments, and clean deleted points. These async tasks are scheduled by `plan_optimizations()` in response to operation count thresholds. This is an internal background processing pattern within the monolith, not an Event-Driven architecture.

## Rejected Styles

| Style | Reason for Rejection |
|-------|---------------------|
| **Microservices** | Single binary, single Docker image, single process. Cluster nodes are identical replicas of the same monolith, not decomposed services. |
| **Event-Driven** | No message broker (no Kafka, RabbitMQ, NATS). WAL and Raft are durability/consensus mechanisms, not async event streams. The `events.rs` file defines internal struct signals (`CollectionDeletedEvent`, `SlowQueryEvent`), not an event bus. |
| **Service-Based** | No coarse-grained service decomposition. Storage, collection, and segment are internal library layers, not independently deployed services. |
| **CQRS** | Read and write paths share the same segment infrastructure. `ShardOperation` trait has both `update()` and `core_search()` methods. No separate read model or write model. |
| **Microkernel** | No runtime plugin loading. No extension points. All functionality is compiled into the binary at build time. Feature flags (gpu, rocksdb) select compiled-in code paths, not loaded plugins. |
| **Pipeline** | No stage-based data transformation topology. Query processing is synchronous fan-out, not a pipeline graph. |
| **Space-Based** | Not a distributed in-memory data grid. Qdrant is a persistent on-disk store with disk-backed segments (mmap). |
| **Serverless** | Runs as a persistent long-lived process; not function-per-request. |
| **Hexagonal** | Code is not organized around explicit named ports/adapters. REST and gRPC handlers are API-layer modules, not interchangeable adapters behind an application port. |
| **Domain-Driven Design** | No bounded contexts, aggregates, domain events, or ubiquitous language conventions. Organized around technical concerns (segment, shard, vector storage) not business domains. |

## Quality Attributes

- **Performance**: HNSW graph-based ANN search; mmap vector storage; jemalloc; separate Tokio runtimes for search, update, and general tasks; GPU acceleration (optional).
- **Scalability**: Horizontal scaling via sharding and replication; Raft-coordinated cluster membership; shard transfer and resharding.
- **Fault tolerance**: Write-ahead log per shard; Raft consensus for metadata; replica sets with configurable replication factor; snapshot/restore support.
- **Maintainability**: Rust workspace with compile-time boundary enforcement; unidirectional dependency graph; each crate has focused responsibility.
- **Security**: JWT RBAC, API key authentication, TLS, per-collection access control, audit logging.
- **Observability**: Prometheus metrics endpoint, telemetry reporting, OpenTelemetry tracing (optional), slow query logging.
- **Extensibility**: Feature flags for GPU, RocksDB, staging, quantization variants; edge runtime crate for embedding Qdrant without consensus.

## Key Evidence

1. `Cargo.toml` `[workspace]` section lists fourteen member crates — all path-referenced, compiled together into one binary.
2. `src/main.rs` is the single entry point; it initializes `TableOfContent`, `Dispatcher`, optional `Consensus`, and then launches actix (REST) and tonic (gRPC) in threads — all in one process.
3. `lib/storage/src/dispatcher.rs` — `Dispatcher` holds a reference to `TableOfContent` (the central collection registry) and an optional `ConsensusStateRef`; the entire cluster routing logic lives in one struct.
4. `lib/collection/src/shards/shard_trait.rs` — `ShardOperation` trait defines both read (`core_search`, `scroll_by`) and write (`update`) on the same interface, confirming no CQRS separation.
5. `lib/collection/src/shards/remote_shard.rs` — gRPC calls to peer nodes use Tonic clients inline; peers run the same binary and expose the same internal API, confirming identical-replica clustering rather than microservices.
6. `src/consensus.rs` — Raft node runs in a dedicated thread within the same process, receiving messages over in-process channels (`tokio::sync::mpsc`).
