## Style Rationales

**Modular Monolith (primary, 0.92):** Shopware is a single deployable PHP application that is explicitly decomposed into bounded domain modules sharing a single database (MySQL/MariaDB). The `src/` tree reveals four named top-level bundles â€” `Core`, `Storefront`, `Administration`, and `Elasticsearch` â€” each registered as a Symfony `Bundle` in `Kernel::registerBundles()`. Within `Core` alone there are thirteen high-cohesion subdomain namespaces: `Framework`, `Checkout`, `Content`, `System`, `Service`, `Profiling`, `Maintenance`, `DevOps`, `Installer`, `Migration`, and more. Each domain sub-module (e.g. `Checkout/Cart`, `Checkout/Payment`, `Checkout/Order`, `Content/Product`, `Content/Cms`, `Content/Flow`, `System/SalesChannel`, `System/Tax`) carries its own `Definition`, `Entity`, `Service`, `DependencyInjection`, and `Migration` directories â€” the classic vertical slice of a modular monolith. The 2022 ADR "Mapping of product area" (`adr/2022-09-28-mapping-of-product-area.md`) formalizes this decomposition into eight explicit product areas (`checkout`, `content`, `inventory`, `services-settings`, `buyers-experience`, `customer-order`, `admin`, `storefront`), enforced by `@Package` annotations on every PHP class. All modules share one Doctrine DBAL connection to a single database schema defined in `src/Core/schema.sql`, and the shared Data Abstraction Layer (`Framework/DataAbstractionLayer/`) provides the common persistence API across all domains. The application is packaged and distributed as a single Composer library (`shopware/platform`).

**Microkernel (secondary, 0.88):** On top of the modular monolith sits a first-class two-track extension system that is architecturally a Microkernel. The core is extended at runtime by *Plugins* (PHP bundles loaded via `KernelPluginLoader` and `KernelPluginCollection`) and *Apps* (manifest-driven, no-code extensions that run backend logic on remote HTTP servers, introduced in ADR `2020-08-12-implement-app-system-inside-platform.md`). Both mechanisms are structured identically to a microkernel: a minimal, stable core (Framework bundle, DAL, SalesChannel, event bus) with extension point protocols for the plug-in periphery. The decoration pattern (`adr/2020-11-25-decoration-pattern.md`) defines the standard plug-in contract: every extensible service exposes an `abstract` base class with a `getDecorated()` method, and plugins chain-wrap services via the Symfony DI container. The `custom/` directory and the App manifest system (`Framework/App/`) provide the runtime loading surface. The 2024 ADR "Transition to an Event-Based Extension System" (`adr/2024-06-18-extended-event-system.md`) represents a further evolution of the microkernel plug-in protocol, formalising events as the primary extension contract. The App scripting system (`Framework/Script/`, ADR `2021-10-21-app-scripting.md`) adds Twig-based sandbox scripting that executes within the kernel at named hook points. Checkout and Context Gateways (`adr/2024-04-01-checkout-gateway.md`, `adr/2025-04-01-context-gateway.md`) provide explicit interface contracts for apps to intercept and modify core workflows.

**Event-Driven (secondary, 0.82):** Asynchronous message processing via Symfony Messenger is a load-bearing architectural seam throughout the platform. `Framework/MessageQueue/` defines the message bus infrastructure: `AsyncMessageInterface`, `LowPriorityMessageInterface`, `DeduplicatableMessageInterface`, `ScheduledTask` (with `ScheduledTaskHandler` and Symfony Scheduler), and middleware/stamps. Heavy operations â€” DAL indexing (`DataAbstractionLayer/Indexing/EntityIndexer` dispatches `EntityIndexingMessage`), product export, import/export, mail delivery â€” are all offloaded to async Messenger queues. The `Infrastructure` redis-backed increment pool for message-queue stats is detailed in ADR `2021-11-09-increment-pattern.md`. The Flow Builder subsystem (`Content/Flow/`) is a business-rules automation engine that reacts to `FlowEventAware` domain events and executes chains of user-configured actions; the 2025 ADR "Move flow execution after business process" (`adr/2025-01-31-move-flow-execution-after-business-process.md`) documents the architectural decision to queue Flow execution after `KernelEvents::TERMINATE`, `WorkerMessageHandledEvent`, and `ConsoleEvents::TERMINATE` â€” making the event bus the primary decoupling mechanism between business logic and side effects. The nested event system (`Framework/Event/NestedEventDispatcher`) and the `BusinessEventRegistry` provide structured, typed domain events throughout the codebase.

**Hexagonal Architecture (secondary, 0.75):** The platform consistently separates domain logic from infrastructure through port-and-adapter abstractions. The Data Abstraction Layer is itself a hexagonal port: `EntityWriteGatewayInterface`, `AbstractKeyValueStorage` (backed by MySQL or Redis), and `AbstractStockStorage` (ADR `2023-05-15-stock-api.md`) are all abstract ports with swappable implementations. The `Framework/Adapter/` directory groups infrastructure adapters by concern: `Cache/`, `Database/`, `Filesystem/`, `Messenger/`, `Redis/`, `Session/`, `Storage/`. The Store API (`System/SalesChannel/SalesChannel/`) is a domain-facing inbound port that is explicitly separated from the Admin API (`Framework/Api/`) and the Storefront (`src/Storefront/`). Abstract gateway interfaces for Checkout (`CheckoutGatewayInterface`) and Context (`AppContextGateway`) define clean inbound ports for external app server integration. Telemetry (`Framework/Telemetry/`, ADR `2024-07-30-add-telemetry-abstraction-layer.md`) is exposed via `MetricTransportInterface` â€” an outbound port whose implementations ship as separate packages. Payment handlers (`SynchronousPaymentHandlerInterface`, `AsynchronousPaymentHandlerInterface`, ADR `2021-10-01-payment-flow.md`) are explicit extension ports that isolate the checkout domain from payment-provider specifics.

## Domain

Open-source e-commerce platform. Core domains: Cart & Checkout, Order Management, Payment, Shipping, Product Catalog, Category/CMS, Customer, Promotions, Pricing/Tax, Storefront, Administration, Sales Channels (multi-tenant headless), Search (OpenSearch), Flow Builder (business automation), App Marketplace, Import/Export, Document generation (invoices, delivery notes). Multi-sales-channel model where each `SalesChannel` is an independent storefront context (language, currency, pricing rules, domain).

## Production Context

- Single deployable PHP 8.2+ / Symfony 7 application; requires MySQL/MariaDB, optional Redis (cache/sessions/queues), optional OpenSearch/Elasticsearch
- Multi-sales-channel: every request is scoped to a `SalesChannelContext` (currency, language, tax, rules, shipping); headless API-first deployment supported
- Store API (`/store-api/`) for headless storefronts; Admin API (`/api/`) for back-office integrations; Storefront (`src/Storefront/`) for traditional server-rendered Twig UI
- Extension ecosystem: 3,100+ plugins and apps available via store.shopware.com; Apps run backend logic on remote servers (zero-trust model, no third-party PHP code on platform server)
- Symfony Messenger workers process async jobs; Docker Compose dev environment includes MariaDB, Valkey (Redis-compatible), OpenSearch, Mailpit
- Versioned as `shopware/platform 6.x`; major versions gated behind feature flags; ADR governance process with `adr/` directory tracking all significant architectural decisions
