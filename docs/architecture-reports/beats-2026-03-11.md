# Architecture Report: Elastic Beats

**Date:** 2026-03-11
**Repository:** https://github.com/elastic/beats
**Classification:** Microkernel, Pipeline
**Confidence:** 0.95

---

## Summary

Elastic Beats is a collection of lightweight, independently deployable data shippers (Filebeat, Metricbeat, Auditbeat, Heartbeat, Packetbeat, Winlogbeat, Osquerybeat) built on a shared Go framework called `libbeat`. The repository is a textbook Microkernel (Plugin) architecture: `libbeat` serves as the core kernel providing common infrastructure, while each Beat and each module within a Beat functions as a plugin registered via well-defined extension interfaces. Within each Beat, data flows through a strict sequential Pipeline — input, processing, queue, output — that constitutes a strongly expressed secondary architectural style.

## Architecture Styles Identified

### 1. Microkernel (Primary, highest confidence)

The `libbeat` framework acts as the microkernel — a minimal core that provides:

- **Plugin loading infrastructure** (`libbeat/plugin/plugin.go`): `PluginLoader` type, `MustRegisterLoader()`, `RegisterLoader()`, and `LoadPlugins()` for dynamic `.so` loading on Linux/Darwin.
- **Core contract** (`libbeat/beat/beat.go`): The `Beater` interface with `Run(*Beat) error` and `Stop()` is the sole requirement for any Beat. `Creator func(*Beat, *config.C) (Beater, error)` is the factory type.
- **Output plugin registry** (`libbeat/outputs/output_reg.go`): `RegisterType(name, Factory)` / `FindFactory(name)` — outputs register themselves via `init()`.
- **Processor plugin registry** (`libbeat/processors/registry.go`): `RegisterPlugin(name, Constructor)` pattern, with a `pluginKey = "libbeat.processor"` keying into the central plugin bus.
- **Input plugin registry** (`filebeat/input/registry.go`): `Register(name, Factory)` / `GetFactory(name)` map-based registry.
- **Metricbeat module/metricset registry** (`metricbeat/mb/registry.go`): `var Registry = NewRegister()` singleton; `ModuleFactory` and `MetricSetFactory` types; modules self-register via `init()`:

```go
// metricbeat/module/apache/status/status.go
func init() {
    mb.Registry.MustAddMetricSet("apache", "status", New,
        mb.WithHostParser(hostParser),
        mb.DefaultMetricSet(),
    )
}
```

- **Build-time plugin discovery** (`filebeat/include/list.go`): Code-generated blank imports (`_ "github.com/elastic/beats/v7/filebeat/module/apache"`) cause `init()` functions to self-register — the canonical Go plugin pattern.
- **Community ecosystem**: Dozens of community-authored Beats extend the framework, which is the defining hallmark of a Microkernel: third parties can build on the kernel without modifying it.
- **Two-level plugin nesting**: Beat → Module → MetricSet/Input forms a two-tier plugin hierarchy, with each Beat itself being a plugin of libbeat, and each module being a plugin of its host Beat.

### 2. Pipeline (Secondary, strongly expressed)

Within each Beat, event data flows through a strict, ordered sequence of stages orchestrated by the publisher pipeline in `libbeat/publisher/pipeline/`:

```
Input/Harvester → Processors → Queue (mem/disk) → Output Controller → Output Client
```

- **Inputs** produce events (file harvesters, network sniffers, API pollers).
- **Processors** transform events (add metadata, filter, decode — `libbeat/publisher/processing/`).
- **Queue** (`libbeat/publisher/queue/`) buffers events with configurable mem or disk backends; the `Queue` interface with `Producer` → `Get` → `Batch` → `ACK` is a textbook pipeline stage contract.
- **Output controller** pulls batches from the queue and distributes them across load-balanced output clients.
- **Outputs** deliver events to Elasticsearch, Logstash, Kafka, Redis, or file.

The pipeline is explicitly documented in source comments: `"The pipeline consists of clients, processors, a central queue, an output controller and the actual outputs."` ACK propagation flows backward through the pipeline to ensure at-least-once delivery semantics.

### Styles Rejected

- **Microservices**: Beats do not communicate with each other. Each ships to external sinks (Elasticsearch/Logstash). No service mesh, inter-service API calls, or service registry.
- **Modular Monolith**: Each Beat is a separate binary with its own `main.go` and `Dockerfile` — not a single deployable monolith.
- **Event-Driven**: Events flow through a pipeline, not via decoupled async event buses between loosely coupled consumers. The internal queue is a pipeline buffer, not an event broker.
- **Hexagonal Architecture**: No explicit ports-and-adapters boundary; interfaces exist but there is no domain core being protected from infrastructure.
- **Domain-Driven Design**: Modules are organized by technology source (apache, redis, docker), not by business domain. No aggregates, bounded contexts, or ubiquitous language.
- **Layered**: There is some layering within individual Beats (input → process → output), but this is subordinate to the plugin and pipeline patterns, not an independent architectural style.

## Quality Attributes

| Attribute | Justification |
|---|---|
| **Extensibility** | New Beats, modules, inputs, processors, and outputs can be added without modifying libbeat. Community ecosystem demonstrates real-world extensibility. |
| **Portability** | Small footprint, no runtime dependencies, cross-platform (Linux, macOS, Windows) with per-OS Dockerfiles and build targets. |
| **Reliability** | At-least-once delivery via ACK propagation through the pipeline; disk-backed queue option for durability across restarts; file state registrar for resume on crash. |
| **Operability** | Centralized configuration via YAML, hot-reload via `reload.ReloadableList`, Fleet/Elastic Agent central management (`libbeat/management/`), built-in self-monitoring metrics. |
| **Performance** | Lightweight by design — minimal memory and CPU footprint is an explicit goal; in-process pipeline avoids serialization overhead between stages; early event encoding in outputs reduces memory pressure. |
| **Modularity** | Clear separation between kernel (libbeat) and plugins; each module is independently testable, buildable, and releasable within the monorepo. |
| **Observability** | Internal monitoring subsystem (`libbeat/monitoring/`), input-level metrics (`libbeat/monitoring/inputmon`), OpenTelemetry collector components in x-pack for telemetry export. |

## Classification Reasoning

The Beats repository is unambiguously a **Microkernel** architecture with **Pipeline** as a co-primary style. The evidence is structural, not inferred: `libbeat` is a published extension framework with explicit plugin loading, multiple named registries, a factory-based extension contract (`Beater`, `Creator`, `Factory` types), and a documented community ecosystem of third-party Beats. Every mechanism of plugin architecture is present — registration, discovery, dynamic loading, and multiple independent extension points.

The Pipeline style is equally concrete: the publisher pipeline is the central runtime artifact of every Beat, with distinct stages separated by typed interfaces (Producer, Queue, Batch, Client) and explicit ACK-based backpressure coordinating the stages.

The previous classification (Microkernel + Layered) was correct on the primary style. This analysis upgrades the secondary style from Layered to Pipeline because the sequential event-processing pipeline in `libbeat/publisher/pipeline/` is an explicit architectural construct with its own interface contracts — not simply "layering" in the abstract sense. Layering is an internal implementation detail within stages of the pipeline, not an independent architectural pattern at the same abstraction level.

**Confidence: 0.95** — very high. Both primary patterns are explicitly expressed in source code with named types, registries, and interfaces. The only uncertainty is whether to name Pipeline or Layered as the secondary style; Pipeline is the stronger evidence.
