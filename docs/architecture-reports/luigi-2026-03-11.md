# luigi — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/spotify/luigi
**Classification:** Pipeline + Microkernel
**Confidence:** 0.89

## Summary

Luigi is a Python batch workflow orchestration framework whose fundamental abstraction is the Task DAG (Pipeline). Users define tasks with `requires()`, `output()`, and `run()` methods that form an explicit directed acyclic graph of data dependencies. At runtime, a central scheduler resolves that graph and dispatches work to one or more workers. The framework's extension surface — the `luigi/contrib/` directory with 50+ pluggable backends (S3, GCS, Hadoop, BigQuery, Spark, Postgres, Redis, Kubernetes, Docker, etc.) — constitutes a deliberate Microkernel model: a stable core engine with loosely coupled peripheral plugins, each extending the base `Task` or `Target` abstract class.

### Core Abstractions

**Task** (`luigi/task.py`): The base unit. Subclasses implement three methods:
- `requires()` — returns upstream Task dependencies (defines the DAG edges)
- `output()` — returns one or more `Target` objects (idempotency check)
- `run()` — executes the unit of work

The `Register` metaclass auto-discovers and caches all Task subclasses globally, forming the plugin registry used by the CLI and scheduler.

**Target** (`luigi/target.py`): Abstract resource with a single required method `exists()`. Completeness of a task is determined entirely by whether its output targets exist — this is Luigi's idempotency contract. New storage systems are integrated by subclassing `Target` and `FileSystem`.

**Scheduler** (`luigi/scheduler.py`): A single-threaded, in-process or remote (HTTP) central authority. Maintains the full dependency graph as a state machine (PENDING → RUNNING → DONE | FAILED). Serializes access to prevent duplicate task execution across multiple workers. Workers poll the scheduler via `get_work()` RPC.

**Worker** (`luigi/worker.py`): Traverses the dependency graph registered by the user, announces all tasks to the scheduler, then repeatedly calls `get_work()` to receive permitted tasks. Executes each in a `TaskProcess` (forked subprocess) for isolation and timeout support.

**Server** (`luigi/server.py`): Tornado-based REST server (`luigid`) exposing the scheduler over HTTP. `rpc.py` implements the client-side `RemoteScheduler` that translates method calls to HTTP requests. Workers can thus operate across machines against a shared scheduler.

**Event System** (`luigi/event.py`): Lightweight callback hooks (SUCCESS, FAILURE, START, PROGRESS, PROCESSING_TIME, DEPENDENCY_DISCOVERED) attached to the Task class. Pure in-process, not a message broker.

### Pipeline Pattern

Luigi's execution model is a textbook pipe-and-filter / Pipeline architecture:

1. **User defines** a Task graph in Python via `requires()` chains
2. **Worker traverses** the graph bottom-up, registering tasks with the scheduler
3. **Scheduler resolves** the DAG, enforces sequencing by dependency completion
4. **Worker executes** each permitted task in order, checking `output().exists()` for idempotency
5. **Targets** are the data artifacts flowing between pipeline stages — files, database records, HDFS paths

This is exactly the GNU Make analogy the authors draw: tasks are rules, targets are file artifacts, dependency declarations define the build graph. The data flowing through the system is the "pipe" content; tasks are the "filters."

### Microkernel Pattern

The `contrib/` package is the extension surface of the Microkernel:

- **Core (kernel):** `task.py`, `target.py`, `scheduler.py`, `worker.py`, `server.py`, `parameter.py`, `task_register.py`
- **Plugin contract:** Subclass `Task` (override `run/requires/output`) or `Target` (override `exists`), and optionally `FileSystem`
- **Built-in plugins:** 50+ modules in `contrib/` covering cloud storage, databases, compute platforms, cluster schedulers, monitoring systems
- **Runtime extension:** `task_register.Register` metaclass captures all `Task` subclasses at import time; CLI can load arbitrary modules via `--module`
- **Metrics plugin:** `luigi/metrics.py` defines `MetricsCollector` ABC; Datadog and Prometheus implementations in `contrib/`; custom collectors via `metrics_custom_import` config

The core engine has no knowledge of S3, Hadoop, Spark, or Postgres — all of that lives in contrib plugins that satisfy the `Task`/`Target` contracts.

### Deployment Model

Luigi runs as a single Python package in one of two modes:
- **Local (--local-scheduler):** Worker and scheduler in the same process (in-memory scheduler); suited for development and single-machine pipelines
- **Central scheduler (luigid):** `luigid` daemon runs the Tornado HTTP server hosting the scheduler; workers on any machine connect via `rpc.RemoteScheduler`; multiple workers coordinate safely through the central scheduler

There are no independently deployed microservices. The "server" is one Tornado process; the "workers" are the same Python process that also does computation. This is a single-application, single-codebase deployment — a Modular Monolith in structure, but classified primarily by its runtime execution model (Pipeline + Microkernel).

### What Was Considered and Rejected

**Not Microservices:** `luigid` and workers are not independently versioned or independently deployable services with separate data stores. They are two operational roles of a single Python package sharing the same scheduler state.

**Not Event-Driven (primary):** The `Event` constants and callbacks are lightweight in-process hooks for observability, not a message broker or event bus. Task coordination flows through the scheduler's state machine (polling), not through event publication/subscription.

**Not Layered (primary):** While there is a rough layering (CLI → worker → scheduler → tasks → targets), the dominant organizing principle is the task DAG pipeline execution model, not a traditional presentation/business/data stack.

**Not CQRS:** There is no command/query responsibility separation. The scheduler maintains a single unified state and serves both mutation (task registration, status updates) and queries (get_work, task graph visualization) from the same in-memory store.

**Modular Monolith (structural, secondary):** Luigi is organized as a well-structured Python package with clear module responsibilities. The `contrib/` separation is clean. But this is a structural observation about code organization, not the dominant architectural driver. The Pipeline and Microkernel patterns better explain how Luigi *behaves* at runtime.

### Quality Attributes

**Extensibility:** The `Task`/`Target`/`FileSystem` ABCs are the stable plugin contract. Adding a new backend requires only implementing these interfaces — no changes to core. The `--module` CLI flag enables dynamic task discovery from any importable Python module.

**Idempotency:** Output targets as the completeness criterion means pipelines are inherently restartable. A re-run of any workflow skips already-completed tasks without re-execution.

**Reliability:** Per-task retry policy (`retry_count`, `disable_hard_timeout`, `disable_window`), batch email notifications, and per-task failure isolation (subprocess per task) limit blast radius of individual failures.

**Simplicity:** The three-method Task contract (`run/requires/output`) and single Python package deployment make Luigi significantly simpler to operate than multi-process orchestrators like Airflow. No separate metadata database required by default.

**Observability:** Built-in execution summary (`luigi/execution_summary.py`), task history (`luigi/task_history.py`, `luigi/db_task_history.py`), Tornado-served web UI for dependency graph visualization, pluggable Datadog/Prometheus metrics, and notification hooks.

**Portability:** Tasks can orchestrate work on any backend — Hadoop, Spark, Kubernetes, cloud storage, RDBMS — without the framework itself having cloud dependencies.
