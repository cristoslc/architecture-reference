# ksql — Architecture Classification Report
**Date:** 2026-03-11
**Repository:** https://github.com/confluentinc/ksql
**Primary Style(s):** Modular Monolith, Layered, Microkernel
**Confidence:** 0.93
**Summary:** ksqlDB is a single-process, horizontally scalable stream processing database engine built on Kafka Streams, organized as a tightly layered modular monolith with a Microkernel-style plugin system for user-defined functions.

## Evidence

- **Multi-module Maven build**: Root `pom.xml` defines 29+ modules including `ksqldb-parser`, `ksqldb-metastore`, `ksqldb-execution`, `ksqldb-engine`, `ksqldb-streams`, `ksqldb-rest-app`, `ksqldb-rest-client`, `ksqldb-cli`, `ksqldb-udf`, `ksqldb-serde`, and supporting modules. All compile and package into a single server artifact.
- **Single deployable image**: `docker-compose.yml` scales a single `ksqldb-server` image horizontally (primary + additional nodes). No separate microservices with distinct codebases. Three supporting infrastructure services (Kafka, Schema Registry, CLI client) are external dependencies, not part of the application itself.
- **Deep horizontal layering**: The engine module alone shows explicit sub-layers: `parser` → `analyzer` → `logicalplanner` → `physicalplanner` → `execution/plan` → `streams` (Kafka Streams builder). `KsqlExecutionContext` is the central interface separating the REST layer from the engine.
- **Kafka-based distributed command log**: `CommandStore` publishes DDL/DML statements to a Kafka command topic. `CommandRunner` consumes and replays these commands across all cluster nodes via `InteractiveStatementExecutor`. This is the mechanism for cluster-wide state consistency, not inter-service messaging.
- **UDF plugin system**: `UserFunctionLoader` scans a configurable `pluginDir` for annotated JARs at startup using ClassGraph. UDFs (`@UdfDescription`/`@Udf`), UDAFs (`@UdafDescription`), and UDTFs (`@UdtfDescription`) are discovered and registered into `MutableFunctionRegistry`. `ExtensionSecurityManager` sandboxes plugin code.
- **Kafka Streams as execution runtime**: `KSPlanBuilder` implements `PlanBuilder` and translates an abstract `ExecutionPlan` DAG into a live Kafka Streams topology. All stateful queries (aggregations, joins, windowed operations) run as Kafka Streams applications inside the same JVM.
- **CQRS-like read/write split**: `KsqlPlan` explicitly separates `getDdlCommand()` (DDL/write) and `getQueryPlan()` (query/read). Push queries (streaming, event-driven delivery) and pull queries (point-in-time reads from materialized state stores) are distinct code paths with their own routing classes (`PushRouting`, `HARouting`).
- **Distributed pull query routing**: `HARouting` and `PushRouting` fan out queries across cluster nodes by consulting Kafka Streams partition metadata to locate state store owners. `HeartbeatAgent` maintains liveness information for all cluster members.
- **Security extension SPI**: `KsqlSecurityExtension` and the `Endpoints` SPI in `ksqldb-rest-app/api/spi` allow external implementations to plug in custom authentication/authorization and endpoint behavior.

## Architecture Styles

### Primary: Modular Monolith
The entire ksqlDB server is a single deployable process packaged from 29+ Maven modules. There are no inter-service network calls between ksqlDB components; all module communication is through direct Java method calls and shared interfaces. The docker-compose topology confirms horizontal scaling of an identical image rather than a decomposed set of services. Module boundaries are enforced via Maven dependency declarations: `ksqldb-rest-app` depends on `ksqldb-engine`; `ksqldb-engine` depends on `ksqldb-parser`, `ksqldb-metastore`, and `ksqldb-execution`; `ksqldb-streams` depends on `ksqldb-execution`. This graph is acyclic and intentionally hierarchical.

### Secondary: Layered
The internal structure of the system follows a strict top-to-bottom dependency chain across technical layers:

1. **API/Transport Layer** (`ksqldb-rest-app`, `ksqldb-cli`): Vert.x-based HTTP server, REST resources (`KsqlResource`, `StreamedQueryResource`), WebSocket endpoint (`WSQueryEndpoint`).
2. **Engine Layer** (`ksqldb-engine`): `KsqlEngine` implements `KsqlExecutionContext`. Orchestrates parsing, analysis, planning, and query lifecycle.
3. **Analysis & Planning Layer** (`ksqldb-engine/analyzer`, `logicalplanner`, `physicalplanner`): `QueryAnalyzer` → `LogicalPlanner` → physical plan nodes.
4. **Execution Plan Layer** (`ksqldb-execution`): Abstract, backend-agnostic execution plan DAG (Step types: `StreamFilter`, `StreamAggregate`, `TableSource`, etc.).
5. **Streams Build Layer** (`ksqldb-streams`): `KSPlanBuilder` translates the abstract plan into a concrete Kafka Streams topology.
6. **Metastore/Schema Layer** (`ksqldb-metastore`, `ksqldb-serde`, `ksqldb-common`): Schema definitions, data source registry, serialization.

### Secondary: Microkernel
The UDF/UDAF/UDTF extension subsystem exhibits Microkernel (plugin) architecture characteristics:
- A stable core function registry (`MutableFunctionRegistry`) acts as the microkernel.
- `UserFunctionLoader` dynamically discovers and loads plugin JARs from a `pluginDir` at startup.
- Plugins implement annotated Java classes; the loader uses ClassGraph for annotation-based discovery.
- `UdfClassLoader` provides isolated classloading per plugin JAR.
- `ExtensionSecurityManager` applies a security sandbox to plugin code.
- The `Endpoints` SPI and `KsqlSecurityExtension` provide additional plugin points for API behavior and security.

## Quality Attributes

- **Scalability**: Horizontal scaling via multiple ksqldb-server instances sharing a Kafka command topic; Kafka Streams partitioning distributes stateful query workload.
- **Availability**: HA pull queries via standby replica routing (`HARouting`, `HeartbeatAgent`); configurable standby replicas absorb failures without rebalancing downtime.
- **Fault Tolerance**: Kafka Streams state store replication to standbys; command topic replay allows new nodes to reconstruct full cluster state.
- **Extensibility**: Microkernel UDF plugin system; security extension SPI; Endpoints SPI for API customization.
- **Maintainability**: Strong module boundary enforcement via Maven; acyclic dependency graph; abstract execution plan separates SQL frontend from Kafka Streams backend.
- **Consistency**: Command log on Kafka topic provides ordered, durable DDL/DML history; all nodes replay the same sequence ensuring eventual consistency.
- **Observability**: JMX metrics (`KsqlEngineMetrics`, `PullQueryExecutorMetrics`, `ScalablePushQueryMetrics`); processing log context; RocksDB metrics collection.
- **Performance**: RocksDB-backed state stores via `ksqldb-rocksdb-config-setter`; push query fan-out routing; rate limiting and concurrency controls in REST layer.

## Classification Reasoning

ksqlDB is architecturally a **Modular Monolith** first. The defining characteristic is deployment: one Docker image, one Java process, 29 Maven modules all compiled into a single server artifact. The `docker-compose.yml` scales identical instances, and the Kafka command topic is the coordination mechanism — not an inter-service message bus. There is no service mesh, no API gateway between components, no independent deployability of subsystems.

Within that monolith, the internal organization is strongly **Layered**. The dependency chain from REST resources down through engine, planner, execution plan, and Kafka Streams builder is explicitly directional and enforced at the build level. The `KsqlExecutionContext` interface is the canonical layer boundary between the API/REST tier and the core engine. The separation of `ksqldb-execution` (abstract plan) from `ksqldb-streams` (Kafka Streams implementation) is a deliberate design decision documented in KLIP-6, enabling backend portability.

The **Microkernel** classification is warranted by the UDF plugin subsystem's sophistication. The ability to load arbitrary JAR files from a directory at runtime, with isolated classloading and security sandboxing, is a textbook plugin architecture. The `Endpoints` and `KsqlSecurityExtension` SPIs extend this pattern to the API and security layers.

**Rejected classifications:**
- **Event-Driven**: ksqlDB processes Kafka events as its domain model but its internal architecture communicates synchronously via direct method calls. The command topic is a coordination log, not a general event bus.
- **CQRS**: While `KsqlPlan` separates DDL commands from query plans, and push/pull queries are distinct execution paths, there are no separate read and write data models in the codebase architecture sense. The same `KsqlEngine` and `MetaStore` serve both paths.
- **Pipeline**: The SQL-to-Kafka-Streams compilation process is pipeline-like, but this describes a compilation workflow, not the overall system architecture.
- **Microservices**: Single deployable process. All modules share a JVM and communicate via method calls.
- **Hexagonal Architecture**: No ports-and-adapters pattern. `KsqlResource` directly references `KsqlEngine`. Dependency injection is present but not structured as hexagonal ports.
