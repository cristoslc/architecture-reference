# squidex — Architecture Report (2026-03-11)

## Summary
Squidex is an open-source headless CMS built with ASP.NET Core (.NET 8) that exposes content through a rich REST API (OData + GraphQL + Swagger) and is deployable as a single containerized process. The system is internally organized as a **Layered Modular Monolith** with CQRS and Event Sourcing applied as key data/persistence patterns across all domain aggregates. All aggregates (App, Schema, Content, Asset, Rule, Team) are modeled as `DomainObject<T>` instances that produce and replay domain events through a pluggable event store (MongoDB, SQL, or EventStoreDB). An in-process `ICommandBus` pipeline dispatches commands through ordered middleware before they reach an aggregate root, while a separate query layer (`IContentQueryService`, `IAssetQueryService`, etc.) serves reads. Extensibility is achieved through a first-class Microkernel plugin system (`IPlugin`) and a rich built-in extensions library covering 20+ action handlers (Kafka, Slack, Algolia, Webhook, ElasticSearch, etc.).

## Evidence

### Layered
- Clear horizontal layer separation enforced by project boundaries: `Squidex.Domain.Apps.Core.Model` → `Squidex.Domain.Apps.Core.Operations` → `Squidex.Domain.Apps.Entities` → `Squidex.Web` → `Squidex` (host)
- ASP.NET Core Areas separate API, IdentityServer, and Frontend concerns (`backend/src/Squidex/Areas/Api`, `Areas/IdentityServer`, `Areas/Frontend`)
- `Startup.cs` registers services in strict layer order: infrastructure first, then domain services, then web/presentation last
- Data access isolated in `Squidex.Data.MongoDb` and `Squidex.Data.EntityFramework` with `IStore<T>` / `IPersistenceFactory<T>` interfaces

### Microkernel
- `IPlugin` interface (`backend/src/Squidex.Infrastructure/Plugins/IPlugin.cs`) defines a two-phase extension hook: `ConfigureServices` and `ConfigureServicesPost`
- `PluginManager` loads external assemblies at runtime via `McMaster.NETCore.Plugins`, sharing core Squidex assemblies
- Built-in extensions in `backend/extensions/Squidex.Extensions/` cover: Actions (Algolia, AzureQueue, ElasticSearch, Kafka, Slack, Webhook, Twitter, Typesense, OpenSearch, Discourse, Medium, Prerender, Script, SignalR, Email, CreateContent, Comment, Fastly, Notification), Asset/Azure metadata, Text indexing
- `services.AddSquidexPluginsPost(config)` runs last in `Startup.ConfigureServices` to allow plugins to override any registered service

### CQRS (qualifier)
- `InMemoryCommandBus` composes an ordered chain of `ICommandMiddleware` implementations; each aggregate is accessed through `AggregateCommandMiddleware<TCommand, TDomainObject>`
- Parallel read services: `IContentQueryService` → `ContentQueryService` reads directly from `IContentRepository` bypassing the command bus entirely
- Content write commands in `backend/src/Squidex.Domain.Apps.Entities/Contents/Commands/` (CreateContent, UpdateContent, DeleteContent, PatchContent, ChangeContentStatus…) are entirely separate from read queries in `.../Contents/Queries/` (ContentQueryService, ContentEnricher, ContentQueryParser)
- Controllers use `ICommandBus.PublishAsync()` for mutations and `IContentQueryService` for reads

### Event Sourcing (qualifier)
- `DomainObject<T>` base class (`backend/src/Squidex.Infrastructure/Commands/DomainObject.cs`) hydrates aggregate state from a persisted event stream using `IPersistenceFactory<T>.WithSnapshotsAndEventSourcing()`
- Uncommitted events accumulate in `List<Envelope<IEvent>>` and are flushed on `SaveAsync()`
- Pluggable `IEventStore` with three implementations: `MongoDb`, `Sql` (EntityFramework), and `GetEventStore` (EventStoreDB) — configured in `EventSourcingServices.cs`
- Rich domain event library in `Squidex.Domain.Apps.Events/`: 25 App events (AppCreated, AppDeleted, AppWorkflowAdded…), plus events for Assets, Contents, Schemas, Rules, Teams
- `EventConsumerManager` + `EventConsumerWorker` process outbound event streams for projections, rule execution, and subscriptions

### DDD (qualifier)
- Domain aggregates organized by bounded context: Apps, Assets, Contents, Schemas, Rules, Teams, each with dedicated Commands/, DomainObject/, Queries/, Repositories/ sub-packages
- Guard classes (`GuardApp`, `GuardSchema`, `GuardContent`) enforce domain invariants before event production
- `AppDomainObject`, `SchemaDomainObject`, `ContentDomainObject`, `AssetDomainObject`, `TeamDomainObject` are explicit aggregate roots
- Domain events separated into `Squidex.Domain.Apps.Events` project; core domain models in `Squidex.Domain.Apps.Core.Model`

## Quality Attributes
- **Extensibility**: Runtime plugin loading with shared assembly contract; 20+ built-in action plugins
- **Auditability**: Full event log per aggregate; every state change is a persisted domain event
- **Polyglot persistence**: Swappable event store and read store (MongoDB or SQL/EntityFramework)
- **Scalability**: Optional worker/non-worker clustering mode; separate messaging channels for backup, rules, and general work with configurable parallelism; optional Kafka transport
- **Observability**: Dedicated `TelemetryServices`, health checks, semantic logging via `ISemanticLog`
- **Security**: Identity Server 4/Duende embedded; JWT + OpenID Connect; OData + GraphQL API with role-based permissions
- **Deployability**: Single Docker image; Helm charts for Kubernetes; cloud deployment manifests for Azure, AWS, GCP, Heroku, Render, Vultr
- **Testability**: Clear layer boundaries with interface-driven dependencies throughout the domain
