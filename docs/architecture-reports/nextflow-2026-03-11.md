## Style Rationales

**Pipeline (primary, 0.95):** Nextflow's entire design is built on the dataflow programming model using GPars `DataflowQueue`, `DataflowVariable`, and `DataflowOperator` primitives (imported throughout `Channel.groovy`, `DataflowHelper.groovy`, and `TaskProcessor.groovy`). Every process communicates exclusively through typed channels — data moves as values through explicitly connected directed-acyclic-graph edges, never via shared mutable state or method calls between processes. The `DAG` class (`modules/nextflow/src/main/groovy/nextflow/dag/DAG.groovy`) builds a literal directed acyclic graph of `PROCESS`, `OPERATOR`, and `ORIGIN` vertices connected by `Edge` objects carrying read/write channel references. The `TaskProcessor` creates a `DataflowOperator` for each process instance and binds its input/output channels so that work flows automatically when data is available. The channel operator library (`extension/`) contains ~25 operators (map, filter, collect, join, branch, mix, groupTuple, etc.) that compose higher-order pipeline graphs. The Nextflow DSL keyword is `workflow`, and a Nextflow script is literally a description of a data pipeline.

**Microkernel (primary, 0.91):** Nextflow uses PF4J as its plugin runtime (`org.pf4j.PluginManager`, `org.pf4j.ExtensionPoint`) — visible in `ExecutorFactory`, `PluginExtensionPoint`, and the `settings.gradle` which declares ten independently versioned plugin projects: `nf-amazon`, `nf-google`, `nf-azure`, `nf-k8s`, `nf-tower`, `nf-wave`, `nf-cloudcache`, `nf-codecommit`, `nf-console`, `nf-seqera`. The core runtime provides three formal extension points via `PluginExtensionPoint`: `@Factory` (new channel factories), `@Operator` (new channel operators), and `@Function` (new DSL functions). Executors are also plugin-extended — `ExecutorFactory` maps built-in executor names (local, slurm, sge, lsf, pbs, k8s, etc.) and also dynamically discovers `Executor` subclasses from the PF4J plugin manager at runtime. The ADR `adr/20250922-plugin-spec.md` formalises a JSON schema-based plugin specification so external tooling (language servers, registries) can introspect plugin capabilities without loading them into the JVM, a canonical microkernel design pattern.

**Modular Monolith (secondary, 0.84):** The repository is a Gradle multi-project monorepo with five internal modules (`nextflow`, `nf-commons`, `nf-httpfs`, `nf-lang`, `nf-lineage`) and ten plugin sub-projects under `plugins/`, all declared in `settings.gradle`. The `nextflow` module is the orchestration core; `nf-lang` contains the DSL parser, type system, and process/workflow definitions; `nf-commons` provides shared utilities; `nf-lineage` implements data provenance tracking. These modules share the same JVM process at runtime (not separate network services) and communicate through direct object references and the GPars dataflow bus, not over a network. The modules are independently compilable and testable units within a single deployable application.

## Quality Attributes

| QA | Evidence |
|---|---|
| **Portability** | Executor abstraction (`Executor.groovy`) decouples pipeline logic from execution backend; 18+ built-in executors (local, slurm, sge, lsf, pbs, condor, k8s, etc.) and cloud plugins (AWS Batch, GCP Batch, Azure Batch via `nf-amazon`, `nf-google`, `nf-azure`); container support (Docker, Singularity, Podman, Apptainer, Charliecloud, Shifter) configured independently of pipeline code |
| **Extensibility** | PF4J plugin system with three formal extension points (`@Factory`, `@Operator`, `@Function`) and plugin-discoverable executors; 10 bundled cloud/platform plugins; JSON plugin spec schema enables third-party plugin ecosystem |
| **Reproducibility** | Content-addressed task caching via `TaskHasher` and `CacheDB`; `CachedTaskHandler` short-circuits re-execution of identical tasks; `nf-lineage` module records full data provenance; container and conda/spack environment pinning |
| **Scalability** | Dataflow model naturally parallelises all processes; `TaskPollingMonitor` and `BatchContext` manage concurrent task submission to HPC schedulers and cloud batch services; Kubernetes executor (`nf-k8s`) scales to pod-per-task on Kubernetes |
| **Observability** | `TraceObserverV2` lifecycle event interface; `TraceFileObserver`, `ReportObserver`, `TimelineObserver`, `GraphObserver` produce trace CSV, HTML report, timeline HTML, and DAG SVG; `nf-tower`/`nf-seqera` plugins stream events to Seqera Platform |
| **Composability** | Workflow DSL supports named sub-workflows with declared `take:` / `emit:` interface blocks; module system (`IncludeDef`, `ScriptMeta`) enables reusable, parameterised workflow components across repositories |

## Domain

Scientific workflow orchestration platform for bioinformatics and computational research. Core domains: dataflow pipeline authoring (DSL), distributed task execution across HPC schedulers and cloud batch services, container and package environment management, task result caching and reproducibility, data lineage and provenance, workflow observability and reporting.

## Production Context

- Published in Nature Biotechnology (2017); widely adopted across bioinformatics, genomics, and life sciences
- Used in production at research institutions, clinical laboratories, and cloud-scale genomics platforms (AWS, GCP, Azure)
- nf-core community maintains 100+ peer-reviewed, production-grade Nextflow pipelines
- Seqera Platform (commercial offering) integrates directly via `nf-tower`/`nf-seqera` plugins
- Supports 18+ HPC schedulers and job runners; Docker, Singularity, Podman, and Conda/Spack for software environments
