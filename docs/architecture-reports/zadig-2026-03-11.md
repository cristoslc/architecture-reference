# zadig — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/koderover/zadig
**Classification:** Microservices + Pipeline
**Confidence:** 0.88

## Summary

Zadig is an AI-powered, cloud-native DevOps platform built as a true microservices system: fourteen independently deployable binaries (each with its own `cmd/` entry point and Dockerfile), coordinated through an API gateway (Gloo Edge), with asynchronous workflow execution expressed as a sequential Pipeline of stages and concurrent job pools. The dominant pattern is Microservices at the deployment/process topology layer, with Pipeline as the primary expression of its core domain value — CI/CD workflow orchestration. LLM integration (OpenAI, DeepSeek, Aliyun Bailian, and others) is present as a feature layer for DevOps analytics, not as an autonomous agent architecture.

### Service Inventory

The Makefile's `MICROSERVICE_TARGETS` enumerates eleven deployable services: `aslan`, `cron`, `executor`, `hub-agent`, `hub-server`, `init`, `jenkins-plugin`, `packager-plugin`, `predator-plugin`, `ua`, `user`, `warpdrive`. Together with `zadig-agent` and `zgctl`, the repository defines fourteen independent deployment units.

Key service roles:

| Service | Role |
|---|---|
| `aslan` | Monolithic core: project, environment, service, workflow, build, delivery, sprint management, and system administration APIs |
| `user` | Authentication, RBAC, Dex federation (LDAP, OAuth2, AD, GitHub) |
| `hub-server` / `hub-agent` | WebSocket reverse tunnel enabling Zadig to manage remote Kubernetes clusters without direct network access |
| `cron` | Scheduled trigger service for recurring workflows and cleanup |
| `jobexecutor` | In-cluster Kubernetes job runner: receives tasks, executes containerised steps |
| `zadig-agent` | VM-based polling agent: registers with server, polls for tasks over HTTP, executes on bare-metal/VMs |
| `reaper` / `predator-plugin` / `packager-plugin` | Specialised plugin executors injected into job pods |

### Communication Patterns

**API Gateway**: Gloo Edge routes external requests to backend services, providing a single ingress point.

**Synchronous (REST/gRPC)**: Services communicate synchronously via HTTP REST. gRPC (`google.golang.org/grpc v1.76.0`) is present as a dependency, primarily for Kubernetes operator interactions and select integrations.

**Asynchronous (Redis pub/sub)**: The `pkg/tool/eventbus/redis` package implements a Redis-backed event bus used for workflow lifecycle signals: cancellation (`workflowctl-cancel-{name}-{id}`) and debug breakpoint control (`workflowctl-debug-lock-{name}-{id}`). Redis is also used for distributed locking (`cache.NewRedisLock`).

**NSQ**: Referenced in code comments as a legacy downstream mechanism for task dispatch. The current primary queue store is MongoDB (`WorkflowQueueColl`), with Redis used for concurrency control on the queue consumer.

**Agent polling**: The `zadig-agent` binary (VM-based) uses an HTTP polling model to obtain tasks from `aslan`, avoiding the need for inbound network access to VMs.

**Reverse tunnel**: `hub-agent` deployed in remote clusters dials back to `hub-server` over a persistent WebSocket (via `remotedialer`), allowing `aslan` to manage multi-cluster Kubernetes environments without VPN or direct connectivity.

### Workflow Engine (Pipeline Architecture)

The workflow engine inside `aslan` is the clearest architectural signal:

- `WorkflowV4` model: a workflow is an ordered list of `Stages`, each containing `Jobs`.
- Stage execution is **sequential** (`RunStages` iterates stages in order, halts on stopped status).
- Job execution within a stage is **concurrent** (worker pool, configurable concurrency level).
- Each job type is dispatched via a factory (`initJobCtl`) that instantiates the correct `JobCtl` implementation (30+ job types: `ZadigBuildJobCtl`, `HelmDeployJobCtl`, `K8sBlueGreenDeployJobCtl`, `ApprovalJobCtl`, etc.).
- Status transitions follow a priority-ordered state machine: `Pause > Reject > Cancelled > Timeout > Failed > Passed > Unstable`.
- Manual approval gates can interrupt stage progression (native, Lark, DingTalk, WorkWX).
- Webhook triggers (Git hooks, Jira hooks, Meego hooks, general webhooks) feed pipeline execution.

This is a **Pipeline** pattern: a directed, ordered sequence of stages, with plug-in job implementations and optional approval gates — not a DAG engine.

### LLM / AI Integration

`pkg/tool/llm` defines an `ILLM` interface supporting eight providers (OpenAI, DeepSeek, DeepSeek-SiliconCloud, Azure OpenAI, Azure AD OpenAI, Aliyun Bailian, Volcengine Ark, Huawei MaaS). All use an OpenAI-compatible client.

Current usage:
- `pkg/microservice/aslan/core/stat/service/ai/ai_analysis.go`: DevOps statistics analysis — sends formatted project metrics to LLM for trend analysis, performance evaluation, and natural language querying of CI/CD data.
- LLM provider credentials managed via CRUD API (`/api/aslan/system/llm/integration`).

The LLM integration is a feature-level enhancement (AI-assisted analytics), not a control-plane or orchestration mechanism. This does not qualify as a Multi-Agent architecture.

### Internal Structure of Aslan

Despite being deployed as a single binary, `aslan` contains well-separated domain modules under `pkg/microservice/aslan/core/`:

```
application/, build/, code/, collaboration/, common/, cron/,
delivery/, environment/, log/, multicluster/, plugin/,
project/, release_plan/, service/, sprint_management/,
stat/, system/, templatestore/, ticket/, vm/, workflow/
```

Each domain module follows a consistent three-layer structure: `handler/` (Gin HTTP handlers) → `service/` (business logic) → `repository/` (MongoDB/MySQL access). This is Layered packaging within the Aslan process, though it does not elevate Aslan to a Modular Monolith at the deployment level — it remains one deployable unit among many.

## Classification Reasoning

**Primary style — Microservices (0.85 weight):**
Fourteen independently buildable and deployable binaries, each with its own Dockerfile, `main.go`, and distinct `pkg/microservice/<name>` implementation package. Services communicate over HTTP REST via an API gateway. The Makefile explicitly names `MICROSERVICE_TARGETS`. Each service has a distinct bounded responsibility (auth, core platform, cluster tunneling, job execution, scheduling).

**Secondary style — Pipeline (0.80 weight):**
The core business value of Zadig is CI/CD workflow orchestration. The workflow engine is architected as a sequential pipeline of named stages, each containing a concurrent pool of heterogeneous jobs dispatched via a factory/strategy pattern. This is not merely a feature — it is the dominant internal topology of the system's primary service. Stage-sequential with intra-stage concurrency, state machine status aggregation, and approval-gate interruption are all canonical Pipeline characteristics.

**Why not Event-Driven as a style:**
Redis pub/sub is used narrowly for workflow control signals (cancel, debug). NSQ appears as a legacy reference. There is no system-wide event backbone; services are not structured around event topics or message contracts. Event-Driven is an architectural qualifier, not the dominant style.

**Why not Modular Monolith:**
Aslan has clear internal modular structure, but the overall system is deployed as multiple independent processes. The internal layering within Aslan is a qualifier (layered internal structure), not evidence of a Modular Monolith at the system level.

**Why not Multi-Agent:**
LLM integration exists as a stateless analytics feature (send metrics, receive insights). There are no autonomous agents, agent loops, tool-calling orchestration, or multi-agent coordination patterns. The `zadig-agent` is a CI/CD execution agent, not an LLM-based AI agent.

## Architecture Qualifiers

- **Layered internal structure**: Each microservice (notably `aslan`) follows a consistent handler → service → repository layering.
- **Event-Driven signals**: Redis pub/sub used for workflow lifecycle events (cancel, debug breakpoints).
- **Reverse tunnel / sidecar**: `hub-agent` uses a WebSocket reverse dial pattern for multi-cluster access.
- **Plugin/extension model**: Job types are factory-dispatched; 30+ job type implementations conform to the `JobCtl` interface.
- **Polling agent**: `zadig-agent` uses HTTP polling for VM-based task dispatch.
- **AI-augmented analytics**: LLM integration for DevOps metric analysis and natural language querying.
- **Multi-cluster topology**: Native support for managing multiple Kubernetes clusters from a single control plane.

## Confidence Notes

Confidence is 0.88. The Microservices classification is unambiguous (14 Dockerfiles, 14 `cmd/` entry points, API gateway). The Pipeline secondary classification is well-supported by the workflow engine design. A small discount is applied because `aslan` handles a very broad surface area (it could arguably be called Service-Based given how much business logic lives in one service), and NSQ's historical role is unclear from current code — the messaging backbone may have been more central in an earlier architecture version.
