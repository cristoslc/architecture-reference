## Executive Summary

Redis is a high-performance in-memory data structure server written entirely in C. Its architecture is best classified as a **Modular Monolith** (primary), **Event-Driven** (secondary), and **Microkernel** (secondary). All functionality is compiled and deployed as a single process; functional concerns are cleanly separated into discrete source modules. The core runtime is a single-threaded non-blocking event loop (`ae.c`) that dispatches all I/O and timer events, with optional multithreaded I/O offload via `iothread.c`. A formal module extension system (`module.c`, `redismodule.h`) allows third-party shared libraries to extend data types, commands, and event hooks at runtime—the clearest evidence for the Microkernel secondary style.

## Architecture Styles

### Primary: Modular Monolith (0.92)

The entire Redis server compiles into a single process (`redis-server`), sharing a unified global state struct `redisServer` (defined in `server.h`, instantiated as `struct redisServer server` in `server.c`). Functional concerns are cleanly separated by source file into cohesive, internally coherent modules:

- **Networking layer**: `networking.c`, `connection.h/c`, `anet.c`, `tls.c`, `socket.c`
- **Event loop engine**: `ae.c` + platform backends `ae_epoll.c`, `ae_kqueue.c`, `ae_evport.c`, `ae_select.c`
- **Data types**: `t_string.c`, `t_hash.c`, `t_list.c`, `t_set.c`, `t_zset.c`, `t_stream.c`
- **Data structures**: `dict.c`, `quicklist.c`, `ziplist.c`, `listpack.c`, `rax.c`, `sds.c`, `adlist.c`, `ebuckets.c`
- **Persistence**: `rdb.c` (RDB snapshots), `aof.c` (Append-Only File), `rio.c` (buffered I/O abstraction)
- **Replication**: `replication.c`
- **Clustering**: `cluster.c`, `cluster_asm.c`, `cluster_legacy.c`, `cluster_slot_stats.c`
- **Pub/Sub**: `pubsub.c`
- **Scripting**: `eval.c` (EVAL/Lua), `functions.c`, `script.c`, `script_lua.c`
- **Security**: `acl.c`, `tls.c`
- **Eviction / Expiry**: `evict.c`, `expire.c`, `defrag.c`
- **Background I/O**: `bio.c` (worker threads for close, fsync, lazy-free)
- **Observability**: `slowlog.c`, `tracking.c`, `latency.c`
- **High Availability**: `sentinel.c` (operates as an alternative server mode)
- **Module runtime**: `module.c`

These modules share `server.h` as a central type manifest and exchange data through well-defined function calls and shared struct pointers. There is no network boundary or process boundary between them; they co-reside in a single address space.

### Secondary: Event-Driven (0.85)

The runtime is structured around an event-driven concurrency model:

- **Event loop core** (`ae.c`): A purpose-built, platform-abstracted event loop supporting file I/O events (readable/writable/barrier) and time events. The loop abstracts `epoll`, `kqueue`, `evport`, and `select` via compile-time inclusion of the appropriate backend file. The main `aeMain()` function drives the entire server lifecycle.
- **`serverCron()`** (`server.c:1483`): A time event registered in the loop that fires at configurable `hz` frequency (default 10 Hz), coordinating expiry, eviction, replication heartbeats, stats collection, and memory checks.
- **`beforeSleep()` / `afterSleep()`** (`server.c:1857`): Hooks called around each event loop poll, handling client writes, AOF flushing, keyspace notifications, incremental hash rehashing, and lazy expiry.
- **Background worker threads** (`bio.c`): Three pthread workers for slow async work: `bio_close_file`, `bio_aof` (fsync), `bio_lazy_free`. They communicate with the event loop via a completion pipe that triggers a readable event, keeping the main loop non-blocking.
- **Multithreaded I/O** (`iothread.c`): Optional pool of IO threads (configurable via `io-threads`, default 1 = single-threaded). IO threads handle reading and writing client buffers concurrently with the main loop, which retains exclusive control over command execution. This is a read/write offload pattern, not a parallel execution model.
- **Keyspace notifications** (`pubsub.c`, `module.c:9162–9412`): The server fires named events (e.g., `set`, `expired`, `del`) that modules and pub/sub subscribers react to asynchronously—a classic event-driven publish/subscribe bus internal to the process.
- **Event Notifier** (`eventnotifier.c/h`): A thin abstraction over `eventfd` or a pipe pair used to wake the event loop from IO threads and background workers, avoiding busy-wait.

### Secondary: Microkernel (0.75)

Redis exposes a formal, versioned extension mechanism via its Modules API:

- **`redismodule.h`** defines the public module API surface (versioned `REDISMODULE_APIVER_1`), including callbacks, data type registration, keyspace event subscriptions, timer registration, server hooks, and custom command registration.
- **`module.c`** (~15,559 lines, the largest file in the codebase) implements the module runtime: shared library loading via `dlopen`, lifecycle management via `RedisModule_OnLoad()` entry points, a plugin isolation layer using `RedisModuleCtx`, shared API cross-module exchange (`sharedapi` dict), and event hook registration (`moduleNotifyKeyspaceEvent`, server event callbacks at `module.c:408`).
- **Built-in internal modules** (`modules/` directory): `redisbloom`, `redisearch`, `redisjson`, `redistimeseries`, `vector-sets` are maintained as loadable module extensions rather than compiled-in features, reflecting genuine microkernel extensibility rather than monolithic inclusion.
- **Module event hooks**: Modules subscribe to `REDISMODULE_EVENT_*` server lifecycle events (server start/end, client connect/disconnect, replication state, cron, keyspace changes) enabling plugin-style reactive behavior.
- **Module types**: Modules can register new Redis object types via `RedisModule_CreateDataType()`, with custom RDB save/load, AOF rewrite, free, copy, and defrag callbacks—extending the core data model.

The core server (`server.c`) acts as a minimal kernel: managing the event loop, the command dispatch table, connection lifecycle, and the persistence/replication machinery. Extensions plug in at runtime via the Module API rather than by modifying core source.

## Quality Attributes

### Performance
Redis is built for sub-millisecond latency. The single-threaded event loop eliminates lock contention on the hot path. All data structures (`dict`, `quicklist`, `listpack`, `rax`, `sds`) are custom-implemented in C for cache efficiency. `jemalloc` is the default allocator. The optional IO-thread model (`iothread.c`) offloads network read/write from the main thread to scale throughput on multi-core systems without sacrificing command serialization guarantees.

### Availability
Sentinel mode (`sentinel.c`, 5,474 lines) provides automatic failover: distributed monitoring, subjective/objective down detection (SRI_S_DOWN / SRI_O_DOWN), leader election, replica promotion, and client notification. Redis Cluster (`cluster_asm.c`, `cluster_legacy.c`) provides partition-aware routing with 16,384 hash slots, gossip-based failure detection, and slot migration.

### Durability / Reliability
Two persistence mechanisms: RDB point-in-time snapshots (child process via `fork()`); AOF write-ahead log with configurable fsync policies (always/everysec/no). The `bio_aof` background thread handles fsync without blocking the event loop. Multi-part AOF manifests (`aof.c`) track BASE + INCR + HISTORY files for safe atomic rewrite.

### Extensibility
The Modules API (versioned, ~15K LOC runtime in `module.c`) allows third parties to add new data types, commands, event hooks, timers, and info sections without modifying core source. Internal modules (`redisearch`, `redisjson`, etc.) demonstrate production-grade extensibility via this mechanism.

### Scalability
Horizontal scalability via Redis Cluster with sharding across up to 1000 nodes. Vertical I/O scalability via configurable IO threads (`io-threads` config, max 128). Pub/Sub shard channels distribute message fan-out within cluster slots (`pubsubshard_channels`).

### Observability
Slow log (`slowlog.c`), latency monitor (`latency.c`), client-side caching invalidation tracking (`tracking.c`), instantaneous metrics (commands/sec, net I/O, event loop cycle/duration), INFO command exposing ~100 server stats, and keyspace notification events for external monitoring integration.

## Architectural Decision Notes

1. **Single-threaded command execution is a deliberate design choice** for atomicity and simplicity. Redis commands are not atomic in the OS sense—they rely on the event loop serializing all command execution. This eliminates the need for in-data-structure locking.

2. **`fork()` for persistence** is a key design pattern: RDB snapshots and AOF rewrites are performed in a forked child process, leveraging OS copy-on-write semantics. The parent continues serving clients with zero blocking (beyond the fork syscall itself).

3. **The Sentinel is a separate operating mode**, not a separate binary. `sentinel.c` is conditionally activated via `server.sentinel_mode`. This is monolithic deployment with mode switching, not a separate service.

4. **Connection abstraction** (`connection.h`) provides a vtable-based polymorphism over TCP, TLS, and Unix socket connections, allowing the networking layer to be transparent to command handlers—a clean separation of transport from protocol.

5. **RESP protocol** (`resp_parser.c`) is the wire protocol for all client/server and inter-node communication, with RESP3 (`CLIENT TRACKING`) for client-side caching invalidation.
