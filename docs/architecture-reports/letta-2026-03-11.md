# letta — Architecture Classification Report
**Date:** 2026-03-11
**Repository:** https://github.com/letta-ai/letta
**Primary Style(s):** Layered, Multi-Agent
**Confidence:** 0.92
**Summary:** Stateful AI agent platform built as a layered monolith with first-class multi-agent orchestration, an extensible plugin/tool-executor subsystem, and a streaming REST/WebSocket API.

## Evidence

- `letta/server/rest_api/routers/v1/` — 30+ FastAPI route files organized by resource (agents, blocks, messages, tools, groups, runs, etc.), constituting a dedicated presentation layer.
- `letta/services/` — ~30 manager/service classes (`AgentManager`, `MessageManager`, `GroupManager`, `ToolManager`, `StreamingService`, `LLMBatchManager`, etc.) forming a cohesive application-service layer.
- `letta/agents/` — Abstract `BaseAgent` + concrete implementations: `LettaAgent`, `LettaAgentV2`, `LettaAgentV3`, `VoiceAgent`, `VoiceSleeptimeAgent`, `EphemeralAgent`.
- `letta/groups/` — Dedicated multi-agent coordination patterns: `RoundRobinMultiAgent`, `SleeptimeMultiAgent`, `DynamicMultiAgent`, `SupervisorMultiAgent`.
- `letta/services/tool_executor/` — Factory-dispatched tool executor hierarchy: `LettaCoreToolExecutor`, `LettaMultiAgentToolExecutor`, `ExternalMCPToolExecutor`, `SandboxToolExecutor`, `ComposioToolExecutor` (Microkernel-style extension point).
- `letta/services/tool_executor/multi_agent_tool_executor.py` — Agents communicate with each other via first-class inter-agent messaging tools: `send_message_to_agent_and_wait_for_reply`, `send_message_to_agent_async`, `send_message_to_agents_matching_tags`.
- `letta/orm/` — SQLAlchemy ORM models with pgvector for semantic memory, representing the data-access layer.
- `letta/plugins/plugins.py` — Protocol-based plugin registry (`SummarizerProtocol`, `DEFAULT_PLUGINS`) supporting runtime-swappable implementations loaded via `importlib`.
- `letta/llm_api/llm_client.py` — Factory selecting from 15+ provider-specific LLM client adapters (OpenAI, Anthropic, Google, Azure, Bedrock, etc.) via a match-case dispatch.
- `letta/schemas/enums.py` — `AgentType` enum lists distinct agent kinds: `memgpt_agent`, `letta_v1_agent`, `react_agent`, `workflow_agent`, `split_thread_agent`, `sleeptime_agent`, `voice_convo_agent`, `voice_sleeptime_agent`.
- `compose.yaml` — Single `letta_server` container + pgvector PostgreSQL + nginx; confirms single-deployable monolith (not microservices).
- `pyproject.toml` — `temporalio`, `apscheduler`, `mcp`, `fastmcp` among dependencies, indicating temporal workflow, scheduled polling, and Model Context Protocol (MCP) integration.
- `letta/jobs/scheduler.py` — Distributed leader-election advisory lock pattern for background batch polling.
- `letta/services/memory_repo/` — Git-backed and memfs-backed hierarchical memory storage per agent.

## Architecture Styles

### Primary: Layered
The entire codebase follows a strict horizontal layering pattern:
- **Presentation**: `letta/server/rest_api/routers/`, `letta/server/ws_api/`, `letta/cli/`
- **Application/Service**: `letta/services/` (all manager classes), `letta/server/server.py`
- **Domain/Agent**: `letta/agents/`, `letta/groups/`, `letta/functions/`
- **Infrastructure/Data**: `letta/orm/`, `letta/adapters/`, `letta/llm_api/`

Dependencies flow strictly downward. REST routers call services; services call ORM; ORM hits PostgreSQL. This pattern is unambiguous and consistent across all 30+ resource types.

### Primary: Multi-Agent
Letta is fundamentally a platform for composing autonomous AI agents that communicate with one another:
- `letta/groups/` implements four distinct multi-agent topologies: round-robin, sleeptime background processing, dynamic (manager selects next speaker), and supervisor (one agent directs many).
- `LettaMultiAgentToolExecutor` exposes inter-agent messaging as first-class callable tools available to any agent at runtime.
- `AgentType` enum has dedicated entries for group/sleeptime/voice coordination variants.
- The platform's stated mission ("stateful agents... that can learn and self-improve") centers on multi-agent self-improvement loops.

### Secondary: Microkernel
A plugin registry (`letta/plugins/plugins.py`) provides protocol-based extension points:
- Pluggable summarizers via `SummarizerProtocol`
- `ToolExecutorFactory` dispatches to seven distinct executor implementations by `ToolType`
- `LLMClient` factory selects from 15+ interchangeable provider backends
- MCP (Model Context Protocol) integration for external tool servers (`letta/services/mcp/`)

## Quality Attributes

- **Statefulness** — Core design goal; per-agent persistent memory (core, archival, filesystem, summary blocks) backed by PostgreSQL with pgvector.
- **Extensibility** — Plugin system, tool executor factory, and LLM provider factory allow behavior to be swapped without modifying core logic.
- **Observability** — OpenTelemetry tracing (`trace_method` decorator throughout), Sentry integration, ClickHouse traces, DataDog metrics.
- **Scalability** — Async-first (asyncio throughout), Redis-backed SSE stream manager, distributed scheduler leader election, batch LLM processing via Anthropic Batch API.
- **Modularity** — Services are individually instantiated and injected; no circular imports enforced by layering.
- **Streaming** — SSE and WebSocket streaming paths are first-class alongside synchronous REST.
- **Multi-tenancy** — Organization/user/project model with per-org access control in ORM mixins.
- **Interoperability** — 15+ LLM provider adapters; MCP client/server support; OpenAI-compatible chat completions interface (`letta/server/rest_api/routers/v1/chat_completions.py`).
- **Sandboxing** — Tool execution isolated via local sandbox, E2B sandbox, and Modal serverless sandboxes.

## Classification Reasoning

Letta (formerly MemGPT) is a **Layered** platform at the structural level and a **Multi-Agent** platform at the functional/product level. These two styles are complementary, not competing.

The layered pattern is the dominant structural organizer: every feature — agents, memory, tools, groups, files, providers — is accessed through the same three-tier stack (REST router → service manager → ORM). This is not a modular monolith with domain boundaries; the layers cut horizontally across all domains. The codebase is a single deployable artifact with one database, confirmed by `compose.yaml`.

The multi-agent pattern is not incidental. `letta/groups/` contains four concrete multi-agent coordination strategies with first-class inter-agent messaging tools. Agents can asynchronously spawn other agents, coordinate in named groups, and have dedicated executor infrastructure (`LettaMultiAgentToolExecutor`). The platform's identity and product differentiation are rooted in multi-agent statefulness. This distinguishes Letta from a simple chatbot backend.

The Microkernel secondary style is well-evidenced but narrower in scope: it applies to the plugin registry, tool executor dispatch, and LLM provider selection — all extension points within the larger layered structure, not a separate architectural tier.

Confidence is 0.92. The layered pattern is unambiguous (0.97). The multi-agent designation is strongly supported (0.92) but Letta the platform is the multi-agent runtime, not itself organized as autonomous agents. The Microkernel secondary (0.85) is clearly present but limited in scope relative to the overall system size.
