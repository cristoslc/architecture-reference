## Summary

Oppia is an open-source interactive learning platform built as a **Layered** monolith (primary) with a secondary **Modular Monolith** quality. The backend is a Python/webapp2 application deployed on Google App Engine, organized into four strict horizontal layers: controllers, domain (services + domain objects), storage (datastore models), and platform (cloud service adapters). A large Angular/TypeScript single-page application serves as the frontend presentation layer. Apache Beam batch jobs running on Google Cloud Dataflow are dispatched via Google Cloud Tasks for background processing.

## Evidence

### Layer Structure (Backend)

The `core/` directory enforces a four-layer hierarchy documented explicitly in `core/README.md`:

```
core/
  controllers/   # HTTP request handlers (webapp2 handlers)
  domain/        # Business logic: *_domain.py (entities) + *_services.py (services)
  storage/       # Datastore models (gae_models.py per aggregate)
  platform/      # Cloud service adapters (datastore, taskqueue, auth, email, cache, search)
```

- **61 controller source files** — one per feature area (editor, reader, admin, blog, classroom, contributor_dashboard, etc.)
- **118 domain source files** — paired `*_domain.py` / `*_services.py` for every major entity (exploration, skill, topic, story, question, suggestion, user, voiceover, etc.)
- **Storage layer** — each aggregate has its own `gae_models.py` (exploration, skill, topic, story, question, blog, user, suggestion, etc.) backed by Google Cloud Datastore (NDB/Firestore in Datastore mode)
- **Platform layer** — thin adapters under `core/platform/` for datastore, taskqueue, cache, auth, email, bulk_email, search, secrets, speech_synthesis, translate, storage; the `models.Registry` singleton provides a dependency injection shim that switches implementations between GAE production and development emulator

### Platform Adapter Pattern

`core/platform/models.py` exposes a `Registry` class with methods like `import_models()`, `import_datastore_services()`, `import_taskqueue_services()`, `import_cache_services()`, etc. All domain and controller code calls through this registry, enabling swap of cloud service implementations without touching business logic. This is the key architectural boundary between domain logic and infrastructure.

### Domain Objects

`core/domain/exp_domain.py` (7,245 lines) documents the pattern: "Domain objects capture domain-specific logic and are agnostic of how the objects they represent are stored." Domain objects are passed up to controllers and down to storage services, never carrying persistence logic themselves.

### Frontend Layer (Angular SPA)

`core/templates/` contains 1,349 TypeScript source files organized by feature page (40+ pages) with shared `services/`, `components/`, `domain/`, `directives/`, and `filters/`. It communicates with the backend exclusively via REST API handlers. The frontend has its own event-bus (`EventBusService` using RxJS `Subject`) for intra-frontend event coordination, but this is local to the SPA and does not constitute a system-level event-driven architecture.

### Background Processing (Apache Beam / Cloud Tasks)

`core/jobs/` defines Apache Beam batch jobs dispatched via Google Cloud Tasks queues (`QUEUE_NAME_BACKUPS`, `QUEUE_NAME_EMAILS`, `QUEUE_NAME_ONE_OFF_JOBS`, `QUEUE_NAME_STATS`, `QUEUE_NAME_VOICEOVER_REGENERATION`). Jobs run on Google Cloud Dataflow (DataflowRunner). Scheduled triggers are declared in `cron.yaml` (12 cron jobs hitting `/cron/` endpoints). This is background job processing bolted onto a monolith, not a standalone event-driven architecture.

### Deployment

`app_dev.yaml` shows a single GAE application (`runtime: python310`, `entrypoint: gunicorn`) serving all routes. Static assets are served from the same GAE app. There is no decomposition into separate services — the entire backend is one deployed unit.

### Extension System

`extensions/` contains pluggable interaction types (drag-and-drop, multiple choice, numeric input, etc.), rich text components, value generators, and answer visualizations. These are registered via Python registry modules (`interaction_registry.py`, `rte_component_registry.py`, etc.) — a lightweight plugin registry that adds extensibility to the monolith without making it a microkernel architecture (the extension points are narrow and domain-specific).

## Rejected Styles

**Not Microservices:** The entire application is a single GAE deployment unit. Controllers, domain services, and storage all share the same codebase and a single Cloud Datastore instance. Cloud Tasks queues are intra-application async dispatchers, not inter-service communication channels.

**Not Event-Driven (primary):** `core/domain/event_services.py` defines `BaseEventHandler` for recording learner interaction events, but these dispatch synchronously to stats_services within the same process. The frontend's `EventBusService` is an intra-SPA RxJS subject — not a system-level message broker. There is no Kafka, Pub/Sub broker, or asynchronous event backbone.

**Not Microkernel:** The `extensions/` plugin system and platform adapter registry exist, but the core extension points are narrow (interaction types, RTE components) and the adapters implement a fixed interface rather than a dynamic plugin lifecycle. This does not rise to a microkernel classification.

**Not CQRS or DDD:** Domain objects and services are organized around feature areas (exploration, topic, skill) but do not enforce bounded contexts, aggregates with explicit aggregate roots, domain events, or separate read/write models in the CQRS sense.

## Quality Attributes

- **Testability** — 61 controller test files, 118 domain test files; test pyramid is thorough and co-located
- **Maintainability** — strict layer separation enforced by import conventions (domain cannot import controllers, storage cannot import domain)
- **Extensibility** — platform adapter registry allows swapping cloud services; interaction plugin registry enables new question types
- **Deployability** — single GAE app with Webpack/Angular build pipeline; cron-based batch triggers
- **Scalability** — GAE auto-scaling for HTTP tier; Cloud Dataflow for batch jobs; Cloud Tasks for async work
- **Observability** — Google Cloud Logging integration; feature flags (`feature_flag_list.py`) for controlled rollouts
