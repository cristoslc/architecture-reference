## Style Rationales

**Pipeline (primary, 0.93):** Prefect's entire design centres on the Flow → Task composition model. A `@flow`-decorated function is an orchestrated pipeline; `@task`-decorated functions are individual pipeline stages. The execution path is strictly sequential: (1) a Flow Run is submitted and its state is validated by the Prefect REST API, (2) a Worker or Runner polls the API for scheduled runs and dispatches them, (3) the Flow Engine (`flow_engine.py`) executes the flow, coordinating with the API before and after every task via state-transition requests, (4) the Task Engine (`task_engine.py`) executes each task and persists results. Every state change — Pending → Running → Completed/Failed — must be approved by the server-side orchestration engine before user code may proceed, making the pipeline inherently observable and resumable. The `runner/` module implements a poller-based process that manages the full scheduling-to-execution pipeline locally (`_scheduled_run_poller.py`, `_flow_run_executor.py`, `_state_proposer.py`). The `server/services/scheduler.py` upstream creates scheduled flow runs from deployment definitions, feeding the top of the pipeline.

**Event-Driven (primary, 0.87):** Prefect has a first-class event subsystem that is architecturally central, not peripheral. The `events/` module (client-side) and `server/events/` (server-side) form a full publish-subscribe pipeline: events are emitted by flows, tasks, and workers using `emit_event()`, published via `EventPublisher` to a pluggable message broker (`PREFECT_MESSAGING_BROKER`), and consumed by background `Service` subclasses — `ReactiveTriggers`, `EventPersister`, `EventLogger`, `TaskRunRecorder`. The `server/events/triggers.py` implements a stateful event-matching engine: it evaluates `EventTrigger`, `CompoundTrigger`, and `SequenceTrigger` automations against the live event stream and fires `TriggeredAction` objects when conditions are met. Actions (`server/events/actions.py`) can cancel flow runs, change flow run state, run deployments, call webhooks, send notifications, or pause/resume work pools — a complete reactive automation loop driven entirely by events. The `ReactiveTriggers` service consumes from an `events` topic via a `Consumer` that wraps either in-memory queues or an external broker, making the broker swappable without altering the trigger logic.

**Modular Monolith (secondary, 0.82):** The codebase is a single Python package (`prefect`) that ships both the server and the client runtime together (`pyproject.toml` bundles server and client dependencies in one distribution). Internally it has strong modular boundaries enforced by directory structure: `server/` owns the REST API (FastAPI), orchestration engine, database layer, scheduler services, and event infrastructure; `client/` owns the orchestration client and schema objects; `workers/` owns infrastructure-agnostic worker base classes; `runner/` owns the process-based local runner; `events/` owns client-side event emission and automation schema. Each module can be used independently — users can run just `prefect server start` (server mode), point workers at a remote API (cloud mode), or run flows purely in-process with an ephemeral API. Despite co-deployment from one package, these layers do not cross-call each other arbitrarily: the client layer communicates with the server only through the HTTP REST API, enforcing a clean boundary. The `integrations/` workspace contains 18 independently versioned collection packages (e.g. `prefect-aws`, `prefect-dbt`, `prefect-kubernetes`) loaded via Python entry points, giving the monolith a light microkernel extension layer.

## Quality Attributes

| QA | Evidence |
|---|---|
| **Observability** | Every flow run and task run state transition is recorded in the metadata DB via the REST API; `emit_event()` publishes lifecycle events; OpenTelemetry (`opentelemetry-api`) and `prometheus-client` are bundled dependencies; `telemetry/` module provides structured instrumentation; UI served from `server/api/ui/` and separate `ui-v2/` |
| **Resilience** | State-machine orchestration with retries, pauses, and crash recovery (`core_policy.py` handles `Crashed`, `Paused`, `Cancelling` transitions); `CancellationCleanupService` and `LateRunsService` repair stale runs; `RepossessorService` reclaims stuck work from crashed workers |
| **Extensibility** | 18 independently versioned integration collections (`prefect-aws`, `prefect-kubernetes`, `prefect-dbt`, etc.) loaded via Python entry points; `Block` system provides pluggable credential/config storage; `TaskRunner` is an abstract base class enabling thread-pool, process-pool, Dask, Ray, and other parallel executors |
| **Scalability** | Work pools (`workers/`) dispatch to Docker, Kubernetes, Serverless, or process-based infrastructure; `TaskRunner` enables parallel task execution within a flow; `Runner` in server mode serves multiple flows; concurrency limits (`concurrency/`) apply global and tag-based flow and task slot controls |
| **Deployability** | Single Dockerfile builds server + dual UI in one image; `prefect server start` launches embedded server; Cloud mode points any worker at Prefect Cloud; work pools abstract infrastructure (process, Docker, Kubernetes, cloud VM); `prefect.yaml` deployment manifest drives GitOps-style deployments |
| **Evolvability** | Alembic migrations in `server/database/_migrations/`; `PrefectDBInterface` singleton abstracts SQLite vs PostgreSQL dialect differences; `PREFECT_MESSAGING_BROKER` and `PREFECT_MESSAGING_CACHE` decouple event broker from orchestration; `_experimental/` namespace for pre-release features (SLAs, bundles) |

## Domain

Workflow orchestration and data pipeline platform. Core domains: flow and task definition, deployment scheduling, state orchestration, infrastructure-agnostic work dispatch (work pools), event-driven automation (triggers + actions), result persistence, concurrency control, asset lineage, and observability.

## Production Context

- Ships as a single Python distribution (`pip install prefect`) that includes both the REST API server and the client SDK
- Self-hosted via `prefect server start` (SQLite default, PostgreSQL for scale) or managed via Prefect Cloud
- 18 integration packages cover AWS, GCP, Azure, Kubernetes, Dask, Ray, dbt, Snowflake, Databricks, Slack, and more
- Workers abstract the infrastructure layer: process, Docker, Kubernetes, and cloud-native work pools allow the same flow code to execute across different environments without modification
- Event-driven automations allow reactive workflows: a flow failure can automatically trigger a remediation deployment, send a Slack notification, or pause a work pool
