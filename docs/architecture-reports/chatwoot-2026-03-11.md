# Architecture Report: Chatwoot

**Date:** 2026-03-11
**Repo URL:** https://github.com/chatwoot/chatwoot
**Classification:** Event-Driven, Layered
**Confidence:** 0.92

---

## Summary

Chatwoot is an open-source customer engagement platform built as a Rails monolith with a deeply integrated, custom event-driven architecture layered on top. The application is deployed as a single unit (web server + Sidekiq worker processes) against a shared PostgreSQL database, but its internal communication model is unmistakably event-driven: domain models publish named events through a singleton Dispatcher, and a structured set of listeners subscribe to react asynchronously. The layered Rails conventions (controllers, services, models, jobs, policies) provide the structural scaffolding; the event-driven dispatcher is the mechanism through which cross-cutting concerns (webhooks, real-time pushes, notifications, automations, analytics) are triggered without tight coupling.

## Evidence from Code Exploration

### Deployment topology
- `Procfile` defines exactly two processes: `web` (Rails/Puma) and `worker` (Sidekiq). There is no service mesh, no inter-service HTTP, no gRPC.
- `config/database.yml` uses a single PostgreSQL database across all environments.
- `config/cable.yml` configures ActionCable over Redis for WebSocket subscriptions.
- `config/sidekiq.yml` defines a rich queue hierarchy (`critical`, `high`, `medium`, `default`, `mailers`, `scheduled_jobs`, `low`, etc.) indicating heavy reliance on background processing.

### Custom event dispatcher (`app/dispatchers/`)
- `Dispatcher` is a Singleton that holds a `SyncDispatcher` and an `AsyncDispatcher`, both inheriting from `BaseDispatcher`.
- `SyncDispatcher#dispatch` immediately publishes the event to `ActionCableListener` and `AgentBotListener` (real-time path).
- `AsyncDispatcher#dispatch` calls `EventDispatcherJob.perform_later(...)`, queuing the event on the `critical` Sidekiq queue. The job resolves the event to a method name and invokes all registered async listeners.
- Async listeners registered: `AutomationRuleListener`, `CampaignListener`, `CsatSurveyListener`, `HookListener`, `InstallationWebhookListener`, `NotificationListener`, `ParticipationListener`, `ReportingEventListener`, `WebhookListener`.

### Event types (`lib/events/types.rb`)
A module `Events::Types` defines 30+ named event constants across domains: `conversation.*`, `message.*`, `contact.*`, `inbox.*`, `notification.*`, `agent.*`, `copilot.*`, `campaign.*`, `webwidget.*`.

### Domain model event publication
34 dispatch call-sites exist across `app/models/` and `app/services/`. Key examples:
- `Conversation` dispatches `CONVERSATION_CREATED`, `CONVERSATION_UPDATED`, `CONVERSATION_BOT_HANDOFF`, and status-change events.
- `Message` dispatches `MESSAGE_CREATED`, `MESSAGE_UPDATED`, `FIRST_REPLY_CREATED`.
- `Contact` dispatches `CONTACT_CREATED`, `CONTACT_UPDATED`, `CONTACT_DELETED`, `CONTACT_MERGED`.
- `Inbox` dispatches `INBOX_CREATED`, `INBOX_UPDATED`.

### Listener pattern (`app/listeners/`)
Each listener class inherits from `BaseListener` and implements named handler methods matching event method-names (e.g., `conversation_created`, `message_updated`). Responsibilities are clearly separated:
- `WebhookListener` — outbound HTTP webhook delivery via `WebhookJob`
- `ActionCableListener` — WebSocket broadcast via `ActionCableBroadcastJob`
- `NotificationListener` — creates in-app notification records
- `AutomationRuleListener` — evaluates and fires automation rules
- `CampaignListener` — campaign trigger logic
- `ReportingEventListener` — analytics and reporting events
- `HookListener` / `InstallationWebhookListener` — integration and platform-level hooks

### Layered Rails structure (`app/`)
The application follows recognizable Rails layering:
- `controllers/api/v1`, `controllers/api/v2` — versioned HTTP API surface (JSON)
- `models/` — ActiveRecord domain entities (Conversation, Message, Contact, Inbox, Account, User, ...)
- `services/` — 48+ service object namespaces for business logic (conversations, contacts, automations, integrations per channel: facebook, slack, whatsapp, twilio, email, etc.)
- `policies/` — Pundit authorization layer
- `builders/` — complex object construction patterns
- `finders/` — encapsulated query objects
- `presenters/` — data shaping for API responses
- `jobs/` — Active Job / Sidekiq background tasks
- `mailers/` — email delivery

### Enterprise extension mechanism (`config/initializers/01_inject_enterprise_edition_module.rb`)
A GitLab-inspired module injection system (`InjectEnterpriseEditionModule`) uses Ruby's `prepend` to conditionally layer enterprise features from the `enterprise/app/` tree over the open-source core. This is a Microkernel-adjacent pattern but is scoped to feature gating, not cross-service communication.

### Frontend (`app/javascript/`)
Vue.js SPA with separate entrypoints for `dashboard`, `widget`, `portal`, `sdk`, and `superadmin`. The frontend communicates via REST API and ActionCable WebSocket, not a secondary architecture style of the backend.

## Architecture Styles Identified

### Primary: Event-Driven
**Strength: Strong**

The custom dispatcher-listener system is the central integration mechanism for all domain side-effects. Events are the first-class citizens connecting models to webhooks, real-time pushes, notifications, automations, and analytics. The system handles both synchronous (low-latency) and asynchronous (queue-backed) event paths explicitly. This is not a secondary concern — it is architecturally intentional and mirrors production event-driven designs (named event types, subscriber registration at boot, job-backed async dispatch).

### Secondary: Layered
**Strength: Strong**

The Rails MVC conventions impose clear horizontal layering: HTTP request handling (controllers) → authorization (policies) → business logic (services, builders) → data access (models/ActiveRecord) → background processing (jobs). This layering is consistently applied and the service object pattern prevents business logic from leaking into models or controllers.

### Tertiary signal: Microkernel (weak)
The `enterprise/` extension injection pattern resembles a microkernel plugin model, but it is not the primary integration mechanism and does not affect the runtime event-driven communication. Insufficient to elevate to primary classification.

## Quality Attributes

- **Extensibility:** High. The listener registration pattern allows new cross-cutting behaviors (new integrations, analytics, automations) to be added without modifying domain models. The enterprise module injection adds a clean feature-extension seam.
- **Scalability:** Medium-High. Sidekiq worker concurrency is externally configurable. Multiple queues allow priority-based processing. ActionCable scales via Redis pub/sub. Single-database constraint is the primary bottleneck.
- **Maintainability:** High for the backend. Clear separation of concerns via service objects, builders, finders, and policies keeps models and controllers focused.
- **Real-time capability:** High. ActionCable over Redis provides low-latency WebSocket delivery to agents and contacts. The sync dispatcher path ensures real-time events reach the client without waiting for the async queue.
- **Testability:** High. The service object pattern and thin controllers are amenable to unit testing. `spec/` directory is large and well-organized.
- **Operability:** Medium. Single database is a simpler operational model. Sidekiq provides monitoring UI. Multiple scheduled jobs (sidekiq-cron) add operational surface area.
- **Resilience:** Medium. Sidekiq retry logic (`:max_retries: 3`) provides basic fault tolerance for async event processing. No circuit breakers or saga patterns observed.

## Classification Reasoning

Chatwoot is classified as **Event-Driven (primary) + Layered (secondary)**.

The event-driven classification is earned by the presence of a purpose-built, first-party dispatcher-listener framework — not just incidental use of callbacks. The system defines a typed event vocabulary (`Events::Types`), routes events through both synchronous and asynchronous paths, and structures all cross-cutting domain reactions (webhooks, real-time pushes, notifications, automation, analytics) as independent, subscribing listeners. This decouples domain models from their side-effects while keeping everything within a single deployable.

The layered classification is earned by the consistent Rails architectural conventions: versioned API controllers, service objects for business logic, policies for authorization, builders and finders as supporting patterns, and Active Job/Sidekiq for background processing.

Rejected alternatives:
- **Modular Monolith**: No enforced module boundaries or namespace isolation between bounded contexts. All models share one database and are freely cross-referenced.
- **Microservices / Service-Based**: Single deployable, single database, no inter-service network communication.
- **CQRS**: No separate read/write model stack; standard ActiveRecord patterns throughout.
- **Hexagonal**: No ports/adapters abstraction; controllers call services and models directly.
- **Domain-Driven Design**: No explicit bounded contexts, aggregates, or value objects in the DDD sense; Rails ActiveRecord models serve as the domain objects.
- **Pipeline**: No sequential data transformation stages.
- **Serverless**: Traditional server + worker deployment model.
