# snuba — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/getsentry/snuba
**Classification:** Pipeline + Layered
**Confidence:** 0.89

## Summary

Snuba is Sentry's analytics storage and query broker, sitting between Sentry's application tier and ClickHouse. Its dominant architectural expression is a **Pipeline** style: every data path — ingestion, query execution, and subscription evaluation — is structured as an explicit chain of typed stages (processors, transformers, planners, executors) where each stage consumes the output of the prior one. A rigorous **Layered** organization underlies every pipeline: Datasets sit atop Entities, which sit atop Storages, which sit atop ClickHouse clusters. The layering is enforced by the initialization order (`storages → entities → datasets`) and by strict one-way dependency rules in the query path. Kafka-based ingestion and subscriptions introduce event-driven mechanics, but these are implementation mechanisms within the pipeline rather than a top-level architectural style. The entire system is deployed as a single Python package with multiple CLI entry points — a Modular Monolith in deployment, but the Pipeline + Layered characterization captures the organizing design force.

### Key Architectural Files

- `snuba/pipeline/query_pipeline.py`: `QueryPipelineStage[Tin, Tout]` is the typed base for all pipeline stages. Each stage implements `_process_data(pipe_input: QueryPipelineData[Tin]) -> Tout` and wraps results in `QueryPipelineResult`, propagating errors through the chain without exceptions.
- `snuba/pipeline/stages/query_processing.py`: `EntityProcessingStage` translates a `Request` into a `ClickhouseQuery | CompositeQuery[Table]` by running entity validators and entity query processors. `StorageProcessingStage` then applies storage-level query processors and selects the best execution plan.
- `snuba/pipeline/stages/query_execution.py`: `ExecutionStage` runs the finalized ClickHouse query and returns a `QueryResult`.
- `snuba/web/query.py`: `_run_query_pipeline` chains `EntityProcessingStage → StorageProcessingStage → ExecutionStage`, demonstrating the pipeline composition at the HTTP request level.
- `snuba/datasets/dataset.py`: `Dataset` is the top layer — a pure facade over a list of `EntityKey` references, with no query logic of its own.
- `snuba/datasets/entity.py`: `Entity` holds the logical schema (`ColumnSet`), join graph, `LogicalQueryProcessor` sequence, subscription processors, and storage connections. It is the integration point between the Dataset layer and the Storage layer.
- `snuba/datasets/storage.py`: `ReadableStorage` / `WritableTableStorage` expose `ClickhouseQueryProcessor` sequences and `AllocationPolicy` references, forming the lowest layer before the database.
- `snuba/core/initialize.py`: Enforces initialization order: `_load_storages() → _load_entities() → _load_datasets()` — a structural declaration of the layering.
- `rust_snuba/ARCHITECTURE.md`: Describes the Rust consumer pipeline as: `HealthCheck → SetJoinTimeout(0) → MessageTransformers → Reduce → SetJoinTimeout(∞) → ClickhouseWriterStep → COGS topic → CommitLog topic`. Each step is a typed processing strategy in Arroyo.
- `snuba/consumers/consumer.py`: Python Arroyo `StreamProcessor` consuming Kafka topics, producing `BytesInsertBatch` rows batched into ClickHouse via `TableWriter`.
- `snuba/subscriptions/executor_consumer.py`: Subscription execution pipeline — an Arroyo consumer that reads scheduled subscription tasks from a Kafka topic, executes the query, and produces results to a result topic.
- `snuba/web/rpc/__init__.py`: Protocol-buffer RPC handler base that selects a `RoutingStrategy` (outcomes-based or outcomes-flex-time) to route queries to full-resolution or downsampled ClickHouse tiers.
- `snuba/datasets/configuration/events_analytics_platform/storages/`: Defines four downsampled storage tiers for the EAP dataset: full, downsample_8, downsample_64, downsample_512 — a pipeline of tiered storage routing based on query time range.

### Patterns Found

**Pipeline (primary):**
Three distinct pipeline expressions exist:

1. **Query execution pipeline** — `Request → EntityProcessingStage → StorageProcessingStage → ExecutionStage → QueryResult`. Each stage is a typed `QueryPipelineStage[Tin, Tout]` that can pass errors downstream without throwing. Entity-level processors (15+ implementations in `query/processors/logical/`) and storage-level processors (20+ in `query/processors/physical/`) are applied in declared sequence. Composite queries follow the same pipeline with parallel composite variants.

2. **Ingestion pipeline** — Kafka consumer → message filter → message processor (per dataset/entity type) → batch accumulator → ClickHouse HTTP batch writer → commit log producer. In Rust: `HealthCheck → SetJoinTimeout → MessageTransformer threadpool → Reduce (open batch) → ClickhouseWriterStep (finalize) → COGS → CommitLog`. Each step is a composable `ProcessingStrategy` from the Arroyo library.

3. **Subscription pipeline** — Kafka scheduler topic → `SubscriptionSchedulerConsumer` → scheduled task topic → `SubscriptionExecutorConsumer` (runs full query pipeline per task) → result topic. The executor consumer uses the same `EntityProcessingStage → StorageProcessingStage → ExecutionStage` chain as HTTP queries.

**Layered (primary structural):**
The system is organized into four strict horizontal layers:
- **API layer**: `snuba/web/` (Flask HTTP, RPC handlers) and `snuba/subscriptions/` (streaming query API)
- **Domain model layer**: `snuba/datasets/` — Datasets, Entities, join graphs, logical query processors, schema declarations
- **Storage abstraction layer**: `snuba/datasets/storages/` and `snuba/clusters/` — ClickHouse cluster selection, allocation policies, physical query processors
- **Infrastructure layer**: `snuba/clickhouse/` (HTTP and native drivers), `snuba/redis.py` (subscription state), `snuba/migrations/` (DDL runner)

Cross-layer dependency is strictly downward. The initialization protocol enforces this at startup. No storage module imports an entity module; no entity module imports a dataset module.

**Event-Driven (mechanism, not primary style):**
Kafka underpins both ingestion and subscriptions. The Arroyo library provides `StreamProcessor` + `ProcessingStrategy` composition for all consumer processes. Multiple named Kafka topics carry distinct event types: ingest topics (errors, transactions, spans, metrics, generic-metrics, EAP items), replacement topic, commit log topic, subscription-scheduled topic, subscription-result topic, COGS accountant topic, DLQ topic. This is an event-driven transport layer enabling the pipeline and layered architecture rather than defining the top-level organizing principle.

**CQRS (structural signal):**
`ReadableStorage` and `WritableTableStorage` are distinct class hierarchies. The write path (`KafkaStreamLoader` → consumer → `TableWriter`) and the read path (`Query → ClickhouseQueryProcessor chain → Reader`) are fully separate code paths with no shared execution logic. ClickHouse's ReplacingMergeTree / CollapsingMergeTree engines handle mutation via a dedicated `replacer` consumer, reinforcing the asymmetric read/write design.

**Modular Monolith (deployment):**
All components — API server, consumers (Python and Rust), admin UI, migration runner, replacer, subscription scheduler/executor — are compiled from the same `snuba` Python package and `rust_snuba` extension. Deployment is via multiple CLI entry points (`snuba api`, `snuba consumer`, `snuba rust-consumer`, `snuba subscriptions-scheduler`, `snuba subscriptions-executor`, etc.) on the same Docker image. There are no separate network-addressable services; inter-component communication is through Kafka topics or shared Redis/ClickHouse.

### Confidence Rationale

Pipeline is chosen as the primary style because the explicit typed stage chain (`QueryPipelineStage[Tin, Tout]`) is the codified organizing pattern for both query execution and data ingestion, documented and enforced in code rather than simply emergent from function composition. Layered is chosen as co-primary because the Dataset → Entity → Storage initialization order is structurally enforced and the entire data model is described in these three layers with strictly downward dependencies. Confidence is 0.89 rather than higher because the system exhibits strong CQRS and Modular Monolith signals simultaneously, and the event-driven Kafka infrastructure is substantial enough that a reasonable observer might weight it differently.

## Quality Attributes

- **Scalability**: Horizontal scaling via Kafka partition assignment; consumer replicas auto-adjust batch time per replica count. ClickHouse distributed clusters with slice-aware topic routing. Downsampled storage tiers (1x, 8x, 64x, 512x) for query cost reduction at scale.
- **Performance**: Rust consumer pipeline for high-throughput ingestion with streaming HTTP batch insertion. Prewhere optimization, bloom filter optimizer, mapping promoter, and 20+ physical query processors applied before ClickHouse execution. Query allocation policies with per-referrer thread limits and byte read caps.
- **Reliability**: Arroyo `SetJoinTimeout(0)` idempotency zone ensures consumer rebalancing cannot corrupt partial batches. DLQ consumer for invalid messages. Subscription executor uses Kafka offset commits only after successful ClickHouse write. Health-check step touches file each event loop iteration for Kubernetes liveness probes.
- **Maintainability**: YAML-driven dataset/entity/storage configuration separates schema from behavior. Typed `QueryPipelineStage[Tin, Tout]` makes the query transformation chain explicit and testable in isolation. Extensive migration system with dangerous-migration detection and rollback support.
- **Observability**: `sentry-arroyo` metrics emitted from every consumer step. Query logging via `querylog` module and `record_query` call after every execution. COGS (Cost of Goods Sold) topic records per-query resource attribution. Allocation policy violations surfaced as structured exceptions with quota usage detail.
- **Extensibility**: New datasets/entities/storages added entirely via YAML config files. Physical and logical query processors registered and sequenced declaratively. RPC endpoint handlers auto-discovered via `RegisteredClass` introspection. Storage routing strategies pluggable via `BaseRoutingStrategy` subclasses.
- **Security**: JWT-based admin authentication with role-based access control. Mandatory condition checkers enforce project-scoping on every storage query. Allocation policies enforce per-organization and per-referrer quotas.
