## Summary

Spree is an API-first open-source e-commerce platform organized as a **Modular Monolith** with strong **Event-Driven** and **Hexagonal Architecture** characteristics. The backend is a single Rails application composed of four versioned Ruby gems (`spree_core`, `spree_api`, `spree_admin`, `spree_emails`) that share a single PostgreSQL database and deploy together. A structured service layer with strict Result/success/failure contracts sits between the API and the domain models, approaching a ports-and-adapters pattern. An explicit publish-subscribe event bus (`Spree::Events`) with async dispatch via ActiveJob/Sidekiq coordinates cross-cutting concerns such as webhooks, email notifications, exports, and analytics without creating direct coupling between subsystems. A pnpm/Turborepo monorepo alongside the Rails server houses TypeScript SDKs (`@spree/sdk`, `@spree/admin-sdk`, `@spree/next`) that communicate with the Rails backend exclusively through its REST API.

## Styles Identified

### Modular Monolith (primary, 0.93)

The entire backend deploys as a single Rails process against a single PostgreSQL database. Four gem modules (`spree_core`, `spree_api`, `spree_admin`, `spree_emails`) form the production application; each is a Rails Engine with isolated namespace, its own controllers, models, services, migrations, and tests. The umbrella `spree.gemspec` composes them via explicit gem dependencies. The `server/` directory is the host Rails application that mounts `Spree::Core::Engine` at `/` and thin-wires all gem engines into one deployable process.

Intra-module boundaries are enforced by gem isolation (separate gemspecs, separate Gemfiles, separate spec suites), not by network calls. There is no inter-process communication between the four Spree engines. The database schema (`server/db/schema.rb`, 2012 lines) is unified — all 160+ ActiveRecord models in `spree_core` map to a single shared database.

The JavaScript packages (`packages/`) are a parallel monorepo (pnpm workspaces + Turborepo) producing consumable npm libraries (`@spree/sdk`, `@spree/admin-sdk`, `@spree/next`). These are client-side delivery artifacts, not separate services; they communicate with the single Rails backend via HTTP.

**Key evidence:**
- `spree/spree.gemspec` — umbrella gem depending on `spree_core` and `spree_api`
- `spree/core/spree_core.gemspec`, `spree/api/spree_api.gemspec`, `spree/admin/spree_admin.gemspec` — four gem modules
- `server/config/routes.rb` — single `mount Spree::Core::Engine, at: '/'`
- `server/db/schema.rb` — 2012-line unified database schema
- `docker-compose.yml` — one `web` container, one `worker` container (Sidekiq), single `postgres` and `redis` services
- `spree/core/lib/spree/core/engine.rb` — `Rails::Engine` with `isolate_namespace Spree`

### Event-Driven (secondary, 0.85)

Spree ships a fully realized pub-sub event system (`Spree::Events`) backed by `ActiveSupport::Notifications` with a pluggable adapter interface (`Spree::Events::Adapters::Base`). Events are published with `Spree::Events.publish(name, payload, metadata)` and consumed by classes that extend `Spree::Subscriber`. The subscriber DSL supports `subscribes_to 'order.*'` wildcard patterns and per-event handler routing via `on 'payment.complete', :handle_complete`. Async delivery is the default: `Spree::Events::SubscriberJob` (a Sidekiq-backed ActiveJob) receives serialized event hashes and reconstructs them before calling the subscriber class.

Cross-cutting concerns are decoupled through this bus:
- `Spree::ExportSubscriber` — reacts to export lifecycle events
- `Spree::InvitationEmailSubscriber` — sends invitation emails on user invite events
- `Spree::ProductMetricsSubscriber` — tracks product view/engagement analytics
- `Spree::ReportSubscriber` — schedules report generation
- Webhook delivery — `Spree::WebhookEndpoint` / `Spree::WebhookDelivery` models record and replay event payloads to external HTTP endpoints

Event subscribers are registered in `Spree.subscribers` (an array on `app.config.spree`) and activated at Rails boot by `Spree::Events.activate!`. In development, Zeitwerk code-reload is handled by resolving subscriber constants fresh after each reload. The system also provides `Spree::Events.disable { }` for scoped suppression in tests and bulk operations.

**Key evidence:**
- `spree/core/lib/spree/events.rb` — full publish/subscribe/unsubscribe API with wildcard support and async flag
- `spree/core/lib/spree/events/adapters/active_support_notifications.rb` — pluggable adapter
- `spree/core/app/jobs/spree/events/subscriber_job.rb` — async subscriber dispatch via ActiveJob
- `spree/core/app/models/spree/subscriber.rb` — subscriber base class with DSL
- `spree/core/app/subscribers/spree/` — event_log, export, invitation_email, product_metrics, report subscribers
- `spree/core/app/models/spree/webhook_endpoint.rb` / `webhook_delivery.rb` — outbound webhook integration

### Hexagonal Architecture (secondary, 0.72)

The service layer in `spree/core/app/services/spree/` is the most structurally significant Hexagonal characteristic. Twenty-nine service namespaces covering the full commerce domain (`Cart`, `Checkout`, `Orders`, `Payments`, `Products`, `Shipments`, `Account`, `Addresses`, `GiftCards`, `Variants`, `Wallet`, etc.) are implemented as callable objects with a uniform `Result` contract defined in `Spree::ServiceModule::Base`. Every service returns `success(value)` or `failure(value, error)`, never raising. Services are composed using the `run(callable)` DSL, which short-circuits on the first failure — a functional pipeline reminiscent of ports-and-adapters input ports.

Dependency injection is explicit. `Spree::Core::Dependencies` defines 60+ named injection points (`cart_add_item_service`, `checkout_complete_service`, `products_finder`, etc.) with default class names that consuming code resolves via `Spree::Dependencies.cart_add_item_service.constantize.call(...)`. This means any service can be swapped out by an application or extension gem without modifying core — a clean secondary-port/adapter seam.

The Hexagonal classification is partial (secondary, not primary) because the domain models use ActiveRecord, which ties persistence to the domain object — there is no explicit repository abstraction or persistence port. The "ports" are the service injection points, not the persistence boundary.

**Key evidence:**
- `spree/core/lib/spree/service_module.rb` — `Result`, `success()`, `failure()`, `run()`, `Callable` contract
- `spree/core/lib/spree/core/dependencies.rb` — 60+ named service injection points
- `spree/core/app/services/spree/` — 29 service namespaces, each a callable returning Result
- `spree/core/app/finders/spree/` — discrete finder objects (addresses, countries, orders, products, taxons, variants) separate from models
- `spree/core/app/services/spree/cart/add_item.rb`, `checkout/complete.rb`, etc. — fine-grained service operations

## Classification Reasoning

Spree's primary architecture is **Modular Monolith**: the backend deploys as a single Rails process, uses a single shared database, and is organized into four well-bounded gem engines. This is not Microservices (no network boundary between engines, shared database, no independent deployment of subsystems). It is not a pure Layered architecture — the primary organizational axis is domain-vertical (gem engines per capability: Core, API, Admin, Emails) even though Rails' MVC stack provides horizontal technical layers within each gem.

The **Event-Driven** secondary classification is earned by the presence of a first-class, fully designed event bus (`Spree::Events`) with async dispatch, pluggable adapters, subscriber base classes with DSL, wildcard pattern matching, and webhook integration built on top of it. This is an architectural mechanism, not incidental use of Rails callbacks. Cross-cutting concerns (emails, exports, webhooks, analytics) are decoupled from business operations through this bus.

The **Hexagonal Architecture** secondary classification reflects the service layer's port-like structure. The 60+ injection points in `Spree::Core::Dependencies`, the uniform `ServiceModule::Result` contract, the discrete finder objects, and the `run()` pipeline composition bring the service layer very close to the hexagonal port/secondary-adapter pattern. The qualification is that ActiveRecord domain models are not persistence-agnostic, so a strict hexagonal boundary is not maintained at the database level.

The system is NOT CQRS (no separate read/write model or command bus — services handle both writes and the finder objects handle reads as a convention, not as a CQRS command/query segregation with distinct models). It is NOT Microkernel in the classical sense — while there are registration points for payment methods and shipping calculators, they are Rails engine mount points and ActiveRecord STI hierarchies rather than a formal kernel/plugin protocol. It is NOT Serverless, Space-Based, Pipeline, or Multi-Agent.
