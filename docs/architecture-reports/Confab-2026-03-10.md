## Architecture Styles

### Primary: Modular Monolith

Seven bounded-context modules — Agendas, Attendances, Conferences, Saga, Speakers, Tickets, Users — each implement `IModule` (Name, Path, Register, Use) and are loaded at startup by `ModuleLoader`, which discovers and wires them from DLLs via reflection. Individual modules can be disabled via `module:enabled` config flags without recompilation. The single `Confab.Bootstrapper` host assembles all modules into one running process (`dotnet run`), and a shared `IModuleClient` enables synchronous in-process cross-module request routing over JSON serialization without shared type references — enforcing module isolation at the type level.

### Secondary: Domain-Driven Design

The Agendas and Attendances modules implement full DDD building blocks: `AggregateRoot<T>` with domain event collection and optimistic versioning, typed value objects (`AggregateId`, `ConferenceId`, `SpeakerId`, `EntityId`), invariant-enforcing entities (`AgendaTrack.ChangeName`, `AddRegularSlot`, time-conflict validation), repository interfaces per aggregate, and domain services. Modules with simpler domains (Users, Conferences, Speakers, Tickets) use a lighter CRUD pattern, matching the project's own documentation of per-module style variation.

### Secondary: CQRS

A shared framework — `ICommand`/`ICommandDispatcher`/`ICommandHandler<T>` and `IQuery<T>`/`IQueryDispatcher`/`IQueryHandler<T>` — is registered across all assemblies at startup. Command handlers are wrapped by `TransactionalCommandHandlerDecorator<T>` (resolved via `UnitOfWorkTypeRegistry`) to automatically scope database transactions. Read and write paths are structurally separated inside every DDD module (e.g., Agendas has distinct `Commands/` and `Queries/` folders with separate handlers), and domain writes surface through a separate query-optimized EF projection.

### Secondary: Event-Driven

Modules communicate asynchronously via events defined per-module as local contracts (e.g., `TicketPurchased`, `ConferenceCreated` in their own namespace) — no shared event project. `IMessageBroker.PublishAsync` routes events simultaneously to RabbitMQ (`IBusPublisher`) for external delivery and `IModuleClient.PublishAsync` for in-process fan-out. The Saga module uses the Chronicle library to coordinate multi-step flows (`InviteSpeakerSaga`) across SignedUp, SpeakerCreated, and SignedIn events. The standalone Tickets service (in `src/Services/`) subscribes to `ConferenceCreated` via RabbitMQ, demonstrating the documented future microservices migration path.

## Domain

Conference / Event Management — manages conference organization including users, speakers, ticket sales, agenda tracks and slots, call-for-papers, submission approval, and attendee registration.

---

## Production Context

- Explicitly positioned as a reference/educational implementation of Modular Monolith architecture, designed to demonstrate a migration path from monolith to microservices (the Tickets service is the proof-of-concept extraction).
- Single deployable .NET 5.0 process backed by one PostgreSQL database; RabbitMQ is the external message bus connecting modules and providing the integration point for the standalone Tickets service.
- Modules are individually togglable via JSON config files (`module.agendas.json`, etc.) without code changes, enabling selective feature enablement in different deployments.
- A companion Blazor web frontend (`Confab.Web`) consumes the exposed Web API; a YARP-based API Gateway routes between the modular monolith and the standalone Tickets microservice.
