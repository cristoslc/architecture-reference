# debezium ‚Äî Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/debezium/debezium
**Classification:** Microkernel + Pipeline
**Confidence:** 0.93

## Summary

Debezium is a Change Data Capture (CDC) platform built as a Maven multi-module mono-repo. Its dominant architectural expression is **Microkernel (Plugin)**: a stable kernel (`debezium-api`, `debezium-core`) defines a rich SPI/plugin contract, and a set of independently deployable connector JARs plug into a host runtime (Kafka Connect or the embedded engine). The internal event-processing flow within each connector follows a **Pipeline** pattern ‚Äî database log reader ‚Üí ChangeEventQueue ‚Üí EventDispatcher ‚Üí Kafka topic ‚Äî making the two styles deeply complementary. Layered dependency ordering underpins both, but it is an implementation constraint rather than a named primary style.

### Microkernel (Plugin) Evidence

1. **Java ServiceLoader throughout**: Every connector registers itself via `META-INF/services/org.apache.kafka.connect.source.SourceConnector`. Additional ServiceLoader files cover `Snapshotter`, `SnapshotLock`, `SnapshotQuery`, `SignalChannelReader`, `NotificationChannel`, `SignalActionProvider`, `PostProcessor`, `HeaderConverter`, and `CloudEventsProvider`. Each is independently loadable without recompiling the kernel.

2. **Explicit SPI module** (`debezium-api`): Defines `Snapshotter`, `CustomConverter`, `PostProcessor`, `TopicNamingStrategy`, `DataCollectionId` ‚Äî the formal extension contracts. Every connector depends on `debezium-api`; `debezium-api` has no dependency on any connector.

3. **Strict dependency hierarchy** (anti-reversal enforced by Maven): `debezium-api` ‚Üê `debezium-core` ‚Üê `debezium-connector-*`. Connectors are leaf modules; the core never imports from connectors.

4. **Multiple plugin dimensions**: Beyond database connectors, pluggability extends to storage backends (eight storage modules), snapshot strategies (nine built-in `Snapshotter` implementations plus user-provided), signal channels (File, Kafka, JMX, Process, Source), notification channels, AI embedding providers, and SMTs.

5. **Two host runtimes** (kernel providers):
   - **Kafka Connect** (distributed, fault-tolerant): connectors deployed as JARs on the Connect worker.
   - **Debezium Embedded Engine** (`debezium-embedded`): same connector JARs loaded in-process; `EmbeddedEngineConfig`, `DebeziumEngineCommon`, async engine variant.

6. **`BaseSourceConnector` ‚Üí `RelationalBaseSourceConnector` ‚Üí `BinlogConnector` ‚Üí `MySqlConnector`/`MariaDBConnector`**: Deep abstract base-class hierarchy shared across connectors provides lifecycle management and validation from the kernel.

### Pipeline (Pipe-and-Filter) Evidence

The internal execution model of every connector is a deterministic pipeline:

```
DB log / change stream
       ‚îÇ
       ‚ñº
StreamingChangeEventSource.execute()   [or SnapshotChangeEventSource]
       ‚îÇ  (ChangeRecordEmitter)
       ‚ñº
EventDispatcher.dispatchDataChangeEvent()
       ‚îÇ  (applies filters, heartbeats, incremental snapshot watermarks)
       ‚ñº
ChangeEventQueue.enqueue()             [bounded blocking queue, back-pressure]
       ‚îÇ  (producer thread)
       ‚ñº
BaseSourceTask.poll()                  [Kafka Connect polling thread]
       ‚îÇ  (applies SMTs, PostProcessors)
       ‚ñº
Kafka topic / embedded consumer
```

Key classes:
- `ChangeEventSourceCoordinator`: sequences the snapshot phase then hands off to streaming; coordinates `SignalProcessor` and `NotificationService` in parallel.
- `EventDispatcher`: central filter/routing node; applies include/exclude lists, heartbeats, transaction metadata enrichment.
- `ChangeEventQueue`: typed bounded queue decoupling producer and consumer threads; enforces back-pressure by size count and byte count.
- `AbstractChangeRecordEmitter` / `RelationalChangeRecordEmitter`: transforms raw DB log entries into structured `Envelope` payloads.

### Layered Dependency Order (supporting constraint)

| Layer | Module(s) |
|---|---|
| Public API / SPI | `debezium-api` |
| Core framework | `debezium-core`, `debezium-ddl-parser` |
| Connector implementations | `debezium-connector-*`, `debezium-connector-binlog` |
| Cross-cutting extensions | `debezium-transforms`, `debezium-storage/*`, `debezium-sink`, `debezium-embedded` |
| Integrations | `debezium-ai`, `debezium-openlineage`, `debezium-scripting` |

The layering is real and strictly maintained, but it is a dependency management constraint that underlies both the Microkernel and Pipeline patterns rather than the primary organizing concern.

### Why Not Other Styles

- **Not Microservices**: All connectors are library JARs sharing a single JVM process within Kafka Connect. There is no service-to-service network communication, no independent deployment of connectors as separate HTTP services.
- **Not Event-Driven (as primary style)**: Debezium *produces* change events externally, but internally components communicate through direct method calls and a shared in-process queue, not through an event bus or message broker mediating between internal components.
- **Not Hexagonal**: The SPIs resemble ports, but the design intent is extensibility (adding new DB connectors, storage backends) rather than isolating a pure domain core from I/O adapters for testability. There is no clear "inside" hexagon with separated left-port and right-port axes.
- **Not Domain-Driven Design**: Module boundaries follow technical roles (api, core, connector-type, storage-backend) not business bounded contexts.
