## Summary

Kill Bill is a production-grade open-source subscription billing and payments platform — the leading OSS entrant in this domain over the past 15 years. It is a Maven multi-module project (16 functional modules, ~1,080 non-test Java source files) that deploys as a single WAR file. Its dominant architectural style is **Microkernel**: the core billing engine defines a rich set of OSGi-backed plugin registries (`OSGIServiceRegistration<T>`) for payment, catalog, invoice, usage, and currency concerns, and every major integration point (payment gateway, tax, currency conversion, usage aggregation, catalog definition) is designed to be satisfied by a runtime-loaded plugin bundle. The secondary style is **Modular Monolith**: all 16 functional modules assemble into a single WAR through `profiles/killbill/pom.xml`, share a single Guice injector hierarchy, and communicate through compile-time API interfaces defined in the `api/` module rather than over a network.

## Styles Identified

### 1. Microkernel (Primary — strong evidence)

The Microkernel verdict is driven by the OSGi-backed plugin registry infrastructure that pervades the codebase. Kill Bill defines a minimal billing kernel — subscription lifecycle management, invoice generation, billing event assembly — and delegates every integration concern to pluggable extensions registered at runtime through `OSGIServiceRegistration<T>` contracts.

**Plugin Registry Infrastructure:**

Each integration domain has its own plugin registry, all backed by `OSGIServiceRegistration<T>` from the `killbill-osgi-api` dependency:

- `DefaultPaymentProviderPluginRegistry implements OSGIServiceRegistration<PaymentPluginApi>` — payment gateways (Stripe, PayPal, Braintree, etc. registered as OSGi bundles)
- `DefaultPaymentControlProviderPluginRegistry implements OSGIServiceRegistration<PaymentControlPluginApi>` — payment routing/retry control plugins
- `DefaultCatalogProviderPluginRegistry` — external catalog source plugins (for tenant-specific catalog loading)
- `DefaultCurrencyProviderPluginRegistry` — currency conversion rate providers
- `DefaultUsageProviderPluginRegistry` — usage data aggregation plugins
- `DefaultCatalogCache` uses `OSGIServiceRegistration<CatalogPluginApi>` for dynamic catalog resolution

```java
// PaymentPluginServiceRegistration.java
public class PaymentPluginServiceRegistration {
    private final OSGIServiceRegistration<PaymentPluginApi> pluginRegistry;

    public Set<String> getAvailablePlugins() {
        return pluginRegistry.getAllServices();
    }
}
```

**Plugin Registration Pattern:**

The `OSGIServiceRegistration` interface defines the plugin lifecycle:
```java
void registerService(OSGIServiceDescriptor desc, T service);
void unregisterService(String serviceName);
T getServiceForName(String name);
Set<String> getAllServices();
```

External OSGi bundles (e.g., `killbill-stripe-plugin`, `killbill-braintree-plugin`) implement the corresponding `*PluginApi` interface and call `registerService` on startup, making themselves available to the kernel. The `KillbillServerModule` injects these registries via Guice `Multibinder` and `OptionalBinder`, allowing absent plugins to degrade gracefully.

**45 non-test source files** import from `org.killbill.billing.osgi.api.*`, confirming the breadth of OSGI-based plugin integration across the codebase.

**State Machine Plugin Integration:**

The payment module demonstrates deep plugin integration via a state machine DSL (`PaymentStates.xml`, `RetryStates.xml`). Every payment operation (authorize, capture, purchase, refund, void, chargeback) transitions through states defined in `PaymentStates.xml`, and at each state transition a plugin is dispatched:

```
// PaymentModule.java imports
import org.killbill.automaton.DefaultStateMachineConfig;
import org.killbill.billing.payment.core.sm.PaymentStateMachineHelper;
import org.killbill.billing.payment.core.sm.PluginControlPaymentAutomatonRunner;
import org.killbill.billing.payment.core.sm.control.ControlPluginRunner;
```

The `PluginControlPaymentAutomatonRunner` and `ControlPluginRunner` classes mediate between the payment state machine and the `PaymentControlPluginApi` registry, applying plugin-driven routing, blocking, and retry logic at each transition.

**Plugin API Definitions in `api/` Module:**

The `api/` module serves as the microkernel's contract layer — it contains `NoOpInvoicePluginApi`, the `BillingApi`, `CatalogApi`, `PaymentApi`, `SubscriptionApi` interfaces, and all `*InternalApi` interfaces. By separating API contracts into a dedicated module with no implementation, the system ensures that functional modules interact through stable plugin interfaces, and external plugins can depend solely on `killbill-api` without pulling in the entire platform.

**Plugin Points per Domain:**

| Domain | Registry | Plugin API |
|---|---|---|
| Payment gateway | `OSGIServiceRegistration<PaymentPluginApi>` | `PaymentPluginApi` |
| Payment control | `OSGIServiceRegistration<PaymentControlPluginApi>` | `PaymentControlPluginApi` |
| Invoice | `InvoicePluginApi` (via NoOp pattern) | `InvoicePluginApi` |
| Catalog | `OSGIServiceRegistration<CatalogPluginApi>` | `CatalogPluginApi` |
| Currency | `OSGIServiceRegistration<CurrencyPluginApi>` | `CurrencyPluginApi` |
| Usage | `OSGIServiceRegistration<UsagePluginApi>` | `UsagePluginApi` |
| Notification | `NotificationPluginApi` (ext bus listener) | `ExtBusEventType` |

### 2. Modular Monolith (Secondary — strong evidence)

Despite the extensibility through OSGi plugins, the core platform itself deploys as a single WAR:

**Single WAR Assembly:**

`profiles/killbill/pom.xml` has `<packaging>war</packaging>` and declares dependencies on all 15 functional modules. At build time, all modules compile into a single web application. `KillbillServerModule` (the top-level Guice module) directly instantiates and assembles all module-level Guice modules:

```java
// KillbillServerModule.java imports include:
import org.killbill.billing.account.glue.DefaultAccountModule;
import org.killbill.billing.beatrix.glue.BeatrixModule;
import org.killbill.billing.catalog.glue.CatalogModule;
import org.killbill.billing.currency.glue.CurrencyModule;
import org.killbill.billing.entitlement.glue.DefaultEntitlementModule;
import org.killbill.billing.invoice.glue.DefaultInvoiceModule;
import org.killbill.billing.jaxrs.glue.DefaultJaxrsModule;
import org.killbill.billing.junction.glue.DefaultJunctionModule;
import org.killbill.billing.overdue.glue.DefaultOverdueModule;
import org.killbill.billing.payment.glue.PaymentModule;
import org.killbill.billing.subscription.glue.DefaultSubscriptionModule;
import org.killbill.billing.tenant.glue.DefaultTenantModule;
import org.killbill.billing.usage.glue.UsageModule;
```

All 13 functional Guice modules are installed into a single injector at servlet startup by `KillbillGuiceListener`.

**API Module as Module Boundary Enforcer:**

The `api/` module contains only Java interfaces and constants — no implementation. This enforces the Modular Monolith discipline: functional modules (`invoice`, `payment`, `subscription`) depend on the `api/` module's interfaces rather than each other's implementations. Cross-module calls go through the `api` contract layer.

**Shared Infrastructure:**

All modules share:
- A single JDBI v2 connection pool (`IDBISetup`) with per-module SQL DAOs
- A single `PersistentBus` instance (the killbill-commons event bus) — 84 non-test source files reference `PersistentBus`, `BusInternalEvent`, or `BusOptimizer`
- A single EhCache / Redis cache manager (`CacheModule`)
- A single Apache Shiro `SecurityManager` (`KillBillShiroAopModule`)
- A single notification queue service (`NotificationQueueService`)

**Module Communication via Internal Bus:**

Modules communicate asynchronously through a database-backed `PersistentBus`. This is not microservices-style network communication — it is in-process publish/subscribe through a shared bus backed by the same MySQL/PostgreSQL database. The `BeatrixListener` in the `beatrix` module subscribes to all `BusInternalEvent` types and translates them to external (`ExtBusEvent`) notifications delivered to plugin notification listeners and push notification webhooks.

### What This Is Not

**Not Microservices:** All modules run inside a single JVM (WAR). There is no service mesh, no container-per-module deployment, no independent scaling unit per domain. The `profiles/killbill` WAR assembles everything.

**Not Layered:** While there is a REST layer (`jaxrs`), service layer, and DAO layer within modules, the dominant organizational principle is plugin extensibility, not horizontal layer separation. Modules are organized by domain (billing, payment, catalog), not by tier.

**Not Event-Driven:** The internal `PersistentBus` is an implementation mechanism for loose coupling within the monolith — it is not the primary architectural driver. The billing engine is fundamentally request/response: a REST call triggers synchronous billing operations, and the bus propagates state-change notifications. The `beatrix` external bus bridges internal events to external plugin callbacks but this is a notification pattern, not an event-driven architecture.

**Not Service-Based:** Kill Bill is a single deployable WAR, not 4–12 independently deployed coarse-grained services.

## Classification Reasoning

Kill Bill's architecture is definitively **Microkernel** as the primary pattern, with **Modular Monolith** as the deployment and module-boundary pattern. The Microkernel verdict is supported by:

1. `OSGIServiceRegistration<T>` is the central extensibility primitive — used across payment, catalog, currency, usage, and control plugin domains. 45 non-test source files import from `org.killbill.billing.osgi.api.*`.
2. The `api/` module serves as the explicit plugin contract layer: it defines the `PaymentPluginApi`, `InvoicePluginApi`, `CatalogPluginApi`, etc. interfaces that external OSGi bundles implement and register.
3. Per-domain plugin registries (`DefaultPaymentProviderPluginRegistry`, `DefaultCatalogProviderPluginRegistry`, `DefaultCurrencyProviderPluginRegistry`, `DefaultUsageProviderPluginRegistry`) implement `OSGIServiceRegistration<T>` and manage runtime plugin lifecycle (register, unregister, lookup by name).
4. The payment state machine (`PaymentStates.xml`) routes every payment operation through plugin invocations — `PluginControlPaymentAutomatonRunner` and `ControlPluginRunner` dispatch to `PaymentControlPluginApi` registrations at each state transition.
5. The platform's documentation explicitly calls this the "platform" (the minimal kernel) + plugins architecture.

The Modular Monolith classification applies because:
1. `profiles/killbill/pom.xml` has `<packaging>war</packaging>` and declares all functional modules as dependencies — single WAR deployment.
2. `KillbillServerModule` directly installs all 13 domain Guice modules into a single injector — no runtime service boundary.
3. Cross-module communication uses direct Guice-injected interface calls (through `api/` contracts) and the in-process `PersistentBus`, not network calls.
4. All modules share a single database connection pool, event bus, cache manager, and security manager.

The prior automated discovery classification (2026-03-09, confidence 0.85) correctly identified Microkernel + Modular Monolith. This deep-analysis confirms that classification with stronger evidence grounding and higher confidence (0.92), using direct code-level evidence from the OSGi service registry infrastructure, state machine plugin dispatch, and Guice module assembly.

Confidence: **0.92** — the Microkernel pattern is unambiguous from the `OSGIServiceRegistration<T>` infrastructure and per-domain plugin registries; the Modular Monolith characterization is equally clear from the WAR assembly profile and single Guice injector pattern.
