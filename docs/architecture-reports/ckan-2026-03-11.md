# Architecture Report: CKAN

**Date:** 2026-03-11
**Repo URL:** https://github.com/ckan/ckan
**Classification:** Microkernel, Layered
**Confidence:** 0.93

---

## Summary

CKAN (Comprehensive Knowledge Archive Network) is a production-grade open-source data management system implemented as a single deployable Python/Flask WSGI application. Its defining architectural characteristic is a formal Microkernel (plugin) system: a minimal, stable core delegates virtually all domain functionality — including features that ship with the product — to runtime-loaded plugins via a rich set of typed interfaces. Within that core, a strict horizontal Layered pattern (Views → Logic/Actions → Model) provides clear separation of concerns. Secondary patterns — signal-based event propagation (Blinker), Redis-backed async job queues (RQ), and Solr for full-text search — are infrastructure integrations, not independent architectural styles.


### Key Architectural Evidence

**Microkernel core:** `ckan/plugins/base.py` defines `Interface`, `Plugin`, `SingletonPlugin` (singleton metaclass), and `implements()`. `ckan/plugins/core.py` performs runtime plugin discovery via `importlib.metadata.entry_points(group='ckan.plugins')`, maintains a global `_PLUGINS_SERVICE` registry, and orders plugins according to `ckan.plugins` in the INI config.

**Extension interface catalogue (ckan/plugins/interfaces.py):** Over 20 typed interfaces — `IActions`, `IAuthFunctions`, `IConfigurer`, `IDatasetForm`, `IGroupForm`, `IBlueprint`, `IResourceController`, `ITemplateHelpers`, `IValidators`, `IMiddleware`, `IAuthenticator`, `ISignal`, `INotifier`, `IForkObserver`, `IApiToken`, `IClick`, `IDomainObjectModification`, `IFeed`, `IFacets`, `IPermissionLabels`, `ITranslation`, `IUploader`, `IResourceView`, `IResourceUrlChange`.

**All features as plugins:** Core capabilities — `activity`, `datastore`, `datapusher`, `tracking`, `stats`, `multilingual`, all view types — are implemented as `ckanext.*` plugins registered in `setup.cfg` under `[options.entry_points]` / `ckan.plugins`. The datastore plugin itself implements `IActions`, `IAuthFunctions`, `IResourceController`, `ITemplateHelpers`, `IBlueprint`, and two datastore-specific interfaces.

**Blanket decorators** (`ckan/plugins/blanket.py`): `@tk.blanket.actions`, `@tk.blanket.auth_functions`, `@tk.blanket.helpers`, `@tk.blanket.blueprints`, `@tk.blanket.validators`, `@tk.blanket.cli`, `@tk.blanket.config_declarations` — convention-over-configuration shortcuts that further reinforce the plugin-centric architecture.

**Plugin toolkit facade** (`ckan/plugins/toolkit.py`): Re-exports `get_action`, `check_access`, `get_validator`, `render`, `abort`, `url_for`, `enqueue_job`, `signals`, `h` (helpers), validation utilities — providing a stable versioned surface to insulate plugins from internal refactoring.

**Layered internal structure:**
- `ckan/views/` — Flask `Blueprint` objects per resource type (`dataset.py`, `resource.py`, `user.py`, `group.py`, `api.py`, `admin.py`, `feed.py`)
- `ckan/logic/action/` — `get.py` (3,198 lines), `create.py` (1,477), `update.py` (1,355), `delete.py` (826) — action functions invoked by both the REST API and the UI via `get_action()`
- `ckan/logic/auth/` — parallel auth functions, checked before every action
- `ckan/model/` — SQLAlchemy ORM models (`Package`, `Resource`, `User`, `Group`, `Tag`, `Activity`, etc.) with `meta.py` managing the session

**Signal system** (`ckan/lib/signals.py`): Blinker-based `Namespace` with predefined signals (`action_succeeded`, `request_started`, `request_finished`, `user_logged_in`, `datastore_upsert`, etc.). The `activity` plugin subscribes to `action_succeeded` to record all mutating operations — demonstrating lightweight event-driven decoupling within the monolith.

**Async jobs** (`ckan/lib/jobs.py`): RQ + Redis queue system (`enqueue`, `get_queue`) used by datapusher and other extensions for background processing. This is an infrastructure concern, not a fundamental architectural style.

**Single deployable unit:** `wsgi.py` creates one Flask WSGI application via `make_app(config)`. All features share one PostgreSQL database, one Solr index, one Redis instance. No service boundaries, no inter-service communication.

## Architecture Styles Identified

### Primary: Microkernel

The plugin system is the organizing principle of CKAN. A minimal kernel (session management, config loading, WSGI app construction, base model) provides hook points through `Interface` subclasses. Extensions — including those bundled with CKAN itself — supply all domain behavior by implementing these interfaces. Plugins are discovered at runtime via Python entry points, ordered by configuration, and managed through a global lifecycle (`load`, `unload`, `plugins_update`). The `SingletonPlugin` pattern and `PluginImplementations` iterator ensure plugins are composable and replaceable. This matches Microkernel precisely: a stable minimal core with pluggable feature modules.

**Confidence evidence:** `ckan/plugins/base.py`, `ckan/plugins/core.py`, `ckan/plugins/interfaces.py` (20+ interfaces), `setup.cfg` entry-point registrations, `ckanext/` directory with 30+ bundled plugins.

### Secondary: Layered

Within the kernel and within plugins that follow conventions, a strict three-tier layer structure is enforced by directory layout and import discipline: Views call `get_action()` → Logic/Actions call auth functions and manipulate models → Models interact with the database via SQLAlchemy. The auth layer (`ckan/logic/auth/`) mirrors the action layer exactly, enforcing the pattern. Cross-layer access (e.g., views directly touching models) is actively discouraged.

**Confidence evidence:** `ckan/views/`, `ckan/logic/action/`, `ckan/logic/auth/`, `ckan/model/` structure; `get_action()` as the sole cross-layer call mechanism.

## Quality Attributes

| Attribute | Rating | Justification |
|---|---|---|
| **Extensibility** | Very High | The entire Microkernel design exists to maximize extensibility. 20+ typed interfaces allow plugins to override, chain, and augment virtually every system behavior without modifying core code. `chained_action` and `chained_auth_function` allow ordered plugin composition. |
| **Maintainability** | High | Clear layer separation (views/logic/model) and the plugin toolkit facade reduce coupling. Action functions are individually testable. Blanket decorators enforce consistent plugin structure. |
| **Modularity** | High | Features are self-contained plugin packages (own actions, auth, views, templates, assets). The `ckanext` namespace enforces physical separation. Plugin-level configuration isolation via `IConfigurer` and `IConfigDeclaration`. |
| **Testability** | High | `use_plugin()` context manager enables per-test plugin loading/unloading. The action layer is easily unit-tested via `get_action()`. pytest fixtures via `ckan.tests.pytest_ckan`. |
| **Interoperability** | High | REST API (v3, JSON-based) exposed via `ckan/views/api.py`; Solr for search; Redis for sessions and queues; DataPusher as optional external service. Well-defined action API used by external clients. |
| **Scalability** | Moderate | Single-process WSGI application; scales via uWSGI workers and load balancers (ckan-uwsgi.ini present). Redis-backed async jobs decouple long-running tasks. Solr handles search at scale. True horizontal scaling requires external infrastructure, not application-level sharding. |
| **Security** | Moderate-High | Per-action authorization via `check_access()` / auth functions; CSRF protection via flask-wtf; API token management (IApiToken, expire_api_token plugin); permission labels for dataset visibility. Authorization is pluggable (IAuthFunctions). |
| **Observability** | Moderate | Blinker signals for internal event tracing; standard Python logging; activity stream via the activity plugin. No built-in distributed tracing or structured metrics framework. |

## Classification Reasoning

CKAN's primary architecture is **Microkernel**. The evidence is unambiguous: the kernel (`ckan/`) provides the application skeleton and a registry of extension interfaces, while the actual domain capabilities (dataset management, user management, data storage, search, visualization) are implemented by plugins loaded at runtime. This pattern is enforced at every level — interface definition, entry-point discovery, singleton plugin lifecycle, the plugin toolkit facade, and blanket decorators. Crucially, even the features that ship with CKAN core are implemented as `ckanext.*` plugins rather than hardcoded behavior, which is the hallmark of genuine Microkernel design.

The **Layered** secondary classification reflects the strict internal discipline within the application. The views → logic/actions → model → database flow is consistently enforced. The auth layer mirrors the action layer exactly. The `get_action()` indirection is the only approved cross-layer call. This layering is subordinate to (and operates within) the plugin extension points.

The application is explicitly **not** Microservices or Service-Based: it is a single deployable WSGI process with one shared database. External services (Solr, Redis, optional DataPusher) are infrastructure dependencies. It is **not** Event-Driven as the primary style: signals are a secondary internal mechanism for plugin-to-plugin communication (especially activity recording), not the primary architectural organizing principle. It is **not** Hexagonal: plugins implement extension interfaces, not ports/adapters for dependency inversion; the model layer uses SQLAlchemy directly with no repository abstraction.
