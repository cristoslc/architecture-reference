# logstash — Architecture Classification Report
**Date:** 2026-03-11
**Repository:** https://github.com/elastic/logstash
**Primary Style(s):** Pipeline, Microkernel
**Confidence:** 0.97
**Summary:** Logstash is a data processing engine whose fundamental topology is a linear Pipeline (Inputs → Filters → Outputs) and whose extensibility is governed by a Microkernel plugin system built on a minimal JRuby/Java core.

### Languages

Primary: **Java** and **JRuby/Ruby** (co-equal; core engine is Java, orchestration/plugin system in JRuby). Secondary: Python (QA scripts).

### Key Configuration

Pipeline configuration is declarative and explicitly linear:
```
input  { plugin_a { ... } }
filter { plugin_b { ... } plugin_c { ... } }
output { plugin_d { ... } }
```

### Plugin Ecosystem

- Over 200 first-party plugins hosted under `github.com/logstash-plugins`
- Each plugin is an independent Ruby gem or Java JAR
- Plugin types: `input`, `filter`, `output`, `codec`
- Installed and managed via `bin/logstash-plugin` CLI (`lib/pluginmanager/`)

## Architecture Styles

### 1. Pipeline (Primary)

Logstash's defining topology is a sequential data processing pipeline with three explicit, ordered stages.

**Evidence:**

- `PipelineIR.java` compiles every user configuration into a directed graph whose vertices are inputs, a queue vertex, filter plugins, and output plugins — in strict topological order.
- `CompiledPipeline.java` (javadoc): "This class represents an executable pipeline, compiled from the configured topology… Each compiled pipeline consists in a graph of `Dataset` that represent either a filter, output or an if condition."
- `WorkerLoop.java`: "Pipeline execution worker, it's responsible to execute filters and output plugins for each `QueueBatch` that [is] pull[ed] out from queue." Workers pull batches from the queue and run them serially through the compiled filter/output graph.
- `java_pipeline.rb` manages `@input_threads` (producers) and `@worker_threads` (filter+output consumers), with an internal queue (`@drain_queue`) acting as the pipe between them.
- The persisted queue (`ackedqueue/Queue.java`) is an explicit backpressure mechanism between the input stage and the filter/output stage, a hallmark of durable pipeline architectures.
- `PipelineBus.java` enables multiple pipelines to be chained: an output plugin in one pipeline routes events to an input plugin of another, extending the pipeline topology across pipeline boundaries.
- The configuration DSL (`config/grammar.treetop`) reflects the three-stage structure syntactically; no other ordering is valid.

### 2. Microkernel (Secondary)

Logstash Core acts as a minimal kernel providing lifecycle management, the pipeline execution engine, the queue, and plugin infrastructure. All domain-specific I/O, transformation, and encoding logic lives in plugins outside the kernel.

**Evidence:**

- `logstash-core-plugin-api` is a dedicated gem that exports a stable plugin contract: `LogStash::Inputs::Base`, `LogStash::Filters::Base`, `LogStash::Outputs::Base`, `LogStash::Codecs::Base`. No business logic lives here; only the protocol.
- `co.elastic.logstash.api` Java package (referenced in `Event.java`, `CompiledPipeline.java`) defines Java-side plugin interfaces (`Input`, `Filter`, `Output`, `Codec`) — a typed extension point API.
- `plugins/discovery/PackageScanner.java` + `PluginRegistry.java` scan for `@LogstashPlugin`-annotated classes at runtime, providing dynamic plugin registration without modifying core.
- `PluginClassLoader.java` gives each Java plugin an isolated classloader, preventing dependency conflicts — the canonical microkernel isolation technique.
- `plugins/registry.rb` (Ruby side): discovers installed gems by scanning `Gem::Specification` for entries with the `logstash_plugin` metadata key. Plugins self-register by dropping a `logstash_registry.rb` hook file.
- `lib/pluginmanager/` provides a full `bin/logstash-plugin install/remove/update/generate` CLI, enabling runtime plugin management without rebuilding the application.
- Logstash Core processes zero events on its own. Without plugins, the pipeline cannot ingest, filter, or emit anything — the kernel is genuinely minimal.
- The `x-pack/` directory itself uses the same plugin mechanism to add enterprise capabilities (monitoring, central pipeline management, fleet integration) without touching core.

## Quality Attributes

- **Extensibility**: Hundreds of community and official plugins; any third party can author one without touching core
- **Throughput**: Multi-threaded worker model; configurable pipeline workers per pipeline
- **Durability**: Persistent Queue (PQ) with disk-backed acked delivery prevents event loss during outages
- **Backpressure**: Queue absorbs speed mismatches between inputs and outputs; configurable `queue.max_bytes`
- **Observability**: Instrumentation subsystem (`instrument/`) + REST monitoring API on port 9600 with per-plugin metrics
- **Resilience**: Dead Letter Queue (DLQ) for failed event routing; pipeline auto-reload on config change
- **Hot reload**: Pipelines can be reloaded without restarting the JVM (`config.reload.automatic`)
- **Isolation**: `PluginClassLoader` prevents plugin dependency conflicts; ephemeral pipeline IDs for state tracking
- **Multi-pipeline**: Multiple independent pipelines run concurrently in a single process; inter-pipeline routing via `PipelineBus`
- **Portability**: JRuby+JVM stack; ships as a self-contained distribution (tar/deb/rpm/Docker)

## Classification Reasoning

Logstash exhibits two deeply co-expressed architectural styles that are inseparable from its design intent.

**Pipeline is the primary style** because the entire application exists to move data from sources to destinations through transformations. Every architectural decision — the queue subsystem, the `WorkerLoop`, the `PipelineIR` graph compiler, the three-section DSL — exists in service of this data flow topology. The pipeline is not an internal implementation detail; it is the user-facing primitive. Users write pipeline configs, monitor pipeline metrics, reload pipelines, and chain pipelines together. The architecture radiates outward from the pipeline concept.

**Microkernel is the secondary style** because the kernel (logstash-core) is genuinely minimal and purely structural: it provides the execution frame (pipeline runner, queue, worker threads, monitoring) but contributes zero domain logic. Every meaningful action — reading from a file, parsing JSON, writing to Elasticsearch — requires a plugin. The plugin API (`logstash-core-plugin-api`), the plugin registry, the classloader isolation, and the plugin manager CLI are not incidental; they are first-class architectural concerns. This pattern is what has enabled the ecosystem of 200+ plugins and broad community adoption.

The two styles are complementary and mutually reinforcing: the Microkernel defines *how extensions are plugged in*; the Pipeline defines *where they are placed and how data flows between them*. Together they compose the distinctive Logstash architecture: a configurable, extensible data processing pipeline whose stages are microkernel plugins.

This is definitively **not** Event-Driven architecture: internally, Logstash uses a synchronous pull-based worker model (workers poll the queue). It is also not Microservices: the entire engine runs in a single JVM process as one deployable artifact. It is not Layered: there are no horizontal presentation/business/data layers — the organization is around vertical plugin extension points within a pipeline topology.
