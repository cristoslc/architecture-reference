# Architecture Report: Outline

**Date:** 2026-03-11
**Source:** https://github.com/outline/outline
**Classification:** Layered + Microkernel
**Confidence:** 0.91
**Method:** deep-analysis

---

## Summary

Outline is a collaborative knowledge-base application (think Notion/Confluence) built as a TypeScript monorepo. Its backend is a Koa/Node.js server with a PostgreSQL database and Redis-backed Bull queue system. The frontend is a React/MobX SPA compiled with Vite. The system is deployed as a single container that boots one or more named runtime "services" (web, worker, collaboration, websockets, cron, admin) selected via CLI flag.

The primary architectural style is **Layered**: every layer from routing through commands, models, policies, and presenters is explicit and horizontally organized. The secondary style is **Microkernel**: a formal `PluginManager` with typed `Hook` enum extends the core with 20+ plugins for auth providers, integrations, processors, and tasks registered at runtime.

## Primary Style: Layered Architecture

The backend follows strict horizontal layering documented in `docs/ARCHITECTURE.md`:

| Layer | Path | Responsibility |
|---|---|---|
| Routes | `server/routes/api/` | HTTP ingress, request parsing, response dispatch |
| Middlewares | `server/middlewares/` | Authentication, CSRF, rate-limiting, tracing |
| Commands | `server/commands/` | Cross-model orchestration (documentCreator, documentMover, etc.) |
| Models | `server/models/` | Sequelize ORM entities with lifecycle hooks |
| Policies | `server/policies/` | cancan authorization — decoupled from business logic |
| Presenters | `server/presenters/` | JSON transformation for API responses |
| Storage | `server/storage/` | Database, Redis, and file storage adapters |

The frontend mirrors this with: Routes → Scenes → Components → Stores → Models.

Data flows strictly downward through layers. Cross-cutting concerns (logging, metrics, tracing) are applied via middleware or utility imports rather than embedded in business logic.

## Secondary Style: Microkernel (Plugin System)

`server/utils/PluginManager.ts` defines a first-class plugin registry with typed hook points:

```typescript
export enum Hook {
  API = "api",              // Mount additional Koa routers
  AuthProvider = "authProvider",  // Register auth strategies
  EmailTemplate = "emailTemplate",
  IssueProvider = "issueProvider",
  Processor = "processor",  // Register event processors
  Task = "task",             // Register scheduled tasks
  UnfurlProvider = "unfurl",
  Uninstall = "uninstall",
}
```

Plugins are glob-loaded at startup from `plugins/*/server/index.[jt]s`. Each registers itself conditionally (usually guarded by env vars) by calling `PluginManager.add(...)`. Example from `plugins/slack/server/index.ts`:

```typescript
PluginManager.add([
  { type: Hook.AuthProvider, value: { router, id: config.id } },
  { type: Hook.API,          value: hooks },
  { type: Hook.Processor,    value: SlackProcessor },
]);
```

A parallel `app/utils/PluginManager.ts` manages client-side hooks (Settings, Imports, Icon). This is a textbook microkernel extension mechanism: a stable core with well-defined extension points.

## Event-Driven Internals (Supporting Pattern)

The system uses a Redis-backed Bull queue system for intra-process async work:

- **`globalEventQueue`** — receives all domain events (e.g., `documents.publish`) emitted by model `AfterSave` hooks
- **`processorEventQueue`** — fan-out queue; the worker routes each event to all applicable `BaseProcessor` subclasses
- **`websocketQueue`** — dedicated for `WebsocketsProcessor` to push real-time updates via Socket.IO
- **`taskQueue`** — handles arbitrary scheduled and background tasks (cleanup, export, email)

Processors declare interest via `static applicableEvents: Event["name"][]`. This resembles Event-Driven Architecture but is scoped entirely within a single deployable unit — it is background job processing, not inter-service communication. It does not qualify as a standalone Event-Driven style.

## Real-Time Collaboration

The `collaboration` service uses **Hocuspocus** (a Y.js CRDT server) with an extension model:

- `AuthenticationExtension` — validates JWT before allowing document access
- `PersistenceExtension` — persists Y.js document state to PostgreSQL
- `ConnectionLimitExtension` — enforces per-document connection limits
- `Redis` extension — enables multi-process collaboration state sharing

The collaboration service communicates over WebSockets (`wss://`) and can run on a separate host from the web server (configured via `COLLABORATION_URL`).

## Runtime Service Model

A single codebase boots multiple named services:

```
# Procfile
web:    yarn start --services=web,websockets,collaboration
worker: yarn start --services=worker
```

Each service (`web`, `worker`, `collaboration`, `websockets`, `cron`, `admin`) is a Node.js module that registers with a shared Koa application and HTTP server. They share one PostgreSQL database and communicate via Redis queues. This is a **multi-process single-deployable** pattern, not microservices — there is one `package.json`, one `Dockerfile`, and one build artifact.

## Quality Attributes

- **Extensibility:** Plugin system with typed hooks and priority ordering enables zero-touch integration of 20+ providers
- **Consistency:** cancan-based policy layer enforces authorization rules uniformly across all API routes
- **Real-time:** Hocuspocus Y.js CRDT provides conflict-free concurrent document editing with Redis scaling
- **Observability:** Structured logging (Datadog-compatible), Sentry error tracking, Prometheus-style Metrics class throughout
- **Scalability:** Web/worker/collaboration roles can scale horizontally independently; Redis queues decouple async work
- **Testability:** Co-located tests, factory-based test fixtures, separate test database

---

## Why Not Other Styles

- **Not Microservices:** One `package.json`, one `Dockerfile`, one shared PostgreSQL database. Runtime service selection (`--services=`) is a deployment concern, not service decomposition.
- **Not Modular Monolith:** No enforced module boundaries (no workspace packages, no internal API contracts). The `plugins/` directory provides extension, not internal module isolation.
- **Not Hexagonal/Ports-and-Adapters:** No explicit port interfaces. The `presenters` layer serializes data but is not an adapter abstraction. Storage and external services are accessed directly.
- **Not Event-Driven (primary):** Bull queues handle background work within the monolith; there is no event bus connecting independent services or bounded contexts.
- **Not CQRS:** No explicit read/write model separation; Sequelize models handle both reads and writes uniformly.
