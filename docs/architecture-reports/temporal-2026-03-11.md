# temporal — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/temporalio/temporal
**Classification:** Service-Based + Event-Driven
**Confidence:** 0.93

## Summary

Temporal is a durable-execution platform implemented as a tightly-coordinated cluster of four coarse-grained internal services (Frontend, History, Matching, Worker) that communicate via gRPC, share a common persistence tier, and can be deployed from a single binary or as independently scaled processes. The defining characteristic of the architecture is Service-Based: a small set of semantically distinct, stateful services with shared-database integration rather than the fine-grained, data-isolated topology of Microservices. A deeply embedded Event-Driven style is the equally foundational second pillar: every workflow execution is driven by an append-only history of events using a transactional outbox pattern to propagate tasks between services, with speculative in-memory task execution as an optimization layer.

### Key Architectural Files

- `service/history/history_engine.go`: `historyEngineImpl` — the central per-shard orchestrator. Holds `queueProcessors` map, `workflowCache`, `executionManager`, `replicator`. Delegates operation logic to specialized `history/api/` handlers.
- `service/history/shard/controller_impl.go`: Acquires shards via Ringpop ownership verification, lingers on ownership loss (up to 1 min), publishes shard count changes.
- `service/history/hsm/sm.go`: `Transition[S, SM, E]` — type-safe state machine transitions with validated source states, destination state, and side-effecting apply function that generates tasks.
- `common/persistence/data_interfaces.go`: Six persistence manager interfaces — `ShardManager`, `ExecutionManager`, `TaskManager`, `MetadataManager`, `ClusterMetadataManager`, `NexusEndpointManager`. All services write to the same database tier.
- `common/membership/interfaces.go`: `Monitor` (cluster participation) and `ServiceResolver` (Ringpop-based shard/key routing). `Lookup(key)` returns which host owns a given shard.
- `service/matching/matching_engine.go` + `forwarder.go`: Task queue partitioning with hierarchical forwarding tree — pollers on empty leaf partitions are redirected to parent partitions, creating eventual matching across the partition tree.
- `service/frontend/workflow_handler.go`: Public gRPC surface. Routes workflow operations to History service; routes task dispatch requests to Matching.
- `chasm/engine.go`: CHASM engine — component-based execution framework providing `StartExecution`, `UpdateComponent`, `ReadComponent`, `PollComponent` with speculative (non-persisted) and durable modes.
- `components/nexusoperations/`: Nexus outbound operation state machine with multi-cursor queue, circuit breaker, per-destination rate limiter; manages long-running cross-namespace RPC calls.

### Patterns Found

**Service-Based (primary):** Four coarse-grained services with distinct ownership of different concerns — not fine-grained Microservices. They share a single persistence tier (configured via `numHistoryShards` and a shared `defaultStore`). A single binary (`cmd/server/main.go`) can run all services or any subset. Inter-service calls are synchronous gRPC. Client beans (`client/client_bean.go`) aggregate typed gRPC client stubs for `HistoryServiceClient`, `MatchingServiceClient`, `FrontendClient`. Each service wires its own Uber-fx dependency graph (`service/*/fx.go`) but shares common infrastructure modules from `common/resource/fx.go`.

**Event-Driven / Event Sourcing (co-primary):** Every workflow execution is represented as a linear, append-only `History Event` sequence. "The sequence of History Events alone is sufficient to recover all other relevant information" for a workflow. The internal task propagation mechanism is a Transactional Outbox: Transfer Tasks, Timer Tasks, Visibility Tasks, and Archival Tasks are written atomically to the database alongside mutable state updates, then read by queue processors and dispatched as RPCs to Matching or other targets. This is the fundamental delivery guarantee mechanism, not a bolt-on feature.

**Hierarchical State Machine (structural pattern):** Two generations of state machine infrastructure coexist. The older `service/history/hsm/` package provides a `Transition[S, SM, E]` framework with validated source states, type-safe destination states, and task regeneration on state-based replication. The newer `chasm/` package introduces a component-based durable execution framework where arbitrarily nested stateful components can be composed into an execution tree, each with lifecycle, visibility attributes, and pure timer-based tasks.

**Shard-Based Data Partitioning:** The History service partitions all workflow executions into a fixed number of History Shards (set at cluster creation, immutable). Ringpop-based consistent hashing assigns shards to History service instances. A `RangeID` generation counter provides fencing for shard ownership transfers. Each shard owns its own in-memory queue processors and workflow execution cache.

**Long-Poll Dispatch:** The Matching service implements a server-side long-poll pattern for worker task delivery. Workers send `PollWorkflowTaskQueue` / `PollActivityTaskQueue` gRPC calls that block server-side until a task is available. Task queues partition into a forwarding tree (default 4 partitions) where pollers on empty leaf partitions are forwarded upward to find waiting tasks at parent partitions. This enables high-throughput matching across a distributed partition set.

**Speculative Execution:** Workflow Update operations use non-persisted "Speculative Workflow Tasks" that skip database writes entirely for rejection scenarios. Speculative tasks live only in the Matching service memory and are discarded if the workflow produces no commands and only rejection messages, achieving zero-write rejection paths.

**Multi-Cluster Replication (supporting):** The `service/history/ndc/` and `service/worker/replicator/` packages implement cross-cluster replication. Global Namespaces designate active/standby clusters. The internal worker service's Replicator consumes replication tasks from remote clusters and applies them locally, enabling geo-distributed active-passive deployments.

## Architecture Styles Identified

### Service-Based (Primary)

The system is explicitly decomposed into four well-bounded services: Frontend, History, Matching, and Worker. Each has a distinct responsibility domain. They share the same persistence tier (a single `defaultStore` with SQL or NoSQL backends). The cluster can run as a single process (all services in one binary) or with each service type independently scaled. gRPC is the inter-service communication protocol, with Ringpop membership providing service discovery and shard routing. This matches the Service-Based architectural style: a small number of coarse-grained services, shared database, gRPC communication, deployable from a single monorepo artifact.

This is explicitly NOT Microservices: services share the database schema, they are not independently versioned or released as separate artifacts, and four services is coarse-grained rather than fine-grained. It is NOT a Modular Monolith because the services communicate over the network (gRPC) with service discovery, not in-process function calls.

### Event-Driven (Co-Primary)

Event sourcing is the foundational data model: the append-only History Event log is the system of record for every workflow execution, with mutable state derivable entirely from replaying it. The Transactional Outbox pattern (Transfer Tasks, Timer Tasks, Visibility Tasks written atomically alongside event appends) is the core delivery mechanism for propagating work between the History and Matching services. This is not merely an implementation detail — the entire consistency model depends on it. Timer-triggered state transitions (activity timeouts, workflow task timeouts, user timers) are all driven through the Timer Task queue, creating a completely event/task-driven execution loop.

### CHASM Component Framework (Emerging Secondary)

The `chasm/` package represents a new-generation component-based execution framework being layered into the History service. Schedules are already implemented on CHASM (`chasm/lib/scheduler/`). This framework provides hierarchical stateful components with pure timer-based tasks, speculative execution modes, and visibility search attribute projection. As CHASM matures it will likely displace portions of the existing HSM-based workflow engine, but at this snapshot it is a secondary emerging pattern alongside the dominant event-sourced service model.

## Quality Attributes

- **Durability:** Core design guarantee. Append-only event log plus transactional outbox ensures no workflow state is lost even if the History service crashes mid-operation. Workflow state is fully recoverable from history replay.
- **Scalability:** History sharding (fixed partition count, ownership via Ringpop) allows horizontal scaling of the History service. Matching service task queue partitioning (default 4 partitions, configurable) allows parallel task dispatch. Frontend service is stateless and horizontally scalable. Dynamic configuration supports runtime tuning of quotas, batch sizes, and concurrency limits without restarts.
- **Consistency:** Database transactions couple mutable state, task creation, and history event appends in a single atomic write. Shard `RangeID` fencing prevents split-brain when ownership transfers. Shard linger (up to 1 minute) allows graceful handoff.
- **Fault Tolerance:** Failed activities and workflow tasks are automatically retried via timer queues. Speculative tasks degrade gracefully to persisted tasks on failure. Multi-cluster replication provides active-passive geo-redundancy. DLQ infrastructure handles unprocessable messages.
- **Extensibility:** The Nexus RPC framework provides cross-namespace and cross-cluster service-to-service calling with durable long-running operation semantics. CHASM provides a composable component framework for building new durable execution primitives. The persistence layer supports multiple database backends (Cassandra, MySQL, PostgreSQL, SQLite) and multiple visibility backends (Elasticsearch, SQL).
- **Observability:** Prometheus-compatible metrics via `common/metrics/`, structured Zap logging throughout, OpenTelemetry tracing support, health check endpoints per service, pprof endpoints for profiling.
- **Maintainability:** Uber-fx dependency injection enforces explicit wiring of all service dependencies. Interface-first design in `common/persistence/data_interfaces.go` separates storage concerns from business logic. Per-service `fx.go` wiring files make dependency graphs inspectable. Extensive co-located test coverage.
- **Interoperability:** Public API is protobuf/gRPC with OpenAPI (Swagger) HTTP gateway on the Frontend. REST (HTTP/JSON) and gRPC both supported. Nexus provides standardized open protocol for cross-system durable RPC. Multiple language SDKs (Go, Java, Python, TypeScript, .NET, Ruby, PHP) connect via the public gRPC API.

## Classification Reasoning

The decisive architectural question for Temporal is whether the primary organizing principle is the *service decomposition topology* (Service-Based) or the *event-driven execution model* (Event-Driven), and the answer is that both are equally fundamental and mutually reinforcing — neither can be removed without destroying the system.

**Service-Based** is the deployment and communication architecture: four named services, shared persistence, gRPC inter-service calls, Ringpop-based service discovery, one binary supporting all topology modes. This is not Microservices (shared DB, coarse-grained, monorepo single binary). It is not a Modular Monolith (gRPC network boundary, not in-process). Service-Based at 0.95 local confidence.

**Event-Driven** is the execution architecture: append-only event sourcing for workflow state, transactional outbox for task propagation, timer queues for all scheduled operations. The entire consistency model is built on atomic event + task writes. This is not merely a pattern used opportunistically — it is the load-bearing structure of every workflow execution. Event-Driven at 0.95 local confidence.

The small confidence gap from 1.0 reflects: (a) the emerging CHASM component model introduces a new execution paradigm that may eventually change the primary style weighting; (b) the Nexus subsystem introduces an additional stateful RPC pattern whose architectural maturity is still evolving (multi-cluster replication not yet implemented for Nexus endpoints); (c) classifying a system that deliberately runs as a single binary but is designed for multi-service scaling requires judgment about the intended versus actual deployment mode.

Confidence: **0.93**
