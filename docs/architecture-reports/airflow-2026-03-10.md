## Style Rationales

**Microkernel (primary, 0.92):** The `providers/` directory contains 84 independently versioned and installable provider packages, each declaring itself via `[project.entry-points."apache_airflow_provider"]` in their `pyproject.toml`. The core discovers providers at runtime through `shared/providers_discovery/src/.../providers_discovery.py` using `entry_points_with_dist("apache_airflow_provider")`. The `shared/plugins_manager/` defines `AirflowPlugin` with formal extension points (hooks, operators, sensors, executors, auth backends, secrets backends, listeners, triggers, fastapi_apps). Executors are loaded dynamically by class path — `CeleryExecutor` and `KubernetesExecutor` come entirely from provider packages, not the core. The `task-sdk/` defines the `BaseOperator`/`BaseHook`/`BaseSensorOperator` contracts that all plugins implement. DAG bundles are also pluggable via `AbstractDagBundle`.

**Pipeline (primary, 0.88):** Airflow's fundamental abstraction is the DAG — a directed acyclic graph of tasks with explicit data-flow dependencies. The five-component runtime enforces a strict one-way pipeline: DAG authors write tasks via the Task SDK → the DAG Processor parses DAGs in isolated subprocesses → the Scheduler reads serialized DAGs and queues task instances → Workers execute tasks via the Task SDK communicating back through the Execution API → the Triggerer handles deferred async tasks. Each stage has a single defined role and the scheduler explicitly never runs user code; workers never touch the metadata database directly.

**Modular Monolith (secondary, 0.82):** The repository is a UV workspace monorepo with clear internal package boundaries: `airflow-core` (scheduler, API server, models), `task-sdk` (DAG authoring and task execution runtime), `shared/` (configuration, secrets, logging, observability, plugin infrastructure), `airflow-ctl` (management CLI), and `clients/` (Python, Go SDK clients). These packages share a single metadata database and deploy from a common codebase; they are independently testable units within a monorepo rather than separate services.

## Domain

Workflow orchestration platform. Core domains: DAG authoring and scheduling, task execution, external-system integration (via providers), asset/dataset lineage, secrets management, human-in-the-loop approval, backfill and retry management, multi-team access control.

## Production Context

- Deployed via Kubernetes Helm chart or Docker Compose; five distinct process types each independently scalable
- 84 provider packages covering AWS, GCP, Azure, Databricks, Snowflake, Kubernetes, Celery, dbt, OpenAI, and many more
- Airflow 3.x introduced the `task-sdk` isolation boundary: workers no longer need access to the metadata database, reducing operational risk and enabling heterogeneous worker environments
- The Execution API (FastAPI, versioned under `execution_api/versions/`) mediates all worker-to-scheduler communication and is considered a stable public contract
