## Style Rationales

**Microkernel (primary, 0.93):** The defining architectural characteristic of dbt-core is its adapter-plugin system, explicitly documented in `ARCHITECTURE.md`: "dbt uses an adapter-plugin pattern to extend support to different databases, warehouses, query engines, etc." The core (`dbt-core`) defines stable extension contracts through the `dbt-adapters` package, and database-specific implementations live in separate packages (`dbt-postgres`, `dbt-snowflake`, `dbt-bigquery`, etc.) that are loaded dynamically at runtime. The `core/dbt/plugins/manager.py` implements `PluginManager`, which uses `pkgutil.iter_modules()` to discover all installed `dbt_*` packages at startup and registers their `dbtPlugin` subclasses. The `core/dbt/config/profile.py` calls `load_plugin(typename)` to dynamically import the adapter matching the `type` field in `profiles.yml`. A second plugin mechanism (`dbtPlugin` base class with `get_nodes()` and `get_manifest_artifacts()` hooks) allows third-party plugins to inject nodes directly into the DAG. The `adapter.dispatch()` mechanism provides polymorphic macro invocation, resolving calls through a priority chain: adapter-specific project → adapter-specific package → default implementation. Materialization macros follow the same override chain, documented in `docs/arch/4.5_Node_Materialization.md`. The core itself never contains database-specific SQL; it remains warehouse-agnostic.

**Pipeline (primary, 0.89):** dbt-core's entire purpose is orchestrating a multi-stage transformation pipeline against data warehouses, implemented as DAG-ordered node traversal. The parsing phase (`ManifestLoader.get_full_manifest()`) is itself a sequential pipeline with seven distinct, ordered stages: Read Files → Load Macros → Parse Project Files → Parse Schema Files → Patch Sources → Process References → Validation. The execution phase follows a strict directed acyclic graph (DAG) order enforced by `GraphQueue`, a topological priority queue in `core/dbt/graph/queue.py` that releases nodes only when all upstream dependencies have completed via `mark_done()`. Within each node execution, a two-phase compile-then-execute pipeline applies: Jinja templates are rendered to produce `compiled_code`, then the compiled SQL is dispatched to the adapter for warehouse execution. The `GraphRunnableTask` in `core/dbt/task/runnable.py` orchestrates parallel node execution via `DbtThreadPool` (a `multiprocessing.pool.ThreadPool`) while preserving DAG ordering. Data flows strictly forward: file parsing produces the Manifest, which drives graph construction, which feeds the execution queue, which produces `RunResult` artifacts written to `run_results.json`.

**Layered (secondary, 0.85):** The internal codebase is organized into well-defined horizontal layers with clear directionality. From outermost to innermost: CLI (`core/dbt/cli/`) parses commands and decorates them with setup requirements; Task layer (`core/dbt/task/`) orchestrates command-level execution using an inheritance hierarchy (BaseTask → ConfiguredTask → GraphRunnableTask → CompileTask → RunTask → BuildTask); Graph layer (`core/dbt/graph/`) constructs the networkx DAG and manages node selection and queuing; Parser layer (`core/dbt/parser/`) reads project files and builds the Manifest; Compilation layer (`compilation.py`) renders Jinja templates; Contracts layer (`core/dbt/contracts/`) defines validated dataclass schemas; Clients layer (`core/dbt/clients/`) wraps external dependencies (Jinja2, agate, YAML). Each layer calls downward and never upward. The Context layer (`core/dbt/context/`) provides a parallel hierarchy of Jinja execution environments. The Events layer (`core/dbt/events/`) is a cross-cutting concern using protobuf-defined event types fired via `fire_event()` throughout all layers.

## Quality Attributes

| QA | Evidence |
|---|---|
| **Extensibility** | Adapter plugin system allows any data warehouse to be supported without changes to core; `dbtPlugin` API allows third-party packages to inject DAG nodes; materialization macros overrideable at project, package, and adapter levels; `adapter.dispatch()` enables macro-level polymorphism |
| **Maintainability** | 7-phase parsing pipeline with distinct parser classes per resource type (`ModelParser`, `SnapshotParser`, `SchemaParser`, etc.) isolates change; task hierarchy separates command orchestration from node execution; contracts layer (`core/dbt/contracts/`) centralizes schema definitions and validation |
| **Performance** | Thread pool parallel execution with topological DAG ordering maximizes concurrency while respecting dependencies; partial parsing (`--partial-parse`, default on) reuses previous manifest on unchanged files; relation cache (`adapter.set_relations_cache()`) reduces `information_schema` queries during execution |
| **Testability** | Unit tests use `mock_adapter.py` to isolate core from warehouse; functional tests invoke real adapters against Postgres; parser layer is stateless and testable without database connections; compiler is testable with a minimal manifest |
| **Observability** | Protobuf-defined event types in `core/dbt/events/types.py` (2,500+ lines); `fire_event()` cross-cutting throughout all layers; event codes organized by execution phase (A=pre-load, I=parsing, Q=node execution, W=testing); structured JSON artifacts: `manifest.json`, `run_results.json`, `catalog.json`, `sources.json` |
| **Portability** | Core is warehouse-agnostic; entire SQL dialect handling delegated to adapter plugins; `dbt-core` runs on any platform with Python 3.10+ and a compatible adapter installed |

## Domain

Data transformation and analytics engineering toolchain. Core domains: project parsing and manifest construction, DAG-based SQL compilation and warehouse execution, adapter-mediated database portability, incremental and stateful execution, package dependency management, documentation generation, data testing, and data freshness monitoring.

## Production Context

- Single deployable PyPI package (`dbt-core`) with a `dbt` CLI entry point; adapters installed separately (e.g., `pip install dbt-core dbt-postgres`)
- `pyproject.toml` declares dependencies on `dbt-adapters>=1.22.8`, `dbt-common>=1.37.3`, `networkx`, `Jinja2`, and `protobuf` as first-class runtime dependencies
- Thread-pool parallelism controlled by `threads` in `profiles.yml`; default is 1, typically set to 4–16 in production
- Artifacts versioned and schema-stable: manifest v12, run_results v5, catalog v1, freshness v3; downstream tools (dbt Cloud, Datafold, Elementary) consume `manifest.json`
- `dbtRunner` class (`core/dbt/cli/main.py`) provides a programmatic Python API for embedding dbt in orchestrators (Airflow, Prefect, Dagster)
- Partial parsing (enabled by default) reduces cold-start latency on large projects by serializing the manifest to `target/partial_parse.msgpack`
