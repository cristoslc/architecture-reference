## Summary

Apache Ignite 2 is a **Microkernel** at its structural core — the `IgniteKernal` class acts as a
central coordinator (the "kernal") that discovers, loads, and lifecycle-manages a large set of
pluggable Managers and Processors via a formal SPI (Service Provider Interface) extension system.
The repository is simultaneously a **Modular Monolith**: 36 Maven modules under a single parent
POM, assembled into one binary distribution (`apache-ignite`), with all modules sharing the same
JVM process at runtime. The runtime topology of any deployed Ignite cluster is **Space-Based**:
homogeneous peer nodes each own a partition of the distributed in-memory data set and execute
computation co-located with that data (affinity colocation), with no central shared database.

Compared to Ignite 3, the style ordering is inverted: Ignite 2's Microkernel identity is primary
because the SPI + plugin system is the dominant structural pattern across the entire codebase,
while Ignite 3 leads with Modular Monolith due to its stricter Gradle-enforced module boundaries
and acyclic graph guarantees.

## Evidence

### 1. Microkernel (primary)

**The Kernal as central coordinator**

`IgniteKernal.java` is the single startup and lifecycle orchestrator for every component in the
system. It constructs and initializes a `GridKernalContext` that aggregates all subsystems and
makes them accessible across the runtime. The startup sequence calls `startProcessor(...)` and
`startManager(...)` in a fixed dependency order:

```java
// IgniteKernal.java (excerpt from startup sequence)
List<PluginProvider> plugins = U.allPluginProviders(cfg, true);

startProcessor(new DiagnosticProcessor(ctx));
startProcessor(new GridInternalSubscriptionProcessor(ctx));
startProcessor(new IgnitePluginProcessor(ctx, cfg, plugins));
startProcessor(new FailureProcessor(ctx));
startManager(new GridTracingManager(ctx, ...));
startManager(new GridIoManager(ctx));
startManager(new GridCheckpointManager(ctx));
startManager(new GridEventStorageManager(ctx));
startManager(new GridDeploymentManager(ctx));
startManager(new GridLoadBalancerManager(ctx));
startManager(new GridFailoverManager(ctx));
startManager(new GridCollisionManager(ctx));
startManager(new GridDiscoveryManager(ctx));
startManager(new GridEncryptionManager(ctx));
startProcessor(new GridAffinityProcessor(ctx));
startProcessor(new GridCacheProcessor(ctx));
startProcessor(new GridQueryProcessor(ctx));
startProcessor(new IgniteServiceProcessor(ctx));
startProcessor(new GridJobProcessor(ctx));
startProcessor(new GridTaskProcessor(ctx));
```

The kernal does not contain business logic itself — it is purely a coordinator and lifecycle
manager. This is the defining characteristic of the Microkernel pattern.

**Plugin SPI system**

`IgniteSpi.java` defines the lifecycle contract for all pluggable subsystems:

```java
public interface IgniteSpi {
    public String getName();
    public Map<String, Object> getNodeAttributes() throws IgniteSpiException;
    public void spiStart(String igniteInstanceName) throws IgniteSpiException;
    public void onContextInitialized(IgniteSpiContext spiCtx) throws IgniteSpiException;
    public void spiStop() throws IgniteSpiException;
    public void onContextDestroyed();
}
```

Nine named SPI categories are defined under `modules/core/src/main/java/org/apache/ignite/spi/`:

| SPI Category | Function |
|---|---|
| `discovery` | Node discovery and cluster membership (`TcpDiscoverySpi`, ZooKeeper) |
| `communication` | Inter-node message transport (`TcpCommunicationSpi`) |
| `checkpoint` | Distributed job checkpoint storage |
| `collision` | Job scheduling and admission control |
| `failover` | Job failover routing |
| `loadbalancing` | Task-to-node load distribution |
| `eventstorage` | Local and remote event storage |
| `deployment` | Peer-to-peer class deployment |
| `indexing` | Pluggable indexing backend (SQL, text) |

Each SPI category has a `GridManager` facade (`GridDiscoveryManager`, `GridIoManager`, etc.) that
wraps the SPI layer, so the kernal logic is decoupled from any particular implementation.

**ServiceLoader-based plugin loading**

`IgniteUtils.allPluginProviders()` in `IgniteUtils.java` uses `ServiceLoader` to discover
`PluginProvider` implementations at startup:

```java
public static List<PluginProvider> allPluginProviders() {
    List<PluginProvider> providers = new ArrayList<>();
    Iterable<PluginProvider> it = loadService(PluginProvider.class);
    for (PluginProvider provider : it)
        providers.add(provider);
    return providers;
}
```

Plugins may also be registered explicitly via `IgniteConfiguration.setPluginProviders(...)`.
`IgnitePluginProcessor` manages the complete plugin lifecycle: initialization, extension
registration, discovery data exchange, and teardown.

**Formal extension point contract**

`PluginProvider.java` javadoc states:

> "Ignite plugins can be loaded using JDK `ServiceLoader` or set up explicitly via
> `IgniteConfiguration#setPluginProviders(PluginProvider[])`. If the `ServiceLoader` approach
> is chosen, the fully-qualified `PluginProvider` class name is registered in
> `META-INF/services/org.apache.ignite.plugin.PluginProvider`."

Plugins implement `IgnitePlugin` (user-facing API), `PluginProvider` (lifecycle + wiring), and
optionally `CachePluginProvider` (per-cache extensions). The `Extension`/`ExtensionRegistry`
interfaces provide typed extension point registration for capability discovery between plugins.

**Optional feature modules as plugins**

Several modules register themselves as optional feature modules discoverable at startup:

- `modules/zookeeper` — `ZookeeperDiscoverySpi` (alternative discovery)
- `modules/calcite` — alternative SQL engine (`QUERY_ENGINE.create(ctx, ...)`)
- `modules/opencensus` — tracing backend
- `modules/kubernetes` — Kubernetes-native discovery
- `modules/platforms` — .NET/C++ native interoperability via `PlatformBootstrapFactory`

The Calcite module is loaded at startup via `createComponent(...)`, which performs classpath
scanning to swap the default H2-based query engine with Apache Calcite when the module is
present.

### 2. Modular Monolith (secondary)

**Single Maven assembly artifact**

`assembly/release-apache-ignite.xml` produces a single binary distribution (`apache-ignite`)
from all modules. All 36 Maven modules under `modules/` run in the same JVM process. There are
no cross-service HTTP/gRPC boundaries between modules at runtime.

**36 Maven modules under a shared parent POM**

`pom.xml` at the root enumerates all modules:

```xml
<modules>
    <module>modules/core</module>
    <module>modules/indexing</module>
    <module>modules/calcite</module>
    <module>modules/spring</module>
    <module>modules/zookeeper</module>
    <module>modules/rest-http</module>
    <module>modules/clients</module>
    <module>modules/kubernetes</module>
    <module>modules/platforms</module>
    ...
</modules>
```

Module dependencies are enforced through Maven POM structure. A BOM (`modules/bom/pom.xml`)
manages version alignment across all modules.

**Internal component architecture**

The `GridKernalContext` provides centralized access to approximately 45+ Processors and 15+
Managers organized by functional domain:

| Layer | Component examples |
|---|---|
| Foundation | `GridClosureProcessor`, `GridResourceProcessor`, `GridMarshallerMappingProcessor` |
| Storage | `GridCacheProcessor`, `IgniteCacheOffheapManagerImpl`, persistence WAL |
| Query | `GridQueryProcessor`, `IndexProcessor`, Calcite engine |
| Compute | `GridTaskProcessor`, `GridJobProcessor`, `GridClosureProcessor` |
| Services | `IgniteServiceProcessor`, `DataStructuresProcessor` |
| Cluster | `GridClusterStateProcessor`, `GridAffinityProcessor` |
| Connectivity | `ClientListenerProcessor`, `GridRestProcessor` |
| Streaming | `DataStreamProcessor`, `GridContinuousProcessor` |

All components share one `GridKernalContext` — there is no separate data store per component.
This is the defining mark of a Modular Monolith, not Microservices.

**Internal package boundaries**

`org.apache.ignite.internal.*` packages encapsulate implementation details. Public APIs live in
`org.apache.ignite.*` (top-level). The `spi` package provides the stable extension contract
layer. However, unlike Ignite 3's strict `-api`/`-impl` Gradle module split, the Ignite 2
boundary enforcement is by convention at the package level, not by build-system visibility rules.

### 3. Space-Based (tertiary — runtime topology)

The runtime deployment of Apache Ignite 2 is a classic data grid: homogeneous peer nodes each
own a partition of the distributed in-memory dataset and execute computation co-located with
that data.

**Partitioned distributed caches (the grid)**

`GridCacheProcessor` manages all caches. The `GridDhtCache` class implements the Distributed
Hash Table layer — data is split into 1024 partitions (configurable) distributed across nodes
via affinity functions:

```
modules/core/src/main/java/org/apache/ignite/internal/processors/cache/distributed/dht/
├── GridDhtCache.java
├── GridDhtCacheAdapter.java
├── GridDhtCacheEntry.java
├── atomic/
└── colocated/
```

`GridAffinityProcessor` maps cache keys to partition owners. `CacheConfiguration.setAffinity()`
allows custom `AffinityFunction` implementations to control data routing.

**Compute-on-data affinity colocation**

`IgniteCompute.affinityRun(...)` / `affinityCall(...)` routes task execution to the node holding
the specified cache key — moving computation to the data rather than the data to the computation.
`ComputeTask.map(List<ClusterNode>, Object)` gives full control over job-to-node assignment.

**Shared-nothing peer topology**

Nodes communicate via TCP (`TcpCommunicationSpi`) and discover each other via `TcpDiscoverySpi`
(ring topology) or `ZookeeperDiscoverySpi`. There is no central coordinator: the cluster
coordinator role rotates via the discovery protocol. Each node holds its own in-memory data
slice plus optional persistence via the native Write-Ahead Log.

**Native persistence (WAL)**

`FileWriteAheadLogManager` implements the WAL for crash recovery without shared storage.
`GridCacheDatabaseSharedManager` manages page memory and checkpoint management, enabling
Ignite to persist its in-memory grid state to local disk on each node independently.

**Change Data Capture (CDC)**

`CdcMain` and `WalRecordsConsumer` in `modules/core/src/main/java/org/apache/ignite/cdc/`
implement change-data-capture over WAL records, enabling downstream event streaming from the
data grid — a lightweight event-driven outbox on top of the Space-Based storage tier.

## What This Architecture Is Not

**Not Microservices**: All modules run in one JVM process per Ignite node. Cluster nodes are
homogeneous peers sharing a single code artifact, not independently bounded-context services
with private data stores. The node forms a processing unit (Space-Based), not a service
boundary (Microservices).

**Not Layered (primary)**: The codebase has horizontal structure (SPI → Manager → Processor →
API), but the dominant organizing principle is the Microkernel plugin system. Layered is a
supporting internal pattern within individual components, not the structural identity of the
system.

**Not Event-Driven (primary)**: Ignite provides `IgniteMessaging` (topic-based pub/sub),
`IgniteEvents` (cluster event listeners), Continuous Queries, and CDC. However, these are
feature capabilities attached to the data grid, not the primary coordination mechanism. The
primary programming model is synchronous in-memory data access, not message-broker-mediated
event flow.

**Not CQRS**: Reads and writes operate against the same cache partition (`GridDhtCacheAdapter`).
Continuous Queries are streaming queries, not a CQRS read-model separation.

**Not Hexagonal Architecture**: SPIs provide swappable implementations for infrastructure
concerns, but the codebase is not organized around explicit inbound/outbound port contracts
isolating an application core. Components have deep interdependencies via `GridKernalContext`.

**Not Domain-Driven Design**: Modules are organized by technical capability (cache, query,
compute, service, stream), not business bounded contexts with aggregate roots or domain events.

## Key Technologies

- **Transport**: `TcpCommunicationSpi` (custom binary protocol over NIO)
- **Discovery**: `TcpDiscoverySpi` (ring topology); `ZookeeperDiscoverySpi` (optional module)
- **Persistence**: Custom page-memory storage + Write-Ahead Log (`FileWriteAheadLogManager`)
- **SQL**: Apache H2 (default indexing module); Apache Calcite (optional `modules/calcite`)
- **Transactions**: Custom 2PC over DHT partitions with MVCC support
- **Serialization**: Custom binary marshaller (`BinaryMarshaller`) with schema registry
- **Compute**: MapReduce-style `ComputeTask` / closure APIs with affinity routing
- **Build**: Maven multi-module, single distribution assembly
