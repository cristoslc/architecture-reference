## Architecture Styles

### Primary: Microkernel

Jellyfin is organized around a stable core that is deliberately incomplete without plugins. The `ApplicationHost` (`Emby.Server.Implementations/ApplicationHost.cs`) acts as the microkernel: it wires the ASP.NET Core DI container, discovers plugin assemblies from the filesystem via `PluginManager`, and surfaces extension points through `GetExports<T>()` and `GetExportTypes<T>()`. Plugins are loaded at startup into isolated `AssemblyLoadContext` instances (`Emby.Server.Implementations/Plugins/PluginLoadContext.cs`), can register their own services via `IPluginServiceRegistrator`, and contribute API controllers through `GetApiPluginAssemblies()` — all without modifying core code.

Plugin contracts are defined in `MediaBrowser.Common/Plugins/`: `IPlugin` (name, version, id, lifecycle), `BasePlugin<TConf>` (configuration storage, data folder), and `IPluginManager`. Extension-point interfaces in `MediaBrowser.Controller/Providers/` define contracts for metadata providers (`IMetadataProvider<T>`, `ILocalMetadataProvider`, `IRemoteMetadataProvider`), image providers (`IImageProvider`, `IDynamicImageProvider`), and subtitle providers — all resolvable at runtime by the DI container. The live-TV tuner system in `src/Jellyfin.LiveTv/TunerHosts/` follows the same pattern: `ITunerHost` implementations are discovered and registered without a hardcoded list.

The `IEventManager` (`MediaBrowser.Controller/Events/IEventManager.cs`) provides async typed event publication (`Publish<T>`, `PublishAsync<T>`), and `IEventConsumer<T>` subscribers (session events, plugin lifecycle events, authentication events) are auto-registered from any assembly — core or plugin. This pub/sub layer is the messaging spine of the microkernel, allowing plugins to react to core domain events without coupling to core services.

### Secondary: Layered

Within the stable core, the codebase is organized in a clear horizontal stack:

- **Presentation / API layer** — `Jellyfin.Api/Controllers/` (60 ASP.NET Core controllers), `Jellyfin.Api/Middleware/` (9 middleware components), and `Jellyfin.Api/WebSocketListeners/` (4 WebSocket listeners for real-time push). The `Jellyfin.Server/Startup.cs` configures this layer with Prometheus metrics, Swagger, and custom auth policies.
- **Service / Application layer** — `Emby.Server.Implementations/` contains the large service implementations: `LibraryManager.cs`, `SessionManager.cs`, `UserDataManager.cs`, `MediaSourceManager.cs`, and the `ScheduledTasks/` task runner.
- **Domain Abstraction layer** — `MediaBrowser.Controller/` is an interface-only project defining `ILibraryManager`, `ISessionManager`, `IUserManager`, `IMediaSourceManager`, and all extension contracts. It has no concrete implementations.
- **Infrastructure / Repository layer** — `Jellyfin.Server.Implementations/` provides EF Core repositories (`BaseItemRepository`, `PeopleRepository`, `ChapterRepository`, `MediaStreamRepository`) against `JellyfinDbContext`. `src/Jellyfin.Database/Jellyfin.Database.Implementations/` houses entity definitions, and `Jellyfin.Database.Providers.Sqlite/` implements the `IJellyfinDatabaseProvider` abstraction, making the database engine swappable.
- **Model / DTO layer** — `MediaBrowser.Model/` holds DTOs and shared value types used across all layers; `Jellyfin.Data/` holds EF Core entities.
- **Cross-cutting subsystems** — `MediaBrowser.MediaEncoding/Transcoding/` and `src/Jellyfin.MediaEncoding.Hls/` encapsulate the FFmpeg transcoding pipeline with HLS playlist generation as a distinct subsystem.

## Rejected Styles

| Style | Reason |
|---|---|
| Microservices | Single Kestrel process, single database, no inter-process communication; deploys as one binary |
| Modular Monolith | No enforced compile-time module boundaries between assemblies; plugin boundary is runtime, not build-time |
| Event-Driven | `IEventManager` is a secondary pub/sub mechanism, not the primary coordination path; most control flow is direct method call through DI |
| Hexagonal / Ports-and-Adapters | Interfaces exist for layering and plugin extension, not to isolate a pure domain from infrastructure |
| Service-Based | No coarse-grained independently deployable services; the live-TV and transcoding subsystems are in-process |

## Domain

Media Server / Streaming — a privacy-first, free-software media server descended from Emby 3.5.2, ported fully to .NET. Manages personal media libraries (movies, TV, music, photos, live TV/DVR), transcodes streams on-the-fly via FFmpeg to match client capabilities, and serves content to first-party and third-party clients (web, Android, iOS, Roku, Fire TV, Kodi, Apple TV) via a REST API.

---

## Production Context

- Deploys as a single Kestrel HTTP server process with embedded SQLite (EF Core) metadata store; no external service dependencies required.
- Plugin ecosystem (installed as `.dll` folders under the configured plugins path) extends metadata providers, subtitle providers, authentication providers, notification targets, and live-TV tuner integrations.
- FFmpeg transcoding pipeline supports hardware acceleration (VAAPI, NVENC, Intel QSV) with HLS/DASH adaptive streaming negotiated per-client from `MediaInfoController`.
- WebSocket listeners push real-time session state, scheduled task progress, activity log entries, and system notifications to connected clients.
- DLNA server for automatic device discovery on local networks; HDHR and M3U tuner host support for live TV.
- 38k+ GitHub stars; one of the most widely deployed open-source media servers; self-hosted by hundreds of thousands of users.
