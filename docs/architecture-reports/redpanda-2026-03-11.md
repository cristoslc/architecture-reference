# redpanda — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/vectorizedio/redpanda
**Classification:** Modular Monolith + Event-Driven
**Confidence:** 0.92

## Summary

Redpanda is an Apache Kafka-compatible streaming data platform implemented as a single deployable C++ binary built on the Seastar async framework. Its internal structure is a Modular Monolith — dozens of well-bounded subsystems (raft, cluster, storage, kafka, pandaproxy, cloud_storage, transform, wasm, datalake, security, rpc, net, config, model, …) each with their own namespace, BUILD target, logger, and forward-declaration header, all linked into one process and coordinated through direct in-process calls and Seastar's sharded-service model. The system's primary data-handling paradigm is Event-Driven: the core product is a distributed log/event-stream engine, the internal transform and cluster-management planes are organized around event notification and state-machine transitions triggered by Raft leadership events, and the Kafka protocol itself is a produce/consume (publish/subscribe) event-streaming API.

### Key Architectural Files

- `src/v/redpanda/application.h` / `application.cc`: The single `application` class (inheriting `ssx::sharded_service_container`) owns and wires every subsystem. All ~30 sharded services are constructed, started, and torn down here in strict dependency order. One process, one binary.
- `src/v/redpanda/application_services.cc`: `wire_up_redpanda_services` — constructs connection cache, shard table, Raft recovery throttle, Raft group manager, partition manager, cluster controller, storage API, Kafka server, PandaProxy, cloud storage, transform manager, and datalake services sequentially.
- `src/v/redpanda/application_rpc.cc`: `add_runtime_rpc_services` — registers all internal RPC service handlers (raft, cluster, tx_gateway, id_allocator, metadata_dissemination, transforms, datalake coordinator, cloud-topics metastore, cluster-link) on a single `rpc::rpc_server`.
- `src/v/redpanda/application_start.cc`: Registers Raft STM factories (tm_stm, id_allocator_stm, rm_stm, log_eviction_stm, archival_metadata_stm, transform_offsets_stm, datalake state machines) — the Raft log is the central event-sourcing backbone.
- `src/v/raft/consensus.cc` (~4 400 lines): Full Raft consensus implementation — leader election, log replication, snapshotting, recovery, heartbeat management. Every subsystem that needs durability or distributed coordination is a Raft state machine.
- `src/v/cluster/controller.cc` (~1 365 lines): Cluster-level controller, owns the controller Raft group, manages topic table, partition leaders, members, config, security, features.
- `src/v/transform/README.md`: Explicitly documents a control-plane/data-plane split with "loosely coupled in an event emitting pattern" — the control plane is a fiber loop responding to Raft leadership events and data-plane errors; the data plane is three fibers (read → transform → write) connected via buffers.
- `src/v/raft/state_machine_manager.{cc,h}`: Central registry and lifecycle manager for all Raft state machines attached to a consensus group.
- `src/v/storage/disk_log_impl.cc`: Local log engine — segment management, compaction, batch cache, index.
- `src/v/kafka/server/connection_context.{cc,h}`: Per-connection Kafka protocol handler; dispatches all Kafka API request types.

### Patterns Found

**Modular Monolith (primary):**

Every functional domain lives in its own namespace and BUILD target — `raft::`, `cluster::`, `storage::`, `kafka::`, `pandaproxy::`, `cloud_storage::`, `transform::`, `wasm::`, `datalake::`, `security::`, `rpc::`, `net::`, `config::`, `model::` — with explicit forward-declaration headers (`fwd.h`) enforcing interface discipline between modules. Modules communicate via injected references to `ss::sharded<T>` service instances; there is no message bus, no service registry, no process boundary. All modules are linked into a single `redpanda` binary and started by the one `application` class. The `rpk` CLI is a separate Go binary but is a management tool, not a runtime service.

Internal RPC (`src/v/rpc/`) exists exclusively for intra-cluster node-to-node communication (Raft replication, cluster management, shard-to-shard forwarding) — not for intra-process module decoupling. Modules on the same node communicate directly.

The Seastar sharded-service model (`ss::sharded<T>`) means each module is instantiated once per CPU core with in-shard access patterns; cross-shard calls use `invoke_on` or `invoke_on_all`. This is a performance optimization within the monolith, not a microservices decomposition.

**Event-Driven (co-primary):**

Redpanda's product identity is a streaming event platform — the fundamental abstraction is the distributed log, and the Kafka produce/consume API is publish/subscribe over topics. This is not incidental; event streaming is the entire purpose of the system.

Internally, the architecture is organized around events at multiple levels:
1. **Raft log as event source:** Every durable state change (topic creation, partition assignment, configuration, transactions, transforms, archival, datalake) is encoded as a Raft command replicated through the controller Raft group, making the log the authoritative event stream for the cluster.
2. **State machines on the log:** `rm_stm` (transactions), `tm_stm` (transaction coordinator), `id_allocator_stm` (producer ID allocation), `archival_metadata_stm` (tiered storage), `transform_offsets_stm`, datalake `state_machine` — each reacts to log entries as events.
3. **Transform control plane:** Explicitly event-driven — a single fiber loop reacts to Raft leadership change notifications and data-plane error events to schedule/start/stop transform processors.
4. **Cluster controller topology events:** `topic_updates_dispatcher`, `partition_leaders_table`, `members_manager` all propagate change notifications that trigger reactive updates across the cluster.
5. **Cloud Topics write path:** Producer → Kafka Frontend → Batcher → Object Storage → Raft replication of placeholder — a sequential event pipeline from write through cloud acknowledgment.

**Supporting observations:**

- No microservices: all runtime services run in-process; there is no inter-service HTTP/gRPC boundary between kafka, pandaproxy, raft, cluster, or storage subsystems on the same node.
- No CQRS: reads and writes share the same log storage; there is no separate read model aside from the batch cache and materialized manifests for tiered storage.
- No Hexagonal: ports-and-adapters is not the organizing principle; modules are wired by direct reference injection, not through abstract port interfaces (the transform subsystem abstracts its IO for testability, but this is local to one module, not a system-wide pattern).
- No Pipeline as primary style: there is no system-wide pipe-and-filter topology; the event-driven streaming and Raft log are the structural spine.

## Classification Rationale

Redpanda compiles to one binary, one process, many internal modules — that is definitionally a Modular Monolith. The internal modules are well-bounded, namespaced, and communicate through injected references rather than a network boundary. The system's raison d'être and its internal organization around the Raft log, topic/partition event streams, state-machine transitions on log entries, and reactive control planes for transforms and archival make Event-Driven the correct co-classification. The two styles are complementary: the monolithic deployment structure is the implementation vehicle, and the event-driven paradigm is both the product interface and the dominant internal coordination mechanism.
