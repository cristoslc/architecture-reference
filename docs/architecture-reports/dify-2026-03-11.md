# Architecture Report: dify

**Date:** 2026-03-11
**Source:** https://github.com/langgenius/dify
**Version analyzed:** 1.13.0 (shallow clone, commit depth 1)
**Classification:** Layered (primary), Event-Driven (secondary), Microkernel (secondary)
**Confidence:** 0.87

---

## Summary

Dify is an LLM application development platform implemented as a **layered modular monolith** with a clear horizontal tier structure, a Celery-based event-driven async layer, and a microkernel-style plugin extension system. Three deployment units (`api`, `worker`, `web`) share the same codebase and database, differentiating Dify from microservices. The graph workflow engine (`dify_graph`) is its most architecturally sophisticated subsystem, applying Domain-Driven Design principles and strict import-linter-enforced layering internally.

---

## Evidence

### 1. Primary Style: Layered Architecture

The entire backend (`api/`) is organized as a strict horizontal stack with unidirectional dependencies:

| Layer | Path | Role |
|---|---|---|
| Controller | `api/controllers/{console,service_api,web,trigger,inner_api,mcp}/` | HTTP request routing via Flask-RESTX |
| Service | `api/services/` | Business logic orchestration (50+ service classes) |
| Core Domain | `api/core/` | Domain logic: agents, RAG, tools, workflow, model runtime |
| Repository | `api/repositories/` | Protocol-backed data access with SQLAlchemy implementations |
| Model | `api/models/` | SQLAlchemy ORM entities |

The `api/.importlinter` file legally enforces these boundaries. The `dify_graph` subpackage has its own nested layering, documented explicitly:

```
graph_engine → graph_events → graph → nodes → node_events → entities
```

The Flask application factory (`api/app_factory.py`) initializes extensions in dependency order, further evidencing the layered initialization pattern.

### 2. Secondary Style: Event-Driven

Two distinct event buses operate:

**Blinker signals** (`api/events/`): In-process synchronous signals for domain events such as `app_was_created`, `app_model_config_was_updated`, `app_published_workflow_was_updated`. Handlers in `api/events/event_handlers/` respond to these.

**Celery + Redis async task queue** (`api/tasks/`, `api/extensions/ext_celery.py`): 30+ task modules handle decoupled async operations:
- Document indexing: `document_indexing_task.py`, `add_document_to_index_task.py`
- Workflow execution: `async_workflow_tasks.py`
- Mail delivery: `mail_*.py` tasks
- Dataset maintenance: `clean_*.py` tasks

The `worker` deployment unit runs the same image as `api` with `MODE=worker`, consuming tasks from Redis. A separate `worker_beat` container handles Celery periodic scheduling. The graph engine itself uses an event-driven internal model: all node executions emit typed events (`NodeRunStartedEvent`, `NodeRunSucceededEvent`, `NodeRunFailedEvent`, `GraphRunStartedEvent/CompletedEvent`) for monitoring.

**Command channels** in `dify_graph/graph_engine/command_channels/` (InMemory and Redis) allow external control of running workflows (abort, pause, resume) via an explicit command pattern over Redis pub/sub.

### 3. Secondary Style: Microkernel (Plugin System)

The plugin subsystem qualifies as a microkernel pattern:

- **Plugin daemon** (`plugin_daemon` in docker-compose): a separate deployment unit (`langgenius/dify-plugin-daemon`) managing plugin lifecycle, installations, and execution sandboxing
- **Inner API** (`api/controllers/inner_api/`): private HTTP API for bidirectional plugin-core communication
- **Extension scanning** (`api/core/extension/extensible.py`): runtime discovery of built-in and custom extensions using filesystem scanning and dynamic import
- **Extensible subsystems**: `core/tools/builtin_tool/`, `core/tools/plugin_tool/`, `core/tools/mcp_tool/`, `core/moderation/`, `core/datasource/` all support pluggable providers
- **Model provider runtime** (`dify_graph/model_runtime/model_providers/`): provider-agnostic interface with per-provider implementations

### 4. Deployment Topology (Service-Based Characteristics)

`docker/docker-compose.yaml` defines these runtime units:

| Service | Image | Role |
|---|---|---|
| `api` | `langgenius/dify-api:1.13.0` | Flask HTTP server (`MODE=api`) |
| `worker` | `langgenius/dify-api:1.13.0` | Celery async workers (`MODE=worker`) |
| `worker_beat` | `langgenius/dify-api:1.13.0` | Celery beat scheduler (`MODE=beat`) |
| `web` | `langgenius/dify-web:1.13.0` | Next.js frontend |
| `plugin_daemon` | `langgenius/dify-plugin-daemon:0.5.3-local` | Plugin lifecycle manager |
| `sandbox` | `langgenius/dify-sandbox:0.2.12` | Isolated code execution |
| `nginx`, `redis`, `db_postgres`/`db_mysql`, vector DBs | various | Infrastructure |

Critically, `api` and `worker` share the same Docker image and share the same PostgreSQL/MySQL database schema. This confirms the architecture is a **modular monolith with split deployment**, not microservices.

### 5. Graph Workflow Engine: DDD and Layered internals

`api/dify_graph/graph_engine/graph_engine.py` uses DDD terminology explicitly ("aggregate root," "domain models") and enforces:
- **Domain isolation**: `dify_graph.graph_engine.domain` cannot import from infrastructure layers
- **Command/event separation**: commands flow in via channels; events flow out as a stream
- **SOLID principles**: responsibilities delegated to `Dispatcher`, `ExecutionCoordinator`, `GraphStateManager`, `EdgeProcessor`, `WorkerPool`, `EventManager`

This is internal-to-module DDD, not a system-wide pattern.

### 6. App Types and RAG Pipeline

Dify supports multiple LLM application patterns, each with its own runner:
- `api/core/app/apps/chat/` — conversational chat
- `api/core/app/apps/workflow/` — agentic workflow execution
- `api/core/app/apps/agent_chat/` — agent + chat hybrid
- `api/core/app/apps/pipeline/` — RAG pipeline (Pipeline runner wraps the graph engine)
- `api/core/app/apps/advanced_chat/` — advanced chat with workflow integration

---

## What This Is Not

- **Not Microservices**: No bounded context isolation; `api` and `worker` share one codebase and one database schema. No independent service-to-service communication beyond plugin daemon.
- **Not Service-Based**: The deployment units are not independent coarse-grained business services; they are operational splits of one monolith.
- **Not CQRS**: No separate read/write model paths or separate query/command databases.
- **Not Hexagonal**: No explicit ports/adapters interface pattern; the controller-service-repository pattern is layered, not hexagonal.
- **Not Pipeline** (system-wide): While a `Pipeline` app type exists within core, the system is not organized as a series of independent processing stages.
- **Not Multi-Agent**: Agent execution is one app type within the platform, not the system's architectural principle.
