## Style Rationales

**Modular Monolith (primary, 0.91):** Ralph is a single Django application deployed as one process (web + background workers) sharing a single relational database. The codebase is partitioned into 25+ domain-vertical Django apps registered in `INSTALLED_APPS`: `ralph.assets`, `ralph.back_office`, `ralph.data_center`, `ralph.networks`, `ralph.licences`, `ralph.domains`, `ralph.deployment`, `ralph.virtual`, `ralph.operations`, `ralph.configuration_management`, and more. Each app owns its models, migrations, admin registrations, REST API viewsets, serializers, and URL patterns â€” a textbook modular organisation within a single deployable unit. The shared `ralph.lib.*` packages (`transitions`, `permissions`, `custom_fields`, `hooks`, `external_services`, `mixins`, `polymorphic`, `visibility_scope`) provide infrastructure that is intentionally decoupled from any single domain module. There are no separate services; all modules are loaded in-process and share the same database connection pool (`DATABASES["default"]`).

**Layered (secondary, 0.83):** Within each domain app a consistent three-tier layer structure is enforced: (1) **Models** (`models/`, `models.py`) â€” fat domain models inheriting from `BaseObject`/`Asset`, owning business logic and validation; (2) **Admin / Views** (`admin.py`, `views.py`) â€” Django admin panel and custom view handlers that act as the presentation layer; (3) **API** (`api/`) â€” Django REST Framework viewsets and serializers that form a separate read/write interface. The `ralph.lib.transitions` subsystem cross-cuts these layers but is isolated in shared lib rather than embedded per-domain. Middleware in `MIDDLEWARE` (session, CSRF, auth, metrics) applies globally as a horizontal concern. This strict inner layering is consistent across all 25+ domain modules.

**Event-Driven (secondary, 0.72):** Ralph integrates Hermes (via `pyhermes`) as an external message-bus bridge. Three publishing modules emit domain events on save: `ralph.data_center.publishers` publishes `DCHost` update events (host lifecycle changes for DNS/DHCP consumers), `ralph.dns.publishers` publishes DNS TXT record updates on commit, and `ralph.domains.publishers` emits domain lifecycle events. Three subscriber modules consume events: `ralph.assets.subscribers` handles `SERVICE_CREATE/UPDATE/DELETE` Hermes topics to sync service catalog data from upstream; `ralph.operations.changemanagement.subscribtions` consumes change-management events and records them as `Operation` objects; a Hermes webhook endpoint is mounted at `/hermes/` when `ENABLE_HERMES_INTEGRATION=True`. Background jobs (PDF generation, async transitions) are dispatched via Redis Queue (`django-rq`), further decoupling long-running work from the request cycle. However, the event bus is limited to specific cross-system integration points; the dominant internal coordination mechanism remains synchronous Django ORM calls and signals, keeping this style secondary.

| **Integration** | Pyhermes (Hermes broker) for outbound/inbound event integration; REST API (DRF) with token/session/basic auth; XML renderer support for legacy consumers; DNS-as-a-Service (DNSaaS) integration; OpenStack sync for virtual asset discovery |

## Domain

IT Asset Management (ITAM), Data Center Infrastructure Management (DCIM), and Configuration Management Database (CMDB). Core domains: physical asset lifecycle (procurement, assignment, decommission), data center spatial management (racks, server rooms, DC visualization), back-office asset tracking, network/IP address management, license management, deployment automation, domain/SSL certificate tracking, change management integration.

## Production Context

- Deployed as a Django/Gunicorn application with Nginx reverse proxy; Docker Compose provided for self-hosting
- Background workers (`django-rq` + Redis) handle PDF report generation and asynchronous lifecycle transitions
- MySQL is the default database; PostgreSQL is also supported (psycopg3 present in dependencies)
- Optional Redis Sentinel support for high-availability queue and cache
- Hermes integration is opt-in (`ENABLE_HERMES_INTEGRATION`), allowing event-driven sync with an upstream service catalog and DNS-as-a-Service
- OpenStack SDK integration (`python-novaclient`, `openstacksdk`) enables automatic import of virtual server inventory from OpenStack clouds
- Used internally at Allegro; source-available under Apache 2.0 since 2025 under a "sources only" maintenance model
