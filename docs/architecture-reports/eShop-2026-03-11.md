# eShop — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/dotnet/eShop
**Classification:** Microservices + Event-Driven + Domain-Driven Design
**Confidence:** 0.95

## Summary

eShop is the official .NET reference implementation of a cloud-native e-commerce application, intentionally built to demonstrate Microservices architecture orchestrated via .NET Aspire. It combines three mutually reinforcing styles: Microservices as the primary deployment topology (10+ independently deployable services each with its own database), Event-Driven Architecture as the primary integration mechanism (RabbitMQ event bus with transactional outbox), and Domain-Driven Design as the internal structural discipline of the ordering bounded context (aggregates, value objects, domain events, repositories).

## Evidence

### Directory Structure

```
src/
  eShop.AppHost/          # .NET Aspire orchestrator — declares all services and their topology
  eShop.ServiceDefaults/  # Shared cross-cutting concerns (OpenTelemetry, health, auth)
  EventBus/               # Abstraction layer: IEventBus, IntegrationEvent base record
  EventBusRabbitMQ/       # RabbitMQ implementation of IEventBus
  IntegrationEventLogEF/  # Transactional outbox: IntegrationEventLogService<TContext>
  Catalog.API/            # Product catalog (REST, PostgreSQL + pgvector, AI search)
  Basket.API/             # Shopping basket (gRPC, Redis-backed)
  Ordering.API/           # Order management (REST, MediatR CQRS, PostgreSQL)
  Ordering.Domain/        # Pure domain model: aggregates, value objects, domain events
  Ordering.Infrastructure/# EF Core repositories, UnitOfWork, domain event dispatch
  Identity.API/           # OIDC identity provider (Duende IdentityServer)
  Webhooks.API/           # Outbound webhook management (REST, PostgreSQL)
  OrderProcessor/         # Background worker: grace period / order state machine
  PaymentProcessor/       # Background worker: payment processing via event bus
  WebApp/                 # Blazor Server frontend (calls basket, catalog, ordering APIs)
  HybridApp/              # .NET MAUI Blazor Hybrid mobile client (via mobile-bff YARP)
  WebhookClient/          # Webhook receiver test app
```

### Key Architectural Files

- `src/eShop.AppHost/Program.cs`: Declares the complete service graph — 10 services, 4 PostgreSQL databases, 1 Redis instance, 1 RabbitMQ event bus. Services reference each other via typed endpoints (no hardcoded URLs). YARP reverse proxy configured as a mobile BFF.
- `src/EventBus/Abstractions/IEventBus.cs`: Single-method `PublishAsync(IntegrationEvent)` contract; consumer registers typed handlers via subscription pattern.
- `src/EventBusRabbitMQ/RabbitMQEventBus.cs`: RabbitMQ implementation with durable exchanges and consumer-side handler dispatch.
- `src/IntegrationEventLogEF/Services/IntegrationEventLogService.cs`: Transactional outbox — events are persisted atomically in the same DB transaction as the domain change before being relayed to RabbitMQ.
- `src/Ordering.Domain/SeedWork/Entity.cs`: Base `Entity` class carries a `List<INotification> _domainEvents`; aggregates append domain events during state transitions, which are then dispatched post-save.
- `src/Ordering.Domain/AggregatesModel/OrderAggregate/Order.cs`: `Order : Entity, IAggregateRoot` — enforces invariants (only methods can change state), raises domain events on status transitions (OrderStartedDomainEvent, OrderShippedDomainEvent, etc.).
- `src/Ordering.Infrastructure/MediatorExtension.cs`: `DispatchDomainEventsAsync` reads accumulated domain events from the EF Core change tracker and dispatches them via MediatR after `SaveChanges`.
- `src/Ordering.API/Application/Commands/`: 9 command/handler pairs (CreateOrderCommand, CancelOrderCommand, ShipOrderCommand, etc.) — CQRS command side.
- `src/Ordering.API/Application/Queries/IOrderQueries.cs` + `OrderQueries.cs`: Raw SQL read model — CQRS query side separate from EF write model.
- `src/Ordering.API/Application/Behaviors/TransactionBehavior.cs`: MediatR pipeline behavior wrapping each command in a DB transaction and triggering outbox flush post-commit.
- `src/Basket.API/Grpc/BasketService.cs`: gRPC service exposing `GetBasket` / `UpdateBasket` over Protobuf (heterogeneous protocol — not all services use REST).
- `src/PaymentProcessor/Program.cs`: Pure event consumer — registers one subscription `OrderStatusChangedToStockConfirmedIntegrationEvent → handler`, no HTTP endpoints.
- `src/OrderProcessor/Program.cs`: Background worker subscribing to grace-period events to progress order state machine.

### Patterns Found

**Microservices (Primary):**
Each service is an independently deployable unit with its own `.csproj`, `Program.cs` entry point, and data store. AppHost enumerates `catalogdb`, `identitydb`, `orderingdb`, `webhooksdb` as separate databases; Basket.API uses Redis exclusively. Services expose heterogeneous APIs — REST (Catalog, Ordering, Webhooks), gRPC (Basket), OIDC (Identity). Deployment is via .NET Aspire container orchestration, with each service scalable and replaceable independently.

**Event-Driven Architecture (Primary co-style):**
All cross-service side effects are mediated through the `IEventBus` / RabbitMQ layer using integration events. The order lifecycle is entirely event-choreographed: `OrderStartedIntegrationEvent` → identity verification → `OrderStatusChangedToAwaitingValidationIntegrationEvent` → Catalog.API stock check → `OrderStockConfirmedIntegrationEvent` / `OrderStockRejectedIntegrationEvent` → PaymentProcessor → `OrderPaymentSucceededIntegrationEvent` / `OrderPaymentFailedIntegrationEvent`. The `IntegrationEventLogService` (transactional outbox) guarantees at-least-once delivery without distributed transactions.

**Domain-Driven Design (Strong secondary):**
The Ordering bounded context is a textbook DDD implementation:
- `Ordering.Domain` is a pure domain layer with zero infrastructure dependencies.
- `Order` and `Buyer` are IAggregateRoot-implementing aggregates protecting their invariants.
- `Address` is a value object persisted as an EF Core owned entity.
- 7 domain events in `Ordering.Domain/Events/` capture state transitions.
- DomainEventHandlers in `Ordering.API` translate domain events into integration events (bridge domain → integration layer).
- `IOrderRepository` / `IBuyerRepository` define the repository ports; `Ordering.Infrastructure` provides EF Core implementations.

**CQRS (Structural pattern within Ordering.API):**
The Ordering.API applies CQRS at the service level: write model uses MediatR command handlers + EF Core aggregates; read model uses `IOrderQueries` with raw SQL projections. This is a service-internal pattern, not a system-level architecture, hence not classified as a primary style.

### Rejected Styles

- **Modular Monolith:** Multiple independently deployable processes with separate databases. Not a single deployable unit.
- **Layered:** No horizontal layer model spanning services. Each service has its own vertical slice.
- **Service-Based:** Services do not share a monolithic database; each owns its own store, which is the defining Microservices distinction.
- **Pipeline:** No data-processing pipeline abstraction. Event flows are choreography, not extract-transform-load.
- **Microkernel:** No plug-in host or core + extension registry pattern.
- **Serverless:** Services are long-running containers, not function-as-a-service.
- **Space-Based:** No tuple space, distributed grid, or processing unit / virtualized middleware model.
- **Hexagonal Architecture:** While Ordering.Domain uses port/adapter–style repository interfaces, this is tactical DDD, not the primary architectural framing of the entire system.
- **Multi-Agent:** No autonomous agent or LLM orchestration model (OpenAI/Ollama integration is optional catalog AI search, not a core architectural style).

## Quality Attributes

- **Scalability:** Each service scales independently; Basket.API stateless via Redis; .NET Aspire supports replicated deployments.
- **Loose Coupling:** Services communicate only through the event bus or typed HTTP endpoints; no shared databases (except Redis for basket state).
- **Fault Isolation:** Service failures are bounded; transactional outbox prevents event loss on publisher crash.
- **Observability:** eShop.ServiceDefaults injects OpenTelemetry tracing, metrics, and structured logging uniformly across all services.
- **Maintainability:** DDD bounded contexts (especially Ordering) enforce encapsulation of business rules; CQRS separates read and write complexity.
- **Extensibility:** Optional AI integration (OpenAI / Ollama) is additive and does not alter the core architecture.
