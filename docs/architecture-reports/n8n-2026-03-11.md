# n8n — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/n8n-io/n8n
**Classification:** Pipeline + Modular Monolith
**Confidence:** 0.88

## Summary

n8n is a self-hostable workflow automation platform whose primary architectural expression is the Pipeline pattern: users compose directed graphs of integration nodes through which data flows step-by-step. The runtime is packaged as a Modular Monolith — all backend logic lives in a single deployable process organized into sharply bounded internal modules (scaling, eventbus, webhooks, modules/, task-runners, etc.) rather than independent services, even though it supports a separate queue-worker process for scaled deployments. Event-Driven and Multi-Agent are significant secondary styles introduced by the event bus infrastructure and the LangChain-based AI agent system.

### Pipeline Pattern (Primary)

The fundamental user-visible abstraction is the directed node graph: workflow nodes are connected in sequence (or with branches/merges), and item arrays flow through each node in turn. The `workflow` package models this as a graph with typed connections; `WorkflowExecute` in `n8n-core` traverses and executes it node-by-node. Each node is a stateless transformation function with typed inputs and outputs — a textbook pipe-and-filter / pipeline topology. The 300+ nodes in `nodes-base` and the LangChain AI nodes in `@n8n/nodes-langchain` are all pipeline stages implementing a common `INodeType` interface. Triggers (webhooks, schedulers, pollers) act as pipeline sources.

### Modular Monolith (Primary)

All backend code runs inside the `cli` package as a single Express process. Feature domains are decomposed into explicit, independently-loaded backend modules under `packages/cli/src/modules/` (each decorated `@BackendModule`), covering MCP, insights, source-control, LDAP, SSO, chat-hub, and workflow-builder. The `@n8n/di` container and `@n8n/decorators` package wire these modules together at startup with formal controller, service, and repository boundaries. The database layer is isolated in `@n8n/db` with its own entity/repository/migration structure. This is a well-bounded modular monolith, not microservices: all modules share a single process, database connection, and in-process DI container.

### Event-Driven (Secondary)

Two distinct event-driven mechanisms operate in parallel:

1. **MessageEventBus** (`eventbus/`) — a durable, file-backed event log that streams execution and audit events to external destinations (syslog, webhooks). It provides at-least-once delivery semantics with crash recovery.
2. **EventService** (`events/event.service.ts`) — an internal typed EventEmitter used for cross-service coordination (AI events, queue metrics, relay events) without coupling service classes directly.

In queue mode, a **Redis pub/sub command bus** (`scaling/pubsub/`) propagates control commands (reload-license, reload-saml-config, community-package-install, get-worker-status, multi-main leader messages) across multiple main instances and worker processes.

### Multi-Agent (Secondary)

The `@n8n/ai-workflow-builder.ee` package implements a multi-agent orchestration system using LangChain's `StateGraph` (LangGraph). A **SupervisorAgent** coordinates a **PlannerAgent**, **BuilderSubgraph**, and **DiscoverySubgraph**, each implemented as autonomous LangGraph state machines. At the workflow level, n8n's AI Agent nodes (v1–v3) support tool-calling and multi-agent orchestration patterns (orchestrator/subagent, MCP tool exposure), documented as a first-class pattern in the node's `builderHint`.

### Scaling Mode

A formal `ScalingService` based on **BullMQ** (Redis-backed job queue) allows n8n to run as a multi-process system: one or more `main` processes handle the API and scheduling, while one or more `worker` processes (started via `n8n worker`) consume jobs from the queue. Multi-main leader election uses Redis TTL keys. This is an optional operational mode — the default single-process deployment is a true monolith.

## Classification Rationale

Pipeline is the dominant style because the workflow DAG and its execution engine are the entire reason n8n exists — every other subsystem (database, auth, scaling, MCP, AI builder) exists to support the creation and execution of pipelines. Modular Monolith is the dominant runtime style: despite a large number of packages in the pnpm workspace, all backend packages resolve into a single deployable process with in-process communication and shared state. Event-Driven is a genuine secondary style (two distinct event systems, a Redis pub/sub bus) but is supporting infrastructure, not the primary organizing principle. Multi-Agent is an emerging secondary style, present in the AI workflow builder and the LangChain agent node palette, but not the product's core execution model.
