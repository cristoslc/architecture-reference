## Style Rationales

**Event-Driven (primary, 0.96):** Pulsar is a distributed pub-sub messaging and streaming platform. Every component is oriented around producing and consuming events on named topics. The `BrokerService` in `pulsar-broker/src/main/java/org/apache/pulsar/broker/service/BrokerService.java` bootstraps a Netty server to handle persistent and non-persistent topic I/O; `PersistentTopic` and `PersistentDispatcherMultipleConsumers` in `pulsar-broker/src/main/java/org/apache/pulsar/broker/service/persistent/` manage fan-out dispatch to subscriber cohorts with per-topic cursors. The `pulsar-io/` module contains 30+ pluggable source/sink connectors (Kafka, Debezium CDC, Cassandra, Elasticsearch, JDBC, Kinesis, MongoDB, RabbitMQ, Redis, etc.) that form an event ingestion and egress pipeline. The `pulsar-functions/` module provides a serverless function-as-a-service layer (`FunctionRuntimeManager`, `RuntimeSpawner`) for arbitrary event processing within the same broker topology. System topics (`pulsar-broker/src/main/java/org/apache/pulsar/broker/systopic/`) carry internal coordination events (namespace policies, topic policies, transaction buffer snapshots). Geo-replication is expressed as a push-based `PersistentReplicator` writing to remote cluster topics. The transaction subsystem (`pulsar-transaction/`) provides exactly-once semantics over the event log via a two-phase-commit coordinator topic. The client API (`pulsar-client-api/`) exposes producer, consumer, reader, and table-view primitives entirely around the event abstraction.

**Layered (primary, 0.88):** Pulsar enforces a strict three-layer storage separation: (1) a stateless broker tier (`pulsar-broker/`) that owns topic ownership, routing, dispatch, and protocol handling but holds no durable state itself; (2) a managed ledger tier (`managed-ledger/`) that abstracts Apache BookKeeper into a sequentially appended, cursor-tracked log abstraction (`ManagedLedger`, `ManagedCursor`); and (3) the Apache BookKeeper persistence tier that physically stores ledger segments across bookie nodes. This separation means broker nodes can be added or removed without data migration — the managed ledger and bookie layers are independently scalable. Above the broker sits a metadata layer (`pulsar-metadata/`) backed by ZooKeeper or etcd, used for namespace ownership, broker registry, load balancing state, and configuration. The proxy tier (`pulsar-proxy/`) adds a stateless routing and TLS-termination layer in front of the broker tier. Clients access the broker through the binary protocol or via the REST/WebSocket adapters (`pulsar-websocket/`).

**Microkernel (secondary, 0.81):** The broker defines multiple named extension points loaded at runtime from NAR (NiFi ARchive) packages. Protocol handlers (`pulsar-broker/src/main/java/org/apache/pulsar/broker/protocol/ProtocolHandler.java`) allow plugging custom wire protocols into the broker (used by KoP — Kafka on Pulsar — and AoP — AMQP on Pulsar) without modifying core code. Entry filters (`EntryFilter.java`, `EntryFilterProvider.java`) allow pluggable per-message filtering on the dispatch path. Broker interceptors (`BrokerInterceptor.java`) hook into the full producer/consumer lifecycle. The IO connector framework (`pulsar-io/core/`) defines a `Source`/`Sink` SPI. Functions runtimes (`pulsar-functions/runtime/`) support thread, process, and Kubernetes runner backends selected at deployment time. Tiered storage offloaders (`tiered-storage/`) are loaded from NAR packages to offload cold data to S3, GCS, Azure Blob, HDFS, or Alluxio. Load manager extensions (`pulsar-broker/src/main/java/org/apache/pulsar/broker/loadbalance/`) support pluggable load ranking and shedding strategies.

## Domain

Distributed messaging and event streaming platform. Core domains: durable topic pub-sub, geo-replication, multi-tenancy and access control, serverless event processing (Functions), event-driven integration (IO connectors), tiered cold storage, transaction coordination, and schema registry.

## Production Context

- Deployed as a cluster of stateless brokers fronted by a proxy, backed by a BookKeeper bookie ensemble and a ZooKeeper (or etcd) metadata store; Kubernetes Helm charts and Terraform/Ansible deployments in `deployment/`
- Broker, Functions Worker, and IO Connector processes run as separate JVM processes but share the same binary distribution; Docker images in `docker/pulsar/` and `docker/pulsar-all/`
- 30+ IO connectors and tiered-storage offloaders packaged as NAR files, installable without broker restart via the package management API (`pulsar-package-management/`)
- KoP (Kafka on Pulsar) and AoP (AMQP on Pulsar) protocol handlers allow Pulsar to serve Kafka and AMQP workloads transparently on the same broker fleet via the ProtocolHandler SPI
- Functions Worker (`pulsar-functions/worker/`) can run co-located inside the broker or as a standalone cluster; supports thread, process, and Kubernetes pod runtimes
