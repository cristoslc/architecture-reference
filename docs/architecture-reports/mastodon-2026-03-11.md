## Summary

Mastodon is a federated, open-source social network server built on the ActivityPub protocol. Its architecture combines a Rails-based **Modular Monolith** as the primary web and API application with a strong **Event-Driven** core powered by Redis pub/sub and Sidekiq queues. A distinct, independently deployed Node.js streaming service handles real-time WebSocket delivery — a clear **Service-Based** separation. The result is a hybrid: a Rails monolith with service-extracted real-time infrastructure and an entirely event-driven internal distribution model.

## Process Topology

Four process types are deployed from this single repository:

| Process | Runtime | Responsibility |
|---|---|---|
| `web` | Ruby / Puma | REST API, ActivityPub endpoints, OAuth, admin UI, web views |
| `sidekiq` | Ruby / Sidekiq | Async job processing: federation delivery, fan-out, notifications, media |
| `streaming` | Node.js / Express + WS | Real-time WebSocket and SSE feeds to clients |
| `vite` (dev only) | Node.js | Frontend asset bundling in development |

All four share the same PostgreSQL database and Redis instance, co-deployed via Docker Compose or Helm chart. The streaming server is independently containerizable (`streaming/Dockerfile`) and is documented as a "separate app" in the Heroku deployment guide.

## Primary Style Evidence: Modular Monolith

The Rails application is organized into clear vertical slices within a single deployable unit:

- **`app/controllers/`** — REST API (`api/v1/`, `api/v2/`, `api/v1_alpha/`), ActivityPub endpoints (`activitypub/`), admin UI, OAuth, auth
- **`app/models/`** — Rich domain model with many `concerns/` for mixins (Cacheable, Redisable, RateLimitable, Paginable, etc.)
- **`app/services/`** — Service object layer (~50+ service classes) encapsulating domain operations: `FanOutOnWriteService`, `FavouriteService`, `BlockDomainService`, `NotifyService`, etc.
- **`app/workers/`** — Sidekiq workers translating queued jobs into service calls; organized by domain: `activitypub/`, `fasp/`, `admin/`, `import/`
- **`app/serializers/`** — ActivityStreams / JSON-LD serialization layer
- **`app/chewy/`** — Elasticsearch index definitions (Chewy gem)
- **`app/policies/`** — Pundit authorization policies
- **`app/presenters/`** — View presenters

The layering is: **Controllers → Services → Models → Database**, with workers as an async trigger layer sitting above services. All code shares one PostgreSQL store and one Redis cache. There is no internal module boundary enforcement (no engine-per-domain), so this is a monolith with good internal organization rather than a strict modular monolith with enforced isolation.

## Primary Style Evidence: Event-Driven

The event-driven character is pervasive and central to how Mastodon functions:

### Sidekiq Queue Architecture

`config/sidekiq.yml` declares seven named queues with explicit priority weights:

```
default (8), push (6), ingress (4), mailers (2), pull (1), scheduler (1), fasp (1)
```

Every significant user action dispatches an async job rather than performing work inline:
- Posting a status → `DistributionWorker` → `FanOutOnWriteService`
- Receiving a federated activity → `ActivityPub::ProcessingWorker`
- Delivering to remote instances → `ActivityPub::DeliveryWorker` (queue: `push`, retry: 16)
- Sending email → standard Rails mailer via `mailers` queue
- Scheduled cleanup, trend refresh, indexing → `Scheduler::*` workers via cron/interval

`FanOutOnWriteService` is the canonical event-dispatch hub: it fans a new status out to follower home feeds, public feeds, and Redis streams in a single coordinated operation triggered by the `DistributionWorker`.

### Redis Pub/Sub for Real-Time

The streaming server (`streaming/index.js`, `streaming/redis.js`) subscribes to Redis channels that the Rails application publishes to via `ActionCable`-compatible messages. Named channels include `user`, `user:notification`, `timeline:public`, `timeline:hashtag:*`, `timeline:list:*`, and others. The streaming server relays these Redis messages over WebSocket or SSE to connected clients in real time.

This is a textbook event broker pattern: Rails publishes events, Redis brokers them, the Node.js streaming service subscribes and pushes to end clients.

### Scheduler Jobs

Sidekiq Scheduler runs 15+ recurring jobs for background maintenance: trend refresh (every 5m), indexing (every 1m), vacuum (nightly), instance refresh (hourly), user cleanup (daily), software update checks (every 30m), and more.

## Secondary Style Evidence: Service-Based

The streaming server is the clearest case of service extraction:

- **Independent codebase:** `streaming/` is a self-contained Node.js package (`@mastodon/streaming`) with its own `package.json`, `Dockerfile`, `tsconfig.json`, and dependencies (Express, ioredis, prom-client, ws)
- **Independent deployment:** Separate container in `docker-compose.yml` (`mastodon-streaming` image), separate port (4000 vs 3000), separate health check endpoint
- **Shared data store:** Connects directly to the same PostgreSQL (for auth token validation) and Redis (for pub/sub channel subscription) — no API contract with the Rails app
- **Single purpose:** Only handles WebSocket/SSE streaming; no business logic, no writes
- **Prometheus metrics:** Exports its own metrics via prom-client, including per-channel counters

The federation delivery subsystem (`ActivityPub::DeliveryWorker` with stoplight circuit-breaker, exponential backoff retry logic, and `DeliveryFailureTracker`) also exhibits service-like autonomy within the monolith, though it is not separately deployed.

## What Was Rejected and Why

**Not Microservices:** All process types (`web`, `sidekiq`, `streaming`) share the same PostgreSQL and Redis — there is no independent data store per service. The Rails web and sidekiq processes run identical code from a single image. The streaming server is a purposeful extraction, not a decomposition into many independent services.

**Not Layered (primary):** While the Rails app has recognizable horizontal layers (controllers → services → models), the dominant architectural concern is the async event queue and federation protocol, not a pure layered request-response stack. The presence of services, workers, serializers, and policies adds vertical decomposition that goes beyond classic layered architecture.

**Not CQRS:** While reads and writes follow different paths in some cases (Elasticsearch for search reads, PostgreSQL for writes), there is no explicit command/query segregation pattern with separate models and handlers enforced architecturally.

**Not Domain-Driven Design:** There is no explicit domain aggregate or bounded context structure. Models are organized by entity type, not by domain boundary, and there is no ubiquitous language enforced in code organization.

## Quality Attributes

### Federation and Interoperability
ActivityPub protocol implementation is first-class. The `app/lib/activitypub/` library handles JSON-LD serialization, signature verification, dereferencing, and inbox/outbox processing. WebFinger, NodeInfo, and host-meta endpoints are all implemented. This enables federation with any ActivityPub-compatible server (Pixelfed, Pleroma, Misskey, etc.).

### Scalability
- Sidekiq concurrency is configurable via `SIDEKIQ_CONCURRENCY` env var
- Multiple Sidekiq processes or workers can run in parallel against the same queues
- The streaming server is independently scalable (horizontal scale behind a load balancer)
- Elasticsearch is optionally integrated via Chewy for full-text search at scale

### Resilience
- `ActivityPub::DeliveryWorker` implements exponential backoff (up to 16 retries) with custom jitter
- `DeliveryFailureTracker` implements a stoplight circuit-breaker (10 failures → 60s cooloff) per inbox URL
- Sidekiq's at-least-once delivery ensures jobs survive process restarts
- Redis Sentinel support is implemented in both the Rails app and the streaming server

### Observability
- Prometheus metrics exported by the streaming server (`prom-client`)
- PgHero integration for PostgreSQL query monitoring (accessible via admin)
- Sidekiq web UI available to admin roles
- Structured logging throughout

### Security
- OAuth2 via Doorkeeper for API access
- HTTP Signatures on all ActivityPub deliveries (verified on ingress)
- Pundit policies for authorization
- Content Security Policy middleware
- WebAuthn / security key support for 2FA

### Deployability
- Docker Compose for single-server production deployment
- Helm chart maintained in a separate `mastodon/chart` repository
- Heroku and Scalingo support via `app.json` / `scalingo.json`
- Single image (`ghcr.io/mastodon/mastodon`) for both `web` and `sidekiq` processes

## Classification Reasoning

Mastodon's architecture cannot be reduced to a single canonical style. The **Modular Monolith** label applies to the Rails application itself — a well-organized single deployable unit with clear internal layering and service objects, but no enforced module boundaries. The **Event-Driven** label applies to its runtime behavior — virtually every significant operation is dispatched asynchronously via Sidekiq queues, and real-time delivery is mediated by Redis pub/sub as an event broker. The **Service-Based** label applies to the deliberate extraction of the streaming function into a separately deployed Node.js service that communicates through a shared data tier rather than through an API.

The event-driven pattern is not incidental; it is the architectural mechanism that makes federation at scale possible. Delivering to potentially thousands of remote ActivityPub inboxes requires durable queuing, retry logic, circuit-breaking, and concurrency control — all implemented via the Sidekiq + Redis event pipeline.

This combination — a Modular Monolith with Event-Driven internals and a Service-Based streaming extraction — is well-suited to Mastodon's requirements: a single operator runs a complete social network instance, needs simple deployment, but also needs real-time delivery and federated distribution across a decentralized network.
