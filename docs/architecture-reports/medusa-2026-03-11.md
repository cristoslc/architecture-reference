## Executive Summary

Medusa is an open-source commerce platform built as a **Modular Monolith** with strong **Event-Driven** and **Domain-Driven Design** characteristics. The codebase is organized as a Yarn workspace monorepo governed by Turborepo, shipping a single deployable Node.js/Express application (`packages/medusa`) composed of 35+ independently defined, DI-container-wired commerce modules. Each module owns its own data models, migrations, and service layer. Cross-module operations are coordinated by a first-class **Workflow Engine** that implements a distributed-transaction (saga with compensation) pattern. An event bus (local or Redis-backed via BullMQ) propagates domain events to subscribers asynchronously. The result is an architecture that deploys as one process but whose internal decomposition mirrors microservices-style bounded contexts without the network hop overhead.

## Architecture Styles Identified

### Primary: Modular Monolith

**Evidence:**

- The root `package.json` declares a Yarn workspace covering `packages/modules/*`, `packages/core/*`, and the main `packages/medusa` application.
- `packages/core/utils/src/modules-sdk/definition.ts` defines 31 named module keys (`Modules` enum) — `CART`, `ORDER`, `PRODUCT`, `PAYMENT`, `FULFILLMENT`, `NOTIFICATION`, etc. — each resolved to a distinct npm package path.
- Each module under `packages/modules/<domain>/src/` owns its own:
  - Data models and MikroORM migrations (`models/`, `migrations/`)
  - Service implementation (e.g., `packages/modules/cart/src/services/cart-module.ts`)
  - Joiner config (GraphQL-like service descriptor for the RemoteQuery federation layer)
- `packages/core/modules-sdk/src/module-loader.ts` bootstraps all modules into a shared **Awilix DI container** at startup. Modules are registered by key and resolved by interface — they share no direct import dependencies at runtime.
- `packages/core/framework/src/medusa-app-loader.ts` orchestrates the full startup: config loading, DI container setup, module loading, link-module registration, subscriber wiring, and workflow registration.
- The `packages/core/modules-sdk/src/remote-query/remote-query.ts` implements a **RemoteJoiner** that federates cross-module queries without inter-module HTTP calls — modules stay in the same process but query each other through a joiner protocol.

**Inter-module linking:**
`packages/modules/link-modules/` defines relationship tables (e.g., cart-product, order-product) that cross module boundaries without breaking module ownership. `packages/core/utils/src/modules-sdk/define-link.ts` exposes the `defineLink()` utility for declaring these relationships declaratively.

### Secondary: Event-Driven

**Evidence:**

- Two swappable Event Bus module implementations: `packages/modules/event-bus-local/` (in-process EventEmitter) and `packages/modules/event-bus-redis/` (BullMQ queue backed by Redis).
- `packages/modules/event-bus-redis/src/services/event-bus-redis.ts` uses BullMQ `Queue` + `Worker` for async, durable event delivery.
- The `@EmitEvents()` decorator (`packages/core/utils/src/modules-sdk/decorators/emit-events.ts`) wraps module service methods — after each write operation, accumulated events from a `MessageAggregator` are flushed to the event bus.
- `packages/core/framework/src/subscribers/subscriber-loader.ts` dynamically loads subscriber files from `packages/medusa/src/subscribers/` (and plugin directories) and wires them to the event bus at startup.
- The `emitEventStep` is a first-class workflow step in `packages/core/core-flows/src/common/steps/emit-event.ts`, used within saga workflows (e.g., `create-carts.ts`) to emit `CartWorkflowEvents` at defined lifecycle points.
- Core subscriber example (`packages/medusa/src/subscribers/configurable-notifications.ts`) listens for `order.created` and triggers notification delivery — a classic reactive event subscriber.

### Secondary: Domain-Driven Design

**Evidence:**

- The `Modules` enum maps 31 bounded contexts that align precisely with commerce domain concepts: `CART`, `ORDER`, `CUSTOMER`, `PRODUCT`, `PRICING`, `PROMOTION`, `FULFILLMENT`, `PAYMENT`, `INVENTORY`, `STOCK_LOCATION`, `REGION`, `SALES_CHANNEL`, `TAX`, `NOTIFICATION`, `AUTH`, `USER`, `RBAC`, `ANALYTICS`, etc.
- Each module has an `IModuleService` interface in `packages/core/types/src/` defining the public API of that bounded context.
- The Link Modules pattern defines explicit cross-aggregate relationships (e.g., `product` ↔ `variant` ↔ `inventory-item`) without coupling service implementations.
- Workflow orchestration in `packages/core/core-flows/src/` is organized by domain: `cart/`, `order/`, `payment/`, `fulfillment/`, `product/`, etc. Each domain directory contains `workflows/` and `steps/` sub-directories that encode domain operations as composable units.
- The DML (Domain Modeling Layer) abstraction in `packages/core/utils/src/dml/` provides a schema-definition API for module entities, further reinforcing bounded-context encapsulation.

## Workflow / Saga Layer

The workflow engine is a defining architectural feature distinct from common modular monolith implementations:

- `packages/core/orchestration/src/transaction/transaction-orchestrator.ts` implements a **distributed transaction orchestrator** (Saga pattern) with forward execution and compensation (rollback) steps.
- `packages/core/orchestration/src/transaction/distributed-transaction.ts` tracks multi-step transaction state across async boundaries, supporting retry, timeout, revert, and checkpoint semantics.
- `packages/core/orchestration/src/workflow/workflow-manager.ts` maintains a registry of named workflows, each with an `OrchestratorBuilder`-defined step graph.
- Two Workflow Engine module backends: `packages/modules/workflow-engine-inmemory/` (for local dev) and `packages/modules/workflow-engine-redis/` (for production with Redis-persisted state).
- `packages/core/workflows-sdk/src/medusa-workflow.ts` + `createWorkflow()` / `createStep()` provide the developer-facing DSL for composing workflows with `parallelize()`, `transform()`, `createHook()`, and `WorkflowResponse`.

A canonical workflow (`packages/core/core-flows/src/cart/workflows/create-carts.ts`) demonstrates the pattern: steps for region resolution, customer lookup, sales channel validation, inventory confirmation, line-item creation, pricing, promotions, tax calculation, and payment collection are composed into a single saga with compensation paths.

## HTTP / API Layer

- Express.js is the HTTP runtime, loaded by `packages/core/framework/src/http/express-loader.ts`.
- Routes are file-system-based: `packages/medusa/src/api/admin/` and `packages/medusa/src/api/store/` each organize routes by domain resource (products, orders, carts, customers, etc.).
- Route handlers are thin: they validate input, invoke a named workflow (e.g., `createProductsWorkflow(req.scope).run(...)`) or execute a RemoteQuery, and serialize the response.
- Middleware and policy loaders (`packages/core/framework/src/policies/`) provide RBAC authorization.
- The Admin UI is a React application in `packages/admin/` served as a separate build artifact.

---

## Plugin / Extension System (Microkernel Influence)

- Module implementations are swappable: e.g., `event-bus-local` ↔ `event-bus-redis`, `cache-inmemory` ↔ `cache-redis`, `workflow-engine-inmemory` ↔ `workflow-engine-redis`, `file-local` ↔ `file-s3`, `notification-local` ↔ `notification-sendgrid`.
- Payment providers (`payment-stripe`), auth providers (emailpass, GitHub, Google), fulfillment providers, and analytics providers are registered as module providers.
- Plugin directories can contribute API routes, subscribers, workflows, and jobs — loaded by the framework's `ResourceLoader` during startup.
- This extensibility mechanism is microkernel-adjacent but does not constitute a full Microkernel architecture since core modules are not pluggable kernel extensions but independently deployable commerce services.

## Classification Reasoning

Medusa is classified primarily as a **Modular Monolith** because:

1. It deploys as a single Node.js process with a shared database (PostgreSQL) and a shared DI container.
2. Its 35 commerce modules are internally well-decomposed with clear bounded contexts but communicate through in-process DI resolution and a RemoteJoiner — not network APIs.
3. Module implementations are interchangeable via configuration (microkernel-like plugin slot), but the core application structure is not itself a microkernel.

The **Event-Driven** secondary style is warranted by the first-class event bus architecture: a `@EmitEvents()` decorator on every write operation, a subscriber lifecycle that wires handlers at startup, and BullMQ-backed async durable event delivery in production.

The **Domain-Driven Design** secondary style is warranted by the explicit bounded-context decomposition (31 named modules), DML entity modeling, joiner config service descriptors, and the domain-organized workflow/step library in `core-flows`.

The system is **not** classified as Microservices because all modules share a process, a PostgreSQL instance, and an Awilix container. It is **not** classified as a Pipeline because the workflow orchestration is a saga coordination mechanism, not a data-transformation pipeline. It is **not** classified as Microkernel because the core framework is not a minimal kernel with plug-in services replacing core functionality.
