# Architecture Report: Zammad

**Date:** 2026-03-11
**Repo URL:** https://github.com/zammad/zammad
**Classification:** Layered, Event-Driven
**Confidence:** 0.91

---

## Summary

Zammad is an open-source helpdesk and customer support platform built as a Ruby on Rails monolith deployed from a single codebase. The primary architectural style is **Layered**: a canonical Rails stack of controllers, service objects, authorization policies, models (ActiveRecord), background jobs, and a Vue.js SPA frontend. On top of this layered structure sits a purpose-built, first-party **Event-Driven** dispatching system: `TransactionDispatcher` + `EventBuffer` capture ActiveRecord lifecycle events (create/update) from across the domain model, buffer them within the database transaction boundary, and fan out to registered sync and async backends — including `Transaction::Trigger`, `Transaction::Notification`, `Transaction::SignatureDetection`, `Transaction::CtiCallerIdDetection`, and `Transaction::ClearbitEnrichment`. This internal event bus is the primary mechanism through which cross-cutting domain side-effects are handled without tight model coupling, making Event-Driven a genuine secondary structural style alongside the layered Rails scaffolding.

## Evidence from Code Exploration

### Deployment topology

- `Procfile` defines three processes: `web` (Rails/Puma), `websocket` (custom `script/websocket-server.rb` on port 6042), and `worker` (`script/background-worker.rb`). All three originate from the same codebase — single deployable unit.
- `Gemfile` defines one monolithic dependency graph; no service-to-service communication.
- Single PostgreSQL database (`config/database/database.yml`); 482+ migration files confirm a unified schema.
- `config/cable.yml` configures ActionCable (backed by Redis) for WebSocket subscriptions.
- Background worker delegates to `BackgroundServices::Cli`, which manages services including `ProcessDelayedJobs`, `ProcessScheduledJobs`, `ManageSessionsJobs`, `ProcessDelayedCommunicationInboundJobs`, and `ProcessDelayedAiJobs`.

### Layered Rails structure (`app/`)

The application follows clear horizontal layering:

- **Controllers** (`app/controllers/`) — HTTP request handling; REST API namespaced by feature domain. 95 modular route files in `config/routes/` organized by domain (ticket, user, channel_email, channel_whatsapp, ai_agent, knowledge_base, etc.) confirm a comprehensive REST API surface.
- **GraphQL** (`app/graphql/gql/`) — A complete GraphQL layer (`Gql::ZammadSchema`) with `mutations/`, `queries/`, `subscriptions/`, `types/`, `resolvers/`, `fields/`, and `context/` subdirectories. Mounted at `/graphql` via POST/OPTIONS. Subscriptions use ActionCable for real-time delivery.
- **Service Layer** (`app/services/service/`) — Services organized by domain area: `ticket/`, `user/`, `auth/`, `channel/`, `ai/`, `ai_assistance/`, `history/`, `link/`, `template/`, `translation/`, `system/`, `external_data_source/`. Services inherit from `Service::Base` with an `execute` method — controllers and GraphQL mutations delegate to services rather than embedding logic.
- **Policy Layer** (`app/policies/`) — 51 items (32 policy files + 19 subdirectories) using Pundit authorization. Every controller action is gated by an authorization check. Covers all major domain models.
- **Models** (`app/models/`) — 114+ ActiveRecord models. Rich shared behavior via 45 concerns (`has_history`, `has_tags`, `has_online_notifications`, `can_search`, `can_selector`, `checks_html_sanitized`, `has_transaction_dispatcher`, etc.).
- **Jobs** (`app/jobs/`) — 41 ActiveJob-backed background jobs covering AI assistance, communication (Facebook, SMS, Telegram, WhatsApp, email), escalation, search indexing, webhook triggering, data privacy, and session management.
- **Mailers** (`app/mailers/`) — Email delivery layer.
- **Frontend** (`app/frontend/`) — Vue.js/TypeScript SPA with separate `desktop` and `mobile` applications, built via Vite, communicating to the backend via REST API and GraphQL/ActionCable.

The depth of the service layer, policy layer, and the strict separation of GraphQL as its own access surface distinguish this from a vanilla Rails MVC app and confirm layered architecture as the primary style.

### Event-Driven internal dispatch (`lib/transaction_dispatcher.rb`, `app/models/concerns/has_transaction_dispatcher.rb`)

This is the most architecturally significant finding beyond the Rails layering:

**Event capture:** `HasTransactionDispatcher` is a Rails concern included in all major domain models. It registers `after_create TransactionDispatcher` and `after_update TransactionDispatcher` hooks — every model change that matters is automatically captured.

**EventBuffer:** On create/update, `TransactionDispatcher.after_create` / `.after_update` push an event descriptor (`{object:, type:, id:, changes:, user_id:, created_at:}`) into `EventBuffer` under the `'transaction'` key. Events accumulate within the database transaction boundary.

**Dispatch (`TransactionDispatcher.commit`):** Called after the transaction commits, this method:
1. Reads all buffered events from `EventBuffer`.
2. Deduplicates and merges changes per object (`get_uniq_changes`).
3. Executes **sync backends** sequentially in-process (retrieved via `Setting.where(area: 'Transaction::Backend::Sync')`).
4. Schedules **async backends** via `TransactionJob.perform_later(item, params)` — an ActiveJob that resolves registered async backends from `Setting.where(area: 'Transaction::Backend::Async')`.

**Registered transaction backends (handlers):**
- `Transaction::Trigger` — evaluates automation triggers against changed objects
- `Transaction::Notification` — generates in-app and email notifications
- `Transaction::SignatureDetection` — detects email signatures in articles
- `Transaction::CtiCallerIdDetection` — caller ID resolution for phone integrations
- `Transaction::ClearbitEnrichment` — contact data enrichment from Clearbit

This architecture is notably similar to Chatwoot's dispatcher-listener pattern: typed events, a named buffer, sync/async routing, and independently registered handler classes. The backend registration via `Setting` (database-stored) adds runtime configurability beyond static registration.

### Sequencer pattern (`lib/sequencer/`)

`lib/sequencer/` implements a workflow execution pipeline: complex multi-step operations (such as data imports) are decomposed into `Unit` objects with named input/output attributes, shared `State`, and a `Sequence` orchestrator. This is used internally for import workflows and is a design pattern layered atop the main architecture, not a separate architectural style.

### Channel ingestion (`lib/`, `app/models/channel/`)

External communication channels (email via IMAP/POP3/SMTP, SMS, Telegram, WhatsApp, Facebook, Microsoft Graph) are modelled as drivers under `app/models/channel/driver.rb`. Inbound messages are pulled by the `ProcessDelayedCommunicationInboundJobs` background service and converted into `Ticket` and `Ticket::Article` records. This funnels asynchronous external input through the domain model, which then triggers the `TransactionDispatcher` fan-out — an event-driven ingestion path.

### Frontend architecture

`app/frontend/apps/` contains two separately compiled SPA applications:
- `desktop/` — full agent/admin interface (Vue.js 3, TypeScript, Vite)
- `mobile/` — mobile-optimized interface

Both are served from the same Rails app as static assets. The frontend communicates exclusively through the Rails REST API and GraphQL endpoint — it is not a separate service.

## Architecture Styles Identified

### Primary: Layered
**Strength: Strong**

The canonical Rails horizontal stack is consistently applied and disciplined: HTTP entry via controllers, business logic isolated in 120+ service objects, authorization in Pundit policies, data access exclusively through ActiveRecord models, and asynchronous work through ActiveJob. The existence of a GraphQL layer (`app/graphql/gql/`) as a second complete API surface reinforces the layered model — both REST and GraphQL controllers delegate downward to the same service and model layers. Single deployable unit, single database.

### Secondary: Event-Driven
**Strength: Strong**

The `TransactionDispatcher` + `EventBuffer` system is a first-party, purpose-built internal event bus. It is not incidental Rails callbacks — it is a typed event vocabulary, a buffer with a commit lifecycle, sync/async routing, and pluggable backend handlers registered via a configuration system. All domain side-effects (triggers, notifications, signature detection, CTI caller ID, enrichment) are handled through this dispatcher rather than through direct model coupling. The pattern closely parallels Chatwoot's dispatcher-listener architecture, which was also classified as Event-Driven.

### Considered and rejected: Microkernel (weak signal)
The `Package` model suggests a plugin system exists for extending Zammad via installable packages. However, there is no evidence that this constitutes a runtime kernel + plugin topology in the Richards & Ford sense — it appears to be an administrative packaging mechanism for distributing configuration and code, not a primary architectural integration mechanism.

## Quality Attributes

- **Extensibility:** High. The transaction backend system allows new cross-cutting behaviors (new automation triggers, new notification strategies, new integrations) to be added as registered handler classes without modifying domain models. The channel driver abstraction (`app/models/channel/driver.rb`) provides clean extension points for new communication channels.
- **Scalability:** Medium. Single PostgreSQL database is the primary bottleneck. Background workers (`delayed_job`-based) can be scaled horizontally. Redis-backed ActionCable scales WebSocket delivery. Elasticsearch offloads search to a dedicated index. The absence of Sidekiq (using custom `delayed_job` fork instead) limits queue management sophistication versus Chatwoot.
- **Maintainability:** High. Service objects prevent business logic leakage into controllers and models. Pundit policies centralize authorization. The layered separation of concerns makes the codebase navigable at scale (5.4k stars, 18k+ commits, active commercial support).
- **Real-time capability:** High. ActionCable (Redis-backed) provides WebSocket delivery. GraphQL subscriptions via the `graphql_channel.rb` ActionCable channel enable push-based updates for the Vue.js SPA. A dedicated WebSocket server process (port 6042) handles real-time client connections.
- **Testability:** High. The service object pattern and thin controllers are amenable to unit testing. Large `spec/` suite with RSpec, Capybara, factory_bot, and VCR. Pundit matchers for policy testing.
- **Operability:** Medium. Single database and single codebase simplify the operational model. The custom `delayed_job` fork (not Sidekiq) means no built-in web UI for queue monitoring. Elasticsearch, Redis, and optional Memcached add infrastructure dependencies. Supports Docker, Kubernetes (Helm), and DEB/RPM package deployments.
- **Resilience:** Medium. `ActiveJobLock` model prevents duplicate job execution. Background services include retry logic. No circuit breaker or saga patterns observed. Delayed job retry is the primary fault-tolerance mechanism for async processing.
- **Multi-channel:** High by design. Email (IMAP/POP3/SMTP), WhatsApp, Telegram, SMS, Facebook, CTI (telephony), and chat are all modelled as first-class channel drivers. Adding a new channel follows a consistent driver interface pattern.

## Classification Reasoning

Zammad is classified as **Layered (primary) + Event-Driven (secondary)**.

**Layered** is primary because the Rails architectural conventions are consistently applied at every layer: versioned REST API controllers, GraphQL as a second access surface, 120+ service objects for business logic isolation, Pundit policies for authorization, and ActiveRecord models for data access. The single deployable unit and single shared database confirm a monolithic topology organized as a horizontal stack. The prior classification of Layered-only by the previous model was correct at the primary level.

**Event-Driven** is secondary because the `TransactionDispatcher` + `EventBuffer` system is a first-party internal event bus, not a side-effect of Rails conventions. It captures typed domain change events, buffers them within transaction boundaries, and fans out to registered handlers through both sync and async paths. The backend handlers are pluggable and registered via configuration — this is architecturally intentional. The previous classification (by `openrouter/z-ai/glm-5`) rejected Event-Driven on the grounds that "components communicate synchronously via database, not through event brokers." This is partially true (no external broker like Kafka), but the `TransactionDispatcher` is functionally equivalent to an internal event broker — it implements the same pattern as Chatwoot's `Dispatcher` singleton, which was correctly classified as Event-Driven.

**Rejected alternatives:**
- **Modular Monolith:** No enforced module boundaries or namespace isolation. The 95 route files and service subdirectories are organizational groupings, not architectural boundaries. Models freely cross-reference each other; no separate schemas per domain.
- **Microservices / Service-Based:** Single deployable, single database, no inter-service network calls. The Procfile's three processes are colocated threads within one application, not independent services.
- **Pipeline:** The `lib/sequencer/` pattern is an internal utility for import workflows, not the primary integration mechanism of the system.
- **Microkernel:** The `Package` model and any plugin capability are not the dominant integration topology.
- **Serverless / Space-Based / Multi-Agent:** No evidence for any of these.
