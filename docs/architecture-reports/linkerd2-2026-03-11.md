# linkerd2 — Architecture Classification Report
**Date:** 2026-03-11
**Repository:** https://github.com/linkerd/linkerd2
**Primary Style(s):** Microservices, Microkernel
**Confidence:** 0.93
**Summary:** A Kubernetes-native service mesh control plane composed of independently deployed, gRPC-communicating control-plane microservices, with a microkernel extension model for optional capability bundles (viz, multicluster, CNI) and a sidecar data-plane proxy managed by admission webhook injection.

## Evidence

- **Control plane components are separate Deployments:** `charts/linkerd-control-plane/templates/` contains distinct Deployment manifests for `destination`, `identity`, `proxy-injector`, `heartbeat`, and `sp-validator` — each with its own Service, RBAC, and container image. They are individually scalable and independently restartable.
- **Distinct binaries via subcommand dispatch:** `controller/cmd/main.go` routes `os.Args[1]` to `destination.Main`, `identity.Main`, `proxy-injector.Main`, `sp-validator.Main`, `heartbeat.Main`, and `service-mirror.Main` — a single Go binary acts as a multi-tenant dispatcher for separately deployed process types.
- **gRPC as the internal communication protocol:** The destination controller exposes a gRPC server on port 8086 (`linkerd-dst` Service). The proxy fetches routing, endpoint, and policy updates from it via a streaming gRPC API defined in `proto/` and the external `linkerd2-proxy-api` Go module. The identity controller issues mTLS certificates over gRPC. Viz metrics-api and tap expose their own gRPC servers (`viz/metrics-api/grpc_server.go`, `viz/tap/api/grpc_server.go`).
- **Policy controller is a separate Rust binary:** `policy-controller/` is a fully independent Rust workspace (`Cargo.toml`) compiled to a separate binary `linkerd-policy-controller`, which runs as its own Kubernetes Deployment and exposes a gRPC API consumed by the destination controller. It has its own crate hierarchy: `core`, `grpc`, `k8s/api`, `k8s/index`, `k8s/status`, `runtime`.
- **Sidecar proxy is fetched from a separate repository:** `Dockerfile.proxy` downloads the proxy binary from `linkerd/linkerd2-proxy` at build time — the data plane is an entirely separate codebase with its own release cycle.
- **Kubernetes-native resource model:** Custom resource definitions are defined in `controller/gen/apis/` for `serviceprofile/v1alpha2`, `link/v1alpha3`, `policy/v1alpha1`, `policy/v1beta3`, `externalworkload`, `server`, and `serverauthorization`. Controllers watch these CRDs via SharedInformers and reconcile Kubernetes state.
- **Microkernel extension model:** `EXTENSIONS.md` documents a formal binary extension protocol: third-party extensions are CLI executables named `linkerd-<name>` found on PATH. First-party extensions (`viz`, `multicluster`, `CNI`) ship as optional Helm charts (`charts/linkerd-crds`, `charts/linkerd-control-plane`, `charts/linkerd2-cni`) installed independently. The `viz` subsystem (`viz/`) adds Prometheus metrics API and tap (live traffic inspection) without modifying the core control plane.
- **Admission webhook for transparent injection:** `controller/proxy-injector/` implements a Kubernetes MutatingAdmissionWebhook that intercepts pod creation and injects the sidecar proxy container and init container without any application code changes.
- **Event-queue-based reconciliation in multicluster:** `multicluster/service-mirror/cluster_watcher.go` uses a `workqueue.TypedRateLimitingInterface` and an event broadcaster (`record.EventBroadcaster`) to process remote cluster events asynchronously — a narrow event-driven pattern within the service mirror component.
- **Dual language codebase:** Go (`go.mod` module `github.com/linkerd/linkerd2`) for the control plane, Rust (`Cargo.toml` workspace) for the policy controller. Primary deployment target is Kubernetes (extensive `k8s.io/api`, `k8s.io/client-go`, `sigs.k8s.io/gateway-api` dependencies).
- **Prometheus-native observability:** Every control plane component exposes Prometheus metrics on a dedicated admin port. `pkg/prometheus/` provides shared instrumentation. A `PodMonitor` manifest is included in the Helm chart (`charts/linkerd-control-plane/templates/podmonitor.yaml`).

## Architecture Styles

### Microservices (Primary, 0.93)
The control plane is decomposed into individually deployed, independently operable services:
- **destination** — endpoint resolution, service profile lookup, and traffic policy distribution to proxies via gRPC streaming
- **identity** — mTLS certificate issuance and rotation using Kubernetes token review
- **proxy-injector** — admission webhook that mutates pods to add the sidecar proxy
- **policy-controller** (Rust) — network authorization policy indexing and gRPC API
- **sp-validator** — admission webhook for ServiceProfile CRD validation
- **heartbeat** — periodic telemetry beacon
- **service-mirror** (multicluster extension) — watches remote clusters and mirrors their services locally

Each runs in a separate Kubernetes Deployment, has its own service account and RBAC, communicates exclusively over the network (gRPC or HTTP), and can be scaled independently. There is no shared in-process memory or shared database between them.

### Microkernel (Secondary, 0.88)
The core control plane (destination + identity + proxy-injector) is minimal. Capabilities are added through a documented extension protocol:
- CLI extensions are discoverable binaries on PATH (`linkerd-<name>`)
- Optional Helm chart bundles (`viz`, `multicluster`, `CNI`) install into the cluster independently
- The `viz` extension adds the metrics API and tap service without modifying core components
- The `policy-controller` Rust service is plugged in to the destination controller via gRPC, not compiled into it

## Quality Attributes

- **Security:** mTLS for all mesh traffic via the identity controller; policy controller enforces network authorization policies; admission webhooks run with minimum privilege; separate service accounts per control plane component; non-root proxy runtime.
- **Observability:** Prometheus metrics on every component; Grafana dashboards (`grafana/`); tap (live request inspection) via the viz extension; structured logging via `logrus`.
- **Scalability:** Control plane components scale horizontally as independent Deployments; the destination controller uses streaming gRPC so proxies maintain long-lived connections rather than polling; workqueue rate-limiting in service mirror.
- **Reliability:** Streaming endpoint updates from destination controller keep proxies current without polling; proxy-side load balancing and retries; gRPC keepalive for connection durability.
- **Deployability:** Full Helm chart deployment (`charts/linkerd-control-plane`, `charts/linkerd-crds`); CLI-driven install workflow; extension-based optional components avoid mandatory complexity.
- **Extensibility:** Documented binary extension protocol; optional Helm chart bundles; CRD-based configuration surface (ServiceProfile, Server, ServerAuthorization, ExternalWorkload, Link).
- **Interoperability:** Gateway API (`sigs.k8s.io/gateway-api`) support; gRPC-based proxy API decouples the data plane from the control plane; multicluster federation via the service mirror pattern.
- **Evolvability:** Separate Go and Rust codebases enable independent evolution of control plane and policy controller; versioned CRD APIs (`v1alpha1`, `v1alpha2`, `v1beta3`); proxy fetched from an external release rather than compiled in-tree.

## Classification Reasoning

Linkerd2 is a **Microservices** architecture at the deployment level: each control plane capability (endpoint resolution, mTLS certificate issuance, admission mutation, policy enforcement) is an independently deployed, network-communicating service. This is not a distributed monolith — there is no shared database, no shared in-process state, and no shared library that couples the components at runtime. The boundary between the Go control plane and the Rust policy controller is a concrete network API (gRPC), not a package import.

The **Microkernel** classification is warranted as a secondary style because the codebase explicitly models the control plane as a minimal core with a formal plug-in extension protocol. The `EXTENSIONS.md` document specifies required flags, required commands (`install`, `uninstall`, `check`), and the naming convention (`linkerd-<name>`) that all extensions must implement. First-party extensions (`viz`, `multicluster`, `CNI`) follow this same contract. The core control plane installs without any of these extensions and remains fully functional for basic mTLS and traffic management.

**Not Layered:** While there is a separation between data plane (proxy) and control plane, and within the control plane between API serving and Kubernetes watching, the architecture is not organized as horizontal presentation/business/data layers. The dominant structure is vertical decomposition by bounded capability.

**Not Event-Driven (primary):** Kubernetes SharedInformers and workqueues are used internally for reconciliation, and the service mirror uses an event queue, but these are implementation mechanisms. There is no message broker or event bus as a first-class architectural element. The dominant inter-component communication pattern is synchronous gRPC.

**Not Modular Monolith:** The components are deployed as separate containers and communicate over the network. They are not co-deployed modules within a single process that could theoretically be split. The Go binary's subcommand dispatch pattern is a build-time artifact; at runtime each subcommand runs as a separate Kubernetes Deployment.

**Not Pipeline:** While the sidecar proxy intercepts and transforms traffic in a chain (inbound proxy → application → outbound proxy → destination), the control plane itself is not structured as a sequential data transformation pipeline.

The combination of Microservices (deployment topology and inter-component communication) and Microkernel (extension model for optional capabilities) accurately captures how Linkerd2 is structured and how it is intended to evolve.
