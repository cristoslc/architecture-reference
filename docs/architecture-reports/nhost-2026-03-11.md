## Style Rationales

**Microservices (primary, 0.87):** The platform is composed of independently deployed, separately containerised services that communicate exclusively over HTTP: `auth` (nhost/auth, port 4000), `storage` (nhost/storage, port 5000), `graphql` (nhost/graphql-engine / Hasura, port 8080), `functions` (Node.js Express runner, port 3000), `ai` / Graphite (nhost/graphite), `dashboard` (Next.js), and `postgres` (PostgreSQL). Each core service lives in its own directory (`services/auth/`, `services/storage/`) with its own Go module entrypoint (`main.go`), its own migrations, its own Docker image, its own versioned CHANGELOG, and its own OpenAPI/GraphQL schema. Traefik acts as the edge reverse proxy routing subdomain-based requests to the appropriate service. The docker-compose canonical deployment (`examples/docker-compose/docker-compose.yaml`) and the CLI's `dockercompose/` code generator (`cli/dockercompose/compose.go`) both treat each service as an independently started container with explicit `depends_on` health conditions. The hosted cloud platform deploys these on Nhost-managed infrastructure; self-hosting uses the same topology via Docker Compose or Kubernetes.

**Layered (secondary, 0.82):** Each individual service is internally organised as a strict three-layer stack: HTTP transport/API (`api/` + OpenAPI-generated handlers), domain controller (`controller/`), and data access (`sql/` with sqlc-generated queries, or `metadata/` using GraphQL to Hasura). The auth service demonstrates this most clearly: `api/server.gen.go` handles routing and request parsing; `controller/controller.go` defines typed interfaces (`DBClient`, `Emailer`, `SMSer`, `Encrypter`, `OAuthProvider`) and contains all business logic; and `sql/query.sql.go` provides the persistence layer against PostgreSQL. The storage service mirrors the pattern: `api/` (OpenAPI), `controller/` (business logic with `MetadataStorage` interface), `metadata/` (Hasura GraphQL adapter), and `storage/` (S3 adapter). Dependency injection is done manually at startup in `cmd/serve.go`, wiring the concrete implementations into the controller through interfaces, which gives clear inversion-of-control without a DI framework.

**Serverless (secondary, 0.75):** The Functions product (`services/functions` node runtime, `cli/dockercompose/functions`) deploys user-defined TypeScript/JavaScript files from a `./functions/` directory as individual HTTP endpoints, auto-routing based on file path. Each function file exports a single default Express handler; the runtime composes these into an HTTP server. The product is explicitly described as "serverless functions" in documentation and positions alongside AWS Lambda / Vercel functions. While the underlying mechanism is a long-lived Express server rather than FaaS cold-start invocations, the developer contract (file-per-endpoint, no server management, scales independently from other services) and the platform abstraction align with the Serverless style.

## Domain

Backend-as-a-Service (BaaS) platform providing managed PostgreSQL, instant GraphQL API (Hasura), authentication, file storage, serverless functions, and AI capabilities. Positioned as an open-source Firebase alternative with a GraphQL-first approach.

## Production Context

- Hosted cloud at `app.nhost.io`; self-hostable via Docker Compose (`examples/docker-compose/`) or Kubernetes; dedicated cluster enterprise option
- Each project gets isolated instances of postgres, graphql (Hasura), auth, storage, functions, and optionally ai (Graphite) and custom Run containers
- Hasura GraphQL Engine is a third-party service (Nhost fork `nhost/graphql-engine`) acting as the unified data API layer; auth and storage services both delegate metadata through Hasura
- Client SDK (`@nhost/nhost-js`) wraps all four service APIs (auth, storage, graphql, functions) with a unified session/middleware chain
- Nhost Run enables arbitrary containerised services (Redis, MongoDB, custom backends) deployed alongside the core stack with full ingress and networking integration
