## Architecture Styles

### Primary: Pipeline (Pipe-and-Filter)

The core execution model is a block-based dataflow graph. `backend/blocks/` contains 275+ discrete `Block` subclasses, each declaring a typed `Input` (BlockSchemaInput) and `Output` (BlockSchemaOutput) schema and an async `run()` method that yields output tuples. Agents are defined as `AgentGraph` objects whose `Node` instances reference block IDs and whose `Link` instances wire a source block's output pin to a sink block's input pin. The `ExecutionManager` in `executor/manager.py` traverses this graph, routing data from one block to the next in dependency order — a textbook pipe-and-filter pattern.

### Secondary: Microservices

The platform is deployed as eight independently containerised processes: `rest_server` (FastAPI REST, port 8006), `websocket_server` (real-time updates, port 8001), `executor` (graph execution engine, port 8002), `copilot_executor` (CoPilot AI processing, port 8008), `database_manager` (database operations proxy, port 8005), `scheduler_server` (cron/scheduled triggers, port 8003), `notification_server` (email/push notifications, port 8007), and `frontend` (Next.js, port 3000). Services discover each other by hostname and communicate via RabbitMQ exchanges (fanout and direct) and HTTP RPC through `BaseAppService`. Each service runs from the same Docker image with a different startup command — a clear service-based decomposition of functionality.

### Tertiary: Multi-Agent

Agents can be nested recursively. `BlockType.AGENT` and the `AgentExecutorBlock` allow any graph node to invoke a separate, independently versioned `AgentGraph` as a sub-agent. The `agents/` directory contains pre-built agent JSON definitions. The platform also exposes a marketplace (Store) for sharing and composing agent templates, and an MCP (Model Context Protocol) tool block type allows agents to call external MCP tool servers — establishing an ecosystem-level multi-agent topology.

## Quality Attributes

| Attribute | Evidence |
|---|---|
| **Deployability** | Each of the 8 services is independently containerised with its own Dockerfile target (`rest`, `executor`, `ws`, `db`, `scheduler`, `notification`). Docker Compose orchestrates health-check dependencies between services. |
| **Modularity** | Blocks are self-contained classes loaded dynamically via `importlib`; a public `backend.sdk` module provides all base classes for external block development. Blocks can be added without touching the executor. |
| **Scalability** | The executor uses a `ThreadPoolExecutor` with a configurable pool size (gauged via Prometheus) and RabbitMQ queues for distributing graph execution tasks. Copilot has a dedicated separate executor service. |
| **Fault Tolerance** | RabbitMQ connection code explicitly handles `BLOCKED_CONNECTION_TIMEOUT`, `SOCKET_TIMEOUT`, and `CONNECTION_ATTEMPTS` (5 retries). `continuous_retry` and `func_retry` decorators wrap execution entry points. Cluster locking via Redis prevents duplicate graph execution. |
| **Observability** | Prometheus metrics (counters, histograms, gauges) are defined for graph executions, block executions, block duration, and WebSocket connections via `monitoring/instrumentation.py`. Sentry captures exceptions. Structured logging with truncation via `TruncatedLogger`. Redis streams execution status events to WebSocket clients in real time. |
| **Evolvability** | LaunchDarkly feature flags gate new frontend features. Agent graphs are versioned (`graph_version` field in schema). Block SDK allows new integrations without modifying core. The `classic/` subtree is preserved as a legacy path alongside the new `autogpt_platform/` tree. |

## Domain

AI Agent Platform — enables non-technical and technical users to build, publish, and run multi-step AI workflows by visually composing typed "blocks" (LLM calls, web search, social media posting, code execution, etc.) into directed graphs that execute autonomously.

## Production Context

- Operated as a managed cloud service (`agpt.co`) with a self-host option via Docker Compose; the platform is not a reference implementation — it is actively monetised (credit system with Stripe, usage tracking, store marketplace).
- RabbitMQ and Redis are integral runtime dependencies; PostgreSQL with pgvector is the primary persistence layer via Supabase.
- Block execution costs are tracked per block type (`BLOCK_COSTS` config), and users are throttled or notified when credit balances are low — evidence of a live production billing and abuse-prevention system.
- The codebase includes ClamAV antivirus scanning for file uploads, OAuth application support (users can issue tokens), and a Content Safety / automod subsystem (`executor/automod/`) — indicating a multi-tenant production posture.
