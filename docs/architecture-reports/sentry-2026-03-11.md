# sentry — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/getsentry/sentry
**Classification:** Modular Monolith + Event-Driven + Service-Based
**Confidence:** 0.92

## Summary

Sentry is a developer debugging and error monitoring platform structured as a deliberate Modular Monolith (a single Django application organized into ~60 domain-focused sub-packages, deployable as MONOLITH, CONTROL, or REGION silos) over an Event-Driven backbone (Kafka with 40+ topics, a custom Arroyo consumer framework, and a post-process forwarder pipeline) and supplemented by a constellation of coarse-grained Service-Based external dependencies (Relay, Snuba, Symbolicator, Vroom, Chartcuterie, Taskbroker). The hybrid-cloud silo architecture — where CONTROL and REGION planes communicate through outbox replication and an RPC delegation layer — is a defining structural element that sets Sentry apart from a naive Django monolith.

## Evidence

### Key Architectural Files

- `src/sentry/silo/base.py`: `SiloMode` enum (`MONOLITH`, `CONTROL`, `REGION`) and `DelegatedBySiloMode` class — the central mechanism for switching service implementations based on deployment topology.
- `src/sentry/hybridcloud/rpc/__init__.py`: `DelegatedBySiloMode` delegates service calls to either a local database-backed implementation or an HTTP RPC stub depending on current silo mode. `DelegatedByOpenTransaction` further discriminates by open transaction.
- `src/sentry/hybridcloud/outbox/base.py`: `RegionOutboxProducingModel` mixin hooks Django model `save`/`update`/`delete` to write transactional outbox entries, enabling eventual-consistency cross-silo replication.
- `src/sentry/conf/types/kafka_definition.py`: `Topic` enum with 40+ named Kafka topics (`INGEST_EVENTS`, `INGEST_TRANSACTIONS`, `INGEST_PROFILES`, `MONITORS_CLOCK_TICK`, `UPTIME_RESULTS`, `TASKWORKER_*` family, etc.); `ConsumerDefinition` TypedDict links topics to strategy factories.
- `devservices/config.yml`: Declares 20+ external services Sentry depends on: Relay (event ingestion gateway), Snuba (ClickHouse query layer), Symbolicator (native stacktrace symbolication), Vroom (profiling), Chartcuterie (chart rendering), Taskbroker, Launchpad, Uptime-Checker — each from a separate GitHub repo, run containerized.
- `src/sentry/taskworker/registry.py`: `TaskNamespace` binds tasks to Kafka topics via `ArroyoTopic`; uses `KafkaProducer` to enqueue work, replacing legacy Celery for async task dispatch.
- `src/sentry/workflow_engine/registry.py`: Three typed registries (`data_source_type_registry`, `condition_handler_registry`, `action_handler_registry`) enabling plugin-style extension of the workflow engine without cross-module coupling.
- `src/sentry/runner/commands/run.py`: `sentry run web`, `sentry run worker`, `sentry run consumer <topic>`, `sentry run taskworker`, `sentry run cron` — each a distinct process role, supporting fine-grained horizontal scaling per role.

### Patterns Found

#### Modular Monolith (Primary)

Sentry is a single Django deployment with one `wsgi.py`/`app.py`, one unified PostgreSQL schema, and shared configuration. The `src/sentry/` tree organizes ~60 domain packages (`issues/`, `ingest/`, `incidents/`, `releases/`, `integrations/`, `workflow_engine/`, `sentry_metrics/`, `replays/`, `profiles/`, `uptime/`, `monitors/`, `dashboards/`, `discover/`, `explore/`, `insights/`, `feedback/`, `flagpole/`, `codecov/`, `seer/`, `assistant/`, `autopilot/`, etc.) into bounded domain areas. These coexist within the single Python process in MONOLITH mode; in SAAS deployment the same codebase splits into CONTROL and REGION silos with the RPC layer mediating cross-silo access rather than direct DB queries.

The modular intent is explicit: `SiloMode`-decorated endpoints and services enforce silo boundaries at the code level, preventing REGION code from accessing CONTROL-only tables even in a single-process test environment.

#### Event-Driven (Strong Secondary)

Kafka is structurally central to both the write path and async processing:

- **Ingest pipeline**: `devservices/config.yml` defines 15 named Kafka consumer processes: `ingest-events`, `ingest-attachments`, `ingest-transactions`, `ingest-monitors`, `ingest-profiles`, `ingest-occurrences`, `ingest-feedback-events`, `ingest-replay-recordings`, `process-spans`, `process-segments`, `ingest-metrics`, `ingest-generic-metrics`, `billing-metrics-consumer`, `uptime-results`, and `post-process-forwarder-*`. Each is an independent Arroyo consumer running `sentry run consumer <topic>`.
- **Post-process forwarder**: After event storage in ClickHouse (via Snuba), a `post-process-forwarder-errors/transactions/issue-platform` consumer reads the Snuba commit log and fires Django signals (`event_processed`, `issue_created`, etc.) that trigger rules evaluation, notifications, and integrations.
- **Taskworker**: The new custom task system (`src/sentry/taskworker/`) uses Kafka topics (`taskworker`, `taskworker-long`, `taskworker-products`, `taskworker-symbolication`, `taskworker-workflows-engine`, plus domain-specific variants) to enqueue and deliver async tasks, all via Arroyo producers/consumers.
- **Django signals**: `src/sentry/signals.py` defines `BetterSignal` instances; `src/sentry/receivers/` implements handlers for analytics, auth, email, outbox flushing, and rule evaluation — an internal pub/sub layer within the process.
- **Monitors clock**: `monitors-clock-tick` and `monitors-clock-tasks` consumers implement a distributed cron-check scheduler entirely via Kafka message flow.

#### Service-Based (Secondary)

Sentry coordinates with coarse-grained external services that each own a discrete responsibility:

- **Relay** (Rust): SDK-facing event ingestion gateway; normalizes, filters, and forwards events to Kafka. Sentry does not receive SDK payloads directly in production.
- **Snuba** (Python/ClickHouse): Query and aggregation layer over ClickHouse. All analytics reads (`discover`, `metrics`, `profiles`, `replays`, `spans`) go through Snuba SDK calls or Snuba RPC.
- **Symbolicator** (Rust): Native stacktrace symbolication and minidump processing.
- **Vroom** (Go): Profiling data processing and flame chart generation.
- **Chartcuterie** (Node.js): Server-side chart rendering for email/Slack alerts.
- **Taskbroker** (Rust): External task queue broker that Sentry's taskworker system connects to.
- **Uptime-Checker** (Rust): Periodic HTTP uptime probes; results flow back via `uptime-results` Kafka topic.
- **Launchpad**: iOS/Android app analysis service.

These services share Kafka infrastructure but have independent codebases, repositories, and container images. They communicate exclusively via Kafka topics or HTTP; there is no shared database at the ORM layer.

#### Pipeline (Internal, within Workflow Engine)

`src/sentry/workflow_engine/processors/` implements a multi-stage processing pipeline: `data_packet` ingestion → `data_source` evaluation → `detector` state machine → `data_condition_group` evaluation → `workflow` trigger → `action` dispatch. Each stage is registered via typed plugin registries, enabling new detector and action types without modifying core orchestration code.

#### Hybrid Cloud / Silo Architecture

The most distinctive structural element is the CONTROL/REGION silo split. In SaaS:

- **CONTROL silo**: Manages global resources (users, authentication, billing, org creation, integrations, API tokens). Runs `SiloMode.CONTROL` services.
- **REGION silo(s)**: Each geographic region runs `SiloMode.REGION` services managing org-scoped data (events, issues, projects, releases, alerts).
- Cross-silo operations use **outbox replication** (transactional outbox pattern via `RegionOutboxProducingModel`) for eventual consistency and **RPC delegation** (`DelegatedBySiloMode`) for synchronous cross-silo service calls proxied via HTTP.
- The API gateway middleware (`hybridcloud/apigateway/`) proxies requests to the correct region when a CONTROL silo receives a request destined for a REGION.

This is not a standard Microservices architecture — there are only two logical silo types (not dozens of fine-grained services), and in self-hosted and single-tenant deployments all silos collapse into MONOLITH mode with full shared-database access.

### What This Is NOT

- **NOT Microservices**: There is no database-per-service. CONTROL and REGION silos each have a single PostgreSQL instance. External services (Relay, Snuba, Vroom, etc.) are infrastructure components, not microservices in the DDD sense.
- **NOT Hexagonal**: No ports-and-adapters abstraction layer. Django ORM is used directly within domain packages; the silo RPC layer is not a hexagonal port.
- **NOT Layered**: The primary organization is vertical domain slices (`issues/`, `incidents/`, `monitors/`, `replays/`, etc.), not horizontal tiers. The `api/` layer is thin presentation on top of domain packages, not a strict tier boundary enforced architecturally.
- **NOT CQRS**: There is no explicit command/query separation at the application level. Snuba as a read store is an infrastructure concern, not a CQRS architectural decision applied to the application model.
- **NOT Serverless**: All components are long-running containers (Django, Arroyo consumers, Relay, Snuba).

## Languages

- **Python** (primary): Django application, all backend domain logic, Arroyo consumers, taskworker (~4,650 `.py` files in `src/`)
- **TypeScript / TSX** (significant): React SPA (`static/app/`) with ~8,500 files, ~300 components, ~160 view scenes

## Quality Attributes

- **Scalability**: Independent horizontal scaling of web, worker, consumer, and taskworker processes; Kafka consumer groups for parallel event processing; per-region REGION silos in SaaS.
- **Fault Tolerance**: Kafka DLQ topics for every consumer (`ingest-events-dlq`, `taskworker-long-dlq`, etc.); outbox pattern for cross-silo eventual consistency; Arroyo backpressure and dead-letter handling.
- **Observability**: Sentry is self-observing (uses its own SDK); DataDog/statsd metrics wired into every instrumented task; structured logging throughout.
- **Deployability**: Single codebase deployable as MONOLITH (self-hosted), or split into CONTROL + REGION silos (SaaS); devservices config supports 10+ named deployment modes.
- **Modularity**: ~60 domain packages with silo-mode decorators enforcing runtime module boundaries; plugin-style registries for workflow engine extensions; integrations as isolated sub-packages.
- **Security**: Data isolation between CONTROL and REGION via RPC; silo-mode decorators prevent cross-silo data leakage; relay as a security boundary for SDK events.
- **Performance**: Kafka-based async ingest decouples SDK write latency from processing; Snuba/ClickHouse for analytics queries; Redis caching; Relay pre-filters invalid events before they reach Sentry.
