## Architecture Style Rationale

### Primary: Modular Monolith (confidence 0.97)

The entire system runs as a single ASP.NET Core process launched from `src/Bootstrapper/Inflow.Bootstrapper`. At startup, `ModuleLoader.LoadAssemblies()` discovers and loads all `Inflow.Modules.*` assemblies; each module registers itself via `IModule.Register()` / `IModule.Use()` against the shared DI container. Critically, no module `.csproj` carries a `ProjectReference` to any other module — cross-module coupling is zero at the compiler level. Each module owns a private PostgreSQL schema (`wallets`, `payments`, `customers`) enforced by separate `DbContext` types, delivering module-level data isolation within a shared server. The README explicitly positions this as a "Modular Monolith" reference implementation with a planned `microservices` branch to illustrate the migration path.

### Secondary: Event-Driven (confidence 0.93)

Inter-module integration is accomplished exclusively through asynchronous events published via `IMessageBroker` and delivered in-process through `InMemoryMessageBroker`. Each module defines its own local contract type for external events it consumes (e.g., Wallets defines its own `CustomerCompleted` class annotated with `[Message("customers")]` rather than importing the Customers module's type), ensuring zero shared contracts. The shared infrastructure provides a transactional Outbox/Inbox pattern (`EfOutbox`, `EfInbox`, `OutboxProcessor`) for at-least-once delivery guarantees. A dedicated Saga module (`NewCustomerBonusFundsSaga` via the Chronicle library) orchestrates cross-module business processes spanning `CustomerVerified` → `WalletAdded` → `DepositCompleted` → `FundsAdded`.

### Secondary: Domain-Driven Design (confidence 0.82)

DDD tactical patterns are applied consistently inside each module. `Wallet` and `Owner` types extend `AggregateRoot<T>` with domain-event collection and optimistic concurrency via a `Version` counter. Value objects (`TransferName`, `TransferMetadata`, `OwnerName`, `TaxId`, `Currency`, `Amount`) enforce invariants at construction. Domain repository interfaces live in the domain layer; EF Core implementations live in the infrastructure layer. The Wallets module demonstrates full Clean Architecture layering: Core (domain) → Application (command/query handlers) → Infrastructure (EF repositories, storage) → Api (controllers). These are DDD bounded contexts that happen to share a process rather than separate deployable services.

## Quality Attributes

- **Modularity**: Zero compile-time coupling between modules enforced by project reference isolation; each module registers independently and can be disabled via configuration; module-private schemas prevent schema bleed.
- **Evolvability**: Local contracts pattern allows each module's event schema to evolve independently; the README explicitly documents a migration path to microservices on the `microservices` branch, showing the architecture is designed to unbundle.
- **Reliability**: Transactional Outbox/Inbox pattern with `OutboxProcessor` background job ensures cross-module events survive process failures and are delivered exactly-once semantics within a transaction boundary.
- **Testability**: Three test project tiers per module (Unit, Integration, EndToEnd); Wallets module ships 20+ tests covering commands, entities, and event handlers; shared `Inflow.Shared.Tests` base classes reduce test setup duplication.
- **Simplicity**: Single deployable unit with a single PostgreSQL instance keeps operational complexity minimal for the reference use case; no message broker, no service mesh, no sidecar proxies.
- **Observability**: Structured logging with request ID, trace ID, user ID, and correlation ID propagated through `MessageContext`; `ILogger` injection at every command handler and event handler boundary.

## Domain

Virtual payments platform: Customers, Payments (deposits/withdrawals to real bank accounts), Wallets (virtual wallet transfers), Users (identity/auth), and a cross-cutting Saga for new-customer bonus funds. Primary purpose is a **reference implementation** for building Modular Monolith systems in .NET, published by the devmentors educational platform.

## Production Context

- Not a production application; distributed as a course companion repository for devmentors' "Building Modular Monolith" course series.
- Single-host deployment; no clustering, no message broker; PostgreSQL is the sole infrastructure dependency.
- `microservices` branch in the same repository demonstrates Modular Monolith → Microservices extraction; full microservices version lives in `Inflow-micro`.
- .NET 6.0 (LTS end-of-life May 2024); code patterns remain current reference material for .NET 8/9.
