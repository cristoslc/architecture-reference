## Style Rationales

**Layered (primary, 0.93):** Flink is organized into a strict, explicitly-documented set of horizontal layers, each depending only downward. The layer stack is: (1) **User API Layer** — `flink-datastream-api/`, `flink-table/flink-table-api-java/`, `flink-table/flink-table-api-scala/`, and `flink-clients/` expose the DataStream API, Table API, and SQL as public surfaces annotated with `@Public`/`@PublicEvolving`/`@Internal`. The `flink-architecture-tests` module enforces via ArchUnit rules (`ConnectorRules.java`) that connector code may only depend on `@Public`/`@PublicEvolving` classes from the core, not internal implementation. (2) **Query Planning Layer** — `flink-table/flink-table-planner/` implements SQL parsing (`flink-sql-parser/`), AST/semantic tree, Calcite-based optimization (`planner/plan/optimize/`), code generation (`planner/codegen/`), and translates relational operations into `Transformation<T>` objects via the `Planner` delegation interface (`flink-table-api-java/.../delegation/Planner.java`). The planner is loaded in an isolated classloader via `flink-table-planner-loader/` to prevent Scala and Calcite dependency conflicts with user code. (3) **Streaming Runtime Layer** — `flink-streaming-java/` bridges the DataStream API to `flink-runtime`: `StreamGraphTranslator` converts user programs to a `StreamGraph`, then to a `JobGraph` (`flink-runtime/.../jobgraph/`). Operators (`StreamOperator`, windowing, functions) live here, strictly above the runtime. (4) **Runtime Layer** — `flink-runtime/` is the core engine: the `Dispatcher` receives `JobGraph` submissions, the `JobMaster`/`ExecutionGraph` tracks execution state, the `AdaptiveScheduler`/`AdaptiveBatchScheduler` allocates tasks to slots, the `ResourceManager` negotiates with YARN/Kubernetes, and `CheckpointCoordinator` manages exactly-once state snapshots. The `TaskExecutor` receives tasks from the `JobMaster` and executes them. (5) **Infrastructure Layer** — `flink-rpc/` (Akka-based RPC), `flink-metrics/`, `flink-core/` (FileSystem abstraction, serialization, plugin loading), and `flink-runtime/.../io/network/` (Netty-based credit-based flow control and shuffle).

**Pipeline (primary, 0.90):** Flink's fundamental execution model is a directed dataflow graph (Pipe-and-Filter) where each node is an operator and edges are bounded or unbounded data streams. The user composes a pipeline using `DataStream.map()`, `.keyBy()`, `.window()`, `.process()`, etc., which builds a logical `StreamGraph`. The `StreamGraph` is compiled to a `JobGraph` (physical plan with chained task vertices and intermediate result sets — `IntermediateDataSet.java`). At runtime the `TaskManager`s execute operator chains as a streaming pipeline with Netty-based network buffers flowing between tasks (`flink-runtime/.../io/network/`). Credit-based flow control (`CreditBasedPartitionRequestClientHandler`) provides natural backpressure: a slow downstream operator withholds credits, causing upstream operators to slow down. The exactly-once checkpoint protocol (`CheckpointCoordinator`, barrier injection) is an intrinsic property of the pipeline: barriers flow through the same network channels as data. The Table API (`flink-table/flink-table-planner/`) compiles SQL into the same `Transformation` pipeline representation, making the pipeline model the universal execution substrate.

**Microkernel (secondary, 0.80):** A formal plugin system (`flink-core/.../core/plugin/`) loads filesystem, metrics-reporter, and resource-plugin JARs from a `plugins/` directory at startup using isolated classloaders (`PluginLoader.java`, `DirectoryBasedPluginFinder.java`). The `PluginManager` interface (`PluginManager.java`) is the SPI gateway: it discovers plugin implementations via Java `ServiceLoader`. Filesystem implementations (`flink-filesystems/`: S3-Hadoop, S3-Presto, GCS, Azure, OSS, HDFS) are drop-in plugins with no core changes required; metrics reporters (`flink-metrics/`: Prometheus, Datadog, InfluxDB, Graphite, Datadog, JMX, OpenTelemetry, StatsD, SLF4J, Dropwizard) similarly. The Table planner itself is loaded via an explicit classloader isolation mechanism (`flink-table-planner-loader/`) to allow independent Scala/Calcite versions. RPC is also loaded as a pluggable module (`flink-rpc/flink-rpc-akka-loader/`). Connector packages (`org.apache.flink.connector.*`, `org.apache.flink.streaming.connectors.*`) are governed by ArchUnit rules that enforce they depend only on public API — the canonical microkernel constraint.

## Domain

Unified stream and batch processing platform. Core domains: dataflow graph compilation and optimization (DataStream API, Table API, SQL), stateful streaming execution with exactly-once guarantees, distributed checkpoint coordination, resource management across YARN/Kubernetes, connector ecosystem for sources and sinks, and Complex Event Processing (`flink-libraries/flink-cep`).

## Production Context

- Three deployment modes: Session Cluster (shared), Per-Job (isolated), Application Mode (main() runs in cluster) across YARN, Kubernetes, and standalone
- Used as the stream-processing layer in the Apache data ecosystem, consuming from Kafka topics and writing to sinks via connector plugins
- `AdaptiveBatchScheduler` eliminates the need to pre-configure parallelism for batch workloads by inferring it from input statistics at runtime
- `flink-python/` (PyFlink) exposes all APIs to Python, compiled to the same Java `Transformation` representation
- ForSt state backend (fork of RocksDB) is the emerging default for large stateful jobs
