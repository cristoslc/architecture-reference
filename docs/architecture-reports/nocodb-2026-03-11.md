## Summary

NocoDB is an open-source Airtable alternative that turns any SQL database into a collaborative spreadsheet interface. Its architecture is a **Layered** monolith with a significant **Microkernel** plugin subsystem and meaningful **Event-Driven** internal coordination. The codebase is a pnpm/lerna monorepo containing two deployable packages (backend and frontend), shipped as a single Docker container.

**Primary styles:** Layered, Microkernel
**Secondary style:** Event-Driven

## Architecture Style Analysis

### Primary Style 1: Layered Architecture

The NestJS backend rigidly separates concerns into horizontal layers. Each domain entity (Base, Table, Column, View, Hook, etc.) has a matching controller, service, and model. Requests flow strictly top-to-bottom.

**Layer breakdown:**

| Layer | Location | Examples |
|---|---|---|
| Presentation | `src/controllers/` | `bases.controller.ts`, `tables.controller.ts`, `hooks.controller.ts` |
| API Versioning | `src/controllers/v3/` | `bases-v3.controller.ts`, `data-v3.controller.ts` |
| Business Logic | `src/services/` | `BasesService`, `TablesService`, `HookHandlerService` |
| Data Access | `src/models/` | `Base.ts`, `Table.ts`, `Column.ts`, `Hook.ts`, `Model.ts` |
| Query/DB Abstraction | `src/db/` | `BaseModelSqlv2.ts` (7,726 lines), `CustomKnex.ts`, `SqlClientFactory.ts` |
| Metadata Store | `src/meta/meta.service.ts` | Knex-based internal schema management |

The `AppModule` root imports `NocoModule`, `JobsModule`, `AuthModule`, `EventEmitterModule`, and `OAuthModule`. `NocoModule` registers all controllers and providers. This classic NestJS hierarchy enforces the layered dependency flow.

**Key evidence:**
- `noco.module.ts` lists ~60 controllers and ~70 services, all organized as flat services per resource.
- No cross-service direct calls; dependency injection wires layers together.
- NestJS guards (`GlobalGuard`) and middleware (`GlobalMiddleware`, `ExtractIdsMiddleware`) implement cross-cutting concerns without polluting layers.

### Primary Style 2: Microkernel (Plugin Architecture)

NocoDB implements a rich plugin ecosystem for storage adapters, notification channels, and integrations. The core application acts as the kernel; plugins slot in without modifying the kernel.

**Plugin categories in `src/plugins/`:**
- **Storage:** S3, GCS, Minio, Backblaze, Linode, Scaleway, OVH Cloud, R2, UpCloud, Vultr, Spaces, GenericS3
- **Email:** SMTP, SES, MailerSend
- **Notifications:** Slack, Discord, Teams, Mattermost, Twilio, Twilio WhatsApp

**Integration framework** (`packages/noco-integrations/`):
- Provides a base `IntegrationWrapper` class and `IntegrationRegistry` for auto-discovery at build time.
- Build-time script (`registerIntegrations.js`) scans `@noco-integrations/*` packages and generates an integration index, supporting third-party plugins.
- Plugin interface defines `init()`, `fileCreate()`, `fileDelete()`, `test()` contracts for storage adapters.

`NcPluginMgrv2` (referenced in MCP service and elsewhere) manages plugin instantiation and lifecycle at runtime.

### Secondary Style: Event-Driven (Internal Coordination)

Event-driven patterns are pervasive in internal coordination, though not the primary communication mechanism.

**Evidence:**

1. **AppHooksService** (`src/services/app-hooks/app-hooks.service.ts`): A strongly-typed, in-process event bus with 80+ named `AppEvents` (e.g., `TABLE_CREATE`, `COLUMN_UPDATE`, `WEBHOOK_TRIGGER`, `DATA_EXPORT`, `USER_SIGNUP`). Services emit events; listeners react to them without direct coupling.

2. **Dual event emitter strategy:** `EventEmitterModule` provides an `IEventEmitter` interface backed by either a fallback `Emittery`-based emitter or `NestJsEventEmitter` (EventEmitter2), selected at startup. Redis pub/sub (`PubSubRedis`) is used for cross-instance coordination in horizontally-scaled deployments.

3. **Async job queue** (`src/modules/jobs/`): Bull (Redis-backed) queue with 30+ `JobTypes` for heavy asynchronous work (data export, thumbnail generation, webhook dispatching, source creation, attachment migration). Fallback in-process `PQueue` queue is used when Redis is absent. Jobs communicate outcomes via `JobsEventService` which re-emits Bull queue events (`OnQueueActive`, `OnQueueCompleted`, `OnQueueFailed`).

4. **WebSocket Gateway** (`src/gateways/socket.gateway.ts`): Socket.io gateway propagates job status updates and notifications to connected browser clients in real time.

5. **Emit handlers** (`src/services/emit-handler/`): Event handlers that listen to internal events and enqueue jobs (e.g., `AttachmentUrlUploadHandler` listens for `HANDLE_ATTACHMENT_URL_UPLOAD` and enqueues a `JobTypes.AttachmentUrlUpload` job).

6. **HookHandlerService**: Listens to data lifecycle events and triggers outbound webhooks to user-configured HTTP endpoints. The webhook dispatch itself is offloaded to the job queue.

7. **MetaDependencyEventHandler**: Listens to metadata lifecycle events (`COLUMN_ADDED`, `COLUMN_UPDATED`, `COLUMN_DELETED`) and triggers dependent model updates (e.g., formula recalculation, rollup invalidation).

## What Was Considered and Rejected

**Modular Monolith:** NestJS modules group code, but there are no enforced domain boundaries, separate schemas per module, or autonomy contracts. All services share a single metadata database and a single process.

**Microservices:** A single Dockerfile produces a single container. Docker Compose deployments show nocodb + a database, not multiple independently deployed services. The `NC_WORKER_CONTAINER` environment variable allows separating worker processes, but this is a configuration of the same monolith — not independent microservices.

**Hexagonal Architecture:** No explicit ports/adapters pattern. Controllers call services directly; services call model classes directly. The `IBaseModelSqlV2` interface provides some adapter-like abstraction over different SQL databases, but this is database-layer only, not a full hexagonal design.

**Domain-Driven Design:** No bounded contexts, aggregates, domain events as first-class citizens, or ubiquitous language formalized in the codebase structure. The model layer is an Active Record pattern, not a rich domain model.

**CQRS:** No command/query separation. Services handle both reads and writes. The V3 API is a versioning concern, not a CQRS pattern.

## Deployment Topology

NocoDB is deployed as a **single monolithic container**. Optional Redis provides queue, cache, and pub/sub. Optional worker containers run the same image with `NC_WORKER_CONTAINER=true` to separate background job processing from the API server. There is no microservices topology — all runtime concerns live within the same NestJS application process.

```
[Browser / API clients]
        |
  [nc-gui (served by nocodb)]  <-->  [nocodb NestJS API + Job Workers]
                                           |              |
                                     [Meta DB]        [Data DBs]
                                   (SQLite/PG/MySQL)  (external sources)
                                           |
                                        [Redis]
                                   (queue/cache/pubsub)
```
