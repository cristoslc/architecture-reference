# sdk-go — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/cloudevents/sdk-go
**Classification:** Microkernel
**Confidence:** 0.90

## Summary

The CloudEvents Go SDK is a **Microkernel** architecture. A minimal, stable core (`v2/`) defines the canonical event model, message abstraction interfaces, and client API. Nine independently-versioned protocol implementations (AMQP, Kafka Sarama, Kafka Confluent, MQTT, NATS, NATS JetStream, Google Pub/Sub, STAN, WebSocket) plug into that core via Go interfaces, each packaged as a separate Go module with its own `go.mod`. The architecture exhibits the defining Microkernel properties: a small, stable kernel that does not import plugins, and plugins that depend only on the kernel. Hexagonal ports-and-adapters is a secondary design approach visible in the kernel's boundary layer but is not the topology-defining force.

## Evidence

### Module Structure

The repository is organized as a multi-module Go workspace. The core is the single `v2/` module:

```
github.com/cloudevents/sdk-go/v2      ← kernel
github.com/cloudevents/sdk-go/sql/v2  ← optional expression-language module
github.com/cloudevents/sdk-go/protocol/amqp/v2
github.com/cloudevents/sdk-go/protocol/kafka_confluent/v2
github.com/cloudevents/sdk-go/protocol/kafka_sarama/v2
github.com/cloudevents/sdk-go/protocol/mqtt_paho/v2
github.com/cloudevents/sdk-go/protocol/nats/v2
github.com/cloudevents/sdk-go/protocol/nats_jetstream/v2
github.com/cloudevents/sdk-go/protocol/nats_jetstream/v3
github.com/cloudevents/sdk-go/protocol/pubsub/v2
github.com/cloudevents/sdk-go/protocol/stan/v2
github.com/cloudevents/sdk-go/protocol/ws/v2
github.com/cloudevents/sdk-go/observability/opencensus/v2
github.com/cloudevents/sdk-go/observability/opentelemetry/v2
```

Each protocol module's `go.mod` replaces the core with a local path (`replace github.com/cloudevents/sdk-go/v2 => ../../../v2`), confirming the one-way dependency: plugins depend on the kernel; the kernel depends on nothing in the plugin tree.

### The Kernel (`v2/`)

The kernel packages and their roles:

| Package | Role |
|---|---|
| `v2/event` | Canonical `Event` struct and `EventContext` (v0.3 / v1.0) |
| `v2/binding` | `Message`, `MessageReader`, `Transformer` — the plugin contract |
| `v2/protocol` | `Sender`, `Receiver`, `Requester`, `Responder`, `Opener`, `Closer` interfaces |
| `v2/client` | Host application facade; accepts any `protocol.*` implementor |
| `v2/types` | Shared scalar types (`Timestamp`, `URIRef`) |
| `v2/context` | Shared context utilities (target URL, retry policy, logger) |
| `v2/extensions` | Optional CloudEvents extensions (tracing, dataref) |
| `v2/observability` | Observability keys only; concrete adapters are separate modules |

The kernel's `v2/go.mod` has zero dependency on any protocol or observability module — confirming the kernel is self-contained.

### Plugin Contract (the Interfaces)

`v2/protocol/inbound.go` and `v2/protocol/outbound.go` define the extension points:

```go
type Receiver interface {
    Receive(ctx context.Context) (binding.Message, error)
}
type Sender interface {
    Send(ctx context.Context, m binding.Message, transformers ...binding.Transformer) error
}
type Requester interface {
    Request(ctx context.Context, m binding.Message, transformers ...binding.Transformer) (binding.Message, error)
}
type Responder interface {
    Respond(ctx context.Context) (binding.Message, ResponseFn, error)
}
type Opener interface {
    OpenInbound(ctx context.Context) error
}
```

The `binding.Message` interface adds lifecycle management (QoS 0/1/2) and encoding negotiation (`ReadStructured`, `ReadBinary`), giving plugins full control over wire-format fidelity.

### Client as Plugin Host

`v2/client/client.go` is the plugin host. It discovers capabilities via Go interface type assertions at construction time — the canonical Microkernel "duck typing" extension point:

```go
func New(obj interface{}, opts ...Option) (Client, error) {
    if p, ok := obj.(protocol.Sender); ok     { c.sender = p }
    if p, ok := obj.(protocol.Requester); ok  { c.requester = p }
    if p, ok := obj.(protocol.Responder); ok  { c.responder = p }
    if p, ok := obj.(protocol.Receiver); ok   { c.receiver = p }
    if p, ok := obj.(protocol.Opener); ok     { c.opener = p }
    ...
}
```

A plugin can implement any subset of the five capability interfaces; unused capabilities are silently absent. Plugins are mixed and matched — a user imports only the protocols they need.

### Plugin Implementations

Each of the nine protocol plugins follows an identical structure:

1. Has its own `go.mod` with a unique module path.
2. Defines a `Protocol` struct implementing whichever kernel interfaces the underlying transport supports.
3. Includes explicit compile-time conformance assertions such as:
   ```go
   var _ protocol.Sender   = (*Protocol)(nil)
   var _ protocol.Receiver = (*Protocol)(nil)
   ```
4. Imports only `github.com/cloudevents/sdk-go/v2/binding` and `v2/protocol`; never imports another plugin.

The two observability modules (`opencensus`, `opentelemetry`) follow the same pattern: they implement `client.ObservabilityService` from the kernel and are imported by the adopter only when tracing is wanted.

### Transformer Pipeline (Within the Kernel)

The `binding.Transformer` interface and `binding.Transformers` (a slice that chains transformers) implement a mini-pipeline within the message encoding path. This is an internal design detail, not a top-level topology. The `v2/binding/transformer/` sub-package ships built-in transformers (add/set/delete metadata, version conversion).

### Built-in Protocols

Two protocol implementations live inside the kernel module itself:

- `v2/protocol/http` — uses `net/http`; bidirectional request/response support
- `v2/protocol/gochan` — in-process Go channel transport for testing

These are convenience inclusions, not violations of the kernel boundary — they depend only on kernel types and standard library.

### Secondary Design Approach: Hexagonal (Ports and Adapters)

The architecture exhibits textbook ports-and-adapters structuring:

- **Ports**: `v2/protocol` interfaces (`Sender`, `Receiver`, etc.) and `v2/binding` interfaces (`Message`, `StructuredWriter`, `BinaryWriter`)
- **Adapters**: Each protocol module adapts an external broker (AMQP, Kafka, NATS) to the port interfaces
- **Core isolation**: The `v2/client` and `v2/event` packages depend only on abstractions, never on concrete protocol implementations

However, the primary architectural driver is the **plugin extension model** — the fact that each protocol is an independently versioned, selectively importable module. Hexagonal isolation serves that Microkernel goal; it is not itself the topology.

## Rejected Styles

| Style | Reason Rejected |
|---|---|
| **Layered** | No horizontal presentation/business/data layering. The vertical plugin axis dominates. |
| **Modular Monolith** | Plugins are independently importable modules, not modules in a single deployable unit. |
| **Event-Driven** | The SDK carries CloudEvents but internally uses direct interface calls, not event-driven messaging between its own components. |
| **Pipeline** | The transformer chain is an intra-message encoding detail, not the system's primary topology. |
| **Microservices / Service-Based** | This is a library, not a set of deployed services. |
| **CQRS / DDD** | No command/query split or domain model organization. |
| **Serverless** | No function-as-a-service topology. |
| **Space-Based** | No in-memory data grid. |
| **Multi-Agent** | No autonomous agents. |
| **Indeterminate** | Architecture is clear and consistent throughout the codebase. |

## Quality Attributes

- **Extensibility**: Primary attribute. Callers can add protocols without forking the SDK.
- **Portability**: Events are transport-agnostic; the same `Event` struct works across all protocols.
- **Interoperability**: SDK implements the CNCF CloudEvents 0.3 and 1.0 specifications, enabling cross-platform event exchange.
- **Modularity**: Independent `go.mod` per plugin enforces module-level isolation and selective dependency.
- **Observability**: Pluggable `ObservabilityService` interface + dedicated OpenCensus and OpenTelemetry adapter modules.
- **Testability**: Go channel protocol and test helper packages (`v2/binding/test`, `v2/client/test`, `v2/protocol/test`) enable fast in-process unit testing.
