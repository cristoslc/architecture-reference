# Twenty тАФ Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/twentyhq/twenty
**Version analyzed:** v1.18.0 (main branch)
**Classification:** Modular Monolith (primary), Event-Driven (secondary)
**Confidence:** 0.90

---

## Summary

Twenty is an open-source CRM platform (a modern alternative to Salesforce) built on an Nx monorepo. Its backend is a **Modular Monolith**: all business logic deploys as a single NestJS server process organized into 278+ strictly bounded NestJS modules. A separate queue-worker process runs from the same Docker image and shares the same PostgreSQL database, confirming there is no service-per-process decomposition. An **Event-Driven** secondary style is present through a `WorkspaceEventEmitter` (EventEmitter2-based internal event bus), a BullMQ/Redis job queue system with 17 named queues, and Redis-backed SSE subscriptions for real-time frontend updates. Notably, Twenty has added significant AI capabilities (multi-model LLM agent execution, workflow AI steps, MCP tool server, and a Model Context Protocol endpoint) but these are embedded subsystems within the monolith rather than an independent architectural tier.

### 1. Primary Style: Modular Monolith

**Single deployable unit:**

`packages/twenty-docker/docker-compose.yml` uses the same image (`twentycrm/twenty:latest`) for both the `server` and `worker` services. The worker starts with `yarn worker:prod` but runs from identical compiled code. A single PostgreSQL instance and single Redis instance are shared. No network boundary separates the two processes at the application code level.

**Internal module boundaries enforced by NestJS DI:**

The `twenty-server` src tree contains 278+ NestJS `@Module` classes. Each module exposes an explicit `imports` array, making cross-module dependencies declarative and traceable. Domain modules under `src/modules/` follow a consistent internal structure:
- `*.module.ts` тАФ NestJS module wiring
- `standard-objects/` тАФ workspace entity definitions
- `services/` тАФ business logic
- `jobs/` тАФ background job processors
- `resolvers/` тАФ GraphQL endpoints
- `repositories/` тАФ data access via TwentyORM

Domain modules: `calendar`, `messaging`, `workflow`, `connected-account`, `contact-creation-manager`, `dashboard`, `favorite`, `note`, `task`, `opportunity`, `company`, `person`, `attachment`, `blocklist`, `match-participant`, `timeline`, `workspace-member`.

**TwentyORM: a custom multi-tenant data access layer:**

`src/engine/twenty-orm/` is a custom abstraction over TypeORM that maintains per-workspace datasources. Each workspace can have its own dynamic schema (defined via metadata modules), and TwentyORM manages the entity/repository lifecycle for each. This is a sophisticated internal infrastructure concern, not a distributed system boundary.

**Nx monorepo with enforced boundaries:**

`nx.json` defines `scope:backend` and `scope:frontend` tags with build caching and dependency enforcement. The fifteen packages in the monorepo are compiled and resolved at build time, not at runtime as separate services.

**Metadata-driven schema engine:**

The `metadata-modules/object-metadata/` system allows workspaces to define custom CRM objects at runtime. GraphQL schema is built dynamically per workspace via `workspace-schema-builder`. This is a runtime metaprogramming pattern within the monolith тАФ not a separate service.

### 2. Secondary Style: Event-Driven

Three distinct event-driven mechanisms operate within the monolith:

**Internal event bus (WorkspaceEventEmitter):**

`src/engine/workspace-event-emitter/workspace-event-emitter.ts` uses `EventEmitter2` (NestJS event-emitter) to broadcast typed `WorkspaceEventBatch` and `CustomWorkspaceEventBatch` objects within the process. Events carry workspace ID, object metadata, and a typed action (CREATED, UPDATED, DELETED, etc.). Consumers register with `@OnWorkspaceEvent` decorators. This is the primary mechanism for decoupling domain actions from side-effects (audit log writes, webhook triggers, contact creation, calendar sync).

**BullMQ job queue (17 named queues):**

`src/engine/core-modules/message-queue/message-queue.constants.ts` defines 17 queue names in a `MessageQueue` enum:

| Queue | Purpose |
|---|---|
| `messagingQueue` | Email/messaging sync |
| `calendarQueue` | Calendar sync |
| `webhookQueue` | Outbound webhook delivery |
| `workflowQueue` | Workflow step execution |
| `aiQueue` | AI agent job processing |
| `triggerQueue` | Automated trigger evaluation |
| `emailQueue` | Transactional email |
| `billingQueue` | Stripe events |
| `contactCreationQueue` | Contact creation from emails |
| `cronQueue` | Scheduled tasks |
| `entityEventsToDbQueue` | Event log persistence |
| `logicFunctionQueue` | Logic function execution |
| `delayedJobsQueue` | Delayed workflow steps |
| `deleteCascadeQueue` | Cascade delete operations |
| `workspaceQueue` | Workspace lifecycle |
| `taskAssignedQueue` | Task assignment notifications |
| `dashboardSyncQueue` | Dashboard data sync |

The BullMQ driver (`bullmq.driver.ts`) wraps `BullMQ` with metrics tracking, job deduplication (via `options.id`), and graceful shutdown. A synchronous fallback driver exists for test environments.

**Redis SSE subscriptions (real-time frontend):**

`src/engine/subscriptions/subscriptions.module.ts` uses `RedisClientModule` and `EventStreamService` to push real-time record change events to the React frontend via Server-Sent Events (graphql-sse). This bridges the internal event bus to the frontend without WebSocket overhead.

### 3. Workflow Engine: Pipeline Pattern (Scoped)

Twenty's workflow subsystem expresses a **Pipeline pattern** internally тАФ but scoped to the workflow feature, not the overall system:

- `workflow-trigger/` тАФ entry points (automated triggers, HTTP triggers)
- `workflow-builder/` тАФ DAG composition
- `workflow-runner/` тАФ dispatches runs to the queue (`RunWorkflowJob`)
- `workflow-executor/` тАФ resolves each step to an action handler via `WorkflowActionFactory`
- Action handlers are pluggable: `code`, `http-request`, `form`, `ai-agent`, `if-else`, `iterator`, `filter`, `delay`, `record-crud`, `mail-sender`, `logic-function`

The workflow builder is exposed visually in the frontend via React Flow (node graph UI). The pattern is Pipeline/DAG at the workflow level but does not define the system's primary runtime topology.

### 4. AI-Native Subsystem (Embedded, Not Architectural Style)

Twenty has built a substantial AI layer that is structurally embedded within the Modular Monolith:

**Multi-model LLM support (7 providers):**

`src/engine/metadata-modules/ai/ai-models/constants/` contains model constant files for: `openai`, `anthropic`, `bedrock`, `google`, `xai`, `groq`, `mistral`. A unified `AI_MODELS` constant aggregates all supported models.

**Agent execution (ReAct / tool-use loop):**

`AgentAsyncExecutorService` (in `ai-agent-execution/`) implements an agentic loop using the Vercel AI SDK's `generateText` with tool calling. The loop:
1. Calls the configured LLM with a curated toolset (DATABASE_CRUD, ACTION, NATIVE_MODEL categories тАФ workflow tools excluded to prevent recursion)
2. Respects role-based permission filtering on available tools
3. Handles malformed tool calls via `experimental_repairToolCall`
4. If the agent defines a JSON output schema, makes a secondary synthesis call to structure the result

**Agent configuration model:**

`AgentEntity` is a first-class metadata entity. Agents are assigned roles (RBAC) and can be used as workflow action steps or chat participants. Applications (`ApplicationEntity`) contain agents + logic functions as composable components.

**MCP (Model Context Protocol) server:**

`src/engine/api/mcp/` exposes an HTTP MCP endpoint allowing external LLM clients to discover and invoke Twenty CRM tools (execute-tool, get-tool-catalog, learn-tools, load-skill) via the standard MCP protocol. This enables Twenty to act as a tool server for external AI agents.

**AI chat:**

`src/engine/metadata-modules/ai/ai-chat/` supports multi-turn chat sessions with agents. The `AgentChatThreadEntity` tracks conversation history.

**Logic functions:**

`src/engine/core-modules/logic-function/` provides a runtime for user-defined code (logic function drivers) that can be triggered by workflow steps or event hooks.

The AI capabilities are extensive but are implemented as NestJS modules within the monolith тАФ they do not introduce an independent inference service or a separate AI process boundary.

### 5. Frontend Architecture

`packages/twenty-front/` is a React SPA:

- **State management:** Jotai (atom-based, local state) + Apollo Client (server state / GraphQL)
- **Data fetching:** Apollo Client with GraphQL subscriptions via `graphql-sse`
- **AI SDK:** Vercel AI SDK (`ai` v6, `@ai-sdk/react` v3) for frontend LLM streaming
- **Workflow builder UI:** React Flow v12 for visual workflow DAG editing
- **Rich text:** BlockNote + TipTap editors

The frontend is a separate compilation unit but communicates exclusively with the backend via GraphQL (data) and SSE (realtime push). It is not independently deployable as a service with its own backend concerns.

### 6. Deployment Topology

```
docker-compose.yml services:
  server   тЖТ twentycrm/twenty (default command: API server)
  worker   тЖТ twentycrm/twenty (command: yarn worker:prod)
  db       тЖТ postgres:16 (single shared database)
```

The `worker` depends on `server` being healthy before starting. Both share the same local storage volume. This confirms: **one codebase, one database, two process roles** тАФ a classic modular monolith with worker sidecar, not microservices.

## What This Is Not

**Not Microservices:** There is no independent deployable service with its own database. `server` and `worker` share one image, one PostgreSQL schema, one Redis. In-process NestJS DI is the communication mechanism, not HTTP or message bus service calls.

**Not Service-Based:** The twenty packages are compilation units in an Nx monorepo, not deployable services. The only runtime processes are server and worker, both from the same binary.

**Not Multi-Agent (architectural style):** AI agents are a feature offered to users тАФ users can configure agents that run as workflow steps or chat participants. The system itself does not run as a fleet of coordinating autonomous agents. The AI subsystem is a module cluster within the monolith.

**Not Pipeline (system-wide):** The workflow engine within Twenty implements a pipeline execution model, but the overall system is not organized as a series of independent processing stages. The pipeline pattern is scoped to the `workflow` module cluster.

**Not Event-Driven (primary):** Events (EventEmitter2 + BullMQ) are used to decouple cross-cutting concerns (audit, webhooks, sync jobs) from domain actions. But the primary organizing principle remains synchronous NestJS DI and GraphQL request/response, not message-passing choreography.
