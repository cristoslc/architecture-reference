## One-Line Summary

Greg Young's canonical "Simplest Possible Thing" reference implementation of CQRS with Event Sourcing — a single-process .NET application demonstrating strict command/query segregation, an in-memory event store, and event-driven read model projection.

---

## Project Overview

The `m-r` repository is the primary reference implementation authored by Greg Young, one of the originators of the CQRS pattern. It is a deliberately minimal inventory management sample built in C# targeting .NET 7, consisting of two projects: `SimpleCQRS` (the core library) and `CQRSGui` (an ASP.NET Core MVC web front end). The solution is named `SimplestPossibleThing.sln`, signaling its intent as a teaching artefact rather than a production system.

The codebase contains approximately 10 source files (excluding static web assets) organized around a single domain concept — an inventory item — and is structured to expose, in the clearest possible form, how CQRS and Event Sourcing interact.

## Structural Evidence

### Layer Organization

The application is organized across recognizable horizontal layers:

| Layer | Location | Role |
|---|---|---|
| Presentation | `CQRSGui/Controllers/HomeController.cs`, `Views/` | MVC controller + Razor views; sends commands, reads from read facade |
| Application / Command | `SimpleCQRS/CommandHandlers.cs` | Receives commands from bus, loads aggregate, calls domain methods, persists |
| Domain | `SimpleCQRS/Domain.cs` | `InventoryItem` aggregate + `AggregateRoot` base class with event-sourced state |
| Infrastructure / Messaging | `SimpleCQRS/FakeBus.cs`, `SimpleCQRS/EventStore.cs` | In-memory command/event bus and event store |
| Read Model | `SimpleCQRS/ReadModel.cs` | Event-handler views + DTOs + facade; completely separate from write model |

### CQRS Implementation

The codebase implements textbook CQRS:

**Write path (Commands):**
- `Commands.cs` defines intent-bearing command objects: `CreateInventoryItem`, `RenameInventoryItem`, `DeactivateInventoryItem`, `CheckInItemsToInventory`, `RemoveItemsFromInventory`, `ChangeMaxQty`.
- `FakeBus.Send<T>()` enforces single-handler routing for commands (throws if handler count != 1).
- `InventoryCommandHandlers` receives commands, loads the `InventoryItem` aggregate from the repository, calls the appropriate domain method, and saves via the repository.
- The `Repository<T>` reconstitutes aggregate state by replaying events from the `EventStore`.

**Read path (Queries):**
- `ReadModel.cs` defines separate projections (`InventoryListView`, `InventoryItemDetailView`) that subscribe to domain events and maintain flat in-memory read state in `FakeDatabase`.
- `ReadModelFacade` exposes `GetInventoryItems()` and `GetInventoryItemDetails()` — pure data retrieval with no domain model involvement.
- The controller reads exclusively through `ReadModelFacade`, never touching the aggregate or event store directly.

**Separation enforced in `Program.cs`:**
```
Write side:  Commands → FakeBus → CommandHandlers → Repository → EventStore
Read side:   EventStore publishes events → FakeBus → InventoryListView, InventoryItemDetailView → FakeDatabase
Query path:  Controller → ReadModelFacade → FakeDatabase
```

### Event Sourcing

`AggregateRoot` implements Event Sourcing as a first-class mechanism:
- `ApplyChange(Event)` dispatches events via private reflection (`AsDynamic().Apply(@event)`) to type-specific `Apply()` methods, then accumulates uncommitted events.
- `LoadsFromHistory(IEnumerable<Event>)` replays a stream of past events to rehydrate aggregate state — the only way aggregates are reconstituted from the repository.
- `EventStore.SaveEvents()` persists events with optimistic concurrency checking (version mismatch raises `ConcurrencyException`) and immediately publishes each event to the bus for downstream read model updates.
- The `OriginalVersion` field carried on every command is the optimistic concurrency token; mismatches are caught in `EventStore` before persistence.

### Domain Model

`InventoryItem` is a simple aggregate with encapsulated business rules:
- `CheckIn(count)` enforces that `AvailableQty + count <= MaxQty`.
- `Remove(count)` enforces that `AvailableQty - count >= 0`.
- `Deactivate()` enforces idempotency (cannot deactivate twice).
- All state changes go through `ApplyChange(new EventType(...))`, making the event the single source of truth for state transitions.

## Deployment Topology

Single deployable unit: one ASP.NET Core process serving both the command-handling infrastructure and the query-serving read model. There are no service boundaries, message brokers, or separate deployables. The `FakeBus` is an in-process dictionary of handlers; the `EventStore` is an in-memory `Dictionary<Guid, List<EventDescriptor>>`. This is an intentional simplification for instructional clarity.

## Why Not Other Styles

- **Not Microservices:** Single solution, single process, no service boundaries or network calls between components.
- **Not Event-Driven (macro):** The bus is internal (`FakeBus`), not a message broker enabling inter-service communication.
- **Not Domain-Driven Design (as primary):** One aggregate, no bounded contexts, no value objects, no domain services. The domain model is a minimal vehicle for demonstrating CQRS mechanics.
- **Not Hexagonal:** No port/adapter separation. `FakeBus` and `FakeDatabase` are inline stubs, not adapters to external systems behind interfaces.
- **Not Modular Monolith:** No module boundaries beyond the two projects. `SimpleCQRS` is a library, not an independent module with its own contract.

## Classification Reasoning

The topology is `Layered`: the application is organized into a presentation layer (MVC), an application/command-handling layer (`CommandHandlers`), a domain layer (`AggregateRoot`, `InventoryItem`), and a storage/infrastructure layer (`EventStore`, `FakeBus`, `FakeDatabase`). Dependencies flow strictly downward with no circular references across layers.

`CQRS` is the dominant design pattern and the repository's explicit purpose: command and query paths are structurally separated at every level — from the bus routing rules, through the handler types, through the storage mechanism, to the controller action methods. The separation is complete and intentional.

Event Sourcing is the persistence strategy that underpins the write side — aggregate state is never updated in place; it is always reconstructed from an ordered stream of events. This is a data-pattern qualifier rather than a topology-defining style.
