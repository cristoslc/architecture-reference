## One-Line Summary

A Hacker News-style forum application that is an explicit teaching reference for Domain-Driven Design tactical patterns (aggregates, value objects, domain events, repositories) layered with Clean Architecture within a single deployable monolith.

---

## Architecture Styles

- **Domain-Driven Design** (primary)
- **Layered** (secondary — Clean Architecture variant)

## Classification Confidence

**0.97**

## Structural Overview

### Deployment Topology

Single deployable unit. One `package.json`, one `Dockerfile`, and a `docker-compose.yml` that provisions only supporting infrastructure (MySQL via Sequelize, Redis for JWT session management, Adminer for DB inspection). No service decomposition exists.

### Source Organization

```
src/
  config/
  index.ts                          # entry point: wires infra + subscriptions
  modules/
    forum/                          # forum bounded context
      domain/                       # aggregates, value objects, domain events, domain services
      dtos/
      infra/http/routes/            # Express REST controllers
      mappers/
      repos/                        # repo interfaces + Sequelize implementations
      subscriptions/                # domain event handlers (AfterUserCreated, etc.)
      useCases/                     # use case classes (CreatePost, UpvoteComment, etc.)
    users/                          # users bounded context
      domain/
      infra/http/routes/
      mappers/
      repos/
      services/redis/               # JWT auth via Redis
      useCases/
  shared/
    core/                           # UseCase<>, Result<>, Guard, AppError
    domain/                         # AggregateRoot<>, Entity<>, ValueObject<>, DomainEvents
    infra/
      database/sequelize/           # ORM models, migrations, Sequelize hooks
      http/                         # Express app, REST v1 router, GraphQL (Apollo)
```

Two bounded contexts (`forum`, `users`) reside within a single process. Cross-context integration occurs exclusively through domain events: `UserCreated` (raised by the `User` aggregate) is consumed by `AfterUserCreated` in the forum context, which calls the `CreateMember` use case synchronously in-process.

## Evidence

### Domain-Driven Design (Primary)

**Aggregates and Aggregate Roots**
- `User extends AggregateRoot<UserProps>` (`src/modules/users/domain/user.ts`)
- `Post extends AggregateRoot<PostProps>` (`src/modules/forum/domain/post.ts`)
- `Member extends AggregateRoot<MemberProps>` (`src/modules/forum/domain/member.ts`)
- `AggregateRoot<T>` base class manages domain event collection and deferred dispatch via `DomainEvents.markAggregateForDispatch`.

**Value Objects**
Extensive use of `ValueObject<T>` for all primitive wrapping: `UserEmail`, `UserName`, `UserPassword`, `PostTitle`, `PostSlug`, `PostText`, `PostLink`, `CommentText`, `MemberId`, `PostId`, `CommentId`, etc.

**Domain Events**
- `IDomainEvent` interface and `DomainEvents` static dispatcher (`src/shared/domain/events/`)
- Events raised: `UserCreated`, `UserLoggedIn`, `UserDeleted`, `PostCreated`, `CommentPosted`, `PostVotesChanged`, `CommentVotesChanged`, `MemberCreated`
- Dispatch is triggered by Sequelize ORM `afterCreate`/`afterUpdate`/`afterSave`/`afterDestroy` hooks (`src/shared/infra/database/sequelize/hooks/index.ts`) — events fire after persistence is committed, not before
- Subscribers: `AfterUserCreated`, `AfterCommentPosted`, `AfterCommentVotesChanged`, `AfterPostVotesChanged`

**Repositories**
- Domain-layer interfaces (`IUserRepo`, `IPostRepo`, `IMemberRepo`, `ICommentRepo`, etc.) in `repos/`
- Concrete Sequelize implementations in `repos/implementations/` (`SequelizeUserRepo`, `PostRepo`, etc.)
- Repo interfaces depend only on domain types; concrete implementations are wired at startup

**Domain Services**
- `src/modules/forum/domain/services/postService.ts` — vote resolution logic

**Bounded Contexts**
- `users` subdomain: identity, authentication, JWT lifecycle
- `forum` subdomain: posts, comments, members, voting
- README explicitly identifies these as the two subdomains

**Mappers and DTOs**
- `UserMap`, `PostMap`, `CommentMap`, `MemberMap`, `PostDetailsMap`, `MemberDetailsMap` translate between domain objects and persistence/transport representations
- Separate DTO types per use case (e.g., `CreateUserDTO`, `CreatePostDTO`)

### Layered Architecture — Clean Architecture Variant (Secondary)

Each bounded context enforces a strict inward dependency rule across four layers:

| Layer | Artifacts |
|---|---|
| Domain | Entities, AggregateRoots, ValueObjects, DomainEvents, DomainServices |
| Application | UseCase classes (`CreatePost`, `Login`, `UpvoteComment`, etc.) |
| Adapter | Repo interfaces, Mappers, DTOs |
| Infrastructure | Sequelize models, Express controllers/routes, Redis auth service, Apollo GraphQL |

Use cases depend on repo interfaces, not concrete implementations. Concrete repos are injected at the module index files (e.g., `src/modules/forum/repos/index.ts`). This is a textbook Clean Architecture/Hexagonal boundary, though the project does not use the term "ports and adapters" explicitly.

### What Is Not Present

- **Hexagonal Architecture:** The project uses the same inward-dependency approach but does not define named ports or adapter interfaces beyond the repo layer. The boundary is implicit, not formalized with explicit port types.
- **Microservices / Service-Based:** Single process, single `Dockerfile`, no inter-service communication.
- **CQRS:** No separate read and write models. Some use cases read projected details (`PostDetails`, `MemberDetails`) but these are query projections built inside the same repository, not a distinct read model updated by events.
- **Event-Driven Architecture (macro):** Domain events are intra-process synchronous callbacks dispatched via Sequelize ORM hooks — not a message broker, not asynchronous messaging infrastructure.
- **Serverless / Pipeline / Space-Based / Microkernel / Multi-Agent:** No evidence of any of these patterns.

## Quality Attributes

- **Maintainability:** High — Clean Architecture layering and explicit DDD tactical patterns make each layer independently changeable. Repo interfaces decouple domain logic from Sequelize.
- **Testability:** High — Use cases accept interface types and can be tested with mock repos; the repo pattern and domain factories support unit testing without infrastructure.
- **Modularity:** Medium — Two bounded contexts are clearly separated by directory, but they share a single database schema and a single process. Cross-context coupling via domain events is controlled but synchronous.
- **Expressiveness / Readability:** High — The codebase is explicitly educational. Every tactical DDD pattern is named and structured in a way that maps directly to DDD literature.
- **Scalability:** Low — Single monolithic process; no horizontal scaling mechanisms beyond vertical resource increases or database connection pooling.

---

## Summary Reasoning

ddd-forum is an unambiguous reference implementation of Domain-Driven Design tactical patterns. Every major DDD artifact — aggregate roots, value objects, domain events, repositories, domain services, bounded contexts, mappers — is present and structurally correct. The Clean Architecture layering (domain → application → adapter → infrastructure) is the secondary organizing principle that gives the DDD domain model a home inside a single Express application. The domain event system (`DomainEvents.register` + Sequelize `afterSave` hooks) provides bounded-context integration without direct coupling, but all of this occurs in-process and synchronously — placing the project firmly in the Layered Monolith category at the deployment level, with DDD as the dominant architectural style.
