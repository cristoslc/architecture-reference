## Architecture Styles

### Primary: Microkernel

The core mechanism is `AFPluginDispatcher` (`frontend/rust-lib/lib-dispatch/src/dispatcher.rs`), a central event bus that routes typed requests to registered plugin handlers. Each domain module â€” Folder, Document, Database, User, AI, Search, Storage â€” exports an `init()` function that returns an `AFPlugin` instance registering event-to-handler bindings. All plugins are assembled in `flowy-core/src/module.rs` via `make_plugins()`. The Flutter side mirrors this with `PluginSandbox` and `registerPlugin()` calls in `lib/startup/tasks/load_plugin.dart`, providing content-type plugins (Document, Grid, Board, Calendar, AIChatPlugin) behind the abstract `Plugin` / `PluginBuilder` interfaces. Feature addition requires no changes to the core dispatcher â€” only a new plugin registration.

### Secondary: Modular Monolith

The Rust workspace (`frontend/rust-lib/Cargo.toml`) enforces strong module boundaries through 30+ crates, each with a clearly scoped interface. Every domain module is paired with a `-pub` crate (e.g., `flowy-folder-pub`, `flowy-document-pub`) that declares public trait ports (`FolderCloudService`, `DocumentCloudService`, `UserCloudService`) â€” a hexagonal ports-and-adapters pattern inside the workspace. The entire system compiles to a single desktop binary (or mobile app) accessed over FFI from Flutter via `dart-ffi`.

## Production Context

- Ships as native desktop (macOS, Windows, Linux), iOS, and Android; all platforms share one Flutter+Rust codebase compiled per target.
- Offline-first: local SQLite (`flowy-sqlite`) + CRDT collaboration layer (`AppFlowy-Collab` using `yrs`/Y.js protocol) â€” cloud sync is an optional overlay via `AppFlowyServer` trait.
- Self-hostable: documented self-hosting path via `AppFlowy-Cloud` (separate repo); client references it via git dependency in `Cargo.toml`.
- Active production release cadence with CHANGELOG, Codemagic CI, Flathub, Snapcraft, and App Store / Play Store distribution.
