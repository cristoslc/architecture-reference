# Architecture Report: Apache NiFi

**Date:** 2026-03-11
**Version Analyzed:** 2.9.0-SNAPSHOT (main branch, depth-1 clone)
**Source:** https://github.com/apache/nifi
**Classification:** Pipeline + Microkernel + Modular Monolith
**Confidence:** 0.95

---

## Summary

Apache NiFi is a data flow automation platform built around a visual, directed-graph pipeline
model where discrete "processors" pass FlowFile records through typed "connections" (queues).
The framework is a microkernel: a thin, stable core (FlowController + FlowEngine + NAR
classloader) that discovers and loads extension bundles (NARs) at runtime. The entire system
ships as a single deployable unit — a Maven multi-module monolith — with 73+ extension
bundles and a rich plugin ecosystem.

## Architecture Analysis

### Primary Style 1: Pipeline (confidence 0.95)

The central abstraction in NiFi is the **data flow graph**: a directed, acyclic-or-cyclic
graph of `Connectable` nodes joined by `Connection` objects that act as bounded queues.
Data units are `FlowFile` records (attributes + content pointer), routed between processors
via typed `Relationship` outputs.

Key evidence:
- `Connectable.java` (`nifi-framework-core-api`) — the core graph vertex interface, extended by
  `ProcessorNode`, `Port`, `Funnel`; each `Connectable` exposes `getRelationships()` and
  `getIncomingConnections()` / `getOutgoingConnections()`.
- `Connection.java` — the graph edge; holds a priority queue of FlowFiles; enforces back-pressure
  via `nifi.queue.backpressure.count` and `nifi.queue.backpressure.size`.
- `FlowEngine.java` — a `ScheduledThreadPoolExecutor` that triggers each processor on its
  configured schedule (timer, event-driven, cron).
- `FlowController.java` — the top-level runtime object wiring all components (processors,
  connections, process groups, repositories) together into a live flow graph.
- `nifi.flow.configuration.file` — the entire flow graph is persisted as `flow.json.gz` and
  restored on startup; the pipeline is a first-class, versioned artifact.
- Stateless mode (`nifi-stateless/`) executes a sub-graph as a bounded pipeline with
  exactly-once guarantees, reinforcing that the pipeline topology is the primary unit of
  composition.

The flow model is not a batch DAG (like Airflow) — it is a **continuous streaming pipeline**
where each processor runs in a tight loop, consuming from upstream queues and producing to
downstream queues, with guaranteed delivery and configurable back-pressure.

### Primary Style 2: Microkernel (confidence 0.92)

NiFi's framework is a thin, stable kernel that discovers and loads plugins at runtime through
the **NAR (NiFi Archive)** classloader system. A NAR is a ZIP archive containing a bundle of
extension classes and their dependencies, each loaded in an isolated `NarClassLoader`.

Key evidence:
- `NarClassLoaders.java` / `NarClassLoader.java` — each NAR gets its own isolated classloader,
  preventing dependency conflicts between extensions.
- `NarAutoLoader.java` / `StandardNarLoader.java` — watches `nifi.nar.library.autoload.directory`
  and loads new NARs at runtime without restart.
- `ExtensionBuilder.java` — instantiates plugin instances (Processor, ControllerService,
  ReportingTask) by resolving the correct NAR classloader context.
- `nifi-framework-api/` — the formal plugin contract: `Processor`, `ControllerService`,
  `ReportingTask`, `FlowAnalysisRule`, `ParameterProvider` interfaces. Extensions implement
  these; the kernel never depends on extension implementations.
- 73 extension bundles in `nifi-extension-bundles/` covering AWS, Azure, GCP, Kafka,
  Hadoop, Elasticsearch, Snowflake, Slack, Salesforce, SNMP, and many more — all loaded
  as NARs, none compiled into the kernel.
- `nifi-framework-bundle/nifi-framework-extensions/nifi-py4j-framework-bundle/` — Python
  extensions via Py4J bridge; processors can be written in Python, loaded as NARs, and
  executed in isolated Python subprocesses.
- `StandardNarManager.java` — manages runtime NAR lifecycle (install, activate, deactivate)
  via a REST API, enabling hot-plugging of extensions.

### Secondary Style: Modular Monolith (confidence 0.87)

Despite the microkernel extension system, NiFi deploys as a **single process** containing
all components. The Maven multi-module structure enforces strong module boundaries within
one deployable artifact:

- The framework bundle, extension bundles, commons libraries, REST API, and UI all
  assemble into one distribution (`nifi-assembly/`).
- Cluster nodes are identical instances of the same binary — no separate services.
- A single `FlowController` object holds all processors, connections, repositories, and
  cluster state within one JVM.
- The `nifi-registry/` module is the only genuinely separate service (flow versioning).
- `nifi-stateless/` provides an embedded library mode but is not a separate deployed service.

The modular monolith boundary is enforced by Maven module dependencies: `nifi-framework-core`
depends on `nifi-framework-core-api` but not on any extension bundle. Extensions can only
reference `nifi-framework-api`.

## Architectural Components

### Core Runtime

| Component | Location | Role |
|---|---|---|
| `FlowController` | `nifi-framework-core` | Central orchestrator; owns all processors, connections, process groups |
| `FlowEngine` | `nifi-framework-core-api` | Scheduled thread pool that triggers processor execution |
| `NarClassLoaders` | `nifi-nar-utils` | Per-NAR isolated classloaders |
| `WriteAheadFlowFileRepository` | `nifi-framework-core` | Durable FlowFile state (WAL-backed) |
| `FileSystemRepository` | `nifi-framework-core` | Content storage for FlowFile payloads |
| `PersistentProvenanceRepository` | `nifi-provenance-repository-bundle` | Indexed event history for data lineage |

### Plugin Extension Points

| Interface | Location | Purpose |
|---|---|---|
| `Processor` | `nifi-framework-api` | Data transformation/routing unit |
| `ControllerService` | `nifi-framework-api` | Shared service (DB connection pool, schema registry) |
| `ReportingTask` | `nifi-framework-api` | Background metrics/alerting task |
| `FlowAnalysisRule` | `nifi-framework-api` | Flow design validation rule |
| `ParameterProvider` | `nifi-framework-api` | External parameter source (HashiCorp Vault, etc.) |

### Infrastructure

| Component | Technology | Role |
|---|---|---|
| Cluster coordination | ZooKeeper (embedded or external) | Leader election, cluster state |
| REST API | JAX-RS (Jetty) | Management, monitoring, flow design |
| Site-to-Site | Custom TCP/HTTP protocol | Inter-NiFi data transfer |
| State management | Local (RocksDB) + cluster (ZooKeeper) | Processor state storage |
| Auth | OpenID Connect, SAML, LDAP, Kerberos | SSO and identity |
| Python extensions | Py4J bridge + subprocess isolation | Native Python processor support |

## Quality Attributes

### Extensibility
73 extension bundles; NAR hot-loading without restart; formal `nifi-framework-api` plugin
contract; Python extensions via Py4J; custom NAR maven archetype for third-party bundles.

### Reliability
Guaranteed delivery via FlowFile repository (WAL-backed); back-pressure prevents data loss;
configurable retry/backoff per processor; content repository with archiving.

### Scalability
Horizontal clustering via ZooKeeper; load-balanced connection queues distribute FlowFiles
across cluster nodes; stateless mode for high-throughput embedded use.

### Observability
Provenance repository records every FlowFile routing decision with full content metadata;
searchable lineage graph from source to destination; ReportingTask extension point for
metrics/alerting; Prometheus endpoint; QuestDB-backed time-series status history.

### Security
TLS by default; RBAC with fine-grained component-level access policies; OpenID Connect and
SAML 2.0 SSO; Kerberos delegation; HashiCorp Vault integration for secrets; encrypted
sensitive properties.

### Deployability
Docker images (`nifi-docker/`); MiNiFi edge agent (`minifi/`) for resource-constrained nodes;
NiFi Registry for versioned flow promotion across environments; stateless mode for embedded/k8s use.

## What Was Rejected and Why

**Not Microservices:** The entire system is one process. Cluster nodes are identical replicas
of one binary sharing the same flow configuration. The registry is the only genuinely
separate service. There is no service mesh, no per-capability deployment boundary.

**Not Event-Driven (primary):** FlowFiles move through queues between processors, but
coordination is synchronous and pull-based (each processor is polled by `FlowEngine`).
There is no event broker (Kafka, RabbitMQ) mediating internal component interactions.
The data _content_ may be event streams, but the runtime architecture is a continuous
streaming pipeline, not event-driven choreography.

**Not Layered:** Organization is vertical by component type (processor, connection, group)
and horizontal by concern (NAR isolation, repository, cluster), not a strict
presentation/business/data horizontal stack.

**Not Service-Based:** While NiFi has a REST API and a registry, the components
(processors, groups, connections) are not deployed as separate services. They are all
graph nodes within one runtime.

## Key Architectural Insights

1. **The pipeline is the product.** NiFi's flow graph is not an implementation detail — it
   is the primary user artifact, versioned in the registry, drag-and-dropped in the browser
   UI, and the unit of deployment between environments.

2. **NAR isolation is a first-class architecture concern.** Classloader isolation between
   NAR bundles is not an implementation detail — it is what enables 73 independent extension
   bundles with conflicting dependencies (different Hadoop versions, AWS SDK versions, etc.)
   to coexist in the same JVM safely.

3. **Guaranteed delivery is a core property.** The FlowFile repository (WAL) + content
   repository + provenance repository form a three-layered durability stack. A processor
   crash or node failure will not lose data — FlowFiles are re-queued from the WAL on
   restart.

4. **Stateless mode is an architectural boundary.** `nifi-stateless/` allows a sub-graph to
   execute as a bounded, exactly-once pipeline without any persistent repositories. This is
   architecturally significant: it means the pipeline model is separable from the durable
   streaming runtime, enabling embedding in Kubernetes operators, AWS Lambda, or testing
   contexts.

5. **MiNiFi is an edge topology.** The `minifi/` subproject embeds the same NAR + pipeline
   runtime in a resource-constrained footprint for edge data collection, demonstrating the
   architecture's portability.
