# auth (Supabase Auth) — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/supabase/auth
**Classification:** Layered + Modular Monolith
**Confidence:** 0.92

## Summary

Supabase Auth (formerly GoTrue) is a focused authentication and identity management service implemented as a single Go binary with a clear three-layer architecture (HTTP API, domain models, data access) and cohesive internal module boundaries enforced by Go's `internal/` package convention. It is deployed as a standalone service within the Supabase platform ecosystem, but internally it is a layered modular monolith with no event bus, no plugin runtime, and no inter-service decomposition.

## Evidence

### Entry Point and Deployment

`main.go` is a single entry point. `cmd/serve_cmd.go` starts one HTTP server, one background `apiworker`, and one optional config reloader — all in the same process. The `Dockerfile` produces one container image (`auth` binary). This is unambiguously a single deployable unit.

### Layered Architecture Evidence

**Layer 1 — HTTP/Presentation (`internal/api/`):** All HTTP routing via go-chi, middleware chain (CORS, request ID, logging, rate limiting, body limit, timeout), and individual handlers split by feature (`signup.go`, `token.go`, `verify.go`, `mfa.go`, `saml.go`, `web3.go`, `external.go`, etc.). The `API` struct wires together all dependencies at construction time.

**Layer 2 — Domain/Business Logic (`internal/models/`, `internal/tokens/`, `internal/hooks/`):** Domain entities (`User`, `Session`, `Identity`, `Factor`, `RefreshToken`, `AuditLogEntry`, `OAuthClient`, `FlowState`, etc.) with business methods. The `tokens.Service` handles JWT minting with hook dispatch for custom claims. The hooks manager (`v0hooks.Manager`) dispatches extensibility points synchronously within the request path.

**Layer 3 — Data Access (`internal/storage/`):** `Connection` wraps gobuffalo/pop ORM providing transaction management, connection pooling, and OpenTelemetry-instrumented SQL execution. Models use pop struct tags and call storage directly — no repository interface abstraction layer.

**Cross-cutting Infrastructure:** `internal/conf/` (envconfig-based flat configuration), `internal/observability/` (OpenTelemetry traces + Prometheus metrics + structured logging), `internal/ratelimit/` (tollbooth-based IP and header rate limiting), `internal/crypto/` (bcrypt, TOTP, PKCE, refresh token generation).

### Modular Monolith Evidence

Go's `internal/` package restriction enforces module boundaries — no package outside the repo root can import these packages. Each package has a single stated responsibility. Interfaces are used at key seams:
- `mailer.Mailer` interface with multiple implementations (`templatemailer`, `taskclient`, `noopclient`, `mockclient`, `validateclient`)
- `mailer.Client` interface for the underlying transport
- `hooks` dispatchers are injected via the manager pattern

Despite this modularity, all packages share a single process, a single PostgreSQL database, and a single configuration root — the defining characteristics of a modular monolith rather than microservices.

### Extensibility Hooks (Not Microkernel)

The hook system (`internal/hooks/`) allows callers to configure external HTTP endpoints or Postgres functions for events like `SendSMS`, `SendEmail`, `CustomAccessToken`, `BeforeUserCreated`, and `AfterUserCreated`. These are synchronous in-request calls to external systems, not runtime-loaded plugins — closer to an extension point pattern than a microkernel plugin architecture.

### What Was Ruled Out

- **Not Event-Driven:** No message broker, no async event queue, no publish/subscribe. The `apitask` package provides lightweight in-process goroutine deferral for email sending within request lifetime only.
- **Not Hexagonal:** Domain models (`internal/models/`) directly reference `storage.Connection` and use ORM tags — no ports/adapters inversion. The mailer interface is an adapter but does not constitute a full hexagonal pattern.
- **Not Microservices:** One binary, one database, no network calls between internal components.
- **Not CQRS:** No read/write model separation, no event sourcing, no command bus.
- **Not DDD:** No bounded contexts, aggregates, or domain events as first-class constructs.

## Architecture Styles Identified

### Layered
The codebase has an explicit three-tier horizontal separation: HTTP presentation layer (`internal/api/`), domain/business layer (`internal/models/`, `internal/tokens/`, `internal/hooks/`), and data access layer (`internal/storage/`). Dependencies flow strictly downward. Cross-cutting infrastructure packages (`conf`, `observability`, `ratelimit`, `crypto`) serve all layers without violating the layer ordering.

### Modular Monolith
Go's `internal/` convention enforces package-level module isolation within a single deployable binary. Each `internal/` sub-package has a focused, named responsibility. Interfaces at module boundaries (the `Mailer` interface, the hooks dispatcher abstraction) allow independent testing and replacement while retaining single-process deployment.

## Quality Attributes

- **Security:** Multi-factor authentication (TOTP, WebAuthn/passkeys, phone OTP), SAML SSO, PKCE, rate limiting per-endpoint, HIBP bloom-filter-cached password breach checking, standard-webhooks HMAC signing for hook payloads, JWT with multiple key support (JWK rotation).
- **Observability:** Full OpenTelemetry trace and metric export (OTLP gRPC/HTTP, Prometheus), structured JSON logging via logrus, request ID propagation, per-endpoint metric counters.
- **Extensibility:** Hook system with HTTP and Postgres function dispatchers for seven lifecycle events; custom OAuth/OIDC provider support; hot configuration reloading via fsnotify or polling without restart.
- **Reliability:** Graceful shutdown with request draining, configurable connection pool limits with percentage-based DB connection budgeting, advisory-lock-protected index worker, Postgres transaction management via pop ORM.
- **Performance:** Request-scoped background task deferral for email sending (`apitask`), OIDC provider discovery caching, bloom filter for HIBP lookups, configurable rate limits per flow type.
- **Maintainability:** Feature-by-feature handler decomposition, embedded SQL migrations (`//go:embed migrations/*`), cobra CLI with subcommands for migration and serving, envconfig for 12-factor configuration.
- **Testability:** Extensive unit and integration tests alongside each package; mock/noop mailer implementations; override hooks for OIDC verifiers and clocks in tests.

## Classification Reasoning

Supabase Auth is confidently classified as **Layered + Modular Monolith** at 0.92 confidence. The layered classification is clear and unambiguous: three distinct horizontal tiers with downward dependency flow, middleware chains managing cross-cutting concerns, and no shared mutable state crossing layer boundaries. The modular monolith classification is equally clear: 16 internal packages with enforced boundaries, interface-based seams at key points, all deployed in a single binary against a single PostgreSQL database.

The 0.92 rather than higher confidence reflects two minor complicating factors: (1) the mailer subsystem uses an interface with multiple implementations in a pattern reminiscent of hexagonal adapters, and (2) the hook system invokes external HTTP services synchronously, which could be loosely read as service-based — but neither pattern rises to the level of a primary or secondary architectural style because they are localized mechanisms rather than organizing principles for the whole system.
