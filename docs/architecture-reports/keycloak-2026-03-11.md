# Architecture Report: Keycloak

**Date:** 2026-03-11
**Repository:** https://github.com/keycloak/keycloak
**Classification:** Microkernel, Modular Monolith
**Confidence:** 0.97
**Model:** claude-sonnet-4-6
**Method:** deep-analysis

---

## Summary

Keycloak is Red Hat's open-source identity and access management (IAM) platform implementing OAuth 2.0, OpenID Connect, and SAML. It is an unambiguous **Microkernel** architecture: the entire system is organized around a Service Provider Interface (SPI) mechanism where a minimal kernel (`server-spi`, `DefaultKeycloakSessionFactory`, `ProviderManager`) discovers and loads all functionality ‚Äî authentication flows, storage backends, cryptographic operations, protocol handling, federation, event listeners, themes ‚Äî via `java.util.ServiceLoader` at startup. The secondary style is **Modular Monolith**: despite the extensibility architecture, the entire codebase is compiled, versioned, and deployed as a single Quarkus-based server process with a single Dockerfile and one entrypoint (`kc.sh`).

## Architecture Styles Identified

### 1. Microkernel (Primary ‚Äî highest confidence)

The codebase is organized around a three-layer plugin kernel:

**Layer 1 ‚Äî The Kernel Contracts (`server-spi/`)**

The minimal kernel defines exactly three interfaces:

```java
// Provider.java ‚Äî every plugin must implement this
public interface Provider {
    void close();
}

// Spi.java ‚Äî defines a named extension point
public interface Spi {
    boolean isInternal();        // internal (no 3rd-party) or external (3rd-party allowed)
    String getName();
    Class<? extends Provider> getProviderClass();
    Class<? extends ProviderFactory> getProviderFactoryClass();
    default boolean isEnabled() { return true; }
}

// ProviderFactory.java ‚Äî manages plugin lifecycle
public interface ProviderFactory<T extends Provider> {
    T create(KeycloakSession session);       // per-request instantiation
    void init(Config.Scope config);          // called once at startup
    void postInit(KeycloakSessionFactory);   // called after all factories are initialized
    void close();                            // shutdown
    String getId();
    default Set<Class<? extends Provider>> dependsOn() { return Collections.emptySet(); }
}
```

The `dependsOn()` method on `ProviderFactory` is a topological-sort mechanism ensuring initialization order across plugins ‚Äî a hallmark of a mature plugin kernel.

**Layer 2 ‚Äî The Extension Points (`server-spi-private/`, 92+ Spi implementations)**

Every major capability in Keycloak is an SPI. A representative sample:

| SPI | What it extends |
|-----|----------------|
| `AuthenticatorSpi` | Authentication step in a login flow |
| `ClientAuthenticatorSpi` | OAuth client authentication method |
| `FormAuthenticatorSpi` | Browser-based form authentication |
| `RequiredActionSpi` | Post-login required action (e.g., set up OTP) |
| `UserStorageProviderSpi` | User storage backend (LDAP, DB, custom) |
| `LoginProtocolSpi` | Login protocol (OIDC, SAML, custom) |
| `IdentityProviderSpi` | Social/enterprise identity brokering |
| `EventListenerSpi` | Audit and event processing |
| `ThemeSpi` | UI theme provider |
| `SignatureSpi` / `HashSpi` / `CekManagementSpi` | Cryptographic operations |
| `ClusterSpi` | Cluster communication |
| `VaultSpi` | Secret vault integration |
| `TracingSpi` | Distributed tracing |
| `ClientModelMapperSpi` | REST representation mapping (even mappers are SPIs) |

The `isInternal()` flag on each SPI formally distinguishes between extension points reserved for internal implementations (`true`) and those explicitly designed for third-party JAR deployment (`false`). `UserStorageProviderSpi.isInternal() = false` is the primary third-party extension point that allows enterprise customers to drop in custom user stores.

**Layer 3 ‚Äî Plugin Discovery and Loading (`services/ProviderManager`, `DefaultKeycloakSessionFactory`)**

```java
// ProviderManager.java ‚Äî uses java.util.ServiceLoader to discover all ProviderLoaderFactory implementations
for (ProviderLoaderFactory f : ServiceLoader.load(ProviderLoaderFactory.class, getClass().getClassLoader())) {
    factories.add(f);
}

// DefaultKeycloakSessionFactory.init() ‚Äî discovers all SPIs and loads provider factories
ProviderManager pm = new ProviderManager(KeycloakDeploymentInfo.create().services(),
    getClass().getClassLoader(), Config.scope().getArray("providers"));
for (Spi spi : pm.loadSpis()) {
    if (spi.isEnabled()) {
        spis.add(spi);
    }
}
```

Providers register themselves via `META-INF/services/` files ‚Äî the standard Java `ServiceLoader` discovery mechanism. For example, `federation/ldap/src/main/resources/META-INF/services/org.keycloak.storage.UserStorageProviderFactory` registers the LDAP storage backend. 200 such files exist in the non-test codebase.

**Runtime provider resolution** occurs through the `DefaultKeycloakSession`, which holds a `Map<List<String>, Provider>` of lazily-instantiated provider instances, looked up by SPI type and provider ID from the session factory's registry. This is the kernel's per-request plugin dispatch mechanism.

### 2. Modular Monolith (Secondary ‚Äî strongly expressed)

Despite its deep extensibility, Keycloak is a single deployable server:

- **Single Dockerfile**: `quarkus/container/Dockerfile` builds one image. `ENTRYPOINT ["/opt/keycloak/bin/kc.sh"]` ‚Äî one process, one entrypoint.
- **Single Maven build**: 31 Maven modules all compiled and released as one version (`999.0.0-SNAPSHOT` in the root `pom.xml`).
- **Single HTTP server**: `KeycloakApplication` (JAX-RS) and `KeycloakMain` (Quarkus) start a single Quarkus server hosting all endpoints ‚Äî Admin REST API, OIDC endpoints, Account Console, Admin Console, and SAML ‚Äî in one JVM.
- **Shared in-process state**: `KeycloakSession` is a request-scoped object that accesses all providers in-process. There is no network hop between the auth engine, storage, and protocol layers.
- **Module interdependencies are version-locked**: The modules (`server-spi`, `services`, `model/jpa`, `federation/ldap`, etc.) are tightly coupled by version ‚Äî they are not independently deployable services.

The Quarkus `deployment/` module and `KeycloakProcessor` perform build-time processing (Quarkus augmentation) to wire providers into the final application, further cementing the monolithic deployment model.

### Styles Rejected

| Style | Reason for Rejection |
|-------|---------------------|
| **Microservices** | Single binary, one Dockerfile, one process. No service mesh, no independent service APIs between subsystems. The Kubernetes Operator is a separate process, but it manages Keycloak ‚Äî it is not a Keycloak subsystem. |
| **Layered** | Layering exists within the module structure (SPI layer ‚Üí service layer ‚Üí REST layer), but this is subordinate to the plugin pattern. The primary organizational principle is the SPI extension point, not unidirectional layer dependencies. |
| **Event-Driven** | Events (`EventListenerSpi`) exist for audit and notifications, but they are a plugin capability ‚Äî not the primary communication mechanism between components. All inter-module calls are direct method calls through `KeycloakSession`. |
| **Hexagonal** | There is no ports-and-adapters boundary protecting a domain core. The SPI pattern is plugin-based ‚Äî providers plug into a named registry ‚Äî not adapter-based where infrastructure adapts to a domain port. |
| **Service-Based** | No set of 4‚Äì12 coarse-grained, independently deployed services sharing a database. This is one server with many internal providers. |
| **Pipeline** | There is no pipeline topology. Authentication flows are stateful graphs, not linear pipelines with typed stage contracts. |

## Classification Reasoning

Keycloak is the canonical textbook example of a **Microkernel** architecture applied to an enterprise security platform. The evidence is structural, not inferred:

1. **Three-interface kernel contract** (`Provider`, `ProviderFactory`, `Spi`) ‚Äî the smallest possible stable API surface for a plugin system.
2. **92 named extension points** in `server-spi-private/` ‚Äî every capability from authentication to cryptography to UI themes is an extension point, not hardcoded logic.
3. **`isInternal()` stability tiering** ‚Äî the kernel formally distinguishes between internal-only and externally extensible SPIs, demonstrating intentional API management for a third-party ecosystem.
4. **`java.util.ServiceLoader` discovery** ‚Äî providers are discovered at runtime from 200 `META-INF/services/` registrations without the kernel knowing about implementations at compile time.
5. **`dependsOn()` topological ordering** ‚Äî the kernel manages inter-plugin initialization order, a feature that only exists in mature plugin systems with cross-cutting concerns.
6. **Community extensibility in practice** ‚Äî Keycloak's documentation explicitly targets custom provider development as a first-class use case, confirmed by the `UserStorageProviderSpi.isInternal() = false` contract and the LDAP/Kerberos/SSSD federation implementations serving as reference implementations.

The **Modular Monolith** secondary style is equally concrete: the single `ENTRYPOINT ["/opt/keycloak/bin/kc.sh"]` in the Dockerfile, the unified Maven version, and the in-process `KeycloakSession` provider dispatch all confirm that the plugin system operates entirely within a single JVM process ‚Äî not across service boundaries.

The previous classification (Microkernel + Layered) was correct on the primary style. This analysis confirms Microkernel as primary and revises the secondary from Layered to **Modular Monolith**, because the module structure (31 Maven modules, single version, single deployment) is a more precise architectural descriptor than "layered" ‚Äî which exists within individual modules but is not the dominant organizing principle of the overall system.

**Confidence: 0.97** ‚Äî the highest warranted. Both styles are expressed explicitly in source code, build configuration, and deployment artifacts. The only remaining uncertainty is marginal: whether to include "Layered" as a tertiary qualifier, which is omitted because it describes intra-module organization rather than the system-level topology.
