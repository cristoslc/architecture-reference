## Style Rationales

**Layered (primary, 0.88):** Maybe is a Rails 7 monolith organized in strict horizontal layers with deliberate vertical decomposition inside each layer. The presentation layer is composed of ERB views, ViewComponent components (`app/components/`), and Hotwire/Stimulus controllers (`app/javascript/controllers/`) for progressive enhancement — there is no separate frontend SPA. The application layer is split between Rails controllers (`app/controllers/`) handling HTTP and background jobs (`app/jobs/`) dispatched via Sidekiq for async work. Controllers delegate to model-layer objects rather than performing business logic directly. The domain/business layer centers on ActiveRecord models organized into named subdomains — `Account::Syncer`, `Balance::Materializer`, `Balance::ForwardCalculator`, `Balance::ReverseCalculator`, `Rule::Registry`, `Rule::ActionExecutor`, `Rule::ConditionFilter`, `PlaidItem::Importer`, `PlaidItem::WebhookProcessor` — each encapsulating a discrete responsibility as a plain Ruby object within its model namespace. The infrastructure layer is abstracted behind the `Provider` hierarchy (`Provider::Plaid`, `Provider::Openai`, `Provider::Synth`, `Provider::Stripe`, `Provider::Github`), each returning a `Provider::Response` value object and selected via `Provider::Registry`. All external integrations are channeled through this registry rather than called directly from controllers or models.

**Modular Monolith (secondary, 0.80):** Although the application deploys as a single Ruby process (confirmed by `Procfile.dev` running `web` and `worker` from the same image, and a single `Dockerfile`), the codebase is internally structured around cohesive domain clusters rather than a single flat namespace. Each major domain concept has its own model directory with sub-objects: `app/models/account/`, `app/models/balance/`, `app/models/assistant/`, `app/models/plaid_item/`, `app/models/rule/`, `app/models/transaction/`, `app/models/trade/`, `app/models/holding/`, `app/models/family/`, etc. These clusters do not share internal files across boundaries — `Account::Syncer` only coordinates `Account::MarketDataImporter` and `Balance::Materializer`. The `compose.example.yml` runs exactly two processes from the same container image: a web process and a Sidekiq worker. A single shared PostgreSQL database is used for all domains.

## Ruled-Out Styles

**Microservices:** The entire application runs in one Ruby process, backed by one PostgreSQL database. No inter-service communication, no independent deployment per domain, no service mesh or service registry.

**Event-Driven (primary):** Sidekiq jobs and ActiveCable channels provide async processing, but these are implementation details of the layered architecture's async tier, not a primary integration mechanism between autonomous components. No message broker (Kafka, RabbitMQ) and no event store.

**Hexagonal Architecture:** The provider abstraction does create port-like interfaces, but controllers interact directly with ActiveRecord models (Active Record pattern), not through domain ports. There are no explicit inbound/outbound port interfaces; the layering is Rails-conventional rather than hexagonally structured.

**CQRS:** All reads and writes go through the same ActiveRecord models against the same database schema. Balances are pre-computed via `Balance::Materializer` as a materialized-view optimization, not a CQRS command/query separation.

**Serverless:** Packaged as a long-running Puma/Sidekiq process; explicitly designed for Docker deployment.
