# Architecture Report: ngx-admin

**Date:** 2026-03-11
**Repository:** https://github.com/akveo/ngx-admin
**Classification:** Layered
**Confidence:** 0.90
**Model:** claude-sonnet-4-6

---

## Summary

ngx-admin is an Angular 15 admin dashboard template produced by Akveo, serving as a reference implementation and starter kit for Angular-based single-page applications. Its dominant architectural style is **Layered**: the `src/app/` directory is partitioned into three well-defined horizontal layers — `@core` (infrastructure/data), `@theme` (presentation/UI), and `pages` (feature/application layer) — with a strict downward dependency flow. Within the data sub-layer, the codebase applies a port/adapter separation (abstract data classes backed by injectable implementations), which qualifies as a **hexagonal design approach** but does not alter the overall topology.

## Styles Identified

### 1. Layered (Primary — strong evidence)

**Evidence:**

The codebase exhibits canonical three-tier layering:

**Core Layer (`@core/`)** — infrastructure and data abstraction:
- `src/app/@core/data/` defines abstract TypeScript classes acting as data contracts (e.g. `export abstract class UserData { abstract getUsers(): Observable<User[]>; }`). These classes are the stable API surface.
- `src/app/@core/mock/` provides concrete implementations that return in-process mock data.
- `src/app/@core/utils/` holds horizontal services: `AnalyticsService` (Google Analytics integration), `LayoutService` (RxJS Subject-based layout resize events), `StateService` (BehaviorSubject-based sidebar/layout state), `SeoService`, and `PlayerService`.
- `core.module.ts` acts as the DI composition root, binding every abstract data class to its mock implementation via `{ provide: UserData, useClass: UserService }` across 19 data services. `CoreModule` is guarded by `throwIfAlreadyLoaded` to enforce singleton semantics.

**Theme Layer (`@theme/`)** — shared UI framework:
- Reusable layout shells (`OneColumnLayoutComponent`, `TwoColumnsLayoutComponent`, `ThreeColumnsLayoutComponent`) used by all feature pages.
- Common components (`HeaderComponent`, `FooterComponent`, `SearchInputComponent`) shared across all feature modules.
- Angular pipes (`CapitalizePipe`, `PluralPipe`, `RoundPipe`, `TimingPipe`, `NumberWithCommasPipe`) used across pages.
- Four theme definitions (default, cosmic, corporate, dark) wired into `NbThemeModule`.
- `theme.module.ts` exports all components and pipes so feature modules only need to import `ThemeModule`.

**Pages Layer (`pages/`)** — feature modules consuming both layers:
- Each feature directory (dashboard, e-commerce, charts, forms, maps, etc.) is a self-contained `NgModule` that imports `ThemeModule` and injects `@core` services.
- Feature modules are lazy-loaded via Angular's `loadChildren` pattern in `pages-routing.module.ts`, reducing initial bundle size.
- The pages layer has no downward knowledge of `@theme` internals and no direct reference to mock service implementations — it depends only on abstract data classes from `@core/data/`.

**Dependency direction is strictly downward:** `pages` → `@theme` → `@core` → (Angular/Nebular libraries). The `module-import-guard.ts` enforces that `CoreModule` is only imported once (at root), preventing layer bypass.

### What This Is Not

**Not Modular Monolith:** Angular's `NgModule` system is an organizational mechanism, not an enforcement of independently deployable or bounded module contracts. Feature modules freely share the `@theme` module and all `@core` services; there are no explicit inter-module contracts beyond Angular's DI.

**Not Microservices or Service-Based:** The entire application is a single SPA compiled to a single `dist/` output. There are no network service boundaries, no separate deployable services.

**Not Event-Driven:** Communication is direct Angular service injection and Observable subscriptions on shared services. `LayoutService` emits layout size events, and `StateService` exposes BehaviorSubjects — but these are in-process reactive utilities, not an event-driven topology.

**Not Microkernel:** No runtime plugin registry, no extension points, no dynamically loaded plug-in mechanism beyond Angular's lazy-loaded routes.

**Not CQRS or DDD:** No aggregates, value objects, bounded contexts, domain events, or read/write model separation. The data layer is a thin mock service layer, not a domain model.

## Hexagonal Design Approach (Qualifier)

Within the `@core` layer, the codebase deliberately applies a port/adapter pattern:

- **Ports:** Abstract TypeScript classes in `@core/data/` (`UserData`, `ElectricityData`, `OrdersChartData`, etc.) declare the interface that consumers depend on.
- **Adapters:** Concrete classes in `@core/mock/` (`UserService`, `ElectricityService`, `OrdersChartService`, etc.) implement the ports with mock data.
- **Binding:** `core.module.ts` binds ports to adapters via Angular DI (`{ provide: UserData, useClass: UserService }`).

This design explicitly enables swapping the mock adapter for a real HTTP adapter without changing any consumer. This is a meaningful structural investment in the hexagonal design approach, but it is confined to the data access layer and does not reconfigure the overall layered topology.

## Quality Attributes

| Attribute | Justification |
|---|---|
| **Maintainability** | Clear layer separation with `@core`, `@theme`, and `pages` makes the codebase easy to navigate and extend. The `ThemeModule` single-import pattern reduces boilerplate across feature modules. |
| **Testability** | The port/adapter separation in `@core` allows replacing mock services with HTTP services or test doubles without touching feature components. Angular's DI is inherently testable. |
| **Extensibility** | Lazy-loaded feature modules can be added without touching the root module. New themes are registered by extending the Nebular theme array. |
| **Modularity** | Feature modules are self-contained Angular modules; the `@theme` and `@core` separation avoids cross-feature coupling at the component level. |
| **Developer Experience** | Serves as a reference template demonstrating Angular, Nebular, SCSS theming, lazy loading, and role-based access control in a coherent single application. |

## Classification Reasoning

ngx-admin's architecture is definitively **Layered** as its primary and sole topology-defining style. The evidence is unambiguous:

1. Three explicit horizontal layers (`@core`, `@theme`, `pages`) with a documented downward dependency constraint.
2. The `module-import-guard.ts` singleton enforcement and `forRoot()` pattern on `CoreModule` and `ThemeModule` are classic layered application infrastructure patterns.
3. A single deployable SPA unit with no network boundaries between components.
4. Lazy-loaded Angular routing is a performance technique, not an architectural style change.

The hexagonal design approach applied within `@core` (19 abstract port classes bound to adapter implementations) is a meaningful qualifier but does not change the topology. The system remains a three-layer client application throughout.

Confidence: **0.90** — the layered structure is unambiguous; the slight uncertainty reflects the fact that the hexagonal data-layer pattern is intentional and structural, though it does not override the dominant topology.
