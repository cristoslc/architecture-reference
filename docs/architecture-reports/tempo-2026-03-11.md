## Executive Summary

Grafana Tempo is a high-volume, minimal-dependency distributed tracing backend. Its primary architecture style is **Modular Monolith**: all components compile into a single binary and are activated via a `-target` flag at runtime. The same binary can run as a true single-process deployment (`-target=all`) or be deployed as independent scaled services in distributed mode. A secondary style of **Microservices** applies in distributed deployment: each functional module communicates exclusively over gRPC/HTTP, uses consistent-hash rings for service discovery and sharding, and can scale independently. A third style, **Pipeline**, is present in the query path: the Query Frontend routes requests through a structured middleware pipeline of sharders, retriers, and combiners before dispatching to Queriers. Kafka-based ingest (`pkg/ingest`, `modules/blockbuilder`, `modules/livestore`) introduces event-driven dataflow as an optional but first-class architectural path.

## Component Breakdown

### Distributor (`modules/distributor`)
Accepts spans in OTLP, Jaeger, Zipkin formats via the OpenTelemetry Collector receiver shim (`receiver/shim.go`). Computes a consistent hash of the `traceID` and forwards batches to the correct Ingester(s) via gRPC using a read ring (`dskit/ring`). Also fans out span data to the Metrics Generator ring. Optionally writes to Kafka (`pkg/ingest`) when the ingest path is enabled.

### Ingester (`modules/ingester`)
Maintains an in-memory map of active traces per tenant. Writes incoming spans to a WAL immediately, then batches them into Parquet blocks (via `tempodb/encoding/vparquetX`) and flushes them to object storage. Participates in a consistent-hash ring for ownership. Supports configurable replication factor (RF) for redundancy. Also responds to trace-by-ID queries for recent data not yet in object storage.

### Block Builder (`modules/blockbuilder`)
Alternative ingest path: consumes raw span records from Kafka partitions (using `twmb/franz-go`), deduplicates spans, and writes Parquet blocks directly to object storage. Coordinates partition ownership via a partition ring. Designed to decouple ingestion throughput from write-ahead durability.

### Live Store (`modules/livestore`)
Also a Kafka consumer. Maintains a recent in-memory trace store backed by a WAL, queryable by the Querier for sub-second-latency reads of the most recent data. Coordinates partition ownership via its own partition ring. Provides the "live" tier in a tiered storage model (live store → block builder → compacted object storage).

### Query Frontend (`modules/frontend`)
Entry point for all read traffic. Implements a layered **pipeline** of middleware (`modules/frontend/pipeline/`):

- **Sharders** (e.g., `search_sharder.go`, `traceid_sharder.go`, `metrics_query_range_sharder.go`): split the block ID space or time range into parallel sub-requests.
- **Async/sync middleware**: deny-list filtering, tenant validation, weight balancing, cache lookup/fill, retry, response code adjustment.
- **Combiners** (`modules/frontend/combiner/`): merge results from parallel Querier responses.

Queriers connect to the Query Frontend via a streaming gRPC connection to pull and process sharded work items.

### Querier (`modules/querier`)
Executes sharded search jobs against both the object storage backend (`tempodb.Reader`) and recent data from the Live Store. Instantiates the TraceQL engine (`pkg/traceql`) for query evaluation. Exposes an internal HTTP endpoint consumed only by the Query Frontend.

### Metrics Generator (`modules/generator`)
Optional component. Receives span data (via gRPC push from Distributor, or from Kafka). Runs processors (`spanmetrics`, `servicegraphs`, `hostinfo`) to derive Prometheus-compatible metrics from trace data. Writes metrics to a configurable remote write endpoint. Uses its own ring for per-tenant instance routing.

### Backend Scheduler + Worker (`modules/backendscheduler`, `modules/backendworker`)
Next-generation compaction/retention system. The Scheduler maintains a job queue (compaction, retention); Workers connect via gRPC (`Next`/`UpdateJob` RPCs) to fetch and execute jobs. Intended to replace the legacy compactor with a more deterministic and deduplicated approach.

### TempoDB (`tempodb/`)
The core storage engine, used by Ingesters, Queriers, Block Builders, and Backend Workers. Key sub-systems:
- **WAL** (`tempodb/wal/`): write-ahead log for crash recovery.
- **Encoding** (`tempodb/encoding/vparquetX/`): Apache Parquet block format with bloom filters, column-based trace storage, and versioned schema.
- **Backend** (`tempodb/backend/`): pluggable object storage interface with implementations for S3, GCS, Azure, and local disk. Cache layer wraps backends with Memcached/Redis.
- **Blocklist** (`tempodb/blocklist/`): in-memory tracking of all block metadata per tenant, used by sharders.

### TraceQL Engine (`pkg/traceql`)
A domain-specific query language for traces. Full AST-based parser and executor. Supports structural pipeline queries (`{ ... } | select(...)`), metric queries (rate, count-over-time, histogram), and span filtering by attributes, duration, status. The engine implements `FetchSpans` via the Parquet encoding layer, making query execution column-push-down aware.

## Deployment Modes

Tempo ships a single binary (`cmd/tempo`) with a `-target` flag that selects which modules to activate:

- **`-target=all`** (SingleBinary): All modules run in-process. Default config. Suitable for development and small deployments.
- **Distributed**: Separate process instances per target (`distributor`, `query-frontend`, `querier`, `ingester`, `metrics-generator`, `block-builder`, `live-store`, `backend-scheduler`, `backend-worker`). Docker Compose and Helm chart examples exist in `example/` and `operations/helm/`.

The `dskit/modules` framework enforces declared inter-module dependency ordering at startup time regardless of deployment mode.

## Key Architectural Decisions

1. **Single binary, multiple deployment modes**: Inherited from the Cortex/Loki lineage. Avoids per-service release complexity while preserving horizontal scalability. The `dskit/modules.Manager` pattern is the implementation mechanism.

2. **Object storage as primary backend**: Tempo requires only object storage (S3/GCS/Azure) for persistence. No databases. Blocks are immutable once written. This eliminates stateful infrastructure at the cost of eventual query freshness.

3. **Consistent-hash ring for stateful routing**: Trace spans for a given `traceID` are always sent to the same Ingesters, enabling in-memory trace assembly without cross-shard joins during ingestion.

4. **Apache Parquet as block format**: Since Tempo 2.0. Column-oriented storage enables TraceQL query push-down at the column level (bloom filters, dictionary encoding). Three versions (`vparquet3`–`vparquet5`) are active in the codebase, with versioned encoding abstracted behind an interface.

5. **Optional Kafka ingest path (block-builder + live-store)**: Adds durability via log persistence. Decouples ingestion throughput from storage write latency. Represents a transition toward a more event-driven, disaggregated architecture.

6. **Query pipeline middleware**: The Query Frontend uses a composable request/response pipeline inspired by HTTP middleware chains. Middleware components are independently testable and reorderable.

7. **TraceQL as a first-class query language**: Enables structural and metric queries against trace data, comparable to LogQL for logs and PromQL for metrics.

| **Cost efficiency** | High. Object storage only; no stateful databases. |

| **Flexibility** | High. Kafka ingest optional; compactor replaceable by scheduler+worker. |

## Architecture Style Conclusion

**Primary:** Modular Monolith — one binary, one codebase, module boundaries enforced by `dskit/modules`, deployable as a single process.

**Secondary:** Microservices — in distributed deployment, each module runs as an independent horizontally-scaled process communicating exclusively over gRPC/HTTP with no shared in-process state.

**Tertiary:** Pipeline — the Query Frontend implements a composable request processing pipeline with distinct middleware stages (sharding, caching, retry, combination).

The Kafka ingest path introduces Event-Driven characteristics but remains optional and does not define the primary architectural style.
