## Executive Summary

Vendure is an open-source, headless commerce framework built on NestJS/TypeScript whose defining architectural characteristic is a **Microkernel (Plugin)** system. A minimal core application is designed from first principles to be extended through a formal plugin contract — the `@VendurePlugin` decorator — which allows external plugins to inject GraphQL schema extensions, database entities, configuration modifications, REST controllers, and UI components, all without modifying the kernel. Within the core, a strict three-layer organisation (API → Service → Data) provides the **Layered** secondary style. A first-class **EventBus** with RxJS observables and a `JobQueue` with a pluggable strategy backend provide **Event-Driven** characteristics used primarily as the communication mechanism between the kernel and its plugins. Vendure deploys as a single Node.js process (with an optional separate worker process for background job execution) against a single relational database via TypeORM. Despite its rich extensibility, it remains a monolithic deployable — not Microservices, not a Modular Monolith with enforced bounded-context isolation.

## Architecture Styles Identified

### Primary: Microkernel (Plugin Architecture)

**Confidence: 0.93**

The dominant organising principle of Vendure is that the core framework is a minimal, stable kernel designed to be extended through a formal plug-in contract. This is not merely a configuration mechanism — it is the stated architectural purpose of the project.

**Evidence:**

**Plugin Contract — `@VendurePlugin` decorator** (`packages/core/src/plugin/vendure-plugin.ts`):

The `VendurePlugin` decorator wraps a NestJS `@Module` and adds Vendure-specific extension metadata:

```typescript
export interface VendurePluginMetadata extends ModuleMetadata {
    configuration?: PluginConfigurationFn;    // mutates VendureConfig before bootstrap
    shopApiExtensions?: APIExtensionDefinition;  // extends Shop GraphQL schema
    adminApiExtensions?: APIExtensionDefinition; // extends Admin GraphQL schema
    entities?: Array<Type<any>> | (() => Array<Type<any>>);  // adds TypeORM entities
    compatibility?: string;  // semver check against Vendure core version
}
```

**Plugin Loading — `PluginModule.forRoot()`** (`packages/core/src/plugin/plugin.module.ts`):

At bootstrap, `getConfig().plugins` is spread into the root module's imports array. The PluginModule collects and re-exports all providers from all plugins so that plugin-provided services are available throughout the application via NestJS DI. This is a runtime dynamic load, not a static compile-time import.

**Bootstrap pipeline** (`packages/core/src/bootstrap.ts`):

The bootstrap sequence enforces the microkernel contract:
1. User configuration is collected (`preBootstrapConfig()`).
2. Plugin-defined entities are merged into the entity registry (with name conflict detection).
3. Plugin `configuration` functions run sequentially (`runPluginConfigurations()`).
4. The NestJS AppModule is created with PluginModule loaded.
5. Plugin compatibility is validated against semver.

**Extent of the plugin surface** (`packages/core/src/config/vendure-config.ts`):

The `VendureConfig` interface exposes 40+ strategy interfaces as extension points. Any of these can be replaced by a plugin-provided implementation:

*Authentication:* `AuthenticationStrategy`, `PasswordHashingStrategy`, `PasswordValidationStrategy`, `VerificationTokenStrategy`

*Assets:* `AssetNamingStrategy`, `AssetStorageStrategy`, `AssetPreviewStrategy`, `AssetImportStrategy`

*Catalog:* `ProductVariantPriceSelectionStrategy`, `ProductVariantPriceCalculationStrategy`, `StockDisplayStrategy`, `StockLocationStrategy`, `CollectionFilter`

*Orders:* `OrderItemPriceCalculationStrategy`, `StockAllocationStrategy`, `OrderMergeStrategy`, `OrderCodeStrategy`, `OrderByCodeAccessStrategy`, `ActiveOrderStrategy`, `OrderSellerStrategy`, `GuestCheckoutStrategy`

*Payment & Fulfillment:* `PaymentMethodHandler`, `PaymentProcess`, `RefundProcess`, `FulfillmentHandler`, `FulfillmentProcess`

*Shipping:* `ShippingEligibilityChecker`, `ShippingCalculator`, `ShippingLineAssignmentStrategy`

*Promotions & Tax:* `PromotionCondition`, `PromotionAction`, `TaxZoneStrategy`, `TaxLineCalculationStrategy`

*Infrastructure:* `JobQueueStrategy`, `JobBufferStorageStrategy`, `SessionCacheStrategy`, `CacheStrategy`, `HealthCheckStrategy`, `ErrorHandlerStrategy`, `SchedulerStrategy`, `EntityIdStrategy`, `MoneyStrategy`, `InstrumentationStrategy`

**Official first-party plugins demonstrating the pattern** (all under `packages/`):

| Plugin | Extends |
|---|---|
| `AssetServerPlugin` | Serves and transforms assets; implements `AssetStorageStrategy` |
| `EmailPlugin` | Subscribes to EventBus; dispatches transactional emails via JobQueue |
| `AdminUiPlugin` | Serves the Angular/React Admin UI |
| `ElasticsearchPlugin` | Replaces default search; implements `SearchStrategy` |
| `JobQueuePlugin` (BullMQ) | Replaces default in-memory queue with Redis/BullMQ |
| `PaymentsPlugin` | Stripe, Mollie, Braintree, PayPal `PaymentMethodHandler` implementations |
| `HardenPlugin` | Cross-cutting security hardening applied at bootstrap |
| `SentryPlugin` | Implements `ErrorHandlerStrategy` and `InstrumentationStrategy` |
| `StellatePlugin` | CDN cache purging via EventBus subscriptions |

This is textbook Microkernel: a stable kernel (`packages/core`) with externalized, interchangeable plug-in components providing all non-trivial functionality.

### Secondary: Layered

**Confidence: 0.88**

Within `packages/core/src/`, there is clear horizontal layering with unidirectional dependency flow.

**Layer structure:**

**Layer 1 — API Layer** (`packages/core/src/api/`):
- GraphQL resolvers organised into `admin/` and `shop/` subdirectories (20+ resolver files each)
- Schema definitions (`api/schema/`)
- Guards, interceptors (`IdInterceptor`, `CustomFieldProcessingInterceptor`, `TranslateErrorResultInterceptor`), and middleware
- `AuthGuard` for request authorization
- Explicitly delegates all business logic to the ServiceModule: "requests come in, are parsed and then handed over to the ServiceModule classes which take care of the business logic"
- The `api/api.module.ts` imports `ServiceModule` and `ConnectionModule` — never the reverse

**Layer 2 — Service Layer** (`packages/core/src/service/`):
- 20+ domain service classes: `OrderService`, `ProductService`, `CustomerService`, `PaymentService`, `FulfillmentService`, `ShippingMethodService`, etc.
- Specialized helpers: `OrderCalculator`, `OrderStateMachine`, `OrderModifier`, `PromotionService`
- Services coordinate repositories, event publishing, and strategy execution
- Transaction Script style: each public method is a complete business transaction
- `ErrorResultUnion<T, Order>` return types replace exception-based control flow for domain errors

**Layer 3 — Data Layer** (`packages/core/src/entity/`, `packages/core/src/connection/`):
- 60+ TypeORM entity classes (`Order`, `OrderLine`, `Product`, `ProductVariant`, `Customer`, `Payment`, `Fulfillment`, `StockLevel`, `Channel`, `Seller`, etc.)
- `TransactionalConnection` wraps TypeORM, carrying transaction scope via `RequestContext`
- `withTransaction()` for explicit transactional boundaries in non-request contexts (JobQueue, scripts)
- Repository access gated through `TransactionalConnection.getRepository(ctx, EntityType)` — raw TypeORM connection not exposed to service layer

**Layer enforcement:** The `api.module.ts` imports `ServiceModule` and `ConnectionModule`. Services import `TransactionalConnection` directly. No upward dependency allowed. This is enforced by TypeScript module boundaries, not a separate enforcement tool — but the pattern is consistent throughout the codebase.

### Secondary: Event-Driven

**Confidence: 0.78**

Vendure contains a substantial event infrastructure used for plugin integration and asynchronous decoupling. It is a secondary style because the event bus is an extension mechanism layered on top of the plugin architecture rather than the primary integration topology.

**EventBus** (`packages/core/src/event-bus/event-bus.ts`):
- RxJS `Subject<VendureEvent>` as the internal stream
- `publish(event)` emits to all subscribers
- `ofType(EventType)` filters stream by class type
- **Transaction-awareness**: subscribers are only called after any active database transaction completes, preventing dirty read issues
- **Blocking handlers**: synchronous handlers that execute within the publishing transaction with dependency-ordered sequencing and circular-dependency detection

**58 named domain events** (`packages/core/src/event-bus/events/`):
`OrderEvent`, `OrderPlacedEvent`, `OrderStateTransitionEvent`, `OrderLineEvent`, `PaymentStateTransitionEvent`, `FulfillmentEvent`, `FulfillmentStateTransitionEvent`, `ProductEvent`, `ProductVariantEvent`, `CustomerEvent`, `LoginEvent`, `LogoutEvent`, `AccountRegistrationEvent`, `PasswordResetEvent`, `StockMovementEvent`, and 43 others — every significant domain state change emits an event.

**Plugin EventBus integration (EmailPlugin example)** (`packages/email-plugin/src/plugin.ts`):

```typescript
for (const handler of EmailPlugin.options.handlers) {
    this.eventBus.ofType(handler.event).subscribe(event => {
        return this.handleEvent(handler, event);
    });
}
```

Plugins subscribe to domain events, translate them into side effects (email, SMS, webhook), and dispatch them to the JobQueue — a clean reactive pattern.

**JobQueue** (`packages/core/src/job-queue/job-queue.service.ts`):
- Strategy-backed: `JobQueueStrategy` interface allows in-memory (default), database, or BullMQ/Redis implementations
- `JobBufferStorageStrategy` for batching and deduplication before queue submission
- Optional separate worker process (`packages/core/src/worker/`) polls queues and processes jobs
- `SchedulerStrategy` for cron-based task execution (worker-process-only by design)
- BullMQ plugin (`packages/job-queue-plugin/src/bullmq/`) uses Redis with pub/sub for distributed job cancellation, enabling horizontal worker scaling

**Scope of event-driven influence:** The event bus coordinates plugin reactions to domain operations; it does not replace synchronous request/response as the primary flow. Order placement, product creation, and all commerce operations flow synchronously through the service layer and publish events as side-effect notifications. This places Event-Driven as a genuine but secondary characteristic.

## Patterns Observed (Qualifiers)

### Strategy Pattern (Pervasive)

Every major behavioural concern in Vendure is abstracted behind a strategy interface. The `VendureConfig` exposes 40+ strategy slots. This is not a DDD or Hexagonal pattern — it is a classic GoF Strategy applied uniformly as the kernel extension mechanism. The strategies are the "plug points" of the microkernel.

### Transaction Script (Service Layer)

Despite rich domain entities (TypeORM classes), the service layer uses Transaction Script style: `OrderService.addItemToOrder()` coordinates query, mutation, validation, calculation, and event publishing in a single procedure. Domain logic lives in service methods and helper classes (`OrderCalculator`, `OrderStateMachine`), not in entity methods.

### State Machine (Order/Payment/Fulfillment)

`OrderStateMachine`, `PaymentStateMachine`, and `FulfillmentStateMachine` enforce valid state transitions for the core commerce workflows. State machines are configured via strategy interfaces (`OrderProcess`, `PaymentProcess`, `FulfillmentProcess`), allowing plugins to add custom states.

### Dual API (Admin / Shop GraphQL)

Two completely separate GraphQL endpoints (`/admin-api`, `/shop-api`) with distinct schema, resolvers, and guard configurations. Both share the service layer but have no schema overlap. This is a clean API-layer concern separation, not a microservices boundary.

### Server / Worker Split

`bootstrap()` and `bootstrapWorker()` create two distinct NestJS application instances from the same codebase. The server handles HTTP/GraphQL; the worker handles job queues, scheduled tasks, and health monitoring. Both connect to the same database. This enables independent horizontal scaling of compute-intensive background work without adding a service boundary.

## Deployment Model

Vendure is a single deployable unit: one Node.js process per role (server, worker). Both roles share:
- One relational database (PostgreSQL recommended, MySQL and SQLite supported via TypeORM)
- One configuration object
- All plugins loaded at bootstrap

Optional horizontal scaling patterns:
- Multiple server replicas behind a load balancer (stateless, session via DB)
- Multiple worker replicas connecting to a shared Redis-backed BullMQ queue
- Redis-backed `CacheStrategy` for cross-instance session caching

There is no service mesh, no inter-service HTTP communication, and no per-module independent deployment. This is explicitly **not** Microservices.

## What This Is Not

**Not Microservices:** Single deployable Node.js process per role. No inter-service HTTP calls. No independent deployment per domain. The server/worker split is an operational scaling optimisation, not a service boundary.

**Not Modular Monolith:** There are no enforced bounded-context module boundaries isolating domain concerns from each other. The core is a cohesive `ServiceModule` containing all services. Plugin packages are external extensions, not bounded-context modules. Compare with Shopware (PHP bundles with per-domain namespaces and formal product-area annotations) or Medusa (35 DI-container-wired bounded-context modules with joiner federation).

**Not Hexagonal/Ports-and-Adapters:** The strategy interfaces resemble ports but are not structured as inbound/outbound ports. There is no formal separation of application core from infrastructure adapters per Hexagonal conventions. Strategy slots are microkernel extension hooks, not hexagonal ports.

**Not DDD:** No bounded contexts, no aggregates, no ubiquitous language organisation. Services are named by entity type (`OrderService`, `ProductService`), not by domain subdomain. Entities are anemic TypeORM classes; business logic lives in Transaction Script services. Domain events exist but serve the plugin bus, not aggregate lifecycle management.

## Classification Reasoning

Vendure is classified primarily as **Microkernel** because the entire framework is designed around a formal plugin contract. The `@VendurePlugin` decorator, `VendureConfig` strategy slots, `EventBus` subscription, and dynamic `PluginModule.forRoot()` loading are not incidental features — they are the stated architectural purpose of the project. The kernel (core) is intentionally minimal; every significant production feature (assets, email, payments, search, admin UI, job queuing) is implemented as a plugin using this contract. This is a stronger Microkernel signal than nopCommerce (where plugins are an extension to a fully-featured core) or Shopware (where plugins are one of two extension tracks alongside a modular core).

**Layered** is the secondary style because the API → Service → Data dependency ordering is rigorous and unidirectional within the core package. It is not the primary organisation principle — the plugin system is — but it is structurally present and architecturally meaningful.

**Event-Driven** is the tertiary style because the EventBus and JobQueue together form a genuine reactive integration layer between the kernel and plugins. With 58 named domain events and a full RxJS subscription model, the event system is load-bearing in production deployments (email, webhooks, search indexing, CDN invalidation all flow through it). It is secondary rather than primary because the synchronous request-response GraphQL API remains the primary interaction model; events are side-effect propagation, not the primary flow.

The classification confidence of **0.92** reflects very strong evidence for Microkernel (the plugin system is unambiguous and central), high confidence in Layered (rigorous layer discipline), and moderate confidence in Event-Driven (substantial but secondary to the plugin model).
