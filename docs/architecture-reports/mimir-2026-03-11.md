## Summary

Grafana Mimir is a horizontally scalable, multi-tenant, long-term storage backend for Prometheus metrics. Its architecture is a textbook microservices system: each named component (Distributor, Ingester, Query Frontend, Querier, Query Scheduler, Store Gateway, Compactor, Ruler, Alertmanager, Block Builder) is independently deployable from a single compiled binary via a `-target` flag. Components communicate over gRPC using explicit protobuf service contracts. Coordination uses consistent hash rings backed by memberlist gossip rather than a central coordinator. The preferred production architecture (v3.0+) adds a Kafka-based ingest pipeline as a durable decoupling layer between the write and read paths, introducing a strong event-driven secondary style.

## Architecture Classification

### Primary Style: Microservices (0.95)

Every component is an independent, horizontally scalable process with:

1. **Separate deployment identity.** `pkg/mimir/modules.go` defines ~40 named modules. The `-target` flag selects which to start. Production deployments run each as an independent Kubernetes Deployment via the `mimir-distributed` Helm chart.

2. **Explicit gRPC service contracts.** Each component that accepts inter-service calls defines a `.proto` file with a named `service`:
   - `pkg/ingester/client/ingester.proto` — `service Ingester`
   - `pkg/distributor/distributorpb/distributor.proto` — `service Distributor`
   - `pkg/storegateway/storegatewaypb/gateway.proto` — `service StoreGateway`
   - `pkg/scheduler/schedulerpb/scheduler.proto` — `service SchedulerForQuerier`, `service SchedulerForFrontend`
   - `pkg/frontend/v2/frontendv2pb/frontend.proto` — `service FrontendForQuerier`
   - `pkg/ruler/ruler.proto` — `service Ruler`
   - `pkg/alertmanager/alertmanagerpb/alertmanager.proto` — `service Alertmanager`
   - `pkg/blockbuilder/schedulerpb/scheduler.proto` — `service BlockBuilderScheduler`

3. **Decentralized service discovery via hash rings.** No central registry. Each component joins its own gossip-backed ring (Ingesters ring, Partitions ring, Store Gateway ring, Distributor ring, Querier ring, Query Scheduler ring). Components discover peers by watching the ring. When instances join or leave, only a bounded token range moves, ensuring consistent hashing without coordination downtime.

4. **Independently scalable failure domains.** The Distributor, Querier, and Query Frontend are stateless and scale horizontally without data migration. The Ingester is semi-stateful (in-memory + WAL) but participates in ring-managed replication. The Store Gateway replicates blocks across a replication factor of 3. Each component can be updated, restarted, or scaled without affecting others.

5. **Module dependency graph enforces component boundaries.** In `pkg/mimir/modules.go`, the `deps` map hard-codes which modules a given component depends on. For example, `QueryFrontend` depends on `{QueryFrontendTripperware, MemberlistKV, Vault}` — not on Ingester or Distributor directly. The Querier finds Ingesters through the ring, not via direct import.

6. **Single-binary, two modes.** Mimir compiles all components into one binary for operational simplicity, but documents two canonical deployment modes: monolithic (`-target=all`, for development) and microservices (each component separate, recommended for production). This is the same pattern used by Loki and Tempo in the Grafana ecosystem — the binary boundary is a build optimization, not an architectural constraint.

### Secondary Style: Event-Driven (0.82)

Starting with Mimir 3.0, the preferred "ingest storage" architecture interposes Apache Kafka (or a Kafka-compatible system such as WarpStream) between the write and read paths:

- **Distributors write to Kafka.** Each incoming remote-write request is sharded across Kafka partitions by consistent hash (`fnv32a`). A write is acknowledged to the client only after Kafka confirms persistence and replication.
- **Ingesters consume from Kafka.** Each ingester owns exactly one Kafka partition and continuously consumes from it. Writes never travel directly to ingesters; the distributor's write path ends at the Kafka broker.
- **Zone-aware replication is Kafka-native.** Multiple ingesters across different availability zones consume the same partition for HA on the read path. There is no ingester-side quorum write; Kafka handles replication.
- **Decoupled read and write paths.** Heavy query load on ingesters no longer disrupts live writes; the write path terminates at the broker. This is the primary motivation for the Kafka layer.
- **Partition lifecycle managed by ingesters.** Partitions have explicit `Pending -> Active -> Inactive` lifecycle transitions managed cooperatively by ingesters and the rollout operator. This is a finite-state machine driving consumer behavior, a characteristic of event-driven systems.
- **Block Builder as alternative consumer.** The Block Builder component is an independent Kafka consumer that builds TSDB blocks directly from Kafka without the in-memory ingester path — a pipeline processor in the event-driven sense.

The Kafka layer is optional (classic architecture uses direct gRPC writes to ingesters), but is the recommended production configuration.

### What Was Considered and Rejected

**Not Modular Monolith.** Although all components compile into one binary, they are explicitly designed and documented as separate services with independent scaling, separate deployment targets, and gRPC contracts. The documentation, Helm chart, and module dependency system all enforce the microservices mental model.

**Not Pipeline (primary).** The query path has a clear pipeline shape (Frontend -> Scheduler -> Querier -> Store Gateway), and the ingest storage write path is a producer-consumer pipeline. However, the system's primary architectural characteristic is the independently scalable, ring-coordinated service mesh, not a sequential processing pipeline.

**Not CQRS (standalone).** Mimir has distinct write and read paths, but this is a consequence of its domain (time-series storage for Prometheus) rather than an application of the CQRS pattern as an explicit design choice. Write models (remote-write protobufs) and read models (PromQL query engine) differ naturally, not through explicit command/query segregation infrastructure.

## Quality Attributes Evidence

**Scalability:** Distributor, Querier, and Query Frontend are stateless and scale horizontally. Ingesters scale by adding partitions in the Kafka ring. The internal design goal is 1 billion active time series per cluster.

**High Availability:** Each component uses ring-based health checking and zone-aware replication. Mimir can sustain failures, rolling restarts, and upgrades with zero downtime. The ingest storage architecture removes quorum dependency from the write path (Kafka handles it).

**Multi-tenancy:** Every API request carries a tenant ID header. Limits, shuffle sharding, and per-tenant overrides are enforced at the Distributor. Tenants are isolated in separate TSDB blocks on object storage.

**Operability:** All components expose `/ready`, `/metrics`, and ring status HTTP endpoints. A `mimir-mixin` package bundles Grafana dashboards, Prometheus alerts, and runbooks for the cluster itself. Rolling updates are zero-downtime by design.

**Durability:** Long-term storage uses S3/GCS/Azure Blob or any S3-compatible object store. The Compactor deduplicates and merges blocks uploaded by multiple ingesters, ensuring correctness at the storage layer.

**Extensibility:** The ingest path supports Prometheus remote write v1/v2, OTLP, and InfluxDB. Object storage backends are pluggable. The runtime configuration system supports per-tenant limit overrides without restarts.
