## Style Rationales

**Microkernel (primary, 0.93):** The defining architectural characteristic of Directus is its extension system — a comprehensive microkernel pattern where a stable core is extensible through fourteen well-typed plugin categories loaded and managed at runtime. The `ExtensionManager` class (`api/src/extensions/manager.ts`) provides the microkernel host: it discovers extensions from three sources (local filesystem, npm modules, and an external registry), compiles them at runtime using Rollup/Rolldown, manages their full lifecycle (`initialize()`, `reload()`, `uninstall()`), and dispatches `broadcastReloadNotification()` across cluster nodes via the Redis-backed message bus. Extensions plug into well-defined contracts through `@directus/types` — Hooks (filter/action/init/schedule handlers), Endpoints (Express Router mounts), Operations (flow step handlers), Interfaces, Displays, Layouts, Modules, Panels, Bundles, and Themes. API extensions run inside `isolated-vm` sandboxes (`api/src/extensions/lib/sandbox/`) with a controlled surface exposed as the `directus:api` module, preventing untrusted code from escaping the process boundary. Infrastructure itself follows the same plugin driver pattern: five storage drivers (`@directus/storage-driver-{local,s3,gcs,azure,cloudinary}`), five auth providers (`api/src/auth/drivers/`: local, oauth2, openid, ldap, saml), and two deployment providers (`api/src/deployment/drivers/`: netlify, vercel) are all interchangeable plugins. The `@directus/extensions-sdk` package provides CLI tooling (`directus-extension`) and templates for scaffolding and building extensions, completing the plugin ecosystem.

**Layered (secondary, 0.90):** Within the single deployable core, Directus is organized into strict horizontal layers. The HTTP entry layer is Express (`api/src/app.ts`, `api/src/server.ts`) which composes 30+ routers (`controllers/items.ts`, `controllers/users.ts`, `controllers/graphql.ts`, etc.) each bound to a dedicated URL path. Controllers delegate immediately to the service layer (`api/src/services/`), where business logic for each entity type is encapsulated (e.g., `items.ts` at 1,239 lines, `fields.ts` at 1,043 lines, `collections.ts` at 844 lines). Services call into the data access layer (`api/src/database/`) via Knex.js, which abstracts across PostgreSQL, MySQL, SQLite, OracleDB, CockroachDB, MariaDB, and MS-SQL without any service-layer branching. Cross-cutting concerns occupy their own layer: `api/src/middleware/` handles authentication, caching, rate limiting, CORS, query sanitization, and schema resolution. This layering is strict — controllers never touch the database, services never emit HTTP responses.

**Pipeline (supporting, 0.82):** The Flows subsystem (`api/src/flows.ts`, `api/src/operations/`) implements a user-configurable data processing pipeline. Each `Flow` has a `TriggerType` (`event`, `schedule`, `webhook`, `operation`, `manual`) and a linked list of `Operation` nodes, each with a `resolve` and `reject` successor forming a DAG of processing steps. Fifteen built-in operation types (`condition`, `exec`, `item-create`, `item-delete`, `item-read`, `item-update`, `json-web-token`, `log`, `mail`, `notification`, `request`, `sleep`, `throw-error`, `transform`, `trigger`) are registered into the `FlowManager` and chainable by users without code. Custom `Operation` types are also an extension point, allowing third-party pipeline steps. The `FlowManager` uses the `useBus()` message bus (Redis pub/sub or local EventEmitter) to coordinate flow reloads across clustered nodes.

## Domain

Headless CMS and data platform. Core function is wrapping any SQL database with an instant REST and GraphQL API, a no-code Vue.js admin dashboard, a fine-grained permission system, and a visual automation engine (Flows). Used as a backend-as-a-service, CMS backbone, and data ops platform.

## Production Context

- Deployed as a single Node.js process per container, managed by PM2 in cluster mode; horizontal scaling via multiple container replicas all sharing one Redis instance and one database
- `docker-compose.yml` is a developer multi-database test fixture (PostgreSQL, MySQL, MariaDB, MS-SQL, OracleDB, SQLite, CockroachDB), not a microservice deployment manifest
- The extension hot-reload mechanism uses the Redis bus to broadcast reload signals across all PM2 workers so every instance stays in sync without a full restart
- MCP integration (`api/src/ai/mcp/`) turns Directus into a tool server for LLM agents, exposing CRUD operations over collections — an emerging architectural surface added on top of the stable microkernel core
- WebSocket subscriptions (`api/src/websocket/`) provide real-time push to clients; this leverages the same emitter/bus infrastructure as the extension hook system, unifying sync and async event delivery
