# consul — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/hashicorp/consul
**Classification:** Modular Monolith + Layered
**Confidence:** 0.92

## Summary

HashiCorp Consul is a network infrastructure tool providing service discovery, health checking, key/value storage, and service mesh (Connect). Its repository produces a single Go binary (`consul`) that runs in either server or client agent mode. The codebase is structured as a **Modular Monolith** — one deployable unit composed of well-bounded internal packages with clear interfaces — organized around a **Layered** horizontal separation: CLI/API interface layer, agent/RPC business logic layer, Raft/FSM consensus layer, and in-memory state store layer. A secondary event-driven mechanism (Serf gossip channels + internal event publisher) handles intra-cluster state synchronization but does not constitute the system's primary architectural style.

### Key Architectural Files

- `main.go`: Single entry point — all 30+ sub-commands fold into one binary via `command.RegisteredCommands`.
- `agent/agent.go`: `Agent` struct is the top-level orchestrator; holds delegate (`*consul.Server` or `*consul.Client`), gRPC server, local state, anti-entropy sync, all health check types, and cache.
- `agent/consul/server.go`: `Server` struct embeds Raft instance, Serf LAN/WAN, FSM, state store, RPC listener, gRPC, AutoPilot, CA manager, and ACL resolver — the core of server-mode operation.
- `agent/consul/fsm/fsm.go`: Implements `raft.FSM`; routes Raft log commands through a dispatch table (`map[structs.MessageType]command`) to state store mutations, then publishes events to the stream layer.
- `agent/consul/state/catalog.go`: `Snapshot.Nodes()` and surrounding functions query the MemDB-backed tables — a pure data-layer concern separate from business logic.
- `agent/consul/stream/event_publisher.go`: `EventPublisher` receives change events from FSM commits and fans them to topic-keyed buffers; subscribers hold `*Subscription` to receive incremental updates.
- `internal/controller/manager.go`: `Manager.Register` / `Manager.Run` — Kubernetes-style reconciliation loop that executes controllers on Raft leader; controllers watch `pbresource.Resource` types and call `Reconciler.Reconcile`.
- `internal/storage/storage.go`: `Storage` interface defines `Read`, `Write`, `List`, `Watch`, `ListByOwner` — the port that both raft and inmem backends implement.
- `docs/README.md`: Self-describes the codebase as organized into "architectural layers" — CLI, HTTP API, Agent Configuration, RPC, Cluster Persistence, V2 Architecture, Client Agent, Service Discovery, Service Mesh, Cluster Membership.

### Patterns Found

**Modular Monolith (primary):** One binary, one Go module with internal `replace` directives for versioned sub-modules (`api/`, `sdk/`, `proto-public/`, `troubleshoot/`, `envoyextensions/`). Every feature (service mesh, ACL, K/V, DNS, federation) is an internal Go package with explicit interface boundaries. Community Edition vs. Enterprise variants are isolated via build-tagged `_ce.go` files, not separate deployables.

**Layered (primary, co-equal):** The codebase self-documents and structurally enforces four horizontal layers with one-direction dependencies:
1. **Interface layer** — `command/` (CLI), `agent/*_endpoint.go` (HTTP REST), `agent/grpc-external/` (gRPC public API), `agent/dns/` (DNS).
2. **Agent/RPC layer** — `agent/consul/server.go`, `agent/consul/client.go`, RPC dispatch, ACL resolution, blocking queries, cache.
3. **Consensus/FSM layer** — `agent/consul/fsm/`, Raft log application, Serf gossip, autopilot.
4. **State layer** — `agent/consul/state/` (MemDB tables), `internal/storage/` (V2 resource backends).

Classic layered dependency: `catalog_endpoint.go` → `Server` → `state.Store` — each layer only calls downward.

**Controller/Reconciliation (secondary):** `internal/controller/` implements a Kubernetes-style watch-reconcile loop for the V2 resource model. Controllers run only on the Raft leader, watch `pbresource.Resource` types via the gRPC Resource Service, and converge state — an observable architectural addition for the resource-oriented V2 API surface.

**Event-Driven (secondary, internal):** Serf gossip events flow through `eventChLAN` / `eventChWAN` channels into `server.go`'s event handlers. The `stream.EventPublisher` allows subscribers to receive incremental catalog and ACL changes via topic-keyed subscriptions rather than polling. Neither mechanism replaces the synchronous Raft write path; both are downstream notification mechanisms.

## Reasoning

Consul's architecture is definitively **Modular Monolith** because it produces one binary from one Go module, where all subsystems (service discovery, K/V, ACL, Connect, DNS, federation) coexist in shared memory space and share a single Raft-replicated state store. This rules out Microservices (no independent deployables), CQRS (single MemDB; no separate read/write models), Microkernel (no runtime plugin loading), and Serverless (persistent long-running daemon).

The Layered classification is equally primary and explicitly self-documented. The dependency graph flows strictly from interface adapters (HTTP endpoints, CLI, gRPC, DNS) down through business logic (Server/Client structs, RPC handlers) to the consensus mechanism (Raft FSM) and finally to the state store (MemDB). The `Catalog` endpoint struct holds a pointer to `Server`, which holds the state store — a textbook layered dependency.

The V2 architecture introduction (`internal/controller/`, `internal/resource/`, `internal/storage/`) adds a controller-reconciliation overlay for the resource-oriented API surface, inspired explicitly by Kubernetes. This is architecturally noteworthy but does not displace the fundamental modular monolith + layered character of the codebase.

Event-driven elements are real but subordinate: Serf gossip channels and the internal EventPublisher handle downstream change notification, not primary writes. The write path remains synchronous Raft consensus followed by FSM application.

Confidence is 0.92 due to the overwhelmingly clear single-binary deployment model, explicit layer documentation in `docs/README.md`, and the straightforward top-down dependency chain throughout the codebase.
