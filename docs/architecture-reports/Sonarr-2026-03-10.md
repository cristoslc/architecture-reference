## Architecture Styles

### Primary: Modular Monolith

Sonarr is a single deployable ASP.NET Core process (hosted via Kestrel, bootstrapped in `src/NzbDrone.Host/Bootstrap.cs` with DryIoc's `AutoAddServices(ASSEMBLIES)`). The solution compiles roughly 27 projects into one binary:

- `NzbDrone.Core` — all business logic (~2,794 C# files)
- `Sonarr.Api.V3` / `Sonarr.Api.V5` — versioned REST controllers (ASP.NET Core MVC)
- `Sonarr.Http` — middleware, authentication, error handling
- `NzbDrone.SignalR` — real-time WebSocket push via `MessageHub`
- `NzbDrone.Common` — shared infrastructure (HTTP client, disk I/O, DI, serialization)
- `NzbDrone.Windows` / `NzbDrone.Mono` — platform-specific shims

Within `NzbDrone.Core`, every functional domain is organized as a vertical slice with its own models, services, repositories, and events:

- `Tv/` — `Series`, `Episode`, `Season` entities; `SeriesService`, `EpisodeService`, `RefreshSeriesService`
- `Download/` — download client abstraction, `CompletedDownloadService`, `FailedDownloadService`, `DownloadEventHub`
- `Indexers/` — indexer request/response pipeline, `RssSyncService`, `FetchAndParseRssService`
- `DecisionEngine/` — chain-of-responsibility specifications (`IDownloadDecisionEngineSpecification`, 20+ rules) driving grab decisions
- `MediaFiles/` — `DiskScanService`, `DownloadedEpisodesImportService`, `EpisodeFileMovingService`
- `Notifications/` — 25 notification targets fanout on grab/import/upgrade/health events
- `Queue/` — in-progress download tracking
- `History/` — persistent download/import audit log
- `ImportLists/` — Trakt, AniList, custom list providers

A single SQLite or PostgreSQL database (accessed via `BasicRepository<T>`, Dapper-backed, through `MainDatabase`) serves all modules. 227 ordered FluentMigrator migration scripts maintain the schema.

### Secondary: Microkernel

The `ThingiProvider` framework (`NzbDrone.Core/ThingiProvider/ProviderFactory.cs`) provides a runtime plugin bus spanning every major extension category. All extension points implement `IProvider` and register via `ProviderDefinition` / `ProviderFactory<TProvider, TDef>`, resolved from DryIoc at request time. Adding a new plugin requires zero changes to the factory core — the DryIoc `AutoAddServices` call discovers it automatically.

Concrete plugin counts:

- **Download clients** (~20): Aria2, Blackhole, Deluge, DownloadStation, Flood, Freebox, Nzbget, Pneumatic, qBittorrent, rQBit, rTorrent, SABnzbd, Transmission, Tribler, uTorrent, Vuze, and more
- **Notification targets** (25): Apprise, CustomScript, Discord, Email, Gotify, Join, Mailgun, MediaBrowser, Notifiarr, Ntfy, Plex, Prowl, PushBullet, Pushcut, Pushover, SendGrid, Signal, Simplepush, Slack, Synology, Telegram, Trakt, Twitter, Webhook, Xbmc
- **Indexer types** (~10): BroadcastheNet, Fanzub, FileList, HDBits, IPTorrents, Newznab, Nyaa, Torrentleech, TorrentRss, Torznab
- **Import lists**: AniList, Custom, Trakt, and others

The internal messaging layer reinforces the plugin model:

- `EventAggregator.PublishEvent<TEvent>` resolves all `IHandle<TEvent>` (synchronous) and `IHandleAsync<TEvent>` (async via `TaskFactory`) subscribers from the container and dispatches to them in order.
- `CommandExecutor` resolves `IExecute<TCommand>` handlers from the container and runs them across three worker threads from a `CommandQueue`.
- Example flow: `RssSyncService` (triggered by `RssSyncCommand`) calls `FetchAndParseRssService` → `DownloadDecisionMaker` (runs 20+ `IDownloadDecisionEngineSpecification` rules) → `ProcessDownloadDecisions` → publishes `EpisodeGrabbedEvent` → `DownloadEventHub` → `NotificationService` fans out to all configured notification targets.

## Domain

Media Automation / TV Series Library Management — an open-source PVR (personal video recorder) that monitors RSS feeds from configured indexers for new TV episodes, applies quality profile and release restriction rules, sends accepted grabs to download clients, imports completed downloads, renames files to a configurable naming scheme, and notifies subscribers via 25 notification targets. Part of the Servarr/*arr ecosystem alongside Radarr (movies), Lidarr (music), Readarr (books), and Prowlarr (indexer proxy).

## Production Context

- Ships as a cross-platform .NET 10 process, default port 8989, deployable as a Windows service, Linux systemd unit, or Docker container.
- Data persistence: SQLite (default) or PostgreSQL; 227-step FluentMigrator migration history.
- The `ThingiProvider`/plugin system enables runtime addition of indexers, download clients, and notification targets from the web UI without restarting.
- React/TypeScript SPA frontend (in `frontend/`) communicates over versioned REST API (V3/V5) and SignalR for live command/download progress.
- The *arr ecosystem is Service-Based at the ecosystem level: Prowlarr syncs indexer configurations to Sonarr via the API; Overseerr, Ombi, and Jellyseerr request series via the API. Internally, Sonarr itself is a self-contained modular monolith.
