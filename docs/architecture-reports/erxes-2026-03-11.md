## Style Rationales

**Microkernel (primary, 0.93):** The erxes platform is explicitly organized around a Core + Plugin extension model, documented in the README as "erxes is composed of 2 main components: Core & Plugins." The core (`backend/core-api/`) provides the canonical host application — contacts, products, segments, automations, documents, permissions, notifications — and a shared runtime library (`backend/erxes-api-shared/`) that defines the plugin contract. Every plugin must call `startPlugin()` from `erxes-api-shared/src/utils/start-plugin.ts`, passing a typed config object that declares its name, port, GraphQL schema, tRPC router, REST endpoints, subscription paths, and metadata hooks (automations, segments, notifications, payments). This is a formalized plugin registration contract. The gateway (`backend/gateway/`) acts as the kernel host, discovering registered plugins via Redis (`erxes-service-${name}` keys), composing their GraphQL subgraphs via Apollo Router into a unified supergraph, and proxying REST requests via `/pl:serviceName` to the relevant plugin address. The `ENABLED_PLUGINS` environment variable controls which plugins the kernel loads at runtime — a classic Microkernel capability-selection mechanism. A `scripts/create-backend-plugin.js` scaffold tool generates the standard plugin directory skeleton (`apollo/`, `trpc/`, `modules/`, `db/models/`), enforcing the extension convention. The frontend mirrors this pattern: `frontend/core-ui/` is the Module Federation host that declares remote plugin UIs in `remotes: process.env.ENABLED_PLUGINS.split(',').map(p => p + '_ui')`, and each plugin UI (e.g., `frontend/plugins/frontline_ui/`) exposes named module entry points (`./config`, `./frontline`, `./frontlineSettings`) via its own `module-federation.config.ts`.

**Microservices (secondary, 0.85):** Each backend plugin runs as an independent Node.js process on a dedicated port (core-api: 3300, gateway: 4000, frontline: 3304, sales: 3305, etc.) and ships its own `Dockerfile` for containerized deployment. The automations engine (`backend/services/automations/`) and log service (`backend/services/logs/`) are additional standalone services. Inter-service communication uses two synchronous channels: GraphQL Federation (subgraph queries routed via Apollo Router) for public API queries, and tRPC over HTTP (`sendTRPCMessage` in `erxes-api-shared/src/utils/trpc/index.ts`) for typed service-to-service calls. Async communication uses BullMQ queues backed by Redis: `sendAutomationTrigger()` enqueues jobs to the `automations-trigger` queue; the automations service runs `triggerHandlerWorker`, `actionHandlerWorker`, and `aiWorker` BullMQ workers. Real-time subscriptions are handled by a gateway WebSocket server aggregating Redis PubSub (`graphqlPubsub` via `graphql-redis-subscriptions`). Service discovery is Redis-based: plugins register their address on startup via `joinErxesGateway()` and the gateway resolves them via `getPlugin(name)`. The critical divergence from pure microservices is that all plugins share a single MongoDB instance (`MONGO_URL`), with tenant isolation via `useDb()` per subdomain rather than per-service databases.

## Domain

Experience management platform (XOS — Experience Operating System) for customer-facing business operations. Core service domains: contacts/CRM, products, segments, automations, documents, notifications, permissions. Plugin domains: customer support / omnichannel conversations (Frontline), sales pipeline, accounting, loyalty programs, tourism booking, POS client, content management, insurance, payment processing, operational project management.

## Production Context

- Self-hosted via Docker Compose or Kubernetes; SaaS version at erxes.io with per-organization plugin entitlements read from a billing registry (`getSaasOrganizationDetail()`)
- All plugins share a single MongoDB instance — this is a deliberate design choice for multi-tenancy via `useDb(subdomain)` rather than per-service databases; this nuance distinguishes the architecture from pure microservices
- Redis serves three concurrent roles: service registry (plugin address store), BullMQ job broker (automations queue), and GraphQL PubSub channel (real-time subscriptions)
- Apollo Router (v1.59.2) is spawned as a child process by the gateway and recomposed when the supergraph changes, enabling zero-downtime plugin registration updates
- The SaaS deployment mode gates plugin availability per organization via purchased/free charge counts, implementing commercial extensibility on top of the Microkernel model
