# lemmy — Architecture Classification Report
**Date:** 2026-03-11
**Repository:** https://github.com/LemmyNet/lemmy
**Primary Style(s):** Modular Monolith, Layered
**Secondary Style(s):** Microkernel
**Confidence:** 0.88
**Summary:** Rust-based fediverse link aggregator structured as a single deployable modular monolith with a strict horizontal layering of routes → API → db_views → db_schema, ActivityPub federation handled via an async channel-based send pipeline, and a WASM plugin extension point giving it a microkernel quality.

## Evidence

**Workspace layout (Cargo.toml):** Single Rust Cargo workspace with 30+ member crates organized into explicit vertical modules: `crates/api/{api,api_crud,api_common,api_utils,routes,routes_v3}`, `crates/apub/{apub,activities,objects,send}`, `crates/db_schema`, `crates/db_schema_file`, `crates/db_views/{comment,community,post,person,…}`, `crates/routes`, `crates/server`, `crates/utils`, `crates/email`. All crates compile into a **single binary** (`lemmy_server`).

**Single binary entry point:** `crates/server/src/main.rs` invokes `start_lemmy_server()` which starts one Actix-web HTTP server, one background `handle_outgoing_activities` Tokio task, one `SendManager` for federation sends, and one `scheduled_tasks` loop — all in the same process.

**Layered request path:** Every incoming HTTP request passes through Actix middleware (rate-limit, CORS, session, idempotency, tracing) → `lemmy_api_routes::config` / `lemmy_apub::http::routes::config` → API handler functions (`lemmy_api`, `lemmy_api_crud`) → db_views read queries or db_schema write mutations → PostgreSQL. This is a textbook presentation → application → domain → data-access stack.

**Read/Write model separation (CQRS-adjacent):** `crates/db_views` contains 20+ read-only view crates (PostView, CommentView, CommunityView, etc.) that aggregate joined query results for API responses. Write paths live in `crates/db_schema/src/impls` and `api_crud`. These are not event-sourced CQRS but demonstrate deliberate read/write path separation.

**ActivityPub federation as async pipeline:** After any write operation, API handlers post a `SendActivityData` enum variant to a `tokio::mpsc::unbounded_channel` (`ActivityChannel`). A dedicated background task `handle_outgoing_activities` consumes that channel and dispatches to `lemmy_apub_activities::match_outgoing_activities`, which serializes the activity, writes it to the `sent_activity` DB queue, and the `SendManager` workers read that queue and deliver to remote instances. This intra-process async pipeline is the federation path.

**WASM Plugin system (Microkernel evidence):** `crates/api/api_utils/src/plugins.rs` implements `LemmyPlugins` using the `extism` WASM runtime. Plugins are loaded from disk paths or HTTP URLs, run in WASM sandboxes, and are called at named hook points: `plugin_hook_before` (can mutate request data), `plugin_hook_after` (fire-and-forget side-effects), and `notification_after_create`. Plugin discovery is via config (`defaults.hjson` `plugins` array). This is a minimal plugin/extension kernel pattern.

**Docker composition:** `docker/docker-compose.yml` runs five containers: nginx (proxy), lemmy (this repo), lemmy-ui (separate TypeScript SPA), pictrs (image service), postgres. The Lemmy backend is a **single service**; there is no intra-backend service mesh or inter-service message bus.

**328 database migrations:** `migrations/` contains 328 Diesel migration scripts. The schema has grown incrementally in a single PostgreSQL database shared by all domain logic — consistent with a monolithic data layer.

**Federation horizontal scaling docs:** CLI flags `--disable-http-server`, `--disable-activity-sending`, `--federate-process-index N`, `--federate-process-count N` allow running multiple instances of the same binary with responsibilities partitioned. This is horizontal process-level scaling of the monolith, not decomposition into independent services.

## Architecture Styles

### Modular Monolith (Primary, 0.90)
- Single Rust Cargo workspace, single compiled binary, single PostgreSQL database
- Crates with strong internal boundaries: `db_schema` (write), `db_views` (read), `apub` (federation protocol), `api` (HTTP handlers), `routes` (HTTP layer, middleware, feeds)
- All modules share the `LemmyContext` struct (pool, HTTP client, secrets, rate-limit) passed as Actix-web app data
- No inter-process RPC between modules; function calls and tokio channels only
- 328 DB migrations in one schema — unified data layer

### Layered (Primary, 0.87)
- Four clearly demarcated layers: (1) Route/HTTP (actix handlers, middleware), (2) API/Application (api, api_crud, api_utils — business logic, validation, auth), (3) Domain/Data-Access (db_schema traits, impls), (4) Persistence (Diesel + PostgreSQL)
- `db_views` acts as a query-result layer between the application and persistence layers
- Layer dependencies are one-directional: routes → api → db_views/db_schema → diesel

### Microkernel (Secondary, 0.72)
- `extism` WASM plugin system with hook points: `plugin_hook_before`, `plugin_hook_after`, `notification_after_create`, `get_captcha`, `validate_captcha`
- Plugins loaded from configurable file paths or HTTP URLs with SHA-256 hash verification
- Plugin sandbox via WASM with allowed-host and allowed-path controls
- Core behavior is complete without plugins; plugins augment or modify data at defined extension points
- Pattern is lightweight (not a full microkernel) — plugins cannot replace core request handling, only hook into it

### Rejected Styles

**Not Event-Driven (primary):** The internal ActivityChannel is an in-process tokio channel, not an external event broker. There are no topics, subscribers, or event schemas outside of the ActivityPub wire format, which is a federation protocol, not an internal event bus.

**Not Microservices:** All domain logic lives in one binary and one database. lemmy-ui, pictrs, and postgres are external dependencies, not services decomposed from the same application.

**Not CQRS:** While reads and writes use different code paths and crates, there is no command/event bus, no separate read store derived from events, and no event sourcing. The split is a code organization choice, not a CQRS architecture.

**Not Domain-Driven Design (formal):** There are no explicit aggregate roots, domain events, bounded contexts, or repositories following DDD conventions. The crate organization reflects technical concerns (api, db_schema, apub) rather than business domains.

## Quality Attributes

- **Deployability:** Single binary, Docker Compose with five containers, Ansible playbook documented; arm64/Raspberry Pi support
- **Scalability:** Process-level horizontal scaling via federation worker index/count flags; DB connection pool (default 30); async Tokio runtime for I/O concurrency
- **Extensibility:** WASM plugin API with before/after hooks; configurable federation allow/block lists; pluggable captcha provider
- **Federatability / Interoperability:** Full ActivityPub protocol implementation (`lemmy_apub` + `activitypub-federation-rust`); WebFinger, NodeInfo, RSS/Atom feeds; signed fetch support
- **Observability:** Prometheus metrics endpoint (`/metrics`); structured JSON logging via tracing-subscriber; tracing-actix-web request spans; per-request idempotency tracking
- **Modifiability:** Strong internal crate boundaries make individual layers replaceable; Diesel schema migrations enable incremental DB evolution
- **Security:** Rate limiting per endpoint class; JWT-based session auth; RSA key signing for ActivityPub; bcrypt password hashing; 2FA TOTP support; WASM plugin sandboxing with hash verification
- **Performance:** Rust + Actix-web; mimalloc global allocator on x86_64; LTO + codegen-units=1 release profile; in-memory Moka cache; async DB pool via diesel-async

## Classification Reasoning

Lemmy is a **Modular Monolith** at its core. The Cargo workspace organizes the codebase into ~30 tightly-coupled crates that all compile into a single `lemmy_server` binary backed by a single PostgreSQL database. The crate boundaries enforce separation of concerns within the monolith — `db_schema` owns write models and ORM mappings, `db_views` owns read aggregations, `apub` owns ActivityPub protocol, `api`/`api_crud` own HTTP handlers — but no crate boundary crosses a network or process boundary.

Within that monolith, the architecture is unmistakably **Layered**. Incoming requests flow through exactly four layers in order: (1) HTTP route handling and middleware (rate limiting, session, CORS, idempotency), (2) API application logic (validation, authorization, business rules in api/api_crud), (3) data access (db_schema impls and db_views queries), and (4) PostgreSQL persistence. Layer dependencies are strictly one-directional. The `LemmyContext` shared-state struct is passed as an Actix-web `Data` wrapper, functioning as a service locator across layers.

The federation subsystem adds a secondary **pipeline quality**: writes trigger `SendActivityData` enum variants enqueued into a tokio channel, consumed by `handle_outgoing_activities`, written to the `sent_activity` database queue, then delivered by `SendManager` per-instance workers. This is an intra-process async pipeline, not a separate service, but it is architecturally distinct within the monolith.

The WASM plugin system gives Lemmy a **Microkernel** quality at the secondary level. The `extism`-based plugin loader (`LemmyPlugins`) supports before-hooks (data mutation), after-hooks (side effects), and pluggable captcha providers loaded from configurable WASM files. This is a functional extension kernel, though not the primary organizing principle of the codebase.

Confidence is 0.88. The modular monolith and layered classifications are unambiguous from the code structure. The microkernel classification reflects a real architectural pattern implemented in `plugins.rs`, but it is additive rather than foundational.
