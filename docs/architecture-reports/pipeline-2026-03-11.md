## One-Line Summary

Kubernetes-native CI/CD pipeline engine that executes DAG-scheduled Tasks as Kubernetes Pods, coordinated by a Kubernetes controller reconciliation loop, with a pluggable resolver framework for remote resource fetching.

## Structural Overview

Tekton Pipelines is a single Go module (`github.com/tektoncd/pipeline`) deployed as three distinct Kubernetes Deployments from three `cmd/` entry points:

- `cmd/controller` — runs three reconcilers: `taskrun`, `pipelinerun`, and `resolutionrequest` in a single process using Knative's `sharedmain` controller framework.
- `cmd/webhook` — runs Kubernetes admission webhooks (defaulting, validation, config validation, and CRD version conversion) as a separate Deployment.
- `cmd/resolvers` — runs five resolver controllers (git, hub, bundle, cluster, http) in a single process via the resolver `framework`.

All three processes share the same compiled Go module but execute independently as Kubernetes-managed services. The entire system is a single-codebase, single-module monorepo — not a microservices system.

### Core Domain Model (CRDs)

Eight Kubernetes Custom Resource Definitions form the type system:

| CRD | Purpose |
|---|---|
| `Pipeline` | Declares a DAG of Tasks with `runAfter` / result-passing dependency edges |
| `PipelineRun` | An execution instance of a Pipeline |
| `Task` | A sequence of Steps (containers) executed on a single Pod |
| `TaskRun` | An execution instance of a Task |
| `StepAction` | A reusable single-Step definition (v1alpha1/v1beta1) |
| `CustomRun` | Extensibility contract for third-party task implementations |
| `ResolutionRequest` | A request to a resolver to fetch a remote pipeline/task definition |
| `VerificationPolicy` | Policy for trusted resource verification |

### Pipeline Architecture (Primary)

The Pipeline style is the project's primary identity and raison d'être. The fundamental abstraction is an **explicit DAG** of Tasks with declared data-flow dependencies:

**DAG scheduling:** `pkg/reconciler/pipeline/dag/dag.go` implements a generic directed acyclic graph (`Node`, `Graph`) where each `PipelineTask` is a node. Edges are derived from `runAfter` (ordering) and result-passing (`$(tasks.X.results.Y)` parameter interpolation). The `pipelinerun` reconciler traverses this DAG on every reconcile loop, identifying which Tasks are ready to start (all predecessors complete), currently running, or blocked.

**Step sequencing within a Task:** Inside each Pod, Tekton injects a custom `entrypoint` binary (`cmd/entrypoint`) into every Step container. The entrypointer wraps the user's command and uses filesystem-based synchronization (`/tekton/run/<N>/out`) to enforce sequential Step execution within a Task Pod. Each Step waits for its predecessor's marker file before starting, implementing a local pipeline within the container group.

**Result passing:** Tasks emit results via termination messages or a sidecar log results mechanism. The `pipelinerun` reconciler collects these results and passes them as parameters to downstream Tasks via `$(tasks.X.results.Y)` substitution (`pkg/substitution`), making data flow through the pipeline graph explicit and typed.

**Finally tasks:** The Pipeline type also supports a `finally` block — Tasks that execute unconditionally after the main DAG completes, providing a structured pipeline termination phase.

### Microkernel Architecture (Secondary)

Tekton employs a microkernel-style extensibility mechanism for remote resource resolution:

**Resolver plugin framework:** `pkg/remoteresolution/resolver/framework` defines a `Resolver` interface that resolver plugins implement. Five built-in resolvers ship with the project (git, Tekton Hub, Artifact Hub, bundle/OCI, cluster, HTTP), each registering as an independent controller via `framework.NewController`. Third-party resolvers can implement the same interface and deploy as additional controllers without modifying the core.

**CustomRun extensibility point:** The `CustomRun` CRD provides a formal contract for third-party custom task implementations. External controllers can watch `CustomRun` objects and implement arbitrary execution behavior while participating in the pipeline's result/status protocol.

**Webhook conversion controllers:** The webhook process handles CRD version conversion between v1alpha1 → v1beta1 → v1, demonstrating a layered versioned API with backward compatibility managed by a hub-and-spoke conversion topology.

### Event-Driven Architecture (Secondary)

The control plane is event-driven via the Kubernetes informer/watch mechanism:

**Kubernetes informer pattern:** The `pipelinerun` and `taskrun` controllers use `AddEventHandler` on informers for PipelineRun, TaskRun, CustomRun, ResolutionRequest, and Pod objects. State changes to any of these objects trigger reconcile queue entries, causing the controller to re-evaluate the pipeline DAG state. This is a reactive, level-triggered event model.

**CloudEvents emission:** `pkg/reconciler/events/cloudevent` implements optional CloudEvents delivery to external sinks. On every significant TaskRun/PipelineRun status transition, the reconciler emits a structured CloudEvent (`tekton.dev/v1/taskrun/started`, etc.), enabling integration with external event-processing systems (Tekton Triggers, notification systems, observability platforms).

**Kubernetes Events:** Standard Kubernetes Events are also emitted on state transitions via `pkg/reconciler/events/k8sevent`, providing operator-visible audit trail in the cluster.

### Internal Package Structure

```
pkg/
  apis/pipeline/          # CRD type definitions (v1alpha1, v1beta1, v1) + validation
  client/                 # Generated Kubernetes client-go clientsets, listers, informers
  reconciler/
    pipeline/dag/         # DAG graph implementation
    pipelinerun/          # PipelineRun reconciler + resource helpers
    taskrun/              # TaskRun reconciler
    events/               # K8s event + CloudEvent emission
    volumeclaim/          # PVC management for workspaces
  pod/                    # Pod builder: converts TaskSpec to Kubernetes PodSpec
  entrypoint/             # Entrypointer binary: Step sequencing within a Pod
  remoteresolution/       # Resolver framework + built-in resolver implementations
  trustedresources/       # Verification of remote tasks/pipelines against VerificationPolicy
  workspace/              # Workspace volume binding logic
  spire/                  # SPIRE/SPIFFE workload identity integration
  pipelinerunmetrics/     # Prometheus/OpenCensus metrics for PipelineRun lifecycle
  taskrunmetrics/         # Prometheus/OpenCensus metrics for TaskRun lifecycle
  substitution/           # Parameter and result reference interpolation engine
```

### What Was Considered and Rejected

**Not Microservices:** The three Deployments (controller, webhook, resolvers) compile from the same Go module and share all type definitions, clients, and business logic. They are deployment-split process boundaries, not independently deployable services with their own data stores. All state lives in the Kubernetes API server (etcd).

**Not pure Modular Monolith:** While the package structure is modular, the defining characteristic of the project is the DAG-scheduled pipeline execution model, not modularity per se. The Pipeline style captures the essential identity.

**Not Layered:** No horizontal presentation/business/data layer stack. The organization is by domain concern (pipeline scheduling, task execution, resolution, events), not by architectural tier.

**Not CQRS:** While the Kubernetes API server provides an implicit read/write separation (lister cache for reads, API write operations for mutations), this is a Kubernetes platform pattern, not a CQRS design decision by the application.

## Quality Attributes Evidence

**Reliability:** The Knative reconciler framework provides level-triggered control loops that re-converge on any state drift. Leader election (`config-leader-election-*.yaml`) ensures HA controller operation. Timeout handling in `pipelinerun/timeout.go` and `taskrun` prevents resource leaks.

**Extensibility:** Resolver plugin framework allows custom remote resource sources without modifying core. CustomRun CRD provides a formal extensibility contract for third-party task types. VerificationPolicy enables trusted-supply-chain controls as a pluggable policy layer.

**Observability:** Prometheus metrics via OpenCensus/OpenTelemetry in `pipelinerunmetrics` and `taskrunmetrics`. Distributed tracing via OpenTelemetry in `tracing.go` reconcilers. CloudEvents delivery to external sinks. Kubernetes Events for cluster-level audit.

**Security:** SPIRE/SPIFFE workload identity integration (`pkg/spire`) for cryptographic task identity. TrustedResources (`pkg/trustedresources`) verifies remote pipeline/task definitions against Sigstore-based signatures. Admission webhooks enforce validation on all CRD writes.

**Scalability:** Tasks execute as independent Kubernetes Pods on any available node, achieving native horizontal scalability through the Kubernetes scheduler. The DAG executor maximally parallelizes independent branches. The resolver service scales independently of the controller.

**Evolvability:** Three-version API lifecycle (v1alpha1 → v1beta1 → v1) with webhook-managed conversion. `api_compatibility_policy.md` documents a formal stability contract. Feature flags (`config-feature-flags.yaml`) allow controlled rollout of new capabilities.
