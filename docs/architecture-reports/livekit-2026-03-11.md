# livekit — Architecture Classification Report
**Date:** 2026-03-11
**Repository:** https://github.com/livekit/livekit
**Primary Style(s):** Modular Monolith, Event-Driven
**Confidence:** 0.88
**Summary:** Single-binary WebRTC SFU server organized as a tightly-coupled set of internal modules, with event-driven signaling and telemetry flows and Redis pub/sub enabling optional horizontal distribution.

## Evidence

- **Single Go module, single deployable binary** (`github.com/livekit/livekit-server`): all packages (`pkg/sfu`, `pkg/rtc`, `pkg/service`, `pkg/routing`, `pkg/agent`, `pkg/telemetry`) are compiled and shipped together as one process (`cmd/server/main.go`).
- **Well-defined internal module boundaries**: `pkg/sfu` (Selective Forwarding Unit — packet routing, bandwidth estimation, codec munging), `pkg/rtc` (room/participant lifecycle, WebRTC session management), `pkg/service` (HTTP API surface, RoomService, RTCService, AgentService, WHIPService), `pkg/routing` (node discovery, Redis router, signal relay), `pkg/telemetry` (analytics events, webhook dispatch, Prometheus metrics), `pkg/agent` (AI agent worker lifecycle) — each has its own interfaces and fakes generated via counterfeiter.
- **Google Wire dependency injection** (`wire.go`, `wire_gen.go`) wires together all modules at startup, confirming a monolith with clear compositional seams rather than runtime service discovery.
- **Protobuf message bus (PSRPC) for intra-cluster events**: `psrpc.MessageBus` abstracts either `LocalMessageBus` (single-node) or `RedisMessageBus` (distributed). Signal relay, room allocation, and participant sessions all flow through this message bus.
- **Redis pub/sub for horizontal scale**: `RedisRouter` registers nodes in Redis, uses Redis pub/sub to route signaling messages across instances. Room-to-node mapping (`NodeRoomKey`) stored in Redis hash. This is an optional operational mode — the server works without Redis as a single node.
- **Event-driven telemetry and webhook subsystem**: `telemetryService` enqueues domain events (`RoomStarted`, `ParticipantJoined`, `TrackPublished`, etc.) that fan out to both analytics event pipelines and outbound webhook notifications — a classic event-driven notification pattern.
- **Message sink/source abstraction in routing**: `MessageSink` and `MessageSource` interfaces model one-way protobuf message channels between signal and RTC nodes, with channel-based reads (`ReadChan() <-chan proto.Message`).
- **Agent worker protocol**: `pkg/agent` implements a WebSocket-based job dispatch protocol for external AI/automation agents that connect to the server as participants. Workers register, receive job assignments, and report status — an event-driven handshake over WebSockets.
- **Twirp RPC API layer**: RoomService, EgressService, IngressService, SIPService, AgentDispatchService are all exposed over HTTP via the Twirp RPC framework with JWT authentication middleware — a clean service interface layer over the internal monolith.
- **Node selector subsystem** (`pkg/routing/selector`): pluggable node-selection strategies (any, sysload, cpuload, regionaware) with multi-region latency-aware routing.
- **Prometheus metrics + OpenTelemetry tracing** integrated at the service boundary layer.
- **Key dependencies**: `pion/webrtc`, `pion/ice`, `pion/dtls`, `pion/sctp` (WebRTC stack); `redis/go-redis` (distributed routing); `livekit/psrpc` (typed pub/sub RPC); `google/wire` (DI); `twitchtv/twirp` (HTTP RPC); `nats-io/nats.go` (alternative message bus).

## Architecture Styles

### Primary: Modular Monolith
All domain concerns — SFU media forwarding, WebRTC session management, room lifecycle, signaling, agent dispatch, ingress/egress integration, telemetry — are packaged into a single Go binary with well-defined package boundaries. Each package exposes typed interfaces and generated fakes for testability, but cross-package calls are direct function calls, not network hops. Google Wire confirms compile-time wiring of all components. The `pkg/sfu` package alone contains ~30 sub-packages, each with a narrow responsibility (bwe, pacer, streamallocator, videolayerselector, etc.), demonstrating deliberate modular decomposition within the monolith.

Supporting evidence:
- One `go.mod`, one binary, one Dockerfile
- Google Wire `InitializeServer` function wires ~25 distinct components
- Inter-package dependencies are import-time, not network-time

### Secondary: Event-Driven
The internal architecture relies heavily on event-driven patterns for two distinct concerns:

1. **Intra-cluster message routing**: PSRPC message bus (backed by Redis pub/sub or local channels) routes signaling messages between the signal-accepting node and the RTC-processing node. `MessageSink`/`MessageSource` channel interfaces express this dataflow.
2. **Telemetry and webhook notifications**: Domain events are enqueued asynchronously and dispatched to analytics backends and configured webhook URLs, decoupling the media-serving hot path from observability concerns.
3. **Agent job dispatch**: The agent worker protocol is a request/response event sequence over WebSocket — workers emit availability, the server emits job assignments, workers emit status updates.

## Quality Attributes

- **Scalability**: Horizontal scale via Redis-backed distributed router; node selector with load-aware and region-aware strategies; per-node track/bandwidth limits
- **Performance**: SFU architecture avoids media mixing; congestion control (BWE), simulcast, SVC codecs, adaptive stream subscription; batch I/O for packet writes; separate UDP/TCP ICE paths
- **Reliability**: Signal relay with configurable retry/backoff; PSRPC with max-attempts and timeout policies; graceful drain on shutdown; keepalive pub/sub for node liveness
- **Availability**: Multi-node/multi-region deployment mode; Redis Sentinel and Cluster support; health check endpoint; node deregistration on shutdown
- **Observability**: Prometheus metrics endpoint; OpenTelemetry tracing; structured zap logging with sampling; pprof endpoints in dev mode; telemetry event pipeline
- **Security**: JWT access token authentication; API key/secret pairs; end-to-end encryption support; TURN server with credential management; CORS policy; TLS for all external interfaces
- **Extensibility**: Pluggable node selectors; agent worker protocol for external AI participants; PSRPC message bus abstraction (local or Redis); webhook notification for room events
- **Operability**: Single binary deployment; Docker/Kubernetes support; Helm charts; YAML configuration; dev mode with placeholder credentials; Prometheus + structured logging

## Classification Reasoning

LiveKit is best classified as a **Modular Monolith** because it is a single deployable artifact (one binary, one Go module) where the internal structure is disciplined into clear functional packages (`sfu`, `rtc`, `service`, `routing`, `agent`, `telemetry`) with interface-based boundaries and generated mocks — classic modular monolith markers. The package dependency graph is a DAG (no cycles enforced), and Google Wire makes the composition graph explicit and verifiable at compile time.

The **Event-Driven** secondary style is warranted because message passing via typed channels and pub/sub is a load-bearing architectural concern — not merely an implementation detail. The PSRPC bus is what enables multi-node operation: the signal server and RTC server can run on different nodes, with all coordination happening through bus messages. Without the event-driven layer, the distributed deployment mode would not exist.

**Why not Microservices**: All concerns live in one process. Egress, Ingress, and SIP are separate repos/binaries — but this repo (livekit-server) itself does not decompose into independent services. It integrates with those external services via RPC clients.

**Why not Layered**: The pkg structure is domain-functional (sfu, rtc, routing, agent) rather than horizontal technical layers (presentation, business, data). Dependencies between packages follow domain relationships, not strict top-down layering.

**Why not Service-Based**: Service-Based implies coarse-grained services sharing a database. Here, all components share one in-process call graph, not a shared database accessed by independent services.

Confidence is 0.88. The Modular Monolith classification is high-confidence; the Event-Driven secondary is also strong but could be seen as an implementation mechanism enabling distribution rather than a top-level architectural style commitment.
