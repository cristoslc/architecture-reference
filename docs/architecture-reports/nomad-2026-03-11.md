## Project Overview

HashiCorp Nomad is a production-grade, distributed workload orchestrator written entirely in Go. It schedules and manages containers (Docker, Podman), non-containerized applications (raw exec, Java), and virtual machines (QEMU) across on-premises and cloud infrastructure at scale. Nomad is licensed under BUSL-1.1 and ships as a single binary that can operate in both server and client roles.

The project contains approximately 2,129 Go source files organised into clearly delineated packages: `nomad/` (server), `client/`, `scheduler/`, `plugins/`, `drivers/`, `api/`, `command/`, `acl/`, `helper/`, `lib/`, and `ui/`.

## Structural Analysis

### Deployment Unit

Nomad ships as a **single binary** (`main.go`) whose `command/` package exposes every CLI verb. The binary runs as either a **server** (control-plane leader / follower) or a **client** (data-plane agent on each worker node). There are no separate deployable services; the entire system is one compiled artefact.

### Top-Level Package Decomposition

| Package | Responsibility |
|---|---|
| `nomad/` | Server: RPC endpoints, Raft consensus, Serf gossip, FSM, evaluation broker, plan application, state store |
| `client/` | Client agent: allocation runner, fingerprinting, plugin managers, service registration |
| `scheduler/` | Scheduling algorithms (service, batch, system, sysbatch) |
| `plugins/` | gRPC plugin interfaces (base, driver, device, CSI) |
| `drivers/` | Built-in task drivers (Docker, exec, Java, QEMU, rawexec) |
| `api/` | Public Go client library / HTTP REST surface |
| `command/` | CLI commands wired to API calls |
| `acl/` | Access control policy engine |
| `helper/` | Shared utilities (TLS, pool, codec, UUID) |
| `lib/` | Low-level libraries (auth, delay heap, file, lang) |
| `internal/` | Test-only utilities |

### Server Architecture (Control Plane)

`nomad/server.go` (2,378 lines) initialises the full control-plane stack:

- **Raft consensus** (`github.com/hashicorp/raft`) — all state mutations flow through a Raft finite-state machine (`nomad/fsm.go`, 3,409 lines). The FSM supports 31 named snapshot types covering every domain entity (nodes, jobs, evaluations, allocations, ACL, CSI, variables, etc.).
- **Serf gossip** (`github.com/hashicorp/serf`) — used for cluster membership and failure detection across servers and clients.
- **RPC multiplexer** (`nomad/rpc.go`) — a single TCP port carries Raft, Serf, and custom msgpack-over-RPC streams, demultiplexed by a magic byte header.
- **Evaluation broker** (`nomad/eval_broker.go`) — an in-memory priority-queue broker that serialises evaluations per job, dequeues them to scheduler workers, and enforces at-least-once delivery with explicit Ack/Nack.
- **Scheduler workers** (`nomad/worker.go`) — goroutine pool that dequeues evaluations, calls into the `scheduler/` package, and submits allocation plans back to the leader's plan queue.
- **Plan applier** (`nomad/plan_apply.go`) — leader-side planner that validates and commits scheduler plans via Raft.
- **Event broker** (`nomad/stream/event_broker.go`) — asynchronous pub/sub layer that converts Raft-committed state changes into typed events for streaming API consumers (`nomad/state/events.go` maps every `MessageType` to a named event type).
- **RPC endpoints** — ~35 endpoint files (`*_endpoint.go`) each implement a narrow slice of API surface (jobs, evals, allocs, nodes, ACL, CSI, deployments, scaling, variables, etc.).

### Client Architecture (Data Plane)

`client/client.go` (3,617 lines) manages the agent running on every worker node:

- **Fingerprinting** (`client/fingerprint/`) — detects node attributes (CPU, memory, network, GPU) for scheduler constraint matching.
- **Plugin managers** (`client/pluginmanager/`) — lifecycle-manages driver and device plugins via the gRPC plugin system.
- **Allocation runner** (`client/allocrunner/`) — runs and supervises each allocated workload; internally delegates to task runners and hook chains.
- **Device manager** (`client/devicemanager/`) — discovers and reports hardware devices (GPU, FPGA) via device plugins.
- **Service registration** (`client/serviceregistration/`) — registers tasks with Consul or the built-in Nomad service discovery.
- **CSI manager** (`client/pluginmanager/csimanager/`) — manages Container Storage Interface plugin lifecycle.

### Plugin / Microkernel Layer

The `plugins/` package defines stable gRPC interfaces (`go-plugin` v1.7.0 over gRPC with protobuf) for:

- **Task drivers** (`plugins/drivers/driver.go`) — `DriverPlugin` interface; built-in implementations in `drivers/` (Docker, exec, Java, QEMU, rawexec).
- **Device plugins** (`plugins/device/`) — hardware device discovery and allocation.
- **CSI plugins** (`plugins/csi/`) — Container Storage Interface.
- **Base plugin** (`plugins/base/plugin.go`) — shared handshake and configuration protocol shared by all plugin types.

External community drivers implement these interfaces as out-of-process gRPC servers, loaded and managed by the client's plugin manager. This is a textbook microkernel extension mechanism.

### Scheduling Pipeline

The evaluation-driven scheduling pipeline is a clearly defined, sequential pipeline:

1. **Trigger** — a job, node, or deployment change produces an evaluation (written to Raft).
2. **Broker** — `EvalBroker` dequeues the evaluation to a scheduler worker, priority-ordered and per-job serialised.
3. **Scheduler** — `scheduler.NewScheduler()` selects one of four built-in algorithms (service, batch, system, sysbatch) and produces an allocation plan.
4. **Plan applier** — leader validates and commits the plan to Raft, updating the state store.
5. **Client** — the allocation is dispatched to the target client node, which runs it via an `AllocRunner`.

### State Management

All authoritative state lives in a single `nomad/state/state_store.go` (memdb, snapshotted to BoltDB/raft-wal). There is no CQRS split or separate read model; reads hit the in-memory memdb directly.

### Event Streaming

`nomad/stream/event_broker.go` provides an async event buffer that decouples Raft commits from API streaming subscriptions. This is an internal, secondary streaming layer — not the primary communication mechanism — so it does not qualify as a primary Event-Driven architecture.

## Architecture Classification

### Primary Style: Modular Monolith

Nomad's defining characteristic is a single deployable binary whose internals are strongly partitioned into cohesive, bounded packages with clear interfaces. The packages — `nomad/`, `client/`, `scheduler/`, `plugins/`, `drivers/`, `acl/`, `api/` — are independently comprehensible and testable, but they are compiled and deployed together as one unit. There are no separately deployable services, no service mesh, no independent data stores per logical component. This matches the Modular Monolith style precisely.

### Secondary Style: Microkernel

The `plugins/` framework constitutes a genuine microkernel extension architecture. The core runtime (Nomad binary) defines stable gRPC plugin interfaces; task drivers, device plugins, and CSI plugins are out-of-process extensions that the kernel discovers, loads, and supervises. The plugin handshake, versioning, and lifecycle management follow the microkernel pattern of a small stable core with a well-defined plugin API.

### Contributing Patterns (not primary styles)

- **Pipeline** — the evaluation broker → scheduler worker → plan applier → client dispatch pipeline is a strong pipeline pattern internal to the modular monolith, but it does not constitute a standalone architecture style for the system as a whole.
- **Event-Driven (partial)** — the `stream.EventBroker` and `state/events.go` mapping provide event streaming to API consumers, but state mutation is synchronous Raft, not event sourcing. The pattern is supplementary, not architectural.
- **Layered** — within the server (`nomad/`), there is a loose horizontal layering (RPC endpoints → FSM / state store → scheduler), but it is not enforced as a strict dependency direction and does not define the system topology.

### Why Not Microservices or Service-Based

Nomad does not decompose into independently deployable, separately versioned, network-addressable services. All server functionality (scheduler, evaluations, ACL, CSI, deployments, variables) ships in one binary, shares one state store, and communicates in-process. The client is a peer role of the same binary, not a separate service.

## Quality Attributes

| Attribute | Evidence |
|---|---|
| **High Availability** | Raft leader election, Serf membership; cluster survives node failures |
| **Scalability** | Optimistic scheduling, 10K+ node clusters in production; evaluation broker per-job serialisation prevents head-of-line blocking |
| **Extensibility** | gRPC plugin system (go-plugin) for drivers, devices, CSI |
| **Operability** | Single binary deployment; no external coordination service required |
| **Consistency** | All mutations through Raft FSM; strong consistency for scheduler state |
| **Observability** | go-metrics instrumentation throughout; structured logging (go-hclog) |
| **Security** | mTLS, ACL policy engine, Vault integration, workload identity (OIDC) |
| **Portability** | Linux, Windows, macOS; driver plugins abstract workload execution |
