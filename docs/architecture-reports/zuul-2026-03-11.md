# zuul — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/Netflix/zuul
**Classification:** Pipeline + Microkernel
**Confidence:** 0.93

## Summary

Netflix Zuul 2 is an L7 application gateway built on Netty, deployed as a single JVM library/application (not a set of microservices). Its primary architectural expression is the Pipeline (pipe-and-filter) pattern: every HTTP request passes through an ordered chain of INBOUND filters, a single ENDPOINT filter (the proxy), and an ordered chain of OUTBOUND filters. The Microkernel pattern is the secondary style: a stable core (`zuul-core`) provides the filter execution engine, Netty channel pipeline, and connection-pooling infrastructure, while all routing, authentication, throttling, and response-shaping logic lives in independently loaded filter plugins. Together these two patterns give Zuul its defining structural signature as an extensible, high-throughput API gateway.

### Key Architectural Files

- `zuul-core/…/filters/FilterType.java`: Enum with three values — `INBOUND("in")`, `ENDPOINT("end")`, `OUTBOUND("out")`. This is the complete pipeline stage taxonomy.
- `zuul-core/…/netty/filter/ZuulFilterChainRunner.java`: `runFilters()` iterates an ordered `ZuulFilter<T,T>[]` array, calls `executeFilter()` on each, passes the output as input to the next. Returns `Pending` (async) or `Complete`. Each stage is a pure function on `ZuulMessage`.
- `zuul-core/…/netty/filter/ZuulFilterChainHandler.java`: Holds `requestFilterChain` and `responseFilterChain` as two separate `ZuulFilterChainRunner` instances. Coordinates their interaction via the `ZuulEndPointRunner` that sits between them.
- `zuul-core/…/netty/server/BaseZuulChannelInitializer.java`: `addZuulFilterChainHandler()` assembles the full pipeline: `requestFilterChain → endPoint → responseFilterChain`. `getFilters()` retrieves sorted filters from `FilterLoader` by type and wraps them with passport-stamping bookend filters.
- `zuul-core/…/filters/ZuulFilter.java`: Plugin interface. Declares `filterType()`, `filterOrder()`, `applyAsync(I input): Observable<O>`, `needsBodyBuffered()`, `overrideStopFilterProcessing()`. Can be implemented via `@Filter` annotation or method override.
- `zuul-core/…/Filter.java`: `@Filter` annotation with `order`, `type`, `sync`, `category`, `constraints`, `@ApplyAfter`, `@ApplyBefore`. Runtime-retained, driving both the `FilterProcessor` and the `FilterLoader`.
- `zuul-core/…/FilterLoader.java` / `DynamicFilterLoader.java`: Plugin registry and runtime loader. `putFiltersForClasses(String[])` loads by class name. `getFiltersByType(FilterType)` returns a `SortedSet` ordered by `filterOrder()`.
- `zuul-processor/…/FilterProcessor.java`: `javax.annotation.processing.AbstractProcessor` that generates `META-INF/zuul/allfilters` at compile time, enabling zero-configuration filter discovery.
- `zuul-core/…/filters/endpoint/ProxyEndpoint.java`: The canonical ENDPOINT filter. Resolves the origin via `OriginManager`, acquires a pooled connection from `DefaultClientChannelManager`, forwards the request, and pipes the response back into the outbound filter chain.
- `zuul-core/…/netty/server/push/PushProtocol.java`: Enum with `WEBSOCKET` and `SSE` variants, each implementing `sendPushMessage`, `sendPing`, `goAwayMessage`. The push subsystem is a separate Netty channel pipeline layered on the same server.
- `zuul-discovery/…/DynamicServerResolver.java`: Wraps `DynamicServerListLoadBalancer` (Ribbon) backed by Eureka for dynamic origin discovery.

### Patterns Found

**Pipeline (Pipe-and-Filter) — Primary:**

Requests flow through exactly three stages in fixed order:
1. `INBOUND` — ordered array of pre-processing filters (authentication, routing decisions, throttling, body mutation).
2. `ENDPOINT` — single filter that proxies to an upstream origin (`ProxyEndpoint`).
3. `OUTBOUND` — ordered array of post-processing filters (response mutation, logging, compression).

`ZuulFilterChainRunner.runFilters()` is the pipe engine. Each filter receives a `ZuulMessage` (input type `I`) and returns an `Observable<O>` (output type `O`). Filters are ordered by `filterOrder()` integer and composed sequentially. `SessionContext` (a `HashMap`) is the shared state carrier threaded through every filter without copying. Filters can buffer the body (`needsBodyBuffered()`), short-circuit with `shouldStopFilterProcessing`, or be conditionally skipped via `shouldFilter()`. The dual `requestFilterChain` + `responseFilterChain` structure in `ZuulFilterChainHandler` makes the bidirectional pipe explicit.

**Microkernel (Plugin) — Secondary:**

`zuul-core` is the microkernel: it provides the Netty server, Netty channel pipeline wiring, filter execution engine, connection pooling, metrics, and SSL infrastructure. It contains no hardcoded routing, authentication, or business logic. All behaviour is contributed by filter plugins:

- **Plugin contract:** `ZuulFilter<I,O>` interface + `@Filter` annotation define the plugin API.
- **Plugin discovery:** `zuul-processor` generates a compile-time `META-INF/zuul/allfilters` manifest; `FilterProcessor` (annotation processor) collects all `@Filter`-annotated non-abstract classes.
- **Plugin loading:** `DynamicFilterLoader.putFiltersForClasses()` loads filters by class name at startup, building a `ConcurrentMap<FilterType, SortedSet<ZuulFilter>>` registry. Filters can be hot-loaded or reloaded at runtime.
- **Plugin instantiation:** `FilterFactory` interface (`DefaultFilterFactory`) creates filter instances, making instantiation injectable.
- **Built-in plugins:** `GZipResponseFilter`, `SurgicalDebugFilter`, `InboundPassportStampingFilter`, `OutboundPassportStampingFilter`, `MissingEndpointHandlingFilter` ship in `zuul-core` as default kernel plugins.
- **Extension classes:** Developers extend `HttpInboundSyncFilter`, `HttpOutboundSyncFilter`, `HttpSyncEndpoint`, or `BaseSyncFilter`/`BaseFilter` to add custom behaviour.

**API Gateway topology:**

Zuul terminates inbound connections (HTTP/1.1, HTTP/2, mutual TLS) and forwards them to registered upstream origins. `ProxyEndpoint` (the ENDPOINT filter) resolves the VIP via `OriginManager`, acquires a pooled Netty connection from `DefaultClientChannelManager` (backed by `DynamicServerResolver`/Eureka), forwards the request, and pipes the response back. The gateway also supports WebSocket and SSE push via `PushConnectionRegistry` and `PushChannelInitializer`. HAProxy protocol, ELB proxy protocol, TLS-PSK, and mutual TLS are all handled as Netty channel handlers registered before the Zuul filter chain.

**Async non-blocking I/O:**

Zuul 2's defining departure from Zuul 1 is full async/non-blocking operation on Netty event loops. `ZuulFilter.applyAsync()` returns `Observable<O>` (RxJava 1.x). `BaseSyncFilter` provides a synchronous wrapper, but the execution engine is async throughout. `EventExecutorScheduler` bridges cross-thread execution boundaries. This makes the pipeline non-blocking by default, a key quality attribute for a high-concurrency gateway.

## Architecture Styles Identified

### Pipeline (Primary)

The pipeline is the mandatory execution path for every request. There is no alternative control flow. The three stages (INBOUND → ENDPOINT → OUTBOUND) are structurally enforced by `FilterType` and `BaseZuulChannelInitializer.addZuulFilterChainHandler()`. Within each stage, filters are composed in strict integer order. The `SessionContext` map is the pipe state. Short-circuiting (`shouldStopFilterProcessing`) and conditional skipping (`shouldFilter()`) are the only divergences, both well-defined within the pipe-and-filter pattern.

This is confirmed at two levels:
- **External API:** Developers write filters classified as `INBOUND`, `ENDPOINT`, or `OUTBOUND` with explicit integer ordering.
- **Internal engine:** `ZuulFilterChainRunner.runFilters()` iterates `filters[]` sequentially with `AtomicInteger` position tracking, explicit `Pending` / `Complete` state per filter.

### Microkernel (Secondary)

`zuul-core` provides a microkernel stable enough that Netflix operates it internally with hundreds of custom filters not present in the open-source repo. The extension model is mature: compile-time annotation processing generates manifests, runtime class loading dynamically populates the filter registry, `FilterFactory` makes instantiation injectable, and `FilterConstraint` provides constraint-enforcement hooks. The `@Filter.ApplyAfter` / `@ApplyBefore` annotations even support declarative ordering relative to other named filters. This is a well-specified plugin architecture, not an ad-hoc extension mechanism.

### What Was Rejected

**Not Microservices:** Zuul is a single deployable JAR/application. Its five Gradle modules are build-time separations, not independently deployable services. `zuul-core` and `zuul-discovery` are libraries. There is no inter-process communication between Zuul components.

**Not Layered:** Packages are organized by concern (`netty/`, `filters/`, `origins/`, `context/`) but the primary organizing principle is the pipeline processing model, not horizontal tiers with strict unidirectional dependencies.

**Not Event-Driven:** Zuul uses RxJava `Observable` for async composition, but components do not communicate via an event broker. The model is synchronous request-response through the filter pipeline; asynchrony is a concurrency mechanism, not an architectural style.

**Not Modular Monolith:** The module separation is shallow (`zuul-core`, `zuul-processor`, `zuul-discovery`, `zuul-sample`). The primary decomposition is filter-level, not module-level.

## Quality Attributes

- **Performance:** Non-blocking Netty event loop; async filter execution via RxJava; native epoll/io_uring on Linux (`netty-transport-native-epoll`, `netty-transport-native-io_uring`); per-server connection pools (`PerServerConnectionPool`) sized per origin; JMH benchmarks in `zuul-core/src/jmh/`.
- **Extensibility:** Formal `@Filter` plugin model; compile-time annotation discovery; runtime class loading; `FilterFactory` DI hook; `@ApplyAfter`/`@ApplyBefore` declarative ordering; `SessionContextDecorator` for context enrichment.
- **Resilience:** Per-origin concurrency limits (`ZuulFilterConcurrencyExceededException`); retry logic in `ProxyEndpoint` with `RequestAttempts`; circuit-breaker-friendly `OriginThrottledException`; graceful shutdown via `ClientConnectionsShutdown`; idle-timeout and connection-expiry handlers.
- **Observability:** Spectator metrics (`HttpMetricsChannelHandler`, `ConnectionPoolMetrics`, `PerEventLoopMetricsChannelHandler`); `CurrentPassport` per-request lifecycle stamping with `PassportState` events; access log publishing; structured error categorization via `ZuulStatusCategory`.
- **Security:** Mutual TLS (`Http1MutualSslChannelInitializer`); TLS-PSK (`TlsPskHandler`); `StripUntrustedProxyHeadersHandler`; `SslClientCertCheck` extension point; ELB and HAProxy protocol support.
- **Scalability:** Stateless filter chain per request; `DynamicServerResolver` dynamically tracks Eureka server lists; per-event-loop connection pools prevent cross-thread contention.
- **Maintainability:** Single Gradle multi-project build; clear package separation by concern; extensive JUnit 5 test coverage alongside every production package; integration test module (`zuul-integration-test`) with `ZuulServerExtension` JUnit extension.
- **Interoperability:** HTTP/1.1, HTTP/2, WebSocket, SSE, TLS-PSK; Eureka service discovery; Ribbon client load-balancing; Archaius dynamic configuration; Spectator metrics compatible with Atlas.

## Classification Reasoning

The decisive question for Zuul's classification is: what is the primary organizing principle of the internal architecture?

The answer is the three-stage pipeline. `FilterType` defines exactly three values. `BaseZuulChannelInitializer.addZuulFilterChainHandler()` assembles them in mandatory order. `ZuulFilterChainRunner` executes each stage as a sequential filter array. `ZuulFilterChainHandler` coordinates the bidirectional (request + response) pipeline. Every line of Zuul's operational code — request reception, filter execution, proxy forwarding, response delivery — exists to implement this pipeline. A developer building on Zuul is primarily writing pipeline stages (filters), not plugins to a kernel.

The Microkernel label is fully merited as a secondary style. `zuul-core` is a genuine microkernel: it provides execution infrastructure and no domain logic. The plugin system has formal contracts (`ZuulFilter` interface, `@Filter` annotation), compile-time discovery (annotation processor), runtime loading (`DynamicFilterLoader`), and injectable instantiation (`FilterFactory`). This is not an escape hatch but the designed extension model through which Netflix operates its internal gateway.

The `gateway` qualifier captures the deployment topology: Zuul terminates client connections and proxies them to upstream origins. This is the explicit function stated in the README ("L7 application gateway") and implemented by `ProxyEndpoint` + `OriginManager` + `DefaultClientChannelManager`.

Confidence is 0.93 (high). The Pipeline + Microkernel combination is unambiguous: both patterns are simultaneously present, mutually reinforcing, and explicitly reflected in the public API that Zuul's adopters interact with. The small confidence gap accounts for the possibility that a Layered label could be argued for the internal package structure of `zuul-core`, though it would be a weaker signal than the pipeline execution model.
