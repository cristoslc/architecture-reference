## Summary

Dagster is a data orchestration platform whose architecture is organized around two reinforcing patterns: a **Pipeline (Pipe-and-Filter)** style as the primary execution model, and a **Microkernel** style as the structural and extensibility model. The pipeline pattern governs how data flows through graphs of computation (ops, assets); the microkernel pattern governs how the core framework is extended via a rich ecosystem of 67+ integration libraries, all registering themselves against a central kernel abstraction.

## Architecture Analysis

### Primary Style: Pipeline (Pipe-and-Filter)

The fundamental computational model of Dagster is a **directed acyclic graph (DAG)** of discrete processing units (Ops and Assets) connected by typed inputs and outputs. This is the textbook pipe-and-filter pattern:

**Evidence:**

- `python_modules/dagster/dagster/_core/execution/plan/plan.py` constructs an `ExecutionPlan` using `toposort` to order steps and maps `StepOutput` handles to `StepInput` sources (`FromStepOutput`, `FromConfig`, `FromDefaultValue`, etc.). The `_PlanBuilder` explicitly models the "pipes" (step dependencies) and "filters" (execution steps).
- `python_modules/dagster/dagster/_core/definitions/dependency.py` defines the `DependencyStructure` abstraction — mapping node outputs to node inputs — which is the formal pipe-and-filter wiring mechanism.
- `python_modules/dagster/dagster/_core/definitions/graph_definition.py` implements `GraphDefinition`, a composable container for nodes and their dependency wiring, using `toposort_flatten` for execution ordering.
- `python_modules/dagster/dagster/_core/definitions/assets/definition/assets_definition.py` exposes `AssetsDefinition`, the Software-Defined Asset model where lineage is expressed as an upstream-downstream dependency graph (assets as nodes, data dependencies as pipes).
- The `@op`, `@graph`, `@job`, and `@asset` decorator DSL (in `dagster/_core/definitions/decorators/`) provides the user-facing API for building pipe-and-filter graphs declaratively.

The pipeline pattern extends beyond simple job orchestration into the declarative automation system (`dagster/_core/definitions/declarative_automation/`) where assets are automatically materialized based on conditions evaluated against the asset graph.

### Secondary Style: Microkernel (Plugin)

Dagster's structural extensibility is built on a textbook **microkernel** pattern: a stable core kernel exposes abstract extension points (ABCs), and a large ecosystem of independently versioned plugin packages registers against the kernel.

**Evidence:**

**Kernel extension points (abstract base classes):**
- `dagster/_core/executor/base.py` — `Executor` ABC with `execute()` method; implementations include in-process, multiprocess, Celery, Dask, K8s variants
- `dagster/_core/launcher/base.py` — `RunLauncher` ABC; implementations include Docker, K8s, process-based launchers
- `dagster/_core/run_coordinator/base.py` — `RunCoordinator` ABC; implementations include `DefaultRunCoordinator` and `QueuedRunCoordinator`
- `dagster/_core/storage/io_manager.py` — `IOManagerDefinition` (extends `ResourceDefinition`); implementations swap how op outputs are stored and loaded
- `dagster/_core/storage/base_storage.py` — `DagsterStorage` ABC covering `EventLogStorage`, `RunStorage`, `ScheduleStorage`; implementations include SQLite, PostgreSQL, in-memory
- `dagster/_core/storage/compute_log_manager.py` — `ComputeLogManager` ABC; pluggable log capture backends
- `dagster.components.component.component.py` — `Component` ABC with `build_defs()` method; user-defined and third-party components contribute asset definitions to the system

**Plugin registration:**
Every integration library in `python_modules/libraries/dagster-*/` calls `DagsterLibraryRegistry.register()` in its `__init__.py`:
```python
# dagster-aws/__init__.py
from dagster_shared.libraries import DagsterLibraryRegistry
DagsterLibraryRegistry.register("dagster-aws", __version__)
```
This pattern is present in all 67 integration libraries (dagster-aws, dagster-dbt, dagster-k8s, dagster-databricks, dagster-snowflake, etc.).

**Dynamic user code loading:**
The gRPC server (`dagster/_grpc/server.py`) dynamically loads user-defined code repositories at runtime via `ReconstructableRepository` and `LoadableTargetOrigin`, allowing kernel extension without modifying the kernel itself.

**Component system:**
`dagster/components/core/component_tree.py` implements a `ComponentTree` that discovers, loads, and composes `Component` instances (from YAML declarations or Python files), building a tree of pluggable definition contributors. Each leaf component implements `build_defs()` returning `Definitions`, contributing to the final assembled system.

### Service Topology (Deployment Layer)

The Docker Compose example (`examples/deploy_docker/docker-compose.yml`) and Helm chart (`helm/dagster/`) reveal a **Service-Based** deployment topology:

- **dagster-webserver** — Starlette HTTP server, serves GraphQL API and React UI
- **dagster-daemon** — Long-running process handling scheduling, sensors, run monitoring, backfills, asset automation
- **dagster-user-code-server(s)** — One or more gRPC servers, each loading a code location; isolated from the control plane
- **Shared database** — PostgreSQL (or SQLite) for run storage, event log, schedule storage

These services share a single database schema and communicate via gRPC (to user code servers) and shared storage. This is **not** microservices (no independent data ownership, no autonomous deployment units) but rather a **service-based monolith** where the services are coordinated deployment units of a single logical system. The primary architectural identity is Pipeline + Microkernel at the code level; the service topology is a deployment concern.

### Layered Structure (Supporting, not Primary)

A clear UI → API → Core → Storage layering exists:
- **UI Layer:** React TypeScript SPA (`js_modules/app-oss/`)
- **API Layer:** GraphQL schema (`dagster-graphql`) exposing query, mutation, subscription roots
- **Core Layer:** `dagster/_core/` containing execution engine, definitions, asset graph, workspace management
- **Storage Layer:** Pluggable implementations of `RunStorage`, `EventLogStorage`, `ScheduleStorage`

This layering supports and organizes the system but is not the dominant architectural identity.

### What Was Considered and Rejected

| Style | Verdict | Rationale |
|-------|---------|-----------|
| Microservices | Rejected | Services share a single DB schema; no independent data ownership; coordinated deployment not autonomous services |
| Event-Driven | Rejected | Dagster orchestrates event-triggered workflows and persists `DagsterEvent` records to an event log, but internal component communication uses synchronous API calls and shared database state, not message brokers. The event log is a read model, not a bus. |
| CQRS | Rejected | The event log pattern has superficial CQRS resemblance (events written on execution, queries read from storage), but the system is not structured around command/query segregation as a design principle |
| Serverless | Rejected | Dagster can launch serverless workloads (via launchers) but the orchestrator itself runs persistent long-lived processes |
| Domain-Driven Design | Rejected | No evidence of bounded contexts, aggregates, or DDD tactical patterns as architectural organizing principles |
| Hexagonal Architecture | Rejected | While storage adapters exist, the system does not organize around a ports-and-adapters philosophy; the layering is conventional, not hexagonal |

## Quality Attributes

- **Extensibility:** The microkernel pattern enables deep extensibility across executors, launchers, IO managers, storage backends, and integration components without modifying the core
- **Modularity:** 67+ independently versioned packages with clear separation between kernel (`dagster`), shared utilities (`dagster-shared`), and integration plugins; internal modules use underscore-prefixed naming to denote private APIs
- **Observability:** First-class event log (all execution events captured as structured `DagsterEvent` records), asset materialization tracking, run history, compute log capture
- **Reliability:** Retry policies, daemon-based monitoring, run health checks, backfill management, and the `QueuedRunCoordinator` for concurrency control
- **Portability:** Pluggable deployment targets (process, Docker, Kubernetes, Celery) via the `RunLauncher` and `Executor` extension points
- **Scalability:** Horizontal scaling via multiple user code gRPC servers (each a separate code location), K8s-native executor, and queue-based run coordination
