## Style Rationales

**Modular Monolith (primary, 0.91):** Radarr is a single ASP.NET Core process bootstrapped in `NzbDrone.Host/Bootstrap.cs` using DryIoc (`AutoAddServices` across 5 assemblies: Host, Core, SignalR, Api.V3, Http). The solution (`Radarr.sln`) compiles ~25 projects into one binary. `NzbDrone.Core` contains 1,538 `.cs` files organized as vertical domain slices: `Movies/`, `Download/`, `Indexers/`, `IndexerSearch/`, `Notifications/`, `MediaFiles/`, `Queue/`, `History/`, `CustomFormats/`, `DecisionEngine/`, `ImportLists/`, etc. All domains share a single SQLite or PostgreSQL database accessed via `BasicRepository<T>` (Dapper-based), with 150 ordered migration scripts under `Datastore/Migration/`.

**Microkernel (secondary, 0.88):** The `ThingiProvider` framework (`NzbDrone.Core/ThingiProvider/`) provides a generic plugin bus. Every extension category — indexers, download clients, notifications, import lists, metadata sources — implements `IProvider`. The generic `ProviderFactory<TProvider, TProviderDefinition>` resolves implementations from the DryIoc container at runtime and publishes lifecycle events (`ProviderAddedEvent<T>`, `ProviderUpdatedEvent<T>`, `ProviderDeletedEvent<T>`) through `EventAggregator`. Concrete plugin counts: ~10 indexer types (Newznab, Torznab, HDBits, FileList, Nyaa, PassThePopcorn, IPTorrents, etc.), ~21 download client implementations (qBittorrent, Transmission, Deluge, SABnzbd, rTorrent, uTorrent, Flood, Aria2, NZBGet, etc.), ~26 notification targets (Discord, Slack, Email, Webhook, Pushover, Telegram, Plex, Apprise, Gotify, etc.), ~14 import list sources (TMDb, Trakt, IMDb, CouchPotato, Plex, Simkl, Radarr lists, etc.). Adding a new plugin requires zero changes to the factory core.

## Quality Attributes

| QA | Evidence |
|---|---|
| **Modularity** | 25+ domain folders in `NzbDrone.Core` each with own Models/Services/Repositories/Events; `ThingiProvider` contracts enforce separation between core and plugins; DryIoc auto-discovery wires all implementations without manual registration |
| **Evolvability** | 150 ordered DB migrations; new indexer/download-client/notification plugins require zero core changes; `IHandle<TEvent>` allows new event reactions without modifying publishers; `CustomFormats` ruleset decouples quality scoring from release parsing logic |
| **Observability** | NLog throughout; SignalR `MessageHub` pushes real-time command/event status to React UI; 33 `HealthCheck/Checks/` covering indexer status, download client config, root folder validity, API key, import list status, naming config, mount points |
| **Deployability** | Cross-platform .NET (Windows service, Linux daemon, Docker); SQLite default, PostgreSQL optional; Kestrel embedded web server; no IIS dependency; single binary output |
| **Fault Tolerance** | `CommandExecutor` catches per-command exceptions and marks commands failed without crashing worker threads; `ProviderFactory.Handle(ApplicationStartedEvent)` removes missing implementations on startup rather than throwing; `DownloadEventHub` tracks download failures via `DownloadFailedEvent` |
| **Scalability** | Single-process, single-host design; vertical scaling only; PostgreSQL support allows external DB for home-lab multi-container setups; `CommandQueue` scales worker threads between 2–10 based on load |

## Domain

Automated movie collection manager and download orchestrator. Core domains: movie metadata (TMDb), download decision engine, quality/custom format profiles, indexer search, download client integration, media file import and rename, collection management.

## Production Context

- Self-hostable via Docker (cross-platform .NET binary); no cloud-managed offering — purely self-hosted
- Single ASP.NET Core process with embedded Kestrel; React SPA served from the same process
- Integrates with external Usenet/torrent indexers and download clients over HTTP; no internal async broker
- Sister project to Lidarr, Sonarr, Prowlarr (same NzbDrone codebase lineage); at the ecosystem level the *arr suite forms a Service-Based architecture, but each member is an independent Modular Monolith
