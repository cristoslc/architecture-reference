# Architecture Report: umami

**Date:** 2026-03-11
**Source:** https://github.com/umami-software/umami
**Model:** claude-sonnet-4-6
**Method:** deep-analysis
**Classification:** Layered
**Confidence:** 0.95

---

## Summary

Umami is a privacy-focused, open-source web analytics platform built as a Next.js 15 full-stack application. It is a single deployable monolith organized strictly into horizontal technical layers: presentation (React/App Router pages), API routing (Next.js route handlers), business logic and utilities (`src/lib`), permissions (`src/permissions`), and a data access layer split across two sub-stacks (`src/queries/prisma` for relational writes/reads and `src/queries/sql` for analytics queries targeting either PostgreSQL or ClickHouse). All data flows synchronously through these layers. Optional Kafka integration provides an asynchronous path for high-volume analytics event ingestion into ClickHouse, but this is a scaling extension to the ingestion path—not a fundamental architectural pattern structuring the system.

---

## Architecture Styles

**Primary: Layered**

The codebase has a canonical textbook Layered architecture. Every request flows downward through:

1. **Presentation layer** — `src/app/(main)/` contains all React page and layout components, client-side state via Zustand (`src/store/`).
2. **API/Controller layer** — `src/app/api/` contains Next.js App Router route handlers. Each handler: parses the request (`parseRequest`), validates with Zod, checks permissions, delegates to the query layer, and returns a response. No business logic lives here.
3. **Permissions layer** — `src/permissions/` contains named authorization functions (`canViewWebsite`, `canUpdateWebsite`, etc.) that encapsulate access rules, called explicitly from every API route.
4. **Business logic / utility layer** — `src/lib/` contains authentication (`auth.ts`), cryptography (`crypto.ts`), request parsing (`request.ts`), detect / geo enrichment (`detect.ts`), date utilities, constants, and infrastructure adapters (Prisma, ClickHouse, Kafka, Redis clients).
5. **Data access layer (DAL)** — `src/queries/` is divided into two namespaces:
   - `prisma/` handles entity CRUD (users, websites, teams, reports, segments, links, pixels) against PostgreSQL via Prisma ORM.
   - `sql/` handles all analytics queries (events, sessions, pageviews, reports) against PostgreSQL or ClickHouse, selected at runtime by `src/lib/db.ts::runQuery()`.

The `runQuery` abstraction is the key architectural pivot: it inspects environment variables (`CLICKHOUSE_URL`) and dispatches to the appropriate backend, making the storage engine swappable without changing the API layer.

**No secondary style rises to architectural status.** The optional Kafka path in `saveEvent` (ClickHouse path only) is a write optimization for the data ingest pipeline — it does not mediate communication between application modules, there are no consumers inside the application, and the system operates identically without it. Event-Driven is therefore not a style of this architecture.

## Evidence

### Layer organization — direct code evidence

```
src/
  app/
    (main)/          # Presentation layer — React pages
    api/             # API/Controller layer — Next.js route handlers
  lib/               # Business logic and infrastructure utilities
  permissions/       # Authorization rules
  queries/
    prisma/          # Entity DAL (relational ORM)
    sql/             # Analytics DAL (raw SQL / ClickHouse)
  store/             # Client-side state (Zustand)
```

### Consistent request flow across all API routes

Every API handler follows the same pattern (from `src/app/api/websites/[websiteId]/metrics/route.ts` and `realtime/[websiteId]/route.ts`):
```typescript
const { auth, query, error } = await parseRequest(request, schema);  // lib/request
if (error) return error();
if (!(await canViewWebsite(auth, websiteId))) return unauthorized();  // permissions
const data = await getSessionMetrics(websiteId, ...);                 // queries/sql
return json(data);                                                    // lib/response
```

### Storage engine runtime dispatch

`src/lib/db.ts::runQuery()` inspects `process.env.CLICKHOUSE_URL` at call time and routes to the appropriate implementation:
```typescript
export async function runQuery(queries: any) {
  if (process.env.CLICKHOUSE_URL) {
    if (queries[KAFKA]) return queries[KAFKA]();
    return queries[CLICKHOUSE]();
  }
  const db = getDatabaseType();
  if (db === POSTGRESQL) return queries[PRISMA]();
}
```

Every `saveEvent`, `createSession`, `getWebsiteStats`, etc. passes a `{ [PRISMA]: ..., [CLICKHOUSE]: ... }` object to this dispatcher.

### Single deployable unit

- One `package.json`, one `Dockerfile`, one `docker-compose.yml`
- `docker-compose.yml` defines exactly two services: `umami` (the app) and `db` (PostgreSQL)
- ClickHouse, Kafka, and Redis are optional — the base system runs on one app + one database

### Prisma schema — shared relational model

`prisma/schema.prisma` defines a single shared schema covering Users, Websites, Sessions, WebsiteEvents, EventData, SessionData, Teams, TeamUsers, Reports, Segments, Revenue, Links, and Pixels. All modules read from and write to this unified schema with no per-domain isolation.

### Kafka as ingestion optimization, not architecture style

`src/queries/sql/events/saveEvent.ts` (ClickHouse branch):
```typescript
if (kafka.enabled) {
  await sendMessage('event', message);   // async Kafka write
} else {
  await insert('website_event', [message]);  // direct ClickHouse insert
}
```
Kafka is purely a write-side performance option for the ClickHouse storage backend. No other module publishes or consumes from Kafka; there are no event-driven subscribers inside the application codebase.

## Architecture Qualifiers

- **Dual-storage engine:** PostgreSQL (Prisma) for entity management; ClickHouse for high-volume analytics queries. The same query layer wraps both via `runQuery`.
- **Read/Write path separation in the analytics DAL:** writes go through `saveEvent`/`createSession`; reads go through named query functions (`getWebsiteStats`, `getPageviewMetrics`, etc.). This is a CQRS-like read/write split at the DAL level, not a full CQRS implementation (no command objects, no separate read models, no event sourcing).
- **Optional Kafka buffer:** The ClickHouse write path can route through Kafka for throughput decoupling, but this is purely operational — the app processes events synchronously otherwise.
- **Optional Redis for auth caching:** `src/lib/auth.ts` uses Redis (when `REDIS_URL` is set) to store short-lived auth keys, reducing JWT payload size.

## What This Architecture Is NOT

**Not Modular Monolith:** There are no enforced domain module boundaries. The directory structure is purely technical (`lib/`, `queries/`, `permissions/`), not domain-oriented (no `analytics/`, `users/`, `teams/` modules with encapsulated internals). Code imports freely across directories. No module facade interfaces.

**Not Event-Driven:** Internal communication is synchronous request/response throughout. Kafka is present only as an optional write buffer on the ClickHouse path — it is not a message bus mediating between application components.

**Not Microservices:** One process, one repository, one deployment artifact.

**Not Service-Based:** No discrete deployable service partitions within the system.

## Classification Reasoning

Umami scores high confidence as Layered (0.95) because:

1. The layer structure is explicit and consistent — every API route follows the same parse → authorize → query → respond pattern.
2. The permissions module is a distinct architectural layer, not just utility functions, separating authorization concerns from both the controller and query layers.
3. The dual-DAL design (`prisma/` vs `sql/`) is a deliberate architectural split driven by the different access patterns of entity CRUD vs. analytical aggregation — a classic read-model optimization within a Layered architecture.
4. The deployment model is unambiguously a single-process monolith.
5. Kafka's presence does not elevate the classification to Event-Driven because it only affects a conditional branch in one data-write function and has no consumers in the application codebase.
