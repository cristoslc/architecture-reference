# Architecture Report: orbit/orbit

**Date:** 2026-03-11
**Source:** https://github.com/orbit/orbit
**Classification:** Pipeline + Microkernel
**Confidence:** 0.88
**Method:** deep-analysis

## Summary

Orbit is a distributed actor framework built by Electronic Arts, designed to host _addressable_ objects (actors) across a mesh of server nodes. The server routes invocation messages through a fixed, ordered sequence of pipeline steps; the extension points in directories, auth systems, and meter registries constitute a microkernel plugin surface. The overall deployment topology is a horizontal cluster of homogeneous server nodes connected by gRPC streams, with etcd-backed distributed directories for node and addressable placement state.

## Architecture Classification

### Primary Style: Pipeline

The server's core processing model is an explicit, ordered pipeline applied to every inbound and outbound message. `PipelineSteps` assembles a fixed array of six `PipelineStep` implementations:

```
IdentityStep → RoutingStep → PlacementStep → VerifyStep → AuthStep → TransportStep
```

`Pipeline` feeds messages into a `RailWorker` (a buffered channel with configurable parallelism, default 32 rails, 10 000-message buffer). Each message flows through the step chain via a `PipelineContext` that calls `context.next(msg)` to advance. Inbound traversal goes up; outbound goes down. This is a textbook bidirectional message pipeline: fixed stages, no dynamic composition, each stage transforms the message or terminates it with an error.

Key evidence:
- `Pipeline.kt` — `RailWorker`-backed channel, `onMessage` dispatches to `PipelineContext`
- `PipelineSteps.kt` — immutable ordered `Array<PipelineStep>`, comment "inbound is up, outbound is down"
- `PipelineStep.kt` — interface with `onInbound(context, msg)` / `onOutbound(context, msg)`
- `PlacementStep` — resolves actor location (locateOrPlace) on inbound invocation requests
- `RoutingStep` — computes shortest-path route via Dijkstra on the cluster graph on outbound
- `TransportStep` — final delivery to a connected client or remote mesh node via gRPC

### Secondary Style: Microkernel

`OrbitServerConfig` exposes three pluggable extension points via `ExternallyConfigured<T>`:

- `NodeDirectory` — defaults to `LocalNodeDirectory`; swap to `EtcdNodeDirectory` for multi-node
- `AddressableDirectory` — defaults to `LocalAddressableDirectory`; swap to `EtcdAddressableDirectory`
- `MeterRegistry` — defaults to `LocalMeterRegistry`; swap to `PrometheusMetrics`

A fourth hook, `containerOverrides: ComponentContainerRoot.() -> Unit`, allows arbitrary DI registrations to be injected after the default container is built.

On the client side, `OrbitClientConfig` exposes:
- `AddressableConstructor` — defaults to `DefaultAddressableConstructor` (reflection); overridden by the Spring plugin via `SpringAddressableConstructor`
- `NodeLeaseRenewalFailedHandler` — pluggable failure policy

The custom `ComponentContainer` (`orbit-util/di/ComponentContainer.kt`) acts as the service locator that resolves all registered singletons at runtime.

### Supporting Observations

**Actor model runtime:** The client implements a virtual actor pattern. `ExecutionSystem` manages a `ConcurrentHashMap<AddressableReference, ExecutionHandle>` of live actors, activating on demand and deactivating on TTL expiry or lease loss. Actors are located on server nodes via lease-based placement and communicated with through the server pipeline. This is the _application model_ built on top of the pipeline infrastructure, not a separate architectural style.

**Cluster mesh topology:** Multiple `OrbitServer` instances form a mesh. Each server node connects to peer mesh nodes via `RemoteMeshNodeManager`, which discovers peers through the shared `ClusterManager` (backed by `NodeDirectory`). The `ClusterManager` builds a directed Dijkstra graph (`jgrapht`) of node visibility for routing. The docker-compose and Helm charts show 2+ server nodes behind an HAProxy load balancer with shared etcd stores for directories.

**gRPC streaming:** The `Connection` service (`connection.proto`) defines a single `OpenStream(stream MessageProto) returns (stream MessageProto)` RPC — a persistent bidirectional stream per client connection. All invocation traffic flows through this stream; node and addressable management use separate unary RPCs.

## Confidence Notes

The pipeline classification is high-confidence (0.90): the six-step ordered chain is the architectural spine of every message processed by the server. The microkernel classification is moderate-confidence (0.82): the plugin surface is genuine and the swap between local/etcd/prometheus backends is the primary operational configuration axis, but the core is not as thin a kernel as frameworks like Eclipse or OSGi — several concerns (auth, routing, placement) are baked into named pipeline steps rather than plugins. The composite score is 0.88.
