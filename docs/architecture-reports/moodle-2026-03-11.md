## Summary

Moodle is a mature open-source Learning Management System (LMS) that is the world's most widely deployed eLearning platform. Architecturally it is a Microkernel system: a stable, capability-rich core runtime provides the execution environment, database abstraction, context/permission system, plugin management, event bus, hook system, caching, task scheduling, and web service infrastructure. Every functional domain — course formats, activity modules, authentication providers, grading tools, themes, report engines, repository adapters, communication backends, analytics predictors, AI providers, and dozens more — is delivered through a richly typed plugin hierarchy of 49 distinct plugin types. All plugins and the core ship as a single deployable PHP process sharing one relational database, making the outer container also classifiable as a Modular Monolith. The runtime layers — HTTP routing, application logic, data access — are horizontally separated in a fashion consistent with Layered Architecture.

## Architecture Styles

### Primary: Microkernel (0.93)

The dominant structural characteristic is an extensible kernel that defines and manages an elaborate plugin ecosystem.

**Plugin type system (49 types):** `public/lib/classes/plugininfo/` contains 49 distinct plugin type descriptors: `mod`, `auth`, `enrol`, `theme`, `block`, `format`, `report`, `gradereport`, `filter`, `editor`, `repository`, `portfolio`, `qtype`, `qbehaviour`, `qformat`, `qbank`, `local`, `tool`, `availability`, `calendartype`, `dataformat`, `search`, `message`, `mlbackend`, `aiprovider`, `aiplacement`, `communication`, `smsgateway`, `paygw`, `contenttype`, and many more. Each type has a well-defined PHP base class and lifecycle contract.

**Subplugin support:** `mod`, `editor`, `tool`, and `local` plugin types support their own subplugin tiers (e.g., `mod/forum` hosts `forumreport` subplugins via `db/subplugins.json`), giving the kernel three levels of extensibility depth.

**Plugin lifecycle management:** `core\plugin_manager` (public/lib/classes/plugin_manager.php) tracks status (`NODB`, `UPTODATE`, `NEW`, `UPGRADE`, `DELETE`), source (`std` vs `ext`), and version of every installed plugin. Upgrade sequences are coordinated across all plugins via `db/upgrade.php` and the DDL abstraction layer (`public/lib/ddl/`).

**Kernel registration contracts:** Every plugin registers against the kernel via standardised db/ declaration files: `access.php` (capabilities), `events.php` (event observers), `hooks.php` (PSR-14 hook callbacks), `services.php` (web service functions), `caches.php` (cache definitions), `tasks.php` (scheduled/adhoc tasks), `messages.php` (message providers).

**Hook system (PSR-14):** `core\hook\manager` implements `EventDispatcherInterface` and `ListenerProviderInterface` from PSR-14. The kernel defines typed hook classes (e.g., `\core_group\hook\after_group_created`) that plugins subscribe to via `db/hooks.php`. The core `hooks.php` file registers 54 hook callback bindings. This is the modern successor to the older observer-based event system.

**Event system (261 event classes):** `public/lib/classes/event/` contains 261 typed event classes all extending `core\event\base`. Plugins declare observers in `db/events.php`. Events carry structured context (component, action, target, objecttable, objectid, contextid, userid) and are persisted to the log store.

**Core subsystems as kernel services:** The core runtime provides: capability/context system (`accesslib.php`, `public/lib/classes/context/`), DML database abstraction (`public/lib/dml/` — MySQL/MariaDB/Aurora/PostgreSQL/MSSQL), MUC caching framework (`public/cache/` — APCu/Redis/file/session/static backends), task scheduler (`public/lib/classes/task/` — 67 tasks), session management (`public/lib/classes/session/` — DB/file/Redis/Memcached), string manager, file storage, web service framework (`public/webservice/`), external API framework (`public/lib/classes/external/`), and DI container (`core\di` backed by PHP-DI 7).

**AMD JavaScript modules as frontend extension points:** `public/lib/amd/src/` provides a RequireJS/AMD module registry so that plugins can declare and consume JavaScript extension points following the same contract as PHP plugins.

### Secondary: Modular Monolith (0.88)

Despite the 49-type plugin system, all code is packaged and deployed as a single PHP process against a single relational schema.

**Single deployable unit:** One `composer.json` manifests all dependencies; one `config.php` configures the single application instance. There are no separate services, containers, or independently deployable binaries.

**Shared relational database:** All subsystems and plugins share one database server (MySQL, MariaDB, Aurora, PostgreSQL, or MSSQL) via the DML abstraction. The DDL layer (`database_manager`, `sql_generator`) manages a unified schema across all plugin installs.

**Module-scale domain isolation:** Each plugin type occupies a distinct filesystem subtree (`mod/forum/`, `auth/ldap/`, `enrol/cohort/`, etc.) with its own `classes/`, `db/`, `lang/`, `templates/`, and `tests/`. This physical isolation matches modular monolith conventions even without independent deployability.

**Subsystem namespacing:** Core subsystems use Frankenstyle namespacing (`core_*`, `mod_*`, `auth_*`) that provides logical module boundaries enforced by the autoloader, without breaking the single-process deployment.

### Secondary: Layered (0.82)

The core and each plugin are internally organised in three horizontal layers.

**Presentation layer:** Mustache templates (`templates/`), AMD JavaScript modules (`amd/`), Boost/Classic themes, output renderers (`core_renderer`, `output/` classes), AJAX handlers, and the RESTful web service endpoints (`public/webservice/rest/`).

**Application layer:** Controllers are thin entry scripts (`index.php`, `view.php`), with business logic in `lib.php`, `locallib.php`, and namespaced `classes/` services. The task scheduler, backup/restore engine, and external API framework operate at this tier.

**Data layer:** DML (`moodle_database` → DBMS-specific drivers), persistent model base (`core\persistent`), and the legacy procedural data functions in `public/lib/datalib.php` and `public/lib/accesslib.php` (5,193 lines).

**Slim 4 HTTP kernel (new in 5.x):** `composer.json` includes `slim/slim: 4.13.0` and `nikic/fast-route`. `public/lib/classes/router/` introduces a modern HTTP routing layer with middleware, request validation, and controller dispatching — a Slim-based PSR-7/PSR-15 stack layered above the legacy `setup.php` bootstrap.

## Ruled Out

**Microservices:** Single deployable process. All plugins share one database, one PHP process, one Composer manifest. No independent deployability, no inter-service communication via network calls.

**Event-Driven (primary):** 261 event classes exist and are heavily used, but they are synchronous in-process notifications (persisted to a log store) not decoupled message-broker events between independent services. The hook system is similarly synchronous. Neither constitutes an event-driven integration architecture.

**Hexagonal/Ports-and-Adapters:** There is no ports-and-adapters boundary. Database access is shared globally via `$DB` (a DML driver injected at bootstrap); there is no domain layer isolated from infrastructure behind an interface boundary in the Hexagonal sense.

**Domain-Driven Design:** No bounded contexts, no aggregate roots, no ubiquitous language segmentation per domain. The data model is a unified relational schema shared across all plugins.

**CQRS:** Read and write paths use the same DML abstraction. No command/query bus or separate read/write data models.

**Pipeline:** No data pipeline processing model; request handling is request-response, not a sequence of filters/stages applied to a data stream.

**Serverless:** Single-server PHP application; no function-as-a-service decomposition.

## Classification Confidence

**0.93** — Confidence is high. Moodle explicitly names itself "Modular Object-Oriented Dynamic Learning Environment" and the entire codebase is structured around a plugin type registry with well-defined extension contracts. The PSR-14 hook system, 49 plugin type classes, and plugin lifecycle manager are unambiguous Microkernel indicators. The shared single-process deployment and unified database confirm the Modular Monolith secondary style. The horizontal layers (presentation/application/data) are clearly separated and the new Slim 4 router solidifies the Layered secondary style.
