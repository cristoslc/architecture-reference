# envoy — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/envoyproxy/envoy
**Classification:** Microkernel + Layered
**Confidence:** 0.93

## Summary

Envoy is a cloud-native L7 proxy built around a Microkernel (Plugin) architecture: a minimal event-driven core — the libevent-backed dispatcher and threading model — exposes a rich set of typed extension points through a factory-registry system, while all functional logic (HTTP filters, network filters, access loggers, tracers, transport sockets, cluster implementations, health checkers, stat sinks, and more) is implemented as independently selectable plugins. Internally, the codebase enforces a strict Layered discipline: pure-virtual public interfaces (`envoy/`), core implementation (`source/common/`), server lifecycle (`source/server/`), and extensions (`source/extensions/`) occupy clearly separated strata. Envoy ships as a single deployable binary; the architecture is explicitly not a microservices decomposition.

### Key Architectural Files

- `envoy/registry/registry.h`: `FactoryRegistry<Base>` (template-based static registry), `RegisterFactory<T, Base>` (static self-registration via C++ static initializers), `FactoryCategoryRegistry` (organizes factories by category string such as `envoy.filters.http`). This is the kernel's extension bus.
- `envoy/server/filter_config.h`: Defines the plugin contracts — `NamedListenerFilterConfigFactory`, `NamedHttpFilterConfigFactory`, `NamedNetworkFilterConfigFactory` — each extending `Config::TypedFactory` with a required `category()` string and a `createFilterFactoryFromProto()` pure virtual method.
- `source/common/event/dispatcher_impl.{cc,h}`: `DispatcherImpl` wraps libevent's `event_base`; every worker thread owns one dispatcher. This is the runtime kernel — all I/O and timer events are dispatched through it.
- `source/common/http/conn_manager_impl.{cc,h}`: `ConnectionManagerImpl` is a `Network::ReadFilter` that instantiates and chains HTTP filter plugins at request time, effectively acting as the microkernel's orchestrator for the HTTP processing pipeline.
- `source/extensions/dynamic_modules/dynamic_modules.{cc,h}`: `DynamicModule` wraps `dlopen`/`dlsym`, providing runtime plugin loading in addition to compile-time static registration. `newDynamicModule()` loads a shared object by path; `newDynamicModuleByName()` resolves by name.
- `source/extensions/extensions_build_config.bzl`: 650-line Bazel map of extension label → build target. Extensions can be selectively compiled out; only `all_extensions.bzl` entries are mandatory.
- `source/extensions/extensions_metadata.yaml`: Each extension declares `categories`, `security_posture` (`robust_to_untrusted_downstream`, `requires_trusted_downstream_and_upstream`, etc.), and `status` (`stable`, `alpha`, `wip`).
- `REPO_LAYOUT.md`: Explicitly documents the `envoy/` (public interface headers, "almost entirely 100% abstract classes") / `source/common/` (kernel library code) / `source/server/` (server runtime) / `source/extensions/` (all plugins) layering.

### Patterns Found

**Microkernel (Plugin) — Primary:**

The entire Envoy extension system is structured around a factory-registry microkernel:

1. **Extension point contracts:** Each extension category defines a pure-virtual `Named*ConfigFactory` interface in `envoy/server/filter_config.h` or analogous headers. There are over 20 named extension categories: HTTP filters, network filters, listener filters, UDP listener filters, access loggers, tracers, clusters, transport sockets, health checkers, stat sinks, load balancing policies, resource monitors, retry policies, rate limit descriptors, gRPC credential plugins, compression codecs, request ID generators, and more.

2. **Factory registry:** `FactoryRegistry<Base>` is a process-global static map keyed by extension name string. The `RegisterFactory<T, Base>` template uses a static initializer to self-register each plugin at program start. Individual extensions invoke `REGISTER_FACTORY(Impl, Base)` at the bottom of their config files.

3. **Compile-time configurability:** `extensions_build_config.bzl` allows entire extension categories or individual extensions to be stripped from the build at compile time, making the "kernel" even smaller in constrained deployments.

4. **Runtime dynamic modules:** `DynamicModule` via `dlopen`/`dlsym` supports loading plugins from shared objects at runtime, extending the microkernel contract beyond compile-time registration.

5. **xDS as configuration microkernel:** The xDS API (CDS, EDS, LDS, RDS, SDS, ADS, RTDS, ECDS) dynamically injects configuration into the running kernel — listeners, filter chains, routes, clusters, and even extension configs can be pushed at runtime without binary changes, keeping the static core agnostic of the deployment topology.

6. **70+ HTTP filters, 20+ network filters:** All functional L7 logic — authentication, authorization, rate limiting, circuit breaking, fault injection, header manipulation, compression, tracing, WebAssembly sandboxed user code, external processing — is implemented as pluggable filters, not baked into the core.

**Layered Architecture — Strong Secondary:**

The repository layout enforces strict horizontal layering:

- **Layer 0 — Public Interfaces (`envoy/`):** Near-exclusively pure-virtual C++ classes. Defines every contract the kernel and extensions use. Nothing at this layer has implementation details.
- **Layer 1 — Core Kernel (`source/common/`):** Implements the event loop (`event/`), HTTP codec (`http/`), network abstraction (`network/`), upstream cluster management (`upstream/`), routing (`router/`), stats (`stats/`), buffer management, and all other cross-cutting facilities. This layer is designed to be embeddable as a library.
- **Layer 2 — Server Runtime (`source/server/`):** Adds server-specific lifecycle: hot restart, worker thread pool, overload manager, watchdog, admin API, option parsing, listener management.
- **Layer 3 — Extensions (`source/extensions/`):** Plugin implementations. Each extension is fully self-contained within its namespace, depends only on Layer 0 interfaces and Layer 1 utilities, and is compiled in or out independently.

**Event-Driven Execution Model (Implementation Mechanism, Not Primary Style):**

The dispatcher (libevent event loop) is the mechanism through which all async I/O and timer callbacks execute on each worker thread. The `Dispatcher::post()` method is used for cross-thread dispatch. This is an implementation detail of how the kernel processes requests, not a primary architectural style that defines how the system is organized.

### What This Architecture Is Not

- **Not Microservices:** Envoy is a single deployable binary. All extensions run in-process on shared memory. There is no inter-service communication between "filter services."
- **Not Modular Monolith:** The defining characteristic is the plugin registry contract and the compile-time/runtime configurability of components, not merely module organization.
- **Not Pipeline:** While requests pass through an ordered filter chain within a single proxy instance, Envoy itself is a proxy *node* in a service mesh pipeline, not a pipeline system. The filter chain is an internal execution mechanism of the microkernel, not the system's primary organizing principle.
- **Not Event-Driven (primary):** The event loop is a runtime execution primitive, not the primary decomposition axis. The architecture does not decompose around event types or message topics.

## Quality Attributes

- **Extensibility:** The factory-registry microkernel makes adding new filter types, protocols, or backends a matter of implementing an interface and registering — no core changes required.
- **Performance:** libevent-backed single-threaded dispatchers per worker, zero-copy buffer management, connection pool reuse, and in-process filter execution keep latency low.
- **Observability:** Built-in stats infrastructure (`stats/`), pluggable tracers (Zipkin, Datadog, OTel, X-Ray), access loggers (OTel, Fluentd, gRPC), and an admin API provide deep telemetry.
- **Reliability:** Hot restart (`source/server/hot_restart_impl.{cc,h}`), overload manager, watchdog, circuit breakers, and retry extensions.
- **Security:** Transport socket plugins (TLS, ALTS), mTLS, JWT authentication filter, external authorization (`ext_authz`), RBAC filter, WASM sandboxed extensions; per-extension security posture declared in metadata.
- **Configurability:** xDS APIs allow full dynamic reconfiguration without binary restart; extensions are selectively compiled in or out at build time.
- **Portability:** Abstracted platform layer; builds on Linux, macOS, Windows; mobile SDK in `mobile/`.
