# traefik — Architecture Classification Report

**Date:** 2026-03-11
**Repo:** https://github.com/traefik/traefik
**Classification:** Microkernel + Pipeline
**Confidence:** 0.91

## Summary

Traefik is a cloud-native HTTP reverse proxy and load balancer written in Go (~92%), deployed as a single self-contained binary. Its primary architectural expression is the Microkernel pattern: a stable core manages entrypoints, configuration watching, and routing lifecycle, while all external integrations (providers) and traffic-shaping logic (middlewares) are pluggable modules attached through formal interfaces. The request-handling path is a textbook Pipeline (pipe-and-filter) in which each inbound connection passes sequentially through an ordered middleware chain before reaching a backend service. The combination of dynamic configuration discovery, runtime-loadable plugins, and a composable middleware chain is the defining structural signature of the system.

### Key Architectural Files

- `cmd/traefik/traefik.go`: `setupServer()` is the composition root — it instantiates the ProviderAggregator, TLS manager, ACME provider, ObservabilityMgr, TCP/UDP entrypoints, plugin builder, configuration watcher, and all listeners. This is the only place all wiring happens.
- `pkg/server/server.go`: `Server` struct holding `tcpEntryPoints`, `udpEntryPoints`, `configurationWatcher`, and `observabilityMgr`. Start/Wait/Stop/Close lifecycle. The core orchestrator knows nothing about routing rules or backends.
- `pkg/server/configurationwatcher.go`: `receiveConfigurations` → channel → `applyConfigurations`. A two-stage reactive pipeline. Providers push `dynamic.Message` into a 100-capacity buffered channel; the watcher transforms and fans-out to registered listeners.
- `pkg/provider/aggregator/aggregator.go`: `ProviderAggregator.Provide()` calls each registered provider's `Provide(configChan, pool)` in a goroutine. All providers satisfy the single `Provider` interface (`Provide` + `Init`).
- `pkg/plugins/builder.go`: `NewBuilder` reads each plugin's manifest, selects Yaegi or Wasm runtime, returns a `Constructor func(ctx, http.Handler) (http.Handler, error)`. This is the formal microkernel extension contract for middleware plugins.
- `pkg/plugins/manager.go`: Downloads and hash-validates plugins from `https://plugins.traefik.io/public/`. Dynamic runtime loading — the binary need not be recompiled to add a plugin.
- `pkg/server/routerfactory.go`: `RouterFactory.CreateRouters(runtimeConfig)` — rebuilds all HTTP/TCP/UDP routers and service managers on every configuration reload. Holds `pluginBuilder` as an explicit dependency.
- `pkg/server/router/router.go`: `Manager.buildRouterHandler` chains middleware with the `alice` library: Recovery → Observability → AccessLog → EncodedDeny → user middlewares → service handler. Classic pipe-and-filter.
- `pkg/server/server_entrypoint_tcp.go`: `TCPEntryPoint` accept loop; `HandlerSwitcher` peeks first byte for TLS detection; bridges to `httpForwarder` (net.Listener over channel). Connection-level pipeline: accept → detect → route → HTTP server.
- `pkg/config/dynamic/config.go`: `Configuration{HTTP, TCP, UDP, TLS}` — the canonical runtime data model. `Message{ProviderName, Configuration}` is the event type flowing through the config pipeline.

### Patterns Found

**Microkernel (Primary):**

1. **Formal Provider interface** — `pkg/provider/provider.go` defines `Provider{Provide(chan<- dynamic.Message, *safe.Pool) error; Init() error}`. All 13+ built-in providers and plugin providers implement this identical contract. The core never references a concrete provider type.

2. **Runtime plugin loading** — `pkg/plugins/manager.go` downloads, validates (SHA256), and installs plugins at runtime from an external registry. `pkg/plugins/builder.go` instantiates them via Yaegi (Go interpreter) or Wazero (Wasm) without recompiling the binary. Two plugin types: `typeMiddleware` and `typeProvider`.

3. **Extension points in RouterFactory** — `routerfactory.go` carries `pluginBuilder PluginsBuilder` as an interface dependency, making plugins first-class in the routing assembly pipeline.

4. **ProviderAggregator** — `pkg/provider/aggregator/aggregator.go` treats all providers polymorphically. Built-in providers (Docker, Kubernetes, Consul, ECS, Nomad, File, HTTP, KV, REST, ACME, Tailscale) and plugin providers are indistinguishable from the aggregator's perspective.

5. **Single deployable binary** — The `Dockerfile` copies a single `traefik` binary. The `go.mod` module is `github.com/traefik/traefik/v3`. The core is fully functional with zero plugins installed.

**Pipeline (Secondary — request handling):**

1. **Middleware chain composition** — Every HTTP request passes through an ordered `alice` chain assembled at router build time: Recovery → Tracing/Metrics → AccessLog → EncodedCharacterDeny → user-defined middlewares (auth, ratelimiter, compress, redirect, headers, etc.) → service handler. The order is deterministic and each stage transforms or short-circuits the request.

2. **TCP connection pipeline** — `TCPEntryPoint` stages: `listener.Accept()` → deadline configuration → protocol sniffing (`HandlerSwitcher`) → TLS termination or passthrough → HTTP server dispatch. Each stage wraps the connection object.

3. **Configuration pipeline** — Provider events flow: `Provider.Provide()` → `dynamic.Message` channel → `receiveConfigurations` (validate + transform + deduplicate) → transformed channel → `applyConfigurations` (merge + model apply + listener notification) → `RouterFactory.CreateRouters()`.

4. **30+ middleware packages** — `pkg/middlewares/` contains independent, composable filter packages: `auth`, `ratelimiter`, `circuitbreaker`, `compress`, `redirect`, `headers`, `retry`, `buffering`, `capture`, `metrics`, `observability`, `ipallowlist`, `forwardedheaders`, `passtlsclientcert`, `grpcweb`, `mirror`, etc.

**Layered (Internal structural support):**

The package layout exhibits clear horizontal layering:
- Configuration Layer: `pkg/config/static/`, `pkg/config/dynamic/`
- Provider Layer: `pkg/provider/` (all discovery adapters)
- Server/Entrypoint Layer: `pkg/server/` (listener management)
- Router Layer: `pkg/server/router/` + `pkg/muxer/` (rule matching)
- Middleware Layer: `pkg/middlewares/` (request/response transformation)
- Service/Proxy Layer: `pkg/server/service/`, `pkg/proxy/` (backend forwarding)
- Transport Layer: `pkg/tcp/`, `pkg/udp/`, `pkg/tls/` (protocol primitives)

Dependencies flow strictly downward; upper layers consume lower layers but not vice versa.

## Architecture Styles Identified

### Microkernel (Primary)

The governing organizational principle is a stable core with formal, dynamically-loadable extension points. The core (`server.go`, entrypoints, config watcher) knows nothing about concrete providers, middleware implementations, or backend types — all are attached via interfaces. The plugin system goes beyond compile-time extension: Yaegi and Wasm runtimes allow third-party code to be injected at startup without binary modification. The `ProviderAggregator` pattern and the `RouterFactory`'s `pluginBuilder` dependency are the two clearest structural markers of microkernel organization.

### Pipeline (Secondary)

The request-handling path is a deterministic, ordered sequence of transformations. Each middleware in an `alice` chain receives an `http.Handler` and returns an `http.Handler` — the canonical pipe-and-filter composition. At the TCP level the connection passes through a similarly staged pipeline (accept → sniff → route → serve). The configuration update path is itself a pipeline with distinct receive, transform, and apply stages. This pattern is pervasive and deliberate, not incidental.

### Layered (Supporting)

The internal package structure is strictly layered. This is a structural property of the codebase rather than a primary architectural statement, but it reinforces the separation of concerns and enforces clean dependency direction from static config through providers and servers down to transport primitives.

## Quality Attributes

- **Extensibility:** Formal provider interface (13+ built-in adapters, unlimited plugin providers); Yaegi + Wasm middleware plugin runtime; Kubernetes Gateway API support alongside Ingress and CRDs.
- **Operability/Observability:** Full OpenTelemetry traces and metrics (`pkg/observability/`); Prometheus, Datadog, StatsD, InfluxDB2, OpenTelemetry metric backends; structured access logging with configurable redaction; REST API + web dashboard exposing live routing state.
- **Reliability:** Circuit-breaker middleware; health-check subsystem with configurable intervals; retry middleware; graceful shutdown with staged 10-second timeout; rate-limiting work queues in provider aggregator.
- **Security:** ACME/Let's Encrypt automatic TLS (`lego v4`); SPIFFE/SVID mTLS support; mutual TLS passthrough; IP allow-list and auth middleware; TLS minimum version and cipher-suite configuration; credential redaction in logged config.
- **Performance:** Multiple load-balancing algorithms (WRR, HRW, Least-Time, Power-of-Two-Choices, failover, mirror); HTTP/2 and HTTP/3 (QUIC via `quic-go`); keep-alive management; connection-level buffering; smart round-tripper.
- **Portability:** Cloud-agnostic; 13+ provider backends; single statically-linked binary; supports Docker, Kubernetes, Nomad, Consul, ECS, etcd, Tailscale; cross-platform (Linux, macOS, Windows, FreeBSD).
- **Maintainability:** Single Go module; 24 well-scoped `pkg/` packages; strict layering; extensive test coverage (`*_test.go` alongside every production file); `pkg/redactor/` for safe config logging.
- **Dynamism:** Full hot-reload of routing configuration without restart; providers independently push updates; configuration watcher merges partial updates from multiple providers into a unified runtime state.

## Classification Reasoning

The decisive classification question for Traefik is whether its primary organizing principle is the plugin extension model (Microkernel) or the request-handling pipeline (Pipeline). Both are strongly present; the question is which drives the macro-structure.

**Microkernel is primary** because it governs how the system is composed and extended at the architectural level: the core binary is intentionally minimal, all provider integrations are pluggable via a single interface, and the plugin subsystem enables runtime injection of both middleware and provider logic without binary modification. This is the reason Traefik can support Docker, Kubernetes, Consul, ECS, Nomad, etcd, Tailscale, and arbitrary future platforms without changes to its core. The architectural investment in `pkg/plugins/manager.go` (download, hash-validate, install lifecycle) and the two runtime environments (Yaegi, Wasm) confirm this is a designed-in extension model, not an accidental one.

**Pipeline is a strong secondary** because it describes the request-handling core precisely: every HTTP request traverses a fixed ordered chain assembled from independently composable filter units. The 30+ middleware packages are not part of the microkernel's extension point — they are built-in filter stages in a pipeline. The config-update flow is also a pipeline. This pattern is load-bearing and pervasive.

**Layered is a supporting qualifier** — the package topology enforces strict downward dependency flow, which is architecturally meaningful but subordinate to the two primary organizing principles.

**What was rejected:**

- *Microservices*: Single binary, single Go module, no inter-service network communication. Ruled out by deployment topology.
- *Modular Monolith*: Package boundaries exist but the runtime plugin loading (Yaegi + Wasm) crosses the line into true microkernel territory. Modular monoliths have compile-time-only extension; Traefik has genuine runtime extension.
- *Event-Driven*: Configuration updates flow through Go channels, which are used as an internal delivery mechanism, not the primary organizing principle. The `dynamic.Message` channel is infrastructure for the config pipeline, not an event bus architecture.
- *Service-Based*: No decomposition into separately deployed services. The web dashboard and API are embedded in the same binary.

Confidence 0.91: The Microkernel + Pipeline classification is strongly evidenced by both macro-structure (plugin manager, provider interface, aggregator) and micro-structure (alice middleware chain, TCPEntryPoint pipeline, config watcher stages). Minor gap accounts for the fact that Traefik is first and foremost a reverse proxy — its architecture is shaped by that domain function — and the Pipeline and Layered patterns are so intertwined that a case for Pipeline-primary could reasonably be argued.
