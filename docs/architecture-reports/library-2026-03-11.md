# library — Architecture Classification Report
**Date:** 2026-03-11
**Repository:** https://github.com/ddd-by-examples/library
**Primary Style(s):** Domain-Driven Design, Modular Monolith, Hexagonal Architecture, CQRS
**Confidence:** 0.97
**Summary:** A reference implementation of a public library system that demonstrates Domain-Driven Design tactical patterns (aggregates, domain events, value objects, bounded contexts) deployed as a Modular Monolith internally structured with Hexagonal Architecture, with explicit CQRS read/write model separation.

## Evidence

### Single deployable unit (Modular Monolith)
- One `pom.xml`, one `Dockerfile`, one `docker-compose.yml` with a single app service.
- `LibraryApplication.java` bootstraps a single Spring Boot process with two child application contexts (`LendingConfig`, `CatalogueConfiguration`) — bounded contexts are packages, not separate processes.

### Explicit modular boundaries enforced by ArchUnit
- `ModularArchitectureTest.java` uses ArchUnit rules to assert `catalogue` does not depend on `lending`, and `commons` depends on neither — module contracts are compile-time-verified.
- README states: "we decided to assign each bounded context to a separate package, which means that the system is a modular monolith."

### Domain-Driven Design: Bounded Contexts
- Two bounded contexts: `catalogue` (CRUD, no complex logic) and `lending` (rich domain model).
- Package layout reflects the linguistic boundary explicitly called out in EventStorming sessions and README.

### Domain-Driven Design: Aggregates, Value Objects, Domain Events
- `Patron` aggregate root encapsulates complex invariants: `placeOnHold`, `cancelHold`, `checkOut` as rich domain methods returning `Either` monads.
- Value objects throughout: `BookId`, `PatronId`, `LibraryBranchId`, `HoldDuration`, `CheckoutDuration`, `ISBN`.
- `PatronEvent` hierarchy (`BookPlacedOnHold`, `BookCheckedOut`, `BookHoldCanceled`, `BookHoldExpired`, `BookReturned`) models all state transitions as domain events.
- `PatronRepository.save(PatronEvent)` — repositories operate on events, making state changes explicit.

### Domain-Driven Design: Business Policies and Ubiquitous Language
- `PlacingOnHoldPolicy` functional interface with four concrete policy instances: `onlyResearcherPatronsCanHoldRestrictedBooksPolicy`, `overdueCheckoutsRejectionPolicy`, `regularPatronMaximumNumberOfHoldsPolicy`, `onlyResearcherPatronsCanPlaceOpenEndedHolds`.
- Type system enforces invariants at compile time (`placeOnHold(AvailableBook)` vs `placeOnHold(Book)`) — state machine modelled as distinct types `AvailableBook`, `BookOnHold`, `CheckedOutBook`.
- `PatronFactory` constructs aggregates with complex initialization.

### Hexagonal Architecture (Lending bounded context)
- `LendingHexagonalArchitectureTest.java` enforces: model has no dependency on application, infrastructure, or UI; application has no dependency on infrastructure or UI.
- Package structure: `model/` (domain), `application/` (use-case orchestration), `infrastructure/` (adapters, Spring, JDBC), `web/` (primary adapters).
- Domain interfaces (`FindAvailableBook`, `Patrons`, `BookRepository`) are implemented by infrastructure adapters (`BookDatabaseRepository`, `PatronsDatabaseRepository`) — dependency inversion enforced.
- Domain model contains no Spring annotations; `@EventListener`, `@Transactional`, `@SpringBootConfiguration` are restricted to infrastructure layer.

### CQRS: Separated read and write models
- README explicitly states: "we could also make a decision that we separate read and write models (CQRS)."
- **Write model**: rich `Patron` aggregate in `patron/model/` with direct JDBC persistence via `PatronsDatabaseRepository`.
- **Read models**: `PatronProfileReadModel` (patron profile queries from `holds_sheet`/`checkouts_sheet`) and `SheetsReadModel` (daily sheets) — separate projections updated by domain events via `@EventListener`.
- `SheetsReadModel` listens to `BookPlacedOnHold`, `BookHoldCanceled`, `BookHoldExpired`, `BookCheckedOut`, `BookReturned` to maintain denormalized read tables independently of the write model.

### Event-driven intra-process communication
- Aggregates communicate via domain events published through `DomainEvents` interface.
- Two consistency models demonstrated: immediate (Spring `ApplicationEventPublisher` via `JustForwardDomainEventPublisher`) and eventual (`StoreAndForwardDomainEventPublisher` with `@Scheduled` polling).
- `PatronEventsHandler` reacts to patron events to update the `BookRepository` state machine.

### No ORM, functional style
- No JPA/Hibernate; uses `JdbcTemplate` and Spring Data JDBC for fine-grained SQL control.
- Vavr library for immutable data structures and monadic types: `Either`, `Option`, `Try`, `List`.
- Pattern matching via Vavr `API.Match` for polymorphic event handling.
- EventStorming-driven process discovery documented in `docs/big-picture.md`, `docs/design-level.md`, `docs/example-mapping.md`.

## Architecture Styles

### Domain-Driven Design (Primary)
The entire codebase is structured around DDD tactical patterns. Bounded contexts (`lending`, `catalogue`) have distinct ubiquitous languages verified in EventStorming. `Patron` is the central aggregate with full behavior encapsulation. Domain events drive cross-aggregate communication. The repository pattern accepts events rather than entities, making state transitions explicit. Business rules are modelled as first-class domain concepts (`PlacingOnHoldPolicy`). The project description, docs, and README all explicitly identify this as a DDD reference implementation.

### Modular Monolith (Primary co-style)
A single Spring Boot process hosts two bounded contexts as separate Spring application contexts sharing one JVM process. The README explicitly names this a modular monolith. ArchUnit tests enforce module isolation rules. Each context has its own `@Configuration` and database configuration, achieving "microservices-style autonomy" within a single deployment unit.

### Hexagonal Architecture (Secondary)
Applied rigorously within the `lending` bounded context. ArchUnit tests (`LendingHexagonalArchitectureTest`) enforce strict layer isolation: `model` → `application` → `infrastructure`, no reverse dependencies. Domain interfaces define ports; infrastructure classes define adapters. Spring annotations are fully absent from the domain model.

### CQRS (Secondary)
Explicit separation of read and write models within the `lending` context. The write model processes commands through the `Patron` aggregate; separate `PatronProfileReadModel` and `SheetsReadModel` projections are maintained by subscribing to domain events. The README explicitly calls out this design decision.

## Quality Attributes

- **Testability**: Domain model is free of Spring and I/O, enabling pure unit tests without mocking. ArchUnit enforces architectural constraints as automated tests. Spock/Groovy BDD specs closely mirror EventStorming scenarios.
- **Maintainability**: Bounded contexts with enforced module isolation reduce cross-context coupling. Ubiquitous language consistency between model and code minimizes the model-code gap.
- **Modularity**: Package-level module boundaries enforced by ArchUnit; each bounded context is independently evolvable and extractable to a microservice.
- **Flexibility**: Two consistency models (immediate/eventual) implemented with a single `DomainEvents` interface, allowing infrastructure swap without domain changes.
- **Correctness by construction**: Type system prevents invalid state transitions at compile time (`AvailableBook` vs `BookOnHold` vs `CheckedOutBook`). Either monads make failure paths explicit.
- **Observability**: Micrometer/Prometheus metrics on `DomainEventPublisher` (`MeteredDomainEventPublisher`). Grafana dashboard included.
- **Evolvability**: Architecture explicitly designed so context-packages can be extracted into microservices. Deferred consistency model choice is a design principle.
- **Separation of concerns**: Read/write model separation (CQRS) isolates query optimization from command processing.

## Classification Reasoning

This repository is an unambiguous, explicitly-documented reference implementation of **Domain-Driven Design** deployed as a **Modular Monolith** using **Hexagonal Architecture** with **CQRS** for read/write separation.

**DDD is the primary organizing principle.** Every structural decision traces back to DDD concepts: bounded contexts discovered via EventStorming, aggregates with rich behavior, domain events as the primary inter-aggregate communication channel, value objects, repositories, factories, and business policies modelled as first-class domain concepts. The README, docs, and package structure are all DDD-first.

**Modular Monolith is the deployment and structural topology.** The project deploys as a single Spring Boot application (`LibraryApplication.java`) with one `pom.xml` and one `Dockerfile`. The two bounded contexts (`lending`, `catalogue`) are package-level modules with ArchUnit-enforced isolation rules, not separate deployable services. The README explicitly states this is "a modular monolith" and notes it is designed to be split into microservices if needed.

**Hexagonal Architecture is the intra-context structural pattern for the `lending` context.** The domain model layer has no Spring, no JDBC, no infrastructure dependencies — enforced by `LendingHexagonalArchitectureTest`. The application layer orchestrates use cases against domain interfaces (ports). The infrastructure layer provides adapters. The `catalogue` context uses a simpler CRUD local architecture, demonstrating that each bounded context can choose its own local architecture.

**CQRS is applied within the `lending` context.** The write model (`Patron` aggregate, `PatronsDatabaseRepository`) is completely separate from the read models (`PatronProfileReadModel`, `SheetsReadModel`). Read models are maintained by domain event subscriptions, not by querying the write model. The README explicitly calls this a CQRS decision.

**What this is not:** Not Microservices (single deployable, bounded contexts are packages, shared JVM process). Not a traditional Layered architecture (no top-down layer organization; hexagonal provides better insulation). Not Event-Driven Architecture in the macro sense (events are intra-process via Spring ApplicationEvents, not a message broker; the primary deployment is a single app, not event-choreographed services).

The high confidence (0.97) reflects that the README, docs, ArchUnit tests, package structure, and code all consistently and explicitly document the architectural choices with zero ambiguity.
