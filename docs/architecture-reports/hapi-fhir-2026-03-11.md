# Architecture Report: hapi-fhir

**Date:** 2026-03-11
**Repository:** https://github.com/hapifhir/hapi-fhir
**Classification:** Layered, Modular Monolith
**Confidence:** 0.92
**Model:** claude-sonnet-4-6

---

## Summary

HAPI FHIR is the reference open-source Java implementation of the HL7 FHIR standard, maintained by Smile CDR. It is a Maven multi-module project with 60+ modules (~2,974 non-test Java source files) that provides both a client library and a full JPA-backed FHIR server. Its dominant architectural style is **Layered**: the entire system is organized into strict horizontal tiers — FHIR model/structures, REST framework, storage API interfaces, JPA implementation, interceptor cross-cut, and batch processing — where each tier has well-defined inbound dependency rules and upward dependencies are prohibited. The secondary style is **Modular Monolith**: while the Maven module graph enforces hard package-boundary contracts, everything compiles and deploys as a single WAR/JAR artifact (the JPA server), with all modules sharing a common Spring application context.

## Styles Identified

### 1. Layered (Primary — strong evidence)

**Evidence:**

The entire module dependency graph is organized as unidirectional horizontal layers. Maven `<dependency>` declarations in each module's `pom.xml` enforce this: `hapi-fhir-jpaserver-base` depends on `hapi-fhir-storage` (the storage API layer), which depends on `hapi-fhir-base` (the core), but never vice versa.

**Layer 1 — Core/Domain:**
`hapi-fhir-base` provides the `FhirContext` singleton (the central registry for FHIR version metadata, parsers, and client/server factories), the interceptor API (`Pointcut` enum with 107 pointcut entries, `IInterceptorService`, `BaseInterceptorService`), and base annotations. The `hapi-fhir-structures-*` modules provide the auto-generated FHIR resource POJOs (Patient, Observation, etc.). This layer has zero framework dependencies beyond Jakarta EE annotations.

**Layer 2 — REST Framework:**
`hapi-fhir-server` provides `RestfulServer` — a `HttpServlet` subclass that dispatches HTTP requests to `IResourceProvider` instances via reflective `BaseMethodBinding` subclasses (`SearchMethodBinding`, `CreateMethodBinding`, `ReadMethodBinding`, etc.). Resource providers are plain POJOs annotated with `@Search`, `@Create`, `@Read`, etc. The server knows nothing about storage; it only knows about the `IResourceProvider` contract. The client framework (`hapi-fhir-client`) mirrors this with `IGenericClient` and pluggable HTTP adapters (Apache HttpComponents 5, OkHttp).

**Layer 3 — Storage API / Domain Model:**
`hapi-fhir-storage` declares the storage contract interfaces: `IFhirResourceDao<T>`, `IFhirSystemDao`, and the full `Pointcut` hook set for storage operations (`STORAGE_PRESTORAGE_RESOURCE_CREATED`, `STORAGE_PRESEARCH_REGISTERED`, etc.). `hapi-fhir-jpaserver-model` defines the JPA entity model: `ResourceTable` (the root resource entity, annotated `@Entity @Indexed` for Hibernate Search), `ResourceIndexedSearchParam*` entities, `TagDefinition`, `ResourceLink`, and all index tables. This layer defines what storage looks like without implementing it.

**Layer 4 — JPA Implementation:**
`hapi-fhir-jpaserver-base` contains the concrete DAO implementations: `BaseHapiFhirResourceDao<T>` extends the abstract DAO base, uses Hibernate/JPA to persist `ResourceTable`, invokes `InterceptorBroadcaster` at every Pointcut, and delegates search to `SearchCoordinatorSvcImpl`. The search coordinator uses `SearchBuilder` (which constructs SQL predicates via a `QueryStack` / Spring Data JPA criteria API), and persists `Search` entities for async paging. The Spring `@Configuration` classes (`JpaConfig`, `SearchConfig`, `Batch2SupportConfig`) wire all dependencies.

**Layered evidence in the interceptor system:**
The `Pointcut` enum categorizes hooks explicitly by layer:
- `INTERCEPTOR_xxx` — interceptor framework hooks
- `CLIENT_xxx` — Layer 2 REST client hooks
- `SERVER_xxx` — Layer 2 REST server hooks
- `SUBSCRIPTION_xxx` — Layer 4 subscription hooks
- `STORAGE_xxx` — Layer 3/4 storage hooks
- `VALIDATION_xxx` — Layer 2 validation hooks
- `JPA_PERFTRACE_xxx` — Layer 4 JPA performance tracing hooks

This explicit namespacing by layer confirms the horizontal decomposition is intentional and enforced.

**Layered evidence in DAOs:**
The inheritance chain is vertical-within-layer:
```
IFhirResourceDao<T>                  (Layer 3 — interface)
  └─ BaseHapiFhirResourceDao<T>      (Layer 4 — abstract, JPA operations)
       └─ BaseJpaResourceProvider    (Layer 4 — provider adapter)
```
`BaseJpaResourceProvider` sits at the junction of Layer 2 (server) and Layer 4 (JPA), implementing `IResourceProvider` (Layer 2) while delegating to `IFhirResourceDao` (Layer 3).

### 2. Modular Monolith (Secondary — strong evidence)

**Evidence:**

Despite being organized as 60 Maven modules, HAPI FHIR's JPA server assembles into a single deployable unit. The `hapi-fhir-jpaserver-uhnfhirtest` module (the reference HAPI test server at hapi.fhir.org) and the `hapi-fhir-cli` module both bundle all modules into a single WAR or Spring Boot fat JAR.

The `JpaConfig.java` Spring `@Configuration` class in `hapi-fhir-jpaserver-base` is the integration hub: it wires hundreds of beans (DAOs, caches, batch jobs, subscription services, search services, validation services, MDM services) into a single Spring application context. There is no runtime service mesh, no independent deployment per module, and no network boundary between modules at runtime.

The Maven module boundaries are enforced at compile time through `<dependency>` declarations and Checkstyle rules (`hapi-fhir-checkstyle`), providing the "modular" discipline. Each module has its own `pom.xml` and produces its own JAR artifact — but these JARs are dependencies of each other, not independently deployable services.

The Batch2 framework (`hapi-fhir-storage-batch2`) illustrates this well: jobs are defined as `JobDefinition` beans in Spring context, steps run as beans within the same JVM, and work-chunk state is stored in the same database (not via external message broker). While an optional `IChannelFactory` SPI allows routing work chunks via a message channel, the default is in-memory/database-backed coordination.

The multi-FHIR-version support is achieved through parallel module sets (`hapi-fhir-structures-r4`, `hapi-fhir-structures-r5`, etc.) all compiled into the same runtime classpath, with `FhirContext.forR4()` / `FhirContext.forR5()` choosing the active model at application startup.

### What This Is Not

**Not Microservices:** All server modules run in a single Spring application context inside a single JVM/WAR. There is no independent deployment pipeline per module. Subscriptions are not independent services — they run within the JPA server's own thread pools.

**Not Microkernel:** Unlike Backstage, there is no plugin registration mechanism, no extension point primitive, and no kernel/plugin isolation boundary. `IResourceProvider` is a strategy interface consumed by the server framework, but providers cannot be hot-installed or replaced at runtime without restarting the application context. The interceptor system uses hook registration (`registerInterceptor`), not a kernel plugin contract.

**Not Event-Driven:** FHIR Subscriptions deliver notifications via HTTP callbacks, WebSockets, or message channels to external systems, but the internal architecture does not use an event bus as its primary integration pattern. The Batch2 work-channel (`IChannelProducer`/`IChannelConsumer`) is an optional async work distribution mechanism, not the dominant communication model. All primary CRUD and search operations are synchronous.

**Not Pipeline:** The Batch2 `JobDefinition` step chain (e.g., `generate-ranges → load-ids → reindex → pending-work-check`) is a genuine pipeline within the batch subsystem, but it is a subordinate implementation pattern, not the system's overarching architecture.

**Not Hexagonal:** There are no ports-and-adapters boundaries. The `IFhirResourceDao` interface is a storage abstraction (similar to a repository pattern), but it is not an explicit "port" separating domain logic from infrastructure — the DAO itself contains infrastructure logic. The `IResourceProvider` interface is a server extension point, not a domain boundary.

## Classification Reasoning

HAPI FHIR's architecture is definitively **Layered** as its primary pattern, with **Modular Monolith** as the secondary packaging and deployment pattern. The layered verdict is supported by:

1. The Maven module dependency graph enforces a strict directional layer hierarchy: core → REST framework → storage API → JPA implementation, with no upward or cross-layer dependencies
2. The `Pointcut` enum's explicit layer-namespaced categories (`CLIENT_`, `SERVER_`, `STORAGE_`, `JPA_PERFTRACE_`) reveal the intentional horizontal decomposition
3. The DAO inheritance chain (`IFhirResourceDao` interface in Layer 3, `BaseHapiFhirResourceDao` implementation in Layer 4) is a textbook layered architecture pattern
4. The FHIR model/structures modules (Layer 1) have zero server or storage dependencies — they are pure domain objects
5. The server framework (Layer 2) knows nothing about JPA or database — it interacts exclusively through `IResourceProvider` and the interceptor contract

The Modular Monolith classification applies because the 60 Maven modules compile down to a single WAR/fat JAR with a unified Spring application context. There is no runtime module isolation, no service mesh, and no independent deployment per module. The `JpaConfig` Spring configuration is the anti-pattern of a modular monolith integration hub: it wires everything together in a single context.

The prior classification (from 2026-03-09) correctly identified Layered and Modular Monolith. This deep-analysis confirms and refines that classification with stronger evidence grounding and higher confidence (0.92 vs 0.85), and corrects the model identifier.

Confidence: **0.92** — the layered pattern is unambiguous from both the Maven module graph and the internal class hierarchy; the modular monolith characterization is equally clear from the single Spring context assembly pattern.
