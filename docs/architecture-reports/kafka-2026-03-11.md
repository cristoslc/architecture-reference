## Style Rationales

**Modular Monolith (primary, 0.95):** Apache Kafka is built as a single deployable unit (one Gradle root project, one `kafka.tgz` tarball, one entry point at `core/src/main/scala/kafka/Kafka.scala` launching `KafkaRaftServer`) with explicit, statically-enforced module boundaries. The `settings.gradle` defines 40+ subprojects organized into a deliberate dependency hierarchy: `clients` (Java/C++ client APIs) → `common` (protocol, serialization, errors) → `server-common`/`metadata`/`storage` (server infrastructure) → `coordinator-common`/`group-coordinator`/`transaction-coordinator`/`share-coordinator` (coordination layer) → `raft` (KRaft consensus) → `core` (Scala broker, now in gradual migration to Java `server` module). Every module boundary is enforced at compile time by 17 separate Checkstyle import-control XML files in `checkstyle/`, each bearing the comment `<!-- THINK HARD ABOUT THE LAYERING OF THE PROJECT BEFORE CHANGING THIS FILE -->`. The root `import-control.xml` (665 lines) establishes the global rule `<disallow pkg="kafka" />` — no Java module may depend on the Scala `kafka.*` core — and enumerates precisely which `org.apache.kafka.*` packages are public surfaces. Dedicated files enforce boundaries for `metadata`, `server`, `server-common`, `storage`, `group-coordinator`, `transaction-coordinator`, `coordinator-common`, `share-coordinator`, and `test-common`. This compile-time boundary enforcement is the canonical distinguishing mark of a Modular Monolith versus an ad-hoc monolith.

**Microkernel (secondary, 0.90):** Two distinct plugin systems coexist within the monolith. First, **Kafka Connect** implements a formal microkernel at `connect/api/` (SPI) and `connect/runtime/` (core system). The `connect/api/` module defines extension interfaces: `Connector`, `Task`, `SourceConnector`, `SinkConnector`, `Transformation<R>`, `Predicate<R>`, `Converter`, and `HeaderConverter`. The runtime loads plugins via `connect/runtime/src/main/java/org/apache/kafka/connect/runtime/isolation/Plugins.java`, which uses `DelegatingClassLoader.java` (a custom classloader with comment: "A custom classloader dedicated to loading Connect plugin classes in classloading isolation") and `PluginClassLoader.java` for per-plugin isolation. Plugin discovery is dual-mode: `ServiceLoaderScanner.java` and `ReflectionScanner.java`. Built-in reference plugins (`connect/file/`, `connect/json/`, `connect/transforms/`) demonstrate 15+ concrete transform implementations (Cast, Filter, Flatten, InsertField, ReplaceField, TimestampConverter, etc.) following the SPI contract. Second, **Kafka Streams** exposes a `Processor` SPI (`streams/src/main/java/org/apache/kafka/streams/processor/api/Processor.java`) and a `ProcessorSupplier` factory pattern, allowing user-defined stateful processors to be injected into the streaming topology via `StreamsBuilder.addProcessor()`. The `ProcessorTopology` class (`processor/internals/ProcessorTopology.java`) composes `ProcessorNode`, `SourceNode`, and `SinkNode` objects into a directed graph executed by the Streams runtime — a textbook microkernel where the core runs an extension-composed topology.

**Pipeline (secondary, 0.85):** Kafka Streams (`streams/`) implements a Pipe-and-Filter execution model directly within the monolith. Users compose a stream-processing pipeline via the high-level DSL (`KStream`, `KTable`, `GlobalKTable`, `StreamsBuilder`) or the low-level Processor API. `StreamsBuilder` translates DSL operations into an `InternalTopologyBuilder` that constructs a `ProcessorTopology` — a directed graph of `ProcessorNode<K,V,KR,VR>` objects (sources, processors, sinks) connected by topic channels. The `StreamThread` (`processor/internals/StreamThread.java`) drives the pipeline: it polls records from Kafka topics (acting as pipe inputs), routes them through the processor graph node by node, and writes outputs back to Kafka topics (pipe outputs) or state stores. This pipeline execution model is intrinsic to the Streams subsystem but does not govern the broker core, which uses direct method calls and synchronous request processing.

## Quality Attributes

| QA | Evidence |
|---|---|
| **Durability** | Append-only log segments with configurable replication factor (default 3); ISR (In-Sync Replicas) protocol ensures writes are committed only when all ISR members have persisted; leader epoch tracking in `storage/` prevents stale leaders from corrupting the log |
| **Scalability** | Horizontal partitioning as the primary scaling unit: topics are split into partitions, each independently replicated and consumed; partition reassignment redistributes load across brokers; consumer groups provide parallel consumption; KRaft (`raft/`) eliminates the ZooKeeper metadata bottleneck |
| **Fault Tolerance** | KRaft Raft consensus (`KafkaRaftClient.java`) provides leader election and log replication for metadata; ISR-based replication for data topics; unclean leader election configurable; exactly-once semantics via idempotent producers and transactional APIs (`transaction-coordinator/`); Kafka Streams checkpoint-based state store recovery |
| **Extensibility** | Kafka Connect plugin SPI with isolated classloaders for connectors, transforms, converters, and predicate filters; Kafka Streams `Processor` SPI for user-defined stateful operators; `LogCleanerPlugin`, authorizer, and quota plugin interfaces in server; 16 import-control files preserve module boundaries for internal extensibility |
| **Throughput** | Zero-copy network transfers via `sendfile()` syscall for consumer fetch paths; batch compression (gzip, Snappy, LZ4, Zstd via `common/compress/`); record batching in producers with `linger.ms`/`batch.size`; sequential disk I/O on log segments for predictable performance |
| **Observability** | JMX metrics exposed via Yammer Metrics throughout broker, clients, and Streams (`ClientMetricsManager.java`); client-side telemetry push via KIP-714 (`ClientMetricsManager` in `server/`); configurable log4j2 logging per component; `connect/runtime/` exposes REST API for connector status and health |
| **Evolvability** | Scala `core/` → Java `server/` migration in progress with import-control boundaries preventing reverse coupling; KRaft removes ZooKeeper operational dependency (KIP-500); `@ApiVersion` protocol versioning on all request/response types in `common/message/`; module boundaries enforced at compile-time allow internal refactoring without external API breakage |

## Domain

Distributed commit log and event streaming platform. Core domains: partitioned append-only log storage and replication (ISR protocol, leader election), KRaft consensus-based metadata management (replacing ZooKeeper), producer/consumer client protocol (fetch, produce, offset management), Kafka Connect (connector plugin framework for data integration), Kafka Streams (embedded stream processing library with stateful operators and exactly-once guarantees), and transaction/share coordinator services.

## Production Context

- KRaft mode is the current production default; ZooKeeper mode is deprecated and scheduled for removal
- Kafka Connect distributed mode (`connect-distributed.properties`) supports multi-broker connector cluster with REST-based management API
- Kafka Streams runs as an embedded library in user applications; multiple instances form a consumer group for parallel partition processing
- The Scala `core/` module is being incrementally migrated to Java `server/`; import-control files enforce that new Java code cannot depend on the Scala core, creating a one-way migration path
- Tiered storage (`storage/` remote log subsystem) allows offloading older log segments to object storage (S3, GCS, Azure) while keeping hot segments local
